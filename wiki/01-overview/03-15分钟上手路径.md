# 15 分钟上手路径

- 状态：当前事实


> 给第一次进入仓库的开发者。15 分钟走完一遍完整循环：打开编辑器 → 跑一次验证 → 读一次日志 → 知道下一步去哪里。

---

## 前置

- Windows 10/11 + PowerShell 7+
- Godot 4.6.1 安装在 `E:\SDK\Godot\`（路径可调，记得同步命令）
- 仓库已 clone 到本地

---

## 第 1 步（2 分钟）：打开项目

方法 A — 编辑器：

```
E:\SDK\Godot\Godot_v4.6.1-stable_win64.exe --path E:\Code\open_pvz
```

方法 B — 只跑一次 headless 冒烟，验证启动无报错：

```powershell
& 'E:\SDK\Godot\Godot_v4.6.1-stable_win64_console.exe' --headless --path 'E:\Code\open_pvz' --quit-after 3
```

预期：控制台有 `[Protocol]` 或 `[RuntimeSnapshot]` 输出，无红色报错即通过。

---

## 第 2 步（3 分钟）：看主入口

用编辑器打开以下三处，知道"项目从哪里长出来"：

- [scenes/main/main.tscn](../../scenes/main/main.tscn) —— 默认启动场景（Showcase Hub）
- [scenes/demo/demo_level.tscn](../../scenes/demo/demo_level.tscn) —— 可操作演示
- [autoload/](../../autoload/) —— 全局单例目录

按 F5 运行主场景：应看到 Showcase Hub 列表，能进入 demo_level 并放植物打僵尸。

---

## 第 3 步（3 分钟）：跑一次单场景验证

```powershell
& 'E:\Code\open_pvz\tools\run_validation.ps1' -Scenario 'res://scenes/validation/minimal_battle_validation.tres'
```

预期输出：

- `artifacts/validation/<timestamp>_minimal_battle_validation/` 下出现：
  - `validation_report.json`（状态 `passed`）
  - `debug_logs.json`（事件 / 触发器 / 效果日志）

如果状态是 `failed`，打开 `validation_report.json` 看 `rule_results` 里哪条不过；再对照 `debug_logs.json` 的 `event_log` 找原因。

---

## 第 4 步（3 分钟）：读一次日志

打开刚产出的 `debug_logs.json`，重点看三处：

1. **`event_log`** —— 倒序列出所有事件。过滤 `event_name == "projectile.hit"` 看命中次数。
2. **`effect_log`** —— 每个效果的 `success` / `notes`，确认 damage / spawn_projectile / explode 执行情况。
3. **`protocol_log`** —— 如果非空，说明有协议违规（模板字段错 / 引用断链等），需要先修。

读懂这三个，90% 的战斗运行时问题都能定位。详细字段见 [调试与日志观察](../02-runtime-protocol/12-调试与日志观察.md)。

---

## 第 5 步（2 分钟）：跑一次批量回归

```powershell
& 'E:\Code\open_pvz\tools\run_all_validations.ps1'
```

53 个场景并行跑完约 1-3 分钟，产物在 `artifacts/validation/batch_<timestamp>/`。

看 `batch_summary.json` 的 `passed` / `failed` 计数。提交主干改动前应当跑到全绿。

---

## 快速故障排查

| 症状 | 先查 |
| --- | --- |
| `run_validation.ps1` 找不到 Godot | 编辑 `tools/run_validation.ps1` 顶部 `$Godot` 路径 |
| 启动时大量 `[Protocol][ERROR]` | `SceneRegistry` 扫描 `data/combat/` 时发现无效 `.tres`，通常是某个字段类型错 |
| 验证 `failed` 但行为看似正确 | `validation_rules` 断言过严或过松，先读 `rule_results` 再看 `event_log` |
| 批量回归里多个场景超时 | 单跑那个场景的 `.tscn`，看是否死循环或事件链深度爆炸 |

---

## 不要做的事

- 不要直接修改 `vendor/` 下的参考实现
- 不要用 `print` 代替 `DebugService`（产物不会进回归）
- 不要改冻结协议（`periodically` / `when_damaged` / `on_death` / `damage` / `spawn_projectile` / `explode`）而不走设计审批
- 不要跳过 `.tres` 直接把内容写死在 GDScript
