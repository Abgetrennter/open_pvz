# Open PVZ 系统版图与规划分层

- 状态：当前事实

> 本文回答“项目当前有哪些系统层、每一层现在处于什么状态”。它不再承担历史阶段叙事，也不再替旧路线做状态判断。

---

## 当前系统分层

### 1. 引擎运行时层

回答“规则如何运行”。

当前包含：

- 事件总线与协议守卫
- Archetype + Mechanic 编译链
- Trigger / Effect 运行时
- Controller / State / Lifecycle
- 投射体轨迹、命中与高度模型
- 确定性随机与 mechanic-local runtime state

当前状态：**已进入稳定主干**

### 2. 战斗玩法层

回答“PVZ-like 战斗如何成立”。

当前包含：

- 阳光与资源
- 棋盘、槽位、占位与放置
- 卡片、费用、冷却与手牌
- 波次、目标与胜败条件
- 状态系统
- 场上物件与机关

当前状态：**已进入主干，并有批量验证覆盖**

### 3. 正式内容与扩展层

回答“具体植物、僵尸、场上物件、关卡和扩展包如何落在主干上”。

当前包含：

- `data/combat/` 正式资源树
- `51` 个 archetype
- `extensions/` 下的最小包、样例包和守卫包

当前状态：**已形成第一轮正式内容基线**

### 4. 展示、验证与治理层

回答“系统如何被观察、验证和维护”。

当前包含：

- Showcase Hub 与 Demo 入口
- `76` 个验证场景
- 批量回归脚本
- Wiki、术语表、ADR 与归档体系

当前状态：**已成为主干组成部分**

### 5. 外围组织层

回答“战斗之外的产品壳如何组织内容”。

当前包含：

- 选关
- 存档与进度
- 图鉴
- 商店
- 花园
- 模式入口与长期解锁结构

当前状态：**已识别，但仍明显后置**

## 当前最容易失衡的地方

### 1. 把 legacy 兼容层重新当成作者入口

当前 `EntityTemplate / TriggerBinding` 只允许以兼容层语义出现，不能再回到正文主叙事。

### 2. 把专项验证误读成 family 已经全部收口

当前虽已有 `Placement` 相关专项验证，但正文仍将其视为待补完 family，而不是已完全收口。

### 3. 把展示层当成正式内容目录

`scenes/showcase/` 负责展示和导航，不负责定义正式内容归属。

---

## 当前结论

从系统层面看，当前项目的核心矛盾已经不是“主线是否切换”，而是：

1. 如何把 `Placement` 从专项路径收口成正式 family。
2. 如何把 legacy 兼容层从“仍然存在”收紧成“只读兼容层”。
3. 如何让内容、扩展和验证继续按同一口径增长。
