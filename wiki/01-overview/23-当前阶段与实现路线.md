# 当前阶段与实现路线

> 这篇文档回答的不是“项目最终会多大”，而是“以当前仓库真实实现状态看，我们现在到底做到哪一步、接下来该按什么顺序推进”。

---

## 文档定位

这篇文档主要回答：

- 当前阶段到底是什么
- 当前已经完成了哪些主干能力
- 当前还缺哪些关键闭环
- 接下来应该先做什么、后做什么

这篇文档不负责：

- 详细列出所有阶段验收
- 替代协议层和验证层文档

---

## 当前阶段的一句话判断

当前项目已经越过“纯概念和纯原型起步期”，但还没有进入“内容扩展期”。

更准确地说，当前处于：

> 最小规则运行时主干已经成立，空间模型和验证入口也已经进入代码，项目正处在“主干稳固期后段，向内容模板期过渡”的阶段。

这句话里的重点有三个：

1. 现在已经不是只有文档和想法。
2. 现在也还不是可以大规模堆内容的时候。
3. 当前最重要的是把主干、验证和模板入口接稳，而不是过早扩张。

---

## 当前阶段最容易出现的误解

### 误解 1：以为现在在做完整开放引擎

不准确。

当前还没有：

- 完整模板体系
- 完整规则协议校验
- 完整错误技生成器
- 完整扩展加载协议

### 误解 2：以为现在只是在做一个 debug demo

也不准确。

当前已经有：

- 结构化事件模型
- 统一触发器和效果执行链
- 投射体连续行为主干
- 逻辑高度与投影模型
- 自动化验证入口

这已经不是单纯 demo，而是一个正在收敛的引擎主干。

### 误解 3：以为当前主线还是“随机植物系统”

不准确。

错误技和随机组合仍然是重要验证方向，但当前主线是：

- 先把开放式规则运行时主干做稳
- 再让错误技建立在同一主干之上

---

## 当前已经完成的关键主干

### 1. 结构化事件与执行链

当前已经具备：

- `EventBus`
- `EventData`
- `RuleContext`
- `TriggerRegistry`
- `EffectRegistry`
- `EffectExecutor`

当前最小闭环已经能跑通：

```text
game.tick
-> Trigger
-> Effect
-> spawn_projectile / damage / explode
-> projectile.hit / entity.damaged / entity.died
```

### 2. 统一实体运行时状态

当前已经有：

- `EntityState`
- `BaseEntity`

实体的 `id / kind / team / lane / status / position / health / values` 已有统一承载，不再完全靠散落字典。

### 3. 连续行为与投射体主干

当前已经有：

- `ProjectileRoot`
- `ProjectileMoveResult`
- 线性、追踪、抛物线三种 movement strategy

而且这部分已经不是旧的“纯 2D 位置补丁”，而是：

- `ground_position`
- `height`
- `projected_position`

这意味着当前连续行为主干已经进入真正的空间模型阶段。

### 4. 命中模型与高度协议

当前已经落地：

- `HeightBand`
- `ProjectileFlightProfile`
- 地面平面 + 高度范围命中
- 终点命中策略

这让远距离抛射体和不同轨迹的命中判断不再依赖屏幕投影偶然重叠。

### 5. 验证与调试设施

当前已经有：

- `minimal_battle_validation`
- `parabola_long_range_validation`
- `height_hit_validation`
- `lane_isolation_validation`
- `swept_segment_validation`
- `terminal_explode_validation`
- `BattleValidationRule`
- `run_validation.ps1`
- `run_all_validations.ps1`
- `DebugService`
- `DebugOverlay`
- runtime snapshot logging

这意味着当前主干已经具备：

- 可重复运行的场景
- 自动判定通过/失败的入口
- 可追踪的事件、触发器、效果和实体状态

---

## 当前还没有完成的关键工作

虽然主干已经成立，但下面几件事仍然没有完成。

### 1. 模板体系还没有真正进入生成主链

当前 `BattleScenario` 已经资源化了：

- 生成条目
- 高度带
- 飞行策略
- 验证规则

当前还新增了一个最小模板接入口：

- `BattleSpawnEntry -> EntityTemplate -> BattleManager -> EntityFactory`
- 模板 id 已可进入运行时事件链

但它仍然不是完整的 `EntityTemplate` 实例化链。

也就是说：

- 场景资源已经不再是纯硬编码脚本
- 但真正的植物模板、僵尸模板、投射物模板入口还没有完全接上

### 2. 规则协议已经有第一版白名单，但还没进入完整协议期

当前已经有：

- `TriggerDef`
- `EffectDef`
- 内置触发器和效果参数白名单
- `spawn_projectile.on_hit` 子效果白名单
- `flight_profile` / `HeightBand` 资源脚本类型校验

但还没有完全完成：

- 非法组合拦截
- 更完整的基础效果和触发器白名单扩展
- 更完整的负向专项验证矩阵

### 3. 错误技还没有正式建立在当前主干之上

当前运行时已经能表达：

- 周期发射
- 受伤反击
- 死亡爆炸

但这还只是最小行为验证，不等于错误技系统已经真正接入。

### 4. 验证体系还没有完全收口

当前已经有：

- 单场景自动化入口
- 批量验证入口
- 验证状态汇总
- 高度命中专项验证
- 多 lane 交叉专项验证
- 高速扫掠命中专项验证
- 终点爆炸专项验证
- 模板实例化专项验证
- 规则协议负向专项验证

但还没有完全补齐：

- 更完整的主干回归记录与状态维护
- 更多非法组合类型的规则协议专项
- 完整模板工厂与更强模板装配能力

---

## 当前阶段的正确推进顺序

结合现在这版实现状态，当前最合理的推进顺序已经不是最初那种“从零开始”的表述，而应该改成下面这样。

### 第一步：继续收紧主干协议

重点：

- 事件字段语义
- `RuleContext` 继承规则
- `TriggerDef / EffectDef` 的真实角色
- 投射体命中与高度规则

目标：

- 让当前已经跑起来的主干不再继续漂移

### 第二步：把模板体系真正接上场景生成链

重点：

- `EntityTemplate` 真正进入实例化流程
- `BattleScenario` 不再直接承担实体细节装配
- 植物 / 僵尸 / 投射物的差异尽量下沉到模板

目标：

- 让当前资源化工作从“局部参数资源”升级成“实体模板资源”

### 第三步：做第一批手工错误技样例

重点：

- 不急着上完整随机生成
- 先做几组明确的非常规行为模板

目标：

- 证明当前运行时不仅能复刻普通射击，还能承载明显不同的行为组合

### 第四步：再考虑扩展加载和工具链

重点：

- 数据包协议
- 更强调试工具
- 编辑器和可视化支持

目标：

- 建立在稳定协议之上，而不是把工具链反过来变成主线

---

## 当前推荐的技术取舍

### 1. Godot 节点 + 组件 + Autoload 仍然是对的

目前没有迹象表明需要回退或推翻这条路线。

当前这套组合已经证明可以承载：

- 事件链
- 实体状态
- 投射体
- 验证场景

### 2. `Resource` 仍然优先于外部 JSON

当前仓库已经进一步证明这点是对的，因为现在已经实际使用：

- `BattleScenario`
- `BattleSpawnEntry`
- `BattleValidationRule`
- `HeightBand`
- `ProjectileFlightProfile`

这说明第一阶段优先把内部资源协议写稳，确实比先冲外部格式更合理。

### 3. “逻辑 3D + 2D 投影”已经成为当前正确方向

至少对投射体主干来说，这已经不是设想，而是现有实现现实。

后续所有涉及弹道、命中、遮挡和高度的问题，都应默认建立在这套模型上，而不是继续回到纯 2D 屏幕坐标补丁。

### 4. 调试与验证已经是主干的一部分

当前不应再把日志、验证场景、自动化入口理解为“辅助工作”。

它们现在已经是：

- 主干正确性的一部分
- 空间与时序问题可解释性的基础

---

## 当前阶段不该成为主线的内容

下面这些内容仍然不应主导当前设计：

- 完整 ECS 化
- 高级渲染与批处理架构
- 大型编辑器工具链
- 社区分享平台
- 排行榜和工坊系统
- 最终版扩展包生态

原因没有变：

- 当前阶段这些方向的收益，仍然远低于把规则主干、模板入口和验证体系收稳

---

## 当前阶段的阶段性验收

如果按现在这版实现来看，当前阶段已经满足的部分包括：

- 事件链可广播并追踪
- 基础效果可稳定执行
- 投射物可移动、命中并回写事件
- 最小验证场景可重复运行
- 自动化专项验证已经存在
- 批量回归入口已经存在
- 高度命中和 lane 隔离专项验证已经存在
- 高速扫掠命中专项验证已经存在
- 终点爆炸专项验证已经存在
- 最小模板实例化专项验证已经存在
- 主干已开始使用正式 `Resource`

但当前阶段还不能宣告“完全完成”，因为还缺：

- 真正的模板实例化链
- 第一批正式错误技样例
- 更多非法组合专项验证与后续协议扩展

所以更准确地说：

> 当前第一阶段已经进入后段，但尚未完全收口。

---

## 下一阶段真正该做什么

当当前阶段继续推进时，最合理的下一组工作不是“继续加功能”，而是：

1. 模板化实体定义
2. 收紧规则协议
3. 建立第一批手工错误技样例
4. 扩充验证矩阵

只有当这些东西成立之后，下面这些内容才值得成为下一阶段重点：

- 更完整的错误技生成器
- 数据包 / 扩展包加载
- 更强的调试面板
- 编辑器和可视化工具

这里的顺序仍然不能颠倒：

- 先稳协议
- 再接模板
- 再证表达力
- 最后做生态
