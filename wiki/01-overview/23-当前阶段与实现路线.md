# 当前阶段与实现路线

> 这篇文档回答的不是“项目最终会多大”，而是“以当前仓库真实实现状态看，我们现在到底做到哪一步、接下来该按什么顺序推进”。

---

## 文档定位

这篇文档主要回答：

- 当前阶段到底是什么
- 当前已经完成了哪些主干能力
- 当前还缺哪些关键闭环
- 接下来应该先做什么、后做什么

这篇文档不负责：

- 详细列出所有阶段验收
- 替代协议层和验证层文档

---

## 当前阶段的一句话判断

当前项目已经完成第四阶段（战斗玩法系统接入期）的核心收口，同时第五阶段（错误技、扩展与工具期）的核心起步也已经完成收口。

更准确地说，当前处于：

> 第一到第四阶段已经形成稳定主干，第五阶段错误技、扩展入口和扩展 effect 已进入正式回归；项目当前主线应转入第五阶段的 `v1` 规范化推进段。

这句话里的重点有三个：

1. 现在已经不是只有文档和想法。
2. 现在也还不是可以无边界堆内容的时候。
3. 当前最重要的是收敛协议、扩展边界、验证分层和内容工作流，而不是回到主干补丁式开发。

---

## 当前阶段最容易出现的误解

### 误解 1：以为现在在做完整开放引擎

不准确。

当前还没有：

- 完整 `v1` 协议冻结
- 完整内容作者工作流
- 完整随机错误技生成器
- 完整社区工坊与完整外循环系统

### 误解 2：以为现在只是在做一个 debug demo

也不准确。

当前已经有：

- 结构化事件模型
- 统一触发器和效果执行链
- 投射体连续行为主干
- 逻辑高度与投影模型
- 自动化验证入口

这已经不是单纯 demo，而是一个正在收敛的引擎主干。

### 误解 3：以为当前主线还是“随机植物系统”

不准确。

错误技和随机组合仍然是重要验证方向，但当前主线是：

- 先把开放式规则运行时主干做稳
- 再让错误技建立在同一主干之上

## 当前已经完成的关键主干

### 1. 结构化事件与执行链

当前已经具备：

- `EventBus`
- `EventData`
- `RuleContext`
- `TriggerRegistry`
- `EffectRegistry`
- `EffectExecutor`

当前最小闭环已经能跑通：

```text
game.tick
-> Trigger
-> Effect
-> spawn_projectile / damage / explode
-> projectile.hit / entity.damaged / entity.died
```

### 2. 统一实体运行时状态

当前已经有：

- `EntityState`
- `BaseEntity`

实体的 `id / kind / team / lane / status / position / health / values` 已有统一承载，不再完全靠散落字典。

### 3. 连续行为与投射体主干

当前已经有：

- `ProjectileRoot`
- `ProjectileMoveResult`
- 线性、追踪、抛物线三种 movement strategy

而且这部分已经不是旧的“纯 2D 位置补丁”，而是：

- `ground_position`
- `height`
- `projected_position`

这意味着当前连续行为主干已经进入真正的空间模型阶段。

### 4. 命中模型与高度协议

当前已经落地：

- `HeightBand`
- `ProjectileFlightProfile`
- 地面平面 + 高度范围命中
- 终点命中策略

这让远距离抛射体和不同轨迹的命中判断不再依赖屏幕投影偶然重叠。

### 5. 验证与调试设施

当前已经有：

- `minimal_battle_validation`
- `parabola_long_range_validation`
- `height_hit_validation`
- `lane_isolation_validation`
- `swept_segment_validation`
- `terminal_explode_validation`
- `BattleValidationRule`
- `run_validation.ps1`
- `run_all_validations.ps1`
- `DebugService`
- `DebugOverlay`
- runtime snapshot logging

这意味着当前主干已经具备：

- 可重复运行的场景
- 自动判定通过/失败的入口
- 可追踪的事件、触发器、效果和实体状态

## 第二阶段完成结果回顾与后续边界

第一阶段已经完成。第二阶段当前也已经完成收口，下面这组内容更适合作为“第二阶段已经完成了什么”和“第三阶段开始前还应关注什么”的回顾。

### 1. 模板体系还需要从“可用”走向“可规模化”

当前 `BattleScenario` 已经资源化了：

- 生成条目
- 高度带
- 飞行策略
- 验证规则

当前已经形成一条最小完整模板链：

- `BattleSpawnEntry -> EntityTemplate -> BattleManager -> EntityFactory`
- 模板 id 已可进入运行时事件链
- 模板定义的运行时 trigger 已可由工厂产出

但它仍然不是大规模内容扩展阶段需要的完整模板系统。

也就是说：

- 场景资源已经不再是纯硬编码脚本
- 但真正的植物模板、僵尸模板、投射物模板入口还没有完全接上

### 2. 规则协议已经有第一版白名单，但还没进入完整协议期

当前已经有：

- `TriggerDef`
- `EffectDef`
- 内置触发器和效果参数白名单
- `spawn_projectile.on_hit` 子效果白名单
- `flight_profile` / `HeightBand` 资源脚本类型校验

但还没有完全完成：

- 非法组合拦截
- 更完整的基础效果和触发器白名单扩展
- 更完整的负向专项验证矩阵

### 3. 错误技已经正式接上主干，但还没进入规模化生成期

当前运行时已经正式落地并接入回归的包括：

- 受伤反击
- 死亡爆炸
- 空地追踪
- 多源级联
- 多目标溅射
- 高速追踪
- 多 lane 反击隔离

同时扩展入口已经承接：

- 命中分裂
- 周期召唤
- 状态控制
- 击退
- 跳链
- 光环
- 延迟触发
- 延迟爆炸
- 标记

这说明错误技和扩展 effect 已经正式建立在当前主干之上；当前真正还没做的是：

- 规模化内容生产工作流
- 更明确的 `v1` 协议冻结
- 完整随机生成器

### 4. 验证体系仍需继续扩展覆盖面

当前已经有：

- 单场景自动化入口
- 批量验证入口
- `latest / history / per-scenario` 回归状态账本
- 验证状态汇总
- 46 个唯一验证场景 ID
- 第四阶段玩法层正式回归区块
- 第五阶段错误技与扩展专项回归区块
- 扩展 effect 守卫专项

但还没有完全补齐：

- 回归层级划分与分层命令
- 新能力进入矩阵的统一维护规则
- 更标准化的内容接入与作者工作流

## 当前阶段的正确推进顺序

结合现在这版实现状态，当前最合理的推进顺序已经不是最初那种“从零开始”的表述，而应该改成下面这样。

### 第一步：收敛 `v1` 协议与稳定边界

重点：

- 事件字段语义
- `RuleContext` 继承规则
- `TriggerDef / EffectDef` 的真实角色
- 模板字段与扩展包 manifest 口径

目标：

- 让当前已经跑起来的主干不再继续漂移，并给后续扩展设定第一版稳定边界

### 第二步：把扩展入口和内容工作流标准化

重点：

- 扩展包目录约定
- manifest 字段约定
- 内容作者接入步骤
- 样例、验证、showcase 的同步维护口径

目标：

- 让“新增一个 pack / 一个样例 / 一个扩展 effect”从个人经验变成标准流程

### 第三步：把验证入口拆成可维护的分层回归

重点：

- `smoke / core / extension / guardrail`
- 主干专项与扩展专项的分层运行命令
- 新能力接入矩阵和负向守卫的入场规则

目标：

- 让“该跑什么验证、何时跑、跑完看什么”不再依赖协作者口头经验

### 第四步：继续收敛主干编排边界，而不是回退到大管理器

重点：

- `BattleManager` 当前职责边界
- 可下沉的运行时编排职责
- 新能力的入场规则

目标：

- 在不做大重构的前提下，降低后续两个月继续把逻辑堆回编排层的风险

### 第五步：只给外循环系统预留接口位

重点：

- 选关与模式入口
- 存档与进度
- 图鉴、商店、花园
- 战斗结果输出与外围数据边界

目标：

- 先让后续系统在整体设计里有位置，而不是现在就抢占第五阶段主线

## 当前推荐的技术取舍

### 1. Godot 节点 + 组件 + Autoload 仍然是对的

目前没有迹象表明需要回退或推翻这条路线。

当前这套组合已经证明可以承载：

- 事件链
- 实体状态
- 投射体
- 验证场景

### 2. `Resource` 仍然优先于外部 JSON

当前仓库已经进一步证明这点是对的，因为现在已经实际使用：

- `BattleScenario`
- `BattleSpawnEntry`
- `BattleValidationRule`
- `HeightBand`
- `ProjectileFlightProfile`

这说明第一阶段优先把内部资源协议写稳，确实比先冲外部格式更合理。

### 3. “逻辑 3D + 2D 投影”已经成为当前正确方向

至少对投射体主干来说，这已经不是设想，而是现有实现现实。

后续所有涉及弹道、命中、遮挡和高度的问题，都应默认建立在这套模型上，而不是继续回到纯 2D 屏幕坐标补丁。

### 4. 调试与验证已经是主干的一部分

当前不应再把日志、验证场景、自动化入口理解为“辅助工作”。

它们现在已经是：

- 主干正确性的一部分
- 空间与时序问题可解释性的基础

---

## 当前阶段不该成为主线的内容

下面这些内容仍然不应主导当前设计：

- 完整 ECS 化
- 高级渲染与批处理架构
- 大型编辑器工具链
- 社区分享平台
- 排行榜和工坊系统
- 最终版扩展包生态

原因没有变：

- 当前阶段这些方向的收益，仍然远低于把规则主干、模板入口和验证体系收稳

- 投射物可移动、命中并回写事件
- 最小验证场景可重复运行
- 自动化专项验证已经存在
- 批量回归入口已经存在
- 高度命中和 lane 隔离专项验证已经存在
- 高速扫掠命中专项验证已经存在
- 终点爆炸专项验证已经存在
- 模板实例化和模板工厂专项验证已经存在
- 规则协议负向专项验证已经存在

## 第三阶段当前最合理的拆解

结合当前路线图、协议输入清单和设计目标，第三阶段当前更适合拆成下面 4 条主线，而不是继续泛化成“把所有规则都协议化”。

### 1. 先收口第一轮冻结范围

当前第三阶段最重要的第一步不是新增更多触发器和效果，而是明确：

- 第一轮只冻结当前高频主链
- 只覆盖真实样例已经用到的字段和组合
- 明确哪些内容后置到第四阶段或下一轮协议扩展

如果这一步不先做，第三阶段会很容易演变成：

- 局部协议越来越细
- 但玩法系统仍然没有正式进入建设

### 2. 冻结当前高频触发器和效果子集

第一轮更适合优先冻结：

- 触发器：`periodically / when_damaged / on_death`
- 效果：`damage / spawn_projectile / explode`

这里的重点不是“数量够多”，而是：

- 字段语义稳定
- 默认值来源稳定
- 资源类型约束稳定
- 当前样例全部能跑在这套正式子集上

### 3. 用验证矩阵把正向和负向边界一起收口

第三阶段不能只靠文档列白名单，还必须让下面两类东西同时成立：

- 正向样例能稳定通过
- 非法组合能稳定被拦截

这意味着第三阶段的核心产出之一，仍然是：

- 协议守卫专项
- 批量回归矩阵
- 明确可解释的失败结果

### 4. 继续克制非主线扩写

当前仍然不建议把下面这些内容拉进第三阶段主线：

- 阳光、卡片、格子、放置、波次和胜负流程协议
- 全量效果宇宙和全量触发器宇宙
- 外部扩展包最终格式
- 大型编辑器、DSL 和工具链

原因不是这些不重要，而是：

- 当前第三阶段的任务是“收口主链协议”
- 不是“提前做完整产品版图”

## 第三阶段当前完成状态

结合当前路线图、协议文档、负向守卫专项和验证矩阵，这一轮回头看，第三阶段第一轮冻结主线已经完成了核心收口。

当前已经成立的结果主要包括：

1. 第一轮冻结范围已经明确，只收口当前高频主链，不再把更多未来系统混入第三阶段。
2. 第一版触发器白名单已经明确：
   - `periodically`
   - `when_damaged`
   - `on_death`
3. 第一版效果白名单已经明确：
   - `damage`
   - `spawn_projectile`
   - `explode`
4. 默认值来源、覆盖优先级、资源类型约束和高频组合约束已经写成正式边界。
5. `TriggerBinding` 已成为当前正式行为主入口，旧兼容入口已经退出当前协议面。
6. 规则协议守卫和模板守卫专项已经能稳定拦截当前高频非法组合。
7. 当前高频内容样例已经基本落在这套冻结子集内，而不是继续依赖新的临时特判。

更准确地说，按第三阶段收官当时的历史交接口径，项目当时已经不是“刚进入第三阶段”，而是：

> 第三阶段已经完成，项目当前应转入第四阶段玩法系统接入前的前置准备。

这段历史交接结论的重点有三个：

1. 第三阶段最关键的协议收口和兼容清理已经完成。
2. 当前下一步应是为第四阶段玩法系统建设做前置准备。
3. 当前最需要警惕的，不是协议不够多，而是玩法系统一接入就重新把主干打散。

按当时阶段切换点，最合理的剩余工作顺序应理解成：

- 先沉淀第三阶段冻结结论
- 再整理第四阶段玩法系统清单与接入依赖
- 最后再让玩法系统正式进入建设

如果要回看第三阶段收官时的交接文档，建议配合阅读：

- [第四阶段前置清单](../../plans/第四阶段前置清单.md)
- [第四阶段第一批可执行任务清单](../../plans/第四阶段第一批可执行任务清单.md)

- 为了“更完整”继续无限扩写触发器和效果宇宙

- 让错误技生成器、扩展包和大型工具链重新变成当前主线

## 第二阶段真正该做什么

第二阶段当前已经完成的收口结果包括：

1. 扩展了 `EntityTemplate` 的内容表达能力和装配边界。
2. 建立了一批可复用的植物 / 僵尸 / 投射体 / 高度带内容样例。
3. 扩展了正向专项验证与负向守卫专项。
4. 整理了第三阶段协议冻结输入清单。

因此当前最合理的下一组工作，已经不再是继续补第二阶段缺口，而是转向第三阶段主线：

- 冻结第一版触发器协议
- 冻结第一版效果协议
- 固化默认值来源、参数上下界和组合约束

只有当这些东西成立之后，下面这些内容才值得成为下一阶段重点：

- 更完整的错误技生成器
- 数据包 / 扩展包加载
- 更强的调试面板
- 编辑器和可视化工具

这里的顺序仍然不能颠倒：

- 先稳协议
- 再接模板
- 再证表达力
- 最后做生态
