# 术语表

- 状态：当前事实


> 这篇文档用于统一项目里的高频名词，避免同一个词在不同 wiki、代码和讨论里逐渐漂移。

---

## 文档定位

这篇文档主要回答：

- 当前项目里常用名词分别指什么
- 哪些词属于概念层，哪些词属于实现层
- 哪些说法已经不建议继续使用

这篇文档不负责：

- 详细展开某一系统的完整设计
- 代替协议文档和方法论文档

---

## 一、项目治理术语

### 主干

指当前项目中最核心、最需要先稳定下来的运行时与验证基础设施。

在当前仓库里，主干至少包括：

- 事件链
- 触发器与效果执行
- Archetype + Mechanic 编译链
- Controller / State / Lifecycle
- 实体状态
- 空间与命中模型
- 验证入口

### 内容

指建立在主干之上的具体单位、场景、archetype 和数值配置。

### 验证场景

主要用于验证某一能力是否成立、是否可重复、是否可自动回归的场景。

### 重大决策记录

用于记录系统级选择的文档。当前模板见 [重大决策记录模板](31-重大决策记录模板.md)，已决定记录见 [决策记录目录](../decisions/README.md)。

### 验证矩阵

用于记录"能力域 -> 场景 -> 命令 -> 观察点 -> 当前状态"的工程表。见 [验证矩阵](../03-content-validation/32-验证矩阵.md)。

## 二、方法论术语

### 概念层

回答"世界里到底有什么"。

例如：

- 地面位置
- 高度
- 事件
- Archetype
- Mechanic

### 协议层

回答"这些概念在数据和接口上如何表达"。

例如：

- `EventData`
- `RuleContext`
- `CombatArchetype`
- `CombatMechanic`
- `RuntimeSpec`

### 实现层

回答"代码里怎样把协议跑起来"。

例如：

- `EventBus.gd`
- `mechanic_compiler.gd`
- `entity_factory.gd`

### 验证层

回答"如何证明实现符合协议"。

### 内容层

回答"主干之上具体生成什么单位和玩法"。

## 三、编译链与 Mechanic 术语

### `CombatArchetype`

顶层内容作者入口，替代旧的 `EntityTemplate` 作为内容到运行时的正式起点。定义实体的身份、节点、放置、战斗属性和行为（通过 `mechanics[]`）。

三个子类型：`PlantArchetype`、`ZombieArchetype`、`ProjectileArchetype`。

### `CombatMechanic`

沿行为维度的可替换模块，通过 `family / type_id / params` 三层结构组织。

- **family**（行为维度）：10 个冻结 family（Trigger / Targeting / Emission / Trajectory / HitPolicy / Payload / State / Lifecycle / Placement / Controller）
- **type**（实现族）：family 下的具体实现，如 `core.bite`、`lane_forward`
- **param**（配置）：type 的具体参数，如 `{ damage: 10, interval: 1.0 }`

### `MechanicCompiler`

核心编译引擎（`scripts/core/runtime/mechanic_compiler.gd`）。将 Archetype 的 Mechanic 数组编译为 `RuntimeSpec`。内置 37 个 mechanic type 的编译逻辑。

### `NormalizedMechanicSet`

编译中间产物。过滤 disabled mechanic、按 priority 排序、合并 params 和 compiler_hints。

### `RuntimeSpec`

编译最终产物（`scripts/core/runtime/runtime_spec.gd`）。由 `EntityFactory` 消费，包含实例化实体所需的全部信息：trigger bindings、controller specs、state specs、mechanic runtime states 等。

### `CombatContentResolver`

内容解析器，负责将 spawn entry 解析为 archetype 或旧 template。

### `MechanicFamilyRegistry`

10 个冻结 Mechanic family 的注册中心（autoload）。

### `MechanicTypeRegistry`

family → type_id 的注册中心（autoload）。委托 `MechanicCompiler` 注册内置 type。

### `MechanicCompilerRegistry`

per-type compiler callable 的注册与分发中心（autoload）。

### 编译链

内容到运行时的多阶段编译流程：

```
Archetype + Mechanic[] → NormalizedMechanicSet → RuntimeSpec → Runtime Nodes
```

## 四、Controller / State / Lifecycle 术语

### `Controller`（Mechanic family）

连续行为状态机 family，将原来硬编码在实体根脚本中的持续行为抽取为可组合模块。通过 `ControllerComponent` 每帧执行。

已注册 type：`core.bite`（僵尸啃食）、`core.sweep`（割草机扫掠）。

### `State`（Mechanic family）

阶段状态 family，管理实体的阶段性状态转换（如 arming、growth、rage）。通过 `StateComponent` 支持时间触发和事件触发转换。

### `Lifecycle`（Mechanic family）

生命周期钩子 family（`on_spawned`、`on_place`、`on_armed` 等）。不是独立组件——编译时映射为 Trigger family 的特殊触发。

### `ControllerComponent`

挂载在实体上的 Controller 运行时组件（`scripts/components/controller_component.gd`）。每帧通过 `ControllerRegistry` 分发执行。

### `StateComponent`

挂载在实体上的 State 运行时组件（`scripts/components/state_component.gd`）。管理状态转换，支持时间触发和事件触发。

### `ControllerRegistry`

Controller 策略注册中心（autoload）。

## 五、确定性随机术语

### 确定性随机协议

所有随机行为通过 seed 派生链保证可复现：

```
GameState.battle_seed → entity_seed → mechanic_seed → 局部 RNG / ShuffleBag
```

### `ShuffleBag`

确定性随机工具类（`scripts/core/runtime/shuffle_bag.gd`）。使用 Fisher-Yates 洗牌，耗尽后自动重洗。作为 mechanic runtime state 绑定到实体上。

### Mechanic-local runtime state

每个 Mechanic 在运行时可持有的私有状态（如 ShuffleBag 实例、burst 队列等）。存储在 `RuntimeSpec.mechanic_runtime_states` 中。

## 六、运行时与规则术语

### `EventBus`

当前运行时事件广播和分发中心。

### `EventData`

结构化事件对象。当前至少承载：

- `event_name`
- `event_id`
- `chain_id`
- `depth`
- `timestamp`

### `RuleContext`

规则执行上下文，用于承载当前事件、源对象、目标对象和运行时派生信息。

### `TriggerDef`

触发器定义。它描述触发器的协议和参数语义。

### `TriggerInstance`

触发器在运行时上的具体实例。

### `TriggerComponent`

挂在实体上的触发器承载组件，负责让实体接入触发器执行链。

### `EffectDef`

效果定义。它描述效果的参数结构和执行约束。

### `EffectNode`

效果树中的一个节点，用来描述某个效果及其子效果关系。

### `EffectRegistry`

效果注册中心。已注册：`damage`、`spawn_projectile`、`explode`、`apply_status`、`produce_sun`、`spawn_entity`。

### `EffectExecutor`

效果执行器，负责按上下文和链深规则真正执行效果。

## 七、实体与内容术语

### `EntityState`

统一的实体运行时状态承载结构。

### `BaseEntity`

实体运行时根节点基类，负责统一挂接实体状态和核心组件协作。

### `EntityTemplate`

旧实体模板资源。当前作为后端兼容层存在，新内容应使用 `CombatArchetype`。

### `EntityFactory`

实体生成工厂。双路径：Archetype → RuntimeSpec 路径优先，旧 EntityTemplate 路径兜底。

### `TriggerBinding`

旧行为绑定资源。已降级为后端兼容层，新内容行为通过 Mechanic 定义。

### `ProjectileTemplate`

投射体内容模板资源。

### `ProjectileFlightProfile`

投射体飞行策略资源。

### `BattleScenario`

战斗场景资源。

### `BattleSpawnEntry`

战斗场景里的单条生成配置。

### `BattleValidationRule`

场景级验证规则资源。

### `DetectionRegistry`

目标发现策略注册中心（autoload）。已注册：`lane_forward`、`lane_backward`、`always`。

## 八、空间与投射体术语

### 投射体

一切"被发射出去、沿某种轨迹持续运动，并在命中、终点或寿命结束时结算结果"的运行时对象。对应 `ProjectileRoot`。

### `ground_position`

逻辑地面位置。用于寻敌、运动、命中。

### `height`

逻辑高度。用于区分离地状态和命中高度规则。

### `projected_position`

投影后的屏幕位置。用于 2D 场景渲染，不应参与命中逻辑。

### `HeightBand`

单位命中高度范围资源。

### `ProjectileMoveResult`

投射体移动结果对象。

## 九、扩展包术语

### 数据扩展包

只提供资源和定义，不提供新执行代码的扩展包。

### 代码扩展包

允许提供新执行逻辑的扩展包。

### 包类型

ADR-005 定义的 5 种包类型：`content_pack`、`rule_pack`、`runtime_pack`、`asset_pack`、`collection_pack`。

### 信任级别

ADR-005 定义的 3 种信任级别：`data_only`、`rule_extended`、`trusted_runtime`。

### 命名空间注册

注册中心使用的命名空间规则：`core.*` 主仓独占，`extension_namespace.*` 扩展包使用。

## 十、状态词与文档词

### 已决定

已成为项目默认约束，除非有新的决策记录，否则不应随意漂移。

### 当前建议

当前阶段推荐这样做，但仍可能根据实现进展调整。

### 未来方向

长期规划或可能的下一步，不代表当前仓库已经实现。

### 部分过时

某篇文档或某个说法仍有参考价值，但不能再直接当作当前规范使用。

## 十一、不建议继续使用的含混说法

### "EntityTemplate 是正式内容入口"

不建议。当前正式入口是 `CombatArchetype + CombatMechanic[]`。`EntityTemplate` 已降级为后端兼容层。

### "TriggerBinding 是唯一行为入口"

不建议。行为通过 Mechanic 定义，编译链将其转化为运行时 trigger/controller/state。

### "屏幕位置就是命中位置"

不建议。当前主干已区分 `ground_position`、`height`、`projected_position`。

### "子弹"可以直接代替"投射体"

不建议在协议和架构讨论里混用。讲统一运行时和模板边界时更推荐说"投射体"。

### "BattleManager 就是内容系统"

不成立。`BattleManager` 是场景运行与验证协调者。
