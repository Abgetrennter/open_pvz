# 原版实体复刻工作流

> 这篇文档定义从参考项目（`vendor/de-pvz` 和 `vendor/PVZ-Godot-Dream`）提取原版 PVZ 实体信息，并将其翻译为 Open PVZ 资源模板的标准化工作流。

---

## 文档定位

这篇文档主要回答：

- 复刻一个原版实体时，从哪里提取什么数据
- 如何将原版数值映射到 Open PVZ 的资源字段
- 按什么顺序复刻实体（按协议可映射性分批）
- 遇到当前协议不支持的行为时怎么处理

这篇文档不负责：

- 重新定义模板字段或协议边界
- 替代 [模板编写约定](35-模板编写约定.md)
- 规划协议扩展的具体实现方案

如果需要先理解模板字段怎么写，请先看：

- [Archetype 编写约定](35-模板编写约定.md)
- [编译链与 Mechanic 系统](../02-runtime-protocol/11-编译链与Mechanic系统.md)

## 一句话原则

> **de-pvz 是规格书，PVZ-Godot-Dream 是参考实现，Open PVZ 的资源模板是目标格式。**

- 提数值和行为规格 -> 查 `de-pvz`
- 理解 Godot 层面如何落地 -> 查 `PVZ-Godot-Dream`
- 落笔写 `.tres` -> 按 Open PVZ 的命名约定和字段规范

## 1. 数据源定位

### 1.1 de-pvz（权威规格）

| 数据类别 | 文件位置 | 提取内容 |
|----------|----------|----------|
| 植物定义表 | `Lawn/Plant.cpp` 的 `gPlantDefs[]`（L24-78） | cost、refresh ticks、subclass、launchRate |
| 植物运行时 HP | `Lawn/Plant.cpp` 的 `PlantInitialize()` | 各植物 HP（默认 300，WallNut=4000 等） |
| 植物行为逻辑 | `Lawn/Plant.cpp` 的 `UpdateShooter()` / `DoSpecial()` 等 | 射击、生产、爆破、接触击杀等行为分支 |
| 僵尸定义表 | `Lawn/Zombie.cpp` 的 `gZombieDefs[]`（L20-54） | value、startingLevel、pickWeight |
| 僵尸运行时 HP | `Lawn/Zombie.cpp` 的 `ZombieInitialize()`（L85-300） | bodyHP、helmHP、shieldHP、各护甲类型 |
| 僵尸行为逻辑 | `Lawn/Zombie.cpp` 的各 Update 方法 | 移动、啃食、特殊能力、阶段转换 |
| 投射物定义表 | `Lawn/Projectile.cpp` 的 `gProjectileDefinition[]`（L14-29） | damage 值 |
| 投射物运动 | `Lawn/Projectile.cpp` 的 `UpdateMotion*()` | 运动参数（速度、抛物线参数等） |
| 全部枚举 | `ConstEnums.h`（1362 行） | SeedType、ZombieType、ProjectileType、ProjectileMotion 等 |
| 游戏常量 | `GameConstants.h`（58 行） | 棋盘尺寸、计时参数、阳光间隔 |
| 波次数据 | `Lawn/Challenge.cpp` 的 `gZombieWaves[]`、`gZombieAllowedLevels[]` | 50 关波次构成、僵尸解锁关卡 |

**关键换算**：原版以 tick 为单位（100 ticks = 1 秒），Open PVZ 以秒为单位。

```
秒 = ticks / 100
refresh_time(s) = mRefreshTime / 100
attack_interval(s) = mLaunchRate / 100
sun_produce_interval(s) = mLaunchRate / 100  （Sunflower=25s, SunShroom=25s, Marigold=25s）
```

## 2. Open PVZ 运行时机制须知

在开始复刻之前，必须理解 Open PVZ 中几个**与原版不同的硬编码行为**。这些不是可选的，而是当前引擎的实际运行方式。

### 2.1 僵尸近战啃食通过 Controller 实现

所有僵尸（无论模板如何配置）都通过 **Controller.bite** Mechanic 获得近战啃食能力（ADR-004）。啃食由 `ControllerComponent` 在 `_physics_process` 中驱动：

- 每 physics 帧扫描同车道、前方的敌人
- 找到目标后停止移动，以指定间隔造成伤害
- 找不到目标则持续向左移动
- 受到状态效果影响（减速、眩晕）

参数通过 Mechanic params 或 Archetype `default_params` 配置：

| 参数 | 默认值 | 配置方式 |
|------|--------|---------|
| `damage` | 10 | Mechanic params 或 default_params |
| `interval` | 1.0 | Mechanic params 或 default_params |
| `range` | 56.0 | Mechanic params 或 default_params |
| `move_speed` | 55.0 | Archetype default_params |

**关键结论**：僵尸不需要 Trigger Mechanic 来啃食。Trigger Mechanic 只用于**远程攻击**和**被动响应**。

### 2.2 向日葵产阳光通过触发器系统实现

产阳光通过 `periodically` 触发器 + `produce_sun` 效果实现，走标准执行链：

```
game.tick → EventBus → TriggerComponent → TriggerInstance(periodically, start_delay) → EffectExecutor → produce_sun → BattleEconomyState.spawn_sun()
```

向日葵的 `plant_produce_sun` TriggerBinding 配置：

| 参数 | 值 | 说明 |
|------|------|------|
| `condition_values.interval` | 10.0 | 产阳光间隔（秒） |
| `condition_values.start_delay` | 10.0 | 首次产阳光延迟（秒） |
| `condition_values.detection_id` | `always` | 不需要检测敌人 |
| `effect_id` | `produce_sun` | 生成阳光效果 |
| `effect_params.value` | 25 | 每次产阳光的值 |
| `effect_params.offset_y` | -36.0 | 生成位置 Y 偏移 |

**关键结论**：产阳光植物**使用 trigger_bindings**。可以与触发器系统完全组合。`Twin Sunflower` 可通过 spawn_overrides 覆盖 `value=50` 实现（单份大阳光而非双份小阳光）。

### 2.3 投射物系统限制

| 能力 | 状态 | 说明 |
|------|------|------|
| 单发直线 | 支持 | `linear` + `swept_segment` |
| 单发抛物线 | 支持 | `parabola` + `terminal_hitbox` / `terminal_radius` |
| 追踪弹 | 支持 | `track` 模式自动寻敌 |
| 溅射爆炸 | 支持 | `on_hit` 嵌套 `explode` 效果 |
| 命中减速/眩晕 | 支持 | `on_hit` 嵌套 `apply_status` 效果 |
| **连射多发** | **已支持** | Emission.shuffle_cycle / burst，multi-payload 编译 |
| **随机效果** | **已支持** | 确定性随机（ShuffleBag） |
| **穿透** | **不支持** | `_on_hit()` 立即 `_consumed = true` |
| **效果链式嵌套** | **受限** | 只有 `spawn_projectile` 有 `on_hit` 子槽 |

### 2.4 检测系统

`periodically` trigger 的 `detection_id` 参数：

| 值 | 行为 |
|----|------|
| `always` | 不检测敌人，无条件触发（用于召唤、自爆等） |
| `lane_forward` | 检测同车道、源实体**右侧**的敌人（植物用） |
| `lane_backward` | 检测同车道、源实体**左侧**的敌人（僵尸远程攻击用；也用于裂荚射手打身后） |

**注意方向**：`lane_forward` = delta.x >= 0（目标在右方），`lane_backward` = delta.x <= 0（目标在左方）。对于植物，`lane_forward` 检测僵尸；`lane_backward` 检测**已经越过植物的僵尸**。

## 3. 数值映射表

### 3.1 植物：原版 -> Open PVZ

| 原版字段 | 原版来源 | Open PVZ 字段 | Open PVZ 资源类型 | 换算 |
|----------|----------|---------------|-------------------|------|
| `mSeedCost` | `gPlantDefs[]` | `sun_cost` | `CardDef` | 直接使用 |
| `mRefreshTime` (ticks) | `gPlantDefs[]` | `cooldown_seconds` | `CardDef` | /100 |
| `mPlantHealth` | `PlantInitialize()` | `max_health` | `EntityTemplate` | 直接使用 |
| `mLaunchRate` (ticks) | `gPlantDefs[]` | `condition_values.interval` | `TriggerBinding` | /100 |
| `mSubClass` | `gPlantDefs[]` | 决定是否需要 attack binding | -- | SHOOTER -> 需要 |
| 投射物 damage | `gProjectileDefinition[]` | `effect_params.damage` 或 `default_params.damage` | `TriggerBinding` 或 `ProjectileTemplate` | 直接使用 |
| 投射物 motion | `ProjectileMotion` enum | `move_mode` + `height_strategy` | `ProjectileFlightProfile` | 见 §3.3 |
| 产阳光间隔 | `mLaunchRate=2500` ticks | `condition_values.interval` | `TriggerBinding` | /100 |
| 产阳光值 | 硬编码 25 | `effect_params.value` | `TriggerBinding` | 直接使用 |

### 3.2 僵尸：原版 -> Open PVZ

| 原版字段 | 原版来源 | Open PVZ 字段 | Open PVZ 资源类型 | 换算 |
|----------|----------|---------------|-------------------|------|
| body + helm + shield HP | `ZombieInitialize()` | `max_health` | `EntityTemplate` | 当前合并为总 HP；护甲分层后拆分 |
| 啃食间隔 | `TICKS_BETWEEN_EATS=4` ticks | `default_params.attack_interval` | `EntityTemplate` | /100 = 0.04s |
| 啃食伤害 | `DAMAGE_PER_EAT=4` | `default_params.attack_damage` | `EntityTemplate` | 直接使用 |
| 移动速度 | `PickRandomSpeed()` + 各僵尸覆盖 | `default_params.move_speed` | `EntityTemplate` | 需实测校准（见下方） |
| 减速系数 | `CHILLED_SPEED_FACTOR=0.4` | apply_status 的 `movement_scale=0.4` | `TriggerBinding.on_hit_effect_params` | 直接使用 |
| `mZombieValue` | `gZombieDefs[]` | 波次点数预算 | 波次定义 | 直接使用 |
| `mStartingLevel` | `gZombieDefs[]` | 关卡解锁 | 关卡/波次配置 | 直接使用 |
| `mPickWeight` | `gZombieDefs[]` | 生成权重 | 波次定义 | 直接使用 |

**僵尸速度换算**：原版僵尸速度以像素/帧为单位（约 60fps），Open PVZ 使用像素/秒。由于两个项目的坐标体系和视口尺寸不同（原版 800x600，Open PVZ 960x540），不能直接换算。建议流程：

1. 以 PVZ-Godot-Dream 的 `MoveComponent` 导出值为参考起点
2. 在验证场景中用 `zombie_basic_walker`（move_speed=55）作为基准
3. 其他僵尸按原版相对速度比例设置
4. 通过验证场景实测确认"看起来对"

## 4. 实体分批复刻计划

根据当前协议的可映射性，将原版 48 植物 + 26 僵尸分为三批。

### Batch A：直接可映射（当前协议完整支持）

这类实体的行为可以用现有的 trigger + effect 系统完整表达，**不需要任何代码或协议修改**。

| Repeater | attack binding + 快速 interval (0.75s) 单发 | 已完成 `plant_repeater_burst` |

> **Gloom-shroom 实现方式**：不需要投射物。使用 `periodically` trigger + `explode` effect（`target_mode=enemies_in_radius`，`radius=90`，`amount=20`），每隔 2.0s 对自身周围 90px 内所有敌人造成伤害。这复用了已有的 `explode` 效果，不需要新代码。

> **Split Pea 实现方式**：两个独立的 `attack` trigger binding。Binding 1：`detection_id=lane_forward`，`direction=Vector2(1,0)` 射击右侧僵尸。Binding 2：`detection_id=lane_backward`，`direction=Vector2(-1,0)` 射击左侧（已越过植物的）僵尸。两个 binding 共用相同的 interval 和 projectile template。

### Batch C：需协议扩展（需要新触发器类型或效果类型）

这类实体涉及当前冻结协议完全不支持的行为模式。标注缺口，延后到专门的设计审批后实施。

**Batch C 植物**（约 14 种）：

| 原版植物 | 缺失能力 | 建议触发器/效果 |
|----------|---------|----------------|
| Cherry Bomb | 放置后即时引爆 3x3 | `on_place` 触发器 + `explode` |
| Potato Mine | 放置后延迟就绪 + 接触引爆 | `on_place` + `on_proximity` 触发器 |
| Jalapeno | 放置后整行焚烧 | `on_place` + `explode`（整行） |

## 5. 单个实体复刻工作流

### 5.0 前置判断

开始前回答三个问题：

1. **它属于哪个 Batch？** 按 §4 的分批表确定。如果是 Batch C，只填写参数表（§6.1），记录协议缺口，停止后续步骤。
2. **它需要哪些资源？** 对照 [Archetype 编写约定](35-模板编写约定.md) §6 判断需要：CombatArchetype / CombatMechanic / ProjectileTemplate / ProjectileFlightProfile / CardDef / HeightBand。
3. **它能复用已有资源吗？** 检查 `data/combat/` 下是否有可共享的投射物模板、飞行配置或高度带。

### 5.1 提取参数

**第一步：从 de-pvz 提取权威数值**

打开以下文件，按映射表（§3）提取数据：

```
1. ConstEnums.h            -> SeedType/ZombieType 枚举值、ProjectileType
2. Plant.cpp gPlantDefs[]  -> cost, refreshTime, launchRate, subclass
3. Plant.cpp PlantInit()   -> HP 值
4. Plant.cpp Fire()        -> 投射物类型、伤害、运动方式
5. Zombie.cpp gZombieDefs[] -> value, startingLevel, pickWeight
6. Zombie.cpp ZombieInit() -> bodyHP, helmHP, shieldHP
7. Projectile.cpp           -> damage 值、运动参数
8. GameConstants.h          -> 通用常量
```

将提取的原始数值记录到参数表中（格式见 §6.1）。

**第二步：必要时从 PVZ-Godot-Dream 交叉验证**

对于 Godot 特有的实现细节（组件拆分、碰撞配置、动画状态），打开对应场景或脚本参考：

```
1. scripts/data/character_registry.gd  -> cost/cooldown（秒）、场景路径
2. scripts/character/plant/plant_XXX.gd  -> 植物特定逻辑
3. scripts/character/zombie/zombie_XXX.gd -> 僵尸特定逻辑
4. scripts/character/components/  -> 组件接口和 export 参数
5. animation/character/  -> 可用的动画状态列表
```

### 5.2 翻译为 Open PVZ 资源

**按照以下顺序创建 Archetype 资源**（遵循 [Archetype 编写约定](35-模板编写约定.md)）：

```
1. HeightBand（如需新建）          -> data/combat/height_bands/
2. ProjectileFlightProfile（如需新建）-> data/combat/projectile_profiles/
3. ProjectileTemplate              -> data/combat/projectile_templates/
4. CombatMechanic（一个或多个）     -> 定义行为（Trigger / Controller / State 等）
5. CombatArchetype                 -> data/combat/archetypes/plants/ 或 zombies/
6. CardDef（如果是植物）            -> data/combat/cards/
```

**翻译时的核心映射逻辑**：

```
原版 mSeedCost        -> CardDef.sun_cost
原版 mRefreshTime/100 -> CardDef.cooldown_seconds
原版 mPlantHealth     -> EntityTemplate.max_health
原版 mLaunchRate/100  -> TriggerBinding.condition_values["interval"]
原版 投射物 damage    -> TriggerBinding.effect_params["damage"]
原版 STRAIGHT         -> ProjectileFlightProfile.move_mode = &"linear"
原版 LOBBED           -> ProjectileFlightProfile.move_mode = &"parabola", height_strategy = &"arc"
原版 HOMING           -> ProjectileFlightProfile.move_mode = &"track"
原版 PUFF(75 ticks)   -> ProjectileTemplate.lifetime = 0.75
原版 产阳光(2500 ticks) -> TriggerBinding.condition_values["interval"] = 25.0
```

**射手植物的典型 Archetype 资源组合**：

```
ProjectileFlightProfile (如已有可复用)
    ↓
ProjectileTemplate (定义 damage、lifetime、speed)
    ↓
CombatMechanic (Trigger.periodically + Targeting + Emission + Trajectory + HitPolicy + Payload)
    ↓
CombatArchetype (引用 mechanic + max_health + hitbox + projectile)
    ↓
CardDef (sun_cost + cooldown)
```

**纯被动植物**（Wall-nut、Tall-nut、Lily Pad、Flower Pot、Pumpkin）：

```
CombatArchetype (max_health + placement_role + hitbox)
    ↓
CardDef (sun_cost + cooldown)
```

不需要 Mechanic。

**产阳光植物**（Sunflower、Sun-shroom、Twin Sunflower、Marigold）：

```
CombatMechanic (Trigger.periodically + Payload.produce_sun)
    ↓
CombatArchetype (引用 mechanic)
    ↓
CardDef (sun_cost + cooldown)
```

**近战型僵尸**（Basic、Flag、Conehead、Buckethead 等）：

```
CombatMechanic (Controller.core.bite)
    ↓
CombatArchetype (max_health + default_params 含 move_speed 等 + mechanic)
```

不需要 Trigger Mechanic。啃食由 Controller.bite 驱动。

**远程型僵尸**（Catapult、bone_thrower 等）：

```
ProjectileFlightProfile + ProjectileTemplate
    ↓
CombatMechanic (Controller.bite + Trigger.periodically + Payload.spawn_projectile)
    ↓
CombatArchetype (引用 mechanic + default_params 含 move_speed 等)
```

啃食依然存在（Controller.bite），远程攻击是额外能力。

### 5.3 创建验证场景

每个新实体必须配套至少一个验证场景，证明其行为正确：

```
1. 在 scenes/validation/ 下创建 BattleScenario .tres
2. 配置 BattleSpawnEntry 放置新实体
3. 配置 BattleSpawnEntry 放置交互对象（如靶子僵尸）
4. 添加 BattleValidationRule 断言关键事件
5. 在 tools/validation_scenarios.json 中注册
6. 更新 wiki/03-content-validation/32-验证矩阵.md
```

### 5.4 运行验证

```powershell
pwsh tools/run_validation.ps1 -Scenario 'res://scenes/validation/<scenario>.tres'
```

检查 `artifacts/validation/` 下的报告，确认所有 validation rule 通过。

### 5.5 更新文档

完成一个实体后，同步更新：

- `wiki/03-content-validation/32-验证矩阵.md`（如新增验证场景）
- 本文档 §4 的状态列（标记"已完成"）
- 如有协议变更，更新相关 wiki 文档

## 8. 完整工作流清单（Checklist）

复刻一个实体时，按以下清单逐项执行。

### 提取阶段

- [ ] 确定实体属于 Batch A / B / C（§4）
- [ ] 如 Batch C：仅填写参数表（§6.1），记录协议缺口，停止后续步骤
- [ ] 从 de-pvz `ConstEnums.h` 确认枚举值
- [ ] 从 de-pvz `gPlantDefs[]` 或 `gZombieDefs[]` 提取 cost、refresh、launchRate
- [ ] 从 de-pvz `PlantInitialize()` 或 `ZombieInitialize()` 提取 HP 值（注意分层：body + helm + shield）
- [ ] 从 de-pvz `Fire()` / 投射物定义提取投射物类型、damage、运动方式
- [ ] tick->秒 换算：/100
- [ ] 如有疑问，从 PVZ-Godot-Dream 对应脚本/场景交叉验证

### 创建资源阶段

- [ ] 确定可复用的已有资源（ProjectileFlightProfile、HeightBand）
- [ ] 确定该实体属于哪种 Archetype 组合：
  - 射手植物：FlightProfile + ProjectileTemplate + Mechanic(attack) + Archetype + CardDef
  - 纯被动植物：Archetype + CardDef
  - 产阳光植物：Mechanic(produce_sun) + Archetype + CardDef
  - 近战僵尸：Mechanic(Controller.bite) + Archetype
  - 远程僵尸：Mechanic(Controller.bite + attack) + Archetype + ProjectileTemplate + FlightProfile
- [ ] 创建 ProjectileFlightProfile（如需新建）-> `data/combat/projectile_profiles/`
- [ ] 创建 ProjectileTemplate -> `data/combat/projectile_templates/`
- [ ] 创建 CombatMechanic（一个或多个）
- [ ] 创建 CombatArchetype -> `data/combat/archetypes/plants/` 或 `zombies/`
- [ ] 创建 CardDef（如果是植物）-> `data/combat/cards/`
- [ ] 检查命名符合约定（plant_role_variant / zombie_role_variant）

### 验证阶段

- [ ] 创建验证场景 -> `scenes/validation/`
- [ ] 添加 BattleValidationRule 断言关键事件
- [ ] 注册到 `tools/validation_scenarios.json`
- [ ] 运行验证通过

### 文档更新阶段

- [ ] 更新 `wiki/03-content-validation/32-验证矩阵.md`
- [ ] 更新本文档 §4 的状态列
- [ ] 填写并归档参数表（§6.1）
