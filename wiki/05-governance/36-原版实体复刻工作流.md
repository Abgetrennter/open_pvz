# 原版植物移植设计方案

- 状态：当前计划

> 本文替代旧版“原版实体复刻工作流”，作为 PVZ1 原版植物移植的正式执行方案。本文只定义植物移植流程；僵尸、场上物件和表现层资产移植应另行成文。

---

## 一、目标与边界

默认目标：

- 覆盖 PVZ1 种子槽中的 `49` 个原版植物。
- 不包含 `Explode-o-nut / Giant Wall-nut / Sprout / Leftpeater` 等特殊非标准 seed。
- 采用“机制优先”验收：优先还原原版数值、触发条件、战斗语义和放置约束。
- 动画、音效、reanim 级别视觉资产可后续补齐，不作为首轮植物移植的阻塞项。
- 新内容必须走 `CombatArchetype + CombatMechanic[]`。
- 不得把 `EntityTemplate / TriggerBinding` 重新作为正式作者入口。

如果原版行为超出当前 family/type 能力边界，应记录协议缺口并停在正确位置；不得把缺失能力硬编码进单个植物脚本。

---

## 二、数据源优先级

### 1. 原版规格来源

`vendor/de-pvz` 是原版规格唯一优先来源，重点检查：

- `ConstEnums.h`
- `Lawn/Plant.cpp`
- `Lawn/Plant.h`
- `Lawn/Board.cpp`
- `Lawn/Projectile.cpp`
- `Lawn/SeedPacket.*`

用于提取：

- `SeedType`
- 卡片费用、冷却、解锁信息
- 生命值、碰撞和放置条件
- 攻击周期、伤害、目标规则
- 投射物、状态效果和特殊交互

### 2. Godot 层参考

`vendor/PVZ-Godot-Dream` 只作为 Godot 实现参考，用于查看：

- 节点组织方式
- 组件拆法
- 动画与场景结构
- 表现层资源落点

它不能覆盖 `de-pvz` 的原版规格结论。

### 3. Open PVZ 正式落点

最终资源只能落到当前正式资源树：

- `data/combat/archetypes/`
- `data/combat/mechanics/`
- `data/combat/projectile_templates/`
- `data/combat/projectile_profiles/`
- `data/combat/cards/`
- `scenes/validation/`
- `tools/validation_scenarios.json`
- `tools/formal_content_validation_map.json`

---

## 三、正式资源映射

### 1. 每个植物的最小资源

每个原版植物至少需要确认或新增：

- `CombatArchetype`
- `CardDef`
- 至少一个 validation 场景

攻击植物按需增加：

- `CombatMechanic`
- `ProjectileTemplate`
- `ProjectileFlightProfile`

### 2. 旧模型归档边界

正式资源不得再新增或保留旧实体身份字段；事件身份、验证断言和目标判定统一使用 `archetype_id / source_archetype_id / target_archetype_id`。

禁止新增依赖：

- `backend_entity_template`
- `backend_entity_template_id`
- 实体旧模板资源
- 实体旧触发绑定资源

### 3. 放置语义

放置型植物必须通过 `Placement` mechanic 或已正式支持的 `placement_spec` 表达。

不得为了单个植物新增 battle 放置特判。

## 四、行为分类映射

### 1. 普通射手

适用于：

- Peashooter
- Snow Pea
- Repeater
- Gatling Pea

默认组合：

- `Trigger.core.periodic`
- `Targeting.core.lane_forward`
- `Emission`
- `Trajectory.core.linear`
- `HitPolicy`
- `Payload.core.spawn_projectile`

### 2. 抛投植物

适用于：

- Cabbage-pult
- Kernel-pult
- Melon-pult
- Winter Melon

默认组合：

- `Trigger.core.periodic`
- `Targeting.core.lane_forward`
- `Trajectory.core.parabola`
- terminal hit policy
- projectile payload

### 3. 产阳光植物

适用于：

- Sunflower
- Sun-shroom
- Twin Sunflower
- Marigold（若首轮只实现阳光/资源语义，应记录金币语义缺口）

默认组合：

- `Trigger.core.periodic` 或 support cycle
- `Payload.core.produce_sun`
- `CardDef`

### 4. 防御植物

适用于：

- Wall-nut
- Tall-nut
- Pumpkin

默认组合：

- 高 `max_health`
- 合理 `hitbox_size`
- 正式放置语义
- 无攻击 mechanic

### 5. 即时植物

适用于：

- Cherry Bomb
- Jalapeno
- Ice-shroom
- Doom-shroom
- Coffee Bean（唤醒语义需先确认协议）

默认组合：

- `Lifecycle.core.on_place`
- `Payload.core.explode`
- `Payload.core.damage`
- `Payload.core.apply_status`

### 6. 水生、屋顶与覆盖植物

适用于：

- Lily Pad
- Tangle Kelp
- Sea-shroom
- Flower Pot
- Pumpkin

默认组合：

- `Placement.core.water_slot`
- `Placement.core.roof_slot`
- `Placement.core.cover_on_primary` 或等价正式 placement spec

### 7. 升级植物

适用于：

- Gatling Pea
- Twin Sunflower
- Gloom-shroom
- Cattail
- Winter Melon
- Gold Magnet
- Spikerock
- Cob Cannon

优先通过以下能力表达：

- `required_present_roles`
- `required_placement_tags`
- 专属 Placement type
- 正式 `placement_spec`

如果无法表达，应登记协议缺口，不得写成单植物放置特判。

### 8. 手动技能植物

适用于：

- Cob Cannon

默认路线：

- 先走 battle mode / input profile 设计。
- 通过 `BattleModeHost / BattleRuleModule / BattleInputProfile` 承接手动输入。
- 不把手动输入逻辑塞进 `BattleManager`。

## 五、单植物调查记录

每个植物开工前必须先建立迁移记录。记录应至少包含：

- 原版 `SeedType`
- 原版名称
- 卡片费用
- 卡片冷却
- 解锁阶段或来源
- 生命值
- 碰撞尺寸
- 是否可被啃咬
- 放置条件：地面、水面、屋顶、覆盖、升级依赖、多格占用
- 行为入口：周期、受击、死亡、放置后、手动点击、状态变化
- 投射物或效果：伤害、速度、弹道、命中范围、目标规则、状态效果
- 当前 Open PVZ 可复用资源
- 是否存在协议缺口

调查记录不要求成为独立资源文件，但 PR 或任务记录中必须能追溯这些结论。

---

## 六、资源落地顺序

每个植物固定按以下顺序执行：

1. 先补或复用 `ProjectileFlightProfile / ProjectileTemplate`。
2. 再补 `CombatMechanic`，优先复用已有 skeleton mechanic。
3. 创建 `CombatArchetype`，绑定 mechanic、生命值、hitbox、placement 和默认 projectile。
4. 创建 `CardDef`，设置费用、冷却、archetype 引用。
5. 创建单植物 validation。
6. 将 validation 加入 `tools/validation_scenarios.json`。
7. 若进入正式 roster，更新 `tools/formal_content_validation_map.json`。
8. 最后补 showcase；showcase 不作为验收依据。

如果任一步发现协议缺口，应先记录缺口，再决定是否补最小 type/effect/validation。

## 八、协议缺口处理

以下行为不得硬编码进单个植物：

- 催眠反转阵营
- Garlic 换道
- Torchwood 改写投射物
- Magnet-shroom / Gold Magnet 吸附物品
- Umbrella Leaf 防护
- Blover 全局驱散飞行单位
- Cob Cannon 手动瞄准与多格占用
- Imitater 复制卡片/复制植物
- 夜间蘑菇睡眠与 Coffee Bean 唤醒
- Spikeweed / Spikerock 持续地面伤害与特殊僵尸交互

处理规则：

1. 若现有 family/type 可表达，只新增资源。
2. 若只缺 payload/effect 行为，新增最小 effect type，并补 guardrail。
3. 若改变输入、放置或 mode 语义，优先扩 `BattleModeHost / BattleInputProfile / Placement`。
4. 不得把 mode 差异回流到 `BattleManager` 特判。
5. 若需要新增 family，必须先写 ADR，不进入植物移植批次。

## 九、验证计划

### 1. 单植物最低验收

每个植物至少满足：

- 单植物 `core` validation：能放置、能触发核心行为、能产生预期事件。
- 卡片 validation：费用、冷却、非法放置、合法放置。
- 交互 validation：至少覆盖一个代表僵尸或场地。
- 若新增 effect/type：补一个正向 validation 和一个 guardrail validation。
- 若进入正式 roster：更新 formal content map，并确保批量验证通过。

### 2. 批次级验收

批次级 validation 建议：

- 批次 A/B：`plant_original_batch_ab_validation`
- 批次 C：夜间、睡眠、唤醒专项 validation
- 批次 D：场地语义、多方向攻击、特殊目标专项 validation
- 批次 E：升级依赖、多格占用、手动技能专项 validation

### 3. 推荐验证命令

每批完成后至少运行：

```powershell
& ".\tools\run_all_validations.ps1" -Layers smoke,core
```

涉及扩展能力、新 effect 或新 guardrail 时运行完整批量验证：

```powershell
& ".\tools\run_all_validations.ps1"
```

验证事实以 `tools/validation_scenarios.json` 和脚本产物为准；Wiki 中手写数量如果冲突，应同步修正。

---

## 十、维护规则

- 原版植物数值以 `vendor/de-pvz` 为准，外部 Wiki 或记忆信息只能作为辅助线索。
- 已存在的 Open PVZ 样板植物可被改名、复用或迁移为原版植物。
- 样板植物不得保留与原版语义冲突的行为。
- showcase 只负责展示，不替代 validation。
- 移植中发现架构缺口时，优先小步补 type/effect/validation。
- 不提前建设大型新架构。
- 不新增未进入当前需求的未来能力。
- 不为单个植物复制粘贴专属运行时逻辑；相似行为必须沉淀为 mechanic、effect 或 mode 能力。
