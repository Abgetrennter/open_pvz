# 模板编写约定

> 这篇文档把第二阶段的模板体系进一步落成到“内容作者怎么写资源”这一层。它不是再解释模板边界，而是直接回答：一个新模板要建哪些文件、放到哪里、字段怎么分、哪些内容不该写进去。

---

## 文档定位

这篇文档主要回答：

- 新增一个实体模板时应新建哪些资源
- `EntityTemplate / ProjectileTemplate / TriggerBinding / BattleSpawnEntry` 分别该怎么写
- 当前推荐的编写顺序是什么
- 哪些字段属于长期定义，哪些字段只适合场景局部覆盖

这篇文档不负责：

- 重新定义模板边界
- 冻结第三阶段协议细节
- 展开第四阶段玩法系统

如果需要先理解“字段为什么这样分”，请先看：

- [模板与装配边界](../02-runtime-protocol/11-模板与装配边界.md)

---

## 一句话原则

编写模板时，默认遵守下面这条规则：

> 先写长期内容定义，再写行为绑定，最后才写场景局部覆盖。

换句话说：

- 不要先写 `BattleSpawnEntry` 再把一个单位拼出来
- 不要先在 `runtime_behavior` 里塞一大坨字典再回头补资源
- 不要把投射体长期定义继续塞进实体模板的零散参数里

---

## 1. 当前正式目录约定

第二阶段起，战斗内容资源默认使用下面这套目录：

- `data/combat/height_bands/`
- `data/combat/projectile_profiles/`
- `data/combat/projectile_templates/`
- `data/combat/trigger_bindings/`
- `data/combat/entity_templates/plants/`
- `data/combat/entity_templates/zombies/`

默认约束：

- 新的实体模板必须进入 `entity_templates/plants` 或 `entity_templates/zombies`
- 新的投射体模板必须进入 `projectile_templates`
- 新的行为绑定必须进入 `trigger_bindings`
- 不要再新增平铺 `templates/` 目录

---

## 2. 新增一个单位时的推荐顺序

推荐按下面顺序写资源：

1. 先确定单位属于植物还是僵尸
2. 再确定它是否需要独立投射体
3. 再确定是否需要独立行为绑定
4. 写 `EntityTemplate`
5. 如果有发射物，写 `ProjectileTemplate`
6. 如果有正式行为，写 `TriggerBinding`
7. 最后在 `BattleScenario` 里用 `BattleSpawnEntry` 摆实例

如果一个单位只是“站桩目标”或“纯受击验证单位”，通常只需要：

- `EntityTemplate`

如果一个单位要发射东西，通常至少需要：

- `EntityTemplate`
- `ProjectileTemplate`
- 一个或多个 `TriggerBinding`

---

## 3. `EntityTemplate` 编写约定

### 3.1 适合写进 `EntityTemplate` 的内容

应该优先写进：

- 身份信息
  - `template_id`
  - `entity_kind`
  - `display_name`
  - `tags`
- 节点和组件信息
  - `root_scene`
  - `visual_scene`
  - `required_components`
  - `optional_components`
- 长期战斗属性
  - `max_health`
  - `hitbox_size`
  - `hit_height_band`
- 默认投射入口
  - `projectile_flight_profile`
  - `projectile_template`
- 正式行为资源入口
  - `trigger_bindings`

### 3.2 当前不建议继续堆进 `EntityTemplate` 的内容

不建议继续长期写在这里：

- 一整套投射体本体参数
- 复杂 on-hit 组合
- 场景里才会变化的位置信息
- 只在某一个验证场景里才成立的覆盖项

### 3.3 命名建议

建议使用：

- 植物：`plant_<role>_<variant>`
- 僵尸：`zombie_<role>_<variant>`

例如：

- `plant_basic_shooter`
- `plant_track_bomber`
- `zombie_lane_dummy`
- `zombie_reactive_bomber`

避免：

- `test1`
- `new_template`
- `pea_01_final_final`

### 3.4 最小示例

```tres
[resource]
template_id = &"plant_basic_shooter"
entity_kind = &"plant"
display_name = "Basic Shooter Plant"
tags = PackedStringArray("plant", "template", "shooter")
required_components = PackedStringArray("TriggerComponent")
optional_components = PackedStringArray("DebugViewComponent")
max_health = 90
hitbox_size = Vector2(42, 54)
hit_height_band = ExtResource("2_height")
projectile_template = ExtResource("4_projectile_template")
trigger_bindings = [ExtResource("5_binding")]
```

## 4. `ProjectileTemplate` 编写约定

### 4.1 什么时候需要单独写 `ProjectileTemplate`

满足下面任意一条时，建议单独建投射体模板：

- 投射体有自己的模板身份
- 投射体默认飞行配置需要复用
- 投射体默认寿命、碰撞半径、表现入口需要固定
- 多个实体会共用这一类投射体

### 4.2 适合写进 `ProjectileTemplate` 的内容

应该优先写进：

- `template_id`
- `display_name`
- `tags`
- `root_scene`
- `visual_scene`
- `flight_profile`
- `lifetime`
- `hitbox_radius`
- `default_params`

### 4.3 不适合写进 `ProjectileTemplate` 的内容

不建议写进：

- “谁发射它”
- “多久触发一次”
- 实体本体血量和高度带
- 只在某个场景里成立的目标位置

### 4.4 命名建议

建议使用：

- `pea_linear`
- `track_bomb`
- `lobbed_shell`
- `air_bolt`

尽量让名字体现：

- 投射体类型
- 轨迹或用途

### 4.5 最小示例

```tres
[resource]
template_id = &"pea_linear"
display_name = "Linear Pea Projectile"
tags = PackedStringArray("projectile", "pea", "linear")
flight_profile = ExtResource("2_profile")
lifetime = 5.0
hitbox_radius = 10.0
default_params = {
"damage": 20,
"speed": 220.0
}
```

## 5. `TriggerBinding` 编写约定

### 5.1 当前推荐把什么行为写成 `TriggerBinding`

第二阶段里，新增正式行为优先写成：

- `attack`
- `when_damaged`
- `on_death`

后续如果再扩：

- `on_spawn`
- `on_hit`
- `support`

但这些后续块当前还不建议在没有验证配套的情况下随意扩写。

### 5.2 `TriggerBinding` 应该承载什么

建议写进：

- `binding_id`
- `behavior_key`
- `trigger_id`
- `event_name`
- `condition_values`
- `effect_id`
- `effect_params`
- `on_hit_effect_id`
- `on_hit_effect_params`
- `projectile_template`

### 5.3 当前写法建议

如果是发射型行为：

- `effect_id` 用 `spawn_projectile`
- 投射体默认差异优先写到 `projectile_template`
- `effect_params` 只保留这一条行为真正需要覆盖的参数

如果是受伤反击或死亡爆炸：

- 不要把实体移动速度、生命值这类实体字段塞进 `effect_params`
- `effect_params` 只保留效果本身需要的字段

### 5.4 最小示例

```tres
[resource]
binding_id = &"plant_attack_basic_linear"
behavior_key = &"attack"
trigger_id = &"periodically"
event_name = &"game.tick"
condition_values = {
"interval": 0.6
}
effect_id = &"spawn_projectile"
effect_params = {
"damage": 20,
"direction": Vector2(1, 0),
"speed": 220.0
}
projectile_template = ExtResource("2_projectile")
```

## 6. `BattleSpawnEntry` 编写约定

### 6.1 `BattleSpawnEntry` 现在主要负责什么

当前应主要只负责：

- 引用 `entity_template`
- 指定 `lane_id`
- 指定 `x_position`
- 提供少量场景级覆盖字段

### 6.2 什么适合写进 `BattleSpawnEntry.params`

只建议写那些：

- 这一局才变化
- 不值得上升为模板长期定义
- 明确是场景局部覆盖

例如：

- 某个验证里临时改一次 `interval`
- 某个验证里临时替换一次 `projectile_template`
- 某个验证里临时覆盖一次 `flight_profile`

### 6.3 什么不适合继续写进 `BattleSpawnEntry`

不建议继续长期写在这里：

- 单位默认生命
- 默认碰撞体积
- 默认高度带
- 正式行为定义
- 一整套默认攻击参数

如果一个字段每次都要在场景里重复写，通常说明它该回收到模板。

### 6.4 最小示例

```tres
[resource]
entity_template = ExtResource("4_plant_template")
lane_id = 0
x_position = 160.0
```

## 7. 当前覆盖规则约定

这里先记录当前版本的约定，后续如果 `Issue B2` 正式收口覆盖优先级，应同步更新这篇文档。

当前可按下面顺序理解：

1. 模板默认值
2. 行为绑定或投射体模板默认值
3. `BattleSpawnEntry.params` 局部覆盖
4. 运行时派生值

对内容作者来说，最重要的结论是：

- 模板里放长期默认值
- 场景里只放局部覆盖
- 不要把“暂时为了验证方便”写成模板长期事实

---

## 8. 新增内容时的最小自检清单

写完一个新模板后，建议至少自查下面这些问题：

1. 它放在正确目录了吗
2. `template_id` 是否稳定且可读
3. 这个字段属于模板长期定义还是场景局部覆盖
4. 是否误把投射体长期定义塞回了实体模板
5. 是否误把正式行为继续塞回了大字典
6. 是否已经有对应验证场景或至少有明确的待补验证位置

---

## 9. 当前不建议进入第二阶段主线的内容

即使以后会做，当前写模板时也不建议把这些内容提前揉进来：

- 阳光、卡片、手牌 UI
- PlantCell 式完整格子规则
- 波次和关卡流程
- 菜单、商店、花园、存档
- 大型表现层特化字段

这些内容在当前阶段容易把模板写回成“功能愿望单”，而不是稳定内容协议。

---

## 10. 推荐工作流

如果要新增一个正式内容样例，推荐按下面节奏：

1. 先确定它是植物还是僵尸
2. 建 `EntityTemplate`
3. 如果需要发射物，建 `ProjectileTemplate`
4. 如果需要正式行为，建 `TriggerBinding`
5. 在验证场景里加 `BattleSpawnEntry`
6. 跑对应专项验证
7. 如果它代表了一类新内容，再补文档和验证矩阵
