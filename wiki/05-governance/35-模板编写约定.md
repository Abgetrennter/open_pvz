# Archetype 编写约定

> 这篇文档说明内容作者如何使用 `Archetype + Mechanic[]` 新作者模型创建实体。

---

## 文档定位

这篇文档主要回答：

- 新增一个实体时应创建哪些资源
- `CombatArchetype` 和 `CombatMechanic` 分别怎么写
- 当前推荐的编写顺序是什么
- 命名约定是什么

这篇文档不负责：

- 解释编译链机制（见 [编译链与 Mechanic 系统](../02-runtime-protocol/11-编译链与Mechanic系统.md)）
- 定义 Mechanic family / type 的协议边界

---

## 一句话原则

编写新实体时，默认遵守：

> 先确定 Archetype 身份和类型，再组装 Mechanic 行为数组，最后写场景条目。

---

## 1. 目录约定

```
data/combat/
├── archetypes/
│   ├── plants/          # 植物 archetype .tres
│   ├── zombies/         # 僵尸 archetype .tres
│   └── field_objects/   # 场上物件 archetype .tres
├── projectile_templates/
├── projectile_profiles/
├── height_bands/
├── cards/
├── battlefields/
├── waves/
└── levels/
```

新增 archetype 必须进入 `archetypes/plants/` 或 `archetypes/zombies/` 或 `archetypes/field_objects/`。

---

## 2. 新增实体时的推荐顺序

1. 确定单位属于植物、僵尸还是场上物件
2. 确定它需要哪些 Mechanic（按 family 思考：需要 Trigger 吗？Controller 吗？State 吗？）
3. 写 `CombatArchetype`（Identity + Chassis + Combat + Mechanic[]）
4. 如果需要投射体，写 `ProjectileTemplate` 和 `ProjectileFlightProfile`
5. 在 `BattleScenario` 里用 `BattleSpawnEntry` 摆实例
6. 跑对应专项验证

---

## 3. `CombatArchetype` 编写约定

### 3.1 适合写进 Archetype 的内容

| 层 | 字段 | 说明 |
|----|------|------|
| 身份 | `archetype_id`, `entity_kind`, `display_name`, `tags` | 这个单位是谁 |
| 节点 | `root_scene`, `visual_scene`, `required_components`, `optional_components` | 实例化用哪个场景和组件 |
| 放置 | `placement_role`, `allowed_slot_types`, `required_placement_tags`, `granted_placement_tags` | 棋盘放置语义 |
| 战斗 | `max_health`, `hitbox_size`, `hit_height_band` | 基础战斗属性 |
| 投射 | `projectile_flight_profile`, `projectile_template` | 默认投射体配置 |
| 行为 | `mechanics` | Mechanic 数组——所有行为的来源 |
| 编译提示 | `compiler_hints`, `default_params` | 传给编译器的额外信息 |

### 3.2 命名约定

建议使用：

- 植物：`plant_<role>_<variant>`
- 僵尸：`zombie_<role>_<variant>`
- 场上物件：`field_object_<name>`

例如：

- `plant_basic_shooter`
- `zombie_basic_walker`
- `field_object_lawn_mower`

### 3.3 最小示例（射手植物）

```tres
[gd_resource load_steps=3 format=3]

[ext_resource type="Resource" path="res://data/combat/height_bands/ground_band.tres" id="1_height"]
[ext_resource type="Resource" path="res://data/combat/projectile_profiles/linear_flat.tres" id="2_profile"]

[resource]
script = ExtResource("CombatArchetype")
archetype_id = &"plant_basic_shooter"
entity_kind = &"plant"
display_name = "Basic Shooter"
tags = PackedStringArray("plant", "shooter")
max_health = 300
hitbox_size = Vector2(42, 54)
hit_height_band = ExtResource("1_height")
projectile_flight_profile = ExtResource("2_profile")
mechanics = [ExtResource("3_mechanic_attack")]
```

### 3.4 最小示例（近战僵尸）

近战僵尸通过 Controller.bite Mechanic 获得啃食能力，不需要 Trigger Mechanic：

```tres
[resource]
script = ExtResource("ZombieArchetype")
archetype_id = &"zombie_basic_walker"
entity_kind = &"zombie"
display_name = "Basic Zombie"
tags = PackedStringArray("zombie", "walker")
max_health = 270
hitbox_size = Vector2(36, 54)
default_params = {
    "move_speed": 55.0,
    "attack_damage": 10,
    "attack_interval": 1.0,
    "attack_range": 56.0
}
mechanics = [ExtResource("3_mechanic_bite")]
```

## 4. `CombatMechanic` 编写约定

### 4.1 三层结构

每个 Mechanic 必须指定：

- `family`：行为维度（10 个冻结 family 之一）
- `type_id`：family 下的具体实现
- `params`：配置参数字典

### 4.2 常见 Mechanic 组合

#### 射手攻击（植物远程攻击）

```
Mechanic(family=Trigger, type_id=periodically)
  + Mechanic(family=Targeting, type_id=lane_forward)
  + Mechanic(family=Emission, type_id=single)
  + Mechanic(family=Trajectory, type_id=linear)
  + Mechanic(family=HitPolicy, type_id=swept_segment)
  + Mechanic(family=Payload, type_id=spawn_projectile)
```

#### 僵尸啃食

```
Mechanic(family=Controller, type_id=core.bite)
```

#### 产阳光

```
Mechanic(family=Trigger, type_id=periodically)
  + Mechanic(family=Payload, type_id=produce_sun)
```

#### 死亡爆炸

```
Mechanic(family=Trigger, type_id=on_death)
  + Mechanic(family=Payload, type_id=explode)
```

#### 阶段状态（如土豆地雷 arming）

```
Mechanic(family=State, type_id=core.armed)
  + Mechanic(family=Lifecycle, type_id=core.on_armed)
```

### 4.3 Mechanic 示例（周期触发）

```tres
[resource]
script = ExtResource("CombatMechanic")
mechanic_id = &"mechanic_attack_periodic"
display_name = "Periodic Attack"
family = &"Trigger"
type_id = &"periodically"
enabled = true
priority = 100
params = {
    "interval": 1.5,
    "detection_id": &"lane_forward",
    "effect_id": &"spawn_projectile",
    "effect_params": {
        "damage": 20,
        "speed": 220.0
    }
}
```

### 4.4 Mechanic 示例（Controller bite）

```tres
[resource]
script = ExtResource("CombatMechanic")
mechanic_id = &"mechanic_bite"
display_name = "Bite Attack"
family = &"Controller"
type_id = &"core.bite"
enabled = true
priority = 100
params = {
    "damage": 10,
    "interval": 1.0,
    "range": 56.0
}
```

## 5. `BattleSpawnEntry` 编写约定

`BattleSpawnEntry` 负责在场景中放置一个 archetype 实例：

```tres
[resource]
entity_archetype_id = &"plant_basic_shooter"
lane_id = 0
x_position = 160.0
spawn_overrides = {}
```

如果需要覆盖参数：

```tres
[resource]
entity_archetype_id = &"zombie_basic_walker"
lane_id = 2
x_position = 800.0
spawn_overrides = {
    "move_speed": 80.0
}
```

## 6. 编写顺序速查

按实体类型选择编写路径：

| 实体类型 | 需要的资源 |
|----------|-----------|
| 射手植物 | Archetype + Mechanic(attack) + ProjectileTemplate + ProjectileFlightProfile + CardDef |
| 纯被动植物 | Archetype（无 Mechanic 或仅 Lifecycle）+ CardDef |
| 产阳光植物 | Archetype + Mechanic(periodically + produce_sun) + CardDef |
| 近战僵尸 | Archetype + Mechanic(Controller.bite) + default_params(move_speed 等) |
| 远程僵尸 | Archetype + Mechanic(Controller.bite + attack) + ProjectileTemplate |
| 场上物件 | Archetype + Mechanic(Controller.sweep 等，按需) |

---

## 7. 新增内容时的最小自检清单

1. Archetype 放在正确目录了吗（plants / zombies / field_objects）
2. `archetype_id` 是否稳定且可读
3. Mechanic 的 family 和 type_id 是否正确
4. 是否已有对应验证场景
5. 如有投射体，是否正确关联了 ProjectileTemplate 和 FlightProfile
