## 1. 决策摘要

项目在 `Mechanic-first` 阶段正式采用：

1. 扩展包默认面向 `Archetype / Mechanic / type` 层接入，而不是直接面向 `TriggerBinding`。
2. 扩展包默认只能新增现有 family 下的 `type`，不得新增一级 family。
3. 迁移策略采用“**受限双轨期迁移**”：短期允许新旧作者模型并存以建立最小验证闭环，但中期目标是切换到新模型为唯一正式入口。

### 当前实现状态（2026-04-22）

截至 2026-04-22，这份 ADR 已经从“迁移策略”进入“主实现已落地”阶段：

- 现有正式扩展包已经能通过 `Mechanic-first` 主链运行：
  - `minimal_chaos_pack`
  - `phase5_chaos_pack`
  - `phase5_guardrail_pack`
- 扩展包 smoke / chaos / guardrail 验证均已通过
- 主线资源入口已经能够优先消费扩展 archetype
- 旧扩展 `EntityTemplate / TriggerBinding` 当前仍保留，但已经降级为：
  - 后端素材
  - legacy 兼容层
  - 迁移对照对象

## 2. 背景与问题

当前扩展包体系已经有基础：

- `content` 资源包
- 最小 chaos 包
- 守卫包

但它们主要还是围绕当前作者模型与 effect 扩展路径运转。

在 `Mechanic-first` 下，扩展包真正想扩的东西发生了变化：

- 新 archetype
- 新 mechanic set
- 新 family 下的 type
- 必要时新的 controller / payload / trajectory runtime

同时，主仓还要面对迁移问题：

- 旧内容资产如何过渡
- 新旧验证如何接住
- 双轨期如何不失控

## 3. 目标

1. 定义 `Mechanic-first` 下扩展包的正式接入方向。
2. 冻结扩展包默认权限边界。
3. 明确迁移策略是短期双轨还是一次性切换。
4. 把迁移策略和验证闭环一起冻结。

## 4. 非目标

本次决策不直接冻结：

- manifest 最终全部字段
- 所有 trust level 的 UI 或产品交互
- 所有旧资源的转换脚本细节

## 5. 备选方案

### 方案 A：扩展包继续主要面向旧作者模型

优点：

- 改造量小

缺点：

- 与新主线脱节
- 后续仍然要在旧模型与新模型之间来回桥接

未选原因：

- 不符合主线切换后的架构方向。

### 方案 B：扩展包面向 Mechanic 层接入

优点：

- 和新架构方向一致
- 更适合错误技生成器和新内容工作流

缺点：

- 需要重建注册中心与守卫逻辑

选择原因：

- 这是让扩展包真正成为平台能力，而不是旧装配模型附庸的必要条件。

### 迁移策略子问题

#### 子方案 A：长期双轨期

优点：

- 风险小

缺点：

- 极易形成长期技术债
- 新旧心智长期并存

未选原因：

- 与本次重构“纠偏而不是长期兼容”目标冲突。

#### 子方案 B：一次性切换

优点：

- 最干净

缺点：

- 短期冲击太大
- 很容易把主干和验证一起打穿

未选原因：

- 对当前重构初期不够稳妥。

#### 子方案 C：受限双轨期迁移

内容：

- 短期允许新旧模型并存
- 双轨只用于建立最小验证闭环
- 中期明确废弃旧作者模型

优点：

- 比一次性切换更稳
- 比长期双轨更可控

选择原因：

- 更符合当前工程节奏。

## 6. 最终决策

### 6.1 扩展包接入方向

正式决定：

扩展包默认面向以下层接入：

- `Archetype`
- `MechanicSet`
- family 下的 `type`
- 必要时高信任运行时代码

不再把 `TriggerBinding` 视为扩展包的主要长期目标接口。

### 6.2 扩展包类型与权限

建议正式包类型：

- `content_pack`
- `rule_pack`
- `runtime_pack`
- `asset_pack`
- `collection_pack`

建议信任等级：

- `data_only`
- `rule_extended`
- `trusted_runtime`

默认权限边界：

- 扩展包可以新增 `type`
- 扩展包默认不能新增一级 family
- 新 family 只能由主仓通过 ADR 决定

### 6.3 注册中心方向

`Mechanic-first` 下建议新增：

- `MechanicFamilyRegistry`
- `MechanicTypeRegistry`
- `MechanicCompilerRegistry`
- `ControllerRegistry`

并要求扩展包注册都走命名空间：

- `core.*`
- `extension_namespace.*`

### 6.4 迁移策略

正式采用：

> **受限双轨期迁移**

解释：

- 短期允许旧作者模型和新作者模型并存
- 双轨期仅用于建立新架构的最小闭环
- 一旦首批 archetype 验证通过，旧作者模型立即进入废弃流程
- 后续新增内容只允许使用新模型

### 当前执行结果补充（2026-04-22）

这条迁移策略在当前仓库里的实际执行结果是：

- 双轨期已经完成其主要目标：
  - 建立 archetype 主路径
  - 建立扩展包主路径
  - 建立对照与 guardrail 验证
- 当前状态已经不再是“新旧模型对等并存”，而是：
  - 新模型是主线
  - 旧模型是有限兼容层

因此，这份 ADR 中的“双轨期”在工程上已经进入尾声。

### 6.5 验证闭环

迁移过程中的验证分三层：

1. 旧验证保底，确保底层 runtime 未被无意打碎
2. 新验证证明 archetype/compiler/controller 链路成立
3. 对照验证证明典型单位迁移前后行为基本一致

### 当前验证状态补充（2026-04-22）

当前验证闭环已经进入整仓稳定通过状态：

- `tools/run_all_validations.ps1`
- 当时结果：`66 / 66 passed`

其中包含：

- 主仓 core / smoke / guardrail
- migration parity
- extension smoke

## 8. 迁移与兼容

### 8.1 双轨期原则

双轨期必须受限：

- 只允许少量 archetype 试点
- 不允许以“双轨方便”为理由继续写大量旧内容
- 必须设置退出信号

### 8.2 退出信号

满足下面条件后，旧作者模型进入正式废弃流程：

1. 首批 archetype 验证通过
2. compiler/runtime assembler 能稳定工作
3. archetype 路径能覆盖一批典型内容
4. 新验证与对照验证稳定进入回归

### 当前退出信号状态（2026-04-22）

这 4 条退出信号当前都已经满足：

1. 首批 archetype 验证通过：已满足
2. compiler/runtime assembler 能稳定工作：已满足
3. archetype 路径能覆盖一批典型内容：已满足
4. 新验证与对照验证稳定进入回归：已满足

因此，从这份 ADR 的角度看：

- “受限双轨期迁移”目标已经达成
- 当前剩余工作只在于是否进一步物理清理 legacy 兼容层，而不是是否继续允许新旧双轨并行作为正式策略

### 8.3 首批迁移对象

建议首批迁移：

- 向日葵
- 豌豆射手
- 卷心菜投手
- 基础僵尸
- 土豆雷或樱桃炸弹

## 10. 风险与未决项

### 风险

1. 双轨期如果没有退出约束，会迅速演化成长期共存。
2. 扩展包如果能绕过命名空间和 family 限制，系统会重新失控。
3. 迁移对照验证如果做得不够，会让“看起来能跑”掩盖真实行为漂移。

### 未决项

- manifest 新字段的正式列表
- 扩展包冲突、停用、版本升级的最终产品策略
