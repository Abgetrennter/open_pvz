## 1. 决策摘要

项目正式采用一层 battle mode 组织层 v1，用于把 mode 的：

1. 身份
2. 输入
3. 规则模块
4. 胜负目标

从 `BattleManager / BattleFlowState / InputRouter` 的零散特判中抽离出来。

当前正式对象为：

- `BattleModeDef`
- `BattleRuleModule`
- `BattleInputProfile`
- `BattleObjectiveDef`
- `BattleModeHost`

## 2. 背景与问题

在保留 PVZ 棋盘 / 槽位 / 车道语义的前提下，项目已经开始需要承接更多风味小游戏与 mode 变体。

如果继续沿用旧路径，差异将不断回流到：

- `BattleManager`
- `BattleFlowState`
- `InputRouter`
- 各个 battle state

这会导致：

- mode 特判蔓延
- battle orchestration 与 mode 规则耦合
- 输入与目标逻辑无法资源化
- 新小游戏缺少固定验证入口

## 3. 目标

1. 为 battle 底盘补一层正式 mode 组织层。
2. 不推翻现有 `BattleScenario / BattleFlowState / InputRouter` 主链。
3. 第一版优先承接“规则偏转型小游戏”。
4. 让 mode 差异默认落在资源与 host 分发层，而不是 manager 特判层。

## 4. 非目标

本次决策不直接要求：

- 自由移动动作底盘
- 独立 `BattleHUDProfile`
- 独立 `BoardRuleLayer`
- 复杂脚本化 plugin 式规则系统
- 把旧 `battle_goal / defeat_conditions` 一次性完全替换

## 5. 备选方案

### 方案 A：继续把 mode 差异写进 `BattleManager`

优点：

- 改动集中

缺点：

- 特判继续积累
- mode 与 battle 主循环继续纠缠
- 输入、目标和规则很难分别治理

未选原因：

- 与当前主线“按层收口”的方向冲突。

### 方案 B：直接重写为完整插件化 mode 框架

优点：

- 理论扩展性最强

缺点：

- 复杂度过高
- 会打穿当前稳定 battle 主链
- 明显超出 v1 目标

未选原因：

- 不符合当前工程阶段。

### 方案 C：在 battle 主链之上叠一层 mode host

内容：

- `BattleScenario` 继续是 battle 资源入口
- `BattleModeDef / InputProfile / RuleModule / ObjectiveDef` 负责 mode 组织
- `BattleModeHost` 作为唯一新增 host，驱动 mode-level runtime

选择原因：

- 改动范围可控
- 可验证
- 可逐步演进

## 6. 最终决策

### 6.1 采用 mode host 叠加模型

正式采用：

- `BattleScenario` 继续定义 battle 内容资源
- `BattleModeDef` 定义 mode 身份与默认配置
- `BattleModeHost` 解析 mode、合并 override、调度 rule modules、评估 objective

### 6.2 输入按 `BattleInputProfile` 配置

`InputRouter` 保持现有主干，但增加 profile gating。

这意味着：

- 标准 battle 在无 profile 时保持旧行为
- mode battle 可以按配置关闭/开启输入能力
- 不再为每个小游戏继续往输入层塞特判

### 6.3 规则通过 `BattleRuleModule + Registry` 分发

第一版不为每个 module 创建独立 Node。

正式采用：

- `BattleModeModuleRegistry`
- `module_id -> handler`
- `on_mode_setup / on_battle_start / on_game_tick / on_event / on_mode_teardown`

### 6.4 目标通过 `BattleObjectiveDef` 追加判定

`BattleFlowState` 继续保留 battle phase 主状态机职责。

`BattleModeHost` 的职责仅限于：

- 解释 mode objective
- 在满足条件时调用 `mark_victory()` / `mark_defeat()`

不直接重写 battle phase 主状态机。

### 6.5 scenario 模块覆盖语义正式冻结

当前正式语义为：

- 同 `module_id`：scenario 模块覆盖 mode 默认模块
- 不同 `module_id`：共同生效
- 执行顺序由 `priority` 决定

## 7. 影响范围

### battle 主链

- `BattleSubsystemHost` 需要创建 mode host
- `BattleManager` 只做获取与生命周期转发

### 输入层

- `InputRouter` 需要接受 mode profile 注入

### 协议与验证

- `BattleScenario` 增加 mode 相关字段
- 协议校验需要覆盖 mode resource 和 mode input request
- 批量验证需要新增 mode 专项场景

### 文档

- 当前状态页、路线页、运行时页、验证矩阵都必须同步 mode 口径
- 完成后的 mode 草案计划应从 `plans/` 根目录转入 archive

## 8. 迁移与兼容

### 8.1 battle 兼容原则

- 无 `mode_def` 的 scenario 必须继续零改动工作
- mode 是叠加层，不是替换层

### 8.2 v1 退出信号

满足下面条件后，可认为 battle mode 组织层 v1 成立：

1. mode 核心对象进入主干
2. input profile 已真实注入运行时
3. 至少一个试点 mode 形成闭环
4. objective / module / guardrail 验证进入批量回归

### 8.3 当前状态（2026-04-27）

上面 4 条当前都已满足：

1. mode 核心对象：已满足
2. input profile 注入链：已满足
3. `manual aim` 试点闭环：已满足
4. mode 验证回归：已满足

因此，从这份 ADR 的角度看：

- battle mode 组织层 v1 已正式成立
- 后续重点转为继续补专项 objective / board / HUD 能力，而不是重新讨论是否需要 mode 层

## 10. 风险与未决项

### 风险

1. 如果后续小游戏继续直接回流 `BattleManager`，mode 层会被架空。
2. 如果未来把 HUD / board rule / runtime state 一次性塞进 v1，会重新抬高复杂度。
3. 如果缺少子类型专项验证，objective 覆盖会停留在“主链可跑”而不是“各类型都被证明”。

### 未决项

- `BattleHUDProfile` 的正式边界
- `BoardRuleLayer` 是否需要独立成层
- `BattleModeRuntimeState` 是否需要从 host 内部状态升级为正式对象

## 11. 后续动作

1. 继续补 `protect_archetype / clear_special_targets / defeat_named_spawn` 的专项验证。
2. 根据后续内容压力，再决定是否正式引入 `BattleHUDProfile`。
3. 继续保持“mode 差异先进 host / module / objective / profile”的治理约束。
