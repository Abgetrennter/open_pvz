## 1. 决策摘要

项目主线从“第七阶段输入准备”切换为 `Mechanic-first` 重构阶段；本次重构采用**保留底层执行骨架、重做上层作者模型与编译装配链**的边界，不进行全栈内核重写。

### 当前实现状态（2026-04-22）

截至 2026-04-22，这份 ADR 对应的路线切换已经在工程上真正落地：

- 主线作者入口已经切到 `Archetype + Mechanic[]`
- `cards / waves / levels / validation / showcase / extensions` 都已补齐 `archetype_id` 主路径
- 当前整仓批量验证为：
  - `66 / 66 passed`
  - 见 `artifacts/validation/batch_20260422_144938`
- 旧 `EntityTemplate / TriggerBinding` 尚未物理删除，但已经降级为：
  - legacy 兼容层
  - 后端素材
  - 迁移对照验证入口

当前仓库状态不再是“准备切主”，而是：

- 逻辑上已经切主
- 工程上仍保留有限兼容层

## 2. 背景与问题

当前仓库已经完成第一到第六阶段主干沉淀，并在 wiki 中以“第七阶段输入准备”为当前主线叙事：

- [当前阶段与实现路线](../01-overview/23-当前阶段与实现路线.md)
- [开发路线图](../04-roadmap-reference/26-开发路线图.md)

这条叙事默认假设：

- 现有 `EntityTemplate / TriggerBinding / ProjectileTemplate` 是可继续扩写的正式作者入口
- 当前主线应继续加厚正式内容与关卡模板

但随着“错误技自由组合植物系统”的讨论推进，出现了更关键的结构性问题：

1. `TriggerBinding` 更像运行时装配件，而不是顶层作者抽象。
2. 连续行为、生命周期、成长、接近触发、随机序列、扩展包新能力等需求难以自然落在 `TriggerBinding-first` 模型中。
3. 如果继续在旧模型上增加内容，后续重构成本会随着资产规模线性上升，并且更容易被局部特判绑架。

换句话说，当前最核心的矛盾已经从“内容还不够多”变为：

> 顶层作者模型是否还能承载项目未来的组合性目标。

## 3. 目标

本次决策明确想达成的目标：

1. 正式终止“继续围绕旧作者模型加厚内容”作为当前主线。
2. 允许主线转向 `Mechanic-first` 重构。
3. 明确重构边界，避免演化成一次失控的“重写整个引擎”。
4. 保护现有底层运行时与验证资产，不因顶层重构而无谓清零。

## 4. 非目标

本次决策不打算直接解决：

- `Mechanic` 的全部正式字段与资源协议
- 一级 family 的最终冻结版本
- 扩展包新注册协议的最终字段
- 首批 archetype 的具体实现细节
- 全部旧资源的一次性迁移计划

这些内容将在后续 ADR 中继续处理。

## 5. 备选方案

### 方案 A：保持当前路线不变

内容：

- 继续按“第七阶段输入准备”主线推进
- `Mechanic-first` 仅保留为讨论草案

优点：

- 不打断当前内容生产
- 风险最低

缺点：

- 会继续在旧作者模型上堆内容
- 未来重构成本更高
- 容易在“短期继续做内容”中错过最佳纠偏窗口

未选原因：

- 当前最核心的结构问题已经不是“内容不足”，而是“作者抽象不对位”。

### 方案 B：切换主路线，进入重构阶段

内容：

- 主线从“继续加厚内容”切换为 `Mechanic-first` 重构
- 将旧路线留档
- 把新主线聚焦于顶层模型、编译链和正式能力位

优点：

- 能在资产规模进一步膨胀前完成抽象纠偏
- 与错误技自由组合目标更一致
- 避免继续被 `TriggerBinding-first` 绑架

缺点：

- 短期会明显降低新增内容速度
- 需要同步修正 wiki 主叙事与工程优先级

选择原因：

- 当前结构问题已经成为主矛盾，继续扩内容只会放大后续迁移成本。

### 方案 C：直接做全栈重构

内容：

- 从顶层作者模型一直重做到最底层运行时主干

优点：

- 理论上最干净

缺点：

- 风险最高
- 周期最长
- 现有验证与 runtime 资产会被大面积打碎

未选原因：

- 当前讨论尚未得出“底层执行骨架也错误”的结论。

## 6. 最终决策

正式选择：

### 6.1 路线选择

选择 **方案 B：切换主路线，进入 `Mechanic-first` 重构阶段**。

这意味着：

- “第七阶段输入准备”不再作为当前主线继续推进
- 后续主线以顶层模型重构、编译链建立、正式能力位冻结为核心

### 6.2 重构范围选择

在重构边界上，明确选择：

> **保留底层执行骨架与验证基础设施，重做上层作者模型与编译装配链。**

即：

**保留：**

- `EventBus`
- `EventData`
- `RuleContext`
- `EffectExecutor`
- 当前投射物空间模型与大部分 projectile runtime
- `BattleBoardState / WaveRunner / Validation` 主框架

**重做：**

- 顶层作者模型
- 编译层与 `RuntimeSpec`
- `EntityFactory` 当前的直接装配流程
- `TriggerBinding` 的顶层作者入口地位
- 目前散落在实体 root 内的专属强逻辑组织方式

## 9. 验证方式

这项决策本身不以“代码通过测试”作为完成标准，而以“工程与文档主线切换完成”作为验证目标。

完成信号包括：

1. 存在正式 ADR，明确路线切换与重构边界。
2. 旧路线已归档。
3. wiki 中出现 `Mechanic-first` 讨论纪要与 ADR 输入文档。
4. 后续正式 ADR 和协议文档开始围绕新主线展开。

## 10. 风险与未决项

### 风险

1. 主线切换后，短期正式内容增量会放缓。
2. 如果后续 family / compiler / controller 治理不严格，`Mechanic-first` 仍可能膨胀失控。
3. 如果双轨期控制不好，可能形成新的长期技术债。

### 未决项

- 顶层作者模型是否彻底替换为 `Archetype + Mechanic[]`
- `Compiler / RuntimeSpec` 的精确层次
- 一级 `Mechanic` family 的正式冻结版本
- 扩展包注册与信任边界
- 迁移期长度与首批验证闭环
