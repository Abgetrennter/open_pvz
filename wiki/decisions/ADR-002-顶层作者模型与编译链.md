## 1. 决策摘要

项目正式采用 `Archetype + Mechanic[]` 作为新的顶层作者模型，并建立显式的多阶段编译链：

`Archetype -> NormalizedMechanicSet -> RuntimeSpec -> RuntimeAssembler -> Runtime Nodes`

旧的 `EntityTemplate / TriggerBinding / ProjectileTemplate` 不再作为正式顶层作者入口继续扩写，而是降级为底层编译产物、后端资源或短期迁移兼容层。

### 当前实现状态（2026-04-22）

截至 2026-04-22，这份 ADR 对应的主实现已经落地：

- 已存在正式 archetype 目录：
  - `data/combat/archetypes/plants`
  - `data/combat/archetypes/zombies`
  - `data/combat/archetypes/field_objects`
- 编译与装配桥接已经接通：
  - `scripts/core/runtime/combat_content_resolver.gd`
  - `scripts/core/runtime/mechanic_compiler.gd`
  - `scripts/core/runtime/runtime_spec.gd`
  - `scripts/battle/entity_factory.gd`
- `EntityFactory` 已经能够消费 archetype 路径产出的 `RuntimeSpec`
- 主仓 cards 已经优先引用 wrapper archetype，而不是 skeleton archetype
- 旧 `EntityTemplate / TriggerBinding / ProjectileTemplate` 当前仍保留，但只作为：
  - 后端资源
  - 编译桥接素材
  - legacy 对照路径

## 2. 背景与问题

当前仓库的正式作者入口主要围绕：

- `EntityTemplate`
- `TriggerBinding`
- `ProjectileTemplate`
- `default_params`

这套模型在“最小事件链 + 第一批正式内容 + 第一批错误技样例”阶段是成功的，但随着目标转向“可自由组合、可扩展、可随机生成的机制系统”，它暴露出结构性问题：

1. `TriggerBinding` 更像运行时规则绑定，而不是作者真正想表达的能力单元。
2. 连续行为、生命周期、状态阶段、随机序列等能力很难自然落在旧模型里。
3. `EntityFactory` 当前既承担实例化，又承担大量隐式装配与兼容逻辑，未来容易继续膨胀。

换句话说，问题已经不是“旧模型还能不能勉强承载”，而是：

> 顶层作者抽象是否已经落在了错误的层级上。

## 3. 目标

本次决策的目标是：

1. 明确新的正式作者模型。
2. 把“作者表达”和“运行时装配”彻底分层。
3. 建立一个能支撑错误技、扩展包和连续行为的显式编译链。
4. 为后续 `Controller / State / Lifecycle / 扩展包 type 注册` 留出稳定落点。

## 4. 非目标

本次决策不直接解决：

- 所有 `Mechanic` family/type 的最终字段细节
- 所有旧资源的具体迁移顺序
- 编译器实现语言风格和目录最终细节
- 所有验证场景的同步改造方案

## 5. 备选方案

### 方案 A：`EntityTemplate + Mechanic[]`

内容：

- 在旧模板上直接增加 `mechanics`
- 旧模板继续保留顶层地位

优点：

- 改动较小
- 目录与心智迁移成本较低

缺点：

- 旧模型仍然占据顶层
- 容易形成 `EntityTemplate` 和 `Mechanic` 双中心
- 作者会持续困惑“哪些能力应该写模板字段，哪些应该写 mechanic”

未选原因：

- 这类做法更像“旧模型继续加一层”，无法真正完成抽象纠偏。

### 方案 B：`Archetype + Mechanic[]`

内容：

- 新建 `PlantArchetype / ZombieArchetype / ProjectileArchetype`
- 顶层作者资源只表达身份、底盘、基础数值和 `Mechanic[]`
- 旧模板体系退为后端资源或迁移兼容层

优点：

- 顶层抽象清晰
- 更符合“能力组合结果”这一项目哲学
- 更适合错误技生成器和扩展包面向机制层工作

缺点：

- 文档与资源体系要同步迁移
- 初期编译链和运行时装配改造量较大

选择原因：

- 它能从根上消除旧模型的层级错位。

### 方案 C：长期双轨共存

内容：

- `EntityTemplate / TriggerBinding` 和 `Archetype / Mechanic[]` 长期并存

优点：

- 迁移阻力最小

缺点：

- 架构复杂度最高
- 很容易拖成长期技术债
- 无法形成统一作者心智

未选原因：

- 双轨可以短期存在，但不应成为正式长期架构。

## 6. 最终决策

### 6.1 顶层作者模型

正式采用：

`Archetype + Mechanic[]`

作为新的唯一正式作者模型方向。

建议的 archetype 类型包括：

- `PlantArchetype`
- `ZombieArchetype`
- `ProjectileArchetype`

其中 `Archetype` 顶层只承载：

- 身份与命名
- 底盘与基础数值
- 放置与阵营基本信息
- `Mechanic[]`

### 6.2 旧模型的定位

`EntityTemplate / TriggerBinding / ProjectileTemplate` 的定位调整为：

- 编译产物
- 运行时后端资源
- 短期迁移兼容层

并明确：

- 不再继续作为正式顶层作者规范扩写
- 后续新增内容不应再优先写旧模型

### 6.3 编译链

正式采用多阶段编译链：

1. `Archetype`
2. `NormalizedMechanicSet`
3. `RuntimeSpec`
4. `RuntimeAssembler`
5. `Runtime Nodes`

其中：

- `NormalizedMechanicSet` 负责规范化、去冲突、预算归一化
- `RuntimeSpec` 负责提供稳定的中间执行规格
- `RuntimeAssembler` 负责把规格实例化成现有 runtime 结构

### 6.4 工厂职责调整

当前 `EntityFactory` 的职责未来应收敛为：

- 消费 `RuntimeSpec`
- 实例化节点与组件
- 不再直接消费顶层作者资源并承担全部兼容逻辑

## 7. 影响范围

### 脚本

- `scripts/battle/entity_factory.gd`
- 新增 `scripts/core/compiler/`
- 新增 `scripts/core/defs/archetype_*.gd`
- 新增 `RuntimeAssembler`

### Resource

- 新增 `Archetype` 资源目录
- 新增 `Mechanic` 资源目录
- 新增 `RuntimeSpec` 资源目录或中间对象

### 场景与验证

- 验证场景未来应优先引用新的 archetype 入口
- 现有验证资产尽量在装配层适配，不直接重写所有场景

### 文档

需要新增或重写：

- `Mechanic-first 架构总览`
- `Archetype 协议`
- `Compiler / RuntimeSpec` 协议
- 原 `模板与装配边界` 进入待替代状态

## 8. 迁移与兼容

本 ADR 明确：

- 新模型是正式方向
- 旧模型仅允许短期迁移兼容

当前不承诺：

- `EntityTemplate` 会长期保留为正式作者入口
- `TriggerBinding` 会继续承担顶层规范职责

兼容期的长度与过渡策略，在 [ADR-005 扩展包接入与迁移策略](ADR-005-扩展包接入与迁移策略.md) 中统一冻结。

## 9. 验证方式

这项决策的完成信号不是“文档存在”，而是下面这些事情成立：

1. 存在新的 `Archetype` 资源类型。
2. 存在 `Mechanic -> RuntimeSpec` 的显式编译链。
3. 工厂或装配器开始消费 `RuntimeSpec`，而不是直接消费顶层作者资源。
4. 至少一批 archetype 能通过验证链运行。

建议的首批验证对象：

- 向日葵
- 豌豆射手
- 卷心菜投手
- 基础僵尸
- 土豆雷或樱桃炸弹

## 10. 风险与未决项

### 风险

1. 如果 `NormalizedMechanicSet` 边界不清，复杂度会重新堆回编译器。
2. 如果 `RuntimeSpec` 过于临时化，后续很难成为稳定中间层。
3. 如果旧模型兼容期过长，实际效果会退化成双轨长期共存。

### 未决项

- `RuntimeSpec` 是否落地为正式 `Resource`
- archetype 目录结构与命名规范
- 卡片、关卡、波次是否统一引用 archetype ID
