## 1. 决策摘要

项目正式将：

- `Controller`
- `State`
- `Lifecycle`

提升为一级 family，并明确：

1. 连续行为不再继续默认为实体 root 内硬编码逻辑。
2. 阶段状态与生命周期节点成为正式一等能力位。
3. 随机与 mechanic-local runtime state 采用确定性协议，纳入正式运行时设计。

## 2. 背景与问题

当前仓库在连续行为和状态层面已经有基础资产：

- 投射物连续行为模型已经成立
- `BaseEntity` 已经有轻量状态与标记系统
- 部分实体 root 中存在硬编码 controller 逻辑，如：
  - `ZombieRoot` 的近战咬
  - `LawnMower` 的扫掠状态机

但这还不是一个正式、统一、可组合的结构。

如果继续保持当前状态，会遇到三类问题：

1. 连续行为只能不断写进 root 脚本，无法形成正式组合能力位。
2. 生命周期行为和阶段状态只能零散实现，难以表达土豆雷武装、Sun-shroom 成长等机制。
3. 随机效果（如乱序轮转抛射物）缺少正式的 mechanic-local runtime state 与可复现协议。

## 3. 目标

1. 为持续状态机提供正式承载位。
2. 为阶段状态和生命周期节点提供正式协议。
3. 让随机机制与私有运行时状态进入统一框架。
4. 为原版复杂行为和错误技行为提供统一落点。

## 4. 非目标

本次决策不直接冻结：

- 所有 controller type 的完整列表
- 所有状态字段的完整数据结构
- 所有随机日志字段的完整协议

## 5. 备选方案

### 方案 A：继续允许写在 root 脚本里

优点：

- 实现最快

缺点：

- 连续行为无法成为正式组合能力
- root 脚本会继续膨胀
- 与 `Mechanic-first` 目标冲突

未选原因：

- 只能延续旧问题，不能解决旧问题。

### 方案 B：正式引入 `Controller / State / Lifecycle`

优点：

- 连续行为、阶段状态、生命周期有了明确边界
- 更适合错误技生成、扩展包和验证

缺点：

- 需要额外设计 controller blackboard、状态切换和触发顺序

选择原因：

- 它是 `Mechanic-first` 能真正覆盖原版复杂行为的必要条件。

## 6. 最终决策

### 6.1 Controller family

正式引入 `Controller` family。

其职责是承载：

- 咬
- 扫掠
- 跳跃攻击
- 冲锋
- 潜水
- 其他每帧持续状态机

实现上建议新增：

- `ControllerRegistry`
- `ControllerComponent`
- controller-local blackboard/runtime state

### 6.2 State family

正式引入 `State` family。

其职责是承载：

- 阶段状态
- 成长
- 武装
- 狂暴
- 持久机制状态

并明确：

- 阶段状态不等同于当前轻量 status effect
- `status effect` 继续保留用于临时控制和数值修正
- `State` 更偏实体阶段与持久模式切换

### 6.3 Lifecycle family

正式引入 `Lifecycle` family。

其职责是承载：

- `on_place`
- `on_spawned`
- `on_armed`
- `on_removed`
- `on_state_enter`
- `on_expire`

### 6.4 随机与 mechanic-local runtime state

正式决定：

- 随机全部采用确定性随机协议
- 允许 mechanic-local runtime state

典型数据包括：

- shuffle bag
- current index
- cycle
- burst queue
- controller blackboard

seed 建议按下面层次派生：

- battle seed
- entity seed
- mechanic seed

## 7. 影响范围

### 连续行为

- `ZombieRoot` 近战逻辑未来应迁移到 controller
- `LawnMower` 扫掠未来应迁移到 controller

### 状态系统

- `BaseEntity` 当前状态系统需要拆分“临时 effect 状态”和“阶段状态”

### 生命周期

- 放置、武装、成熟等行为不再只靠外部事件和局部脚本拼接

### 随机

- 错误技随机效果需要纳入统一可复现体系

## 8. 迁移与兼容

当前决定：

- 短期允许旧 root 逻辑继续存在
- 但它们属于待迁移对象，而不是正式长期结构

优先迁移目标建议：

1. `ZombieRoot` bite
2. `LawnMower` sweep
3. 土豆雷武装
4. Sun-shroom 成长

## 9. 验证方式

建议新增或更新专项验证：

1. `bite_controller_validation`
2. `lawn_mower_controller_validation`
3. `lifecycle_on_place_validation`
4. `state_transition_validation`
5. `deterministic_shuffle_cycle_validation`

完成信号：

- 至少一类连续行为从 root 迁移到 controller
- 至少一类阶段状态行为通过 `State + Lifecycle` 表达
- 至少一类随机机制通过固定 seed 稳定复现

### 第二阶段完成状态（2026-04-22）

截至 2026-04-22，第二阶段最小闭环已经落地，并满足下列完成信号：

1. 已新增最小 `ControllerRegistry + ControllerComponent` 路径。
2. zombie archetype 已可通过 `Controller.bite` 进入共享 bite 运行时。
3. 已新增最小 `Lifecycle.on_spawned` 路径。
4. lifecycle archetype 已可在 `entity.spawned` 时编译并执行一次共享 payload。
5. 已通过的第二阶段专项验证：
   - `archetype_lifecycle_validation`
   - `archetype_zombie_runtime_validation`

当前边界仍然保持为：

- lifecycle 目前只正式打通 `on_spawned`
- controller 目前只正式打通 `core.bite`
- `State` family 仍未进入正式编译落地
- 随机机制的正式协议与日志体系仍属于后续阶段

## 10. 风险与未决项

### 风险

1. controller 如果设计成万能脚本容器，会重新变成无边界扩展口。
2. `State` 和临时状态效果边界如果不清，会让协议非常混乱。
3. 如果随机状态没有统一日志，很难调试组合行为。

### 未决项

- 一个实体是否允许多个 controller 并存
- controller 与 rule/payload 的优先级顺序
- 生命周期节点是否全部映射为事件，还是部分走内部运行时钩子

5. 补 deterministic random 的日志与验证出口。
