# 当前阶段与实现路线

## 这页解决什么问题

现有 wiki 已经把很多想法拆成了专题，但仓库当前几乎还是文档先行状态。真正需要先明确的，不是“系统理论上能扩到多大”，而是“现在第一版到底先做什么”。

这页的目标是把当前思路收束成一条可执行主线。

---

## 一句话定位

`Open PVZ` 当前最合理的定位不是“完整开放式引擎”，而是：

> 先做一个基于 Godot 的 PVZ-like 原型，用触发器 + 效果树 + 随机生成去验证“错误技系统”能否形成稳定、有趣、可扩展的核心循环。

这意味着：

- 先验证玩法闭环
- 再抽象引擎层
- 最后再考虑 ECS、编辑器、Mod 生态和社区分享

---

## 当前主线思路

### 1. 把系统拆成两条线

当前设计里其实混着两件事：

- 一条是“生成线”：如何随机造出一株植物
- 一条是“运行线”：这株植物进入战场后如何响应事件并执行效果

当前阶段应优先保证运行线可用，再让生成线接上来。

更具体地说：

1. 先把事件、触发器、效果树、执行器跑通
2. 再把随机生成器接到这些运行时结构上
3. 最后才讨论更完整的内容生产工具

### 2. 运行时核心保持简单

当前最稳的运行时抽象仍然是这几个对象：

- `TriggerDef` / `TriggerInstance` / `TriggerStrategy`
- `EffectDef` / `EffectNode` / `EffectStrategy`
- `Context` / `State`
- `EventManager`

这套抽象已经足够支撑第一版，不需要一开始就引入完整 ECS、复杂消息总线或大而全的渲染分层。

### 3. 随机生成是内容来源，不是第一优先难点

随机生成当然是项目特色，但它不应该压在最前面。

正确顺序应该是：

1. 先准备少量手写的触发器和效果
2. 先验证这些机制在运行时能稳定执行
3. 再引入三层生成器，把“手写组合”换成“随机组合”

否则会很容易出现一个问题：

- 你分不清 bug 来自生成器，还是来自运行时本身

## MVP 应该长什么样

### 最小闭环

第一版应只追求跑通下面这条链：

1. 生成或手动创建一株植物
2. 该植物至少拥有 1 个触发器
3. 触发器绑定 1 棵效果树
4. 游戏内出现一个明确事件，例如 `game.tick` 或 `projectile.hit`
5. 触发器条件满足后执行效果树
6. 效果树至少能产出一种可见结果：
   - 发射投射物
   - 直接造成伤害
   - 触发一次爆炸
7. 至少有一种连锁能继续触发后续效果

只要这条链稳定，项目就进入了“可以持续迭代”的状态。

### 建议的 MVP 范围

建议第一阶段只保留这些内容：

- 事件：
  - `game.tick`
  - `plant.damaged`
  - `projectile.hit`
  - `plant.died`
- 触发器：
  - `periodically`
  - `when_damaged`
  - `on_death`
- 效果：
  - `shoot`
  - `damage`
  - `explode`
  - 可选 `summon`
- 连续行为：
  - 直线飞行
  - 一种简单附加轨迹，例如正弦偏移或追踪
- 调试能力：
  - 打印事件链
  - 打印触发器命中情况
  - 打印效果树执行顺序

### 暂时不需要追求的内容

在 MVP 跑稳之前，以下内容都不应该成为阻塞项：

- 完整 ECS 化
- 高级渲染管线
- 可视化编辑器
- 热更新 Mod 系统
- 分享码与社区工坊
- 异步 PvP
- 复杂叙事包装

这些内容不是没价值，而是现在做会稀释主线。

## 当前推荐的工程策略

### 1. 先用 Godot 原生方式搭骨架

仓库当前还没有成熟运行时代码，所以第一版更适合：

- 用 Godot 节点和脚本快速搭原型
- 用清晰的数据结构表达 `Def / Instance / Strategy`
- 用少量场景验证投射物、命中、伤害、死亡

只有当运行时热点和结构痛点已经明确时，再考虑是否向 ECS 迁移。

结合 [外部项目调研：PVZ-Godot-Dream](24-外部项目调研-PVZ-Godot-Dream.md) 的结果，这里再进一步收紧为：

- 全局服务优先放在 `Autoload`，例如事件总线、场景注册、调试服务
- 战斗单位优先采用“角色根节点 + 行为组件”组织
- 投射物优先采用“根节点 + MovementComponent”组织
- 第一阶段不要为 ECS 预埋太多抽象层

这不是退而求其次，而是更符合 Godot 的实际开发效率。

### 2. 把“可观测性”当作核心功能

这个项目天然会产生大量组合与连锁，不把调试能力做好，后续开发会非常痛苦。

第一版就应该具备：

- 事件广播日志
- 触发器命中日志
- 效果树 DFS 执行日志
- 连锁深度日志

这部分的重要性不低于新增一个效果类型。

### 3. 用硬边界防止系统失控

现有文档里已经有共识，这部分应该直接进入第一版实现：

- 效果树深度上限
- 事件连锁深度上限
- 冷却与防刷屏
- 参数范围白名单

这不是后期优化，而是系统设计的一部分。

### 4. 定义层先用 Godot Resource，不强行先上外部 JSON

原始设计里多次把 `TriggerDef`、`EffectDef` 写成外部 JSON，这在长期是合理的，但第一阶段不必强求。

结合参考实现，更实际的顺序是：

1. 先用 Godot `Resource` 或脚本内静态表定义最小 `Def`
2. 跑通编辑器内可配置、可实例化的运行时闭环
3. 再决定是否补 JSON 导入、Mod 热加载和外部包格式

这样做的好处是：

- 更贴近 Godot 工作流
- 少一层解析和校验链路
- 更容易在编辑器中直接调试
- 能把注意力放在规则运行时，而不是文件格式

## 建议的阶段划分

### Phase 1: 机制闭环

目标：

- 跑通单个战斗演示
- 跑通触发器到效果执行
- 跑通至少一种投射物命中连锁

交付物：

- 最小事件系统
- 最小触发器系统
- 最小效果系统
- 最小投射物移动组件
- 一套演示场景

### Phase 2: 随机生成接入

目标：

- 将植物能力从手写配置切换到随机生成
- 验证生成结果是否可玩、可解释、可控

交付物：

- 三层生成器
- 基础权重池
- 命名与展示的简化版本

### Phase 3: 内容扩展与稳定性

目标：

- 增加更多触发器、效果和轨迹
- 做性能与安全防护
- 建立回归验证方式

交付物：

- 更完整的效果库
- 验证脚本与统计结果
- 调试面板或日志视图

### Phase 4: 工具链与生态

目标：

- 让系统从“能开发”走向“能扩展”

交付物：

- 编辑器
- Mod 规范
- 分享与导入导出能力

## 当前结论

目前最清晰的思路可以收束为四句话：

1. 先把“植物响应事件并产生连锁结果”这条链做实
2. 随机生成是第二步接入，不是第一步难点
3. 现阶段优先做 Godot 原型，不要过早追求完整引擎化
4. ECS、编辑器、社区生态都应该建立在可玩的最小闭环已经成立之后

如果后续要继续整理 wiki，优先检查每篇文档是否回答了这个问题：

> 它是在帮助当前原型落地，还是只是在扩写未来愿景？
