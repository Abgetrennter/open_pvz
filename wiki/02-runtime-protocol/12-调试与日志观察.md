# 调试与日志观察

- 状态：当前事实


> 本文是 `DebugService` 的使用手册，回答"出问题时看什么、怎么开关、日志格式是什么"。`DebugService` 是整个运行时的观察枢纽，**任何空间、事件、时序相关的 bug 都应优先从日志入手，而非目测 Demo**。

相关源码：[autoload/DebugService.gd](../../autoload/DebugService.gd)

---

## 一、五类日志

`DebugService` 内部维护 5 个环形缓冲：

| 日志类别 | 触发源 | 默认开关 | 容量 |
| --- | --- | --- | --- |
| `event_log` | `EventBus.event_pushed` 信号（所有事件） | 开 | 128 |
| `trigger_log` | `record_trigger_execution` | 开 | 128 |
| `effect_log` | `record_effect_execution` | 开 | 128 |
| `runtime_snapshot_log` | `record_runtime_snapshot`（每帧或步进） | 开 | 128 |
| `protocol_log` | `record_protocol_issue`（协议违规） | 开 | 64 |

所有开关以 `DebugService.enable_*_logging` 形式存在，可在运行时切换。环形缓冲超限时最旧条目丢弃，**不会抛错**。

---

## 二、读哪一份日志

根据问题类型选择入口：

| 现象 | 先看 |
| --- | --- |
| 抛射体未命中 / 命中错目标 | `event_log` 过滤 `projectile.hit` + `runtime_snapshot_log` 对比位置 |
| 触发器没响 | `trigger_log`（`fired=false` 说明订阅到了但条件不过） |
| 效果执行异常 / 数值不对 | `effect_log` 查 `success` 与 `notes` |
| mode 行为不对 / 手动技能不触发 | `event_log` 过滤 `battle.mode_*` / `input.action.*`，再对照 `runtime_snapshot_log` |
| 模板定义加载失败 | `protocol_log`（严重等级 `warning` / `error`） |
| 实体数量 / 位置 / 血量不对 | `runtime_snapshot_log` 按 frame 追 |
| 事件链深度爆炸 | `event_log` 过滤 `depth >= 5` |

## 三、事件链追踪字段

`event_log` 每条包含：

```
event_name  = 事件名（StringName）
event_id    = 本次事件唯一 ID
chain_id    = 所属事件链 ID（用于关联因果）
depth       = 事件链深度（当前最外层通常从 1 开始）
source      = 源实体调试名
target      = 目标实体调试名
value       = 载荷（damage / amount / …）
tags        = 事件标签数组
```

**chain_id 是定位"这一发伤害从哪来"的关键**：同一个外部 tick 触发的链路共享 chain_id，通过过滤可以把"Trigger → Effect → Projectile → Hit → Damage"拼成完整叙事。

## 四、控制台打点

以下场景会**直接 `print` 到控制台**（与环形缓冲独立）：

- 协议违规：`[Protocol][SEVERITY][scope] message`
- 运行时快照：`[RuntimeSnapshot] frame=… time=… scenario=… entities=…`
- 事件追踪（白名单事件）：`[EventTrace] projectile.spawned/hit/expired、entity.damaged/died`

控制台流式输出适合**验证场景运行时快速扫读**；需要结构化分析则读 `build_export_payload()` 返回的 JSON。

## 五、在验证场景里使用

验证场景通过 `BattleManager` 调用 `build_export_payload()`，把 5 类日志一次性写入：

- 单次运行：`artifacts/validation/<timestamp>_<label>/debug_logs.json`
- 批量运行：`artifacts/validation/batch_<timestamp>/<scenario>/debug_logs.json`

JSON 结构：

```json
{
  "config": { "enable_event_logging": true, ... },
  "event_log": [ ... ],
  "effect_log": [ ... ],
  "trigger_log": [ ... ],
  "runtime_snapshot_log": [ ... ],
  "protocol_log": [ ... ]
}
```

典型分析流程：

1. 用 `jq` 或脚本过滤 `event_log` 中 `event_name == "projectile.hit"`，核对命中次数与目标。
2. 比对 `runtime_snapshot_log` 中 `frame` 对应的实体 lane / position / health。
3. 若断言失败，从 `protocol_log` 查是否有定义级别问题被静默吞掉。

## 六、常见坑

### 1. 不要用 `print` 代替 `DebugService`

`DebugService` 的日志会进入批量回归产物，`print` 不会。新增调试信息一律走 `record_*` 方法。

### 2. 环形缓冲会丢历史

单场景运行时间长、事件密集时，可能丢掉早期条目。**把 `MAX_EVENTS` 临时调大**（在 `DebugService.gd` 顶部常量），或缩短场景。

### 3. `enable_event_logging = false` 会让所有分析失效

性能测量前可以关，但关掉后 `debug_logs.json` 会是空 —— 别把关掉后的回归产物当作"通过"。

### 4. `_print_event_trace` 只打白名单事件

想看 `game.tick` 这类高频事件的流，得去读 `event_log`，不要等控制台 —— 控制台白名单里没有它。

## 七、与验证场景的协作

- 验证场景的通过判据（`BattleValidationRule`）本质上是**对 `event_log` 做模式匹配**。
- 如果一个场景"看起来对但 pass 失败"，先打开 `debug_logs.json`，对齐事件名 / tags / value / 次数范围。
- 如果一个场景"pass 但行为不对"，说明断言规则太宽 —— 补强 `validation_rules`，而不是放过。
