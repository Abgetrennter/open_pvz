# 事件模型

- 状态：当前事实


> 事件模型首先服务开放规则引擎，其次才服务植物、僵尸等具体内容表达。

---

## 事件模型的职责

事件模型回答的是：

- 什么叫“发生了一件事”
- 一件事在事件链里如何被表示
- 后续效果如何安全地继续传播

如果事件模型只围绕植物和僵尸的现有行为列清单，就会很快失去扩展性。

---

## 推荐的事件视角

当前更推荐用**语义事件**而不是“单位清单事件”来组织系统。

例如：

- `entity.spawned`
- `entity.damaged`
- `entity.died`
- `projectile.hit`
- `game.tick`

必要时，具体类型信息放在 `Context` 或 `event.core` 中，而不是直接把引擎事件名绑死在某个现有单位体系上。

这能带来的好处：

- 更利于扩展
- 更利于迁移旧内容
- 更利于让错误技系统复用同一套事件主干

---

## 三阶段模型

每个高层语义事件都可以拆成三个阶段：

| 阶段 | 作用 | 可修改性 |
|------|------|----------|
| `BeforeX` | 准备 / 修改输入 | 可写 |
| `OnX` | 事件正式发生 | 只读 |
| `AfterX` | 结果 / 连锁 | 只读 |

例子：

```text
BeforeDamage -> 调整伤害、取消
OnDamage -> 伤害正式生效
AfterDamage -> 连锁、附加行为
```

这套模型的意义在于：

- 保持语义清晰
- 允许有限度修改
- 为连锁提供稳定落点

---

## 当前建议的核心事件

第一阶段建议先只稳定少量高价值事件：

### 游戏级事件

- `game.tick`

### 通用实体事件

- `entity.spawned`
- `entity.damaged`
- `entity.died`

### 投射物事件

- `projectile.spawned`
- `projectile.hit`
- `projectile.expired`

如果后续要区分植物和僵尸，应优先通过实体类型字段表达，而不是立刻裂变出大量专用事件名。

---

## EventData / Context 结构

建议的基础结构仍然是：

```json
{
  "core": {
    "source": "entity_id",
    "target": "entity_id",
    "value": 10,
    "tags": ["projectile"]
  },
  "runtime": {
    "event_id": "uuid",
    "depth": 1,
    "timestamp": 123456
  },
  "ext": {}
}
```

### `core`

存放引擎可理解的语义字段。

### `runtime`

存放事件链追踪和安全边界信息。

### `ext`

存放扩展内容的附加字段。

当前建议不要再默认使用面向“mod 私有对象树”的复杂结构，第一阶段先把核心协议稳定下来。

---

## 事件传播规则

当前建议采用以下规则：

1. 事件由 `EventBus` 广播
2. 触发器或阶段逻辑消费事件
3. `EffectExecutor` 处理行为
4. 行为可以继续追加新事件
5. 新事件回到 `EventBus`

核心链路：

```text
EventBus -> Trigger / Phase Logic -> EffectExecutor -> PushEvent -> EventBus
```

---

## 事件链边界

事件模型必须从一开始就带边界。

当前第一阶段就应明确：

- 链深上限
- 时间戳追踪
- 事件唯一 ID
- 调试记录

否则一旦进入错误技连锁，很快就会无法定位问题。

---

## 当前阶段最重要的结论

事件模型应首先服务：

- 开放规则引擎
- 效果组合
- 实体模板
- 连续行为回写

而不是首先服务：

- 某一批具体植物或僵尸的事件清单

这条原则不立住，后续扩展和错误技都会被反向绑架。
