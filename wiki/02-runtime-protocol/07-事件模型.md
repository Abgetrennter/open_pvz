# 事件模型

- 状态：当前事实


> 当前事件模型首先服务开放规则引擎，其次才服务植物、僵尸、battle mode 等具体内容表达。

---

## 事件模型的职责

事件模型回答的是：

- 什么叫“发生了一件事”
- 一件事在事件链里如何被表示
- 后续效果如何安全地继续传播

如果事件模型只围绕植物和僵尸的现有行为列清单，就会很快失去扩展性。

---

## 当前事件视角

当前主干实际采用**语义事件**而不是“单位清单事件”来组织系统。

例如：

- `entity.spawned`
- `entity.damaged`
- `entity.died`
- `projectile.hit`
- `game.tick`

必要时，具体类型信息放在 `EventData.core` 或执行期 `RuleContext` 中，而不是直接把引擎事件名绑死在某个现有单位体系上。

这能带来的好处：

- 更利于扩展
- 更利于迁移旧内容
- 更利于让高组合内容、扩展能力和 battle mode 复用同一套事件主干

---

## 当前基础事件结构

当前 `EventData` 采用三段结构：

| 区块 | 作用 |
|------|------|
| `core` | 引擎和策略直接消费的语义字段 |
| `runtime` | 事件链追踪字段 |
| `ext` | 扩展字段 |

当前 `core` 已包含的关键字段包括：

- `source_id` / `target_id`
- `source_archetype_id` / `target_archetype_id`
- `source_projectile_template_id` / `target_projectile_template_id`（仅抛射体）
- `source_lane` / `target_lane`
- `source_team` / `target_team`
- `value`
- `tags`

当前 `runtime` 已包含的关键字段包括：

- `event_name`
- `event_id`
- `chain_id`
- `depth`
- `timestamp`

---

## 当前核心事件族

当前主干已经稳定并长期使用的高价值事件至少包括：

### 游戏级事件

- `game.tick`

### 通用实体事件

- `entity.spawned`
- `entity.damaged`
- `entity.died`

### 投射物事件

- `projectile.spawned`
- `projectile.hit`
- `projectile.expired`

### battle / 输入 / mode 事件

- `battle.phase_changed`
- `battle.victory`
- `battle.defeat`
- `input.action.card_selected`
- `input.action.cell_clicked`
- `input.action.entity_clicked`
- `battle.mode_initialized`
- `battle.mode_rule_applied`

如果后续要区分植物和僵尸，应优先通过实体类型字段表达，而不是立刻裂变出大量专用事件名。

---

## 事件传播规则

当前主干采用以下规则：

1. 事件由 `EventBus` 广播
2. 触发器或阶段逻辑消费事件
3. `EffectExecutor` 处理行为
4. 行为可以继续追加新事件
5. 新事件回到 `EventBus`

核心链路：

```text
EventBus -> Trigger / Phase Logic -> EffectExecutor -> PushEvent -> EventBus
```

---

## 事件链边界

事件模型必须从一开始就带边界。

当前第一阶段就应明确：

- 链深上限
- 时间戳追踪
- 事件唯一 ID
- 调试记录

否则一旦进入高频连锁、扩展 effect 或 mode 规则叠加，很快就会无法定位问题。

---

## 当前阶段最重要的结论

事件模型应首先服务：

- 开放规则引擎
- 效果组合
- 编译后运行时
- 连续行为回写
- battle mode 组织层

而不是首先服务某一批具体植物、僵尸或小游戏特判清单。
