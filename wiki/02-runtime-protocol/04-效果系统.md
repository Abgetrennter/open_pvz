# 效果系统

- 状态：当前事实

> 效果系统负责承接离散执行链中的“发生了什么”。在当前主线中，它仍是共享运行时层，但已经不再等于正式顶层作者模型。

---

## 文档定位

本文主要回答：

- 效果系统在当前架构中的职责。
- 它与 `Payload` family、扩展 effect、legacy 资源的关系。
- 为什么它仍然重要，但不该再覆盖作者模型叙事。

本文不负责：

- 把旧 effect 装配路线重新写成顶层入口。
- 代替 `Payload` family 的完整协议说明。

---

## 当前定位

当前效果系统应理解为：

> Trigger 与编译链落到运行时后的共享执行层。

在当前正文里，效果系统同时承担三种来源：

1. legacy 路径中仍直接引用的 effect 定义。
2. `Payload` family 编译后落到运行时的离散效果。
3. 扩展包注册的新 effect 策略。

因此它是主干的一部分，但不是正式作者的第一落点。

---

## 与 `Payload` family 的关系

当前正式作者写的是 `Payload` mechanic，而不是直接写 effect 树作为主入口。

编译后，payload 会映射到共享效果执行层，典型能力包括：

- `damage`
- `spawn_projectile`
- `spawn_entity`
- `explode`
- `produce_sun`
- `apply_status`

这也是为什么当前文档必须同时成立两句话：

- 效果系统仍是核心主干能力。
- 正式作者入口已经切换到 `Archetype + Mechanic[]`。

---

## Damage Layer Policy 与暴露态

`damage`、`spawn_projectile` 与 `explode` 当前都需要遵守暴露态命中口径：

- 未显式声明时，`target_exposure_states` 默认是 `["ground"]`。
- `flying`、`airborne`、`submerged`、`underground` 等特殊暴露态必须由效果、投射体模板或 runtime override 显式 opt-in。
- `ProjectileRoot` 在候选目标阶段也会按 `target_exposure_states` 过滤，避免“不能打空中”的投射物只靠最终伤害层拦截。

`damage` 与 `spawn_projectile` 还支持 `damage_layer_policy`，用于把一次伤害如何进入 `HealthComponent` 的多层 HP 路由说清楚。v1 只冻结两个字段：

| 字段 | 说明 |
|------|------|
| `bypass_layer_kinds` | 跳过指定 layer kind，例如投手类绕过 `shield`，但不绕过 `attachment` 或 `helm` |
| `spillover` | 是否允许过伤继续传递；默认允许 |

`spawn_projectile` 会把 policy 放进 projectile runtime overrides。投射体直接伤害和 on-hit damage 都继承同一 policy，因此“投手绕过门板”不需要在 projectile、zombie 或 BattleManager 中写特判。

`dispel_flying` 当前也按统一口径过滤：先看 `target_exposure_states`，再看 `min_height` 与 `max_weight_class`。这使 Blover 类效果只影响显式处于飞行/空中暴露态、且重量允许的目标。

### `spawn_entity`

`spawn_entity` 是共享效果层的正式能力位，可由 `Payload` mechanic 编译生成。当前参数支持：

| 字段 | 说明 |
|------|------|
| `archetype_id` | 要生成的实体 archetype |
| `x_offset` / `x_position` / `spawn_position` | 横向偏移、绝对横向位置或上下文位置 |
| `lane_id` / `lane_offset` | 显式 lane 或基于 owner/target lane 的相对偏移 |
| `spawn_overrides` | 传给 `BattleSpawner` 的受控覆盖参数 |
| `spawn_reason` | 生成原因，进入运行时记录与验证上下文 |

多 lane 召唤、巨人一次性投 Imp 等行为应通过 `spawn_entity` + trigger/State 守卫表达，不在父实体脚本或 `BattleManager` 中写特判。

## 与扩展包的关系

扩展包当前可以在受控边界内继续扩 effect 能力位。

当前已进入稳定回归的扩展 effect 样例包括：

- `hit_split`
- `spawn_entity`
- `apply_status`
- `knockback`
- `chain_bounce`
- `aura`
- `delayed_trigger`
- `delayed_explode`
- `mark`

扩展 effect 仍必须满足：

- 走注册中心
- 走验证场景
- 走 guardrail

不能旁路主干执行链。

---

## 与旧模型归档的关系

当前仓库中的 effect 运行时服务：

- archetype 编译结果
- 扩展 effect 策略

旧 effect 装配方式只允许作为历史背景出现，不再作为运行时兜底或新增内容路线。

---

## 当前验证入口

效果系统当前的重要验证包括：

- `terminal_explode_validation`
- `sun_resource_validation`
- `archetype_multi_payload_validation`
- `damage_layer_policy_bypass_shield_validation`
- `hit_policy_exposure_ground_default_validation`
- `hit_policy_exposure_flying_validation`
- `hit_policy_exposure_hidden_validation`
- `force_weight_filter_validation`
- 扩展包全部 effect 专项验证
- `extension_manifest_guardrail_validation`
- `extension_slot_guardrail_validation`

这些验证共同保证：

- 主仓 effect 能力位可用
- 扩展 effect 可用
- 无效配置会被 guardrail 拦截
