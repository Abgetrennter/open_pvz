# 战斗模式组织层

- 状态：当前事实

> 本文记录当前主干里已经落地的 battle mode 组织层：它如何挂在现有 `battle` 底盘之上、负责哪些职责、目前已经支持到什么程度，以及哪些内容仍然刻意留在 v1 之外。

---

## 这层解决什么问题

当前 battle 主链已经具备：

- 阳光与资源
- 棋盘、槽位与放置
- 卡片、冷却与手牌
- 波次、目标与胜败条件

但如果直接把更多小游戏差异继续塞进：

- `BattleManager`
- `BattleFlowState`
- `InputRouter`
- 各个 battle state

系统很快会退化成 mode 特判堆。

因此，当前主干正式采用一层 **battle mode 组织层**，用于把下面 4 件事资源化并组合起来：

1. 这一局是哪种 mode
2. 这一局允许哪些输入
3. 这一局附加哪些规则模块
4. 这一局如何解释 mode-level 胜负条件

---

## 当前核心对象

### 资源定义

- `BattleModeDef`
  - 模式身份与默认配置入口
- `BattleInputProfile`
  - 输入能力开关
- `BattleRuleModule`
  - 规则模块最小单元
- `BattleObjectiveDef`
  - mode-level 胜负条件

### 运行时宿主

- `BattleModeHost`
  - 解析 `BattleScenario.mode_def`
  - 合并 scenario override
  - 驱动规则模块生命周期
  - 处理 mode 输入请求
  - 评估 mode objective

### 挂接位置

- `BattleSubsystemHost`
  - 在标准 battle 子系统之后创建 `BattleModeHost`
- `BattleManager`
  - 只负责生命周期转发与只读获取，不承接 mode 特判

---

## 当前 BattleScenario 扩展

当前 `BattleScenario` 已经正式支持以下 mode 相关字段：

- `mode_def`
- `mode_rule_modules`
- `objective_override`
- `input_profile_override`
- `mode_input_requests`

其中：

- `mode_def` 是默认 mode
- `mode_rule_modules` 允许 scenario 级追加或覆盖默认模块
- `objective_override` / `input_profile_override` 允许局部覆盖 mode 默认配置
- `mode_input_requests` 用于验证和脚本化驱动 mode 输入语义

---

## 当前合并规则

`BattleModeHost` 当前按以下顺序解析 mode：

1. 读取 `scenario.mode_def`
2. 读取 `mode_def.input_profile / objective_def / rule_modules`
3. 用 `scenario.input_profile_override` 覆盖默认 input profile
4. 用 `scenario.objective_override` 覆盖默认 objective
5. 用 `scenario.mode_rule_modules` 按 `module_id` 覆盖或扩展默认模块

当前对规则模块的语义已经明确：

- 同 `module_id`：**scenario 模块覆盖 mode 默认模块**
- 不同 `module_id`：共同参与 mode 调度
- 最终执行顺序仍由 `priority` 排序

---

## 当前输入层接入

`InputRouter` 当前已经被 mode 配置化：

- `setup(...)` 支持 `input_profile`
- 可按 profile gating：
  - `enable_card_select`
  - `enable_card_place`
  - `enable_slot_click`
  - `enable_entity_click`
  - `enable_slot_drag`
  - `enable_swap`
  - `enable_manual_skill`
  - `enable_rhythm_hit`
  - `enable_cancel`

并且 battle mode v1 已补上两个 mode 入口：

- `request_entity_click(...)`
- `request_slot_drag(...)`

当前 `BattleModeHost` 负责把解析后的 `resolved_input_profile` 注入给实际 UI 输入接线入口。

---

## 当前规则模块模型

当前 mode 规则不走“每个 module 单独挂 Node”的方向，而是采用：

- `BattleModeHost`
- `BattleModeModuleRegistry`
- `module_id -> handler`

的轻量分发模型。

当前模块生命周期已正式存在：

- `on_mode_setup`
- `on_battle_start`
- `on_game_tick`
- `on_event`
- `on_mode_teardown`

### 当前已落地模块

- `manual_entity_skill`
  - 已能完成“选中实体 -> 点击目标格 -> 结算一次 mode 规则”的试点闭环
- `conveyor_cards`
  - 已有最小可观测实现，可通过 tick 轮转当前手牌顺序

### 当前未进入 v1 的模块

下面这些仍然只属于后续扩展方向，不视为当前事实：

- `slot_swap`
- `combo_score`
- `vase_populator`
- `izombie_role_flip`
- `bowling_lane_limit`

---

## 当前 objective 模型

`BattleObjectiveDef` 当前已经进入协议与运行时主链的类型包括：

- `all_waves_cleared`
- `survive_duration`
- `protect_archetype`
- `score_threshold`
- `combo_threshold`
- `clear_special_targets`
- `collect_resource`
- `defeat_named_spawn`

其角色边界是：

- `BattleFlowState` 仍然保留 battle phase 主状态机职责
- `WaveRunner` 仍继续负责旧 battle goal / defeat condition 主链
- `BattleModeHost` 只做 **mode-level 追加判定**
- `BattleModeHost` 只通过 `mark_victory()` / `mark_defeat()` 改变 battle 结果

这意味着当前 mode objective 是 **叠加器**，不是重写整套 battle phase。

---

## 当前试点与验证

battle mode 组织层当前已经不只是“文件存在”，而是有一组固定批量验证入口：

- `mode_basic_validation`
- `mode_manual_aim_lane_validation`
- `mode_input_profile_guard_validation`
- `mode_objective_score_validation`
- `mode_objective_protect_archetype_validation`
- `mode_objective_clear_special_targets_validation`
- `mode_objective_defeat_named_spawn_validation`
- `mode_module_override_validation`
- `mode_conveyor_cards_validation`
- `mode_no_mode_guardrail`
- `mode_protocol_guardrail_validation`

它们共同覆盖：

- mode 解析
- input profile gating
- 实体选择与目标格点击
- score objective 进度
- scenario module 覆盖语义
- conveyor 最小能力
- 无 mode 回退
- 协议层 guardrail

---

## 当前刻意不做的事

下面这些内容已经在设计中被识别，但当前仍不属于 mode 组织层 v1：

- 独立的 `BattleModeRuntimeState` 类
- 独立的 `BattleHUDProfile`
- 独立的 `BoardRuleLayer`
- 脱离棋盘语义的自由移动 action substrate
- 大型 UI / HUD mode 专属编排系统

这不是遗漏，而是当前 v1 的范围控制。

---

## 当前结论

当前 battle mode 组织层应按下面的口径理解：

1. 它已经是 battle 主干的一部分，不再只是草案。
2. 它负责 mode 的身份、输入、规则、目标四类组织问题。
3. 它的第一版重点是“规则偏转型小游戏”，而不是脱离 PVZ 棋盘语义的全新动作底盘。
4. 后续新增风味小游戏，默认应优先落在 mode 层，而不是重新回流到 `BattleManager` 特判。
