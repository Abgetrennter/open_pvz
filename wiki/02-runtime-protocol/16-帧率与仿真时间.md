## 核心判断

Open PVZ 不应把游戏规则推进绑定到显示器刷新率，也不应为了现代 UI 而锁死渲染帧率。

更合适的判断是：

> 渲染帧可以自由适配设备，规则仿真必须固定步长。原版 100Hz tick 是可迁移数值、确定性随机、验证复现和回放能力的共同基础。

因此，帧率体系应明确拆成三层：

1. 规则仿真帧：固定 100Hz，作为战斗运行时唯一权威时间。
2. Gameplay 物理帧：第一阶段与规则仿真帧对齐，使用 100Hz。
3. 渲染与 UI 帧：可变、解锁，只消费仿真结果，不决定规则结果。

这与 [战斗距离与棋盘度量](15-战斗距离与棋盘度量.md) 中的三层坐标模型对应：

| 维度 | 规则层 | 运行时连续层 | 显示层 |
|------|--------|--------------|--------|
| 时间 | 固定 simulation tick | 固定 `fixed_dt` 连续行为 | 可变 render delta |
| 空间 | slot / lane 语义 | combat world unit | UI 像素与相机投影 |

## 参考实现观察

### 原版 de-pvz

原版使用 `mFrameTime = 10`，即 100 tick/s。游戏世界以固定 10ms 步长推进。

关键特征：

- 全局逻辑计数器 `mAppCounter` 以 100Hz 增长。
- 攻击间隔、动画帧、状态持续、波次节奏都可以以 tick 表达。
- VSync 和渲染刷新率通过累积器适配，不改变逻辑 tick 语义。
- 慢动作和快进本质上是改变单位真实时间内消费的逻辑 tick 数。
- 累积器设置上限，避免卡顿后无限补帧。

这说明原版数值资产默认有两个强假设：

1. 时间粒度是 10ms。
2. 游戏规则与渲染刷新率解耦。

### PVZ-Godot-Dream

移植版主要依赖 Godot 默认回调：

- `_process(delta)` 处理部分移动、UI 和状态。
- `_physics_process(delta)` 默认 60Hz，处理部分攻击、HP 流失和投射物。
- Timer 节点处理攻击冷却、波次、阳光、状态效果。
- `Engine.time_scale` 控制整体速度。

这种方式接近 Godot 默认项目结构，但问题是时间语义分散：

- 多处逻辑隐含 60Hz 物理帧假设。
- Timer、process、physics process 混用后，确定性和验证复现较弱。
- `Engine.time_scale` 缩放 delta 和 Timer，但不等同于精确的规则倍速协议。

### Open PVZ 当前状态

当前 Open PVZ 已有固定语义 tick 的雏形，但尚未成为权威仿真时钟：

- `BattleManager.tick_interval = 0.25`，当前 `game.tick` 约为 4Hz。
- `GameState.current_time` 当前由 `_process(delta)` 推进，受渲染帧 delta 影响。
- 连续行为在 `_physics_process(delta)` 中执行，默认依赖 Godot 物理帧。
- 部分系统使用 `GameState.current_time`，少量逻辑仍使用墙钟时间，如 `Time.get_ticks_msec()`。

主要问题不是方向错误，而是时间口径尚未集中：

- `game.tick` 频率太低，不适合作为统一仿真 tick。
- 规则时间受渲染帧影响，不利于确定性。
- 固定验证、倍速、暂停、回放和原版数值迁移缺少统一基础。

## 三层帧模型

建议将帧率和时间明确分成三层。

### 1. 规则仿真帧

规则仿真帧描述“战斗世界推进了多少确定性步骤”，不描述显示器刷新。

建议参数：

- `SIMULATION_TICK_HZ = 100`
- `SIMULATION_DT = 0.01`
- `current_tick: int`
- `current_time = current_tick * SIMULATION_DT`

用途：

- `game.tick`
- 触发器 interval / start_delay
- 状态持续时间
- 卡片冷却
- 阳光、波次和模式目标
- 确定性随机派生
- 验证 deadline
- 回放与调试快照

约束：

- 不从渲染 delta 推导权威时间。
- 不使用墙钟时间决定玩法结果。
- 每个仿真 tick 的输入、随机、事件顺序应稳定。
- tick 事件中应携带 `tick_index`、`fixed_dt`、`game_time`。

### 2. Gameplay 物理帧

Gameplay 物理帧负责连续对象更新，但仍属于规则运行时，而不是显示层。

当前项目的连续行为主要是：

- 僵尸移动
- 抛射体 movement
- ControllerComponent
- Hitbox / swept segment 检测
- 割草机与地面伤害

第一阶段建议将 Godot physics tick 设置为 100Hz，让 `_physics_process` 与规则仿真帧对齐。这样可避免 100Hz 逻辑帧和 60Hz 物理帧之间再做一次插值或调度。

约束：

- Gameplay `_physics_process` 使用固定 `SIMULATION_DT` 语义。
- 不在物理帧中引入可变真实 delta 影响规则。
- 若 Godot 传入 delta 有微小误差，以仿真时钟的 `fixed_dt` 为准。
- 复杂渲染插值可在显示层处理，不反向影响 gameplay 状态。

### 3. 渲染与 UI 帧

渲染与 UI 帧描述“当前设备显示了多少画面”，不描述规则推进。

用途：

- UI 动画
- hover / drag 反馈
- 调试面板刷新
- 视觉特效
- 死亡淡出、抖动、屏幕反馈
- 相机适配与 CanvasLayer 缩放

约束：

- `application/run/max_fps` 不用于限制规则速度。
- UI 可以解锁高刷新，也可以由 VSync 控制显示体验。
- UI 可使用 `_process(delta)`，但不得修改权威规则时间。
- 视觉 Tween 和粒子可以使用真实时间或显示时间，但命中、伤害、冷却不得依赖它们。

## SimulationClock / GameState 分工

建议将权威时间集中到一个薄对象。可以扩展 `GameState`，也可以新增 `SimulationClock`。

推荐最小职责：

- 保存 `tick_hz`
- 保存 `fixed_dt`
- 保存 `current_tick`
- 派生 `current_time`
- 处理暂停和倍速下的 tick 消费
- 提供 `reset()`、`advance_tick()`、`snapshot()`

不应承担的职责：

- 不管理实体列表。
- 不执行触发器。
- 不处理波次。
- 不做 UI 动画。
- 不读写战斗内容资源。
- 不知道具体植物、僵尸或扩展包能力。

如果保留在 `GameState` 中，应避免让 `GameState` 膨胀为战斗协调器。`GameState` 只保存跨系统共享的运行时事实，调度仍由 `BattleManager` 或专门的 runtime driver 完成。

建议事件字段：

```gdscript
tick_event.core["tick_index"] = GameState.current_tick
tick_event.core["fixed_dt"] = GameState.fixed_dt
tick_event.core["game_time"] = GameState.current_time
```

## 原版数值迁移口径

采用 100Hz 固定仿真后，原版时间数值可以直接或简单映射。

时间映射：

| 原版表达 | Open PVZ 表达 |
|----------|---------------|
| `N` tick | `N` simulation tick |
| `N` tick | `N / 100.0` 秒 |
| 10ms | `SIMULATION_DT = 0.01` |
| 每 tick 执行一次 | 每 `game.tick` 执行一次 |

空间映射应与 [战斗距离与棋盘度量](15-战斗距离与棋盘度量.md) 结合：

| 原版表达 | 迁移语义 |
|----------|----------|
| `80 px` 横向格距 | `1.0 slot` |
| `px_per_tick` | `slots_per_sec = px_per_tick / 80.0 * 100.0` |
| 原版射程 px | `range_slots = px / 80.0` |

最终运行时换算：

```text
seconds = original_ticks / 100.0
slots = original_px / 80.0
world_distance = slots * BattlefieldMetrics.slot_spacing
world_speed = slots_per_sec * BattlefieldMetrics.slot_spacing
```

这意味着：

- 原版时间数值按 tick 直通。
- 原版水平距离先进入 slot 语义。
- Open PVZ 的 UI 分辨率、Camera 缩放和素材尺寸不参与规则迁移。

## 与距离语义化路线的联动

[战斗距离与棋盘度量](15-战斗距离与棋盘度量.md) 中的 `*_slots` 字段和本文的固定仿真时间应共同定义运行时数值语义。

建议拆分推进：

1. `scan_range_slots`、`impact_radius_slots`、`distance_slots` 只依赖棋盘度量，可先接入 `BattlefieldMetrics`。
2. `speed_slots_per_sec`、`move_speed_slots_per_sec` 同时依赖棋盘度量和权威仿真秒，应等 `SimulationClock.fixed_dt` 或等价机制落地后再批量迁移。
3. legacy `speed`、`move_speed` 在迁移期解释为 legacy world unit/s，但运行时推进应逐步改为 fixed tick 消费。

统一公式：

```text
world_distance = slots * BattlefieldMetrics.slot_spacing
world_speed = slots_per_sec * BattlefieldMetrics.slot_spacing
world_delta_per_tick = world_speed * SimulationClock.fixed_dt
```

验证口径：

- 射程/半径验证：改变 `slot_spacing` 后，world range 按比例变化。
- 速度验证：固定 tick 数下，位移稳定等于 `slots_per_sec * elapsed_simulation_seconds * slot_spacing`。
- 不使用渲染帧数、墙钟时间或 UI Tween 结果判断 gameplay 位移。

## 倍速与暂停

倍速应由仿真时钟控制，而不是依赖 `Engine.time_scale` 作为规则协议。

推荐语义：

| 模式 | 规则含义 |
|------|----------|
| 1x | 每真实秒消费 100 个仿真 tick |
| 2x | 每真实秒消费 200 个仿真 tick |
| 0.5x | 每真实秒消费 50 个仿真 tick |
| pause | 不消费仿真 tick，UI 可继续运行 |
| step | 手动消费 1 个或 N 个仿真 tick |

实现建议：

- 使用 accumulator 累积真实时间。
- 按 `time_scale` 消费固定 tick。
- 设置每帧最大补 tick 数，避免卡顿后长时间追帧。
- validation/headless 可使用固定步进模式，不依赖真实时间流逝。

当前实现入口：

- `GameState.set_simulation_speed(speed)`：设置战斗规则倍速，负数由调用侧避免，运行时最小钳到 `0.0`。
- `GameState.set_simulation_paused(paused)`：暂停或恢复规则仿真；暂停状态不阻止手动固定步进。
- `GameState.step_simulation_ticks(count)` / `BattleManager.step_simulation_ticks(count)`：手动消费 N 个仿真 tick，用于单步、回归验证和未来调试 UI。
- `BattleScenario.simulation_speed_override`、`simulation_paused_on_start`、`validation_fixed_step_count`、`validation_finish_after_fixed_steps`：验证场景可声明固定步进，不依赖真实时间。
- `simulation.control` 事件和 `DebugService.simulation_log`：记录倍速、暂停、手动步进和 max-step clamp，便于验证与调试。

当前验证入口：

- `simulation_control_validation`：验证 2x、pause-on-start、headless 固定 3 tick，并确认 `game.tick` 正好发出 3 次。
- `tools/check_runtime_metrics_time_guardrails.ps1`：扫描 gameplay 中的墙钟/Timer 误用，并扫描变更内容资源里的裸 legacy 距离/速度字段。

`Engine.time_scale` 的建议定位：

- 可用于视觉调试。
- 不作为战斗规则倍速的权威机制。
- 不用于确定性验证、回放或内容平衡。

## 累积器与卡顿策略

渲染帧解锁后，runtime driver 需要累积器。

推荐口径：

```text
accumulator += real_delta * simulation_speed
while accumulator >= SIMULATION_DT and steps_this_frame < max_steps:
    advance_simulation_tick()
    accumulator -= SIMULATION_DT
```

建议 guardrail：

- `max_accumulated_time`：例如 0.2s，对齐原版“最多补约 200ms”的思想。
- `max_steps_per_frame`：例如 8 到 20，根据 headless 和交互运行分别配置。
- 超出补帧上限时记录 debug/protocol warning，而不是悄悄改变规则。

需要区分两种运行模式：

1. 交互运行：保护响应性，避免一次补太多 tick 卡住画面。
2. 验证运行：优先确定性和完整推进，可允许更高 steps 上限或纯固定步进。

## 与 Godot 物理帧的关系

第一阶段建议：

- 设置 Godot `physics/common/physics_ticks_per_second` 或等价项目配置为 100。
- Gameplay 连续行为继续放在 `_physics_process`，但内部使用 `GameState.fixed_dt`。
- `game.tick` 与 gameplay physics 同步发出。

不建议第一阶段采用：

- 100Hz 逻辑帧 + 60Hz 物理帧双时钟。
- gameplay 逻辑在 `_process(delta)` 里直接推进。
- 为每个系统单独维护 Timer 节点。

原因：

- 当前项目不依赖复杂刚体模拟。
- 抛射体已经有自定义 movement 和 swept segment。
- controller、detection、状态、波次都更适合统一仿真 tick。
- 双固定帧会引入额外调度和插值成本，不符合 KISS。

## 当前不做

- 不锁死渲染 FPS。
- 不把 UI 帧率作为规则时间。
- 不用 `Engine.time_scale` 作为正式战斗倍速协议。
- 不一次性重写所有 `_process` / `_physics_process`。
- 不把所有视觉 Tween 改成仿真 tick。
- 不修改冻结 Mechanic family。
- 不引入复杂回放系统作为第一阶段前置。
- 不为了原版迁移逐帧复刻所有动画细节。

|------|----------|
| 规则仿真频率 | 100Hz，`fixed_dt = 0.01` |

## 新问题建议处理

| 问题 | 建议处理 |
|------|----------|
| `tick_interval = 0.25` | 视为 legacy 语义 tick，迁移后不再作为 `game.tick` 间隔 |
| 周期触发器最小间隔 | 以 100Hz tick 支持 0.01s 粒度；Validator 可限制内容层最小值 |
| 状态持续时间 | 存储秒或 tick 均可，但运行时比较基于 `current_tick/current_time` |
| validation deadline | 使用仿真时间，不使用渲染时间 |
| wall clock 用途 | 只用于性能统计、日志实际时间，不决定玩法 |
| UI 动画暂停 | UI pause policy 独立于 battle simulation pause |
| headless 验证 | 支持固定步进跑满 N tick，减少机器性能差异 |
| debug 输出 | runtime snapshot 输出 `tick_hz`、`current_tick`、`fixed_dt`、`simulation_speed` |

## 额外遗漏点

### 1. 事件顺序与同 tick 调度

固定 tick 后，同一 tick 内可能发生多个事件链。

建议处理：

- 明确同 tick 内事件按发射顺序同步处理。
- 如果未来需要延迟到下一 tick，使用显式 deferred queue。
- 事件数据记录 `tick_index`，便于验证和回放定位。

### 2. 随机调用顺序

100Hz 只提供确定性基础，不自动保证随机调用顺序稳定。

建议处理：

- 所有随机继续走 `battle_seed` 派生链和 `ShuffleBag`。
- 避免在渲染/UI 中消费玩法随机。
- 随机结果记录必要 debug 信息。

### 3. 动画事件与规则事件

现代 UI 和动画不应反向决定伤害、命中或冷却。

建议处理：

- 规则事件由仿真 tick 决定。
- 动画可以跟随规则状态播放。
- 如果需要动画命中点，应把命中点编译为 tick offset，而不是读取当前渲染帧。

### 4. 输入采样

高刷新 UI 下，同一仿真 tick 前可能收到多个输入事件。

建议处理：

- 输入层将玩家请求排队，附带接收时的 UI 时间和目标 slot。
- 仿真 tick 开始时按确定顺序消费输入请求。
- 放置合法性仍由 `BattleBoardState` 判定。

### 5. 扩展包边界

扩展包不应替换核心时钟算法。

建议处理：

- 扩展内容可以声明 tick/seconds 参数。
- 扩展 runtime strategy 接收 `fixed_dt` 和 `GameState.current_time`。
- 不开放 clock replacement slot，除非未来有明确多人同步或回放需求。

### 6. 浮点稳定性

即使固定 tick，浮点位置仍可能出现边界问题。

建议处理：

- 时间比较使用统一 epsilon 或 tick 比较。
- 可用 tick 表达的 cooldown 优先用整数 tick。
- 距离边界验证覆盖边界内、边界上、边界外。

## 设计原则

### KISS

一套权威仿真时钟，一套固定步长。先让 gameplay physics 对齐 100Hz，避免引入双固定帧调度。

### YAGNI

不提前实现完整回放、网络同步或任意时间轴系统。先满足验证、迁移、倍速和 UI 解耦。

### DRY

把冷却、波次、触发器、状态持续和验证 deadline 统一到 `GameState.current_tick/current_time`，避免 Timer、墙钟和渲染 delta 各自解释时间。

### SOLID

- 单一职责：时钟只负责时间推进和查询。
- 开闭原则：新增模式、扩展内容和 UI 表现消费统一时间接口，不修改各自内部时间算法。
- 依赖倒置：规则系统依赖抽象仿真时间，而不是依赖 Godot 渲染帧或具体 Timer 节点。
