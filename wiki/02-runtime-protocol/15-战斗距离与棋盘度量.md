# 战斗距离与棋盘度量

- 状态：当前设计
- 背景：整理原版 `de-pvz`、移植版 `PVZ-Godot-Dream` 与 Open PVZ 当前实现中关于距离、像素、棋盘坐标和 UI 现代化的设计判断。
- 关联文档：
  - [帧率与仿真时间](16-帧率与仿真时间.md)
  - [战斗距离与仿真时间统一重构执行计划](../../plans/archive/runtime-metrics-time/战斗距离与仿真时间统一重构执行计划.md)

---

## 核心判断

Open PVZ 不应把原版的 `px` 直接理解为现代屏幕物理像素，也不应让战斗判定随 UI 缩放、窗口尺寸或显示器 DPI 改变。

更合适的判断是：

> 原版 `px` 是固定 800x600 设计画布中的规则单位。Open PVZ 应继承它背后的棋盘比例和玩法语义，而不是继承逐像素坐标。

因此，战斗距离应属于规则运行时和棋盘度量，UI 现代化应属于显示层和交互层。两者可以共享位置结果，但不能共享缩放责任。

## 参考实现观察

### 原版 de-pvz

原版在 `800x600` 固定画布内运行。`GameConstants.h` 中定义：

- `BOARD_WIDTH = 800`
- `BOARD_HEIGHT = 600`
- `LAWN_XMIN = 40`
- `LAWN_YMIN = 80`

`Board.cpp` 中的棋盘换算体现了固定设计单位：

- X 方向格子宽度约为 `80`
- 普通草地 Y 方向格子高度约为 `100`
- 泳池和屋顶多使用 `85`
- 屋顶按 X 位置追加坡度修正
- 植物、僵尸、投射物大量依赖 `mPosX/mPosY`、`GridToPixelX/Y`、`Rect` overlap

这说明原版的 `px` 同时承担了三件事：

1. 棋盘格距
2. 运行时碰撞和距离判定
3. 最终绘制坐标

这个设计适合固定分辨率时代，但不适合 Open PVZ 的现代化 UI 和可扩展规则引擎。

### PVZ-Godot-Dream

移植版更接近 Godot 场景坐标驱动。植物格子由场景节点组织，植物容器、碰撞区域和全局位置共同参与运行。

这种实现比原版更贴近现代引擎，但仍有明显局限：

- 棋盘坐标和场景节点结构绑定较深
- 距离语义主要来自节点位置和裸数值
- UI、交互和运行时位置较容易互相耦合

它适合作为迁移参考，但不适合作为 Open PVZ 的最终距离模型。

### Open PVZ 当前状态

当前 Open PVZ 已经具备更好的基础：

- `BattleBoardState` 使用 `board_slot_origin_x + slot_index * board_slot_spacing` 构造格子世界位置
- `BoardSlot` 保存 `lane_id`、`slot_index`、`world_position`
- `DetectionRegistry` 使用实体 `ground_position` 做距离检测
- `ProjectileMovementBase` 和投射物系统已区分地面位置和投射高度

主要问题不是方向错误，而是距离数值仍然偏“裸 world unit”：

- `scan_range = 900.0`
- `impact_radius = 20.0`
- `move_speed = 55.0`
- `speed = 220.0`
- `board_slot_spacing = 80.0 / 88.0 / 96.0`

这些数值可以工作，但缺少统一解释口径。随着棋盘形态、UI 适配和内容规模增长，裸数值会增加调参成本。

## 三层坐标模型

建议将坐标和距离明确分成三层。

### 1. 棋盘语义坐标

棋盘语义坐标描述“玩法位置”，不描述屏幕像素。

典型字段：

- `lane_id`
- `slot_index`
- `slot_type`
- `slot_tags`
- `placement_role`

用途：

- 放置合法性
- 占位关系
- 行列目标选择
- 关卡和模式规则

### 2. 战斗世界单位

战斗世界单位是规则运行时使用的连续坐标。它可以继续使用 Godot `Node2D` 的 world unit，但文档语义上应称为 `world unit` 或 `combat unit`，不要称为 UI 像素。

用途：

- 移动
- 距离检测
- 投射物轨迹
- hitbox / impact radius
- continuous controller

约束：

- 不随窗口大小变化
- 不随 UI 缩放变化
- 不随显示器 DPI 变化
- 验证场景应能在 headless 中稳定复现

### 3. 显示与 UI 像素

显示层负责把战斗世界投影到屏幕。

用途：

- Camera 适配
- CanvasLayer UI
- 高分辨率素材
- 视觉反馈
- 交互 hover 和点击呈现

约束：

- UI 缩放不得改变战斗判定
- 视觉节点不得成为放置合法性的主事实
- 像素级验证不应成为规则验证的默认依赖

## BattlefieldMetrics / BoardGeometry

建议引入一个集中式棋盘度量对象，但必须严格限制职责，避免演变成 God Class。

推荐命名：

- `BattlefieldMetrics`
- 或 `BoardGeometry`

它应是一个很薄的运行时值对象或纯查询对象，只负责“棋盘语义距离如何映射到战斗世界单位”。优先实现为 `RefCounted` 运行时对象；只有当后续确实需要独立保存或编辑 metrics 资源时，再考虑 Resource 化。

### 应承担的职责

- 保存或派生 `slot_origin_x`
- 保存或派生 `slot_spacing`
- 保存或派生 lane 的 Y 坐标
- 保存 playable 边界，如 `playable_min_x` / `playable_max_x`
- 提供 `slot_position(lane_id, slot_index)`
- 提供 `slots_to_world(slot_count)`
- 提供 `world_to_slot_index(x)` 之类纯几何换算

### 不应承担的职责

- 不管理实体列表
- 不判断种植是否合法
- 不扫描目标
- 不处理投射物命中
- 不管理波次
- 不读写 `GameState`
- 不发事件
- 不知道具体植物或僵尸类型
- 不负责 UI layout

它应该像尺子，而不是裁判。

### 与 BattlefieldPreset 的关系

推荐分工：

- `BattlefieldPreset`：数据入口，声明棋盘尺寸、slot 数、lane 数、特殊格配置和基础度量参数
- `BattlefieldMetrics`：运行时只读查询对象，从 preset / scenario 派生
- `BattleBoardState`：继续负责占位、放置、slot 状态和 placement 事件
- `DetectionRegistry` / `Projectile` / `Controller`：消费 metrics 提供的换算结果，而不是散落硬编码距离

`BattlefieldPreset` 不应直接长出大量运行时查询方法。它保持“配置数据”的角色；metrics 承担“尺子”的角色。这样可以避免 preset 既是配置、又是几何服务、又被运行时系统到处依赖。

### lane_y 来源

长期建议将 lane 的 Y 坐标纳入 battlefield 几何配置，而不是继续由 `BattleManager` 作为最终事实源。

迁移期可以采用兼容策略：

1. `BattlefieldPreset` 增加 `lane_origin_y` / `lane_spacing` 或 `lane_y_positions` 之类正式字段。
2. `BattlefieldMetrics` 创建时优先读取 preset / scenario。
3. 缺失配置时 fallback 到当前 `BattleManager.get_lane_y()` 逻辑。
4. 后续让 `BattleManager.get_lane_y()` 反向委托 metrics。

这样可以保持现有验证稳定，同时逐步把棋盘几何从战斗协调器中抽离。

## 距离表达建议

后续内容不建议继续优先写裸 `scan_range = 900.0`。更好的表达是语义距离。

候选字段：

- `scan_range_slots`
- `impact_radius_slots`
- `speed_slots_per_sec`
- `hitbox_size_slots`
- `distance_slots`
- `range_mode = full_lane`

字段命名采用 `_slots`，不采用更抽象的 `_board_units`。原因是 `slots` 直接表达“几格”，更符合 PVZ-like 内容作者的直觉；`board_units` 过于泛化，容易在后续文档和数据中失去明确含义。

示例口径：

| 行为 | 推荐语义 |
|------|----------|
| 豌豆射手整行索敌 | `range_mode = full_lane` |
| 小喷菇 / 海蘑菇 | `scan_range_slots = 1.5` |
| 大喷菇 | `scan_range_slots = 4.0` |
| 土豆雷近身触发 | `scan_range_slots = 0.8` |
| 普通豌豆命中半径 | `impact_radius_slots = 0.2 ~ 0.3` |
| 普通僵尸移动 | `speed_slots_per_sec = 0.6 ~ 0.8` |
| 普通直线投射物 | `speed_slots_per_sec = 3.0 ~ 4.0` |

这些值不是最终平衡表，只是表达方向：规则应优先绑定“几格远”“整行”“近身”，再由 metrics 转成 world unit。

速度字段中的 `per_sec` 指 **simulation second**，即由固定仿真时钟派生的规则秒，不是渲染帧 delta、墙钟时间或 UI 动画时间。结合 [帧率与仿真时间](16-帧率与仿真时间.md) 后，推荐解释为：

```text
world_speed = speed_slots_per_sec * BattlefieldMetrics.slot_spacing
world_delta = world_speed * SimulationClock.fixed_dt
```

原版 `px_per_tick` 迁移到 Open PVZ 时，应先按 100Hz 仿真 tick 转为 slot/s：

```text
speed_slots_per_sec = original_px_per_tick / 80.0 * 100.0
```

这保证空间语义和时间语义同时脱离 UI 像素与渲染帧率。

`full_lane` 应作为 targeting / detection 参数中的语义模式表达，例如：

```gdscript
{
	"detection_id": &"lane_forward",
	"range_mode": &"full_lane"
}
```

detection strategy 收到 `range_mode = full_lane` 后，通过 metrics 计算从 source 到 playable edge 的范围。不得用 `99999.0`、`4000.0` 这类超大数值模拟整行射程。

## 兼容策略

不建议一次性重写所有 `.tres`。

推荐渐进策略：

1. 保留现有 `scan_range`、`impact_radius`、`speed`、`move_speed`，视为 legacy world unit。
2. 新增可选语义字段，如 `scan_range_slots`、`speed_slots_per_sec`。
3. 编译链优先消费语义字段；缺失时回退 legacy 字段。
4. 新内容默认使用语义字段。
5. 旧内容按验证覆盖分批迁移。

这种方式避免打断现有验证，也避免为了坐标系统一次性重开所有内容资源。

## 设计原则

### KISS

不要引入复杂坐标引擎。先做一个小而稳定的 `BattlefieldMetrics`，只覆盖当前需要的 slot / lane / world 换算。

### YAGNI

不要提前支持任意曲面棋盘、六边形网格或完整编辑器。当前只需要支撑 PVZ-like 横向 lane + slot 模型，以及已有屋顶/水路扩展空间。

### DRY

将 `900.0`、`96.0`、`20.0` 这类重复散落的距离解释集中到 metrics 和编译层，避免每个 registry、controller、projectile resolver 自己解释“几格远”。

### SOLID

- 单一职责：metrics 只负责几何度量。
- 开闭原则：不同 battlefield preset 可以替换度量参数，不需要修改检测、投射物和 controller。
- 依赖倒置：运行时系统依赖抽象几何查询，而不是依赖具体场景节点或裸数字。

## 当前不做

- 不做逐像素复刻。
- 不让 UI 缩放影响战斗判定。
- 不把 `BattlefieldMetrics` 做成实体管理器。
- 不在第一步迁移所有 `.tres` 数值。
- 不改变冻结 Mechanic family。
- 不把坐标系统变成新的扩展点，除非后续确有跨内容包需求。

## 新问题建议处理

| 问题 | 建议处理 |
|------|----------|
| lane Y 配置形式 | 支持两层字段：`lane_origin_y + lane_spacing` 作为简洁默认；`lane_y_positions` 作为显式覆盖，存在时优先 |
| playable 边界 | 拆分为 `playable_min_x` / `playable_max_x`、`plant_zone_min_x` / `plant_zone_max_x`、`spawn_x`、`projectile_despawn_x`；第一阶段只消费 playable 与 despawn |
| X/Y 度量 | `slot_spacing` 明确只表示 X 方向格距；Y 方向使用 `lane_spacing` 或 `lane_y_positions` |
| hitbox 是否语义化 | `impact_radius`、`scan_range`、`detection_range` 优先支持 `_slots`；`hitbox_size` 默认继续使用 world unit，仅保留未来可选 `hitbox_size_slots` |
| 速度迁移范围 | 内容层移动和投射物支持 `*_slots_per_sec`，按固定仿真秒解释；视觉 Tween、粒子、UI 动画继续使用视觉/世界单位 |
| `range_mode` 范围 | 第一阶段只要求 `lane_forward` / `lane_backward` 支持 `full_lane`；multi-lane 复用每条 lane 的 full lane；`global_track` 不使用 full lane，继续用 global / explicit range 语义 |
| 迁移验证样本 | 按“整行射手、短射程、近身触发、投手、移动实体、割草机”六组推进 |
| debug 可观察性 | metrics 应进入 `BattleBoardState.get_debug_snapshot()` 或 validation report，只输出几何参数和版本，不输出实体关系 |

## 额外遗漏点

### 1. 朝向与阵营语义

当前 `lane_forward` 主要依赖 X 方向正负判断。后续如果出现僵尸侧可放置、反向植物、双向关卡或镜像模式，`forward` 不能只等价于 `+X`。

建议处理：

- metrics 或 battle context 提供 lane direction / team direction 查询。
- detection 参数保留 `lane_forward` / `lane_backward` 语义，但内部通过 direction policy 转换成 X 方向。
- 第一阶段仍维持当前 `+X` / `-X`，但文档标记为默认方向策略。

### 2. 非均匀棋盘与屋顶坡度

原版屋顶不是简单等距网格；同一 lane 的可视 Y 可能受 X 影响。Open PVZ 当前投射物已有 ground position / height 分离，未来屋顶、坡面和特殊地形可能需要更强的投影模型。

建议处理：

- 第一阶段 metrics 只提供 `slot_position()` 和 lane Y。
- 预留纯查询方法名：`ground_position_for(lane_id, x)`。
- 不在第一阶段实现复杂坡面，只避免把 lane Y 写死到多个系统。

### 3. 地面位置、视觉位置与高度

战斗距离应基于 ground position，但视觉节点可能有高度、影子、动画偏移和 hit height band。

建议处理：

- 文档明确：距离和索敌默认用 `get_ground_position()`。
- 高度命中继续由 `hit_height_band`、`max_hit_height`、projectile height 处理。
- 不把视觉 sprite 偏移纳入 metrics。

### 4. 数据版本与迁移可追踪性

一旦引入 `_slots` 字段，内容资源会进入 legacy 字段和语义字段并存阶段。

建议处理：

- 给 migration ledger 或文档表记录每批迁移范围。
- ProtocolValidator 先做 warning 风格提示，不立即硬禁 legacy。
- 后续可在资源或验证报告中记录 `distance_semantics_version`，但第一阶段不强制。

### 5. 扩展包边界

扩展包可能希望定义自己的 battlefield preset 或内容距离。metrics 如果过早开放为可替换服务，会增加安全和兼容成本。

建议处理：

- 扩展包可以贡献 battlefield preset 数据。
- 扩展包不得替换核心 `BattlefieldMetrics` 算法，除非未来正式开放 geometry slot。
- 扩展内容可以使用 `_slots` 字段，编译结果由核心 metrics 解释。

### 6. 输入映射与规则坐标一致性

输入层的 `BoardCoordinateMapper` 和战斗层 metrics 如果各自实现换算，可能出现点击格子与规则格子不一致。

建议处理：

- 输入映射应消费 `BattleBoardState` / metrics 暴露的 slot position。
- 不允许 UI 自己维护一套独立棋盘几何常量。
- 视觉 overlay 可以做 hover 区域，但合法性仍以 `BattleBoardState` 为准。

### 7. 浮点稳定性与边界判定

`slots_to_world()` 会产生浮点值，边界处可能影响“刚好进入范围”的判定。

建议处理：

- detection / hit 判定使用明确的 `<=` 边界口径。
- validation 增加边界内、边界上、边界外三个样本。
- metrics 不负责四舍五入；具体判定系统负责阈值比较。

### 8. 时间语义依赖

距离、半径和棋盘位置可以先语义化，但速度字段必须依赖统一仿真时间。若 `speed_slots_per_sec` 在权威 fixed tick 落地前大量进入内容，会出现“字段名是每秒，运行时却按可变 delta 消费”的口径风险。

建议处理：

- `scan_range_slots`、`impact_radius_slots`、`distance_slots` 可先进入编译链。
- `speed_slots_per_sec`、`move_speed_slots_per_sec` 应与 `SimulationClock.fixed_dt` 或等价权威时间同步接入。
- 速度验证同时检查距离和 tick 数：给定 `speed_slots_per_sec`、`slot_spacing` 和 `N` 个 fixed tick，最终位移应稳定等于预期 world distance。
