# 波次与组波系统

- 状态：当前事实

> 本文记录当前主干中 `WaveRunner`、显式 `WaveDef[]` 与规则化 `WaveRecipeDef` 的职责边界。历史设计讨论已归档到 `plans/archive/wave-runner-2026-05/`。

---

## 当前结论

当前波次系统采用三层模型：

1. **显式波次层**：`BattleScenario.wave_defs` 直接提供 `WaveDef[]`，用于 validation、showcase、剧情关和精调关卡。
2. **规则化组波层**：`BattleScenario.wave_recipe` 提供 `WaveRecipeDef`，由 `WaveComposer` 在 battle setup 阶段编译成普通 `WaveDef[]`。
3. **确定性执行层**：`WaveRunner` 只消费已经解析好的 wave plan，按 simulation time 推进 wave、投放 spawn entry、发事件，并连接现有胜败条件。

核心边界是：

> WaveRunner 支持自定义结果，不执行任意自定义脚本。

## 当前资源入口

### `BattleScenario`

当前 battle scenario 同时支持：

- `wave_defs`
  - 显式 `WaveDef[]`
  - 保持旧验证、showcase 和手工波次兼容
- `wave_recipe`
  - 可选 `WaveRecipeDef`
  - setup 阶段由 `WaveComposer.compile(...)` 编译成普通 `WaveDef[]`

二者可以共存。`WaveRunner.setup(...)` 会收集显式 `wave_defs`，再追加 recipe 编译结果，并按 `start_time` 排序。

### `WaveDef`

当前字段：

| 字段 | 说明 |
| --- | --- |
| `wave_id` | 稳定波次 id |
| `start_time` | 默认开始时间 |
| `spawn_entries` | `WaveSpawnEntry[]` |
| `wave_kind` | `normal` / `flag` |
| `advance_policy` | 可选 `WaveAdvancePolicyDef` |

### `WaveRecipeDef`

当前字段：

| 字段 | 说明 |
| --- | --- |
| `recipe_id` | 组波规则 id |
| `total_waves` | 编译生成的波次数 |
| `waves_per_flag` | 每几波标记一次 flag wave；小于等于 0 时不生成 flag wave |
| `start_delay` | 第一波延迟 |
| `base_spacing` | 默认波次间隔 |
| `base_budget` | 第一波基础预算 |
| `budget_per_wave` | 每波递增预算 |
| `flag_budget_multiplier` | flag wave 预算倍率 |
| `pool_def` | `WavePoolDef` 候选池 |
| `flag_entry` | flag wave 额外入口 |
| `advance_policy` | 编译后挂到每个 `WaveDef` 的推进策略 |
| `special_injection_rules` | 受控特殊注入规则 |

### `WavePoolDef` 与 `WavePoolEntryDef`

`WavePoolDef.entries` 保存候选 `WavePoolEntryDef[]`。

`WavePoolEntryDef` 当前字段：

| 字段 | 说明 |
| --- | --- |
| `archetype_id` / `archetype` | 目标僵尸 archetype；支持 id 或局部内联 resource |
| `power` | 消耗预算 |
| `weight` | 加权抽取权重 |
| `first_allowed_wave` | 从第几波开始允许出现，使用 0-based wave index |
| `lane_id` / `x_position` | 生成位置基础参数 |
| `required_spawn_tags` | `spawn.medium.*` 等入口需求 |
| `spawn_overrides` | 透传到 `BattleSpawnEntry.spawn_overrides` |

当 `required_spawn_tags` 为空时，`WaveComposer` 会尝试从 archetype tags 中提取 `spawn.medium.ground`、`spawn.medium.water`、`spawn.medium.roof`。

### `WaveAdvancePolicyDef`

当前字段：

| 字段 | 说明 |
| --- | --- |
| `policy_id` | 策略 id |
| `policy_kind` | 当前支持 `absolute_time` 与 `timer_with_health_threshold` |
| `min_wave_duration` | 提前推进前的最短运行时间 |
| `health_ratio_threshold` | 当前 wave objective-counted health 比例阈值 |
| `huge_warning_lead_time` | flag wave 前置提示时间 |

`timer_with_health_threshold` 使用 `GameState.current_time` 与当前 wave 已生成实体的有效血量比例，不使用 Godot `Timer` 或墙钟时间。

### `WaveInjectionRuleDef`

当前字段：

| 字段 | 说明 |
| --- | --- |
| `rule_id` | 注入规则 id |
| `wave_index` | 目标波次，0-based |
| `entry` | 要注入的 `WavePoolEntryDef` |
| `spawn_time_offset` | 注入 entry 的波内 offset |

它只负责把配置化 entry 注入到指定 wave，不开放任意 composer 脚本。

## 运行时数据流

```text
BattleScenario
  ├─ wave_defs
  │    └─ WaveRunner consumes as-is
  └─ wave_recipe
       ├─ WaveComposer.compile(recipe, battle_seed, battlefield_preset)
       ├─ emits ordinary WaveDef[]
       └─ WaveRunner consumes compiled waves

WaveRunner on game.tick
  ├─ update advance policy
  ├─ emit wave.huge_approaching when flag wave warning is due
  ├─ start due waves and emit wave.started
  ├─ resolve spawn via BattleSpawnResolver
  ├─ spawn via BattleSpawner
  ├─ complete waves and emit wave.completed
  └─ evaluate legacy battle goal / defeat conditions
```

`WaveComposer` 的随机性来自 `GameState.battle_seed` 派生的 `RandomNumberGenerator`，同一 recipe 与 seed 应得到稳定输出。

## 生成入口边界

`WaveComposer` 可以在编译阶段根据 spawn contract 预过滤不满足入口需求的 pool entry。当前支持的入口标签为：

- `spawn.medium.ground`
- `spawn.medium.water`
- `spawn.medium.roof`

运行时最终合法性仍由 `BattleSpawnResolver` 裁判。也就是说：

- recipe 负责减少明显不可能的候选；
- `BattleSpawnResolver` 负责实际 spawn zone、lane、位置和入口能力匹配；
- `WaveRunner` 不根据地形自行挑行，也不绕过 resolver。

## 自定义分层

| 自定义方式 | 作者维护什么 | 适用场景 | 运行时边界 |
| --- | --- | --- | --- |
| 显式波次 | `WaveDef[]` | validation、showcase、剧情关、精调关卡 | WaveRunner 直接执行 |
| 规则化组波 | `WaveRecipeDef + WavePoolDef + WaveAdvancePolicyDef` | 普通正式关卡、Adventure-like、Survival-like | setup 编译为普通 `WaveDef[]` |
| 受控注入 | `WaveInjectionRuleDef[]` | flag wave 额外 entry、少量特殊僵尸注入 | 只注入配置化 entry |
| 模式级改写 | `BattleModeDef + BattleRuleModule` | 锤僵尸、我是僵尸、墓碑关等特殊规则 | BattleModeHost 驱动 |

不属于当前开放能力：

- 扩展包通过任意 GDScript 接管 `WaveRunner` loop；
- `WaveRunner` 解释原版关卡 allowed table；
- `WaveRunner` 直接管理 GridItem、墓碑生命周期或特殊小游戏完整流程。

## 当前验证覆盖

波次系统当前进入以下固定验证：

| 验证场景 | 覆盖点 |
| --- | --- |
| `wave_flow_validation` | 显式波次、spawn、wave start/completion 与基础胜利闭环 |
| `wave_recipe_compile_validation` | recipe 编译成普通可执行 waves，包含 flag wave 插入 |
| `wave_advance_health_threshold_validation` | `timer_with_health_threshold` 提前推进与 `wave.huge_approaching` |
| `wave_recipe_spawn_zone_filter_validation` | recipe-level spawn tag 预过滤 |
| `wave_recipe_roof_spawn_zone_validation` | roof spawn zone 下的 recipe 过滤与 runtime resolve |
| `wave_recipe_special_injection_validation` | 受控特殊注入，不开放任意脚本 |
| `wave_guardrail_validation` | 既有波次守卫 |

正式内容映射归属见 `tools/formal_content_validation_map.json` 的 `wave_and_level_structure_v1`。

## 后续观察项

下面内容尚未升格为当前事实：

- 将 `WaveRecipeDef` 输出保存为调试 artifact；
- 从 zombie archetype metadata 自动派生完整 power / weight；
- survival endless 的长期预算曲线；
- trusted `wave_composer` 扩展 slot；
- 特殊小游戏对自然波次的完整禁用或替换协议。

这些方向需要单独设计与验证，不应隐式塞回 `WaveRunner`。
