# 性能基线实测

- 状态：当前事实


> 本文回答"当前项目的性能数字是多少"。它不是性能规范（规范见 [性能与安全防护](10-性能与安全防护.md)），而是**实测基线记录**。目的是让每次主干性能相关改动都能对照上一次基线，避免"感觉还行"。

---

## 一、为什么需要这篇

项目已经积累了 53 条验证场景与较多抛射体运动、事件链、效果执行路径。缺乏实测基线会导致：

- 协议级别的性能回退被淹没在功能验证里。
- 抛射体数量 / 事件链深度 / Effect 嵌套的容量假设只停留在文档。
- 无法回答"引擎还能塞多少内容"这个最常被问到的问题。

---

## 二、当前基线状态

**当前仓库尚未建立正式性能基线**。本页先定义测量协议，等首次完整测量完成后填入数字。

已知硬上限（来自现有代码约束）：

| 约束 | 数值 | 来源 |
| --- | --- | --- |
| 事件链最大深度 | 5 | `EffectExecutor` / `ProtocolValidator` |
| 事件历史缓冲 | 256 | `EventBus` |
| 调试事件 / 效果 / 触发器日志缓冲 | 各 128 | `DebugService` |
| 协议日志缓冲 | 64 | `DebugService` |

## 三、测量协议

任何后续性能测量必须遵循统一协议，结果才能纵向对比。

### 硬件基准

记录以下字段，放在报告顶部：

- OS / CPU 型号 / 核数 / 主频
- 内存容量
- GPU（即使项目是 mobile 渲染也要记，避免后续切桌面渲染时误判）
- Godot 版本（当前默认 `4.6.1-stable`）
- 物理引擎：Jolt

### 启动模式

- 默认使用 `--headless` 避免渲染噪声
- `--quit-after` 用于控制场景时长
- 关闭 `DebugService.enable_event_logging` 等开关前后各测一次（日志自身也耗时）

### 指标

| 指标 | 采集方式 | 说明 |
| --- | --- | --- |
| 帧平均耗时 (ms) | `Performance.TIME_PROCESS` + 场景内采样器 | 过滤前 60 帧预热 |
| 物理帧耗时 (ms) | `Performance.TIME_PHYSICS_PROCESS` | 投射体数量敏感 |
| 事件链平均深度 | `event_log` 统计 | depth 分布 p50/p90/p99 |
| Effect 平均嵌套 | `effect_log` 按 chain_id 聚合 | 检测是否触及硬上限 |
| 内存（GDScript 对象数） | `Performance.OBJECT_COUNT` | 检测实体/投射体泄漏 |

### 固定测量场景

为保证可比性，选定以下场景作为**性能基线套件**（实际接入前先创建这些验证场景）：

- `perf_baseline_idle`：空场景 5 秒，测基础开销
- `perf_baseline_linear_swarm`：单车道 20 个持续发射的 basic_shooter，60 秒
- `perf_baseline_parabola_swarm`：3 车道 lobber 混合 + 地面波次，60 秒
- `perf_baseline_chain_explosion`：级联爆炸场景（复用 `chain_explosion_cascade_validation`），30 秒
- `perf_baseline_full_demo`：demo_level 满载波次，单局到结束

这些场景**不进入默认 smoke 批**，仅按需或定期手动跑。

## 四、测量流程

```powershell
# 1. 清理历史产物
Remove-Item -Recurse -Force artifacts/perf

# 2. 依次运行基线场景（headless，日志开）
& 'E:\SDK\Godot\Godot_v4.6.1-stable_win64_console.exe' `
  --headless --path 'E:\Code\open_pvz' `
  'res://scenes/validation/perf_baseline_linear_swarm.tscn' `
  --validation-auto-quit `
  --validation-output-dir=artifacts/perf/linear_swarm

# 3. 整理报告（手工或脚本）
```

报告产物放在 `artifacts/perf/<timestamp>/`，同时在本页"五、历史基线"下追加一行。

## 六、何时必须重跑基线

下列任一情况，合并前应跑对应场景并更新表格：

- 修改 `EventBus` / `EffectExecutor` / `TriggerInstance` / `ProjectileRoot._physics_process` 的核心路径
- 修改 `Performance`-sensitive 的 `DebugService` 字段
- 新增高频事件（每帧触发、每实体触发）
- 一次性新增 5+ 个投射物模板或 10+ 个触发绑定
- 切换渲染后端、物理后端或 Godot 版本
