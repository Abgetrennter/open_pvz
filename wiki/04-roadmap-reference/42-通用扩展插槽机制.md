# 通用扩展插槽机制

- 状态：已落地 v1
- 实现基线：`ProjectileMovement` 插槽、`MechanicCompilerDef` 扩展接入、manifest trust/capabilities、extension/guardrail 验证

---

## 一、定义

通用扩展插槽是主仓提供的受控扩展入口：

```text
extension.json
-> contributor Resource
-> Slot Registry
-> Runtime dispatch by id
-> DebugService / validation
```

扩展包提供“贡献项”，主仓 registry 负责校验、注册和分发。运行时只依赖 registry，不为某个扩展能力追加硬编码 `match` 分支。

---

## 二、v1 已落地插槽

| Slot | 贡献项 | Registry | 运行时入口 | 说明 |
|------|--------|----------|------------|------|
| `projectile_movement` | `ProjectileMovementDef` | `ProjectileMovementRegistry` | `ProjectileRoot` | `linear / parabola / track` 已注册为 `core.*`，兼容旧别名 |
| `movement` | `MovementDef` | `MovementRegistry` | `MovementComponent` | `walk / leap_once` 已注册为 `core.*`，用于实体自主运动命令 |
| `mechanic_compilers` | `MechanicCompilerDef` | `MechanicCompilerRegistry` | `MechanicCompiler` | 扩展包可在冻结 family 下新增 type |
| `effects` | `EffectDef` | `EffectRegistry` | `EffectExecutor` | 保留现有成熟路径，并补齐基础 registry 行为 |
| `triggers` | `TriggerDef` | `TriggerRegistry` | `TriggerComponent` | 6 内置 trigger，扩展包可通过 strategy_script 注册 |
| `detections` | `DetectionDef` | `DetectionRegistry` | `TriggerRegistry` | 6 内置 detection，扩展包可通过 strategy_script 注册 |
| `controllers` | `ControllerDef` | `ControllerRegistry` | `ControllerComponent` | 4 内置 controller，扩展包可通过 strategy_script 注册 |

所有 registry 已统一继承 `RegistryBase`，共享注册、去重、信任检查、来源追踪和协议错误记录逻辑：

- `register_def()` / `unregister()` / `has()` / `get_def()` / `get_entry()` / `list_ids()` / `list_entries()` / `rebuild_registry()`
- contributor 字段统一为 `id`（废弃 `move_mode` / `effect_id` / `trigger_id` / `type_id`）和 `param_defs`（废弃 `condition_params`）
- 扩展包不得覆盖 `core.*`，重复 id 拒绝并记录 `protocol.issue`

---

## 三、硬约束

- `core.*` 由主仓独占，扩展包不得注册或覆盖。
- 扩展包不得新增 Mechanic family，只能在 11 个冻结 family 下新增 type。
- 默认不做跨包 override，重复 id 直接拒绝。
- 所有扩展注册失败必须进入 `DebugService.record_protocol_issue()`。
- 运行时代码插槽需要 `trust_level = "trusted_runtime"`。
- `data_only` 包只能注册资源内容，不能注册 projectile movement、compiler、controller 等运行时代码。

---

## 四、ProjectileMovement 插槽

扩展包通过资源注册 movement：

```text
extensions/<pack>/data/combat/projectile_movements/*.tres
```

资源脚本为 `ProjectileMovementDef`（继承 `RegistryContributorDef`），关键字段：

- `id`（统一 contributor id，如 `my_pack.zigzag`）
- `movement_script`
- `param_defs`
- `default_hit_strategy`
- `default_terminal_hit_strategy`
- `tags`

`ProjectileRoot` 通过 `ProjectileMovementRegistry.create_component(move_mode)` 创建组件。找不到 `move_mode` 时记录 `protocol.issue(scope = projectile_movement)`，并安全回退到 `core.linear`。

内置 movement 注册为：

- `core.linear`
- `core.parabola`
- `core.track`

兼容别名保留：

- `linear -> core.linear`
- `parabola -> core.parabola`
- `track -> core.track`

---

## 五、MechanicCompiler 扩展接入

扩展包通过资源注册 compiler：

```text
extensions/<pack>/data/combat/mechanic_compilers/*.tres
```

资源脚本为 `MechanicCompilerDef`（继承 `RegistryContributorDef`），关键字段：

- `id`（统一 contributor id，如 `my_pack.zigzag`）
- `family`
- `compiler_script`
- `param_defs`
- `tags`

compiler script 必须提供：

```gdscript
func compile(mechanic, archetype, merged_params: Dictionary) -> Dictionary
```

例如 `Trajectory.my_pack.zigzag` 表达为：

```text
family = Trajectory
id = my_pack.zigzag
```

compiler 输出 `movement_mode = my_pack.zigzag` 和 `amplitude / frequency` 等参数后，`spawn_projectile` 会按 `movement_mode` 查询 `ProjectileMovementDef.param_defs`，只放行该 movement 声明过的额外参数。

## 六、manifest 字段

v1 manifest 支持：

```json
{
  "pack_id": "my_pack",
  "enabled_by_default": false,
  "register": ["resources", "projectile_movement", "mechanic_compilers"],
  "trust_level": "trusted_runtime",
  "capabilities": ["projectile_movement", "mechanic_compiler"]
}
```

新增 register kind：

- `projectile_movement`
- `mechanic_compilers`

信任级别：

- `data_only`
- `rule_extended`
- `trusted_runtime`

未声明 `trust_level` 的旧包按保守规则兼容映射；现有 `minimal_chaos_pack / phase5_chaos_pack / phase5_guardrail_pack` 保持可运行。

## 七、新增扩展能力工作流 (RegistryBase 规范)

新增扩展 slot 必须遵循 `RegistryBase + RegistryConfig + ContributorDef` 规范，不再允许独立实现注册逻辑：

1. 定义 slot 边界：确认是否必须新增主仓插槽，不能用现有 registry 解决时再新增。
2. 定义 contributor Resource：必须继承 `RegistryContributorDef`（统一字段 `id`、`tags`、`param_defs`），额外字段通过 `@export` 追加。
3. 创建 Registry 单例：继承 `RegistryBase`，重写 `_make_registry_config()`（声明 slot_id、def_script、register_kind、extension_dir、required_trust）。
4. 实现 `_register_builtin_defs()`：构造内置 contributor def，调用 `register_def()`。
5. 实现 `_on_def_registered(entry)`：注册策略 callable / strategy_script 实例化。
6. 接入运行时 dispatch：运行时只通过 `get_def(id)` 获取 contributor，通过注册策略分派。
7. 接入 manifest：在 `ExtensionPackCatalog.ALLOWED_REGISTER_KINDS` 中添加 register_kind。
8. 新增验证场景：
   - smoke 场景：验证 registry 初始化无协议问题（`min_count=0, max_count=0` on `protocol.issue`）。
   - guardrail 场景：验证重复 id / core 命名空间 / 信任级别 被拒绝。
9. 跑 `extension` 与 `guardrail` 分层批量验证。

## 八、验证基线

该机制落地时已覆盖 121 场景基线；当前完整验证矩阵已扩展到 145 场景。通用插槽相关覆盖包括：

- `projectile_movement_zigzag_smoke_validation`
- `projectile_movement_guardrail_validation`
- `extension_runtime_trust_guardrail_validation`
- `extension_slot_guardrail_validation`
- `registry_duplicate_id_guardrail` — RegistryBase 重复 id 拒绝
- `registry_core_override_guardrail` — RegistryBase core.* 命名空间保护
- `registry_trust_level_guardrail` — RegistryBase data_only 信任级别拒绝
- `registry_slot_extension_probe` — 扩展 detection/controller/trigger/effect 正向分发、`get_entry().source` 和 rebuild 稳定性
- `registry_strategy_script_guardrail` — 扩展 trigger/detection/controller 非法 strategy script 拒绝
- `detection_registry_smoke` — DetectionRegistry 初始化无协议问题
- `controller_registry_smoke` — ControllerRegistry 初始化无协议问题
- `trigger_registry_smoke` — TriggerRegistry 初始化无协议问题
- 既有 `extension` 与 `guardrail` 分层批量回归
