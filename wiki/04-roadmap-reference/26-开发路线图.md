# 开发路线图

- 状态：当前路线

> 本文是唯一的前进路线页，只回答“从当前状态继续推进，顺序应该是什么”。当前已经成立的事实请看 [当前阶段与实现路线](../01-overview/23-当前阶段与实现路线.md)。

---

## 当前路线判断

当前路线的中心不是再解释旧阶段，而是围绕三个明确目标继续推进：

1. 收紧 legacy 兼容层，避免新内容回流旧模型。
2. 保持正式内容、扩展边界和批量回归同步增长。
3. 在 battle mode 组织层上继续加厚小游戏与 objective 覆盖。

补充事实：

- battle 模式组织层 v1 已经从草案进入正式实现，当前应沿 `BattleModeHost / RuleModule / InputProfile / Objective` 继续扩展，而不是把小游戏差异重新塞回 `BattleManager`、`BattleFlowState` 或输入分发特判。

---

## 当前优先级

### P1. legacy 兼容层职责收紧

路线目标不是立即物理删除旧资源，而是先把职责边界写死：

- 新内容不再允许优先走旧模型。
- 旧模型只承担兼容、对照和后端素材职责。
- `backend_entity_template*` 的存在不再被解读为主线双轨。

### P2. 正式内容与验证同步

后续每次新增 archetype、扩展能力或正式内容，都默认同步补：

- 正向验证
- 必要的 guardrail
- 正文中的当前事实

### P3. battle mode 专项能力继续加厚

当前 battle mode 组织层 v1 已成立，但后续仍适合继续补：

- 更多 objective 子类型专项验证
- 更多 rule module 样例
- 更明确的 HUD / board / mode 边界

### P4. 扩展系统可观察性和包治理

在不重开双轨的前提下，继续补：

- 扩展注册失败报告
- 包级禁用/白名单
- manifest 核心字段形式化

---

## 当前退出信号

Mechanic-first 收口阶段的退出信号应理解为：

1. 当前 archetype 不再默认依赖 legacy 后端字段作为主链路说明。
2. 旧作者模型只剩只读兼容层职责，不再承担新增内容入口。
3. 相关验证与文档口径同步完成，没有正文与 ADR 打架的情况。

---

## 后续方向怎么排

一旦上面的退出信号成立，后续优先级建议按下面理解：

### 方向 A：继续加厚正式战斗内容

前提是当前主干与验证已经稳定，不再需要靠 legacy 兼容层兜主线。

### 方向 B：补作者工作流与轻量工具

包括：

- 更清晰的 archetype 编写体验
- 更稳定的验证接入体验
- 扩展包治理和可观察性

### 方向 C：再考虑外围组织层

如选关、存档、图鉴、商店、花园等外围组织层，应继续后置，不能反过来挤压当前主线。

---

## 当前不应优先的方向

- 重新讨论旧路线是否继续作为当前正文主线。
- 把完整社区工坊、完整编辑器或大型 UI 包装当成当前优先项。
- 为了快速出内容而放松对 legacy 回流的约束。
- 在 mode 层已经成立之后，又重新把小游戏差异回流到 `BattleManager` 或输入特判。
