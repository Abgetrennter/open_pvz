## 一、使用原则

### 1. 两个参考项目的角色

| 参考项目 | 本地路径 | 角色 | 适合回答 |
|----------|----------|------|----------|
| de-pvz | `vendor/de-pvz/` | 原版规格优先来源 | 原版数值、枚举、tick 语义、资源 ID、棋盘坐标、行为条件 |
| PVZ-Godot-Dream | `vendor/PVZ-Godot-Dream/` | Godot 表达参考 | 节点组织、组件拆法、场景资源、动画/表现、Godot 工程落点 |
| Open PVZ | 当前仓库正式资源树 | 目标格式 | Mechanic-first、Resource 数据、验证场景、扩展插槽 |

`de-pvz` 的结论优先级高于 `PVZ-Godot-Dream`。`PVZ-Godot-Dream` 只能解释 Godot 中如何组织，不应覆盖原版规格。

### 2. 禁止事项

- 不直接复制 vendor 代码进入 Open PVZ。
- 不修改 `vendor/`。
- 不把 `PVZ-Godot-Dream` 的具体单位脚本模式移植成 Open PVZ 的实体特判。
- 不从外部 wiki 或记忆直接取最终数值；原版数值应回到 `de-pvz` 源码确认。
- 不用参考项目结论绕过 `CombatArchetype + CombatMechanic[]` 主链。

## 二、de-pvz 常用锚点

| 问题类型 | 优先文件/目录 | 主要锚点 | 用途 |
|----------|---------------|----------|------|
| 植物枚举与基础定义 | `vendor/de-pvz/ConstEnums.h` | `SeedType`、`PlantSubClass` | 确认原版类型、分类、枚举命名 |
| 植物费用/冷却/发射周期 | `vendor/de-pvz/Lawn/Plant.cpp` | `gPlantDefs[]` | 提取 `mSeedCost`、`mRefreshTime`、`mLaunchRate` |
| 植物行为与特殊交互 | `vendor/de-pvz/Lawn/Plant.cpp`、`Plant.h` | `Plant::Update`、各类 `Do*` / `Fire*` 函数 | 确认攻击、状态、生产、死亡、特殊植物语义 |
| 僵尸定义与生成权重 | `vendor/de-pvz/Lawn/Zombie.cpp` | `gZombieDefs[]` | 提取 value、starting level、pick weight |
| 僵尸行为与护甲 | `vendor/de-pvz/Lawn/Zombie.cpp`、`Zombie.h` | `Zombie::Update`、状态/动画/伤害相关函数 | 确认移动、攻击、护甲、特殊僵尸语义 |
| 棋盘坐标与格子度量 | `vendor/de-pvz/GameConstants.h`、`vendor/de-pvz/Lawn/Board.cpp`、`Board.h` | `LAWN_XMIN`、`MAX_GRID_SIZE_*`、`GridToPixel*` | 转换像素、lane、slot、距离 |
| 卡槽与种植流程 | `vendor/de-pvz/Lawn/SeedPacket.cpp`、`SeedPacket.h`、`Board.cpp` | seed packet 状态、放置检查 | 费用、冷却、可种植判断、鼠标输入流程 |
| 抛射体与命中 | `vendor/de-pvz/Lawn/Projectile.cpp`、`Projectile.h` | projectile update、collision、damage | 速度、轨迹、命中、状态弹 |
| 关卡/模式/小游戏 | `vendor/de-pvz/Lawn/Board.cpp`、`Challenge.cpp`、`Challenge.h` | challenge、game mode、wave 逻辑 | 模式差异、胜负条件、特殊规则 |
| 资源 ID 与表现资源 | `vendor/de-pvz/Resources.cpp`、`Resources.h` | image、reanim、sound、font resource ids | 资源命名、原版素材映射、表现层查证 |
| Reanim/Attachment | `vendor/de-pvz/Sexy.TodLib/Reanimator.*`、`Attachment.*` | reanim track、attachment | 动画结构、挂载点、表现层工具链 |
| PAK 与资源加载 | `vendor/de-pvz/PakLib/` | pak interface | 素材包、解包、资源来源分析 |

如果 `rg --files` 在 `vendor/de-pvz` 下发现不完整，先用 PowerShell `Get-ChildItem -LiteralPath "vendor/de-pvz" -Recurse` 枚举目录，再回到具体文件检索。

| 子弹系统 | `vendor/PVZ-Godot-Dream/scripts/autoload/global/bullet_registry.gd`、`docs/子弹说明文档.md` | 查看子弹注册、创建和说明文档 |

使用时只借鉴 Godot 工程经验。若它的数值、行为或特例与 `de-pvz` 冲突，以 `de-pvz` 为准。

## 五、常用检索命令

```powershell
rg -n "gPlantDefs|SeedType|mSeedCost|mRefreshTime|mLaunchRate" "vendor/de-pvz"
rg -n "gZombieDefs|ZombieType|mZombieValue|mStartingLevel|mPickWeight" "vendor/de-pvz"
rg -n "GridToPixel|LAWN_XMIN|MAX_GRID_SIZE" "vendor/de-pvz"
rg -n "RESOURCE_ID|IMAGE_|REANIM_|SOUND_" "vendor/de-pvz/Resources.cpp" "vendor/de-pvz/Resources.h"
rg -n "class_name|extends|component|manager|registry" "vendor/PVZ-Godot-Dream/scripts"
```

Windows 下路径始终用双引号包裹。需要查看 vendor 大目录结构时，优先：

```powershell
Get-ChildItem -LiteralPath "vendor/de-pvz" -Recurse -File
Get-ChildItem -LiteralPath "vendor/PVZ-Godot-Dream" -Recurse -File
```

## 六、引用到草案或计划时的写法

在 `plans/draft/` 或正式计划里引用参考项目时，建议写清三件事：

1. 参考项目路径和语义锚点，例如 `vendor/de-pvz/Lawn/Plant.cpp:gPlantDefs[]`。
2. 该锚点提供的是“数值/行为规格”还是“Godot 实现参考”。
3. Open PVZ 的目标落点，例如 `CombatMechanic`、`BattleBoardState`、`VisualProfile` 或 validation 场景。

推荐格式：

```markdown
- 原版规格：`vendor/de-pvz/Lawn/Zombie.cpp:gZombieDefs[]`，用于确认生成权重和解锁关卡。
- Godot 参考：`vendor/PVZ-Godot-Dream/scripts/character/`，仅用于观察组件拆分。
- Open PVZ 落点：`data/combat/archetypes/zombies/` + `tools/validation_scenarios.json`。
```
