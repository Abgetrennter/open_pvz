# 扩展系统总体规划-v1

> 这份文档用于把 `Open PVZ` 的扩展系统从“当前已有的最小扩展入口”提升为“平台级核心能力”的正式规划文档。它回答的不是“某个扩展包现在怎么放进仓库”，而是“扩展系统长期应包含什么、允许开放到什么程度、包类型如何划分、主仓需要冻结什么、以及后续应按什么顺序落地”。

---

## 文档定位

这篇文档主要回答：

- 为什么扩展系统现在应该上升为核心架构议题
- 扩展系统长期应该包含哪些类型的包
- 每种包分别负责什么、不负责什么
- 扩展系统允许开放到什么程度
- 哪些能力必须允许扩展，哪些核心不变量必须冻结
- 主仓后续应新增哪些 registry、manifest 字段和验证边界
- 应按什么阶段逐步落地

这篇文档不负责：

- 直接定义最终版扩展包 UI
- 直接定义最终版工坊或分享平台
- 直接实现完整安装器
- 代替具体某一种包格式的全部技术细节
- 承诺所有未来扩展都默认安全、平衡或高性能

## 一句话结论

对 `Open PVZ` 来说，扩展系统不应继续被理解成：

- 一个“后面再补”的附属功能
- 或一个“只能扫外部模板目录”的资源入口

更准确的定位应当是：

> 扩展系统是规则化引擎的平台层，负责承载外部内容、外部素材、外部规则能力和外部集合包，并通过受控注册、兼容检查和验证体系把它们接回统一运行时。

---

## 为什么现在必须开始正式规划

当前仓库已经不再处于“只有最小原型”的阶段，而是同时具备了下面 4 个前提：

1. 运行时主干已经稳定。
2. 模板主链已经稳定。
3. 第一版扩展入口已经落地。
4. 项目目标已经从“验证主干可行”转向“让后续开发逐步切换到内容作者模式”。

这意味着如果现在还不把扩展系统总图定下来，后面会出现 4 个问题：

### 1. 内容包、素材包、规则包会混成一个“万能扩展包”

短期看方便，长期会让：

- 权限边界模糊
- 依赖关系混乱
- manifest 越长越不清晰

### 2. 主仓会越来越难判断什么能开放，什么不能开放

如果没有一开始就明确：

- 哪些字段允许扩展
- 哪些 registry 允许外部注册
- 哪些核心语义不可改

那么后面每新增一个能力都要重新争论一次边界。

### 3. 素材包问题会反向污染内容结构

如果素材加载不被视为扩展系统的一部分，后续很容易变成：

- 内容包直接绑死真实素材路径
- 素材包和内容包无法解耦
- 无法做到同一内容切多个主题

### 4. 集合包和内容作者工作流无法稳定落地

你后面真正想切换的是：

- 开发者模式
  - 改主仓
  - 改运行时
  - 改验证体系

逐步转向：

- 内容作者模式
  - 做内容包
  - 做素材包
  - 做集合包
  - 做主题包

这件事只有在扩展系统先变成平台层之后才成立。

## 当前已有基础与当前不足

### 当前已有基础

当前仓库已经具备：

- `extensions/*` 扫描入口
- `extension.json` 最小 manifest
- `resources / effects` 两类注册位
- 内容资源与扩展效果的最小 smoke test
- 一批扩展 effect 样例和守卫专项

具体可参考：

- [第五阶段-扩展入口最小主链](../../plans/archive/第五阶段/第五阶段-扩展入口最小主链.md)
- [extension_pack_catalog.gd](../../scripts/core/runtime/extension_pack_catalog.gd)

### 当前不足

当前仍然缺少：

- 正式的包类型体系
- 明确的信任分级
- 明确的覆盖策略
- 素材包体系
- 集合包体系
- 本地化包与 UI 主题包预留
- 面向平台层的统一 manifest 规范
- 统一的包级验证模型

换句话说：

> 当前已经有“扩展入口”，但还没有“扩展平台”。

## 扩展系统的长期目标

扩展系统长期应同时服务 4 个目标：

### 1. 外部内容接入

支持外部：

- 单位
- 卡片
- 投射物
- 波次
- 关卡
- 战场模板

### 2. 外部规则能力接入

支持外部：

- 新效果定义
- 新触发定义
- 新检测定义
- 新 battle goal / defeat condition
- 必要时的新执行策略

### 3. 外部素材主题接入

支持外部：

- 图片
- 动画
- 音效
- UI 图标
- 主题资源

并允许：

- 同一内容包搭配不同素材包

### 4. 集合包与分发组织

支持：

- 把多个内容包、素材包、规则包组合成一个主题集合
- 为后续内容作者模式和内容发布模式提供组织层

---

## 包类型体系

当前建议把扩展系统正式拆成下面 6 类包。

---

### A. 内容包 `content_pack`

负责：

- `EntityTemplate`
- `ProjectileTemplate`
- `TriggerBinding`
- `CardDef`
- `WaveDef`
- `BattleScenario`
- `BattlefieldPreset`
- 图鉴元数据

不负责：

- 新运行时代码
- 新素材加载逻辑
- 新执行器主链逻辑

这是未来最常见的包类型，也是内容作者模式下的主力包类型。

---

### B. 素材包 `asset_pack`

负责：

- 逻辑素材 ID 到真实资源的映射
- 贴图
- 动画
- 音效
- UI 图标
- 可选材质与 shader 资源

不负责：

- 玩法内容定义
- 规则执行逻辑

这类包的核心价值是：

- 主仓不内置原版素材
- 同一内容包可切换多个素材主题

---

### C. 规则包 `rule_pack`

负责：

- `EffectDef`
- `TriggerDef`
- `Detection`
- battle goal / defeat condition 定义
- capability / tag 扩展
- 必要时的 effect / trigger `strategy`

不负责：

- 大量单位内容本身
- 单位美术资源

这类包代表：

- “平台能力的扩展”

而不是：

- “单个关卡或单个单位的扩展”

---

### D. 集合包 `collection_pack`

负责：

- 聚合多个 pack
- 定义启用顺序
- 定义推荐素材主题
- 定义关卡组、章节组或主题组入口

不直接负责：

- 具体内容细节
- 具体素材映射

这类包是你后续“内容作者模式”的核心产物之一。

它更像：

- 一个主题内容集合
- 一个玩法模组集合
- 一个战役包

---

### E. 本地化包 `localization_pack`

负责：

- 单位名称
- 描述文本
- 图鉴文本
- UI 文本
- 关卡文本

不负责：

- 玩法逻辑
- 素材本体

建议尽早预留这类包，因为：

- 内容作者模式后期一定会遇到多语言与命名管理问题

---

### F. UI 主题包 `ui_theme_pack`

负责：

- 字体
- 主题颜色
- UI 图标风格
- 按钮样式
- HUD 主题资源

不负责：

- 战斗内容定义

这类包可以后置实现，但最好现在就纳入总体规划，避免未来 UI 被主仓写死。

---

## 当前建议的开放分级

不建议只讨论“支不支持扩展”，而应明确 3 级开放度。

---

### 级别 A：数据扩展

允许：

- 新模板
- 新卡片
- 新波次
- 新关卡
- 新战场模板
- 新素材映射

特点：

- 风险最低
- 最适合广泛开放
- 应作为默认扩展模型

---

### 级别 B：协议扩展

允许：

- 新 `EffectDef`
- 新 `TriggerDef`
- 新 `Detection`
- 新 capability / tag
- 新 goal / defeat 定义

特点：

- 仍偏声明式
- 已经会影响协议集合
- 必须走严格注册和兼容检查

---

### 级别 C：策略扩展

允许：

- 新 effect `strategy`
- 新 trigger `strategy`
- 新局部运行时扩展脚本

特点：

- 能力最强
- 风险最高
- 本质上等于执行外部代码

建议：

- 默认不要对所有外部包开放
- 更适合作为：
  - 本地开发者扩展
  - 受信任扩展
  - 内部扩展

---

## 信任分级

如果后续允许策略扩展，扩展系统必须引入信任模型。

当前建议至少分成下面 3 档：

### 1. `data_safe`

允许：

- 数据资源
- 素材资源
- 本地化资源

不允许：

- 执行自定义脚本

适合：

- 普通内容包
- 普通素材包

### 2. `trusted_script`

允许：

- 受控的策略脚本

要求：

- manifest 显式声明
- registry 显式允许
- 校验器显式记录

适合：

- 规则包
- 开发者级功能包

### 3. `internal_only`

允许：

- 更深层的运行时扩展

适合：

- 主仓内部实验
- 不对外分发的高风险扩展

---

## 哪些能力必须允许扩展

长期看，下面这些能力应被视为“可扩展区域”：

- 单位模板
- 投射物模板
- 触发绑定
- 卡片
- 波次
- 关卡
- 战场模板
- 素材映射
- 图鉴元数据
- 本地化文本
- 效果定义
- 触发定义
- 检测定义
- battle goal / defeat condition 定义

---

## 哪些核心不变量必须冻结

长期看，下面这些能力应被视为“核心稳定区”：

- 事件语义
- `EventData` 核心字段语义
- `RuleContext` 核心字段语义
- 链深保护和最小安全边界
- 核心 registry 加载流程
- 核心验证接入规则
- 主仓保留命名空间
- 主仓协议兼容检查流程

换句话说：

> 可以扩内容和能力，但不能让外部包随意改写主干语义。

---

## 命名空间与覆盖规则

当前最需要明确的是：

### 1. 主仓命名空间

建议保留：

- `core/*`
- 或主仓现有稳定 ID 集合作为保留区

### 2. 扩展包命名空间

建议规则：

- 扩展定义默认应带 `pack_id/*`
- 或至少在 manifest 中声明自身命名空间前缀

### 3. 覆盖模式

后续不建议继续只靠“同名冲突直接拒绝”这一条。

建议显式区分：

- `add`
  - 只新增
- `override`
  - 覆盖某个非核心项
- `patch`
  - 对现有定义做受控修改
- `theme_override`
  - 只覆盖素材映射

其中默认策略建议仍然是：

- 主仓核心内容默认不可被覆盖

---

## registry 体系的建议演进

当前已有：

- `SceneRegistry`
- `EffectRegistry`
- `ExtensionPackCatalog`

后续建议明确拆成：

### 1. `PackRegistry`

负责：

- 加载所有 pack manifest
- 解析依赖
- 决定启用顺序
- 决定 trust level

### 2. `ContentRegistry`

负责：

- 关卡、模板、卡片、波次、战场模板等内容注册

### 3. `RuleRegistry`

负责：

- `EffectDef`
- `TriggerDef`
- `Detection`
- goal / defeat 定义

### 4. `AssetRegistry`

负责：

- 逻辑素材 ID -> 实际资源映射
- 回退链
- 主题切换

### 5. `LocalizationRegistry`

负责：

- 文本资源注册
- 语言切换

---

## 素材包应如何进入扩展系统

素材包不应被当成“普通资源目录扫描的附属物”，而应视为正式包类型。

### 核心原则

内容资源不应直接写死具体素材文件路径。

应当写：

- `visual_profile_id`
- `icon_id`
- `animation_set_id`
- `sfx_profile_id`
- `ui_skin_id`

再由 `AssetRegistry` 解析成真实资源。

### 回退链建议

建议固定为：

1. 当前激活素材包
2. 依赖素材包
3. 默认占位素材包

### 必须有默认占位包

主仓虽然不应内置原版素材，但必须内置：

- 合法占位 sprite
- 合法占位 icon
- 合法占位音效
- 最小可运行动画资源

否则后续内容作者工作流会被素材问题反向卡死。

---

## 集合包的作用

集合包不是“又一种内容包”，而是：

> 用来组织多个包如何一起构成一个主题化体验。

一个集合包可以定义：

- 依赖哪些内容包
- 依赖哪些素材包
- 依赖哪些规则包
- 推荐使用哪个 UI 主题包
- 暴露哪些关卡组 / 章节组入口

这会直接对应你后面想做的：

- 一个玩法集合包
- 一个战役集合包
- 一个主题内容包

---

## manifest 最小建议字段

当前扩展系统后续应至少演进到下面这组 manifest 字段：

```text
pack_id
pack_type
version
display_name
description
author
license
namespace
dependencies
optional_dependencies
compatible_core_version
register
trust_level
capabilities
load_priority
entry_points
```

其中不同 `pack_type` 的 `entry_points` 可以不同：

- `content_pack`
  - 内容目录入口
- `asset_pack`
  - asset index
- `rule_pack`
  - rule defs / strategy scripts
- `collection_pack`
  - pack composition / level set entry

## 自动化与验证边界

扩展系统要成为平台层，必须有包级验证。

建议至少覆盖：

### 1. manifest 校验

- 字段完整性
- 类型正确性
- `pack_type` 合法性
- `trust_level` 合法性

### 2. 依赖校验

- 必需依赖是否存在
- 版本是否兼容
- 是否循环依赖

### 3. 注册校验

- ID 冲突
- capability/tag 冲突
- 不允许覆盖核心区

### 4. 资源校验

- 素材映射缺失
- 逻辑素材 ID 未解析
- 本地化 key 缺失

### 5. 包级 smoke test

- 内容包是否最小可玩
- 素材包是否至少能解析关键素材
- 集合包是否能装配成功

---

## 当前建议的分阶段落地顺序

### 第一步：统一总图

当前就应该完成：

- 包类型体系
- 信任分级
- manifest 总结构
- 覆盖规则

### 第二步：先做素材包系统

优先原因：

- 版权问题已经明确存在
- 素材包会直接影响内容包设计方式

### 第三步：升级当前内容扩展系统

从当前：

- `resources / effects`

演进到：

- `content / rule / asset / collection`

### 第四步：建立包级验证和回归

这一步要早于扩展 UI。

### 第五步：最后再做包管理 UI

因为：

- UI 只是平台界面的表现层
- 不是平台规则本身

---

## 当前文档层面的正式结论

如果把当前这轮规划收成一句话，最合理的正式结论是：

> `Open PVZ` 的扩展系统应被正式视为引擎平台层，而不是附属功能；它长期应至少包含内容包、素材包、规则包、集合包、本地化包和 UI 主题包，并通过包类型体系、信任分级、命名空间、覆盖规则、受控 registry 和包级验证模型来维持开放性与稳定性的平衡。
