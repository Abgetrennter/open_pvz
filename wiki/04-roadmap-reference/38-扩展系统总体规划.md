# 扩展系统总体规划

> 本文是 `Open PVZ` 扩展系统的**单页总纲**，只回答"当前已经落地什么、下一步应该做什么、什么仍是草案"。详细设计草案统一归入 [`plans/draft/extension-system/`](../../plans/draft/extension-system/README.md)，在开始实装前不作为仓库协议。

---

## 一、当前已落地的扩展能力

以仓库现状（2026-04-20）为准：

- **扫描入口**：`extensions/` 目录在启动时由 [extension_pack_catalog.gd](../../scripts/core/runtime/extension_pack_catalog.gd) 扫描注册。
- **最小 manifest**：每个包提供 `extension.json`，字段由现有代码消费，未形式化规范。
- **注册位**：当前允许扩展的两类：
  - 资源类（模板、触发绑定、卡片、波次、关卡等 `.tres`）
  - 效果类（通过现有 `EffectRegistry` 注册）
- **样例包**：
  - [`extensions/minimal_chaos_pack`](../../extensions/minimal_chaos_pack) —— 最小内容包
  - [`extensions/phase5_chaos_pack`](../../extensions/phase5_chaos_pack) —— 错误技样例
  - [`extensions/phase5_guardrail_pack`](../../extensions/phase5_guardrail_pack) —— 负向样例，用于守卫回归
- **回归覆盖**：验证矩阵中 `extension` 与 `guardrail` 分层共 15 条，持续入批量回归。

一句话：**当前已有"扩展入口"，还没有"扩展平台"**。

---

## 二、扩展系统的长期定位（不变量）

无论后续细节如何演进，以下定位不应动摇：

- 扩展系统是规则化引擎的**平台层**，不是"后补功能"。
- 核心协议（事件名、冻结触发器 / 效果、模板字段）**由主仓独占**，扩展包不得重定义。
- 扩展包必须走与主仓内容**同一套运行时与验证**，不允许旁路执行路径。
- 扩展包的信任级别、覆盖策略、依赖图**可以后做**，但底线是**受控注册 + 可关闭**。

## 三、下一步应做什么（优先级降序）

这是当前扩展系统的行动路线，粒度到"能否进入一个阶段任务清单"。

### 1. 扩展包加载失败的可观察性

- 当前失败路径依赖日志观察，缺少统一报告结构。
- 目标：在 `extension_pack_catalog` 层产出**分类失败清单**（缺字段 / 类型错 / 依赖缺失），供验证场景断言。

### 2. 扩展包白名单与禁用开关

- 目前所有 `extensions/` 子目录一视同仁加载。
- 目标：支持通过启动参数或配置资源禁用指定包，便于 A/B 验证与回归。

### 3. manifest 字段最小形式化

- 现状：`extension.json` 字段没有显式协议。
- 目标：冻结 `id / version / description / content_roots / effect_scripts` 五字段，其他留空。
- **非目标**：信任分级、依赖声明、主题资源、本地化—— 这些全部仍在 draft。

### 4. 扩展包与主仓内容的命名冲突策略

- 当前策略：未冲突过 = 未测试。
- 目标：定义"同 id 冲突时主仓优先，扩展包发出守卫级错误"并加验证场景。

## 五、判断题：新需求应该进扩展系统吗

当新需求落到扩展相关问题时，先回答：

1. **当前主仓是否已支持这类内容？** 没支持：先进主仓，再谈扩展。
2. **问题属于运行时协议还是内容组织？** 协议层变化不通过扩展系统修改。
3. **是否有对应验证场景？** 没有：先加验证，再讨论扩展。

三个问题任一为"否"，都应先回到主干，而不是打开扩展系统草案。
