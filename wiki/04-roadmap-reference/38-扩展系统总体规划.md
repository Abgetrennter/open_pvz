## 一、当前已落地的扩展能力

以仓库现状为准：

- **扫描入口**：`extensions/` 目录在启动时由 [extension_pack_catalog.gd](../../scripts/core/runtime/extension_pack_catalog.gd) 扫描注册。
- **通用扩展插槽**：主仓已形成 `Slot / Contributor / Registry / Runtime dispatch / Guardrail` 模型，扩展包通过资源化贡献项注册能力。
- **Mechanic 层注册**：扩展包通过 `MechanicCompilerDef` 在已有 family 下注册新 type，并由 `MechanicCompilerRegistry` 分发。
- **ProjectileMovement 插槽**：`ProjectileMovementDef + ProjectileMovementRegistry` 已落地，扩展包可以新增 `my_pack.zigzag` 这类 movement，而不需要修改 `ProjectileRoot` 分支。
- **注册中心全景**：
  - `MechanicFamilyRegistry`：10 个冻结 family（扩展包不能新增 family）
  - `MechanicTypeRegistry`：family → type_id 注册
  - `MechanicCompilerRegistry`：per-type compiler callable 分发
  - `ProjectileMovementRegistry`：projectile movement def 注册与组件创建
  - `ControllerRegistry`：Controller 策略注册
  - `EffectRegistry`：效果注册（支持扫描 `data/combat/effects` 扩展效果）
  - `DetectionRegistry`：目标发现策略注册
  - `TriggerRegistry`：触发器定义与策略注册
- **manifest 能力声明**：`extension.json` 支持 `register`、`trust_level`、`capabilities`，其中运行时代码插槽需要 `trusted_runtime`。
- **命名空间规则**：`core.*` 由主仓独占，扩展包使用 `extension_namespace.*` 前缀注册；重复 id 默认拒绝，不做 override。
- **样例包**：
  - [`extensions/minimal_chaos_pack`](../../extensions/minimal_chaos_pack) —— 最小内容包
  - [`extensions/phase5_chaos_pack`](../../extensions/phase5_chaos_pack) —— 错误技样例
  - [`extensions/phase5_guardrail_pack`](../../extensions/phase5_guardrail_pack) —— 负向样例，用于守卫回归
  - [`extensions/my_pack`](../../extensions/my_pack) —— `ProjectileMovement + MechanicCompiler` 插槽正向样例
  - [`extensions/slot_guardrail_pack`](../../extensions/slot_guardrail_pack) —— 通用插槽负向样例
- **回归覆盖**：验证矩阵中 `extension` 与 `guardrail` 分层持续入批量回归。

一句话：**当前已有受控通用插槽机制，还没有完整扩展平台和安装器体验**。

## 二、包类型与信任级别

ADR-005 定义的包类型体系：

### 包类型

| 类型 | 说明 | 典型内容 |
|------|------|---------|
| `content_pack` | 纯数据内容包 | archetype、投射物、卡片、波次、关卡 |
| `rule_pack` | 规则扩展包 | 新 Mechanic type、新 Effect 策略 |
| `runtime_pack` | 包含运行时代码的包 | 新 Controller、新 Movement 策略 |
| `asset_pack` | 纯素材包 | 图片、动画、音效 |
| `collection_pack` | 其他包的集合 | 整合包 |

### 信任级别

| 级别 | 说明 |
|------|------|
| `data_only` | 只能提供 .tres 资源，不能注册新执行代码 |
| `rule_extended` | 可以注册受控 Effect 策略 |
| `trusted_runtime` | 可以注册 projectile movement、mechanic compiler、controller 等运行时代码 |

### 权限边界

- 扩展包默认只能新增 **Mechanic type**（在已有 family 下）和主仓开放的 slot contributor
- 新增 **family** 需要主仓 ADR 审批
- 扩展包 target **Archetype / Mechanic 层**，不 target 旧 TriggerBinding
- 运行时代码 contributor 必须声明 `trust_level = trusted_runtime`

## 三、双轨迁移状态

ADR-005 定义了受约束双轨期，4 个退出信号已全部满足：

- [x] 首批 archetype 验证通过
- [x] Compiler / RuntimeAssembler 稳定
- [x] Archetype 路径覆盖典型内容
- [x] 新增 + parity 验证进入回归

**结论**：双轨期已达成目的。旧模型已归档为历史记录，新内容统一使用 Archetype + Mechanic[]。

## 四、扩展系统的长期定位（不变量）

无论后续细节如何演进，以下定位不应动摇：

- 扩展系统是规则化引擎的**平台层**，不是"后补功能"。
- 核心协议（事件名、冻结 family、Mechanic 三层边界）**由主仓独占**，扩展包不得重定义。
- 扩展包必须走与主仓内容**同一套运行时与验证**，不允许旁路执行路径。
- 扩展包的信任级别、覆盖策略、依赖图**可以后做**，但底线是**受控注册 + 可关闭**。

## 五、下一步应做什么（优先级降序）

### 1. 扩展包启停与包白名单

- 目标：支持通过启动参数或配置资源禁用指定包。

### 2. Registry 可观察性进一步增强

- 目标：注册失败时产出结构化失败清单，并在 debug overlay 中可查询。

### 3. 跨包依赖与版本范围

- 目标：在不开放 override 的前提下，表达包依赖、版本范围和冲突诊断。

## 七、判断题：新需求应该进扩展系统吗

当新需求落到扩展相关问题时，先回答：

1. **当前主仓是否已支持这类内容？** 没支持：先进主仓，再谈扩展。
2. **问题属于运行时协议还是内容组织？** 协议层变化不通过扩展系统修改。
3. **是否有对应验证场景？** 没有：先加验证，再讨论扩展。

三个问题任一为"否"，都应先回到主干，而不是打开扩展系统草案。
