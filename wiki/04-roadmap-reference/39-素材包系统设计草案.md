# 素材包系统设计草案

> 这份文档用于承接 [扩展系统总体规划-v1](38-扩展系统总体规划-v1.md) 中“素材包系统应优先落地”的结论。它回答的不是“当前仓库里放几张贴图”，而是“在主仓不内置原版素材、后续希望以内容作者模式制作集合包的前提下，素材包系统应该如何设计，才能既满足版权隔离，又满足内容扩展、主题切换和后续产品化入口的需要”。

---

## 文档定位

这篇文档主要回答：

- 为什么素材包必须被视为正式扩展类型
- 素材包和内容包、规则包、集合包是什么关系
- 内容资源里应该如何引用素材
- 逻辑素材 ID 应该如何设计
- `AssetRegistry` 应该负责什么
- 素材解析和回退链应如何组织
- 素材包 manifest 至少应包含什么
- 当前应按什么顺序逐步落地

这篇文档不负责：

- 直接实现素材导入器
- 直接实现素材包 UI
- 定义最终版动画工作流
- 承诺所有素材包都能自动兼容所有未来内容包

## 一句话结论

素材包系统不应被理解成：

- 一组额外图片资源
- 或一个“后面再补的素材加载功能”

更准确地说，它应被理解成：

> 扩展系统中的正式包类型，负责把逻辑素材 ID 解析成真实视觉、音频和主题资源，并通过统一回退链把“无版权主仓 + 可切换主题 + 内容包解耦”这三件事同时成立起来。

---

## 为什么素材包现在必须进入主设计主线

当前如果不把素材包系统正式设计出来，后面会出现 4 个问题：

### 1. 内容资源会被真实路径绑死

一旦内容资源直接引用：

- `res://some_pack/sprites/pea.png`

那么后果是：

- 内容包和素材包无法解耦
- 同一内容无法自由切主题
- 后续做集合包时会非常别扭

### 2. 版权边界无法真正收口

如果主仓不内置原版素材，但内容和 UI 又默认写死原版素材路径，那实际上仍然在架构上默认依赖了某套具体素材。

这会让：

- 主仓无法独立工作
- 占位工作流缺失
- 版权隔离只停留在口头层面

### 3. 后续 UI 和图鉴会没有稳定素材入口

后面无论你做：

- 卡片图标
- 单位预览
- 图鉴
- 关卡预览
- 主题切换

都必须先回答：

- 这些界面到底通过什么 ID 找素材

如果现在不定，以后这些入口会各自长出一套私有路径规则。

### 4. 内容作者模式无法成立

你后面真正想切换的是：

- 不再一直改主仓
- 而是像作者一样做内容包、素材包、集合包

这要求：

- 内容作者可以不碰主仓逻辑
- 素材作者可以不碰内容逻辑

这两个目标都依赖素材包系统先被正式设计出来。

## 素材包在扩展系统中的位置

在当前建议的包类型体系中：

- `content_pack`
  - 定义玩法内容
- `rule_pack`
  - 定义规则能力
- `asset_pack`
  - 提供真实素材
- `collection_pack`
  - 组织多个 pack

素材包的角色非常明确：

> 它不定义玩法内容，也不定义规则能力，它只负责为逻辑内容提供可替换、可覆盖、可回退的表现资源。

---

## 内容包与素材包的边界

### 内容包负责什么

内容包负责：

- 单位是什么
- 卡片是什么
- 波次怎么组织
- 关卡怎么玩
- 战场如何约束

也就是：

- 逻辑定义
- 数值定义
- 组合结构

### 素材包负责什么

素材包负责：

- 单位长什么样
- 动画如何播放
- 卡片图标是什么
- 命中和爆炸有什么音效
- HUD 和 UI 主题长什么样

也就是：

- 视觉资源
- 音频资源
- 主题资源

### 关键原则

内容资源和素材资源之间，不应通过“真实文件路径”耦合，而应通过：

> 逻辑素材 ID

进行解耦。

---

## 逻辑素材 ID 体系

这是素材包系统的核心。

当前建议：

- 内容资源里不直接写具体图片路径
- 改为写逻辑素材槽位 ID

例如：

- `entity_visual_id`
- `card_icon_id`
- `animation_set_id`
- `sfx_profile_id`
- `ui_skin_id`

这些字段的值再写成逻辑 ID，例如：

- `entity.plant.frost_pea.visual`
- `card.plant.frost_pea.icon`
- `animation.zombie.basic.walk`
- `sfx.projectile.pea.hit`
- `ui.theme.classic.panel`

---

## 逻辑素材 ID 的命名建议

建议遵守下面几个原则：

### 1. 先按语义域分前缀

例如：

- `entity.*`
- `card.*`
- `projectile.*`
- `animation.*`
- `sfx.*`
- `vfx.*`
- `ui.*`
- `bgm.*`

### 2. 中段按内容对象组织

例如：

- `entity.plant.basic_shooter.visual`
- `entity.zombie.bone_thrower.visual`
- `card.plant.spore_summoner.icon`

### 3. 尾段按用途组织

例如：

- `.visual`
- `.icon`
- `.idle`
- `.attack`
- `.walk`
- `.death`
- `.spawn`
- `.hit`

### 4. 不把 pack 名写进逻辑素材 ID

因为逻辑素材 ID 应描述：

- “这是什么素材槽位”

而不是：

- “它现在从哪个包里来”

否则主题切换时会再次耦合。

---

## 推荐的素材 ID 示例

### 单位可视

- `entity.plant.frost_pea.visual`
- `entity.plant.spore_summoner.visual`
- `entity.zombie.bone_thrower.visual`

### 卡片图标

- `card.plant.frost_pea.icon`
- `card.plant.spore_summoner.icon`

### 动画组

- `animation.plant.frost_pea.idle`
- `animation.plant.frost_pea.attack`
- `animation.zombie.tar_spitter.walk`
- `animation.zombie.tar_spitter.attack`

### 音效

- `sfx.projectile.pea.hit`
- `sfx.projectile.bone.launch`
- `sfx.entity.zombie.tar_spitter.attack`
- `sfx.ui.card.select`

### UI 主题

- `ui.panel.default`
- `ui.button.primary`
- `ui.icon.sun`

---

## 内容资源应该补哪些字段

当前建议在后续内容资源中逐步补下面这些字段：

### `EntityTemplate`

建议补：

- `entity_visual_id`
- `animation_set_id`
- `sfx_profile_id`
- `card_icon_id`

### `ProjectileTemplate`

建议补：

- `projectile_visual_id`
- `projectile_animation_set_id`
- `projectile_sfx_profile_id`

### `BattlefieldPreset` / 关卡

建议补：

- `background_visual_id`
- `background_music_id`
- `battle_theme_id`

### UI / 图鉴 / 关卡入口

建议补：

- `preview_image_id`
- `portrait_id`
- `ui_skin_id`

这些字段现在不一定要一次性都实现，但命名方向应该尽快定下。

## `AssetRegistry` 的职责

当前建议后续新增一个正式的 `AssetRegistry`，至少负责下面 5 类能力。

### 1. 加载素材包 manifest

负责：

- 识别启用的 `asset_pack`
- 解析其依赖和优先级

### 2. 维护逻辑素材 ID 到资源的映射

例如：

- `entity.plant.frost_pea.visual -> res://asset_packs/.../frost_pea_idle.tscn`

### 3. 提供统一解析入口

例如：

- `resolve_texture(asset_id)`
- `resolve_audio(asset_id)`
- `resolve_scene(asset_id)`
- `resolve_theme_resource(asset_id)`

### 4. 提供回退链

当当前素材包没有提供某个逻辑素材 ID 时，自动回退到下一层。

### 5. 提供缺失素材日志

方便：

- 内容作者查缺
- 素材作者补包
- UI 提示“当前主题缺少某素材”

---

## 素材解析回退链

当前建议统一采用下面回退顺序：

1. 当前激活素材包
2. 当前激活素材包的依赖素材包
3. 默认占位素材包

其中：

- 不建议直接回退到主仓散落资源
- 也不建议让内容资源自己处理 fallback

因为 fallback 本身就是素材平台层的职责。

---

## 默认占位素材包

主仓虽然不应内置原版素材，但必须内置一个：

> 合法、可分发、可运行的默认占位素材包

它至少应提供：

### 视觉

- 基础单位占位图
- 基础投射物占位图
- 基础卡片图标
- 基础 UI 图标

### 动画

- 最小 idle / attack / hit / death 占位

### 音效

- 最小点击音
- 最小发射音
- 最小命中音

### UI 主题

- 最小默认面板
- 最小按钮样式
- 最小字体配置

这样做的意义是：

- 没有任何外部素材包时，项目仍可跑
- 验证和内容工作流不会被版权素材反向卡住

---

## 素材包 manifest 建议字段

当前建议 `asset_pack` 至少包含下面字段：

```text
pack_id
pack_type = asset_pack
version
display_name
description
author
license
namespace
compatible_core_version
dependencies
optional_dependencies
load_priority
asset_index
theme_tags
```

其中：

- `asset_index`
  - 核心字段
- `theme_tags`
  - 用于标识：
    - `classic`
    - `placeholder`
    - `pixel`
    - `sci_fi`
    - `minimal`

---

## `asset_index` 的建议结构

当前建议采用：

- 逻辑素材 ID -> 实际资源路径

例如：

```json
{
  "entity.plant.frost_pea.visual": "sprites/entities/frost_pea_idle.tres",
  "card.plant.frost_pea.icon": "sprites/cards/frost_pea_icon.png",
  "animation.zombie.bone_thrower.attack": "animations/zombie_bone_thrower_attack.tres",
  "sfx.projectile.bone.launch": "audio/sfx/bone_launch.wav",
  "ui.icon.sun": "ui/icons/sun.png"
}
```

后续如有需要，再考虑扩展成：

- 一个逻辑素材 ID 对应多资源变体
- 或带 metadata 的结构

但第一版不建议过度复杂。

---

## 当前对素材包的覆盖规则建议

素材包的覆盖需求和内容包不同。

建议素材包主要使用：

- `theme_override`

也就是：

- 允许覆盖同一个逻辑素材 ID 的提供者
- 但不改变玩法内容

不建议素材包承担：

- 内容 patch
- 规则 patch

因为素材包的覆盖目标应只限于表现层。

---

## 素材包与集合包的关系

集合包应当可以指定：

- 必需内容包
- 必需规则包
- 推荐素材包
- 默认主题包

这样后面一个集合包可以表达：

- “这是一套玩法内容”
- “推荐配这套视觉素材”

而不是把内容和素材绑死。

这正是后续“内容作者模式”的关键体验之一。

---

## 当前最小落地顺序

我建议素材包系统按下面顺序推进。

### 第一步：先写规范，不先写 UI

先定：

- 包结构
- 逻辑素材 ID
- `AssetRegistry` 职责
- 回退链

### 第二步：做默认占位素材包

这是整个系统真正可启动的前提。

### 第三步：给核心内容资源补素材槽位字段

优先补：

- `EntityTemplate`
- `ProjectileTemplate`
- `CardDef`

### 第四步：实现 `AssetRegistry` 最小版

只要求：

- 读 manifest
- 建 index
- 解逻辑 ID
- 做 fallback

### 第五步：再做素材包切换 UI

因为：

- 主题切换界面不是素材系统本身
- 只是素材系统的上层产品入口

---

## 当前正式建议

如果把这份草案收成一句话，当前最合理的正式建议是：

> 主仓应尽快把素材包系统提升为正式扩展类型，并以“逻辑素材 ID + AssetRegistry + 默认占位包 + 统一回退链”为核心方案，让内容包、素材包和集合包在版权隔离前提下真正解耦。
