# 参考实现迁移策略

> 本文回答的问题不是“参考项目好不好”，而是：在 `Open PVZ` 当前的引擎路线下，`PVZ-Godot-Dream` 已有的哪些内容值得迁移，应该怎么迁，哪些不要碰。

---

## 结论先说

可以利用参考项目，但方式应当是：

- 迁移「通用 Godot 基础设施」
- 借鉴「成熟的战斗节点拆法」
- 参考「资源组织方式」
- 不迁移「强绑定原版 PVZ 规则的业务层」

一句话总结：

> 把它当作一套已经验证过的 Godot PvZ 实战基建库来拆解，而不是当作我们项目的直接母体。

---

## 为什么不能整块迁

`PVZ-Godot-Dream` 的核心目标是“高质量复刻原版 PVZ”，不是“开放式、可组合的规则引擎”。

它的很多系统默认前提是：

- 植物、僵尸、子弹都有明确的原版类别
- 行为入口主要是具体角色脚本和具体攻击组件
- 规则变化主要体现在“对象特化”，不是“通用触发器 + 效果树”
- 地图、格子、地形、卡槽、关卡流程都是原版 PvZ 语义优先

所以如果直接 fork 业务层，我们会立刻继承一堆不适合当前目标的假设：

- 植物能力被写死在专用攻击组件里
- 子弹/角色类型枚举过重
- 关卡管理器过大
- 格子与原版地形规则耦合太深

---

## 可直接复用

这些内容和我们当前方向不冲突，可以直接吸收实现思路，必要时甚至可以近似重写。

### 1. Autoload 组织方式

参考文件：

- `vendor/PVZ-Godot-Dream/scripts/autoload/event_bus.gd`
- `vendor/PVZ-Godot-Dream/scripts/autoload/scene_registry.gd`

可迁移点：

- 用 `Autoload` 承担跨场景公共服务
- `SceneRegistry` 集中缓存散落场景引用
- `EventBus` 统一订阅、退订、历史记录和调试

对我们的改进建议：

- 保留现有 `autoload/` 结构
- 在当前 `EventBus` 上补 `priority`、`once`、`filter`
- 在 `SceneRegistry` 中加入演示实体、特效、UI overlay 的统一场景注册

不建议直接照搬的点：

- 参考项目的 `EventBus` 是“动态信号名 + Array payload”模型
- 我们当前更适合保留 `EventData` 结构化事件体

结论：

- 迁“能力”
- 不迁“接口形式”

### 2. 子弹移动组件拆法

参考文件：

- `vendor/PVZ-Godot-Dream/scripts/bullet/component/movement/bullet_movement_base.gd`
- `vendor/PVZ-Godot-Dream/scripts/bullet/component/movement/bullet_movement_linear.gd`
- `vendor/PVZ-Godot-Dream/docs/子弹说明文档.md`

可迁移点：

- 子弹根节点统一驱动移动组件
- 移动组件只封装移动算法，不自己抢 `_physics_process`
- 线性、抛物线、追踪是同一层抽象的不同实现

这和我们当前路线高度一致。

对我们的改进建议：

- 把当前 `MovementComponent` 继续拆成：
  - `projectile_movement_linear.gd`
  - `projectile_movement_parabola.gd`
  - `projectile_movement_track.gd`
- 让 `ProjectileRoot` 只负责：
  - 生命周期
  - 命中
  - on-hit 效果触发
- 让移动组件只负责轨迹更新

结论：

- 这是当前最值得优先迁移的轮子

### 3. 角色根节点 + 组件节点结构

参考文件：

- `vendor/PVZ-Godot-Dream/scripts/character/character_000_base.gd`
- `vendor/PVZ-Godot-Dream/scripts/character/components/...`

可迁移点：

- 角色根节点负责统一身份、方向、死亡、速度因子、公共信号
- 攻击、动画、血量、受击、检测等能力拆为组件节点
- 子类只覆盖少量特有逻辑

对我们的改进建议：

- 保持当前 `BaseEntity -> PlantRoot/ZombieRoot/ProjectileRoot`
- 继续补齐可插拔组件，而不是把逻辑写回根节点：
  - `AttackIntentComponent`
  - `TargetingComponent`
  - `DeathFeedbackComponent`
  - `LaneAgentComponent`

注意：

- 不要照搬参考项目那种“大型通用角色基类 + 大量原版状态字段”
- 我们只迁“节点组织法”，不迁“原版角色语义集合”

### 3.5 动画驱动层与离线资源迁移

参考文件：

- `vendor/PVZ-Godot-Dream/scripts/character/components/anim_component/component_anim.gd`
- `vendor/PVZ-Godot-Dream/scripts/character/components/anim_component/component_anim_norm.gd`
- `vendor/PVZ-Godot-Dream/scripts/character/components/anim_component/component_anim_player.gd`
- `vendor/PVZ-Godot-Dream/addons/R2Ga_PVZ/PVZ_reanim2godot_animation_x86_x64.exe`

可迁移点：

- 先抽象统一动画驱动层，再决定底层是 `AnimationPlayer`、`AnimationTree` 还是空实现
- 把“动画播放速度”“速度倍率修正”“动画结束通知”统一从角色逻辑里抽出来
- 把“原版 reanim 资源转换”视为离线工作流，而不是运行时前置能力

对我们的改进建议：

- 在视觉层中保留统一 `AnimDriver` 抽象
- 第一轮只实现：
  - `NullAnimDriver`
  - `AnimationPlayerDriver`
  - `SpriteFramesDriver`
- `AnimationTreeDriver` 和原版 reanim 转换链都放到后续阶段
- 运行时只消费 Godot 原生可播放资源；如果未来确实需要原版动画，再单独接离线转换工具链

不要直接迁移的点：

- 不要把参考项目的具体角色动画业务脚本整块搬过来
- 不要让“能否导入原版 reanim”变成当前视觉层建设的前置条件

结论：

- 迁“动画驱动分层”和“离线转换分离”
- 不迁“原版角色动画业务层”

## 适配后可借

这些内容不能直接搬，但经过抽象后很有价值。

### 4. BulletRegistry / SceneRegistry 风格的注册表

参考文件：

- `vendor/PVZ-Godot-Dream/scripts/autoload/global/bullet_registry.gd`

参考项目的做法是：

- 用枚举 + `PackedScene` 字典注册子弹场景

这套形式不适合我们直接沿用，因为我们不想把未来扩展能力锁死在编译期枚举里。

但它可以启发我们做两层注册：

- 引擎层：
  - `EffectRegistry`
  - `TriggerRegistry`
  - `MovementStrategyRegistry`
- 内容层：
  - `EntityTemplateRegistry`
  - `ProjectileTemplateRegistry`
  - `VisualPresetRegistry`

建议：

- 运行时策略仍用 `StringName` 注册
- 场景/模板引用统一交给注册表，不在 gameplay 脚本里散落 preload

### 5. Resource 承载规则和内容约束

参考文件：

- `vendor/PVZ-Godot-Dream/scripts/resources/plant_condition/condition_000_plant_base.gd`
- `vendor/PVZ-Godot-Dream/scripts/resources/level/level_data.gd`

可借的不是具体字段，而是思路：

- 用 `Resource` 持有可编辑、可存盘的规则数据
- 让关卡/实体/种植条件先成为编辑器可见数据，而不是纯代码常量

对我们的适配建议：

- 第一阶段先做 3 类 Resource：
  - `EntityTemplateResource`
  - `TriggerBindingResource`
  - `ProjectileTemplateResource`
- 第二阶段再做：
  - `LaneRuleResource`
  - `SpawnWaveResource`
  - `ConditionRuleResource`

不要直接迁移的点：

- 参考项目的 `ResourcePlantCondition` 强绑定原版格子种植规则
- 我们需要把它升级成更通用的“放置约束 / 目标约束 / 生成约束”

### 6. PlantCell / lane 数据结构

参考文件：

- `vendor/PVZ-Godot-Dream/scripts/main_game_item/plant_cell.gd`

这个文件不能直接迁，原因很明确：

- 原版 PVZ 的格子结构过重
- 里面混了地形、紫卡前置、墓碑、花盆、梯子、存档、鼠标交互、铲子选择等大量规则

但它提供了两个值得保留的抽象：

- lane 是核心战斗语义，不只是位置
- 放置约束应当有明确的数据层承载

对我们的改进建议：

- 现在已引入 `lane_id`，这是正确方向
- 后续不要急着做完整 `PlantCell`
- 先做一个更薄的 `LaneSlot` / `BoardSlot`：
  - `lane_id`
  - `slot_index`
  - `placement_tags`
  - `occupants`

这会比直接迁移 `PlantCell` 更适合引擎化目标

### 7. 原版交互语义与失败原因枚举

参考文件：

- `vendor/de-pvz/ConstEnums.h`
- `vendor/de-pvz/Lawn/CursorObject.h`
- `vendor/de-pvz/Lawn/CursorObject.cpp`
- `vendor/de-pvz/Lawn/Board.cpp`

可借的不是原版 `Board` 的实现形态，而是它把交互语义明确建模成稳定枚举：

- `CursorType`
- `PlantingReason`
- `GridSquareType`
- `BackgroundType`

这带来的直接好处是：

- “当前手里拿着什么”是正式状态，而不是临时变量
- “为什么不能种”是正式语义，而不是零散 if
- 预览、提示、音效和放置校验都能围绕同一套 reason 工作

对我们的改进建议：

- 让 `InputRouter` 或等价场景级输入层显式持有交互模式 / 光标语义
- 继续把 `placement.rejected` 的原因收敛成稳定 reason 集
- 让 `BattleBoardState` 的失败原因成为 UI 提示和输入反馈的正式输入，而不是只做日志

不要直接迁移的点：

- 不要把这些语义重新塞回 `BattleManager`
- 不要照搬原版用一个 `Board` 同时管理输入、提示、放置和光标绘制

### 8. 原版 Foley / Music 的语义层

参考文件：

- `vendor/de-pvz/Sexy.TodLib/TodFoley.h`
- `vendor/de-pvz/Lawn/System/Music.h`
- `vendor/de-pvz/Lawn/System/Music.cpp`

可借点：

- `FoleyType` / `FoleyFlags` 说明音效不只是“播什么”，还包括：
  - 是否循环
  - 是否互斥
  - 暂停时是否静音
  - 是否跟随音乐音量
  - 是否避免连续重复
- `MusicTune` 说明 BGM 应按战场语义和模式切换，而不是只按 “running / victory / defeat” 三态
- 主旋律 / 鼓点 / 镲片的分层控制，说明音乐可以按节奏强度做轻量动态变化

对我们的改进建议：

- 给 `AudioCueDef` 预留与 `FoleyFlags` 等价的语义位，而不只保留音量和 pitch
- BGM 未来可按 `battlefield_preset`、关卡模式和阶段语义选曲
- 继续坚持事件驱动音频，不照搬原版到处直接 `PlayFoley()` 的调用方式

### 9. 原版资源清单与分组加载

参考文件：

- `vendor/de-pvz/SexyAppFramework/properties/resources.xml`
- `vendor/de-pvz/Resources.h`

原版值得借的不是具体资源格式，而是：

- UI、背景、声音、reanim、商店、图鉴等资源都有明确分组
- 延迟加载分组 (`DelayLoad_*`) 是正式概念
- 资源 ID、资源域和加载时机之间是有组织关系的

对我们的改进建议：

- 以后 `AssetRegistry` 应按资源域分组，而不是只做“路径查找器”
- 正式区分：
  - 战斗 HUD 资源
  - 战场背景资源
  - 动画资源
  - 音频 cue 资源
  - 外围页面资源
- 为未来正式关卡、图鉴、商店等外围资源预留延迟加载分组概念

## 明确不要迁

### 1. MainGameManager 大一统流程

参考文件：

- `vendor/PVZ-Godot-Dream/scripts/manager/main_game_manager.gd`

这个管理器承载了太多原版流程：

- 选卡
- 戴夫对话
- 相机
- 多轮关卡
- BGM
- 存档
- 游戏胜负
- 多种小游戏模式

对我们来说，这是最典型的“不值得搬”的层。

建议：

- 保留我们的轻量 `BattleManager`
- 以后拆成小 manager，而不是长成单个 Godot 场景总控巨石

### 2. 原版业务枚举和角色 ID 体系

例如：

- `BulletRegistry.BulletType`
- `CharacterRegistry.PlantType`
- `CharacterRegistry.ZombieType`

这些对复刻项目很合理，但对开放引擎不合适。

我们未来更需要：

- 模板 ID
- 标签
- 能力组合
- Resource / 数据包加载

而不是把一切先编码成大枚举。

### 3. 原版专用攻击组件业务层

参考文件：

- `vendor/PVZ-Godot-Dream/scripts/character/components/attack_behavior_component/component_attack_bullet_base.gd`
- `vendor/PVZ-Godot-Dream/scripts/character/components/attack_behavior_component/component_attack_bullet_three_pea.gd`

这些文件很优秀，但它们的问题是：

- 攻击逻辑入口是“某个特定植物的攻击组件”
- 变化维度主要是“原版植物特例”

我们不该直接迁移这种业务层。

更适合的方法是把其中可抽象的部分拆出来：

- 发射点数组
- lane 偏移规则
- 子弹参数装配
- 攻击冷却

然后把这些作为：

- `shoot` 效果的参数模板
- `ProjectileSpawnPattern`
- `TargetSelector`
- `FirePointProvider`

## 推荐迁移顺序

### 第一优先级

1. 子弹移动组件体系  
目标：把当前 `ProjectileRoot + MovementComponent` 升级为可扩展轨迹系统

2. `EventBus` 的优先级/一次性订阅/过滤器机制  
目标：增强当前运行时而不改变事件模型

3. `SceneRegistry` / 模板注册表  
目标：收敛散落 preload

### 第二优先级

4. 角色根节点 + 组件拆分规范  
目标：减少逻辑继续堆在 `PlantRoot/ZombieRoot`

5. Resource 驱动的条件和模板定义  
目标：逐步替换硬编码 demo

### 第三优先级

6. BoardSlot / LaneSlot 抽象  
目标：为未来放置规则、范围规则、生成规则打地基

---

## 具体落地建议

### 近期就可以做

1. 新建 `scripts/projectile/movement/`，按参考项目方式拆分轨迹实现
2. 给当前 `EventBus` 加 `priority / once / filter`
3. 做一个轻量 `ProjectileTemplateResource`
4. 做一个轻量 `SceneRegistry`，统一 projectile / effect / debug overlay 场景入口

### 暂时不要做

1. 不要引入完整 `PlantCell` 体系
2. 不要引入原版角色大枚举
3. 不要照搬 `MainGameManager`
4. 不要把攻击逻辑继续固化成具体植物专用组件树

---

## 最终判断

`PVZ-Godot-Dream` 最有价值的部分不是它“已经复刻了很多内容”，而是它已经踩过一遍 Godot PvZ 项目里最常见的工程化坑：

- Autoload 怎么组织
- 子弹场景怎么拆
- 角色和组件怎么挂
- Resource 怎么承载规则数据
- 场景引用怎么集中管理

这些都值得迁。

但它对我们来说不是“内容母仓”，而是“工程经验仓”。

所以正确策略不是：

- 把现成业务层搬过来

而是：

- 只迁通用基础设施
- 借成熟节点组织法
- 把原版专用逻辑重新抽象成我们的引擎层接口
