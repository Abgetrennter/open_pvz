# 参考项目综合对比分析

> 本文档对 `vendor/` 目录下的四个参考项目进行系统性对比，定位每个项目的架构特征、实现成熟度、与 Open PVZ 的差异，以及可借鉴的方向。

---

## 文档定位

这篇文档主要回答：

- 四个参考项目各是什么，用什么语言、什么引擎、什么架构
- 它们与 Open PVZ 在实体系统、状态管理、数据驱动、扩展性等维度上的本质差异
- 每个项目的突出亮点和结构性短板
- Open PVZ 可以从中借鉴什么、应该避免什么
- 综合视角下 Open PVZ 的定位和后续方向

这篇文档不负责：

- 逐文件级别的代码索引（见 [PVZ-Godot-Dream 代码索引](pvz-godot-dream/00-代码索引总表.md)）
- 具体的迁移实施计划（见 [参考实现迁移策略](25-参考实现迁移策略.md)）
- 替代各参考项目自身的 README

---

## 一句话结论

> 四个参考项目代表了同一游戏（PVZ）的四种工程实现路径——Godot OOP 组件化、Godot C# 继承、C++ 原版重建（×2）。它们在"功能完整度"上远超 Open PVZ，但在"架构可组合性"、"可测试性"和"无代码扩展性"上均不及 Open PVZ 的设计目标。Open PVZ 的核心挑战是把架构优势转化为内容规模。

---

## 一、四个参考项目速览

### 1. PVZ-Godot-Dream

| 维度 | 内容 |
|------|------|
| **本地路径** | `vendor/PVZ-Godot-Dream` |
| **语言** | GDScript |
| **引擎** | Godot 4.x |
| **架构** | OOP 继承 + 组件化 |
| **内容规模** | ~674 脚本、~230 场景、大部分原版内容 |
| **目标** | 高质量复刻原版 PVZ |
| **许可证** | Custom Non-Commercial License（禁止商业使用） |
| **已有调研** | [外部项目调研：PVZ-Godot-Dream](24-外部项目调研-PVZ-Godot-Dream.md)、[参考实现迁移策略](25-参考实现迁移策略.md) |

**核心架构特征：**

- `Character000Base` → `Plant000Base` / `Zombie000Base` 继承体系
- 角色行为通过具体组件拆分（`AttackComponentBase`、`HpComponent`、`ComponentMove`）
- 每个角色一个独立脚本（如 `component_attack_bullet_three_pea.gd`）
- `MainGameManager` 中央协调器，承担多种游戏阶段管理
- `CharacterRegistry` 集中管理角色数据
- EventBus 存在但非唯一通信通道，大量使用 Godot Signal + 直接方法调用

### 2. Godot-PVZ

| 维度 | 内容 |
|------|------|
| **本地路径** | `vendor/Godot-PVZ` |
| **语言** | C# |
| **引擎** | Godot 4.x |
| **架构** | 经典 OOP 继承 |
| **内容规模** | ~30 脚本、10+ 植物、5+ 僵尸 |
| **目标** | PVZ 学习/参考实现 |
| **许可证** | 未明确标注 |

**核心架构特征：**

- `Entity` → `HealthEntity` → `Plants` / `Zombie` 继承链
- C# 强类型带来更严格的接口约束
- `MainGame.Instance` 上帝对象管理几乎所有游戏逻辑
- 自定义 `HitBox` 接口 + Area2D 碰撞
- `StateMachine<TState>` 状态机管理游戏流程
- 手动对象池（数组 + 栈系统）
- 加权随机波次系统，含动态难度（基于场上僵尸 HP 百分比调整生成间隔）

### 3. cpp-PVZ

| 维度 | 内容 |
|------|------|
| **本地路径** | `vendor/cpp-PVZ` |
| **语言** | C++（C++23 标准，CMake 构建） |
| **引擎** | 自研 SexyAppFramework + TodLib |
| **架构** | 传统 OOP + switch 分发 |
| **内容规模** | 49 种子 + 30+ 僵尸（原版完整规模） |
| **目标** | 原版 PVZ 源码的 C++ 现代化重建 |
| **平台** | Windows（DirectDraw/Direct3D7） |

**核心架构特征：**

- `GameObject` → `Plant` / `Zombie` / `Projectile` / `Coin` / `LawnMower` / `GridItem`
- `Plant::UpdateAbilities()` 用巨型 switch 按 `SeedType` 分发到 `UpdatePeaShooter()` 等函数
- `Board` 类是宇宙中心——网格、波次、碰撞、渲染、阳光全部管理
- 自研 `DataArray<>` 模板对象池（预分配 + 空闲链表 + 位打包 ID）
- TodLib `Reanimation` 自研骨骼动画系统
- 无事件总线，实体间通过 `mBoard->IterateZombies()` 直接遍历

### 4. de-pvz

| 维度 | 内容 |
|------|------|
| **本地路径** | `vendor/de-pvz` |
| **语言** | C++（VS .sln 构建） |
| **引擎** | 自研 SexyAppFramework + TodLib |
| **架构** | 传统 OOP + switch 分发 |
| **内容规模** | 与 cpp-PVZ 基本一致 |
| **目标** | 原版 PVZ 源码的原始形态保留 |
| **平台** | Windows（DirectDraw/Direct3D7/8） |

**与 cpp-PVZ 的关系：** 两者共享同一代码主干（SexyAppFramework + TodLib + Lawn/），是原版源码的不同构建分支。cpp-PVZ 用 CMake 现代化构建，de-pvz 保留原始 VS 工程。架构和逻辑实现高度一致，以下分析将两者合并讨论，仅标注关键差异。

**cpp-PVZ / de-pvz 关键差异：**

| 维度 | cpp-PVZ | de-pvz |
|------|---------|--------|
| 构建系统 | CMake | Visual Studio .sln |
| C++ 标准 | C++23 | 传统 C++ |
| 平台层 | 相同（SexyAppFramework） | 相同 |
| 游戏逻辑 | 相同（Lawn/） | 相同 |
| 额外工具 | 资源解密、元数据提取 | 无 |

## 二、多维度对比

### 1. 架构模式

| 项目 | 模式 | 通信方式 | 状态管理 |
|------|------|----------|----------|
| **Open PVZ** | 事件驱动 + 模板装配 | EventBus 唯一通道 | 四协调者分解（Economy/Board/Card/Flow） |
| PVZ-Godot-Dream | OOP 继承 + 组件化 | EventBus + Signal + 直接调用 | MainGameManager 中央集权 |
| Godot-PVZ | 经典 OOP 继承 | Godot Signal + 直接调用 | MainGame 上帝对象 |
| cpp-PVZ / de-pvz | 传统 OOP + switch | 直接遍历 + 回调 | Board 宇宙中心 |

**关键差异：**

Open PVZ 是唯一使用"事件驱动 + 数据驱动模板"的项目。其他四个项目都走 OOP 继承路线，实体行为通过子类 override 或 switch 分发实现。这意味着 Open PVZ 的实体没有专属脚本，行为完全由 TriggerBinding + EffectDef 组合定义。

### 2. 实体系统

| 项目 | 实体创建方式 | 行为定义方式 | 新增角色成本 |
|------|------------|------------|------------|
| **Open PVZ** | EntityTemplate (.tres) → EntityFactory | TriggerBinding + EffectDef 组合 | 零代码（编辑器改 .tres） |
| PVZ-Godot-Dream | 场景实例化 + Registry | 角色脚本 + 组件 override | 新建 GDScript + 注册 |
| Godot-PVZ | 场景实例化 | 子类 override + 状态机 | 新建 C# 类 + 编译 |
| cpp-PVZ / de-pvz | DataArray 池 + NewPlant/NewZombie | switch 分发 + Update 函数 | 改 switch + 写函数 + 编译 |

**关键差异：**

Open PVZ 是唯一实现"零代码内容扩展"的项目。但它的表达力受限于已注册的 trigger/effect 策略类型。参考项目中任何实体都可以执行任意逻辑（C++/C#/GDScript），Open PVZ 中实体只能执行已注册策略的组合。

这个上限通过 `rule_pack` + `trusted_script` 扩展包设计来突破，但该机制尚未实现。

### 3. 数据驱动程度

| 项目 | 角色属性 | 行为逻辑 | 波次/关卡 | 素材 |
|------|---------|---------|----------|------|
| **Open PVZ** | .tres 模板 | .tres 触发绑定 | .tres 战斗场景 | 待设计 asset_pack |
| PVZ-Godot-Dream | Registry | 角色脚本（代码） | level_data.gd | 资源文件 |
| Godot-PVZ | 硬编码 C# | 硬编码 C# | 硬编码 C# | ResourceDB |
| cpp-PVZ / de-pvz | 硬编码 C++ | 硬编码 C++ | 硬编码数组 | 资源 ID 系统 |

**关键差异：**

Open PVZ 是唯一将行为逻辑完全数据化的项目。参考项目最多做到了"属性数据驱动"，行为逻辑始终需要编程。

### 4. 功能完整度

| 功能 | Open PVZ | PVZ-Godot-Dream | Godot-PVZ | cpp-PVZ / de-pvz |
|------|----------|-----------------|-----------|-------------------|
| 植物种类 | ~10 模板 | 大部分原版 | ~10+ | **49 种** |
| 僵尸种类 | ~5 模板 | 大部分原版 | ~5+ | **30+ 种** |
| 地形类型 | 草地 | 草地/泳池/屋顶 | 草地 | **草地/泳池/屋顶/夜间** |
| 游戏模式 | 验证场景 | 大部分原版模式 | 主战斗 | **冒险/生存/挑战/小游戏** |
| 存档系统 | 无 | 有 | 无 | **有（多用户档案）** |
| 图鉴 | 无 | 有 | 无 | **有** |
| 商店 | 无 | 有 | 无 | **有** |
| 迷雾系统 | 无 | 有 | 无 | **有** |
| 冰道系统 | 无 | 有 | 无 | **有** |
| 墓碑系统 | 无 | 有 | 无 | **有** |
| 割草机 | 基础 | 完整 | 基础 | **完整（含泳池/屋顶清扫）** |
| 阳光机制 | 有 | 完整 | 有 | **完整（天降+向日葵+掉落）** |

**关键差异：**

原版源码重建项目（cpp-PVZ / de-pvz）在功能完整度上对所有其他项目形成压倒性优势。PVZ-Godot-Dream 次之，但也覆盖了大部分原版内容。Open PVZ 和 Godot-PVZ 均处于早期阶段。

### 5. 可测试性

| 项目 | 自动化测试 | 调试工具 | CI 支持 |
|------|-----------|---------|--------|
| **Open PVZ** | **52 个验证场景** | DebugService + 事件历史 + 运行时快照 | 支持（headless + JSON 报告） |
| PVZ-Godot-Dream | 无 | EventBus debug 模式 | 无 |
| Godot-PVZ | 无 | GD.Print + TextEdit 调试组件 | 无 |
| cpp-PVZ / de-pvz | 无 | TOD_ASSERT + debug 模式 | 无 |

**关键差异：**

Open PVZ 是唯一拥有系统性自动化测试的项目。这个优势在所有维度中最具结构性——因为当游戏复杂度增长到数十种实体、数百种交互组合时，人工回归测试的可靠性会急剧下降。原版 PVZ 有 49 种植物、30+ 僵尸和无数交互，却完全依赖 15 年的玩家人工验证。

### 6. 扩展与 Mod 能力

| 项目 | Mod 机制 | 内容扩展 | 规则扩展 | 素材替换 |
|------|---------|---------|---------|---------|
| **Open PVZ** | 4 类扩展包 + 3 级信任 + 命名空间（设计中） | .tres 零代码 | rule_pack + trusted_script | asset_pack + 主题系统 |
| PVZ-Godot-Dream | 无 | 新建角色脚本 + 注册 | 改管理器逻辑 | 替换资源文件 |
| Godot-PVZ | 无 | 新建 C# 类 + 编译 | 改 MainGame | 替换资源文件 |
| cpp-PVZ / de-pvz | 无 | 改 C++ + 编译 | 改 Board | 替换二进制资源 |

**关键差异：**

Open PVZ 是唯一从架构层面设计 Mod/扩展支持的项目。其他项目的内容扩展都意味着源码修改。但 Open PVZ 的扩展包体系目前实现完成度约 30%，设计完备但落地仍需持续推进。

### 7. 对象池与性能

| 项目 | 对象池机制 | 内存管理 |
|------|-----------|---------|
| **Open PVZ** | 无显式池，委托 Godot 场景实例化 | Godot GC |
| PVZ-Godot-Dream | 无 | Godot GC |
| Godot-PVZ | 手动数组 + 栈回收 | C# GC |
| cpp-PVZ / de-pvz | **自研 DataArray<> 池**（预分配 + 空闲链表 + 位打包 ID） | 手动 placement new / 析构 |

### 8. 渲染与动画

| 项目 | 渲染后端 | 动画系统 |
|------|---------|---------|
| **Open PVZ** | Godot mobile | Godot AnimatedSprite2D / AnimationPlayer（未大量使用） |
| PVZ-Godot-Dream | Godot mobile | Godot 原生 + 自定义组件 |
| Godot-PVZ | Godot mobile | Godot 原生 |
| cpp-PVZ / de-pvz | DirectDraw/Direct3D7 | **自研 TodLib Reanimation**（骨骼动画 + 轨道 + 挂载 + 粒子） |

**关键差异：**

原版 PVZ 的 TodLib Reanimation 是经过数百万玩家验证的生产级动画系统。Open PVZ 在动画管线方面尚处于空白阶段，到内容填充期需要认真规划。

### 9. 波次系统

| 项目 | 波次定义 | 难度调整 | 特殊机制 |
|------|---------|---------|---------|
| **Open PVZ** | .tres 数据驱动 | 无自适应 | WaveRunner + 胜败检测 |
| PVZ-Godot-Dream | level_data 资源 | 预定义 | 多种原版模式 |
| Godot-PVZ | 硬编码 | **动态 HP 反馈自适应** | 大波次 2.5x、加权随机 |
| cpp-PVZ / de-pvz | 硬编码数组 | ZombiePicker 渐进 | 旗帜波、空投、墓碑钻出 |

**关键差异：**

Godot-PVZ 的"基于场上僵尸剩余 HP 百分比动态调整生成间隔"是所有项目中唯一的实时自适应难度机制，值得 Open PVZ 借鉴。

## 三、各参考项目的突出亮点

### PVZ-Godot-Dream

1. **Godot 生态最完整的 PVZ 实现**——大部分原版内容已复刻
2. **角色组件化拆分成熟**——攻击组件按子弹类型独立，移动组件按行为独立
3. **文档和代码注释相对完善**——有独立的子弹说明文档、开发文档
4. **EventBus 实现可直接参考**——优先级订阅、过滤器、历史记录

### Godot-PVZ

1. **动态难度波次系统**——场上 HP 反馈调整生成间隔，所有项目中唯一
2. **C# 强类型约束**——编译期类型安全，IDE 支持更好
3. **手动对象池**——性能意识明确，虽然侵入性强但思路正确
4. **状态效果系统**——冰冻/减速/狂暴等，简洁实用

### cpp-PVZ / de-pvz

1. **内容规模天花板**——49 种植物、30+ 僵尸，完整原版功能集
2. **DataArray 对象池**——预分配 + 空闲链表 + 位打包 ID，经过数百万玩家验证
3. **TodLib Reanimation**——自研骨骼动画系统，关键帧 + 轨道 + 挂载 + 粒子
4. **游戏模式全覆盖**——冒险/生存/挑战/小游戏/禅境花园
5. **多种地形完整实现**——草地/泳池/屋顶/夜间，含迷雾、冰道、墓碑、矿车

## 四、Open PVZ 相对参考项目的结构性优势

### 1. 架构可组合性

Open PVZ 的 trigger/effect 策略系统允许行为通过组合产生涌现效果。参考项目的实体行为是孤立的代码块，无法在不修改源码的情况下产生新的交互组合。

已验证的组合案例（错误技验证场景）：
- 级联爆炸型错误技（多源反击 + 死亡爆炸链）
- 溅射打击型错误技（终端爆炸多目标溅射）
- 高速追踪型错误技（追踪高速移动目标）
- 多车道反击型错误技（跨车道反击隔离）

### 2. 可测试性

52 个验证场景覆盖主干、第四阶段玩法层、第五阶段错误技与扩展专项以及第六阶段正式内容专项，支持 CI headless 运行，JSON 报告输出。参考项目全部没有自动化测试。

### 3. 零代码扩展

新增角色只需编辑 .tres 文件，不需要编程。参考项目全部需要源码修改和编译。

### 4. 扩展包平台

4 类扩展包（content/rule/asset/collection）+ 3 级信任 + 命名空间隔离 + 依赖解析，是从架构层面设计的 Mod 平台。参考项目全部没有 Mod 支持。

### 5. 架构纪律

冻结协议 + ProtocolValidator + ADR 决策记录 + 术语表，确保系统在演进过程中不腐化。参考项目完全依赖开发者自觉。

## 五、Open PVZ 相对参考项目的主要差距

### 1. 内容规模

~15 模板 vs 49+30（原版完整），这是最直观的差距。需要持续的 .tres 内容填充来缩小。

### 2. 功能完整度

存档、图鉴、商店、迷雾、冰道、墓碑、多种地形、多种游戏模式均未实现。这些是"完整游戏"的必要组成，但不一定是"规则引擎"的核心——需要根据项目定位决定优先级。

### 3. 特殊行为表达力

当某个角色的行为无法用已有 trigger/effect 策略组合表达时，需要注册新策略。这比参考项目中直接写任意逻辑要慢。`rule_pack` 设计了解决通道，但尚未实现。

### 4. 动画管线

原版 PVZ 的 TodLib Reanimation 是生产级骨骼动画系统。Open PVZ 尚未到内容填充阶段，动画管线基本未规划。

### 5. 自适应难度

Open PVZ 的波次系统是纯预定义配置。Godot-PVZ 展示了基于场上状态的动态难度调整，这是一个值得引入的方向。

### 6. 对象池

EntityFactory 层面没有显式池化机制。当实体规模增长时可能成为性能瓶颈，但可以在需要时透明引入。

## 六、可借鉴方向

### 从 PVZ-Godot-Dream 借鉴

| 方向 | 原因 | 建议 |
|------|------|------|
| EventBus 实现细节 | 优先级、过滤器、历史记录的 Godot 落地方式 | 已完成借鉴，可作为参考对照 |
| 投射物移动组件拆分 | Linear / Parabola / Track 的组件分离 | 已在 projectile/ 模块落地 |
| 角色本体 + 组件组织 | 根节点管基础属性，组件管行为 | 已在 entities/ + components/ 落地 |
| 原版 PVZ 运行时行为参考 | 查验某个具体角色的原版行为参数 | 作为查询手册使用 |

### 从 Godot-PVZ 借鉴

| 方向 | 原因 | 建议 |
|------|------|------|
| 动态难度波次 | 基于场上 HP 百分比调整生成间隔 | 作为 WaveRunner 增强方向 |
| 状态效果系统 | 冰冻/减速/狂暴，简洁实用 | 参考 Open PVZ 已有的 status 验证场景，评估是否需要更通用的抽象 |
| 对象池思路 | 手动池化的性能意识 | 在 EntityFactory 层面设计透明池化接口，不侵入业务逻辑 |

### 从 cpp-PVZ / de-pvz 借鉴

## 八、核心判断

### 四个参考项目的定位

- **PVZ-Godot-Dream**：Godot 生态中最完整的 PVZ 复刻，适合作为"原版行为参数查询手册"和"Godot 工程经验参考"
- **Godot-PVZ**：轻量级 C# 参考实现，突出亮点在动态难度和状态效果
- **cpp-PVZ / de-pvz**：原版 PVZ 源码重建，内容规模天花板，适合作为"完整功能对照表"和"底层优化思路参考"

### Open PVZ 的定位

Open PVZ 不是"又一个 PVZ 实现"，而是一个开放式规则引擎。它的价值不在于功能完整度，而在于：

1. 行为可组合（trigger + effect 策略组合产生涌现效果）
2. 内容零代码（.tres 模板驱动，编辑器即可扩展）
3. 运行时可验证（52 个验证场景 + CI）
4. 架构可演进（扩展包平台 + 冻结协议 + 治理体系）

### 核心挑战

> **把架构可组合性转化为内容规模。**

设计层面的完备性已经达到，但实现完成度仍需持续推进。关键风险在于：从当前实现到完整平台需要的工程量，是否能在保持架构纪律的同时持续推进，而不至于因为"先用硬编码快速出活"的诱惑而绕过扩展体系。
