# 验证清单

> 这篇文档记录当前项目的真实验证入口、验证场景和观察点。它的目标不是再列一组抽象原则，而是把“验证什么、怎么验证、看什么结果”落成工程清单。

---

## 文档定位

这篇文档主要回答：

- 当前主干应该至少做哪些验证
- 当前仓库已经有哪些验证场景
- 每类问题应该用什么入口复现
- 关键日志和通过条件应该怎么看

这篇文档不负责：

- 重新定义路线图阶段
- 重新解释空间模型或触发器协议

---

## 当前验证目标

当前验证不是为了证明“demo 看起来差不多”，而是为了证明下面 4 件事：

1. 当前规则运行时主干可重复运行。
2. 连续行为和事件链能共用同一主干。
3. 资源化配置不会破坏主干闭环。
4. 出问题时能够通过日志和自动化场景定位原因。

## 当前固定验证入口

### 1. 默认主场景

当前主场景：

- [`scenes/validation/minimal_battle_validation.tscn`](../../scenes/validation/minimal_battle_validation.tscn)

项目入口：

- [`project.godot`](../../project.godot)

它当前用于：

- 默认回归
- 可视化调试
- 运行时事件链观察
- `R` 键重复重置

### 2. 最小主干验证资源

当前场景资源：

- [`scenes/validation/minimal_battle_validation.tres`](../../scenes/validation/minimal_battle_validation.tres)

它当前验证：

- `game.tick`
- 线性投射物生成
- 追踪投射物生成
- 抛物线投射物命中
- `entity.damaged`
- `entity.died`
- 死亡爆炸伤害

### 3. 长距离抛物线专项验证

当前专项场景：

- [`scenes/validation/parabola_long_range_validation.tres`](../../scenes/validation/parabola_long_range_validation.tres)

它当前验证：

- 长距离抛射体生成
- 长距离抛射体终点命中
- 命中后 `entity.damaged` 写回

### 4. 自动化脚本入口

当前脚本：

- [`tools/run_validation.ps1`](../../tools/run_validation.ps1)

默认行为：

- 以 headless 方式运行 `parabola_long_range_validation`
- 自动打印验证报告
- 自动按验证结果退出

## 当前推荐验证流程

### A. 所有主干改动后的基础检查

1. Headless 启动检查
2. 默认主场景回归
3. 相关专项场景回归

### B. 涉及时空、命中、连锁问题时

除了基础检查，还应补：

1. 运行时快照日志
2. 事件链日志
3. 专项场景自动化验证

## 当前命令入口

### 1. 默认 headless 启动

使用你本机配置的 Godot console 可执行文件运行：

```powershell
& '<GodotConsole>' --headless --path '<project_root>' --quit-after 3
```

当前仓库本地常用路径已记录在：

- [`agent.md`](../../agent.md)

### 2. 跑默认自动化验证

```powershell
& '.\tools\run_validation.ps1'
```

### 3. 指定验证场景

```powershell
& '.\tools\run_validation.ps1' -Scenario 'res://scenes/validation/minimal_battle_validation.tres'
```

### 4. 打开运行时快照日志

```powershell
& '<GodotConsole>' --headless --path '<project_root>' --quit-after 240 -- --runtime-snapshot-log --runtime-snapshot-interval=30
```

### 5. 关闭 overlay 并只打印验证结果

```powershell
& '<GodotConsole>' --headless --path '<project_root>' -- --validation-scenario=res://scenes/validation/parabola_long_range_validation.tres --validation-auto-quit --validation-print-report --validation-no-overlay
```

## 当前验证矩阵

| 能力点 | 场景 | 入口 | 通过信号 |
|------|------|------|------|
| 事件总线基本闭环 | `minimal_battle_validation` | 默认主场景或指定场景运行 | 出现 `game.tick`，并能看到完整事件链 |
| 基础投射物生成 | `minimal_battle_validation` | 默认主场景 | `projectile.spawned` 出现线性、追踪两种 `move_mode` |
| 抛物线终点命中 | `minimal_battle_validation` | 默认主场景 | 出现 `projectile.hit` 且 `move_mode = parabola` |
| 长距离抛物线稳定性 | `parabola_long_range_validation` | `tools/run_validation.ps1` | `projectile.spawned -> projectile.hit -> entity.damaged` 都出现 |
| 死亡触发回写 | `minimal_battle_validation` | 默认主场景 | 出现 `entity.died` 和爆炸伤害链 |
| 日志与调试可解释性 | 默认主场景 | overlay 或日志 | 能看到触发器、效果、实体状态和验证状态 |

## 当前应该观察的日志

### 1. 事件日志

来源：

- `DebugService.event_log`
- `EventTrace` 控制台输出

重点看：

- `event_name`
- `source`
- `target`
- `depth`
- `tags`

### 2. 触发器日志

来源：

- `DebugService.trigger_log`

重点看：

- 哪个触发器 `fire`
- 哪个触发器 `skip`
- 当前事件深度

### 3. 效果日志

来源：

- `DebugService.effect_log`

重点看：

- 执行了哪个 `effect_id`
- 是由哪个事件驱动的
- 是否 `success`

### 4. 运行时快照

来源：

- `DebugService.runtime_snapshot_log`
- `BattleManager --runtime-snapshot-log`

重点看：

- 实体位置
- `status`
- `health`
- `values`
- 投射体的 `ground_position / height / target_position / hit_strategy`

## 当前通过标准

### 主干通过的最低标准

- `EventBus` 能稳定广播并记录结构化事件
- 至少一条完整链路能稳定出现：
  - `game.tick`
  - `projectile.spawned`
  - `projectile.hit`
  - `entity.damaged`
- 死亡链能回写：
  - `entity.died`
  - 后续效果
- 默认验证场景可重复重置，不出现随机漂移

### 连续行为通过的最低标准

- 线性和追踪投射物能稳定生成
- 抛物线投射物能在终点命中预期目标
- 长距离抛物线专项场景可自动通过

### 观测能力通过的最低标准

- 出问题时能通过日志定位是：
  - 事件问题
  - 触发问题
  - 效果问题
  - 弹道/命中问题

## 当前还缺什么

当前验证体系已经起步，但还没有完全完成。

下一步仍需要补：

- 批量验证入口，例如 `run_all_validations.ps1`
- 把 [验证矩阵](32-验证矩阵.md) 持续和验证场景同步
- 已通过 / 待通过 / 失败回归的状态记录
- 专门覆盖高度命中和多高度单位的验证场景

换句话说，当前验证体系已经不再是“肉眼看 demo”，但还没有到“完整回归平台”的程度。
