# 命名与可视化

> 这篇文档说明当前阶段真正应该稳定的命名对象和展示入口，同时把早期“熵灌注命名法”降级为远期创意方向，不再作为当前规范。

---

## 文档定位

这篇文档主要回答：

- 当前项目里哪些 id 和显示名属于正式命名对象
- 当前调试、验证和展示应该优先依赖什么可视化入口
- 早期命名创意为什么不能继续当成当前实现要求

这篇文档不负责：

- 设计一整套未来 UI 风格系统
- 代替术语表或模板编写约定
- 把生成器专属展示方案写成当前主干

---

## 当前状态

### 已决定

- 当前正式命名以稳定资源 id 为主，以 `display_name` 为辅。
- 当前最重要的“可视化”不是稀有度 UI，而是调试日志、验证报告和运行时观测。
- 任何未来的花名、别名或图标系统，都不能替代正式 id。

### 当前建议

- `template_id / scenario_id / binding_id` 等正式 id 保持 ASCII、稳定、可搜索。
- `display_name` 负责面向人阅读，不负责承担协议唯一性。
- 文件名尽量与资源 id 对齐，降低定位成本。

### 未来方向

- 为自动生成内容提供展示别名
- 为内容浏览器补更直观的摘要卡片
- 在不破坏稳定 id 的前提下，引入更有表现力的 UI 别名系统

---

## 当前正式命名对象

### 1. `EntityTemplate`

当前正式字段见：

- [`scripts/core/defs/entity_template.gd`](../../scripts/core/defs/entity_template.gd)

其中应明确区分：

- `template_id`：主键、引用键、日志定位键
- `display_name`：面向人阅读的显示名

当前更推荐的理解是：

- 代码、资源引用、验证日志优先看 `template_id`
- 展示面板、列表标题、样例说明可显示 `display_name`

### 2. `ProjectileTemplate`

当前正式字段见：

- [`scripts/core/defs/projectile_template.gd`](../../scripts/core/defs/projectile_template.gd)

同样应区分：

- `template_id`
- `display_name`

不要再把“投射物叫什么”混成：

- 文件名
- 渲染节点名
- 运行时唯一键

这三者应由稳定资源 id 收口。

### 3. `BattleScenario`

当前正式字段见：

- [`scripts/battle/battle_scenario.gd`](../../scripts/battle/battle_scenario.gd)

关键字段：

- `scenario_id`
- `display_name`
- `description`

这里的重点不是做炫酷标题，而是保证：

- 自动化脚本能稳定引用 `scenario_id`
- 人读报告时能看懂 `display_name`

### 4. `TriggerBinding`

当前正式字段见：

- [`scripts/core/defs/trigger_binding.gd`](../../scripts/core/defs/trigger_binding.gd)

关键字段：

- `binding_id`
- `behavior_key`
- `trigger_id`
- `effect_id`

其中：

- `binding_id` 负责资源级唯一标识
- `behavior_key` 更适合承担作者语义入口

不要再靠乱码式名称去承载运行时协议意义。

## 当前推荐的命名约定

### 1. 正式 id 要稳定、可搜索、可复用

当前更推荐：

- 使用 ASCII
- 使用 snake_case
- 让资源 id 和文件名尽量一致

例如：

- `plant_basic_shooter`
- `pea_linear`
- `minimal_battle_validation`

### 2. 显示名要服务阅读，不要反向污染协议

`display_name` 可以更友好，但应保持：

- 可读
- 简洁
- 不承担主键职责

也就是说，`display_name` 可以变，正式 id 不应轻易漂移。

### 3. 不要把未来花名系统写进当前主干

当前不建议作为正式 id 使用：

- 特殊符号
- 希腊字母
- 基于随机数的乱码名
- 只对某个展示页有意义的短别名

这些内容如果未来要做，也应只做“展示别名层”。

## 当前真正的可视化主线

当前项目里最重要的可视化入口，其实已经存在，而且它们直接服务开发。

### 1. Debug Overlay

当前实现见：

- [`scripts/debug/debug_overlay.gd`](../../scripts/debug/debug_overlay.gd)

它负责把下面这些信息直接暴露出来：

- 最近事件
- 触发器命中
- 效果执行
- 运行时状态

### 2. 事件、触发器和效果日志

当前实现见：

- [`autoload/DebugService.gd`](../../autoload/DebugService.gd)
- [`autoload/EventBus.gd`](../../autoload/EventBus.gd)

这套日志比“好看的名字”更重要，因为当前真正需要的是：

- 看清发生了什么
- 看清它为什么发生
- 看清它有没有偏离协议

### 3. 验证报告和回归产物

当前入口见：

- [`tools/run_validation.ps1`](../../tools/run_validation.ps1)
- [`tools/run_all_validations.ps1`](../../tools/run_all_validations.ps1)
- [`artifacts/validation`](../../artifacts/validation)

这部分是当前最稳定的“工程可视化”。

## 对“熵灌注命名法”的当前结论

早期“熵灌注命名法”更适合作为：

- 远期内容浏览器创意
- 自动生成内容的展示别名实验

而不适合作为当前正式规范。

原因很直接：

1. 当前主干还在收口协议和玩法系统边界。
2. 现阶段最重要的是可定位、可验证，而不是风格化命名。
3. 生成内容如果没有稳定主键，只会增加调试和维护成本。

如果未来真的恢复这条方向，至少应满足：

- 有稳定主键，不被展示别名替代
- 花名可逆或可追踪到 seed / 模板来源
- 验证报告和日志里仍能看见正式 id
- 别名系统只影响展示层，不影响协议层和资源层

---

## 当前不应主导实现顺序的内容

- 稀有度前后缀系统
- 专属乱码名称生成器
- 图标语法和徽章宇宙
- 为了展示名去反推协议结构

这些内容不是不能做，而是明显后置于：

- 正式资源命名收口
- 模板和验证体系稳定
- 第四阶段玩法系统接入
