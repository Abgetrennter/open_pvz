# 06-阶段路线图与完成定义

## 文档定位

本文把视觉反馈层任务串成阶段路线，说明每阶段目标、依赖、可并行任务、验证要求和完成定义。

## 背景与目标

视觉反馈层应逐步落地，避免一次性引入过多表现抽象。路线图以最小可验证闭环为优先：先事件 cue，再实体 actor，再层级环境，最后扩展槽和展示。

## 非目标

- 不要求一个阶段完成所有视觉体验。
- 不把正式素材接入作为前置阻塞。
- 不在早期开放 runtime visual script。
- 不因视觉路线阻塞规则引擎主线。

## Phase 0：文档与边界确认

目标：

- 确认视觉层不改变规则结果。
- 确认 v1 只开放 data-only cue/profile/fx/audio。
- 确认不新增 Mechanic family。

可并行任务：

- 整理当前文档包。
- 对齐既有 `plans/Open PVZ 视觉反馈层设计与路线图.md`。
- 标注后续需要同步的 wiki 文档。

串行依赖：

- 必须先完成 [00-总览与边界.md](./00-总览与边界.md) 的边界确认。

必须新增/更新的验证：

- 无。

阶段完成定义：

- 后续实现 agent 能明确视觉层输入、输出、禁止行为。

不应在该阶段做的事：

- 不写代码。
- 不新增 registry。
- 不新增验证场景。

## Phase 1：VisualCue 最小事件反馈

目标：

- 建立 `VisualCueDef / VisualFxDef / AudioCueDef`。
- 建立 `VisualCueRegistry`。
- 建立 `VisualFeedbackHost` 和 `VisualActionRunner`。
- 让核心事件产生可记录 action request。

可并行任务：

- def 与 registry 可并行设计。
- 内置 action runner 可与 debug log 并行。
- 内置 cue 可在 registry 结构完成后补入。

串行依赖：

- `VisualCueDef` 必须先于 `VisualCueRegistry`。
- `VisualFeedbackHost` 依赖 registry。
- cue smoke 依赖 debug log。

必须新增/更新的验证：

- `visual_registry_smoke`
- `visual_cue_projectile_hit_smoke`

阶段完成定义：

- `projectile.hit` 能触发 visual action request。
- cue 匹配失败不影响战斗。
- data-only cue 不执行脚本。

不应在该阶段做的事：

- 不实现复杂 actor。
- 不移除 fallback `_draw()`。
- 不开放 runtime action strategy。

## Phase 2：VisualProfile 实体表现

目标：

- 建立 `VisualProfileDef / VisualProfileRegistry`。
- 建立 `VisualActorComponent`。
- `EntityFactory` 可选挂载 actor。
- 支持状态到动画映射、受击闪白、血量阶段视觉变化。

可并行任务：

- profile def 与 placeholder actor 可并行。
- damage stage 与状态动画可并行。
- projectile actor 可在基础 actor 完成后独立推进。

串行依赖：

- `VisualProfileRegistry` 先于 `EntityFactory` 接入。
- `VisualActorComponent` 先于 profile smoke。
- projectile actor 依赖 layer policy 的最小接口或本地 fallback。

必须新增/更新的验证：

- `visual_actor_profile_smoke`
- `visual_projectile_projection_smoke`

阶段完成定义：

- 无 profile 的实体仍可显示。
- 有 profile 的实体能实例化 actor。
- `entity.damaged` 可触发闪白。
- projectile 本体/影子来源清晰。

不应在该阶段做的事：

- 不强制所有 archetype 配 profile。
- 不把 actor 节点用于命中判定。
- 不让 animation call track 直接调用规则方法。

## Phase 3：VisualStageLayer 层级环境

目标：

- 建立 `VisualLayerPolicy`。
- 建立 `VisualStageLayerService`。
- 统一 lane-based z_index。
- 接入 fog/weather/screen_fx host。

可并行任务：

- layer policy 与 host 节点结构可并行。
- projectile 分层与 world_fx 放置可并行。
- battlefield visual preset 可后置接入。

串行依赖：

- layer policy 先于 service。
- service 先于 actor/action 的正式 z_index 接入。

必须新增/更新的验证：

- 更新 `visual_projectile_projection_smoke`。
- 可补充 layer-specific smoke。

阶段完成定义：

- 层级计算有集中入口。
- 同 lane 内对象排序稳定。
- screen_fx 与 fog_weather 有独立宿主。

不应在该阶段做的事：

- 不把 roof/pool 碰撞规则挪到视觉层。
- 不让层级服务改变 `BattleBoardState`。

## Phase 4：扩展槽与素材包接入

目标：

- 为 `visual_cues`、`visual_fx`、`audio_cues`、`visual_profiles` 接入扩展注册。
- 更新 `ExtensionPackCatalog.ALLOWED_REGISTER_KINDS`。
- 补 extension smoke 和 guardrail。

可并行任务：

- 四类 registry 的 extension dir 配置可并行。
- 正向扩展包和负向 guardrail 包可并行准备。

串行依赖：

- registry 基础功能先于扩展包 smoke。
- trust_level 规则先于 guardrail。

必须新增/更新的验证：

- `visual_slot_guardrail`
- `visual_extension_pack_smoke`
- 更新 `tools/validation_scenarios.json`

阶段完成定义：

- data-only 扩展包可新增 visual 资源。
- `core.*` 覆盖被拒绝。
- 重复 id 被拒绝。
- runtime visual strategy 未开放。

不应在该阶段做的事：

- 不实现跨包 override。
- 不开放 trusted runtime visual action。
- 不做素材包安装器。

## Phase 5：展示场景与文档同步

目标：

- 增加展示场景或 showcase，演示最小视觉反馈。
- 同步 wiki 或 plans 中的实现事实。
- 明确下一阶段 backlog。

可并行任务：

- showcase 内容与文档同步可并行。
- 视觉 debug overlay 增强可并行。

串行依赖：

- 至少完成 Phase 1 和 Phase 2 的 smoke 后再做 showcase。

必须新增/更新的验证：

- 确认 visual smoke 与 guardrail 进入批量验证。
- showcase 不作为自动验证唯一依据。

阶段完成定义：

- 开发者能在展示场景看到 projectile hit、entity damaged、entity died、placement accepted 的基础反馈。
- 文档同步说明已落地能力和仍为草案的能力。

不应在该阶段做的事：

- 不把 showcase 结果当成规则正确性证明。
- 不为了演示绕过正式 registry。

## 最终完成定义

视觉反馈层完成 v1 时必须满足：

- 视觉反馈层可关闭。
- 无视觉资源时有 fallback。
- 核心事件能触发 data-only cue。
- 实体可通过 profile 接入 actor 表现。
- 投射体本体/影子符合当前 3D 逻辑 + 2D 投影模型。
- 扩展包可新增视觉资源和 cue，但不能覆盖 `core.*`。
- visual smoke 与 guardrail 进入批量验证。

## 风险与边界

- 若实现中需要修改规则协议，必须重新评估，不应在视觉任务中顺手完成。
- 若某个视觉需求需要脚本策略，先记录 backlog，不进入 v1。
- 若素材缺失，使用 placeholder 或 no-op，不阻塞系统能力验证。
