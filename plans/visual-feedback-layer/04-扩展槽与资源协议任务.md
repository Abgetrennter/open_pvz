# 04-扩展槽与资源协议任务

## 文档定位

本文拆解视觉反馈层与通用 Registry/Slot 生产线的关系。目标是让视觉资源和 cue 可以通过扩展包注册，同时保持 data-only 安全边界。

## 背景与目标

项目已落地 `RegistryBase + RegistryConfig + ContributorDef` 规范。视觉反馈层新增扩展能力时必须复用这条生产线，不允许新建独立注册逻辑。

目标：

- 为 visual cue、fx、audio cue、profile 建立 registry。
- 接入 `ExtensionPackCatalog.ALLOWED_REGISTER_KINDS`。
- 明确 trust_level。
- 补 smoke 与 guardrail。

## 非目标

- v1 不开放任意 visual action script。
- 不实现素材包安装器。
- 不实现跨包 override。
- 不允许扩展包注册或覆盖 `core.*`。

## 任务清单

### VF-SLOT-01：为 `visual_cues` 接入 `RegistryBase`

要求：

- contributor 为 `VisualCueDef`。
- register kind 固定为 `visual_cues`。
- required trust 默认为 `data_only`。
- `_register_builtin_defs()` 注册核心 cue。

### VF-SLOT-02：为 `visual_fx` 接入 `RegistryBase`

要求：

- contributor 为 `VisualFxDef`。
- register kind 固定为 `visual_fx`。
- `fx_scene` 缺失时注册可失败或 action no-op，必须记录清晰诊断。

### VF-SLOT-03：为 `audio_cues` 接入 `RegistryBase`

要求：

- contributor 为 `AudioCueDef`。
- register kind 固定为 `audio_cues`。
- 可先只作为 visual action 的请求目标。

### VF-SLOT-04：为 `visual_profiles` 接入 `RegistryBase`

要求：

- contributor 为 `VisualProfileDef`。
- register kind 固定为 `visual_profiles`。
- profile 缺失或 actor_scene 缺失不得中断实体创建。

### VF-SLOT-05：扩展 `ExtensionPackCatalog.ALLOWED_REGISTER_KINDS`

新增 register kind 固定为：

```text
visual_cues
visual_fx
audio_cues
visual_profiles
```

要求：

- data-only 扩展包可以注册这些资源。
- 不影响已有 register kind。

### VF-SLOT-06：补充 trust_level 规则

默认信任级别：

```text
data_only
```

后置预留但 v1 不实现：

```text
visual_action_strategies -> trusted_runtime
```

要求：

- 如果 manifest 尝试通过 data-only 包注册 runtime visual strategy，必须拒绝。
- 所有拒绝都进入 `DebugService.record_protocol_issue()`。

### VF-SLOT-07：补充 guardrail 场景

至少覆盖：

- 重复 id 拒绝。
- `core.*` 覆盖拒绝。
- 非法 def 类型拒绝。
- data-only 包注册 runtime visual script 拒绝。

## 实现说明

### 扩展包目录建议

建议扩展包路径：

```text
extensions/<pack>/data/combat/visual_cues/*.tres
extensions/<pack>/data/combat/visual_fx/*.tres
extensions/<pack>/data/combat/audio_cues/*.tres
extensions/<pack>/data/combat/visual_profiles/*.tres
```

具体路径可由 registry config 决定，但必须在文档和 guardrail 中一致。

### 命名空间规则

内置贡献项：

```text
core.*
```

扩展贡献项：

```text
<pack_id>.*
```

禁止：

- 扩展包使用 `core.*`。
- 跨包覆盖同名 id。
- 通过 visual cue 引用未声明或未注册的 runtime script。

### 缺失资源策略

素材资源缺失时：

1. 记录 visual warning。
2. action 降级为 no-op。
3. 不抛出 fatal。
4. 不影响战斗验证。

## 推荐实现顺序

1. 先实现 def 与 registry。
2. 接入内置 `core.*` 资源。
3. 接入 ExtensionPackCatalog register kind。
4. 新增最小扩展包 smoke。
5. 新增 guardrail。
6. 更新文档和验证分组。

## 验收标准

- 四类 visual register kind 均可被扫描。
- data-only 扩展包可新增 visual cue/fx/audio/profile。
- 扩展包不得覆盖 `core.*`。
- 重复 id 被拒绝并记录 `protocol.issue`。
- v1 不允许扩展包注册任意视觉脚本 action。

## 验证要求

建议验证场景：

- `visual_registry_smoke`
- `visual_slot_guardrail`
- `visual_extension_pack_smoke`

验证重点：

- registry 列表中包含内置项。
- 正向扩展包贡献项可查询。
- 负向扩展包贡献项被拒绝。

## 风险与边界

- 不要绕开 `RegistryBase` 自建扫描逻辑。
- 不要用 visual slot 变相开放规则脚本能力。
- 不要在 v1 支持跨包 override；这会显著增加冲突诊断复杂度。
