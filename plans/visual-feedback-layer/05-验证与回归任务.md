# 05-验证与回归任务

## 文档定位

本文拆解视觉反馈层的验证与回归任务。视觉验证的重点不是截图像素，而是 registry 状态、action request、关键节点和 guardrail。

## 背景与目标

项目当前以验证场景为主要测试机制，没有单元测试框架。视觉反馈层必须进入同一套验证体系，但不能让普通规则验证依赖肉眼观察。

目标：

- 为 visual registry 增加 smoke。
- 为 visual cue 增加正向事件触发验证。
- 为 VisualProfile 增加 actor 挂载验证。
- 为投射体投影增加本体/影子验证。
- 为扩展槽增加 guardrail。

## 非目标

- 不做像素级截图对比。
- 不要求所有视觉 FX 真正可见才算规则验证通过。
- 不用视觉验证替代现有逻辑验证。
- 不把 `vendor/` 素材纳入验证基线。

## 任务清单

### VF-TEST-01：visual registry smoke

场景名建议：

```text
visual_registry_smoke
```

验证：

- `VisualCueRegistry` 初始化无 `protocol.issue`。
- `VisualFxRegistry` 初始化无 `protocol.issue`。
- `AudioCueRegistry` 初始化无 `protocol.issue`。
- `VisualProfileRegistry` 初始化无 `protocol.issue`。
- 至少能查询到一个 `core.*` 内置项。

### VF-TEST-02：visual cue smoke

场景名建议：

```text
visual_cue_projectile_hit_smoke
```

验证：

- 构造 projectile 命中。
- 触发 `projectile.hit`。
- visual debug log 中记录 cue matched。
- visual debug log 中记录 `spawn_fx` 或 `flash_actor` action request。

### VF-TEST-03：projectile visual projection smoke

场景名建议：

```text
visual_projectile_projection_smoke
```

验证：

- 抛物线 projectile 处于高于地面的状态。
- 本体位置来自 `projected_position`。
- 影子位置来自 `ground_position`。
- 本体与影子在 y 轴上可区分。

### VF-TEST-04：visual profile actor smoke

场景名建议：

```text
visual_actor_profile_smoke
```

验证：

- 带 profile 的实体被创建。
- `VisualActorComponent` 被挂载。
- actor scene 被实例化。
- 状态变化产生动画 request。

### VF-TEST-05：visual slot guardrail

场景名建议：

```text
visual_slot_guardrail
```

验证：

- 重复 id 被拒绝。
- 扩展包覆盖 `core.*` 被拒绝。
- 非法 def 类型被拒绝。
- data-only 包注册 runtime visual script 被拒绝。

### VF-TEST-06：extension visual pack smoke

场景名建议：

```text
visual_extension_pack_smoke
```

验证：

- 扩展包注册 visual cue。
- 扩展包注册 visual fx。
- 扩展包注册 visual profile。
- 事件触发后能匹配扩展 cue。

### VF-TEST-07：更新 `tools/validation_scenarios.json`

要求：

- 将 visual smoke 放入合适分层。
- 将 visual guardrail 放入 guardrail 分层。
- 不改变现有验证场景语义。

## 实现说明

### Visual debug log 要求

为便于验证，视觉层应记录结构化日志：

```text
cue_id
event_name
matched
skip_reason
action_type
target_id
result
```

可落点：

- `DebugService.record_visual_event()`，如果实现该接口。
- 或临时 visual debug log，但必须能被验证场景读取。

### 非视觉验证兼容

所有现有验证场景默认应满足：

- 不设置 visual profile 也能运行。
- visual host 不存在也能运行。
- visual action 失败不改变 pass/fail。

### 验证粒度

优先验证事实：

- registry 是否有项。
- action request 是否产生。
- 节点是否挂载。
- z_index 是否符合公式。

不验证：

- 具体粒子形状。
- 贴图颜色是否精确。
- 动画帧是否与原版一致。

## 推荐实现顺序

1. 先实现 visual debug log。
2. 补 registry smoke。
3. 补 cue smoke。
4. 补 profile actor smoke。
5. 补 projectile projection smoke。
6. 补 extension pack smoke。
7. 补 guardrail。
8. 更新批量验证入口。

## 验收标准

- 规则验证不依赖视觉像素。
- 视觉验证只验证 action request、registry 状态和关键节点存在。
- 视觉失败不应让非视觉验证失败。
- DebugService 能输出 visual cue 匹配和 action 执行记录。
- visual smoke 与 guardrail 可在批量验证中稳定运行。

## 验证要求

建议场景名固定为：

```text
visual_registry_smoke
visual_cue_projectile_hit_smoke
visual_actor_profile_smoke
visual_projectile_projection_smoke
visual_slot_guardrail
visual_extension_pack_smoke
```

## 风险与边界

- 不要把视觉验证写成“运行场景但无断言”。
- 不要要求截图或人工观察作为完成标准。
- 不要让 visual smoke 依赖版权素材。
- 不要让 guardrail 修改全局 registry 后污染后续验证；必须隔离或重建 registry。
