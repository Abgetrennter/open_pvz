# 00-总览与边界

## 文档定位

本文定义视觉反馈层的设计思想、输入输出、禁止行为和参考项目吸收点。后续所有任务文档都必须遵循本文边界。

视觉反馈层是规则引擎之外的正交支撑层。它只解释已经发生的事实，不参与产生事实。

## 背景与目标

Open PVZ 当前主干已经形成：

- 事件链：`EventBus -> TriggerComponent -> EffectExecutor -> Runtime Action -> EventBus`
- 编译链：`CombatArchetype + CombatMechanic[] -> RuntimeSpec -> EntityFactory`
- 连续行为链：`ProjectileRoot + ProjectileMoveResult + ControllerComponent`
- 扩展槽：`RegistryBase + ContributorDef + ExtensionPackCatalog`

视觉反馈层应建立在这些事实之上，而不是反向改写它们。

目标：

- 用统一入口订阅规则事实。
- 将事件、状态、投影、层级解释为动画、FX、音频和 actor 显示变化。
- 保持视觉可关闭、可降级、可验证。
- 支持 data-only 扩展包新增视觉资源和 cue。

## 非目标

- 不做完整渲染管线替换。
- 不做 ECS 化。
- 不做原版 PVZ 一比一复刻。
- 不直接复用 `vendor/` 代码或素材。
- 不把视觉表现写入 Effect 策略。
- 不让验证结果依赖肉眼观察。

## 视觉层输入

视觉层只允许读取以下事实来源：

- `EventBus`
  - 事件名与 `EventData.core/runtime/ext`
  - 例如 `projectile.hit`、`entity.damaged`、`entity.died`
- `EntityState`
  - `status`
  - `health / max_health / health_ratio`
  - `values`
  - `status_effects`
- `ProjectileMoveResult`
  - `previous_position`
  - `current_position`
  - `previous_height`
  - `current_height`
  - `terminal_reason`
- `RuntimeSpec`
  - 可选视觉 profile id
  - 可选 actor/fallback 参数
- battlefield / mode visual preset
  - 背景、雾、雨、泳池、屋顶、screen_fx 等表现设定

## 视觉层输出

视觉层可以产生以下表现输出：

- 动画播放
  - `AnimationPlayer.play`
  - actor 状态机动画切换
- FX 场景生成
  - 局部 FX
  - projectile 命中 FX
  - screen overlay
- 音频 cue 请求
  - 只发起播放请求，不直接定义音频系统主协议
- actor 闪白 / 染色 / 状态 overlay
  - 受击闪白
  - 冰冻/魅惑/睡眠/灰烬等视觉层叠
- `z_index` / layer 调整
  - lane-based 排序
  - projectile 本体/影子分层

## 禁止行为

视觉层不得：

- 修改伤害值。
- 重新选择目标。
- 影响 `EffectExecutor` 执行顺序或结果。
- 改变状态持续时间。
- 改写 `EventData` 中的规则事实。
- 阻止实体死亡或销毁。
- 创建额外规则事件来替代正式逻辑事件。
- 让非视觉验证必须依赖可见像素或截图。

## 参考项目吸收点

### `de-pvz`

可吸收：

- `EffectSystem` 统一管理粒子、动画和 attachment。
- `Reanimation` 作为 actor 动画核心载体。
- `TodParticle` 表达短生命周期 FX。
- `Attachment` 支持局部挂点跟随。
- `RenderItem sort` 使用显式 render order，而不是依赖对象创建顺序。

不吸收：

- `Board / Plant / Zombie / Projectile` 内部直接硬编码表现逻辑。
- 伤害逻辑与特效播放混写。
- 用具体关卡和具体实体类型驱动视觉分支。

### `PVZ-Godot-Dream`

可吸收：

- `AnimationPlayer` 作为 Godot 侧 actor 动画落点。
- `BodyCharacter` 的颜色叠加和闪白思路。
- `ResourceBodyChange` 的血量阶段表现资源化思路。
- `SoundManager` 的音频池化、防重复播放经验。
- FX scene 作为短生命周期表现对象。

不吸收：

- 角色脚本直接连接大量表现信号。
- 子弹脚本直接持有命中特效和音效分支。
- 具体植物/僵尸类脚本里散落视觉策略。

## 推荐实现顺序

1. 先实现事件型 VisualCue，验证视觉层可以监听事实并产生可记录 action request。
2. 再实现 VisualProfile，让实体可选挂载 actor 表现。
3. 再实现 VisualStageLayer，统一层级、环境和投射体本体/影子。
4. 最后接入扩展槽和素材包。

## 验收标准

- 文档和实现均明确视觉层不改变规则结果。
- 任一视觉子系统失败时，战斗逻辑仍能继续。
- 无 visual profile / cue / fx 时，现有 fallback 表现仍可运行。
- DebugService 或 visual debug log 能记录 cue 匹配和 action request。

## 验证场景要求

本文不要求新增场景，但后续验证必须覆盖：

- visual registry 初始化无协议错误。
- visual cue 匹配后产生 action request。
- projectile 本体/影子坐标来源不同且符合投影模型。
- guardrail 拒绝 `core.*` 覆盖、重复 id、非法信任级别。

## 风险与边界

- 最大风险是视觉层侵入规则层。实现时任何“为了播放动画而改变规则状态”的改动都应拒绝。
- 第二风险是过早开放 runtime visual script。v1 应坚持 data-only cue 和内置 action。
- 第三风险是层级规则分散。`z_index` 计算必须集中到 VisualStageLayerService 或同等单点。
