# 第五阶段-错误技样例第一批清单

> 这份清单把第五阶段第一批错误技样例从“方向”收口为“已落地样例 + 下一批明确缺口”。它直接服务于路线图里“先用手工样例验证表达力，而不是先做完整生成器”的目标。

---

## 对齐 wiki 设计目标

结合 [当前阶段与实现路线](../wiki/01-overview/23-当前阶段与实现路线.md) 、 [开发路线图](../wiki/04-roadmap-reference/26-开发路线图.md) 和系统版图文档，这一批样例必须同时满足 4 条约束：

1. 差异主要来自模板、触发器、效果和资源组合，而不是新增业务脚本。
2. 必须继续复用当前主干：
   - 事件链
   - 模板实例化主链
   - 投射体与命中模型
   - 验证矩阵
3. 不为了样例反向污染第四阶段玩法边界。
4. 明确区分：
   - 现有系统可直接组合完成的样例
   - 需要扩展入口或新增效果位的样例

## 已可直接由现有系统组合完成的样例

### 1. 受伤反击型

- 场景：
  - [`reactive_retaliation_chaos_validation`](../scenes/validation/reactive_retaliation_chaos_validation.tres)
- 关键样例：
  - 两个 `plant_repeater_burst`
  - 一个 `zombie_reactive_bomber`
- 主要观察点：
  - 连发伤害能重复命中同一目标
  - `when_damaged` 反击能准确回打事件源
  - 样例仍通过统一模板与效果链完成

### 2. 死亡爆炸型

- 场景：
  - [`death_blossom_chaos_validation`](../scenes/validation/death_blossom_chaos_validation.tres)
- 关键样例：
  - `plant_basic_shooter`
  - `zombie_reactive_bomber`
  - 两个 `plant_wall_barrier`
- 主要观察点：
  - 普通伤害可以自然接出 `entity.died`
  - `on_death -> explode` 可同时命中多个附近植物
  - 没有为“错误技”写专用战斗脚本

### 3. 追踪抛射型

- 场景：
  - [`tracking_barrage_chaos_validation`](../scenes/validation/tracking_barrage_chaos_validation.tres)
- 关键样例：
  - `plant_track_bomber`
  - `plant_air_interceptor`
  - `zombie_basic_walker`
  - `zombie_air_scout`
- 主要观察点：
  - 地面追踪与空中追踪能并行存在
  - 同一追踪运行时可承载不同高度带样例
  - 差异仍由模板、飞行 profile 和目标条件决定

### 4. 级联爆炸型（交叉攻击链）

- 场景：
  - [`chain_explosion_cascade_validation`](../scenes/validation/chain_explosion_cascade_validation.tres)
- 关键样例：
  - `plant_basic_shooter`（线性投射）
  - `plant_track_bomber`（追踪投射）
  - `zombie_reactive_bomber`
  - `plant_wall_barrier`
- 主要观察点：
  - 两种不同投射类型的植物同时攻击同一个 reactive bomber
  - bomber 通过 when_damaged 分别向两个不同事件源反击
  - bomber 死亡后 on_death explode 伤害附近墙壁
  - 整条攻击-反击-爆炸链由模板组合驱动，无需专用脚本

### 5. 溅射打击型

- 场景：
  - [`splash_zone_cascade_validation`](../scenes/validation/splash_zone_cascade_validation.tres)
- 关键样例：
  - `plant_melon_lobber`
  - `zombie_bucket_tank`
  - `zombie_basic_walker`
- 主要观察点：
  - melon 抛物线终端 on_hit explode 同时伤害坦克和步兵
  - 溅射跨越不同目标类型（tank 500HP vs walker 270HP）
  - 差异来自模板的飞行 profile 和 on_hit 效果组合

### 6. 高速追踪型

- 场景：
  - [`fast_pursuit_cascade_validation`](../scenes/validation/fast_pursuit_cascade_validation.tres)
- 关键样例：
  - `plant_track_bomber`
  - `zombie_brisk_runner`（移动速度 80px/s）
- 主要观察点：
  - 追踪投射物可靠命中高速移动目标
  - 多发追踪投射物积累伤害
  - 追踪运行时对速度无需特判

### 7. 多车道反击型

- 场景：
  - [`multi_lane_retaliation_cascade_validation`](../scenes/validation/multi_lane_retaliation_cascade_validation.tres)
- 关键样例：
  - `plant_basic_shooter`（lane 0）+ `zombie_reactive_bomber`（lane 0）
  - `plant_repeater_burst`（lane 1）+ `zombie_reactive_bomber`（lane 1）
- 主要观察点：
  - lane 0 bomber 反击 shooter，lane 1 bomber 反击 repeater
  - 反击不串道，event_source 解析正确
  - 两条独立反击链通过共享运行时并行运行

## 新增扩展入口已承接的样例

### 1. 命中分裂型

- 场景：
  - [`hit_split_chaos_validation`](../extensions/phase5_chaos_pack/scenes/validation/hit_split_chaos_validation.tres)
- 当前落点：
  - 扩展包新增 `split_projectiles` effect
  - 通过扩展 `on_hit` 能力位在命中点分裂后续投射物
- 当前意义：
  - 已证明扩展入口可以承接新的 hit-response 原子，而不需要回头修改业务脚本

### 2. 周期召唤型

- 场景：
  - [`periodic_summon_chaos_validation`](../extensions/phase5_chaos_pack/scenes/validation/periodic_summon_chaos_validation.tres)
- 当前落点：
  - 扩展包新增 `spawn_entity` effect
  - 通过共享 spawn 主链周期生成新的召唤单位
- 当前意义：
  - 已证明扩展包不只会加模板，还能加更强的内容表达力

## 当前结论

第一批、第二批与扩展样例已经形成一条完整的第五阶段错误技验证路径：

- 已落地 7 个正式错误技验证场景
- 已落地 2 个扩展能力样例
- 第二批 4 个样例全部由现有系统组合完成，无需新增效果或触发器
- 新增 2 个样例则证明扩展入口已经能承接新的 effect 原子能力
- 已覆盖的能力域：受伤反击、死亡爆炸、追踪投射、级联传导、多目标溅射、高速追踪、多车道隔离、命中分裂、周期召唤

这满足了第五阶段持续验证要求：

> 先用一批正式、可回归的手工错误技样例验证主干表达力，再把真正需要的新原子能力推入扩展入口，而不是回到 `BattleManager` 特判。
