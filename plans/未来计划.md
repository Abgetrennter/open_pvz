# Open PVZ 未来基础设施演进计划

> 状态：当前观察项
> 更新：2026-05-09
> 适用阶段：规则基础设施第二轮已完成，后续只保留尚未被真实内容需求证明的低优先级方向。

本文原本承载实体活跃性、空间索引、高度过滤与 tick 预算监控的实施总览。对应实现已经完成并进入当前基线，原正式计划与草稿统一归档到：

- [rules-infrastructure-2026-05](./archive/rules-infrastructure-2026-05/README.md)

## 已完成基线

### 实体活跃性与状态控制

- `BaseEntity` 已建立多维 liveness profile。
- 旧 `is_combat_active()`、`is_attack_blocked()`、`blocks_attack` 运行时 API 已删除。
- 外部语义拆分为 `is_targetable()`、`is_damageable()`、`is_collidable()`、`is_counted_for_objectives()` 等明确接口。
- `StateComponent` 已统一转换入口，并支持 `entity.state_exited`、`entity.state_transition_rejected`。
- liveness 变更通过 `entity.liveness_changed` 进入事件观测链。

### SpatialIndex 与 HeightBand 过滤

- `BattleManager.spatial_query(params)` 已成为目标查询热路径的统一入口。
- Detection、Projectile、Zombie、Controller、Effect、LawnMower 已迁移到空间查询路径。
- `height_range` overlap 已进入空间查询标准过滤。
- `SpatialIndex` 只负责基础分组与空间/高度过滤，不缓存业务活跃性，不判断目标能否被攻击。
- 热路径全量扫描 fallback 已清理；`get_runtime_combat_entities()` 保留为 rebuild 数据源、验证基线和调试入口。

### 仿真与时序可观测性

- `BattleManager.step_simulation_tick()` 已记录 tick 耗时。
- 超阈值时通过 `simulation.control` / DebugService 记录 `tick_budget_warning`。
- 验证场景可覆盖 warning / critical 阈值，避免绑定机器性能数值。

### 验证基线

- 当前 `tools/validation_scenarios.json` 包含 `140` 个验证场景。
- 已补专项验证：
  - `liveness_system_validation`
  - `spatial_index_consistency_validation`
  - `tick_budget_monitor_validation`
- 最近一次完整矩阵结果：`140 passed / 0 failed`。

## 仍然暂缓的方向

### 对象池

当前不实施。

理由：

- 目前性能压力还没有被真实内容规模证明。
- tick 预算监控已经存在，可在高频 projectile 场景出现压力后再评估。
- 对象池属于性能优化层，不改变规则协议；只要继续保持 projectile 创建入口集中，后续补对象池的影响可控。

后续触发条件：

- 高频 projectile 场景稳定触发 tick budget warning。
- profiling 明确显示 `instantiate()` / `queue_free()` 成为主要开销。
- 需要大规模弹幕、密集连发或长时间 endurance 场景。

### 碰撞层声明

当前不实施。

现有 `team + liveness + HeightBand + query filter` 已覆盖近期目标、伤害、碰撞差异。等出现同队内不同碰撞规则、特殊免疫阵营或复杂友军交互时，再引入 collision group / collision matrix。

### BoardSlot modifier

当前不实施。

Doom-shroom 坑洞、屋顶斜面、水池深度等需求出现时，再在 BoardSlot 层引入 modifier。现阶段只要求新代码不要假设 slot 属性永久不变。

### 帧间插值

当前不实施。

规则验证不依赖表现平滑。等视觉表现层进入正式质量阶段后，再评估 previous/current transform 与 render alpha 插值。

## 当前维护约束

- 新目标查询默认优先使用 `BattleManager.spatial_query()`。
- 新状态或 status 影响实体行为时，优先表达为 liveness override，而不是新增单一布尔开关。
- 新验证场景若依赖时间推进，优先使用固定步进配置，不依赖真实时间。
- 新扩展点仍必须走 `RegistryBase + RegistryConfig + ContributorDef`。
- 不新增 Mechanic family；冻结 family 仍为 `Trigger / Targeting / Emission / Trajectory / HitPolicy / Payload / State / Lifecycle / Placement / Controller`。
