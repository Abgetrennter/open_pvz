# Open PVZ 未来基础设施演进计划

> 状态：正式计划
> 更新：2026-05-09
> 适用阶段：前期开发环境，允许一次性破坏旧 API，不要求兼容旧接口。

本文是未来基础设施工作的总览索引。详细方案已经从 `plans/draft/` 中收敛为正式计划：

| 计划 | 目标 | 优先级 |
|------|------|--------|
| [实体活跃性与状态控制计划](./实体活跃性与状态控制计划.md) | 删除分散的活跃性 API，建立多维 liveness 与统一状态转换入口 | P0 |
| [空间索引与高度过滤计划](./空间索引与高度过滤计划.md) | 用车道 + X 轴排序索引替代热路径全量扫描，并统一 HeightBand 过滤 | P0 |
| [仿真与时序层计划](./仿真与时序层计划.md) | 保持 100Hz 固定步进主模型，仅补低成本 tick 预算监控 | P1 |

## 设计约束

- 不新增 Mechanic family。仍然保持 Trigger / Targeting / Emission / Trajectory / HitPolicy / Payload / State / Lifecycle / Placement / Controller 十个冻结 family。
- 不为了兼容保留旧 API。`is_combat_active()`、`is_attack_blocked()`、`blocks_attack`、`FieldObjectRoot.activate()` / `deactivate()` 可以直接删除。
- 不把旧的单一 `combat_active` 语义换成另一个单一布尔值。必须拆成目标可见性、可受伤、可碰撞、运行时存活、目标统计等明确语义。
- 空间索引只管“在哪里”，不缓存活跃性、不知道目标是否可被攻击。调用方通过 `filter: Callable` 注入业务过滤。
- 状态事件只做通知，不驱动核心逻辑。状态转换、liveness override 的 pop/push 必须在 StateComponent 的统一转换入口内同步完成。
- 所有热路径重构必须保持确定性。排序结果同分时使用 `entity_id` 或稳定实例序作为 tie-breaker。
- 新增扩展点仍必须走 `RegistryBase + RegistryConfig + ContributorDef`。基础设施内部重构不绕过现有扩展插槽规范。

## 当前基础设施缺口

### P0：规则运行时基础设施

1. **实体活跃性与状态控制**
   - 当前 `is_combat_active()` 同时承担目标过滤、受伤过滤、胜败统计、占位判断等多种语义。
   - `is_attack_blocked()` 与 status 的 `blocks_attack` 只覆盖攻击暂停，不能表达睡眠、埋地、冻结、割草机待机等多维状态。
   - StateComponent 只有 `entity.state_entered`，缺少 exit / rejected 事件，也没有统一转换入口。

2. **空间索引与统一查询**
   - Detection、Projectile、Zombie、Controller、Effect、LawnMower 多处热路径都在全量扫描。
   - PVZ-like 车道结构适合使用 lane 分组 + x 排序，不需要四叉树。
   - 查询层需要同时支持 team、lane、tag、kind、x/radius、height、业务 filter。

### P1：仿真与时序可观测性

1. **tick 预算监控**
   - 当前已有 100Hz 固定步进、最大步数保护和线性 `simulation_speed`。
   - 缺少 per-tick 耗时记录。实体规模增加后无法定位超预算 tick。

### P2：按内容需求推进

1. **对象池**
   - 投射体高频创建销毁会成为性能热点，但需要先明确 ProjectileRoot 生命周期和 liveness 回收语义。
2. **碰撞层声明**
   - 当前 team + liveness + HeightBand 可以覆盖多数需求。等出现同队内不同碰撞规则时再引入 collision group。
3. **场地变形**
   - Doom-shroom 坑洞、屋顶斜面、水池深度等需求出现时，再实现 BoardSlot modifier。
4. **帧间插值**
   - 当前规则验证不依赖表现层。等视觉层进入正式表现质量阶段后再评估。

## 推荐实施顺序

1. 实体活跃性基础设施：新增多维 liveness，删除旧活跃性 API。
2. StateComponent 转换入口统一：`_execute_transition()` 负责 exit / enter / rejected / liveness 同步切换。
3. 全仓迁移调用点：Detection、Projectile、Zombie、Health、Trigger、WaveRunner、BattleModeHost、BoardSlot 等改用明确语义 API。
4. SpatialIndex 核心：实现 rebuild、分组索引、车道排序、统一 `query(params)`。
5. SpatialIndex 接入热路径：先 DetectionRegistry，再 Projectile/Zombie/Controller/Effect/LawnMower。
6. HeightBand 过滤统一：空间查询标准支持 `height_range` overlap。
7. tick 预算监控：在 `BattleManager.step_simulation_tick()` 记录超阈值 DebugService 事件。
8. 跑完整验证矩阵，并补专项验证场景。

## 验证要求

- 现有 `tools/validation_scenarios.json` 中所有验证场景通过。
- 新增 liveness 专项验证：
  - 蘑菇 sleeping 后 triggers/controllers 关闭，wake 后恢复。
  - Potato Mine arming 期间 targetable/damageable/collidable 关闭，active 后恢复。
  - frozen status 关闭 movement/controllers，但不影响 damageable。
- 新增空间索引专项验证：
  - 同一查询结果与全量扫描基准一致。
  - 同距离目标选择顺序稳定。
  - lane_forward、radius、tags_any、height_range 组合过滤正确。
- 新增性能观测验证：
  - 50+ 实体场景输出 tick 耗时统计，不要求固定性能阈值，只要求事件和汇总字段存在。
