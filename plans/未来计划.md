于对代码库、wiki 文档和设计草案的全面扫描，以下是宏观层面的基础结构缺口分析：

一、仿真与时序层
已有
100Hz 固定时间步 (simulation_tick_hz=100, fixed_dt=0.01)
game.tick 事件驱动所有定时行为
仿真速度控制 (simulation_speed)、暂停支持
帧累加器 + 最大步数保护 (防死亡螺旋)
缺口
缺口	说明	影响
帧间插值	仿真以 100Hz 独立步进，但渲染是 60Hz。抛射体移动和实体动画可能出现"抖动"——仿真步和渲染帧不对齐时位置跳变。当前没有 interpolation 层把仿真位置平滑到渲染帧	视觉流畅度
时间预算监控	没有 per-tick 耗时统计。当实体数量增多时，某个 tick 超过 10ms 预算会导致连续降步，但没有任何告警机制	性能可观测性
慢动作/变速曲线	仅有线性 simulation_speed，没有 ease-in/ease-out 时间曲线。Ice-shroom 冻结全场的"减速→恢复"如果走时间缩放，需要曲线支持	表现力
暂停粒度	只有全局暂停，没有"暂停但 UI 继续响应"或"暂停战斗但保留特效播放"的能力	UI/UX
二、空间与距离层
已有
BattlefieldMetrics 提供格数→世界单位换算
ProjectileFlightProfile 管理射程、碰撞半径、高度
5 车道 × N 槽位的网格坐标系
投射体 3D 逻辑 + 2D 投影模型（ground_position + height）
缺口
缺口	说明	影响
空间索引	所有范围查询（enemies_in_radius、scan_range）都是全量遍历。没有四叉树/网格哈希。当场上有 50+ 实体时，每 tick 每个实体的索敌都是 O(n²)	性能瓶颈
高度系统不统一	投射体有 height，实体有 hit_height_band，但缺少统一的高度层模型。Tall-nut 阻挡跳跃单位、Cactus 对空/对地切换、Blover 驱散飞行——这些行为的高度语义是分散在各处参数中硬编码的	规则一致性
碰撞层未形式化	没有 physics layer 定义。哪些实体会被哪些投射体命中、哪些实体会相互阻挡，目前靠分散的 flag 和 type 判断，没有统一的碰撞矩阵	扩展性
场地变形	Doom-shroom 坑洞 (G-29)、屋顶斜面、水池深度——场地本身没有可变属性的模型	高级关卡
三、实体生命周期层
已有
EntityFactory 从 RuntimeSpec 实例化
BattleSpawner 波次生成
queue_free() 销毁
entity.spawned / entity.died 事件
缺口
缺口	说明	影响
对象池	投射体高频创建/销毁（Peashooter 每秒 4 发），全部走 queue_free() 没有 pooling。GC 压力和实例化开销在高强度战斗中会累积	性能
实体分组/查询	没有 EntityManager 或 EntityGroup 系统。"查找所有飞行僵尸""查找所有金属装备僵尸"需要每次全量遍历+标签匹配	查询效率
生命周期钩子不完整	有 on_spawned、on_death，但缺少 on_disable、on_enable、on_recycle。Coffee Bean 唤醒实际上需要"暂停实体逻辑"而不只是状态切换	状态管理
四、状态机与行为控制层
已有
StateComponent 支持时间/事件驱动状态转换
ControllerComponent 每帧执行连续行为策略
10 个冻结 family 中有 State 和 Lifecycle
缺口
缺口	说明	影响
分层状态机	当前状态是扁平的。Chomper 需要"正常→攻击中→消化中"，Potato Mine 需要"布防→激活→触发→爆炸"，这些有优先级和互斥关系的状态用扁平模型很难正确表达	复杂行为正确性
状态转换约束	没有声明式转换表。哪些状态能转换到哪些状态、转换条件是什么，都散落在各 component 的代码里	可维护性
状态事件链	状态变化没有统一事件链。entity.state_entered 存在但下游消费不够系统化，视觉层/音频层/效果层各自监听不同事件	表现层驱动
五、表现层（设计完成，实现缺失）
系统	设计状态	实现状态	关键缺口
视觉反馈层	有 6 份计划文档，VisualCue/VisualProfile/VisualStageLayer 架构清晰	仅骨架（VisualFeedbackHost + 几个 registry）	无实际视觉策略实现，占位 sprite 无法表达复杂表现
音频系统	有完整设计：AudioCueDef、池化、动态 BGM 强度层、3 总线布局	AudioCueRegistry 仅注册表壳，无音频资源、无播放逻辑	完全空白
视觉表现层	有 VisualComponent/AnimDriver/FeedbackStack 设计	仅基础 sprite + 简单 AnimationPlayer	无动画状态机、无骨骼动画、无 reanim 导入
输入交互层	有 InputRouter + BoardCoordinateMapper + InteractionProfile 设计	InputRouter 存在但功能有限，触摸/手柄未实现	缺输入重映射、触屏适配
这四个系统的共同特征是：设计文档已经写好，但实现优先级被后置了。当前所有验证场景不依赖表现层，这是一个合理的选择，但如果要让游戏"可玩"而不只是"规则正确"，这些是绕不过去的。

六、完全未考虑的基础设施
系统	说明	为什么重要
存档/持久化	零实现。GameState 只持有运行时数据，没有序列化到磁盘的能力。关卡进度、解锁状态、设置偏好无处可存	完整游戏必需
关卡/流程管理	只有 ShowcaseHub 做场景切换，没有关卡序列、解锁门控、章节划分。BattleFlowState 管理"一场战斗内"的流程，但"关卡之间"的流程完全不存在	游戏循环
资源加载策略	所有资源在场景加载时一起加载。没有流式加载、按需加载、资源卸载策略。97 个 archetype + 127 个验证场景全在内存中	内存/加载时间
错误恢复	运行时错误（entity 引用失效、effect 执行异常）没有 graceful degradation。一个 effect 失败可能导致整条链断裂	稳健性
国际化/文本	没有文本管理系统。当前所有 UI 文字、debug 输出都是硬编码字符串	可扩展性
性能分析工具	没有 per-frame 性能指标、没有 entity count 监控、没有 tick 耗时热力图。DebugService 只记录逻辑事件，不记录性能数据	开发效率
七、优先级建议
从"补齐基础结构"的角度，我建议按这个优先级推进：

第一梯队：规则引擎基础设施
空间索引系统 — 直接影响性能上限，且索敌/范围效果/近距触发都依赖它
对象池 — 投射体高频创建是确定的性能热点
帧间插值 — 100Hz 仿真到 60Hz 渲染的平滑过渡
第二梯队：行为表达基础设施
分层状态机 — Chomper 消化、Potato Mine 布防、蘑菇睡眠等复杂行为的正确性依赖
高度层模型 — 统一 Cactus/Blover/Tall-nut/飞行单位的高度语义
实体分组查询 — 替代全量遍历，支撑 Metal tag、Flying tag 等按标签查询
第三梯队：表现层基础设施
视觉反馈层实现 — 已有详细设计，是"可玩性"的关键
音频系统实现 — 同上
输入交互层完善 — 触屏适配
其中第一梯队的三项是纯引擎层改动，不影响现有 archetype 资源和验证场景，风险最低。而且它们解决的是"规模问题"——当实体数量从当前的十几二十个增长到原版 PVZ 的 40-50 个时，没有空间索引和对象池会明显卡顿。

八、草案交叉冲突分析与解决

对四份设计草案做了交叉对比，发现以下冲突并确定了统一方案。

已解决的冲突

1. 两份空间文档并存——空间索引与车道系统设计草案是空间与距离层设计草案的演进版本（合并了实体分组/查询、更完整的查询接口），后者标记为"已被替代"。空间与距离层草案中唯一需要保留的取舍说明（不使用 dirty flag 懒重建，理由是投射体每帧移动使 dirty 几乎每帧为 true）已补充到前者中。

2. is_combat_active() 引用冲突——空间索引草案的 rebuild 引用了 is_combat_active()，活跃性草案计划删除它。解决方式：空间索引不缓存活跃性状态。rebuild 只做空间索引（lane + x 排序）和分组索引（team / lane / tag / kind）。活跃性检查放在查询时的 _apply_filters() 中，通过调用方提供的 filter 回调执行。这样 SpatialIndex 自始至终不知道 is_combat_active() 或 is_targetable() 的存在，两个系统完全解耦。

3. combat_active_only 语义 vs 活跃性三维度——空间索引查询参数的 combat_active_only: bool 替换为可选的 filter 回调（params["filter"]: Callable）。调用方自己决定过滤逻辑：无活跃性时用 is_combat_active()，有活跃性后用 is_targetable() / is_collidable() / is_damageable()。空间索引不需要改动。

4. Rebuild 策略——强制每 tick 重建，不用 dirty flag。50 个实体排序开销可忽略，强制重建更简单可预测。

5. 回退策略——迁移完成后删除旧查询路径，不维护全量扫描回退。

核心原则

空间索引只管"在哪里"，活跃性只管"能不能打"。两者通过调用方的 filter 回调连接，互不知道对方的存在。

实施顺序

空间索引（filter 回调用 is_combat_active）→ 活跃性系统（正常删除 is_combat_active）→ 调用方 filter 改为 is_targetable 等（空间索引本身不需要改动）
