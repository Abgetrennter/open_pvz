# 输入交互层设计讨论

> 日期：2026-04-19
> 状态：设计讨论阶段
> 背景：在当前仓库已经具备可玩 demo、正式战斗内容和专项回归的前提下，讨论如何把 demo 级输入链路提升为一套可复用的**支撑层规范化方案**。本文不把输入交互层视为当前主干的 P0 阻塞项。视觉表现层设计见 [视觉表现层设计讨论](视觉表现层设计讨论.md)。

## 1. 设计前提

从项目 wiki 和代码中提取的关键约束：

1. **逻辑与交互分离** — 输入层属于"呈现与治理"层，不直接驱动规则，而是翻译为游戏层接口调用
2. **事件可观测** — 输入动作应通过 EventBus 可追踪（调试、验证、回放）
3. **Resource 驱动** — 按键绑定、交互配置走 `.tres`，不硬编码
4. **验证场景兼容** — 已有验证场景通过 `card_play_requests` 程序化驱动，输入层是替代路径，不得破坏程序化驱动
5. **不依赖具体 UI 节点** — 场景级输入规范化层不应被 demo 级节点结构锁死

### 现有输入实现分析

项目已有一套可工作的 demo 级输入链路：

```
CardBar (Control) ─card_selected→ InputBridge ─→ BattleCardState.play_card()
BoardCellVisual (Area2D) ─cell_clicked──↗              │
SunCollectible (Area2D) ─clicked→ BattleEconomyState.collect_sun()
BattleManager ─R key→ restart
```

**已实现的能力：**
- 卡片点击选择（`CardBar._on_slot_input`）
- 格子点击放置（`BoardCellVisual._input_event` → `InputBridge._on_cell_clicked`）
- 阳光点击收集（`SunCollectible._on_click_area_input`）
- ESC 取消选择（`InputBridge._unhandled_input`）
- 卡片冷却覆盖层、费用变色（`CardBar` 内部）
- 格子 hover 高亮、无效放置闪烁（`BoardCellVisual` 内部）
- 程序化卡片放置（`BattleCardState._on_game_tick` 处理 `card_play_requests`）

**核心问题：**

| 问题 | 现状 | 影响 |
|------|------|------|
| 无交互模式管理 | `InputBridge` 用单一 `_selected_card_id` 字符串追踪 | 无法扩展铲子、拖拽等模式 |
| 输入逻辑散落在 demo 层 | `InputBridge`、`CardBar`、`BoardCellVisual` 都在 `scripts/demo/` | 无法被正式场景复用，引擎层无法测试 |
| 棋盘映射依赖 Area2D 节点 | 每个 `BoardCellVisual` 创建 Area2D + CollisionShape2D | 必须先创建节点才能检测点击，无法纯逻辑测试 |
| 无键盘快捷键 | 仅 ESC 和 R | 缺少卡片快捷键（1-9）、铲子快捷键 |
| 无输入事件可观测性 | 输入动作不经过 EventBus | 调试和回放无法追踪玩家操作 |
| 无铲子交互 | 完全缺失 | 无法移除已放置的植物 |

### 参考项目（PVZ-Godot-Dream）的输入架构

参考项目使用状态机 + 信号传递的模式：

```
HandManager (状态机: Null / Character / Item)
├── Null → 正常交互（收集阳光）
├── Character → 卡片选中，hover 预览，点击放置
└── Item → 铲子模式，hover 高亮植物，点击移除

CardSlotRoot._unhandled_key_input()  → 键盘快捷键 1-9, 0
PlantCell (Button click)            → 格子点击
Sun (Button click)                  → 阳光收集
UIShovel (Button click)             → 铲子切换
```

**参考项目的局限：**
- 每个格子是一个 Button 节点（`plant_cell.gd`），输入耦合在 UI 节点内
- `HandManager` 硬编码三种状态，不可扩展
- 键盘快捷键用 `Input.is_action_just_pressed` 轮询，不响应 InputMap 动作映射

### 原版 de-pvz 的交互语义补充

原版 `de-pvz` 虽然实现形态不适合直接迁移，但它在交互语义上有两个非常值得借鉴的点：

1. **“当前手里拿着什么”是正式状态。**
   - 原版把交互光标建成 `CursorType`，不仅区分普通放置，还区分铲子、炮击目标、花园工具等。
   - 这说明 Open PVZ 不应长期停留在“只有一个 `selected_card_id` 字符串”的状态表示上。

2. **“为什么不能放”是正式语义。**
   - 原版用 `PlantingReason` 表达 `only_in_pool / needs_pot / needs_upgrade / not_on_grave` 等失败原因，并让 tooltip / 提示 / 音效围绕这套 reason 工作。
   - 这对 Open PVZ 当前的 `placement.rejected` 很有参考价值：reason 不应只是日志字段，而应成为输入反馈和 UI 提示的正式输入。

## 2. 架构方案：输入路由器模式

### 2.1 核心思路

引入 `InputRouter` 作为**场景级 / 表现层级**的输入路由器。它管理交互模式状态机，将原始输入翻译为语义动作，通过信号和 EventBus 双通道分发。

```
┌─────────────────────────────────────────────────────┐
│                   Presentation Layer                │
│  CardBar ─signal→ ┐                                 │
│  BoardVisual ───→ │                                 │
│  SunNode ────────→│                                 │
│  Raw Input ──────→│                                 │
│                   ▼                                 │
│            ┌─────────────┐    ┌──────────────────┐  │
│            │ InputRouter │───→│ EventBus         │  │
│            │ (mode FSM)  │    │ input.action.*   │  │
│            └──────┬──────┘    └──────────────────┘  │
│                   │ signal                         │
│                   ▼                                 │
│  ┌──────────────────────────────────────────────┐   │
│  │           Gameplay Interface Layer            │   │
│  │  BattleCardState.play_card()                  │   │
│  │  BattleEconomyState.collect_sun()             │   │
│  │  (未来) BattleBoardState.remove_entity()       │   │
│  └──────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

**关键原则：**

- **InputRouter 不持有游戏状态** — 它只管理交互模式（idle/card/shovel/disabled）
- **InputRouter 不做游戏校验** — 费用不足、冷却中、格子被占等由游戏层判定，InputRouter 只负责传递
- **双通道输出** — 信号给直接消费者（高效），EventBus 给全局观测者（调试/回放）
- **显式交互模式优先** — 借鉴原版 `CursorType` 思路，后续扩展模式时优先新增语义态，而不是继续叠加临时布尔和字符串

### 2.2 职责边界

**InputRouter 负责：**
- 维护交互模式状态机（idle / card_selected / shovel / disabled）
- 接收来自 UI 节点的信号和 `_unhandled_input`
- 转发动作到游戏层接口
- 发射 `input.action.*` 可观测事件到 EventBus
- 在终端状态下自动阻止输入（battle victory/defeat）

**InputRouter 不做：**
- 不判定费用是否足够（`BattleCardState` 负责）
- 不判定格子是否可放置（`BattleBoardState` 负责）
- 不管理卡片冷却状态（`BattleCardState` 负责）
- 不决定 UI 长什么样（CardBar / BoardVisual 的职责）

### 2.3 交互模式状态机

```
         ┌───────────────────────┐
         │        IDLE           │ ← ESC / 右键 / 放置完成 / 放置拒绝
         │  (可点击阳光/卡片)     │
         └───┬──────────┬────────┘
             │          │
      点击卡片 │          │ 点击铲子 / S键
             ▼          ▼
    ┌─────────────┐  ┌─────────────┐
    │CARD_SELECTED│  │SHOVEL_ACTIVE│
    │(选格子放置) │  │(选格子移除) │
    └──────┬──────┘  └──────┬──────┘
           │                │
    点击格子 │         点击格子 │
           ▼                ▼
    play_card()      remove_entity()
           │                │
           └───→ IDLE ←─────┘

    任意状态 ── battle 终局 ──→ DISABLED
```

## 3. 组件设计

### 3.1 InputRouter（输入路由器）

```gdscript
# scripts/input/input_router.gd
class_name InputRouter
extends Node

## 交互模式
enum Mode {
	IDLE,            # 默认：可点击卡片、阳光
	CARD_SELECTED,   # 卡片已选：等待格子点击
	SHOVEL_ACTIVE,   # 铲子激活：等待格子点击
	DISABLED,        # 输入锁定：战斗终局、场景过渡
}

## 直接消费者信号
signal card_selected(card_id: StringName)
signal card_deselected
signal cell_clicked(lane_id: int, slot_index: int)
signal cell_hovered(lane_id: int, slot_index: int)
signal cell_unhovered(lane_id: int, slot_index: int)
signal shovel_activated
signal shovel_deactivated
signal sun_clicked(sun_id: int)
signal cancel_requested

## 当前模式（只读外部）
var mode: Mode = Mode.IDLE
var selected_card_id: StringName = StringName()

## 内部引用
var _card_state: Node = null       # BattleCardState
var _economy_state: Node = null    # BattleEconomyState
var _flow_state: Node = null       # BattleFlowState
var _board_mapper: RefCounted      # BoardCoordinateMapper
var _action_mapper: RefCounted     # InputActionMapper
var _profile: Resource             # InteractionProfile
```

**核心方法：**

func request_card_select(card_id: StringName) -> void:

	# 模式转换: IDLE → CARD_SELECTED

	# 任何非 IDLE 模式 → IDLE

## 4. 数据流设计

### 4.1 卡片选择 → 放置

```
玩家点击卡片（CardBar signal）或按快捷键（1-9）
  ↓
InputRouter.request_card_select(card_id)
  ↓ mode = CARD_SELECTED, selected_card_id = card_id
  ├→ emit card_selected(card_id)        → CardBar 高亮选中
  ├→ emit input.action.card_selected     → EventBus（可观测）
  ↓
玩家点击格子（BoardVisual signal 或坐标映射）
  ↓
InputRouter.request_cell_click(lane_id, slot_index)
  ↓ mode == CARD_SELECTED
  ├→ BattleCardState.play_card(selected_card_id, lane_id, slot_index)
  ↓
  ├→ 成功: emit card_deselected → mode = IDLE
  │       EventBus: placement.accepted → InputRouter 自动取消选择
  └→ 失败: EventBus: card.play_rejected → InputRouter 自动取消选择
```

### 4.2 阳光收集（任意模式可用）

```
玩家点击阳光（SunCollectible signal）
  ↓
InputRouter.request_sun_click(sun_id, collectible)
  ↓ 不改变当前模式（阳光收集是"中断式"操作）
  ├→ BattleEconomyState.collect_sun(collectible, null)
  └→ emit input.action.sun_collected → EventBus
```

### 4.3 铲子模式（Phase 5+ 预留）

```
玩家按 S 或点击铲子按钮
  ↓
InputRouter.request_shovel_toggle()
  ↓ IDLE → SHOVEL_ACTIVE 或 SHOVEL_ACTIVE → IDLE
  ├→ emit shovel_activated / shovel_deactivated
  ↓ (SHOVEL_ACTIVE 状态)
玩家点击格子
  ↓
InputRouter.request_cell_click(lane_id, slot_index)
  ↓ mode == SHOVEL_ACTIVE
  ├→ (未来) BattleBoardState.remove_entity_at(lane_id, slot_index)
  └→ mode = IDLE
```

### 4.4 EventBus 可观测事件

所有玩家输入动作都发射到 EventBus，使用 `input.action.*` 命名空间：

| 事件名 | core 字段 | 触发时机 |
|--------|----------|---------|
| `input.action.card_selected` | card_id, source("click"/"keyboard") | 卡片被选中 |
| `input.action.card_deselected` | reason("cancel"/"placed"/"rejected") | 卡片取消选择 |
| `input.action.cell_clicked` | lane_id, slot_index, mode | 格子被点击 |
| `input.action.cell_hovered` | lane_id, slot_index | 格子被 hover |
| `input.action.cell_unhovered` | lane_id, slot_index | 格子取消 hover |
| `input.action.sun_collected` | sun_id, source_type | 阳光被收集 |
| `input.action.shovel_activated` | source("click"/"keyboard") | 铲子激活 |
| `input.action.shovel_deactivated` | reason("cancel"/"used") | 铲子取消 |
| `input.action.cancel` | from_mode | ESC/右键取消 |

这些事件**纯观测性质**，不影响游戏逻辑。DebugService 可订阅用于调试日志。

### 4.5 与现有事件的关系

输入事件和游戏事件是**因果关系**，不是替代关系：

```
input.action.cell_clicked  →  card.play_requested  →  card.play_rejected
(观测：玩家点了格子)          (游戏：卡片出牌请求)      (游戏：放置被拒绝)
```

调试时可以追踪完整因果链：玩家操作 → 游戏请求 → 游戏结果。

补充建议：

- 后续可考虑把 `placement.rejected.reason` 与 `input.action.*` 观测事件串起来，形成稳定的“交互失败原因链”。
- 这样在 UI 层即可根据正式 reason 显示提示，而不是继续在输入层里写临时文案判断。

## 5. 与现有系统的集成

### 5.1 与 BattleCardState 的关系

`BattleCardState.play_card()` 签名不变。InputRouter 只是另一个调用者：

```
已有调用者: BattleCardState._on_game_tick (card_play_requests, 程序化)
新增调用者: InputRouter.request_cell_click (玩家交互)
```

**不改变 `play_card()` 的内部逻辑** — 验证、费用、冷却、放置 commit 全部在 BattleCardState 内完成。

### 5.2 与 BattleBoardState 的关系

`BattleBoardState` 提供 `get_slot_world_position()` 和棋盘参数（`board_slot_origin_x`、`board_slot_spacing`）。`BoardCoordinateMapper` 从这些参数构建映射。

### 5.3 与 SunCollectible 的关系

`SunCollectible` 已有 `clicked` 信号。集成方案：

```
方案 A: SunCollectible.clicked → InputRouter.request_sun_click → economy_state.collect_sun
方案 B: SunCollectible 保持直接调用 economy_state.collect_sun，InputRouter 仅发射观测事件
```

**倾向方案 B** — 阳光收集的交互非常简单（直接点击收集），没有模式依赖。强制路由到 InputRouter 增加了不必要的间接层。InputRouter 可订阅 `sun.collected` 事件来发射 `input.action.sun_collected` 观测事件。

### 5.4 与现有 demo 层的关系

| 现有 demo 组件 | 改造方向 |
|---------------|---------|
| `InputBridge` | 由 `InputRouter` 替代，保留为 demo 层薄壳（只做场景节点连线） |
| `CardBar` | 保留，信号从连 `InputBridge` 改为连 `InputRouter` |
| `BoardCellVisual` | 保留，继续负责视觉反馈（hover 高亮），坐标判定可选迁移到 `BoardCoordinateMapper` |
| `SunCollectible` | 保留，不改动 |
| `validation_card_place` | 保持程序化驱动，可选额外验证 InputRouter 集成 |

### 5.5 验证场景兼容性

- 所有已有验证场景通过 `card_play_requests` 驱动，**完全不依赖输入层**
- `validation_card_place` 和 `validation_sun_click` 已有模拟输入的代码（`viewport.push_input`），改造时需要适配 InputRouter 的接口
- 新增 `input_router_validation` 场景验证交互模式状态机转换

## 6. 铲子交互（Phase 5+ 预留设计）

铲子功能需要 `BattleBoardState` 增加 `remove_entity` 接口。当前设计为 InputRouter 预留模式：

```gdscript
# InputRouter 中预留
func _handle_shovel_click(lane_id: int, slot_index: int) -> void:
	# 检查格子是否有可移除的实体
	# 调用 battle_board_state.remove_entity_at(lane_id, slot_index)
	# 发射 input.action.shovel_used 观测事件
	# 模式回到 IDLE
	pass
```

铲子不阻塞当前实现。`InteractionProfile.enable_shovel` 默认 `false`，模式机中的 `SHOVEL_ACTIVE` 分支不生效。

## 7. 实施路线

| 阶段 | 内容 | 验证标准 |
|------|------|---------|
| **0** | 定义 Resource 类（`InteractionProfile`） | 可在编辑器中创建 .tres |
| **1** | `BoardCoordinateMapper` 纯数学实现 | 独立单元级验证：给定棋盘参数和屏幕坐标，返回正确格子 |
| **2** | `InputActionMapper` 键盘映射 | 按键→动作名映射正确 |
| **3** | `InputRouter` 骨架：模式状态机 + card_selected/cell_clicked 路径 | 替换现有 `InputBridge`，demo 场景功能不退化 |
| **4** | EventBus 观测事件集成 | `input.action.*` 事件可通过 DebugService 追踪 |
| **5** | 键盘快捷键（1-9 选卡，ESC 取消） | 快捷键操作等效于鼠标点击 |
| **6** | 铲子模式（需 BattleBoardState 配合） | `SHOVEL_ACTIVE` 模式可激活，点击格子可移除植物 |
| **7** | `input_router_validation` 验证场景 | 模式转换、快捷键、取消、终局锁定均有覆盖 |

每阶段独立可验证。阶段 0-3 是最小可用路径，阶段 4-5 是体验增强，阶段 6-7 是扩展功能。

## 8. 开放问题

1. **鼠标 vs 触摸统一** — 是否需要在 Phase 4 就考虑触摸输入？当前视口 960x540 适配 PC，移动端输入可能需要额外处理（长按 vs 短按、手势）。
2. **BoardCoordinateMapper 与 Area2D 并存策略** — 是否保留 `BoardCellVisual` 的 Area2D 检测作为主要路径，`BoardCoordinateMapper` 仅作为备用？还是完全迁移到坐标映射？
3. **输入回放** — `input.action.*` 事件是否应包含足够信息以支持未来回放？需要哪些额外字段（时间戳精度、输入来源标记）？
4. **多卡片快速放置** — 参考项目有"列模式"（一整列同时种植）。是否需要在模式机中预留？
5. **卡片拖拽放置** — 参考项目支持从卡片栏拖拽到格子直接放置（不经选中态）。是否纳入设计？
6. **输入优先级** — 当 UI 面板（暂停菜单、设置）打开时，`_unhandled_input` 是否自动被 Godot 的输入处理层级拦截，还是需要 InputRouter 显式管理？

## 10. 与视觉表现层的协作

输入交互层和视觉表现层有两个关键协作点：

1. **Hover 预览** — `cell_hovered` 信号需要触发 VisualComponent 的放置预览（半透明精灵跟随鼠标）。这是 `InputRouter.cell_hovered` → `VisualComponent` 的单向通知，不形成循环依赖。

2. **交互反馈** — 卡片选中时卡片放大（`CardBar` 已有）、格子无效放置闪烁（`BoardCellVisual` 已有）。这些反馈逻辑在 UI 层，不需要 InputRouter 介入。

两个系统通过信号/事件松耦合，不存在直接依赖。
