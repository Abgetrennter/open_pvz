# 音频系统设计

> 为 Open PVZ 设计事件驱动的音频支撑层规范化方案。不涉及具体音效内容，只定义播放管线、资源结构和系统集成方式。本文不将音频系统视为当前主干推进的 P0 阻塞项，而是作为与视觉、UI、输入并行收口的表现层基础设施。

## 一句话结论

> 音频系统应作为 EventBus 的事件订阅者存在，通过 Resource 定义的 AudioCueDef 将游戏事件映射为音效播放请求，再经 AudioService 统一调度池化播放器执行，作为当前阶段的**支撑层规范化方案**接入现有正式内容与回归主链，而不是反向定义游戏逻辑主线。

---

## 设计原则

1. **事件驱动，不直接调用** — 游戏逻辑不调用 `AudioService.play_xxx()`，而是通过 EventBus 事件间接触发。AudioService 订阅事件，查表决定播什么。
2. **Resource 配置驱动** — 音效映射规则写在 `AudioCueDef`（.tres）中，可随扩展包增删，无需改代码。
3. **逻辑素材 ID 解耦** — AudioCueDef 引用 `sfx.projectile.pea.hit` 这样的逻辑 ID，实际音频文件由 AssetRegistry 按素材包回退链解析。
4. **池化 + 防抖** — AudioStreamPlayer 池化复用，帧级冷却防同帧重复播放。
5. **对齐现有模式** — 遵循 autoload 单例模式、ProtocolValidator 验证模式、DebugService 日志模式。

## 与参考项目（PVZ-Godot-Dream）的关键差异

| 维度 | 参考项目 | Open PVZ 设计 |
|------|---------|--------------|
| 触发方式 | 游戏代码直接调用 `SoundManager.play_xxx()` | EventBus 事件订阅，查表播放 |
| 配置方式 | 硬编码在 GDScript 字典中 | AudioCueDef (.tres) Resource 驱动 |
| 扩展性 | 修改代码才能加新音效 | 扩展包放 .tres 即可 |
| 素材解耦 | 直接引用 `res://assets/audio/...` 路径 | 逻辑素材 ID → AssetRegistry 回退链 |
| 防抖 | 帧级 cooldown（25帧） | 帧级 cooldown（可配置，默认 15 帧） |
| 池化 | 按需创建，无正式回收 | 预分配池 + 自动回收 |
| 总线 | Master → BGM → SFX | Master → BGM → SFX → (UI) |
| 动态 BGM | 无（静态单曲播放） | 多层并行 + burst 状态机（借鉴原版 de-pvz） |

参考项目的直接调用方式在快速原型阶段很实用，但会导致音频调用散布在所有游戏代码中，无法替换或扩展。Open PVZ 选择事件驱动 + 查表模式，牺牲少量间接调用开销，换取完全可配置的音频映射。

## 与原版 de-pvz 的补充借鉴

原版 `de-pvz` 最值得借的不是“直接播放 API”，而是它已经把声音语义整理成了两层：

### 1. Foley 语义层

`TodFoley` 把音效类型与播放策略明确建模为：

- `FoleyType`
- `FoleyFlags`
- `MAX_FOLEY_INSTANCES`

其中最有借鉴价值的语义包括：

- `ONE_AT_A_TIME`：同类音效互斥
- `MUTE_ON_PAUSE`：暂停时自动静音
- `USES_MUSIC_VOLUME`：跟随音乐音量
- `DONT_REPEAT`：避免同一变体连续重复

这说明 Open PVZ 的 `AudioCueDef` 后续不应只停留在：

- `volume_db`
- `pitch_scale`
- `cooldown_frames`

而应进一步预留“互斥 / 暂停策略 / 去重策略 / 音量归属”这类语义位。

### 2. MusicTune 语义层

原版 `Music` 并不是简单的“战斗播一首、胜利播一首”，而是按：

- 白天 / 夜晚 / 泳池 / 雾夜 / 屋顶
- 迷你游戏 / 传送带 / Boss / 花园

这些场景语义切换 `MusicTune`。

这对 Open PVZ 的启发是：

- 未来 BGM 不应只按 `battle.phase_changed` 三态选择
- 可以进一步按 `battlefield_preset`、关卡模式和战斗内容域选曲
- "战场语义 -> 音乐语义" 应成为正式扩展方向

### 3. 动态 BGM 层（burst 系统）

原版 `Music.cpp` 的 burst 系统是最值得借鉴的动态音乐架构：

- **三层并行**：同一 .mo3 加载 3 次（主旋律/鼓组/踩镲），通过通道静音实现分轨
- **突发状态机**：`OFF → STARTING → ON → FINISHING`，由僵尸数量（≥10 触发，<4 结束）驱动
- **小节边界同步**：鼓组切入/切出等待下一个 order boundary，保证音乐连贯性
- **两种方案**：叠加型（白天等，鼓组淡入叠加到主旋律上）和替换型（夜晚，鼓组替换主旋律）
- **BPM 微调同步**：`MusicResyncChannel()` 检测多层间漂移，微调 BPM 修正

Open PVZ 无法直接复用 .mo3 tracker 机制，但可以用"多个 AudioStreamPlayer 并行 + 独立音量 Tween"实现等价效果。详见下方"动态 BGM（强度层混合）"章节。

## 总线布局

```
Master (Bus 0)
├── BGM  (Bus 1) — 背景音乐，多层并行（动态 BGM 时最多 3 个 AudioStreamPlayer），循环，支持淡入淡出
└── SFX  (Bus 2) — 音效，多实例池化
    └── UI   (Bus 3) — UI 音效子总线，独立音量控制
```

BGM 总线挂载 1-3 个 `AudioStreamPlayer`（静态 BGM 用 1 个，动态 BGM 用多个并行层），SFX 总线挂载播放器池。

通过 `project.godot` 或 `default_bus_layout.tres` 定义，AudioService 启动时应用。

## 核心组件

### 1. AudioService（autoload 单例）

```
位置：autoload/AudioService.gd
职责：音频系统的唯一入口，管理播放器池、订阅事件、查表播放
```

**核心接口：**

```gdscript
# ---- 生命周期 ----
func _ready() -> void:
    _init_bus_layout()
    _init_player_pool()
    _load_cue_tables()
    _subscribe_game_events()

# ---- 播放 API（主要供内部查表使用，也允许代码直接调用） ----
func play_cue(cue_id: StringName, override_position: Variant = null) -> void
func play_bgm(asset_id: StringName, fade_duration: float = 1.0) -> void
func stop_bgm(fade_duration: float = 1.0) -> void

# ---- 音量控制 ----
func set_bus_volume(bus_index: int, linear: float) -> void
func get_bus_volume(bus_index: int) -> float

# ---- 池管理 ----
func _acquire_player() -> AudioStreamPlayer
func _release_player(player: AudioStreamPlayer) -> void
```

**事件订阅策略：**

AudioService 在 `_ready()` 中订阅所有已知游戏事件，通过 `_on_game_event()` 统一分发：

```gdscript
func _subscribe_game_events() -> void:
    # 实体事件
    EventBus.subscribe(&"entity.damaged", _on_game_event)
    EventBus.subscribe(&"entity.died", _on_game_event)
    EventBus.subscribe(&"entity.spawned", _on_game_event)
    EventBus.subscribe(&"entity.status_applied", _on_game_event)
    # 抛射体事件
    EventBus.subscribe(&"projectile.spawned", _on_game_event)
    EventBus.subscribe(&"projectile.hit", _on_game_event)
    # 资源事件
    EventBus.subscribe(&"sun.collected", _on_game_event)
    # 战斗事件（SFX）
    EventBus.subscribe(&"battle.victory", _on_game_event)
    EventBus.subscribe(&"battle.defeat", _on_game_event)
    EventBus.subscribe(&"wave.started", _on_game_event)
    # 放置事件
    EventBus.subscribe(&"placement.accepted", _on_game_event)
    # BGM 专用事件（走独立回调）
    _subscribe_bgm_events()

func _on_game_event(event_data: Variant) -> void:
    # 查表匹配 cue → 播放
    pass
```

不对每个事件写独立回调，而是统一入口查表。这样增加新音效映射只需改 .tres 配置，不改代码。

**池化实现：**

```gdscript
const POOL_SIZE := 8
const POOL_GROW_STEP := 4
const POOL_MAX := 32

var _pool: Array[AudioStreamPlayer] = []
var _active: Array[AudioStreamPlayer] = []

func _init_player_pool() -> void:
    for i in POOL_SIZE:
        _pool.append(_create_player())

func _create_player() -> AudioStreamPlayer:
    var player := AudioStreamPlayer.new()
    player.bus = &"SFX"
    player.finished.connect(_on_player_finished.bind(player))
    add_child(player)
    return player

func _acquire_player() -> AudioStreamPlayer:
    if _pool.is_empty():
        if _active.size() < POOL_MAX:
            _grow_pool()
        if _pool.is_empty():
            # 池耗尽：回收最早的活跃播放器
            var oldest := _active.pop_front()
            oldest.stop()
            _pool.append(oldest)
    return _pool.pop_back()

func _release_player(player: AudioStreamPlayer) -> void:
    _active.erase(player)
    _pool.append(player)

func _on_player_finished(player: AudioStreamPlayer) -> void:
    _release_player(player)
```

**帧级防抖：**

```gdscript
var _frame_cooldowns: Dictionary = {}  # cue_id -> last_played_frame
var _current_frame: int = 0

func _process(_delta: float) -> void:
    _current_frame = Engine.get_physics_frames()

func _is_cooldown_active(cue_id: StringName) -> bool:
    if not _frame_cooldowns.has(cue_id):
        return false
    return (_current_frame - _frame_cooldowns[cue_id]) < _get_cue_cooldown(cue_id)
```

### 2. AudioCueDef（Resource 定义）

```
位置：scripts/core/defs/audio_cue_def.gd
职责：定义一个音效触发规则——什么事件、什么素材、什么参数
```

```gdscript
extends Resource
class_name AudioCueDef

## 唯一标识，用于查表和日志
@export var cue_id: StringName = StringName()

## 监听的事件名（如 &"projectile.hit"）
@export var event_name: StringName = StringName()

## 事件过滤条件（可选）：
## - "entity_kind": 只匹配特定实体类型（如 "plant", "zombie"）
## - "tags_any": 事件 tags 中包含任一即匹配
## - "tags_all": 事件 tags 中包含全部才匹配
@export var event_filter: Dictionary = {}

## 逻辑素材 ID，如 &"sfx.projectile.pea.hit"
## 实际资源由 AssetRegistry 解析
@export var asset_id: StringName = StringName()

## 直接引用（不需要 AssetRegistry 解析时的后备）
@export var stream: AudioStream = null

## 播放参数
@export var volume_db: float = 0.0
@export var pitch_scale: float = 1.0
@export var pitch_randomness: float = 0.0        # 音高随机偏移范围
@export var max_polyphony: int = 3                # 同 cue 最大并发数

## 防抖：同一 cue 两次播放之间的最小帧间隔
@export var cooldown_frames: int = 15

## 位置模式
## - "global": 全局播放（UI、BGM 等）
## - "entity": 跟随事件中的实体位置（需要 AudioStreamPlayer2D 支持，Phase 2）
@export var position_mode: StringName = &"global"

## 优先级（数值越低越重要，池满时低优先级被优先回收）
@export var priority: int = 0
```

**event_filter 匹配逻辑：**

```gdscript
func matches_event(event_name: StringName, event_data: Variant) -> bool:
    if self.event_name != event_name:
        return false
    if event_filter.is_empty():
        return true
    # entity_kind 过滤
    if event_filter.has("entity_kind"):
        var kind := StringName(event_filter.entity_kind)
        var data_kind := StringName(event_data.core.get("entity_kind", ""))
        if data_kind != kind:
            return false
    # tags_any 过滤
    if event_filter.has("tags_any"):
        var event_tags = event_data.core.get("tags", PackedStringArray())
        var has_any := false
        for tag in event_filter.tags_any:
            if tag in event_tags:
                has_any = true
                break
        if not has_any:
            return false
    return true
```

## 事件 → 音效映射流程

```
EventBus.push_event(&"projectile.hit", event_data)
  │
  ▼ AudioService._on_game_event(event_data)
  │
  ├─ 提取 event_name = "projectile.hit"
  ├─ 查索引 _cue_index["projectile.hit"] → [cue_a, cue_b, ...]
  │
  ├─ for each cue:
  │   ├─ cue.matches_event() → filter 匹配
  │   ├─ _is_cooldown_active(cue.cue_id) → 防抖检查
  │   └─ play_cue(cue.cue_id)
  │
  ▼ play_cue(cue_id)
    ├─ 获取 cue 定义
    ├─ 解析音频资源：AssetRegistry.resolve_audio(cue.asset_id) 或 cue.stream
    ├─ _acquire_player() → 从池中取播放器
    ├─ 设置 stream / volume / pitch / bus
    └─ player.play()
```

## 与现有系统的集成点

### 与 EventBus 集成

AudioService 是 EventBus 的纯消费者。它订阅事件、读取 event_data，但不产生新事件。这保证了音频系统不会干扰游戏逻辑链。

当前已有的事件可以直接映射：

| 事件 | 典型 cue 示例 | 类别 |
|------|-------------|------|
| `entity.damaged` | `sfx.entity.generic.hit` | SFX |
| `entity.died` | `sfx.entity.zombie.death` | SFX |
| `entity.spawned` | `sfx.entity.plant.place` | SFX |
| `entity.status_applied` | `sfx.status.freeze.apply` | SFX |
| `projectile.spawned` | `sfx.projectile.pea.launch` | SFX |
| `projectile.hit` | `sfx.projectile.pea.hit` | SFX |
| `sun.collected` | `sfx.resource.sun.collect` | SFX |
| `wave.started` | `sfx.wave.alert` | SFX |
| `placement.accepted` | `sfx.ui.card.place` | SFX |
| `placement.rejected` | `sfx.ui.card.reject` | SFX |
| `battle.phase_changed` | `bgm.battle.prepare` / `bgm.battle.main` | BGM |
| `battle.music_select` | `bgm.battle.day` / `bgm.battle.night` / ... | BGM 选曲 |
| `wave.huge_approaching` | 触发 burst 状态机 | BGM 动态 |
| `battle.threat_level_changed` | burst 强度判断 | BGM 动态 |
| `battle.victory` | `bgm.battle.victory` | BGM |
| `battle.defeat` | `bgm.battle.defeat` | BGM |

### 与 AssetRegistry 集成

AudioCueDef 的 `asset_id` 字段（如 `sfx.projectile.pea.hit`）直接对接素材包系统的逻辑素材 ID：

```gdscript
func _resolve_stream(cue: AudioCueDef) -> AudioStream:
    # 优先使用直接引用（开发/测试用）
    if cue.stream != null:
        return cue.stream
    # 通过 AssetRegistry 解析逻辑素材 ID
    if cue.asset_id != StringName():
        var asset_path = AssetRegistry.resolve_audio(cue.asset_id)
        if asset_path != null and asset_path != "":
            return load(asset_path) as AudioStream
    # 无资源时静默失败（不阻塞游戏）
    return null
```

AssetRegistry 尚未实现时，`cue.stream` 直接引用字段可作为临时后备。

### 与 DebugService 集成

```gdscript
# 播放成功
DebugService.record_audio_event(&"play", cue.cue_id, {
    "asset_id": String(cue.asset_id),
    "event_name": String(event_name),

# 防抖跳过

# 资源缺失
DebugService.record_audio_event(&"missing_asset", cue.cue_id, {

```

### 与 EntityTemplate 集成

素材包设计草案建议 EntityTemplate 补充 `sfx_profile_id` 字段。AudioCueDef 的 `event_filter` 可以按 `entity_kind` 或 `template_id` 过滤，实现"同一事件对不同实体播放不同音效"：

```
AudioCueDef: event=projectile.hit, filter.tags_any=["pea"], asset=sfx.projectile.pea.hit
AudioCueDef: event=projectile.hit, filter.tags_any=["bone"], asset=sfx.projectile.bone.hit
AudioCueDef: event=entity.died, filter.entity_kind="zombie", asset=sfx.entity.zombie.death
```

## 与 Effect 系统的关系

当前不将 `play_sound` 设计为 EffectDef。原因：

1. 音效是**表现层**关注点，不属于"行为效果层"（damage/spawn_projectile/explode）
2. Effect 执行链的核心是修改游戏状态，音效不修改任何状态
3. 将音效绑入 Effect 链会使表现层和逻辑层耦合

音效通过事件订阅自动触发即可：`damage` effect → `take_damage()` → `entity.damaged` 事件 → AudioService 播放受击音效。这条路径已经存在，无需在 effect 层面增加音频调用。

如果未来确有"效果播放专属音效"的需求（如某个特殊 effect 需要独特音效），可以通过在 effect 执行后 push 一个专用事件来实现，而不是直接在 effect 策略中调 AudioService。

## BGM 管理

BGM 播放与 SFX 不同：

- 同一时间只有一组 BGM（可能由多个并行层组成）
- 支持淡入淡出切换
- 与战斗阶段联动
- 支持动态强度层切换（原版 burst 系统）

### 静态 BGM 切换

最基础的 BGM 模式——场景/阶段变化时整首替换。

```gdscript
var _bgm_player: AudioStreamPlayer

func play_bgm(asset_id: StringName, fade_duration: float = 1.0) -> void:
    var stream := _resolve_stream_by_asset_id(asset_id)
    if stream == null:
        return
    if _bgm_player.playing:
        _fade_transition(_bgm_player, stream, fade_duration)
    else:
        _bgm_player.stream = stream
        _bgm_player.volume_db = -80.0
        _bgm_player.play()
        _fade_in(_bgm_player, fade_duration)

func _fade_transition(player: AudioStreamPlayer, new_stream: AudioStream, duration: float) -> void:
    var tween := create_tween()
    tween.tween_property(player, "volume_db", -80.0, duration * 0.5)
    tween.tween_callback(func():
        player.stream = new_stream
        player.play()
    )
    tween.tween_property(player, "volume_db", 0.0, duration * 0.5)
```

BGM 切换通过事件触发：

```
battle.phase_changed(preparing) → play_bgm("bgm.battle.prepare")
battle.phase_changed(running)   → play_bgm("bgm.battle.main")
battle.victory                   → play_bgm("bgm.battle.victory")
```

### 动态 BGM（强度层混合）

> 参考来源：`vendor/de-pvz/Lawn/System/Music.cpp` — 原版 burst 系统

原版 PVZ 的 BGM 不是简单的"一首曲子从头播到尾"，而是使用 **多轨道并行 + 音量动态混合** 实现战斗强度感知。

#### 原版架构分析

原版使用 BASS 音频库加载 tracker 格式（.mo3），核心手法：

1. **同一首 .mo3 加载 3 次**，分别作为主旋律层（main）、鼓组层（drums）、踩镲层（hihats）
2. 每个副本通过 `BASS_MusicSetAttribute` 静音不同的轨道通道（共 29 通道），使每个副本只播放特定乐器声部
3. 独立控制每层的音量，实现"叠加"而非"切换"的动态效果

**突发状态机**（`MusicBurstState`，Music.cpp:492-650）：

```
OFF ──(僵尸≥10)──→ STARTING ──(400 tick 计时)──→ ON ──(僵尸<4 且 800 tick 最小时长)──→ FINISHING ──→ OFF
```

**鼓组子状态机**（`MusicDrumsState`），在音乐小节边界（order boundary）同步切入：

```
OFF ──→ ON_QUEUED ──(下一个小节边界)──→ ON ──→ OFF_QUEUED ──(小节边界)──→ FADING ──→ OFF
```

**两种突发方案**：

| 方案 | 适用曲调 | 平静态 | 突发态 | 过渡手法 |
|------|---------|--------|--------|---------|
| 方案 1 | DAY / POOL / FOG / ROOF | 主旋律=1.0, 鼓组=0, 踩镲=0 | 主旋律=1.0, 鼓组=1.0, 踩镲=1.0 | 叠加：淡入踩镲(400tick)，在小节边界切入鼓组，结束反之 |
| 方案 2 | NIGHT (Moongrains) | 主旋律=1.0, 鼓组=0 | 主旋律=0, 鼓组=1.0 | 替换：主旋律淡出，鼓组在小节边界跳入并替换主旋律 |

**触发条件**（Board.cpp:5411-5427, Music.cpp:518）：
- 大波来袭倒计时 400（方案 1）或 700（方案 2）tick 时 `StartBurst()`
- 自动检测：每帧检查 `CountZombiesOnScreen() >= 10` 自动触发
- 退出：僵尸 < 4 且已播放至少 800 tick

#### Open PVZ 的适配方案

Godot 不支持 tracker 格式的通道级控制，但可以用 **多个 AudioStreamPlayer 并行播放独立音频文件 + 独立音量 Tween** 实现等价效果。

核心思路：把原版"同一文件不同通道"替换为"不同文件同时播放"，把"通道静音"替换为"整个 player 的音量控制"。

##### BgmLayerDef — BGM 层定义

```
位置：scripts/core/defs/bgm_layer_def.gd
职责：定义一个 BGM 层——什么素材、什么强度级别、默认音量
```

```gdscript
extends Resource

@export var layer_id: StringName = StringName()
@export var asset_id: StringName = StringName()
@export var stream: AudioStream = null
@export var intensity_level: int = 0
@export var base_volume_db: float = 0.0
```

`intensity_level` 定义此层应在什么强度下激活：
- `0` = 始终播放（基础旋律层）
- `1` = 突发时叠加（鼓组/踩镲层）

##### 动态 BGM 运行时

```gdscript
enum BgmBurstState {
    OFF,
    STARTING,
    ON,
    FINISHING,
}

var _bgm_layers: Array[AudioStreamPlayer] = []
var _bgm_dynamic_def: BgmDynamicDef = null
var _burst_state: BgmBurstState = BgmBurstState.OFF
var _burst_counter: int = 0

func play_dynamic_bgm(def: BgmDynamicDef) -> void:
    _stop_dynamic_bgm()
    _bgm_dynamic_def = def
    for layer_def in def.layers:
        var player := _create_bgm_layer_player(layer_def)
        _bgm_layers.append(player)
        if layer_def.intensity_level == 0:
            player.volume_db = layer_def.base_volume_db
        else:
            player.volume_db = -80.0
        player.play()

func _create_bgm_layer_player(layer_def: BgmLayerDef) -> AudioStreamPlayer:
    var player := AudioStreamPlayer.new()
    player.bus = &"BGM"
    player.stream = _resolve_stream_from_layer(layer_def)
    return player

func _resolve_stream_from_layer(layer_def: BgmLayerDef) -> AudioStream:
    if layer_def.stream != null:
        return layer_def.stream
    if layer_def.asset_id != StringName():
        var path = AssetRegistry.resolve_audio(layer_def.asset_id)
        if path != null:
            return load(path) as AudioStream
    return null
```

##### 强度切换（burst）驱动

与原版不同，Open PVZ 不让 AudioService 直接查询"屏幕上有多少僵尸"。而是通过 EventBus 事件驱动：

```
WaveRunner 检测到大型波来袭
  → EventBus.push(&"wave.huge_approaching", {countdown: 400})

BattleManager 每帧更新
  → EventBus.push(&"battle.threat_level_changed", {
       zombie_count: N,
       previous_count: M
     })

BattleManager 胜利/失败
  → EventBus.push(&"battle.victory", {})
  → EventBus.push(&"battle.defeat", {})
```

AudioService 订阅这些事件，维护 burst 状态机：

```gdscript
func _subscribe_bgm_events() -> void:
    EventBus.subscribe(&"wave.huge_approaching", _on_bgm_event)
    EventBus.subscribe(&"battle.threat_level_changed", _on_bgm_event)
    EventBus.subscribe(&"battle.victory", _on_bgm_event)
    EventBus.subscribe(&"battle.defeat", _on_bgm_event)
    EventBus.subscribe(&"battle.phase_changed", _on_bgm_event)

func _on_bgm_event(event_data: Variant) -> void:
    match event_data.core.get(&"event_name"):
        &"wave.huge_approaching":
            _start_burst()
        &"battle.threat_level_changed":
            var count: int = event_data.core.get(&"zombie_count", 0)
            if _burst_state == BgmBurstState.OFF and count >= _bgm_dynamic_def.burst_threshold:
                _start_burst()
        &"battle.victory":
            _stop_dynamic_bgm()
            play_cue(&"bgm.battle.victory")
        &"battle.defeat":
            _stop_dynamic_bgm()
            play_cue(&"bgm.battle.defeat")

func _start_burst() -> void:
    if _burst_state != BgmBurstState.OFF:
        return
    _burst_state = BgmBurstState.STARTING
    _burst_counter = _bgm_dynamic_def.burst_start_frames
    for i in _bgm_layers.size():
        var layer_def = _bgm_dynamic_def.layers[i]
        if layer_def.intensity_level > 0:
            var target_db = layer_def.base_volume_db
            var tween := create_tween()
            tween.tween_property(_bgm_layers[i], "volume_db", target_db,
                _bgm_dynamic_def.burst_fade_in_frames / 60.0)

func _process(delta: float) -> void:
    _current_frame = Engine.get_physics_frames()
    if _burst_counter > 0:
        _burst_counter -= 1
    if _burst_state == BgmBurstState.STARTING and _burst_counter <= 0:
        _burst_state = BgmBurstState.ON
        _burst_counter = _bgm_dynamic_def.burst_min_duration_frames
```

##### 与原版差异的映射

| 原版概念 | Open PVZ 对应 | 说明 |
|---------|-------------|------|
| .mo3 通道静音 | 独立 AudioStreamPlayer 音量 | 不依赖 tracker 格式，用独立文件 |
| `CountZombiesOnScreen()` | `battle.threat_level_changed` 事件 | 不让 AudioService 查询 Board，纯事件驱动 |
| `BASS_MusicSetAttribute(BPM)` | 不实现 | Godot AudioStreamPlayer 无 BPM 概念，多文件需在制作时保证 BPM 一致 |
| 小节边界同步（order boundary） | 不实现 Phase 1 | 需要音频元数据支持，Phase 3 可通过 AudioStreamOGGVorbis 的 playback position 检测近似实现 |
| `MusicResyncChannel()` BPM 微调 | 不实现 | Godot 无等效机制，素材制作时保证时长对齐即可 |
| `MusicTune` 枚举（13 值） | `BgmDynamicDef` Resource + 事件映射 | 用 Resource 配置代替硬编码枚举，可扩展 |
| `StartGameMusic()` 优先级瀑布 | BgmCueDef 事件过滤 + 优先级 | 同一事件匹配多个 BgmCueDef 时，高优先级胜出 |

##### 素材制作约束

动态 BGM 要求多个层文件在时长、BPM、结构上严格对齐。这是素材作者的责任，不是引擎责任。引擎只需保证：
- 多层同时 `play()` 后保持同步（Godot 的 AudioStreamPlayer 在同一帧启动误差 <1ms）
- 各层可以独立 Tween 音量

### BGM 选曲策略

原版 `StartGameMusic()` 使用一个 10 级优先级瀑布选曲。Open PVZ 将其映射为事件过滤 + 优先级：

```
AudioCueDef: event=battle.music_select, filter.tags_any=["final_boss"],
             asset=bgm.battle.final_boss, priority=-100
AudioCueDef: event=battle.music_select, filter.tags_any=["minigame"],
             asset=bgm.battle.minigame, priority=-50
AudioCueDef: event=battle.music_select, filter.tags_any=["night"],
             asset=bgm.battle.night, priority=-10
AudioCueDef: event=battle.music_select, filter.tags_any=["pool"],
             asset=bgm.battle.pool, priority=0
AudioCueDef: event=battle.music_select, filter.tags_any=["roof"],
             asset=bgm.battle.roof, priority=0
AudioCueDef: event=battle.music_select, filter.tags_any=["fog"],
             asset=bgm.battle.fog, priority=-5
AudioCueDef: event=battle.music_select, filter={},
             asset=bgm.battle.day, priority=100
```

BattleManager 在战斗开始时 push `battle.music_select` 事件，附上当前战场标签（如 `["pool", "night"]`）。AudioService 查表匹配所有 cue，选 `priority` 最高（数值最低）的那个播放。默认兜底的 priority 最大，确保总有一首匹配。

原版的"暴风雨关卡 = 无 BGM"也通过此机制实现：设置一个 priority 极低的静音 AudioCueDef 即可。

## 与扩展系统的集成

> 配套阅读：`wiki/04-roadmap-reference/38-扩展系统总体规划-v1.md`、`wiki/04-roadmap-reference/41-扩展包-manifest-规范-v1.md`

音频系统横跨扩展系统的两类包，需要明确归属与边界。

### 资源归属

| 资源类型 | 举例 | 归属包类型 | 理由 |
|---------|------|-----------|------|
| 音频素材文件 | `sfx.projectile.pea.hit` 的 .ogg | `asset_pack` | 扩展总规明确把"音效"归入素材包职责 |
| 音效映射规则 | `AudioCueDef`（事件→素材ID 的映射） | `content_pack` | "什么事件触发什么声音"是玩法内容，不是素材本身 |
| BGM 映射 | `bgm_mappings.tres` | `content_pack` | 战斗阶段→BGM 的映射属于内容配置 |

### manifest 入口

`asset_pack` 的 `entry_points` 应包含 `audio_index`（逻辑素材 ID → 文件路径映射）：

```json
{
  "pack_type": "asset_pack",
  "entry_points": {
    "audio_index": "audio_index.tres"
  }
}
```

`content_pack` 的 `entry_points` 应包含 `cue_tables`（AudioCueTable 目录）：

```json
{
  "pack_type": "content_pack",
  "entry_points": {
    "cue_tables": "data/audio/cue_tables/"
  }
}
```

两者解耦后，同一个内容包可以搭配不同音效风格的素材包。

### 扩展包添加音频的完整路径

```
asset_pack（素材包）
├── manifest.json
│   ├── pack_type: "asset_pack"
│   ├── entry_points: { "audio_index": "audio_index.tres" }
│   └── namespace: "my_mod"
├── audio_index.tres
│   "sfx.my_mod.laser.fire" → "assets/audio/laser_fire.ogg"
│   "sfx.my_mod.laser.hit"  → "assets/audio/laser_hit.ogg"
└── assets/audio/
    ├── laser_fire.ogg
    └── laser_hit.ogg

content_pack（内容包）
├── manifest.json
│   ├── pack_type: "content_pack"
│   ├── entry_points: { "cue_tables": "data/audio/cue_tables/" }
│   └── namespace: "my_mod"
└── data/audio/cue_tables/
    └── my_mod_weapon_cues.tres
        cue: event=projectile.hit, filter.tags_any=["laser"],
             asset_id="sfx.my_mod.laser.hit"
```

### AudioService 的扩展包扫描

当前设计只扫描 `data/audio/`。扩展系统集成后，AudioService 需要在 `PackRegistry` 加载完成后，汇入所有 `content_pack` 的 `cue_tables` 路径：

```gdscript
func _load_cue_tables() -> void:
    _load_cue_tables_from_dir("res://data/audio/cue_tables/")
    if PackRegistry != null:
        for pack in PackRegistry.get_packs_by_type(&"content_pack"):
            var cue_dir = pack.get_entry_point(&"cue_tables")
            if cue_dir != null:
                _load_cue_tables_from_dir(cue_dir)
```

### 信任分级

音频系统的所有组成部分——AudioCueDef、AudioCueTable、音频素材文件——都是纯数据资源，不包含可执行脚本。全部属于 `data_safe` 信任级别，适合广泛开放给内容作者。

### 命名空间与覆盖规则

`cue_id` 和 `asset_id` 应遵守扩展总规的命名空间规则：

| 层级 | 命名空间 | 举例 |
|------|---------|------|
| 主仓核心 | `core/*` 或无前缀 | `sfx.projectile.pea.hit` |
| 扩展包 | `pack_id/*` | `sfx.my_mod.laser.hit` |
| 素材覆盖 | `theme_override` | 同一素材 ID 指向不同 .ogg 文件 |

覆盖模式：

- `add`：新扩展包添加新 cue（默认行为）
- `theme_override`：素材包替换同一逻辑素材 ID 的实际音频文件（不改变映射规则）
- **不允许** `override` 主仓核心 cue 定义（如 `sfx.projectile.pea.hit` 的事件过滤条件）

这保证了"换皮不换规则"——素材包可以改变音色，但不能改变"什么事件触发什么音效"的游戏逻辑。

### 包级验证补充

扩展总规列出的 5 类包级验证中，需要补充的音频校验：

| 验证类型 | 音频相关补充 |
|---------|------------|
| 资源校验 | 检查 AudioCueDef 引用的 `asset_id` 能否通过 AssetRegistry 解析 |
| 注册校验 | 检查 `cue_id` 是否冲突、是否侵入主仓保留命名空间 |
| 包级 smoke test | 加载扩展包的 AudioCueTable，验证至少能触发一个 cue 播放 |

### 与 rule_pack 的边界

音频设计明确"不将 `play_sound` 设计为 EffectDef"，音效属于表现层，EffectDef 属于 `rule_pack`（级别 B 协议扩展），音频映射属于 `content_pack`（级别 A 数据扩展）。

如果未来有"某个特殊 effect 需要独特音效"的需求，应通过三层解耦实现：

```
rule_pack 定义新 EffectDef
  → EffectExecutor 执行后 push "entity.custom_effect_applied" 事件
    → content_pack 的 AudioCueDef 订阅该事件并播放对应音效
      → asset_pack 提供实际音频文件
```

各层各司其职，不跨层耦合。

## 开放问题

1. **AudioService 是否需要知道战斗上下文？** 当前设计是纯事件驱动，不依赖 GameState。如果需要"战斗内才播放战斗音效"的逻辑，可以通过事件是否存在来隐式判断，不需要显式依赖。

2. **cue 表热重载？** 开发阶段可能需要运行时修改 cue 表。可以在 DebugService 中暴露 `reload_cue_tables()` 方法。

3. **与视觉表现层的同步？** 状态效果（冰冻变色 + 冰冻音效）需要同一事件同时触发视觉和音频。当前事件驱动模式天然支持这一点：`entity.status_applied` 事件同时被 AudioService 和未来的 VisualService 订阅。

4. **音频资源加载时机？** Phase 1 使用 `cue.stream` 直接引用（Godot 随场景加载）。Phase 2 接入 AssetRegistry 后，需要决定是预加载还是按需加载。建议 SFX 按需加载（音效短），BGM 预加载（音乐长）。

5. **AudioService 与 PackRegistry 的初始化顺序？** AudioService 需要在 `PackRegistry` 加载完成后才能汇入扩展包的 cue_tables。需要确认 autoload 的加载顺序，或在 AudioService 中监听 PackRegistry 的加载完成信号。

6. **扩展包 cue 冲突时的优先级？** 当两个 content_pack 注册了相同 `event_name` + 相同 `event_filter` 的 cue 时，应如何决定优先级？建议按 `load_priority` 排序，或直接拒绝同名 cue_id 冲突。

7. **音频验证场景是否纳入扩展包 smoke test？** 建议在 `tools/validation_scenarios.json` 中增加"扩展包音频集成"验证场景，覆盖：扩展包 cue 加载、素材解析、防抖、池回收。

8. **动态 BGM 层的同步精度？** 原版通过 BPM 微调实现多轨道同步，Godot 无等效机制。多层同时 `play()` 后的漂移量需要实际测量。如果漂移 > 50ms，可能需要在 Phase 3 引入定期 re-sync（停止并重新对齐播放位置）。

9. **BgmCueDef 与 AudioCueDef 是否合并？** 当前 BGM 选曲和 SFX 触发都使用 `AudioCueDef`，但 BGM 播放走 `play_bgm()`/`play_dynamic_bgm()` 而非 `play_cue()`。可以考虑引入 `bgm: bool = false` 字段区分，或保持两个独立的查询路径。前者简化数据结构，后者保持关注点分离。

10. **静态 BGM 与动态 BGM 的共存策略？** 如果素材包只提供单文件 BGM（无分层素材），应回退到静态 BGM 模式。AudioService 需要在 `play_dynamic_bgm()` 失败时自动回退到 `play_bgm()`。这意味着 `data/audio/bgm/core_bgm_mappings.tres`（静态映射）应始终存在作为兜底。
