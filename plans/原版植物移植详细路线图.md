# 原版植物移植详细路线图

- 状态：执行计划
- 对应治理文档：`wiki/05-governance/36-原版实体复刻工作流.md`

> 本计划把“原版植物移植设计方案”拆成可执行路线图。目标是在不新增大架构、不回退 legacy 作者模型的前提下，按风险递增完成 PVZ1 原版 49 个标准植物的机制优先移植。

---

## 一、总目标

### 1. 范围

本路线图覆盖 PVZ1 种子槽中的 `49` 个原版植物：

- 冒险主线植物
- 夜间植物
- 水池与屋顶植物
- 商店升级植物
- `Imitater`

不覆盖：

- `Explode-o-nut`
- `Giant Wall-nut`
- `Sprout`
- `Leftpeater`
- 迷你游戏专属非标准 seed
- 完整 reanim / 音频 / 原版 UI 资产复刻

### 2. 验收等级

默认采用“机制优先”：

- 必须还原原版核心战斗机制。
- 必须还原卡片费用、冷却和放置约束。
- 必须补 validation，并进入批量回归。
- 表现层允许使用当前占位视觉。

### 3. 技术硬约束

- 新内容正式入口只能是 `CombatArchetype + CombatMechanic[]`。
- `EntityTemplate / TriggerBinding` 只能作为 legacy 兼容背景，不得新增为正式资源。
- `legacy_template_id` 只能用于身份、事件和迁移对照，不得驱动行为。
- 放置语义优先走 `Placement` mechanic / `placement_spec`。
- mode 差异优先走 `BattleModeHost / BattleRuleModule / BattleInputProfile / BattleObjectiveDef`。
- 不允许为了单个植物向 `BattleManager` 增加特判。

## 二、阶段路线

### 阶段 0：迁移底账与基线校准

目标：建立可追踪的 49 植物迁移底账，避免边做边猜。

任务：

1. 从 `vendor/de-pvz/ConstEnums.h` 提取标准 `SeedType 0..48` 清单。
2. 为每个植物建立迁移记录，至少包含：
   - 原版 `SeedType`
   - 原版名称
   - 费用
   - 冷却
   - 生命值
   - 放置条件
   - 攻击或效果入口
   - 投射物或状态效果
   - 可复用 Open PVZ 资源
   - 协议缺口
3. 对齐当前已有样板资源：
   - `archetype_basic_shooter`
   - `archetype_sunflower`
   - `archetype_wall_barrier`
   - `archetype_frost_pea`
   - `archetype_repeater_burst`
   - `archetype_cabbage_lobber`
   - `archetype_melon_lobber`
   - `archetype_pumpkin_cover`
   - `archetype_flower_pot_surface`
4. 生成首版“原版植物迁移状态表”，建议存为 `plans/original-plant-migration-ledger.md` 或后续正式工具 JSON。
5. 校准 Wiki 中手写验证数量，确保以 `tools/validation_scenarios.json` 为准。

产物：

- 49 植物迁移底账。
- 第一轮协议缺口清单。
- 当前可复用资源映射表。

验收：

- 每个植物都有明确批次归属。
- 每个植物都能判断“只需资源”“需最小 effect/type”“需协议设计”三类之一。

## 四、批次 B：即时与定时

植物：

- Cherry Bomb
- Potato Mine
- Squash
- Jalapeno
- Chomper
- Tall-nut
- Pumpkin

目标：

- 覆盖放置后触发、延迟激活、近距离吞噬、行范围伤害和覆盖防御。

主要任务：

1. Cherry Bomb
   - 使用 `Lifecycle.core.on_place + Payload.core.explode`。
   - 校准爆炸半径、伤害和触发延迟。
2. Potato Mine
   - 使用 `State.core.arming` 表达准备时间。
   - 使用 on-contact 或近距触发效果；若缺少触发入口，登记协议缺口。
3. Squash
   - 需要近距目标搜索、跳跃延迟和落点范围伤害。
   - 优先复用现有 targeting/effect；不足时登记最小 effect 缺口。
4. Jalapeno
   - 行范围爆炸。
   - 若缺少整行目标筛选，补最小 effect/type 或登记缺口。
5. Chomper
   - 近距离吞噬 + 消化状态。
   - 消化期间禁用攻击应走 State/Controller，不写单植物脚本特判。
6. Tall-nut
   - 高生命值与较高 hitbox。
   - 若跳跃阻挡未有协议，登记“高度/越过交互”缺口。
7. Pumpkin
   - 复用或迁移 `archetype_pumpkin_cover`。
   - 通过 cover placement 表达覆盖防御。

需要新增或更新的验证：

- `plant_original_batch_b_validation`
- 即时爆炸 validation
- 延迟激活 validation
- 覆盖防御 validation
- 近战吞噬/消化 validation

批次验收：

- 即时植物不依赖 battle 特判即可触发。
- Potato Mine 的未激活/已激活状态可观测。
- Pumpkin 覆盖放置和承伤链路进入回归。
- 批量运行 `smoke,core` 通过。

## 九、验证策略

### 1. 单植物验证模板

每个单植物 validation 至少验证：

- 卡片可被加入手牌。
- 阳光足够时可放置。
- 阳光不足时被拒绝。
- 非法槽位被拒绝。
- 核心行为会产生预期事件。
- 代表目标会受到预期影响。

### 2. 批次验证模板

每个批次 validation 至少验证：

- 同批多个植物可共同存在。
- 同批代表行为可以在同一战斗中稳定触发。
- 不出现 protocol issue。
- 验证结果能通过 `validation_report.json` 判定，而不是依赖肉眼观察。

### 3. 回归命令

每个植物完成后建议运行相关单场景：

```powershell
& ".\tools\run_validation.ps1" -Scenario "res://scenes/validation/<scenario>.tres"
```

每个批次完成后至少运行：

```powershell
& ".\tools\run_all_validations.ps1" -Layers smoke,core
```

涉及扩展 effect、guardrail 或 mode 输入时运行：

```powershell
& ".\tools\run_all_validations.ps1"
```

## 十、风险与处理

### 1. 资源重复风险

风险：

- 当前已有 phase/demo 样板植物，与原版植物语义部分重叠。

处理：

- 能直接对齐原版的资源应迁移命名或补正式映射。
- 不能对齐的保留为 showcase/chaos/sample，不进入 original roster。

### 2. 协议膨胀风险

风险：

- 为每个复杂植物新增一次专属运行时逻辑。

处理：

- 相似行为必须沉淀到 mechanic、effect、placement 或 mode 能力。
- 单个植物不得成为新系统入口。

### 3. 验证滞后风险

风险：

- 先做大量资源，最后才补 validation，导致问题难定位。

处理：

- 每个植物必须先有单体验证，再进入批次验证。
- 每批完成即跑 `smoke,core`。

### 4. 原版表现误差风险

风险：

- 首轮占位表现与原版动画不一致。

处理：

- 只要机制验证通过即可进入首轮完成。
- reanim、音频、视觉 QA 进入后续表现层计划。

## 十一、完成定义

单植物完成定义：

- 有正式 `CombatArchetype`。
- 有正式 `CardDef`。
- 攻击或效果行为由 mechanic/effect 表达。
- 放置语义由 placement 表达。
- 有至少一个 validation。
- 不新增 legacy 作者入口。

批次完成定义：

- 批次内所有植物达到单植物完成定义。
- 批次 validation 通过。
- `tools/validation_scenarios.json` 已更新。
- `tools/formal_content_validation_map.json` 已更新。
- 相关协议缺口已记录。
- `smoke,core` 批量验证通过。

全量完成定义：

- 49 个标准植物均有正式资源。
- 五个批次 validation 全部通过。
- 所有可表达行为进入回归。
- 所有暂不可表达行为有缺口记录和 guardrail。
- 完整批量验证通过。
