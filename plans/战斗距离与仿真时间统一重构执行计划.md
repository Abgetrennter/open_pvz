# 战斗距离与仿真时间统一重构执行计划

- 状态：当前计划
- 目标：在不破坏现有验证基线的前提下，把战斗距离语义化与 100Hz 固定仿真时间合并推进，形成可迁移原版数值、可验证、可现代化 UI 的运行时基础。
- 关联设计：
  - [战斗距离与棋盘度量](../wiki/02-runtime-protocol/15-战斗距离与棋盘度量.md)
  - [帧率与仿真时间](../wiki/02-runtime-protocol/16-帧率与仿真时间.md)
- 历史草案归档：`plans/archive/runtime-metrics-time/`

---

## 执行判断

距离和时间不能完全拆开执行：

- 射程、半径、格子位置可以先语义化，因为它们只依赖 `BattlefieldMetrics`。
- 速度、移动和投射物位移必须等固定仿真时间落地后再批量迁移，因为 `*_slots_per_sec` 的 `per_sec` 必须指 simulation second。
- `game.tick` 从 4Hz 升级到 100Hz 后，离散事件和连续行为必须同 tick 推进，不能只多发事件而不推进实体 movement。

因此实际执行顺序是：

> 先冻结口径和观测点，再落 `BattlefieldMetrics` 与 `SimulationClock`，随后统一 gameplay step，最后迁移内容字段和 guardrail。

---

## 阶段 0：口径冻结与基线审计

目标：不改行为，先把时间与距离的 legacy 现状固定下来。

任务：

- 扫描并记录 legacy 距离字段：
  - `scan_range`
  - `impact_radius`
  - `collision_padding`
  - `speed`
  - `move_speed`
  - `detection_range`
  - `board_slot_spacing`
- 扫描并记录 legacy 时间入口：
  - `BattleManager.tick_interval`
  - `GameState.advance_time(delta)`
  - `Time.get_ticks_msec()` / `Time.get_ticks_usec()`
  - `EffectDelayRunner`
  - `SunCollectible.auto_collect_delay`
  - validation deadline / auto quit
- 选定代表验证样本：
  - board placement
  - 基础射击链
  - proximity 触发
  - 波次刷新
  - 卡片冷却
  - 状态持续
  - 投射物命中
  - 割草机 sweep

退出信号：

- 不改变任何运行时行为。
- 当前裸 world unit 和可变时间入口有清单。
- 后续迁移样本和验证入口明确。

## 阶段 1：落 BattlefieldMetrics 最小对象

目标：集中棋盘几何换算，但保持现有行为一致。

建议改动：

- 新增 `scripts/battle/battlefield_metrics.gd`。
- 从 `BattlefieldPreset` / scenario 派生：
  - `slot_origin_x`
  - `slot_spacing`
  - `lane_y_positions`
  - `playable_min_x`
  - `playable_max_x`
- 提供最小查询：
  - `slot_position(lane_id, slot_index)`
  - `slots_to_world(slot_count)`
  - `world_to_slot_index(x)`
- `BattleBoardState.setup()` 创建 metrics。
- `BattleBoardState.get_slot_world_position()` 委托 metrics。
- `BattleManager.get_lane_y()` 迁移期继续可用，后续再反向委托 metrics。

不做：

- 不接管放置合法性。
- 不接管 detection。
- 不处理投射物命中。
- 不读写 `GameState`。

验证：

- board placement 相关验证通过。
- 新增或复用 metrics smoke：slot origin/spacing 改变后，slot 位置按预期变化。
- runtime snapshot 或 validation report 能看到 metrics 基础参数。

退出信号：

- metrics 接入前后行为一致。
- 棋盘几何解释不再散落在 `BattleManager` / `BattleBoardState` 裸字段中。

---

## 阶段 2：落 SimulationClock / GameState 时间字段

目标：新增固定仿真时间事实，但暂不大改 `game.tick`。

建议改动：

- 优先扩展 `autoload/GameState.gd`：
  - `simulation_tick_hz := 100`
  - `fixed_dt := 0.01`
  - `current_tick := 0`
  - `simulation_speed := 1.0`
  - `is_simulation_paused := false`
- 新增方法：
  - `reset_simulation_time()`
  - `advance_simulation_tick()`
  - `get_simulation_snapshot()`
- `current_time` 由 `current_tick * fixed_dt` 派生。
- 迁移期保留 `advance_time(delta)`，但标记为 legacy 或仅用于兼容。

不做：

- 不立即把 `game.tick` 改成 100Hz。
- 不立即修改所有 `_physics_process(delta)`。
- 不引入完整 replay。

验证：

- clock smoke：
  - reset 后 `current_tick = 0`
  - advance 1 tick 后 `current_time = 0.01`
  - advance 100 tick 后 `current_time = 1.0`
- 现有 smoke/core validation 结果不变。

退出信号：

- 权威 tick 字段稳定存在。
- debug snapshot 能观察 tick/time。

---

## 阶段 3：建立中央 gameplay step

目标：让离散事件和连续行为可以按每个仿真 tick 同步执行。

建议改动：

- 引入中央 runtime driver 或在 `BattleManager` 中收敛 step 调度。
- 定义 `step_simulation_tick()`，单 tick 内按固定顺序执行：
  - 输入请求消费
  - `GameState.advance_simulation_tick()`
  - `game.tick` 发射
  - controller / movement / projectile / field object step
  - 状态和延迟队列处理
  - validation 状态更新
- 明确 Godot `_physics_process` 只作为 1x 过渡 tick source。
- 在支持 2x、pause、单步前，实体、投射物、controller 不能继续只依赖各自 `_physics_process` 回调次数。

关键约束：

- 不允许只让 `BattleManager` 多发 `game.tick`，而连续运动仍按 Godot 回调频率运行。
- `fixed_dt` 是 gameplay step 的唯一 delta。
- 单 tick 内事件顺序保持同步和可观察。

验证：

- 1x 下行为与阶段 2 基线等价。
- 手动推进 N tick 后，事件数和位移符合预期。
- 投射物、僵尸移动、割草机 sweep 与 `fixed_dt` 对齐。

退出信号：

- gameplay 连续行为可由中央 step 驱动。
- 后续 2x、pause、单步具备实现基础。

---

## 阶段 4：`game.tick` 升级为 100Hz 权威事件

目标：让 `game.tick` 成为每个仿真 tick 的权威事件。

建议改动：

- 将 `game.tick` 从 `_process(delta)` 的 `tick_interval` 累积器迁出。
- 项目 physics tick 设为 100Hz，作为 1x 交互运行的外部节拍。
- 每个 tick event 携带：

```gdscript
tick_event.core["tick_index"] = GameState.current_tick
tick_event.core["fixed_dt"] = GameState.fixed_dt
tick_event.core["game_time"] = GameState.current_time
```

兼容策略：

- `tick_interval = 0.25` 视为 legacy 字段，不再作为 `game.tick` 间隔。
- 如果需要旧 4Hz subsystem 行为，新增明确 subsystem interval。
- 更新只依赖 tick 事件次数的 validation。

验证：

- `periodically` / `proximity` interval 行为稳定。
- 波次、卡片、状态、资源、validation deadline 读 simulation time。
- 无明显 event storm 或 detection 性能回退。

退出信号：

- `game.tick` 100Hz 稳定。
- 核心验证通过。

---

## 阶段 5：迁移规则时间消费者

目标：清理会影响玩法结果的可变 delta 和墙钟时间。

优先迁移：

- `TriggerRegistry` interval / start_delay。
- `ControllerRegistry` 中的墙钟时间。
- Projectile lifetime / movement。
- Entity movement / attack cooldown。
- `EffectDelayRunner`、延迟投射物、延迟爆炸、burst interval。
- `SunCollectible.auto_collect_delay`。
- validation deadline。

保留显示层时间：

- debug overlay 刷新。
- UI hover / drag。
- card bar 视觉 overlay。
- visual feedback、particles、Tween。
- 纯表现死亡淡出。

验证：

- 改变渲染 FPS 不改变玩法事件顺序和最终状态。
- pause 后规则事件停止，UI 可继续响应。
- 延迟效果和 auto collect 在固定 tick 下稳定触发。

退出信号：

- gameplay 结果不依赖 `_process(delta)`。
- 墙钟时间不再决定战斗规则。

---

## 阶段 6：接入距离语义字段

目标：让新内容可使用 slot 语义表达距离和速度。

分两批：

第一批只依赖 metrics：

- `scan_range_slots`
- `impact_radius_slots`
- `collision_padding_slots`
- `distance_slots`
- `detection_range_slots`
- `range_mode = full_lane`

第二批依赖 metrics + fixed tick：

- `speed_slots_per_sec`
- `move_speed_slots_per_sec`

编译策略：

- 语义字段优先，legacy 字段回退。
- `full_lane` 由 detection / targeting 结合 playable edge 解释。
- 不用 `99999.0` / `4000.0` 模拟整行射程。
- `per_sec` 一律指 simulation second。

验证：

- 改变 `slot_spacing` 后，射程/半径 world 值按比例变化。
- 固定 N tick 下，速度位移稳定等于：

```text
speed_slots_per_sec * elapsed_simulation_seconds * slot_spacing
```

退出信号：

- 新内容可以不写裸 `scan_range` / `speed`。
- legacy 资源行为保持不变。

---

## 阶段 7：样本内容迁移

目标：用代表内容验证语义字段和固定仿真时间适合实际玩法。

优先迁移：

- 整行射手：豌豆射手、寒冰射手、双发射手。
- 短射程植物：小喷菇、海蘑菇、大喷菇。
- 近身触发植物：土豆雷、魅惑菇、缠绕海草。
- 投射物 profile：`linear_ground`、`linear_fast_swept`、`parabola_arc`、`parabola_terminal_blast`。
- 移动实体：普通僵尸、快速僵尸、割草机。

规则：

- 一次只迁移一组语义相近内容。
- 不混入额外平衡调整。
- 同步更新迁移记录和 validation。

验证：

- 基础射击命中。
- 短射程边界。
- 近身触发边界。
- 抛物线终点半径。
- 僵尸到达、啃咬和割草机扫线。

退出信号：

- 样本内容验证通过。
- 原版 tick / slot 映射口径可用于后续迁移。

---

## 阶段 8：倍速、暂停、单步与 guardrail

目标：完成时间协议能力，并防止新代码重新扩散 legacy 时间/距离口径。

能力：

- `simulation_speed`
- `simulation_paused`
- `step_simulation_ticks(count)`
- headless 固定步进验证。

guardrail：

- 扫描 gameplay 中新增 `Time.get_ticks_msec()` / `Time.get_ticks_usec()`。
- 扫描 battle/runtime 中新增规则 Timer。
- 扫描新增 `.tres` 中裸 `scan_range` / `speed`。
- ProtocolValidator 对负数语义距离、不合理 interval 做提示。
- DebugService 记录 tick overrun、max step clamp、pause/speed change。

退出信号：

- 2x、pause、单步可被 validation 观察。
- 新 gameplay 默认使用 simulation time 和 slot 语义。
- 文档、验证和工具能发现典型误用。

---

## 总完成定义

完成后应满足：

1. 战斗规则由 100Hz 固定仿真 tick 推进。
2. 战斗距离以 slot 语义表达，再由 `BattlefieldMetrics` 转 world unit。
3. 渲染 FPS、VSync、窗口大小、UI 缩放不影响玩法结果。
4. 原版 tick 可直通，原版 px 先映射为 slot。
5. gameplay 连续行为、延迟效果、可收集物延迟都使用 simulation time。
6. 新内容默认使用语义距离字段，legacy 字段仅作兼容或明确 world unit 用途。
7. 批量验证能覆盖时间、距离、速度和边界样本。
