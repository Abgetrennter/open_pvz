# 第二阶段可执行任务清单

> 这份清单把“第二阶段（内容模板期）”进一步拆成可直接开工的 issue/checklist。它不代替路线图，而是把路线图里的第二阶段目标下沉成一组可以排期、认领、验收的工作项。

---

## 文档定位

这篇文档主要回答：

- 第二阶段现在可以直接开干的任务有哪些
- 每项任务的产出物是什么
- 每项任务依赖什么前置条件
- 每项任务完成后如何判定“算做完”

这篇文档不负责：

- 代替阶段路线图
- 冻结第三阶段协议细节
- 直接决定第四阶段玩法系统顺序

配套阅读建议：

- [当前阶段与实现路线](../wiki/01-overview/23-当前阶段与实现路线.md)
- [开发路线图](../wiki/04-roadmap-reference/26-开发路线图.md)
- [验证矩阵](../wiki/03-content-validation/32-验证矩阵.md)
- [术语表](../wiki/05-governance/33-术语表.md)

---

## 使用方式

建议把下面的条目按两层理解：

- `Epic`：一组需要连续推进的主任务
- `Issue`：可直接开工、可单独验收的任务

如果当前人力有限，推荐先按下面顺序推进：

1. `Epic A` 模板边界与资源层补齐
2. `Epic B` 场景生成链收口
3. `Epic C` 内容样例库首批建设
4. `Epic D` 模板专项验证扩容
5. `Epic E` 文档、目录和作者工作流整理

---

## Epic A：模板边界与资源层补齐

### Issue A1：补一版模板边界说明

- 目标：
  - 明确哪些字段属于 `EntityTemplate`
  - 明确哪些字段应该从 `BattleSpawnEntry` 回收
  - 明确“投射体飞行配置”和“投射物模板”边界
- 产出物：
  - 一份边界说明文档
  - 对应术语和路线图引用更新
- 依赖：
  - 无
- 完成判定：
  - 文档中至少覆盖 `EntityTemplate / BattleSpawnEntry / ProjectileFlightProfile / 投射物模板`
  - 后续新增模板时不需要再重复口头解释“字段该写哪”
- checklist：
  - [x] 梳理当前 `EntityTemplate` 已有字段
  - [x] 梳理当前 `BattleSpawnEntry` 已有字段
  - [x] 标出模板长期字段和场景覆盖字段
  - [x] 标出第二阶段暂不引入的字段类型

### Issue A2：给 `EntityTemplate` 补第一版字段分层

- 目标：
  - 把当前模板从“最小可用”推进到“正式内容入口”
- 产出物：
  - 更新后的 `EntityTemplate`
  - 至少 1 份字段说明文档
- 依赖：
  - `Issue A1`
- 完成判定：
  - 模板字段能区分身份层、节点层、战斗层、投射层、行为层
  - 不再把所有内容默认堆进 `default_params`
- checklist：
  - [x] 增补 `tags` 或等价分类字段
  - [x] 评估是否需要“必需组件 / 可选组件”表达
  - [x] 评估表现占位字段是否需要模板入口
  - [x] 更新协议校验和文档

### Issue A3：引入轻量 `ProjectileTemplateResource`

- 目标：
  - 把“谁发射”和“发射出去的东西是什么”分开
- 产出物：
  - `ProjectileTemplateResource` 或等价资源脚本
  - 至少 2 份投射物模板样例
- 依赖：
  - `Issue A1`
  - `Issue A2`
- 完成判定：
  - 至少一类投射体差异不再只能靠 `EntityTemplate.default_params` 表达
  - 模板链可以引用投射物模板而不是只引用飞行配置
- checklist：
  - [x] 定义最小字段集合
  - [x] 明确与 `ProjectileFlightProfile` 的边界
  - [x] 接入 `EntityFactory` 或等价装配链
  - [x] 补 2 份样例资源

### Issue A4：引入轻量行为绑定资源

- 目标：
  - 减少 `runtime_behavior` 无限膨胀成任意字典
- 产出物：
  - `TriggerBindingResource` 或等价资源
  - 至少 1 类行为从裸字典迁移到资源化入口
- 依赖：
  - `Issue A2`
- 完成判定：
  - 至少一类模板行为不再只能靠匿名字典维护
  - 文档中明确写清楚“哪些行为仍允许继续放在 `runtime_behavior`”
- checklist：
  - [x] 选定第一批行为块
  - [x] 定义资源字段
  - [x] 接到工厂或运行时构建链
  - [x] 补文档和样例

## Epic B：场景生成链收口

### Issue B1：收紧 `BattleSpawnEntry` 的职责

- 目标：
  - 让场景条目变成“模板实例化条目”，而不是一份大参数表
- 产出物：
  - 更新后的 `BattleSpawnEntry`
  - 对应协议校验更新
- 依赖：
  - `Issue A1`
- 完成判定：
  - `BattleSpawnEntry` 主要只保留模板引用、位置、lane 和少量覆盖项
  - 明显属于模板长期定义的字段已回收到模板
- checklist：
  - [x] 列出当前字段的归属判断
  - [x] 标记应废弃或后移的覆盖字段
  - [x] 更新 `ProtocolValidator`
  - [x] 更新相关 wiki

### Issue B2：明确模板默认值与场景覆盖优先级

- 目标：
  - 防止后续模板链出现“同一个参数到底谁生效”的混乱
- 产出物：
  - 一份覆盖优先级规则
  - 对应实现和验证
- 依赖：
  - `Issue B1`
- 完成判定：
  - 至少明确三层来源：
    - 模板默认值
    - spawn entry 覆盖值
    - 运行时临时值
  - 验证场景可以证明优先级符合预期
- checklist：
  - [x] 梳理当前参数合并顺序
  - [x] 明确保留覆盖的字段名单
  - [x] 补一个优先级专项验证
  - [x] 更新文档

### Issue B3：建立模板/场景轻量注册入口

- 目标：
  - 避免内容越来越多后继续散落路径引用
- 产出物：
  - `SceneRegistry`、模板注册表或等价入口
- 依赖：
  - `Issue B1`
- 完成判定：
  - 至少验证场景和一类模板资源可以通过注册表统一索引
  - gameplay 脚本里散落的路径引用开始收敛
- checklist：
  - [x] 选定注册入口位置
  - [x] 接入至少一类模板
  - [x] 接入至少一类验证场景
  - [x] 更新使用说明

### Issue B4：继续剥离 `BattleManager` 的内容装配职责

- 目标：
  - 让 `BattleManager` 只做运行、挂接、验证，不再继续长内容特判
- 产出物：
  - 精简后的 `BattleManager`
  - 对应回归验证
- 依赖：
  - `Issue A2`
  - `Issue B1`
  - `Issue B2`
- 完成判定：
  - 新增一个基础单位无需修改 `BattleManager`
  - 内容差异主要来自模板和工厂
- checklist：
  - [x] 标出 `BattleManager` 里剩余内容耦合点
  - [x] 把能下沉的逻辑迁到模板/工厂
  - [x] 跑相关验证场景
  - [x] 更新路线图/阶段判断

## 最小可交付批次建议

如果只先做一轮“最小可交付”，建议以这 8 个 issue 为第一批：

- [ ] `Issue A1` 补一版模板边界说明
- [ ] `Issue A2` 给 `EntityTemplate` 补第一版字段分层
- [ ] `Issue B1` 收紧 `BattleSpawnEntry` 的职责
- [ ] `Issue B2` 明确模板默认值与场景覆盖优先级
- [ ] `Issue C1` 补齐高度带样例库
- [ ] `Issue C2` 补齐投射体轨迹样例库
- [ ] `Issue C3` 补第一批植物模板样例
- [ ] `Issue D1` 为内容样例补正向专项验证

完成这 8 项后，第二阶段就会从“路线图存在”推进到“内容模板期真的开始成形”。

---

## 完成定义

第二阶段不要求把所有模板系统一次性写满，但至少应满足：

- 新增基础单位主要通过模板完成
- 场景主要摆模板实例，而不是重写参数表
- 已有模板样例能覆盖多种投射体和反应行为
- 第二阶段新增内容已进入批量回归
- 第三阶段协议冻结已经有足够真实输入，而不是空想字段
