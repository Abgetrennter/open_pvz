## 当前结论

当前 `tools/validation_scenarios.json` 已覆盖 52 个唯一验证场景 ID，并已为每个条目增加 `layers` 字段。

`layers` 是数组而不是单值，原因是：

- `smoke` 不是一个独立能力域，而是一组快速基线
- 某些场景同时属于 `smoke + core` 或 `smoke + guardrail`
- 后续如果要增加 `perf`、`visual` 等层级，可以兼容扩展

---

## 四层定义

| Layer | 目标 | 典型内容 | 建议频率 |
|------|------|------|------|
| `smoke` | 快速判断主干是否明显断裂 | 最小战斗、模板工厂、资源/棋盘/波次、扩展 smoke、关键守卫 | 每次较小改动后优先跑 |
| `core` | 覆盖主仓内建运行时和玩法主链 | 投射体、模板、玩法层、内建错误技样例 | 合并前或大改后跑 |
| `extension` | 覆盖外部扩展内容和扩展 effect | 扩展包 smoke、9 组扩展 effect 正向样例 | 改扩展入口或 effect 后跑 |
| `guardrail` | 覆盖负向协议和非法配置 | 协议、模板、波次、manifest、effect 守卫 | 改协议、注册、校验逻辑后跑 |

## 常用命令

### 1. 快速 smoke

```powershell
& ".\tools\run_all_validations.ps1" -Layers smoke
```

适用场景：

- 小改动后快速确认主干没断
- 文档外的轻量代码调整
- 进入更大回归前的第一道门

### 2. 主干 core

```powershell
& ".\tools\run_all_validations.ps1" -Layers core
```

适用场景：

- 修改运行时、模板、投射体、玩法层
- 修改 `BattleManager` 或核心 Autoload
- 合并前确认主仓内建能力稳定

### 3. 扩展 extension

```powershell
& ".\tools\run_all_validations.ps1" -Layers extension
```

适用场景：

- 修改扩展包扫描
- 修改 `EffectRegistry`
- 新增或调整扩展 effect

### 4. 守卫 guardrail

```powershell
& ".\tools\run_all_validations.ps1" -Layers guardrail
```

适用场景：

- 修改 `ProtocolValidator`
- 修改 manifest 校验
- 修改注册表冲突处理
- 修改非法配置拦截路径

### 5. 全量回归

```powershell
& ".\tools\run_all_validations.ps1"
```

适用场景：

- 大范围重构前后
- 阶段收口
- 准备发布或打标签前

## 新增能力接入规则

以后每个新增能力都必须先回答下面问题：

1. 它属于哪个能力域？
2. 它至少进入哪个 layer？
3. 是否需要加入 `smoke`？
4. 是否需要一个负向守卫？
5. 是否需要更新 Showcase？
6. 是否需要更新内容工作流？

### 最小门槛

新增能力进入主线前至少需要：

- [ ] 1 个正向验证场景
- [ ] 1 条 `tools/validation_scenarios.json` 记录
- [ ] 1 个明确的 `layers` 标记
- [ ] 1 行验证矩阵说明

新增协议边界进入主线前至少需要：

- [ ] 1 个正向验证场景
- [ ] 1 个负向守卫场景
- [ ] 失败时进入 `protocol.issue` 或等价守卫路径
- [ ] 文档说明新增字段或新增入口为什么需要

新增扩展入口进入主线前至少需要：

- [ ] manifest 或注册边界说明
- [ ] 正向扩展样例
- [ ] 负向扩展守卫
- [ ] 内容工作流说明

## 分层维护规则

### 1. `smoke` 应保持小而代表性强

`smoke` 不是“所有重要场景”，而是最快速的代表性切片。

新增场景只有满足下面条件之一才建议进入 `smoke`：

- 能代表一个大系统的最小正向路径
- 能代表一个最容易断的协议守卫
- 能代表扩展包入口是否还可用

### 2. `core` 应覆盖主仓正式能力

主仓内建运行时和玩法层能力，默认进入 `core`。

如果一个能力只存在于扩展包中，不应放入 `core`。

### 3. `extension` 只覆盖扩展包路径

凡是依赖 `extensions/*` 中资源、effect 或 validation 场景的条目，默认进入 `extension`。

扩展包 smoke test 可以同时进入 `smoke + extension`。

### 4. `guardrail` 专门覆盖负向行为

非法配置、协议拒绝、manifest 失败、注册冲突、schema 错误等，默认进入 `guardrail`。

如果一个守卫代表非常关键的最小防线，可以同时进入 `smoke + guardrail`。
