# 第五阶段-主干编排收口计划

> 这份文档用于完成 `第五阶段-v1规范化-4到8周执行路线图` 中的 `Epic E：主干编排收口与架构减压`。它不追求大重构，而是回答：当前 `BattleManager` 到底承担了什么、哪些职责应保留、哪些职责值得在下一轮演进中下沉、以及未来新增能力该从哪里进入。

---

## 文档定位

这篇文档主要回答：

- `BattleManager` 当前职责有哪些
- 哪些职责属于合理编排层
- 哪些职责已经接近或跨过编排边界
- 第一批最值得收口的目标是什么
- 新增能力应该如何选择入口

这篇文档不负责：

- 直接发起大范围代码重构
- 推翻现有 Godot 节点 + 组件 + Autoload 路线
- 把所有运行时逻辑拆成过细 service
- 解决所有长期架构问题

---

## 当前判断

`BattleManager` 当前已经同时承担了：

1. 战斗主循环
2. 场景解析与实体生成编排
3. 验证状态机
4. 调试与导出辅助逻辑
5. 一部分投射体目标解析与参数组装

这在当前阶段是可运行的，但继续增长下去会带来两个风险：

- 新需求更容易直接堆进 manager
- 验证、导出、投射体辅助、CLI 选项等非核心编排逻辑会继续膨胀

因此当前最合理的策略不是“大范围重构”，而是：

> 先定义边界、锁定第一批收口目标、建立新增能力入场守则，并用低风险代码拆分验证边界可执行，避免后续两个月继续无约束堆积。

---

## 当前职责盘点

### A. 合理保留的编排职责

这些职责仍应由 `BattleManager` 保留：

- 战斗生命周期：
  - `_ready`
  - `_exit_tree`
  - `reset_battle`
- 主循环与 tick：
  - `_process`
  - `_physics_process`
- 当前场景解析与生成入口：
  - `_resolve_scenario`
  - `_spawn_scenario`
  - `_spawn_entry`
  - `_spawn_entry_internal`
- 子系统挂接：
  - `_reset_runtime_services`
  - `_bind_runtime_triggers`
- 对外统一战斗接口：
  - `get_runtime_entities`
  - `try_spend_sun`
  - `spawn_card_entity`
  - `spawn_wave_entry`

当前判断：

- 这些职责直接决定“这一局战斗如何开始、推进和结束”
- 它们属于典型编排层职责
- 当前不建议拆散

### B. 适合逐步下沉的职责

下面职责当前还在 `BattleManager` 内，但更像“辅助能力”：

- 投射体目标解析与预测：
  - `_resolve_projectile_target_node`
  - `_resolve_projectile_target_position`
  - `_predict_target_position`
  - `_estimate_entity_velocity`
  - `_estimate_target_tracking_budget`
- 投射体 effect 参数归并：
  - `_resolve_projectile_effect_params`
  - `_build_projectile_movement_params`
  - `_movement_params_from_flight_profile`
- 验证导出与报告：
  - `_report_validation_result`
  - `_export_validation_artifacts`
  - `_build_validation_report`
- CLI 运行时选项解析：
  - `_apply_runtime_options`

当前判断：

- 它们确实支撑当前运行时
- 但不一定应该长期留在总编排器里
- 这些职责更像：
  - `ProjectileSpawnResolver`
  - `ValidationArtifactExporter`
  - `BattleRuntimeOptions`
  - 或等价轻量 service

### C. 高风险膨胀点

下面区域最容易继续被塞需求：

- `spawn_projectile_from_effect`
- `spawn_entity_from_effect`
- 协议 issue 汇报与导出
- lane 和 board 位置辅助
- 调试输出与 runtime snapshot

这些区域的共同特征是：

- 很容易被“先加个参数、先补个特判”侵入
- 表面上还在 `BattleManager` 里，看起来方便
- 长期会让 manager 既像 orchestrator，又像 util 集合

## 第一批收口目标

当前 4~8 周窗口内，建议只处理 2 个高价值目标。

### 目标 1：验证报告与导出逻辑外提

优先理由：

- 当前验证已经是主干能力，而不是附属功能
- 导出、summary、JSON 拼装不应继续挤在编排器里
- 这部分改动相对独立，回归风险可控

建议归宿：

- `ValidationArtifactExporter`
- 或 `BattleValidationReporter`

本轮代码级收口已经实现第一步拆分：

- 新增 `BattleValidationReporter`
- `BattleManager` 只保留验证完成时机和状态机
- 验证报告构建、导出目录解析、JSON/text 写入已迁出

后续如果继续深入，应保持：

- 输入是什么
- 输出是什么
- manager 只保留“触发导出”的职责

### 目标 2：投射体 effect 参数组装与目标预测逻辑外提

优先理由：

- 这是当前最容易继续膨胀的数学/组装逻辑区
- 未来新增投射体模式或新 effect 最容易继续堆在这里
- 这部分职责天然可复用，也最像 service

建议归宿：

- `ProjectileEffectResolver`
- 或 `ProjectileTargetingResolver`

本轮代码级收口已经实现第一步拆分：

- 新增 `BattleProjectileEffectResolver`
- 投射体模板默认参数归并已迁出
- 投射体 movement params 组装已迁出
- 抛物线目标预测、追踪目标解析和速度估算已迁出

后续如果继续深入，应保持：

- 组装输入参数的边界
- 预测逻辑和战斗编排的边界
- manager 只保留“调用解析器并挂接结果”的职责

## 当前明确不做的事

为了避免架构收口变成大重构，当前明确不做：

1. 不拆散当前战斗生命周期主链
2. 不把所有函数都强行下沉
3. 不把现有 Phase 4 子系统再切成更细 service
4. 不为了“更干净”而牺牲当前验证稳定性
5. 不在没有回归基线的情况下大规模搬迁方法

## 新增能力入场守则

以后新增能力时，先按下面顺序判断入口。

### 1. 它是否属于编排层？

满足下面任意一条，优先进入编排层：

- 直接决定战斗生命周期
- 直接决定场景解析和生成时机
- 直接决定子系统装配顺序
- 只是把已有 service 串起来

如果是，允许进入 `BattleManager`。

### 2. 它是否属于可复用运行时能力？

满足下面任意一条，优先进入 runtime service：

- 多个系统都会用到
- 更像参数解析、目标解析、导出、计算或路由
- 不依赖完整战斗生命周期
- 即使离开 `BattleManager` 也能独立测试或复用

如果是，不应继续直接堆进 `BattleManager`。

### 3. 它是否属于内容表达差异？

满足下面任意一条，优先作为扩展能力接入：

- 只是新 effect 原子
- 只是新模板组合
- 只是新错误技样例
- 不需要改主干编排

如果是，优先进入：

- 模板
- 扩展包
- validation scene

### 4. 它是否只是一次性验证样例？

如果它只是：

- 暂时性的验证拼装
- 不值得长期维护的演示逻辑

那么应优先留在：

- validation scene
- showcase 包装层

而不是进入主干运行时。
