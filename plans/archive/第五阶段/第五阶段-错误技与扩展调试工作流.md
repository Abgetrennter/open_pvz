# 第五阶段-错误技与扩展调试工作流

> 这份工作流把第五阶段最常见的三类问题整理成固定排查顺序：错误技组合问题、扩展包加载问题、战局流程问题。

---

## 先看什么

当前推荐先看 4 个面：

1. 调试 overlay
2. `debug_logs.json`
3. `validation_report.json`
4. `godot.log`

---

## 当前 overlay 观察点

当前 overlay 已补齐下面几类信息：

- 当前资源：
  - `Sun`
  - 活跃阳光数量
- 当前战局阶段：
  - `Phase`
  - 当前激活波次
  - 已完成波次数
- slot 角色占位：
  - `Slot Occupancy`
- 活跃状态：
  - `Active Statuses`
- 最近事件、触发器、效果：
  - `Recent Events`
  - `Recent Triggers`
  - `Recent Effects`

这让第五阶段排错时不再只靠事件日志猜状态。

## 错误技组合问题排查顺序

1. 先看 `validation_report.json`
   - 哪条规则没满足
2. 再看 `debug_logs.json`
   - 有没有 `protocol.issue`
   - 触发器是否 `fire`
   - 效果是否执行
3. 再看 overlay
   - 目标是否存在
   - 状态是否阻断攻击
   - slot/phase 是否正确
4. 最后看 runtime snapshot
   - 投射体 `profile_id / move_mode / target_id`
   - 实体 `active_statuses / health / lane`

---

## 扩展包加载问题排查顺序

1. 先确认目录结构是否符合：
   - `extensions/<pack>/data/combat/...`
   - `extensions/<pack>/scenes/validation/...`
2. 再确认资源 ID 是否唯一
   - 重复 ID 当前会被拒绝并记入 `protocol.issue`
3. 再确认场景是否通过 `entity_template_id` 解析到扩展模板
4. 最后跑 `extension_pack_smoke_validation`

常见故障优先级：

1. 路径放错
2. `template_id / binding_id / scenario_id` 为空
3. 与主仓 ID 冲突
4. 资源脚本类型不匹配

---

## 战局流程问题排查顺序

1. 看 overlay 的 `Phase / wave`
2. 看 `battle.phase_changed / wave.started / wave.completed`
3. 看 `WaveRunner` 快照：
   - `started_wave_count`
   - `completed_wave_count`
4. 再看实体是否提前死亡或越线

## 输出物位置

单场景验证默认输出到：

- `artifacts/validation/<timestamp>_<label>/validation_report.json`
- `artifacts/validation/<timestamp>_<label>/validation_summary.txt`
- `artifacts/validation/<timestamp>_<label>/debug_logs.json`
- `artifacts/validation/<timestamp>_<label>/godot.log`

批量验证汇总输出到：

- `artifacts/validation/latest_summary.json`
- `artifacts/validation/regression_status.json`
- `artifacts/validation/regression_status.md`

分层回归额外可见：

- `requested_layers`
- 每个场景的 `layers`
- summary 中的 `LayerGroup`

## 给新协作者的最短建议

如果你刚接手第五阶段问题，先做下面 3 步：

1. 跑对应验证场景
2. 看 `validation_report.json` 和 `debug_logs.json`
3. 再回到 overlay 和 runtime snapshot 对照

不要一开始就去改 `BattleManager`。
