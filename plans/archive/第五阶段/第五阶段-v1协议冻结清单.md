## 当前判断

当前项目已经具备继续讨论 `v1` 冻结的现实基础：

1. 主干运行时已经稳定承载大量正向样例和负向守卫。
2. 模板主链、第四阶段玩法层和第五阶段错误技样例都已进入正式回归。
3. 扩展包与扩展 effect 已不再只是概念，而是已经有：
   - `minimal_chaos_pack`
   - `phase5_chaos_pack`
   - `phase5_guardrail_pack`
4. 现阶段最大的风险已经不是“没有能力”，而是：
   - 字段继续自由漂移
   - manifest 口径不统一
   - 新增扩展入口没有同步守卫

因此这份清单的任务是：

> 先冻结“已经被大量样例和验证依赖的边界”，再给仍处在扩张期的内容保留受控试验空间。

## 使用方式

建议把下面的边界按三层理解：

- `冻结候选`：已经应优先保持兼容
- `受控扩展`：仍允许新增，但必须同步守卫
- `后置决定`：当前不应在 `v1` 阶段提前锁死

## A. 核心协议冻结候选

### 1. 事件与上下文语义

下面内容进入 `v1` 冻结候选：

- 事件总线作为统一运行时事件入口
- `protocol.issue`
- `projectile.spawned / projectile.hit / projectile.expired`
- `entity.damaged / entity.died`
- `placement.accepted / placement.rejected`
- `resource.changed / resource.spend_failed`
- `battle.phase_changed / wave.started / wave.completed`
- `entity.status_applied / entity.status_removed`
- `field_object.spawned / field_object.activated / field_object.expired`

下面字段语义进入 `v1` 冻结候选：

- `RuleContext.source_node`
- `RuleContext.target_node`
- `RuleContext.owner_entity`
- `RuleContext.event_name`
- `RuleContext.chain_id`
- `RuleContext.depth`
- `RuleContext.runtime`
- `EventData.core.scope`
- `EventData.core.severity`
- `EventData.core.tags`

当前结论：

- 可以继续加新事件
- 不能随意改已有核心事件的语义
- 新协议事件必须同步补矩阵和守卫

### 2. 模板主链

下面资源主链进入 `v1` 冻结候选：

- `EntityTemplate`
- `ProjectileTemplate`
- `TriggerBinding`
- `BattleSpawnEntry`

当前应优先保持兼容的语义包括：

- `template_id / binding_id / scenario_id` 作为注册表主身份
- `TriggerBinding.behavior_key / trigger_id / event_name` 对应关系
- `ProjectileTemplate.flight_profile`
- `BattleSpawnEntry.entity_template_id` 与模板解析链
- `BattleSpawnEntry.spawn_overrides` 的白名单约束

当前结论：

- 模板链允许继续补字段
- 但新增字段必须先回答“属于模板本体、覆盖入口还是运行时上下文”
- 不允许把业务特判重新塞回模板链伪字段

### 3. 基础 effect / trigger 协议

下面 effect / trigger 协议进入 `v1` 冻结候选：

- `EffectDef.effect_id`
- `EffectDef.param_defs`
- `EffectDef.slots`
- `EffectDef.tags`
- `EffectDef.strategy_script`
- `TriggerDef.trigger_id`
- `TriggerDef.event_name`
- `TriggerDef.condition_params`

当前结论：

- `damage / spawn_projectile / explode` 已可视为第一版稳定基础 effect
- 新 effect 家族仍允许新增
- 但新增 effect 不能绕过：
  - schema 校验
  - slot 约束
  - strategy 注册
  - 正负向验证

## B. 扩展包 manifest v1 边界

### 1. 第一版 manifest 必填字段

当前第一版 manifest 至少应包含：

| 字段 | 类型 | 当前要求 | 说明 |
|------|------|------|------|
| `pack_id` | string | 必填，且不得为空 | 扩展包全局身份 |
| `enabled_by_default` | bool | 可选；若存在则必须是 bool | 控制默认激活 |
| `register` | array[string] | 必填，且必须是数组 | 声明扩展包注册的内容类型 |

### 2. 第一版 manifest 可选字段

当前第一版允许的可选字段包括：

| 字段 | 类型 | 当前要求 | 说明 |
|------|------|------|------|
| `activation_scenario_ids` | array[string] | 若存在必须是字符串数组 | 按验证场景激活 |
| `activation_cli_flags` | array[string] | 若存在必须是字符串数组 | 按命令行标志激活 |

### 3. `register` 第一版允许值

当前第一版只允许：

- `resources`
- `effects`

语义约束如下：

- `resources`：允许主仓扫描模板、投射体模板、绑定和验证场景
- `effects`：允许扫描扩展 `EffectDef` 以及对应 `strategy_script`

当前第一版明确不允许：

- 任意自定义 register kind
- 用 manifest 声明未受控的新运行时注入入口
- 用 manifest 覆盖主仓稳定 ID

### 4. manifest 当前不支持什么

当前第一版 manifest 明确不支持：

- 依赖解算
- 版本范围协商
- 热加载
- 远程下载与安装
- 运行时脚本注入权限声明
- 覆盖主仓正式内容

## C. 扩展注册边界

### 1. 当前允许注册的内容

当前第五阶段 `v1` 边界内正式允许注册：

- 验证场景
- `EntityTemplate`
- `ProjectileTemplate`
- `TriggerBinding`
- `EffectDef`
- 带 `execute(context, params, node)` 的扩展 effect `strategy_script`

### 2. 当前不允许注册的内容

当前第一版明确不允许：

- 任意脚本注入式运行时代码扩展
- 直接改写主仓执行器
- 不经注册表的旁路资源接入
- 无 manifest 的隐式扩展包
- 复用主仓稳定 ID 并尝试覆盖

### 3. 当前目录约定

`resources` 类扩展包当前约定目录：

```text
extensions/<pack_id>/
  data/combat/
    entity_templates/
    projectile_templates/
    trigger_bindings/
  scenes/validation/
```

`effects` 类扩展包当前约定目录：

```text
extensions/<pack_id>/
  data/combat/effects/
  scripts/effects/
```

当前结论：

- 目录约定进入 `v1` 冻结候选
- 未来如需新增目录，应先更新注册逻辑、文档和守卫，再允许进入主线

## D. 守卫覆盖清单

### 1. 当前正向基线

| 验证项 | 作用 |
|------|------|
| `extension_pack_smoke_validation` | 验证最小扩展包能扫描、注册并进入统一主链 |
| `hit_split_chaos_validation` | 验证扩展命中分裂 effect |
| `periodic_summon_chaos_validation` | 验证扩展召唤 effect |
| `apply_status_chaos_validation` | 验证扩展状态控制 effect |
| `knockback_chaos_validation` | 验证扩展位移 effect |
| `chain_bounce_chaos_validation` | 验证扩展跳链 effect |
| `aura_chaos_validation` | 验证扩展光环控制 effect |
| `delayed_trigger_chaos_validation` | 验证扩展延迟单体 effect |
| `delayed_explode_chaos_validation` | 验证扩展延迟范围 effect |
| `mark_chaos_validation` | 验证扩展独立标记生命周期 |

### 2. 当前负向守卫

| 验证项 | 当前覆盖 |
|------|------|
| `protocol_guardrail_validation` | 运行时协议非法组合、非法效果参数、非法 target_mode |
| `template_guardrail_validation` | 模板资源缺陷、错误 override 资源和不一致组合 |
| `extension_effect_guardrail_validation` | 重复 effect_id、缺失 effect_id、非法 schema、缺失 execute、slot 不匹配、坏参数类型、无 strategy |
| `extension_manifest_guardrail_validation` | 缺失 manifest、无效 JSON、错误 register 结构、不支持的 register kind、空 pack_id |

### 3. 当前守卫结论

从 `Epic B` 结束时开始，新增扩展入口要想进入主线，至少必须满足：

1. 有 manifest 或注册边界说明
2. 有正向验证
3. 有负向守卫
4. 会进入 `protocol.issue` 或等价守卫路径
5. 已接入 `tools/validation_scenarios.json`
