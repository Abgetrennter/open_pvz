# 第五阶段-v1内容工作流

> 这份文档用于完成 `第五阶段-v1规范化-4到8周执行路线图` 中的 `Epic D：内容工作流与脚手架`。它把“新增内置样例、扩展包样例、扩展 effect”三条路径整理成可直接执行的工作流，并给出最小扩展包脚手架。

---

## 文档定位

这篇文档主要回答：

- 新增一个内容样例应该改哪些目录
- 新增一个扩展包应该怎么起步
- 新增一个扩展 effect 应该补哪些资源和验证
- 命名、目录、验证和 Showcase 应如何同步
- 常见失败点如何排查

这篇文档不负责：

- 代替 `v1` 协议冻结清单
- 设计完整编辑器
- 承诺最终版扩展包发布流程
- 展开完整社区平台

## 当前结论

从 `v1` 规范化开始，新增内容不再只看“能不能跑”，而应同时满足：

1. 内容目录正确
2. 资源 ID 唯一
3. 验证场景可自动运行
4. `tools/validation_scenarios.json` 有明确 `layers`
5. 验证矩阵有对应行
6. 若涉及新协议边界，必须有负向守卫

---

## 路径 A：新增内置样例

适用场景：

- 样例验证主仓核心运行时
- 样例需要成为后续主干基线
- 样例不依赖外部扩展包

### 推荐目录

```text
data/combat/
  entity_templates/
  projectile_templates/
  trigger_bindings/
scenes/validation/
scenes/showcase/
```

### 执行步骤

1. 先写或复用 `EntityTemplate`
2. 如需要新投射体，再写 `ProjectileTemplate`
3. 如需要新行为，再写 `TriggerBinding`
4. 在 `scenes/validation/` 增加验证场景
5. 在 `tools/validation_scenarios.json` 增加条目，通常标记为 `core`
6. 如需要演示入口，再增加 `scenes/showcase/` 包装场景
7. 更新验证矩阵

### 完成判定

- 能通过 `run_validation.ps1`
- 能被 `run_all_validations.ps1 -Layers core` 覆盖
- 不向 `BattleManager` 添加样例专用特判

## 路径 B：新增扩展包样例

适用场景：

- 样例用于证明外部内容包边界
- 样例不应成为主仓内置内容
- 样例可以通过 `extensions/<pack_id>` 独立组织

### 推荐目录

```text
extensions/<pack_id>/
  extension.json
  data/combat/
    entity_templates/
    projectile_templates/
    trigger_bindings/
  scenes/validation/
```

### 执行步骤

1. 从 `tools/templates/extension_pack_v1/` 复制脚手架
2. 把目录改名为 `extensions/<pack_id>/`
3. 全局替换：
   - `__PACK_ID__`
   - `__PLANT_ID__`
   - `__BINDING_ID__`
   - `__SCENARIO_ID__`
4. 调整模板数值和验证目标
5. 在 `tools/validation_scenarios.json` 增加条目，通常标记为 `extension`
6. 如果它是最小代表路径，可以额外标记 `smoke`
7. 更新验证矩阵

### 最小脚手架

当前已提供：

- `tools/templates/extension_pack_v1/README.md`
- `tools/templates/extension_pack_v1/extension.json`
- `tools/templates/extension_pack_v1/data/combat/entity_templates/plants/plant_template_stub.tres`
- `tools/templates/extension_pack_v1/data/combat/trigger_bindings/plant_attack_template_stub.tres`
- `tools/templates/extension_pack_v1/scenes/validation/template_pack_smoke_validation.tres`

### 完成判定

- 能通过指定场景：

```powershell
& ".\tools\run_validation.ps1" -Scenario "res://extensions/<pack_id>/scenes/validation/<scenario_id>.tres"
```

- 能被分层回归覆盖：

```powershell
& ".\tools\run_all_validations.ps1" -Layers extension
```

## 路径 C：新增扩展 effect

适用场景：

- 需要新增一个 effect 原子
- 现有内置 effect 组合无法表达
- effect 能复用当前运行时主链

### 推荐目录

```text
extensions/<pack_id>/
  extension.json
  data/combat/effects/
  scripts/effects/
  scenes/validation/
```

`extension.json` 必须包含：

```json
{
  "register": [
    "resources",
    "effects"
  ]
}
```

### 执行步骤

1. 新增 `EffectDef` 资源
2. 新增 `strategy_script`
3. 确保脚本暴露：

```gdscript
func execute(context, params: Dictionary, node) -> Variant:
```

4. 补一个正向验证场景
5. 如果新增 schema、slot 或 manifest 边界，补一个负向守卫
6. 更新 `tools/validation_scenarios.json`
7. 更新验证矩阵和本工作流

### 完成判定

- 正向样例进入 `extension`
- 新边界进入 `guardrail`
- 失败路径进入 `protocol.issue`
- 不绕过 `EffectRegistry` 和 `EffectExecutor`

## 命名规则

### 扩展包

- 推荐：`<theme>_pack`
- 示例：`minimal_chaos_pack`
- 禁止：`test`, `new`, `pack1`

### 模板

- 植物：`plant_<pack_or_role>_<variant>`
- 僵尸：`zombie_<pack_or_role>_<variant>`
- 场上物件：`field_object_<role>_<variant>`

### 绑定

- 推荐：`plant_attack_<template_id_suffix>`
- 反击：`zombie_when_damaged_<template_id_suffix>`
- 死亡：`zombie_on_death_<template_id_suffix>`

### effect

- 资源文件：`effect_<effect_id>.tres`
- 脚本文件：`effect_<effect_id>.gd`
- `effect_id` 使用 snake_case

### 验证场景

- 主仓：`<ability>_validation`
- 错误技：`<ability>_chaos_validation`
- 扩展包 smoke：`<pack_id>_smoke_validation`
- 守卫：`<ability>_guardrail_validation`

## Showcase 同步规则

新增 Showcase 不是每个样例的必选项。

建议进入 Showcase 的内容：

- 能代表一个能力家族
- 适合人工观察
- 能帮助新协作者理解系统表达力

不建议进入 Showcase 的内容：

- 只验证负向守卫
- 只补边缘协议
- 没有可视化价值的失败路径

如果新增 Showcase：

1. 场景放入 `scenes/showcase/`
2. 名称和验证场景保持可追溯
3. 在 `scripts/main/showcase_hub.gd` 中接入分组
4. 不把 Showcase 逻辑写成运行时协议

## 常见失败点

| 现象 | 优先检查 |
|------|------|
| 扩展包没有被扫描 | `extension.json`、`enabled_by_default`、`activation_scenario_ids` |
| 模板无法解析 | `template_id`、目录位置、资源脚本类型 |
| effect 没有执行 | `EffectDef.effect_id`、`strategy_script`、`execute()` |
| on-hit 组合被拒绝 | slot tag 是否匹配 |
| 场景进不了批量回归 | `tools/validation_scenarios.json` 是否有条目和 `layers` |
| 验证通过但矩阵缺项 | 更新 `wiki/03-content-validation/32-验证矩阵.md` |

## smoke baseline 规则

只有下面类型适合进入 `smoke`：

- 最小引擎骨架
- 代表性玩法层正向路径
- 最小扩展包正向路径
- 最小协议守卫路径

普通内容样例默认不要进入 `smoke`，避免快速验证退化成全量验证。

---

## 主仓与扩展包边界

继续保留在主仓的内容：

- 核心运行时
- 基础协议守卫
- 代表性内建内容样例
- 第四阶段玩法层基线
- 最小验证和基础 Showcase

优先放进扩展包的内容：

- 新 effect 原子样例
- 非主仓基线的错误技组合
- 用于验证扩展边界的内容包
- 可独立启停的实验样例
