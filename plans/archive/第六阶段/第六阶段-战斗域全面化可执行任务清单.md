# 第六阶段-战斗域全面化可执行任务清单

> 这份清单把“第六阶段-战斗域全面化开发方案”拆成可直接认领的 `Epic / Issue / checklist`。它的目标不是继续围绕样例和协议修饰打转，而是在当前主干已经稳定、前置结构收口已经完成的基础上，正式进入植物、僵尸、交互矩阵、战场语义、波次结构和正式关卡内容的持续建设。

---

## 文档定位

这篇文档主要回答：

- 第六阶段现在可以直接开干的主任务有哪些
- 每项任务的产出物是什么
- 每项任务依赖哪些前置条件
- 每项任务完成后如何判断“算做完”
- 哪些任务应优先串行，哪些任务可以并行推进

这篇文档不负责：

- 代替完整项目路线图
- 一次性定义最终版数值平衡
- 提前展开完整选关、存档、图鉴、商店和花园
- 把社区工坊、编辑器产品化和外循环 UI 一起拉进主线
- 代替“正式植物与僵尸内容基线清单”

## 当前判断

结合当前仓库状态，第六阶段的起点应按下面前提理解：

1. 第五阶段已经证明：
   - 运行时主干可行
   - 扩展主链可行
   - 验证和守卫体系可持续
2. 第六阶段前置结构收口已经完成第一轮高价值动作：
   - `battle` 主路径接口已收口
   - 共享内容解析链已抽出
   - 正式内容目录语义已开始建立
   - demo 关卡已经完成第一轮卡片 / 波次资源拆分示范
3. 当前主线最需要的已经不是：
   - 新一轮协议冻结
   - showcase 扩编
   - effect 原子数量继续增长
4. 当前最需要的是：
   - 正式植物谱系
   - 正式僵尸谱系
   - 正式交互矩阵
   - 正式地图 / 棋盘 / 关卡语义
   - 正式波次和关卡模板

因此第六阶段当前最合理的定义是：

> 用当前已经稳定的模板主链、战斗玩法层和验证体系，开始沉淀真正可长期迭代的正式战斗内容，而不是继续围绕样例驱动开发。

---

## 使用方式

建议把下面的条目按两层理解：

- `Epic`：一组需要连续推进的主任务
- `Issue`：可直接认领、可单独验收的任务

如果当前人力有限，推荐按下面顺序推进：

1. `Epic A` 正式植物 roster v1
2. `Epic B` 正式僵尸 roster v1
3. `Epic C` 正式交互矩阵 v1
4. `Epic D` 战场语义与棋盘语义正式化
5. `Epic E` 波次与关卡模板正式化
6. `Epic F` 正式战斗内容回归区块 v1
7. `Epic G` 文档、清单与阶段收口

这个顺序背后的原则是：

- 先补齐玩家与敌方的正式内容基线
- 再整理这些内容之间为什么成立
- 再把地图、波次和关卡真正接进内容设计
- 最后把它们进入稳定回归和阶段输出

---

## 第六阶段边界提醒

当前应坚持下面 6 条边界：

1. 新增内容默认走模板和内容链，不走 `BattleManager` 内容特判。
2. 不把 showcase 数量当成内容深度替代品。
3. 不把“新增 effect 原子数量”当作本阶段主 KPI。
4. 不为未来不确定玩法提前做过度抽象。
5. 不优先做完整外循环 UI。
6. 每个新内容族都必须同时带上：
   - 至少 1 个正向验证
   - 至少 1 个关键交互验证
   - 必要时的守卫验证

## Epic A：正式植物 roster v1

当前这一组任务的收口文档建议统一沉淀到：

- `plans/第六阶段-正式植物与僵尸内容基线清单.md`

### Issue A1：确定植物正式大类与第一批代表单位

- 目标：
  - 把现有植物内容从“若干可运行样例”收敛成正式 roster 结构
- 产出物：
  - 一份植物正式大类和第一批代表单位清单
- 依赖：
  - 无
- 完成判定：
  - 至少明确下面大类：
    - 基础输出型
    - 高频输出型
    - 抛投覆盖型
    - 资源生产型
    - 防御 / 拦截型
    - 控制型
    - 支援 / 光环型
    - 召唤 / 分体型
  - 每类至少 1 个正式代表
- checklist：
  - [x] 列出现有可复用植物模板与能力归属
  - [x] 标出哪些植物可以直接升级为正式代表
  - [x] 标出哪些大类仍缺正式代表
  - [x] 为每个大类补一句“战术职责”

### Issue A2：补齐第一批缺口植物并进入正式目录

- 目标：
  - 把植物 roster 的缺口补到可支撑正式对抗设计的最低完整度
- 产出物：
  - 一批正式 `EntityTemplate / TriggerBinding / CardDef`
- 依赖：
  - `Issue A1`
- 完成判定：
  - 第一批正式植物不再只停留在 validation 语义
  - 卡片、模板和必要投射物资源进入正式内容目录
- checklist：
  - [x] 为缺口大类新增正式模板
  - [x] 补必要的卡片资源
  - [x] 补必要的投射物或绑定资源
  - [x] 避免把内容差异写成业务脚本分支

### Issue A3：为植物 roster v1 建立专项验证

- 目标：
  - 让植物正式内容进入稳定回归，而不是只在 demo 中“能跑”
- 产出物：
  - 植物 roster v1 的正向验证与关键交互验证
- 依赖：
  - `Issue A2`
- 完成判定：
  - 每个植物大类至少有 1 个有效验证落点
  - 关键植物不只验证“会攻击”，还验证它的战术职责
- checklist：
  - [x] 补植物 roster 正向验证
  - [x] 补植物对不同压力类型的交互验证
  - [x] 接入 `tools/validation_scenarios.json`
  - [x] 更新正式内容映射

## 第六阶段完成定义

这一阶段不要求一次性做完所有未来玩法，但至少应满足：

- 正式植物大类已经成形
- 正式僵尸大类已经成形
- 至少一轮正式交互矩阵已经成立
- 地图 / slot / board 语义已进入正式内容设计
- 正式 `WaveDef` 模板族已经建立
- 已有一批正式关卡模板
- 新正式内容仍全部走模板和内容链接入
- 正式内容已经进入稳定回归区块
- 项目已明显从“规则引擎 + 样例”转向“正式战斗内容开发平台”
