# 第六阶段-前置结构收口清单

> 这份清单把“进入第六阶段全面内容开发前，是否需要先做一次结构收口”拆成可直接认领的 `Epic / Issue / checklist`。它的目标不是发起一次大重构，而是在不打散当前稳定主干的前提下，先处理那些已经开始影响正式内容生产效率的结构摩擦点。

---

## 文档定位

这篇文档主要回答：

- 第六阶段开始前，哪些结构问题必须先收口
- 哪些收口项适合在 1 周内完成
- 哪些收口项适合在 2 周窗口内并行推进
- 哪些结构问题应该明确延期，避免抢占战斗内容主线
- 每项结构收口的产出物、依赖和完成判定是什么

这篇文档不负责：

- 代替第六阶段完整战斗域执行路线图
- 发起推倒重来的整体架构重写
- 把 ECS、完整编辑器或外循环产品化一起拉进当前主线
- 为了“更漂亮的架构”而修改已经稳定的协议语义
- 代替正式植物、僵尸、地图、波次和关卡内容基线清单

## 当前判断

结合当前仓库真实状态，第六阶段前的结构问题应按下面前提理解：

1. 当前主干已经具备正式进入战斗域全面化开发的最低条件：
   - 运行时协议主链已稳定
   - `EntityTemplate / ProjectileTemplate / TriggerBinding` 主链已可用
   - 战斗玩法闭环与验证矩阵已建立
   - 扩展入口与扩展 effect 主链已落地
2. 当前真正的风险已经不再是“还能不能继续做样例”，而是：
   - 正式内容一旦开始快速增长，当前结构是否会持续放大维护成本
3. 当前仓库已经出现几类明确的结构摩擦点：
   - `battle` 层横向通信过度依赖 `call()` / `has_method()`
   - `EntityFactory` 与 `ProtocolValidator` 存在重复的解析和归并逻辑
   - 正式内容、验证内容、showcase 内容的目录语义还没有完全分层
   - 关卡资源开始出现单文件内联膨胀趋势
4. 这些问题已经足以影响第六阶段的持续扩张效率，但还没有严重到必须整体重写。

因此当前最合理的策略不是“先停下来做大重构”，而是：

> 先用一个短窗口完成结构收口，把最会拖慢第六阶段内容开发的几处边界先整理清楚，然后立刻切回正式内容生产。

---

## 使用方式

建议把下面的条目按三层理解：

- `1 周必须先做`：不先收口，后续两周内容开发很容易边写边乱
- `2 周可并行推进`：适合和正式内容基线建设并行，不必全部阻塞主线
- `可延期`：确实有价值，但当前不应抢第六阶段主线

如果当前人力有限，推荐按下面顺序推进：

1. `Epic A` 战斗编排层接口收口
2. `Epic B` 内容解析与校验链去重
3. `Epic C` 正式内容目录语义建立
4. `Epic D` 关卡与波次资源拆分
5. `Epic E` 验证、showcase 与正式内容分层
6. `Epic F` 长期结构议题归档

---

## 当前边界提醒

第六阶段前置结构收口最容易失衡的地方，不是“做得不够多”，而是把不该现在做的事一起卷进来。

当前应坚持下面 6 条边界：

1. 这轮收口只处理已经开始阻碍正式内容开发的结构问题。
2. 不为了结构收口重写事件系统、Effect 系统或 Trigger 系统。
3. 不为了统一风格而大规模搬迁已经稳定的运行时代码。
4. 不让 showcase、验证和扩展样例继续反向主导正式内容目录结构。
5. 不把 ECS 化、完整编辑器化、完整外循环 UI 作为当前收口目标。
6. 每个 Issue 都必须同时回答：
   - 当前具体痛点是什么
   - 为什么它会在第六阶段被放大
   - 完成后新增正式内容会变得更容易在哪里

## 1 周必须先做

这一组任务的目标是：

> 先处理那些会直接降低第六阶段正式内容开发效率的接口和结构问题。

## Epic A：战斗编排层接口收口

当前这一组任务的收口结果建议主要落在：

- `scripts/battle/`
- 必要时补一份配套说明文档

### Issue A1：收缩 `BattleManager` 对外门面，减少横向字符串接口

- 目标：
  - 让 `BattleManager` 更接近稳定协调器，而不是继续充当“谁都可以随时 call 的万能门面”
- 当前痛点：
  - `battle_card_state.gd`
  - `battle_board_state.gd`
  - `wave_runner.gd`
  - `battle_spawner.gd`
  - `battle_subsystem_host.gd`
  - 以及部分实体脚本
  - 当前都在依赖大量 `call()` / `has_method()`
- 产出物：
  - 一组更明确的 `battle` 层稳定接口
  - `BattleManager` 保留职责清单
  - 子系统调用入口收敛结果
- 依赖：
  - 无
- 完成判定：
  - 常见主路径不再需要层层 `has_method()` 防御式调用
  - `BattleManager` 只保留：
    - 生命周期
    - 子系统装配
    - 场景解析和重置编排
    - 对外稳定战斗门面
  - 新增内容不再需要继续往 manager 塞内容级特判
- checklist：
  - [ ] 盘点 `BattleManager` 当前真实保留职责
  - [ ] 标出哪些调用应改为稳定显式接口
  - [ ] 收缩 `battle` 子系统之间的跨层动态调用
  - [ ] 为后续新增子系统能力建立入场守则

### Issue A2：明确 `battle` 子系统边界和所有权

- 目标：
  - 让正式内容开发能清楚知道“放置、出牌、波次、状态、场上物件”分别该改哪一层
- 当前痛点：
  - 当前 `BattleCardState / BattleBoardState / WaveRunner / BattleSpawner` 的边界虽然已初步分出，但新增需求仍容易跨层扩散
- 产出物：
  - 一份子系统边界表
  - 必要的轻量代码整理
- 依赖：
  - `Issue A1`
- 完成判定：
  - 能明确回答：
    - 放置合法性归谁
    - 实体生成归谁
    - 波次推进归谁
    - 资源消耗归谁
    - 胜负结算归谁
  - 第六阶段新增单位和关卡时，不需要反复猜入口
- checklist：
  - [ ] 为 `economy / board / card / flow / wave / spawner / field_object` 列职责边界
  - [ ] 标出跨层读写的最小允许路径
  - [ ] 禁止内容逻辑直接回流到 `BattleManager`
  - [ ] 为后续代码 review 建立边界判断口径

## Epic B：内容解析与校验链去重

当前这一组任务的收口结果建议主要落在：

- `scripts/battle/entity_factory.gd`
- `scripts/core/runtime/protocol_validator.gd`
- 新增的共享 resolver / normalizer 模块

### Issue B1：提取模板、投射体和 spawn override 的共享解析逻辑

- 目标：
  - 消除 `EntityFactory` 和 `ProtocolValidator` 中重复的模板、投射体和参数归并逻辑
- 当前痛点：
  - 生成侧和校验侧分别维护一套相似逻辑，后续每加一个正式字段都要多点修改
- 产出物：
  - 一个共享的内容解析模块
  - 统一的归并规则来源
- 依赖：
  - 无
- 完成判定：
  - 模板解析、projectile template 解析、flight profile 解析、spawn override 归并逻辑不再双份演化
  - 新增正式内容字段时，生成和校验能共享同一语义来源
- checklist：
  - [ ] 盘点当前重复逻辑的真实范围
  - [ ] 提取共享 resolver / normalizer
  - [ ] 让 `EntityFactory` 依赖共享逻辑而不是自维护
  - [ ] 让 `ProtocolValidator` 依赖同一套归并规则

### Issue B2：明确“内容生成语义”和“协议校验语义”的分层

- 目标：
  - 避免去重之后又把生成和校验搅成一个超大工具文件
- 当前痛点：
  - 当前重复问题的根源之一，是生成层和校验层都在处理相似对象，但边界没有彻底说明白
- 产出物：
  - 一份共享语义边界说明
  - 必要的命名整理
- 依赖：
  - `Issue B1`
- 完成判定：
  - 能明确回答：
    - 哪些逻辑是“解析/归并”
    - 哪些逻辑是“校验/守卫”
    - 哪些逻辑是“运行时实例化”
  - 后续不会把共享模块继续膨胀成第二个 `ProtocolValidator`
- checklist：
  - [ ] 明确共享模块只负责归并和解析，不吞并验证职责
  - [ ] 明确 `ProtocolValidator` 继续负责守卫、白名单和失败口径
  - [ ] 明确 `EntityFactory` 继续负责实例化和运行时装配
  - [ ] 为新增字段定义统一接入顺序

## Epic C：正式内容目录语义建立

当前这一组任务的收口结果建议主要落在：

- `data/combat/`
- `scenes/`
- 配套索引文档

### Issue C1：把正式内容、验证内容、展示内容的目录语义拆开

- 目标：
  - 让第六阶段的植物、僵尸、地图、波次、关卡不再主要围绕 validation/showcase 命名增长
- 当前痛点：
  - 当前目录更适合第五阶段样例和验证，不够适合第六阶段正式内容规模化沉淀
- 产出物：
  - 更新后的内容目录约定
  - 必要的新目录骨架
- 依赖：
  - `Epic A`
- 完成判定：
  - 正式内容目录和 validation/showcase 目录有明确分工
  - 新增正式内容时不再优先想到“先放到 validation/showcase”
- checklist：
  - [ ] 明确 `data/combat` 下正式内容的一级分类
  - [ ] 明确 `scenes/validation` 只承载验证场景
  - [ ] 明确 `scenes/showcase` 只承载展示包装
  - [ ] 为正式战斗内容建立独立的组织入口

### Issue C2：建立第六阶段正式内容基线目录

- 目标：
  - 为植物、僵尸、波次、关卡、地图语义建立长期可扩张的目录落点
- 当前痛点：
  - 当前正式内容还偏扁平，继续增长容易找不到“哪类内容放哪里”
- 产出物：
  - 建议目录及说明，例如：
    - `data/combat/cards`
    - `data/combat/waves`
    - `data/combat/levels`
    - `data/combat/battlefields`
    - 或等价分层
- 依赖：
  - `Issue C1`
- 完成判定：
  - 第六阶段后续内容清单能够直接映射到目录结构
  - 不再需要把正式关卡模板继续内联塞进 demo 资源
- checklist：
  - [ ] 为正式卡片数据建立目录落点
  - [ ] 为正式波次模板建立目录落点
  - [ ] 为正式关卡模板建立目录落点
  - [ ] 为正式战场语义资源建立目录落点

## Epic D：关卡与波次资源拆分

当前这一组任务的收口结果建议主要落在：

- `scenes/demo/demo_level.tres`
- 正式 `BattleScenario` 资源组织方式

### Issue D1：拆分大体量内联场景资源

- 目标：
  - 避免 `BattleScenario` 继续演化成一个把卡片、阳光、波次、刷怪、场上物件全部内联的超大资源
- 当前痛点：
  - 当前 demo 场景已经出现单文件内联增长趋势，不利于复用和对比
- 产出物：
  - 一套可复用的子资源拆分策略
  - 至少一个示范性拆分案例
- 依赖：
  - `Issue C2`
- 完成判定：
  - `BattleScenario` 更多负责装配，而不是承载全部细节
  - 波次、卡组、场上物件等可以独立维护或复用
- checklist：
  - [ ] 明确哪些资源应独立化
  - [ ] 明确哪些仍保留在场景内联更简单
  - [ ] 用 1 个已有关卡做拆分示范
  - [ ] 保持读取和验证入口兼容

### Issue D2：建立正式关卡模板和波次模板的复用口径

- 目标：
  - 让第六阶段新增关卡时是“组合模板”，不是“复制一份再改”
- 当前痛点：
  - 如果不先定义复用口径，正式内容一上量就会在波次和关卡里复制粘贴
- 产出物：
  - 关卡模板和波次模板复用说明
  - 第一批模板骨架
- 依赖：
  - `Issue D1`
- 完成判定：
  - 能区分：
    - 可复用模板
    - 关卡实例
    - 验证场景
  - 第六阶段正式内容不再默认走“复制 demo_level”
- checklist：
  - [ ] 明确模板和实例的命名约定
  - [ ] 明确波次模板如何复用到多个关卡
  - [ ] 明确哪些字段适合实例覆盖
  - [ ] 明确验证场景和正式关卡的关系

## Epic E：验证、showcase 与正式内容分层

当前这一组任务的收口结果建议主要落在：

- `tools/validation_scenarios.json`
- `scenes/showcase/`
- 第六阶段内容工作流文档

### Issue E1：建立正式内容与验证场景的一对多映射口径

- 目标：
  - 让验证继续服务正式内容，而不是继续成为内容主组织方式
- 当前痛点：
  - 当前验证场景命名和第五阶段样例命名较强，正式内容成长后容易失焦
- 产出物：
  - 正式内容到 validation 的映射约定
  - 一份示例映射表
- 依赖：
  - `Issue C2`
- 完成判定：
  - 每个正式内容族都能回答：
    - 它对应哪些正向验证
    - 它对应哪些关键交互验证
    - 它是否需要守卫验证
  - `validation_scenarios.json` 继续承担回归入口，而不是内容目录
- checklist：
  - [ ] 明确“正式内容 ID”与“验证场景 ID”的映射规则
  - [ ] 区分 smoke/core/guardrail 与正式内容域的关系
  - [ ] 明确新增正式内容时验证接入顺序
  - [ ] 避免 validation 命名继续反向定义内容分类

### Issue E2：把 showcase 从“内容入口”收回到“展示入口”

- 目标：
  - 保留 showcase 的价值，但避免它继续承担正式内容目录和阶段组织职责
- 当前痛点：
  - 当前 showcase 仍强烈保留第五阶段样例分组语义
- 产出物：
  - showcase 分层说明
  - 必要的分组调整建议
- 依赖：
  - `Issue E1`
- 完成判定：
  - showcase 只负责：
    - 演示
    - 人工观察
    - 导航包装
  - 正式内容清单和关卡清单不再依赖 showcase 命名来组织
- checklist：
  - [ ] 区分“验证用场景”和“展示用场景”
  - [ ] 区分“样例展示”和“正式关卡展示”
  - [ ] 明确 showcase 对正式内容只做包装，不拥有内容定义权
  - [ ] 为后续正式关卡入口预留展示位

  - [ ] 定义“允许边开发边整理”的范围
  - [ ] 定义什么情况才允许新增结构性重构任务
  - [ ] 把“内容推进优先”写成明确执行口径

## 完成口径

这轮前置结构收口不要求把所有历史结构问题一次性解决，但至少应满足：

- `BattleManager` 和 `battle` 子系统边界进一步清晰
- 关键主路径不再过度依赖字符串动态调用
- 模板、投射体和 spawn override 的共享解析逻辑已完成第一轮去重
- 正式内容、验证内容、showcase 内容的目录语义已经分清
- `BattleScenario` 大资源拆分策略已经有示范，不再只能继续内联增长
