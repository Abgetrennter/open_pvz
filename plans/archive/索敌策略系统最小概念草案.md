# 索敌策略系统最小概念草案

> 这份草案已经按当前仓库的真实落地范围收口。它回答的不是“未来完整索敌 AI 怎么做”，而是“第一批怎样以最小改动解决植物空射，并把检测结果接入现有运行时主链”。

---

## 文档定位

这篇文档主要回答：

- 第一批索敌系统到底落在哪一层
- 当前已经真正实现了哪些检测策略
- 检测结果如何进入 `Trigger -> RuleContext -> BattleManager` 主链
- 第一批协议字段是什么
- 当前专项验证覆盖了什么

这篇文档不负责：

- 完整目标优先级排序
- 三线、五向、星形、多段索敌
- 空中专用索敌
- Area2D 实时范围感知
- 完整效果层 fan-out 发射

---

## 一句话结论

当前已采用一套最小可落地方案：

1. 新增 `DetectionRegistry` 作为检测策略注册表。
2. 当前第一批只内置 `always` 和 `lane_forward` 两个策略。
3. `periodically` 在时间条件通过后，可选执行检测条件。
4. 检测结果不只决定“是否触发”，还会把 `primary_target` 写入 `RuleContext.target_node`。

这意味着当前系统已经不再只是“挡住空射”，而是已经补上了：

> 检测结果进入上下文，供追踪和抛射投射物复用。

---

## 当前问题

在没有索敌条件之前，`periodically` 只检查时间间隔：

```text
game.tick
-> TriggerInstance.should_trigger()
-> interval satisfied
-> spawn_projectile
```

这会导致：

1. 没有敌人时也持续发射。
2. 相邻车道的敌人无法被明确隔离。
3. `track / parabola` 这类投射物在触发时缺少正式目标入口，只能回退到运行时兜底扫描。

---

## 当前设计选择

## 1. 落点仍在触发器层

索敌的第一职责仍然是“是否放行周期触发”，所以当前把它放在：

- `TriggerRegistry.periodically`
- `TriggerBinding.condition_values`

而不是放到：

- `spawn_projectile`
- `BattleManager` 特判

这样做的好处是：

- 语义上仍然是触发条件
- 不会让效果层承担“该不该执行”的职责
- 向后兼容原有 `attack -> periodically` 映射

## 2. 第一批不使用 `Dictionary` 条件参数

为了保持和当前 validator 一致，第一批没有采用：

- `detection_params: Dictionary`

而是收口成两个正式条件字段：

- `detection_id: StringName`
- `scan_range: float`

这样可以直接接入当前 `TriggerDef.condition_params` 白名单，不需要同时扩展新的参数类型系统。

## 3. 第一批只做单线前向扫描

当前真正落地并进入正式白名单的检测策略只有：

- `always`
- `lane_forward`

后续候选但当前未落地：

- `lane_bidirectional`
- `three_lane_forward`
- air-only / height-aware detection

---

## DetectionRegistry

当前新增：

- [`autoload/DetectionRegistry.gd`](../autoload/DetectionRegistry.gd)

结构保持和现有 registry 体系一致：

- `register_strategy(detection_id, callable)`
- `get_strategy(detection_id)`
- `evaluate(detection_id, owner, params) -> Dictionary`

返回值结构当前统一为：

```gdscript
{
    "has_target": bool,
    "targets": Array,
    "primary_target": Node2D,
}
```

当前第一批会把它用于两件事：

1. 判断触发器是否应继续执行。
2. 把 `primary_target` 回写到运行时上下文。

---

## 当前内置检测策略

### `always`

语义：

- 始终通过
- 不提供检测目标

用途：

- 向后兼容老绑定
- 不需要目标的周期行为

### `lane_forward`

语义：

- 只扫描当前实体所在车道
- 只扫描前方目标
- 按地面位置距离升序返回

当前筛选条件：

- `candidate != owner`
- `candidate.team != owner.team`
- `candidate.lane_id == owner.lane_id`
- `candidate.has_method("take_damage")`
- `candidate is Node2D`
- 若实体实现 `is_combat_active()`，则要求其为 `true`

当前参数：

| 参数 | 类型 | 默认值 |
|---|---|---|
| `scan_range` | `float` | `900.0` |

---

## 主链接入方式

当前接入点在：

- [`autoload/TriggerRegistry.gd`](../autoload/TriggerRegistry.gd)
- [`scripts/core/runtime/trigger_instance.gd`](../scripts/core/runtime/trigger_instance.gd)

执行链现在变成：

```text
game.tick
-> periodically interval satisfied
-> DetectionRegistry.evaluate(detection_id, owner, { scan_range })
-> has_target == false -> stop
-> has_target == true
-> primary_target 写入 TriggerInstance.pending_context_overrides
-> TriggerInstance.execute()
-> RuleContext.from_event_data(...)
-> 把 target_node 覆盖成检测结果
-> EffectExecutor
```

这一步是当前版本最重要的收口点：

- 检测不再只是返回一个 `bool`
- 检测结果已经成为 `RuleContext` 的正式输入之一

---

## 为什么这版是可行的

当前仓库里已经有下面这些能力：

1. `RuleContext.target_node`
2. `spawn_projectile -> BattleManager._resolve_projectile_target_node()`
3. `track / parabola` 对 `context.target_node` 的消费

因此第一批最小方案只需要补：

- 检测条件注册
- 检测结果传递

而不需要重写：

- 效果系统
- 投射物系统
- BattleManager 主干

---

## 当前明确不承诺的能力

下面这些能力目前仍然不应被当成“这份草案已经覆盖”：

1. 三线射手“按各 lane 分别发射”
2. 分裂豆前后双向目标分配
3. 星星果五向检测
4. 只扫描空中单位
5. 基于检测结果直接 fan-out 多个 `spawn_projectile`

原因很直接：

- 这些能力都不只是“是否触发”
- 它们还要求效果层理解“多个目标 / 多个车道 / 多个方向”

当前第一批尚未补这条表达链。

---

## 当前协议字段

当前 `periodically` 第一批正式条件字段为：

| 字段 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `interval` | `float` | 无 | 周期时间间隔 |
| `detection_id` | `string_name` | `always` | 检测策略 id |
| `scan_range` | `float` | `900.0` | 当前检测半径 |

当前 `detection_id` 白名单为：

- `always`
- `lane_forward`

这意味着：

- 未知 `detection_id` 会在协议层报错
- 白名单外条件字段会继续报错
- 老内容即使不写 `detection_id`，也会被默认归一化为 `always`

---

## 当前数据示例

以基础射手为例：

```text
condition_values = {
    "interval": 0.6,
    "detection_id": &"lane_forward",
    "scan_range": 900.0
}
```

不需要检测的行为可继续省略：

```text
condition_values = {
    "interval": 0.6
}
```

运行时会自动把它归一化成：

```text
detection_id = always
scan_range = 900.0
```

## 当前已接入的内容

当前第一批已经接到绑定资源上的攻击类型包括：

- `plant_attack_basic_linear`
- `plant_attack_repeater_burst`
- `plant_attack_track_bomb`
- `plant_attack_cabbage_arc`
- `plant_attack_melon_blast`

当前故意没有一起推进的内容包括：

- `plant_attack_air_spike`

因为空中专用索敌还没有正式落地，当前不应把“只打空中目标”的需求混进第一批。

## 当前专项验证

当前新增专项：

- [`scenes/validation/target_acquisition_validation.tres`](../scenes/validation/target_acquisition_validation.tres)

覆盖 3 件事：

1. 同 lane 有敌人时，基础射手会发射。
2. 相邻 lane 有敌人时，追踪投弹手不会误触发。
3. 卷心菜投手的检测结果会进入 `RuleContext.target_node`，并在本 lane 正确命中。

同时已接入：

- [`tools/validation_scenarios.json`](../tools/validation_scenarios.json)

## 下一步建议

如果后续要继续扩展，建议按下面顺序推进：

1. `lane_bidirectional`
2. `three_lane_forward`
3. 高度带 / 空中目标过滤
4. 效果层 fan-out 发射

顺序不能反过来，因为：

- 没有目标结果传递，多 lane 检测没有真实落点
- 没有效果层 fan-out，多 lane 检测也只能停留在“是否触发”
