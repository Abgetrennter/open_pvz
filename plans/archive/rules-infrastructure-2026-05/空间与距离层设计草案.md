### 核心缺口：所有空间查询是 O(n) 全量扫描

所有索敌和碰撞检测都调用 `get_runtime_entities()` / `get_runtime_combat_entities()` 返回全量数组后逐个过滤。高频调用点：

| 位置 | 操作 | 每帧调用次数 |
|------|------|-------------|
| `DetectionRegistry._scan_enemies()` | 策略索敌（lane_forward 等） | 每个攻击实体每 tick |
| `zombie_root._find_attack_target()` | 僵尸寻找攻击目标 | 每帧每僵尸 |
| `zombie_root._find_attack_target_with_range()` | 带范围的目标搜索 | 每帧每僵尸 |
| `projectile_root._find_terminal_hit_target()` | 投射体终端碰撞 | 每帧每投射体 |
| `projectile_root._find_segment_hit_target()` | 投射体线段碰撞 | 每帧每投射体 |
| `lawn_mower._check_zombie_proximity()` | 割草机邻近检测 | 每帧每割草机 |
| `lawn_mower._sweep()` | 割草机扫荡伤害 | 每帧每割草机（激活后） |
| `battle_projectile_effect_resolver._find_nearest_enemy()` | 效果解析最近敌人 | 按需 |

N 个僵尸 + M 个投射体的场景下，每帧扫描总量约为 O((N+M) × (N+M))。

## 缺口分析与方案

### 缺口一：空间索引

#### 设计目标

将 O(n) 全量扫描降为 O(log n + k)（k 为结果数），支撑 50+ 实体场景。

#### 方案：车道分组 + X 轴排序索引

PVZ 场景的空间特征决定了不需要通用四叉树：

- **车道内查询占 90%**（lane_forward / lane_backward）
- 车道间几乎不交叉，Y 轴方向天然分区
- X 轴是主要运动方向，实体密集排列

因此用 `Dictionary[lane_id -> Array[Entity]]`（按 x 排序）是最优选择。

#### 数据结构

```gdscript
class_name SpatialIndex
extends RefCounted

var _lanes: Dictionary = {}          # int -> Array[Dictionary]
var _all_entities: Array = []        # 跨车道查询用
var _dirty := true

func rebuild(entities: Array) -> void:
    _lanes.clear()
    _all_entities.clear()
    for entity in entities:
        if not entity is Node2D: continue
        var lane_id: int = entity.get("lane_id")
        var x: float = _ground_x(entity)
        if not _lanes.has(lane_id):
            _lanes[lane_id] = []
        _lanes[lane_id].append({"node": entity, "x": x})
        _all_entities.append({"node": entity, "x": x, "lane": lane_id})
    for lane_id in _lanes:
        _lanes[lane_id].sort_custom(_sort_by_x)
    _all_entities.sort_custom(_sort_by_x)
    _dirty = false
```

#### 查询接口

```gdscript
# 车道内前向查询：二分起点 → 向右遍历到 range 截止
func query_lane_forward(source_lane: int, source_x: float, range: float) -> Array

# 车道内后向查询
func query_lane_backward(source_lane: int, source_x: float, range: float) -> Array

# 半径查询：确定涉及车道 → 每车道做范围二分
func query_radius(source_lane: int, source_x: float, source_y: float, radius: float) -> Array

# 全局最近查询
func query_nearest(source_position: Vector2, team_filter: StringName) -> Node2D
```

#### 重建策略

- 每 tick 结束时重建一次（实体位移在 tick 内是确定性的）
- 实体 spawn / death 标记 dirty，下次查询前按需重建
- 开销：50 个实体排序 ~50 × log(50) ≈ 280 次比较，每 tick 一次，可忽略

#### 放置位置

独立模块：`scripts/battle/spatial_index.gd`

由 `BattleSubsystemHost` 持有，在 `game.tick` 后重建。`DetectionRegistry._scan_enemies()` 改为调用 `SpatialIndex` 的查询接口。

#### 车道数兼容性

结构是 `Dictionary[lane_id -> sorted_array]`，车道数完全透明：

- 5 车道 → 5 个 key
- 6 车道（泳池） → 6 个 key
- 3 车道（水空分割） → 3 个 key
- 查询逻辑不依赖车道总数

当前车道配置由 `BattlefieldPreset.lane_count` 驱动，已有 2/3/5 车道的 preset。新增 6 车道泳池只需新增一个 `BattlefieldPreset` 资源文件。

#### 迁移路径

1. 新建 `SpatialIndex` 类，不改动现有代码
2. `BattleSubsystemHost` 在 tick 后调用 `rebuild()`
3. `DetectionRegistry._scan_enemies()` 内部优先走 `SpatialIndex`，回退到全量扫描
4. 各实体（zombie_root / projectile_root / lawn_mower）的私有扫描方法逐步迁移到调用 `DetectionRegistry` 统一接口
5. 验证：现有 127 个验证场景全部通过，无行为变化

### 缺口二：高度过滤不统一

#### 问题

`HeightBand` 已经定义了连续高度区间，`ProjectileFlightProfile.max_hit_height` 也用于碰撞过滤，但两者没有统一对接：

- `DetectionRegistry._scan_enemies()` 不过滤高度
- `zombie_root._find_attack_target()` 不过滤高度
- `projectile_root` 的碰撞检测部分使用 `max_hit_height`，但不走 `HeightBand` overlap
- 效果系统（`explode` 等范围效果）没有高度维度

#### 方案：HeightBand overlap 成为碰撞/检测的标准过滤步骤

不做新模型，而是在所有空间查询路径中统一加入 HeightBand overlap 判断。

**判断规则**：

```
攻击者的 effective_height_range 与 目标的 height_band 有 overlap → 可命中
```

攻击者的 effective_height_range 来源：
- 投射体：`ProjectileFlightProfile` 当前高度 ± hitbox_radius
- 近战/范围效果：archetype 关联的 HeightBand 的 [min_height, max_height]
- AOE：效果参数中定义的高度范围（默认无限）

**实施位置**：

1. `DetectionRegistry._scan_enemies()` 增加可选的高度过滤参数
2. `projectile_root` 碰撞检测统一走 HeightBand overlap
3. `SpatialIndex` 的查询接口支持 height_range 过滤

**不改动的部分**：

- `HeightBand` 资源定义不变（已有 5 个 band，区间重叠设计合理）
- `ProjectileFlightProfile` 不变
- archetype 中已关联的 height_band 引用不变

### 缺口三：碰撞层未形式化

#### 问题

哪些实体会被哪些攻击命中，目前靠分散判断：
- `DetectionRegistry._scan_enemies()` 用 team 过滤（同队跳过）
- 实体 `has_method("take_damage")` 判断是否可受伤
- 投射体 hit_policy 和 collision_padding 控制碰撞行为
- 没有统一的碰撞矩阵

#### 方案：在 archetype 中增加 collision_group 字段

不新建系统，在现有数据定义上扩展：

```gdscript
# CombatArchetype 新增
@export var collision_group: StringName = &"default"
```

内置 collision_group 值及碰撞规则：

| group | 能被命中 by |
|-------|-----------|
| `plant` | `zombie_melee`, `zombie_projectile` |
| `zombie` | `plant_projectile`, `lawn_mower`, `aoe` |
| `collectible` | 无（不可被攻击命中） |
| `field_object` | `aoe`（溅射可波及） |
| `structure` | `zombie_melee`（近战可破坏） |

这更多是**数据声明**而非运行时系统。碰撞检测函数从 `if team == source_team: continue` 演变为 `if not _can_collide(source.collision_group, target.collision_group): continue`。

**优先级**：低于空间索引和高度过滤。等需要新碰撞语义的实体（Cactus 对空/对地切换、Blover 驱散飞行）实际开发时再做。

### 缺口四：场地变形

#### 问题

Doom-shroom 坑洞、屋顶斜面、水池深度——场地本身没有可变属性模型。

#### 方案：远期，预留接口

当前 `BattleBoardState` 的 slot 已经有 `slot_type`（ground/water/roof/air）。场地变形的核心是"某个 slot 的属性在运行时发生了变化"。

**预留方式**：确保新代码不假设"slot 属性不可变"。

实际实现时，在 `BoardSlot` 中增加运行时 modifier：

```gdscript
# BoardSlot 可能的扩展（远期）
var modifiers: Dictionary = {}  # {"crater": {"remaining_ticks": 180}}
```

现阶段不实现。等 Doom-shroom 等需要场地变形的实体开发时再设计。

## 与已有验证场景的关系

空间索引和高度过滤是**纯引擎层优化**，不改变任何 archetype 资源文件和验证场景定义。验证策略：

1. 所有 127 个验证场景必须全部通过（行为不变）
2. 新增专项验证场景：大量实体（40+）场景下的索敌正确性和性能基准
3. 高度过滤新增验证场景：不同 HeightBand 组合下的碰撞判定
