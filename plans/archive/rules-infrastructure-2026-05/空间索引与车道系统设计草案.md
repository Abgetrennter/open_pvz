# 空间索引、分组索引与车道系统设计草案

> 状态：已归档草案 | 创建：2026-05-09 | 更新：2026-05-09
> 该方向已完成实现，正式计划归档于本目录的 `空间索引与高度过滤计划.md`；当前事实以 `wiki/02-runtime-protocol/17-实体活跃性与空间查询.md` 为准。
>
> 本文档合并了"未来计划"中第二部分"空间与距离层 · 空间索引"和第三部分"实体生命周期层 · 实体分组/查询"两个缺口的分析。两者在实现上属于同一个问题——所有热路径的查询都是"分组过滤 + 空间过滤"的联合操作，拆成两个独立系统会增加不必要的接口复杂度。

## 一、背景与问题

当前所有实体查询（索敌、范围效果、碰撞检测）均为 O(n) 全量遍历，且每条热路径都需要多维度过滤。集中在以下调用点：

| 位置 | 操作 | 频率 | 过滤维度 |
|------|------|------|----------|
| `zombie_root.gd` `_find_attack_target()` | 攻击目标搜索 | 每帧每僵尸 | team + lane + x_range + direction |
| `zombie_root.gd` `_find_attack_target_with_range()` | 带范围攻击目标搜索 | 每帧每僵尸(Controller) | team + lane + x_range + direction |
| `projectile_root.gd` `_find_segment_hit_target()` | 扫掠碰撞检测 | 每帧每投射体 | team + segment + height |
| `projectile_root.gd` `_find_terminal_hit_target()` | 终点碰撞检测 | 每帧每投射体 | team + radius + height |
| `DetectionRegistry._scan_enemies()` | 目标发现（6 种策略） | 按触发器 | team + lane + tags + range + direction |
| `EffectRegistry` `enemies_in_radius` | 范围伤害效果 | 按效果 | team + radius |
| `EffectRegistry` `dispel_flying` | 驱散飞行效果 | 按效果 | tags("flying") + radius |
| `ControllerRegistry` sweep/bite | 控制器目标搜索 | 每 0.1s | team + radius |
| `lawn_mower.gd` proximity/sweep | 割草机检测与扫荡 | 每帧每割草机 | team + lane + x_distance |
| `battle_projectile_effect_resolver.gd` `_find_nearest_enemy()` | 找最近敌人 | 按需 | team |

实体获取统一走 `battle.get_runtime_entities()`（拼接 entity_root + collectible_root 子节点），无任何索引。每次查询对所有候选实体逐一执行 `has_method()` + `get("team")` + `get("lane_id")` + `get("tags")` 等反射调用。

问题有两个维度：
- **空间维度**：O(n) 全量遍历，无 lane/x 排序，无法利用 PVZ 车道结构剪枝
- **分组维度**：team（plant/zombie/neutral）、lane_id、tags（flying/metal 等）、entity_kind（plant/zombie/projectile）等分组信息没有索引，每次全量扫描后逐个判断

当实体从当前十几二十个增长到 40-50 个时，O(n²) 的总开销会明显影响帧时间。

## 二、现有车道系统分析

  └── 被 battlefield_preset.lane_count 覆盖（如果存在）
       └── BattleManager._rebuild_lane_config() 据此生成 lane_y_map

已有配置：

| `battlefield_ground_baseline` | 5 | 标准五道白天场景 |

位于 `BattleManager._rebuild_lane_config()`：

- `lane_count == 2` 时硬编码 `{0: 220.0, 1: 320.0}`
- 其他情况：`top_y = 185.0`，`spacing = 60.0`，均匀分布

### 2.3 槽位系统

- 每个槽位有 `slot_type`（ground / water / roof / air）和 `placement_tags`

## 三、空间索引与分组索引设计

### 3.1 为什么不用四叉树 + 为什么不用独立 EntityRegistry

PVZ 场景的空间特征非常特殊：

- 车道是离散的平行带，Y 轴方向几乎不交叉
- X 轴方向实体密集排列，间距相对均匀
- 实体总数上限约 50-80（含投射体），不需要极端优化

四叉树在这种场景下反而不如车道分组高效：四叉树的分裂/合并开销、跨车道的边界判断都不必要。

同样，**不采用独立的 EntityRegistry + SpatialIndex 两层架构**。原因：

1. 所有热路径的查询都是"分组过滤 + 空间过滤"的联合操作（见第一节表格）。拆成两个系统意味着每次查询穿两层，接口复杂度翻倍
2. 空间索引的 per-tick rebuild 策略天然适合同步建分组索引——重建时顺手多插几个字典，成本可忽略
3. `DetectionRegistry._scan_enemies()` 已经是单次遍历做全部过滤。合并后只需一次 `query()` 调用

**结论：SpatialIndex 统一承担空间索引 + 分组索引 + 实体注册表三重职责。**

### 3.2 核心数据结构

```gdscript
# scripts/battle/spatial_index.gd
class_name SpatialIndex
extends RefCounted

# --- 分组索引 ---
var _by_team: Dictionary = {}       # team_name -> Array[Node]
var _by_lane: Dictionary = {}       # lane_id -> Array[Node] (按 x 排序)
var _by_tag: Dictionary = {}        # tag -> Array[Node]
var _by_kind: Dictionary = {}       # entity_kind -> Array[Node]
var _combat_active: Array = []      # 所有 combat_active 实体缓存

# --- 全量缓存 ---
var _all: Array = []                # 所有已索引实体

# --- 查询快照 ---
var _snapshot_version: int = 0
```

### 3.3 Rebuild 策略

在 `game.tick` 末尾一次性重建，而非每次实体移动都更新：

- 投射体每帧移动，逐次更新的代价远大于批量重建
- 单帧内查询顺序不影响正确性（同一帧所有查询看到同一快照）
- 重建代价：N 个实体 × 4 个字典 insert + 车道排序 ≈ O(N log N)，N=50 时可忽略

```gdscript
func rebuild(entities: Array) -> void:
    _by_team.clear()
    _by_lane.clear()
    _by_tag.clear()
    _by_kind.clear()
    _combat_active.clear()
    _all.clear()

    for entity in entities:
        if entity == null or not is_instance_valid(entity):
            continue
        if not (entity is Node2D):
            continue
        _all.append(entity)
        _index_grouping(entity)

    # 车道内按 x 排序（空间查询依赖）
    for lane_id in _by_lane:
        _by_lane[lane_id].sort_custom(_compare_by_x)

    _snapshot_version += 1


func _index_grouping(entity: Node2D) -> void:
    # team
    var team: StringName = entity.get("team") if entity.get("team") != null else &""
    if team != StringName():
        if not _by_team.has(team):
            _by_team[team] = []
        _by_team[team].append(entity)

    # lane
    if entity.has_method("get") and entity.get("lane_id") != null:
        var lane_id: int = int(entity.get("lane_id"))
        if lane_id >= 0:
            if not _by_lane.has(lane_id):
                _by_lane[lane_id] = []
            _by_lane[lane_id].append(entity)

    # tags
    if entity.has_method("get"):
        var tags: Variant = entity.get("tags")
        if tags is PackedStringArray or tags is Array:
            for tag in tags:
                var tag_key: String = String(tag)
                if not _by_tag.has(tag_key):
                    _by_tag[tag_key] = []
                _by_tag[tag_key].append(entity)

    # entity_kind
    if entity.has_method("get"):
        var kind: Variant = entity.get("entity_kind")
        if kind != null:
            var kind_key: String = String(kind)
            if not _by_kind.has(kind_key):
                _by_kind[kind_key] = []
            _by_kind[kind_key].append(entity)

    # combat_active
    if entity.has_method("is_combat_active"):
        if entity.call("is_combat_active"):
            _combat_active.append(entity)
    else:
        _combat_active.append(entity)
```

### 3.4 统一查询接口

```gdscript
# 查询参数说明：
#   team_exclude: StringName   排除该 team 的实体（最常用：排除己方）
#   team_include: StringName   只包含该 team
#   lane_ids: PackedInt32Array 限定车道（空 = 所有车道）
#   tags_any: PackedStringArray 实体必须包含至少一个指定 tag
#   tags_all: PackedStringArray 实体必须包含所有指定 tag
#   kinds: PackedStringArray   限定 entity_kind
#   center: Vector2            空间查询中心点
#   radius: float              空间查询半径
#   x_min: float               X 轴范围下限
#   x_max: float               X 轴范围上限
#   direction: StringName      "forward" / "backward" / "both"
#   height_range: Vector2      高度区间过滤 (min, max)
#   combat_active_only: bool   只返回 combat_active 实体
#   sort_by_distance: bool     按距离排序结果
#   max_results: int           最多返回 N 个结果

func query(params: Dictionary) -> Array:
    var candidates := _resolve_candidate_pool(params)
    var filtered := _apply_filters(candidates, params)
    if params.get("sort_by_distance", false) and params.has("center"):
        var center: Vector2 = params["center"]
        filtered.sort_custom(func(a, b): return _dist(a, center) < _dist(b, center))
    var max_results: int = int(params.get("max_results", 0))
    if max_results > 0 and filtered.size() > max_results:
        filtered = filtered.slice(0, max_results)
    return filtered
```

**`_resolve_candidate_pool`** 利用分组索引快速缩小候选集：

```gdscript
func _resolve_candidate_pool(params: Dictionary) -> Array:
    # 优先使用最窄的分组索引作为初始候选集
    var pool: Array = []

    if params.has("tags_any") and not params["tags_any"].is_empty():
        # tag 索引通常最窄（如 "flying" 可能只有 5 个实体）
        pool = _pool_from_tags(params["tags_any"])
    elif params.has("team_exclude"):
        # team_exclude → 取对立 team 的索引
        var exclude: StringName = params["team_exclude"]
        for team in _by_team:
            if StringName(team) != exclude:
                pool.append_array(_by_team[team])
    elif params.has("team_include"):
        var include: StringName = params["team_include"]
        if _by_team.has(String(include)):
            pool = _by_team[String(include)]
    elif params.has("kinds"):
        pool = _pool_from_kinds(params["kinds"])
    elif not params.get("lane_ids", PackedInt32Array()).is_empty():
        pool = _pool_from_lanes(params["lane_ids"])
    else:
        pool = _combat_active if params.get("combat_active_only", true) else _all

    return pool
```

### 3.5 查询模式映射

| 热路径 | query() 参数 | 优化效果 |
|--------|-------------|----------|
| `zombie_root._find_attack_target` | `team_exclude=team, lane_ids=[lane_id], x_min=x-range, x_max=x, direction=forward, combat_active_only=true` | O(n) → O(log n + k)，利用 lane 排序二分 |
| `zombie_root._find_attack_target_with_range` | 同上，x_max=x, x_min=x-range, direction=forward | 同上 |
| `projectile_root._find_segment_hit_target` | `team_exclude=team, x_min=start.x, x_max=end.x, height_range=[h1,h2], combat_active_only=true` + segment filter | team 索引剪掉一半实体 |
| `projectile_root._find_terminal_hit_target` | `team_exclude=team, center=pos, radius=R, height_range=[h,h], combat_active_only=true` | 同上 |
| `DetectionRegistry._scan_enemies` (lane_forward) | `team_exclude=team, lane_ids=[lane_id], x_min=x, direction=forward, tags_any=[...], combat_active_only=true` | lane + team 双重剪枝 |
| `DetectionRegistry._scan_enemies` (radius_around) | `team_exclude=team, center=pos, radius=R, combat_active_only=true` | team 剪枝 |
| `EffectRegistry.dispel_flying` | `tags_any=["flying"], center=pos, radius=R` | tag 索引直接命中，从 O(n) 降到 O(k) |
| `EffectRegistry.enemies_in_radius` | `team_exclude=team, center=pos, radius=R, combat_active_only=true` | team 剪枝 |
| `ControllerRegistry` sweep/bite | `team_exclude=team, center=pos, radius=R, combat_active_only=true` | team 剪枝 |
| `lawn_mower` proximity/sweep | `team_exclude=team, lane_ids=[lane_id], x_min=x-range, x_max=x, direction=forward` | lane + team 双重剪枝 |

### 3.6 对车道数量的兼容性

索引结构是 `Dictionary[int, Array]`，车道数量对它透明：

- 5 车道：5 个 key
- 6 车道：6 个 key
- 任意 N 车道：N 个 key

`radius_around` 确定涉及车道的逻辑：

```gdscript
func _affected_lanes(center_lane: int, radius: float) -> PackedInt32Array:
    var result: PackedInt32Array = []
    var lane_spacing := _get_lane_spacing()
    for lane_id in _by_lane.keys():
        if absf(_lane_y(lane_id) - _lane_y(center_lane)) <= radius:
            result.append(lane_id)
    return result
```

6 车道场景下 radius 较大时可能跨 3 条车道，但查询逻辑完全一致。

### 3.7 与现有系统的集成点

SpatialIndex 作为 `BattleSubsystemHost` 的内部服务，对外通过 `BattleManager` 暴露查询接口：

```
现有调用链：
  zombie_root._find_attack_target()
    -> get_parent().get_children()  [O(n) 全量遍历]
  projectile_root._find_segment_hit_target()
    -> battle.get_runtime_entities()  [O(n) 全量遍历]
  DetectionRegistry._scan_enemies()
    -> battle.get_runtime_combat_entities()  [O(n) 全量遍历]

替换后：
  所有调用点
    -> battle.spatial_query(query_params)
      -> SpatialIndex.query(params)
        -> _resolve_candidate_pool()  [分组索引剪枝]
        -> _apply_filters()           [空间 + 高度过滤]
```

数据源注意：`zombie_root._find_attack_target()` 当前直接用 `get_parent().get_children()`，不经过 `get_runtime_entities()`。迁移时必须统一改用 `battle.spatial_query()`，否则 rebuild 数据源不一致。

分阶段迁移策略：
1. 先实现 SpatialIndex + rebuild，不接入任何调用点，验证索引正确性
2. 接入最全面的调用点 `DetectionRegistry._scan_enemies()`（覆盖 team/lane/tags/direction/distance），用验证场景确认行为不变
3. 逐个迁移其他调用点

### 3.8 验证策略

空间索引和分组索引是纯性能优化，不改变行为语义。验证方式：
- 现有验证场景全部通过（行为不变）
- 新增 benchmark 场景：50+ 实体对比有/无索引的 tick 耗时
- 新增索引一致性测试：对同一帧的查询结果与全量遍历结果对比，确认无遗漏

## 四、高度过滤统一

### 4.1 已有连续高度区间模型

不需要新增枚举。已有 `HeightBand` 资源（`scripts/core/defs/height_band.gd`）定义了连续高度区间：

| band_id | min_height | max_height | 用途 |
|---------|-----------|-----------|------|
| `ground_unit` | 0 | 24 | 普通地面单位 |
| `ground_unit_large` | 0 | 48 | 大型地面单位（Tall-nut 等） |
| `air_unit_low` | 18 | 44 | 低空飞行 |
| `air_unit_medium` | 28 | 72 | 中空飞行 |
| `air_unit_high` | 40 | 96 | 高空飞行 |

区间有重叠，碰撞时做 overlap 判断。这比离散枚举（GROUND / LOW_AIR / HIGH_AIR）更精确。

### 4.2 真正的缺口：过滤未统一

`HeightBand` 和 `ProjectileFlightProfile.max_hit_height` 已存在，但没有在所有碰撞/检测路径上统一使用：

- `DetectionRegistry._scan_enemies()` 不过滤高度
- `zombie_root._find_attack_target()` 不过滤高度
- `projectile_root` 碰撞检测部分使用 `max_hit_height`，但不走 HeightBand overlap
- 效果系统（`explode` 等范围效果）没有高度维度

### 4.3 方案：HeightBand overlap 成为标准过滤步骤

在所有空间查询路径中统一加入 HeightBand overlap 判断。攻击者的 effective_height_range 与目标的 height_band 有 overlap → 可命中。

SpatialIndex 的查询接口支持 height_range 过滤参数，与空间过滤一起执行。

### 4.4 实施节奏

- 与空间索引同步实施（P1），因为两者改动在同一查询路径上
- 不改动 HeightBand 资源定义和现有 archetype 中的 height_band 引用

## 五、碰撞层（暂不实施）

当前 team 过滤（同队不打同队）覆盖了 90% 的场景。只有同一个 team 内部出现碰撞差异需求时（目前不存在），才需要声明式碰撞矩阵。暂不实施，等实际需求驱动。

## 六、场地变形

### 6.1 问题

Doom-shroom 坑洞、屋顶斜面等场景要求格子属性在战斗中途被修改和恢复。当前 `BoardSlot` 初始化后不变。

### 6.2 方案：Slot Modifier

在 `BoardSlot` 上增加运行时 modifier 数组。modifier 可以临时覆盖 slot_type、增删 placement_tags、设置有效期。每 tick 倒计时，到期自动移除，slot 恢复原样。

Doom-shroom 爆炸时在目标格子上添加一个 crater modifier，移除放置标签并设持续 18 秒。到期后 modifier 自动消失，格子恢复可放置。

改动范围仅 `BoardSlot` 一个文件加 `BattleBoardState._on_game_tick()` 调用倒计时。

### 6.3 实施节奏

远期（P3）。等 Doom-shroom 等需要场地变形的实体开发时再实施。现阶段确保新代码不假设"slot 属性不可变"即可。

board_slot_configs = [
    # 车道 0-1: ground（默认，无需配置）
