# 仿真与时序层计划

> 状态：正式计划
> 更新：2026-05-09

## 结论

仿真与时序层当前主模型正确，不需要重构。保留 100Hz 固定时间步、帧累加器、最大步数保护、线性 `simulation_speed` 和全局暂停。近期只补一个低成本能力：tick 预算监控。

## 现状

已有能力：

| 能力 | 说明 |
|------|------|
| 固定步进 | `GameState.simulation_tick_hz = 100`，`fixed_dt = 0.01` |
| 中央调度 | `BattleManager.step_simulation_tick()` 推进 EventBus、mode、runtime nodes、validation |
| 最大步数保护 | `_max_steps_per_physics_frame = 5`，防止死亡螺旋 |
| 速度控制 | `GameState.simulation_speed` 线性缩放 |
| 暂停 | `GameState.is_simulation_paused` 全局暂停 |
| 调试事件 | 已记录 `manual_step`、`max_step_clamp` 等 simulation event |

这些能力已经足够支撑当前规则验证和前期玩法开发。

## 立即实施：tick 预算监控

### 目标

记录每个 simulation tick 的实际耗时，在超过阈值时通过 DebugService 记录告警，帮助定位实体数量增加后的性能问题。

### 推荐阈值

| 字段 | 默认值 | 说明 |
|------|--------|------|
| `tick_budget_ms` | `10.0` | 100Hz 固定步进的理论预算 |
| `warning_threshold_ms` | `8.0` | 提前告警阈值 |
| `critical_threshold_ms` | `10.0` | 超预算阈值 |

### 实现位置

在 `BattleManager.step_simulation_tick()` 中包裹计时：

```gdscript
var started_usec := Time.get_ticks_usec()
# 原 step_simulation_tick 内容
var elapsed_ms := float(Time.get_ticks_usec() - started_usec) / 1000.0
if elapsed_ms >= warning_threshold_ms:
  DebugService.record_simulation_event(&"tick_budget_warning", GameState.get_simulation_snapshot(), {
    "elapsed_ms": elapsed_ms,
    "warning_threshold_ms": warning_threshold_ms,
    "critical_threshold_ms": critical_threshold_ms
  })
```

### 节流要求

避免每 tick 刷日志：

- warning 级别每 0.5 秒最多记录一次。
- critical 级别可以每次记录，但批量验证输出应汇总。

### 验证

- 新增或扩展一个验证场景，确认启用监控时 debug payload 中存在 tick budget 字段。
- 不要求固定性能数值，因为 CI 和本地机器差异较大。

## 暂缓事项

### 帧间插值

当前规则验证不依赖视觉平滑。插值需要实体维护 previous/current transform，并在 `_process` 中按 accumulator alpha 渲染，会扩大改动面。

暂缓到视觉表现层进入正式质量阶段后再做。

### 慢动作 / 变速曲线

大多数 gameplay 减速不应通过全局 `simulation_speed` 实现，而应走 per-entity status：

- 冻结：`movement=false` 或 `movement_scale=0`
- 减速：`movement_scale<1`
- 攻击暂停：`controllers=false`

全局慢动作属于表现或关卡导演需求，当前不做通用曲线系统。

### 暂停粒度

全局暂停足够支持当前验证。暂停战斗但 UI 继续响应、暂停仿真但特效继续播放等需求应等 UI/视觉层成熟后再设计。

未来可考虑：

```gdscript
PAUSE_SIM
PAUSE_INPUT
PAUSE_UI
PAUSE_VISUAL
```

当前不实现。

## 与其他计划的关系

- SpatialIndex 接入后，tick 预算监控用于观察查询优化收益。
- liveness 接入后，不应通过暂停仿真表达实体睡眠、埋地、冻结等行为。
- 对象池实施前后，tick 预算监控可用于比较高频 projectile 场景的耗时变化。

## 验收标准

- `step_simulation_tick()` 超过阈值时记录 `tick_budget_warning` 或等价 DebugService 事件。
- 正常验证场景不产生大量重复告警。
- 现有验证矩阵通过。

## 不做事项

- 不重写中央 step 架构。
- 不引入 variable timestep。
- 不把 status / entity liveness 建模为全局时间缩放。
