# 空间索引与高度过滤计划

> 状态：正式计划
> 更新：2026-05-09
> 适用阶段：前期开发环境，允许一次性替换旧查询 API。

## 目标

用统一的 SpatialIndex 替代热路径中的全量扫描，并把 HeightBand overlap 变成检测、碰撞、范围效果的标准过滤步骤。

SpatialIndex 的职责是“实体在哪里、属于哪些基础分组”。它不判断实体能不能被打、能不能受伤、是否参与胜败统计。业务语义由调用方通过 `filter: Callable` 注入。

## 当前问题

当前多条热路径都在全量遍历：

| 调用点 | 当前问题 |
|--------|----------|
| DetectionRegistry `_scan_enemies()` | 每次索敌遍历所有 combat entities |
| ZombieRoot `_find_attack_target()` | 每帧每僵尸遍历兄弟节点 |
| ProjectileRoot `_find_terminal_hit_target()` | 每投射体遍历 runtime entities |
| ProjectileRoot `_find_segment_hit_target()` | 每投射体扫掠碰撞遍历 runtime entities |
| ControllerRegistry sweep / bite | 控制器内部继续全量扫描 |
| EffectRegistry `enemies_in_radius` / `dispel_flying` | 范围效果和 tag 查询没有索引 |
| LawnMower proximity / sweep | 每帧遍历 combat entities |

当实体数量上升到 50+ 时，这些路径会形成 O(n²) 压力。

## 设计选择

### 使用车道 + X 轴排序，不使用四叉树

PVZ-like 战场具有强结构：

- 车道离散。
- 大部分查询限制在同车道或少数相邻车道。
- X 轴是主要运动方向。
- 实体规模通常是几十级别。

因此 lane 分组 + x 排序比四叉树更简单、更确定，也更符合 KISS。

### 合并空间索引与分组索引

热路径查询通常同时需要 team、lane、tag、kind、x/radius、height 过滤。拆成 EntityRegistry + SpatialIndex 两层会增加接口复杂度。

SpatialIndex 统一维护：

```gdscript
var _all: Array = []
var _by_team: Dictionary = {}
var _by_lane: Dictionary = {}
var _by_tag: Dictionary = {}
var _by_kind: Dictionary = {}
var _snapshot_version: int = 0
```

不维护：

```gdscript
var _combat_active
```

活跃性由调用方 filter 处理。

## 实体数据源

SpatialIndex rebuild 不能直接索引 `get_runtime_entities()` 的所有返回值，因为该接口还包含 economy、board、card、mode_host 等子系统节点。

推荐数据源：

1. 优先使用 `BattleManager.get_runtime_combat_entities()`。
2. rebuild 内部仍严格过滤：
   - 必须是 `Node2D`。
   - 必须有 `entity_kind` / `team` / `lane_id` 等可查询字段之一。
   - 必须是战斗可查询实体，排除纯 subsystem 节点。

Projectile 是否纳入索引由实际查询需求决定。若投射体需要被其他效果查询，纳入；否则只作为查询发起者，不作为候选目标。

## Rebuild 时机

采用每 tick 一次强制 rebuild，不使用 dirty flag。

原因：

- 投射体和僵尸可能每 tick 移动，dirty 几乎总是 true。
- N=50 到 80 的排序成本可接受。
- 强制 rebuild 更可预测，便于验证。

推荐 tick 生命周期：

1. 战斗初始化后执行一次 rebuild，保证第一帧查询可用。
2. `game.tick` 触发链读取上一轮完成后的稳定快照。
3. `_step_runtime_nodes()` 推进实体位置。
4. tick 末尾 rebuild，生成下一 tick 使用的快照。

这样所有同一 tick 内查询看到同一版本，避免查询顺序影响结果。

## 查询 API

BattleManager 暴露：

```gdscript
func spatial_query(params: Dictionary) -> Array
```

SpatialIndex 实现：

```gdscript
func query(params: Dictionary) -> Array
```

参数：

| 参数 | 类型 | 说明 |
|------|------|------|
| `team_exclude` | StringName | 排除某 team |
| `team_include` | StringName | 只包含某 team |
| `lane_ids` | PackedInt32Array | 限定车道 |
| `tags_any` | PackedStringArray | 至少包含一个 tag |
| `tags_all` | PackedStringArray | 必须包含全部 tag |
| `kinds` | PackedStringArray | 限定 entity_kind |
| `center` | Vector2 | 半径或距离排序中心 |
| `radius` | float | 半径过滤 |
| `x_min` | float | X 范围下限 |
| `x_max` | float | X 范围上限 |
| `direction` | StringName | `forward` / `backward` / `both` |
| `height_range` | Vector2 | 查询方有效高度区间 |
| `filter` | Callable | 调用方业务过滤 |
| `sort_by_distance` | bool | 是否按距离排序 |
| `max_results` | int | 最大返回数量 |

禁止使用：

```gdscript
"combat_active_only": true
```

调用方示例：

```gdscript
battle.spatial_query({
  "team_exclude": source.team,
  "lane_ids": PackedInt32Array([source.lane_id]),
  "x_min": source_x,
  "x_max": source_x + scan_range,
  "direction": &"forward",
  "filter": func(entity): return entity.has_method("is_targetable") and entity.is_targetable(),
  "sort_by_distance": true,
  "max_results": 1
})
```

## 确定性排序

所有排序必须稳定：

1. 主排序键：距离、x 或调用方指定顺序。
2. 次排序键：`entity_id`。
3. 无 `entity_id` 时使用本次 rebuild 的插入序号。

同距离、同 x 时不能依赖 Dictionary key 顺序或 Godot 节点顺序。

## HeightBand 统一过滤

已有 `HeightBand` 和实体 `get_hit_height_range()`。本计划不新增高度枚举。

标准规则：

```text
查询方 height_range 与候选实体 get_hit_height_range() 有 overlap -> 通过
```

来源：

| 查询方 | height_range 来源 |
|--------|------------------|
| Projectile | 当前高度段或 `Vector2(0, max_hit_height)` |
| 近战攻击 | 攻击者自身 HeightBand 或攻击参数 |
| AOE 效果 | effect 参数，默认不限制高度 |
| Detection | detection 参数，默认不限制高度 |

SpatialIndex 只负责执行 overlap，不决定查询方应该用哪个高度区间。

## 调用点迁移

| 调用点 | 新查询语义 |
|--------|------------|
| DetectionRegistry | `filter = is_targetable()` |
| ProjectileRoot terminal hit | `filter = is_collidable()` + terminal hitbox/radius |
| ProjectileRoot segment hit | `filter = is_collidable()` + segment/hitbox |
| ZombieRoot attack target | `filter = is_targetable()`，实际伤害由 HealthComponent 再检查 `is_damageable()` |
| EffectRegistry damage radius | `filter = is_damageable()` |
| EffectRegistry dispel flying | `tags_any=["flying"]` + effect 专用 filter |
| ControllerRegistry sweep/bite | 根据 controller 语义选择 `is_targetable()` 或 `is_collidable()` |
| LawnMower sweep | `filter = is_collidable()` |

## 实施阶段

### Phase 1：SpatialIndex 核心

- 新增 `scripts/battle/spatial_index.gd`。
- 实现 rebuild、分组索引、lane x 排序。
- 实现 `query(params)` 与稳定排序。
- BattleManager 暴露 `spatial_query()`。
- 战斗初始化后和 tick 末尾 rebuild。

### Phase 2：一致性验证

- 新增只读验证工具或场景：对比 SpatialIndex 查询结果与全量扫描结果。
- 覆盖 team、lane、tag、kind、radius、x range、height_range。

### Phase 3：接入 DetectionRegistry

- `_scan_enemies()` 改为 `battle.spatial_query()`。
- 删除内部全量扫描逻辑。
- 保持 detection 策略输出顺序确定。

### Phase 4：迁移热路径

- ZombieRoot 攻击目标查询。
- ProjectileRoot terminal / segment hit 查询。
- ControllerRegistry 内部查询。
- EffectRegistry 范围目标查询。
- LawnMower proximity / sweep 查询。

### Phase 5：HeightBand 标准化

- 所有空间查询支持 `height_range`。
- ProjectileRoot 删除重复高度判断或收敛为查询后的精确确认。
- AOE / Detection 根据需要传入高度范围。

### Phase 6：清理旧路径

- 删除热路径全量扫描回退。
- 保留 `get_runtime_combat_entities()` 作为 rebuild 数据源和 debug 工具。

## 验收标准

- 所有目标查询热路径都走 `battle.spatial_query()`。
- `rg "combat_active_only" scripts autoload` 无结果。
- SpatialIndex 不引用 `is_targetable()`、`is_damageable()`、`is_collidable()`、`is_combat_active()`。
- 同一验证场景多次运行目标选择结果一致。
- 完整验证矩阵通过。

## 后置事项

### 对象池

Projectile 对象池建议在 SpatialIndex 和 liveness 稳定后再做。池化实体需要明确 recycle 时如何清理 liveness override、状态、事件订阅和索引快照。

### 碰撞层声明

暂不引入 collision group。当前 team + liveness + HeightBand 足够覆盖近期需求。

### BoardSlot modifier

Doom-shroom 坑洞、屋顶斜面等场地变形需求出现时，再在 BoardSlot 层实现 modifier。当前只要求新代码不假设 slot 属性永久不变。
