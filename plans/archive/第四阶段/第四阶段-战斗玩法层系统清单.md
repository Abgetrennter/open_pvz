# 第四阶段-战斗玩法层系统清单

> 这份文档是第四阶段 `Epic A` 的正式收口产物。它不负责展开具体协议字段，而是先回答：第四阶段有哪些玩法系统域、哪些进入第一批、它们各自依赖什么、以及什么条件下才允许正式进入实现。

---

## 文档定位

这篇文档主要回答：

- 第四阶段战斗玩法层到底包含哪些系统域
- 哪些系统进入第一批，哪些继续后置
- 每个系统依赖当前哪些已冻结主干能力
- 每个系统最小验证入口应该落在哪里
- 每个系统何时才算满足“可以进入实现”

这篇文档不负责：

- 直接定义阳光、棋盘、卡片、波次的全部协议字段
- 直接实现完整玩法代码
- 提前引入 UI 表现层、拖拽交互和外循环流程
- 为状态系统和场上物件写第一轮详细草案

## 一句话结论

第四阶段战斗玩法层已经不缺“系统名字”，当前真正要做的是把 6 个系统域分成“第一批正式范围”和“后续批次”，并为前四个系统写清依赖和入场门槛。

结合当前主干状态，第一批范围为：

1. 阳光与资源
2. 棋盘、占位与放置
3. 卡片、费用、冷却与手牌
4. 波次、战局阶段与胜负条件

其中第一批完成前置准备后，建议优先进入实现的系统为：

1. 阳光与资源
2. 棋盘、占位与放置
3. 卡片、费用、冷却与手牌

`波次、战局阶段与胜负条件` 在第一批中先完成概念和验证入口预留，再作为下一顺位实现项进入。

## 当前主干可复用能力

第四阶段玩法层不是从零开始，它应建立在当前已经稳定的主干之上：

| 主干能力 | 当前载体 | 第四阶段的意义 |
|------|------|------|
| 场景与生成主链 | `BattleScenario` / `BattleSpawnEntry` / `BattleManager` / `EntityFactory` | 提供内容如何进入战局的现成入口，不需要重新发明实体生成流程 |
| 模板主链 | `EntityTemplate` / `ProjectileTemplate` / `TriggerBinding` | 提供植物、僵尸、投射物和行为模板的正式资源入口 |
| 规则协议守卫 | `ProtocolValidator` | 提供玩法层输入的运行前拦截能力，避免非法组合等到运行中才炸 |
| 事件链 | 当前事件总线与运行时日志 | 提供资源变化、放置成功/失败、出牌结果、波次推进等统一语义承载位 |
| 验证矩阵 | `wiki/03-content-validation/32-验证矩阵.md` | 提供每个新玩法系统的专项验证落点 |
| 批量回归 | `tools/validation_scenarios.json` / `tools/run_all_validations.ps1` | 提供第四阶段后续把专项场景纳入自动回归的统一入口 |

第四阶段玩法层应消费这些主干，而不是反向改写它们的职责边界。

## 系统域分组

### 第一批正式范围

| 系统域 | 当前为什么必须进入 | 当前为什么不先做完整 | 当前批次定位 |
|------|------|------|------|
| 阳光与资源 | 没有资源闭环，就没有战斗经济入口 | 动画、多资源体系、外循环经济都还不是主线 | 第一批草案，第一顺位实现 |
| 棋盘、占位与放置 | 没有放置空间模型，就没有玩家指令入口 | 原版 `PlantCell` 过重，完整地形规则会拖慢主线 | 第一批草案，第二顺位实现 |
| 卡片、费用、冷却与手牌 | 没有卡片层，放置行为缺正式上游 | 完整选卡界面、卡槽表现和卡池系统都还不该先做 | 第一批草案，第三顺位实现 |
| 波次、战局阶段与胜负条件 | 没有敌方节奏和结束条件，就还不算真正玩法层 | 完整 Adventure 流程和选关外层都明显后置 | 第一批草案，下一顺位实现 |

### 后续批次

| 系统域 | 当前为什么先不进入第一批主线 | 后续切入时机 |
|------|------|------|
| 状态系统 | 已落地 slowed / stunned 最小状态闭环，并验证其对移动与攻击的影响 | 后续再扩展更多状态类型与统一施加来源 |
| 场上物件与机关 | 已落地割草机代表性物件，并验证其与战局胜负连接点 | 后续再扩展脑子、障碍物和更多 board object |

---

## 当前明确不要做的事

第四阶段 `Epic A` 收口后，仍应明确克制下面几件事：

1. 不照搬 `MainGameManager`，不把资源、卡片、放置、波次、胜负重新塞回单个 Godot 节点。
2. 不照搬完整 `PlantCell`，先坚持薄 `BoardSlot / LaneSlot` 路线。
3. 不为了玩法层回退成植物、僵尸、投射物大枚举。
4. 不把卡牌 UI、拖拽 UI、阳光动画和菜单流程当成本批主线。
5. 不让玩法层重新定义实体生成方式、效果执行方式或命中空间模型。

---

## 系统依赖顺序

### 推荐建设顺序

1. 阳光与资源
2. 棋盘、占位与放置
3. 卡片、费用、冷却与手牌
4. 波次、战局阶段与胜负条件
5. 状态系统
6. 场上物件与机关

### 依赖关系图

| 系统域 | 依赖的既有主干能力 | 上游玩法依赖 | 最小验证入口 | 当前建议 |
|------|------|------|------|------|
| 阳光与资源 | 事件链、运行时快照、验证矩阵 | 无 | `sun_resource_validation` 或等价命名 | 第一批写草案后立即进入实现 |
| 棋盘、占位与放置 | 场景生成主链、实体模板主链、事件链、验证矩阵 | 无 | `board_placement_validation` 或等价命名 | 第一批写草案后进入实现 |
| 卡片、费用、冷却与手牌 | 模板主链、事件链、验证矩阵、协议守卫 | 阳光与资源、棋盘与放置 | `card_flow_validation` 或等价命名 | 第一批写草案后进入实现 |
| 波次、战局阶段与胜负条件 | 场景生成主链、模板主链、事件链、验证矩阵、协议守卫 | 无硬前置，但更适合在前三者后进入实现 | `wave_flow_validation` 或等价命名 | 第一批先完成草案与验证位，下一顺位实现 |
| 状态系统 | 事件链、模板主链、协议守卫 | 至少一个稳定的攻击/放置/波次主循环 | 待后续定义 | 暂不进入第一批主线 |
| 场上物件与机关 | 场景生成主链、棋盘抽象、战局阶段与胜负连接点 | 棋盘、战局阶段与胜负条件 | 待后续定义 | 暂不进入第一批主线 |

## 各系统入场门槛

### 1. 阳光与资源

当前为什么要先做：

- 它是战斗经济闭环的最小入口
- 它会直接影响卡片费用和出牌结果

进入实现前必须具备：

- 已明确资源状态归属对象
- 已明确 `sun.spawned / sun.collected / resource.changed / resource.spend_failed` 或等价事件语义
- 已明确资源数值是否进入统一快照和验证日志
- 已明确至少一个正向和一个负向验证口径

当前非目标：

- 阳光动画和拾取表现
- 多资源体系
- 商店与外循环经济

### 2. 棋盘、占位与放置

当前为什么要跟进：

- 它决定玩家指令如何进入战斗世界
- 它是卡片落地和后续场上物件接入的基础数据层

进入实现前必须具备：

- 已选定薄 `BoardSlot / LaneSlot` 或等价抽象
- 已明确占位语义和扩展位
- 已明确放置合法性校验属于哪一层
- 已明确占位成功 / 失败事件与最小验证口径

当前非目标：

- 完整原版地形规则
- 花盆、荷叶、墓碑、梯子等完整交互
- 完整拖拽 UI

### 3. 卡片、费用、冷却与手牌

当前为什么排在第三位：

- 它需要资源和放置两条链都已经讲清
- 它是玩家正式上游控制入口，但不应越级直接驱动实体实例化

进入实现前必须具备：

- 已明确 `CardDef` 或等价资源最小字段
- 已明确卡片和模板、资源消耗、放置请求之间的关系
- 已明确冷却计时归属对象
- 已明确手牌状态和失败语义
- 已明确正向 / 负向验证口径

当前非目标：

- 完整选卡界面
- 全套卡槽表现层
- 收藏、解锁和卡池管理

### 4. 波次、战局阶段与胜负条件

当前为什么进入第一批但不抢前三个实现位：

- 它决定一局战斗何时开始、推进和结束
- 但它比资源、棋盘、卡片更容易回退成大管理器流程拼装

进入实现前必须具备：

- 已明确 `WaveDef` 或等价资源最小结构
- 已明确最小战局阶段集合
- 已明确敌方生成入口和胜负条件归属
- 已明确无效波次定义的拦截方式
- 已明确正向 / 负向验证口径

当前非目标：

- 完整 Adventure 流程
- 多模式小游戏流程
- 菜单和选关 UI

### 5. 状态系统

当前已完成的最小结果：

- 已实现 `slowed / stunned` 两个代表状态
- 已实现状态施加、过期移除和可观测事件
- 已验证状态对移动与攻击的影响

后续继续扩展前建议具备：

- 保持当前主循环稳定
- 再选 1 到 2 个新增状态而不是一次性扩写
- 明确状态与效果系统、目标选择的进一步连接方式

### 6. 场上物件与机关

当前已完成的最小结果：

- 已实现场上物件运行时底座
- 已实现割草机代表性物件
- 已验证其与战局胜负连接点的基本闭环

后续继续扩展前建议具备：

- 继续保持与棋盘抽象和战局流程分层
- 再选脑子、障碍物等一个子域深入
- 明确哪些物件走 slot-bound、lane-bound 或 goal-bound

---

## 第一批结束后的正式实现范围

如果 `Epic A` 到 `Epic F` 的第一批前置准备全部完成，建议立刻进入实现的系统为：

1. 阳光与资源
2. 棋盘、占位与放置
3. 卡片、费用、冷却与手牌

同一时点不建议并发实现的内容：

- 波次、战局阶段与胜负条件
- 状态系统
- 场上物件与机关

原因是：

- 波次系统虽然已进入第一批草案范围，但更容易牵出大流程编排，应放在前三者跑通后接入
- 状态和物件都更适合作为主循环稳定后的代表性扩展
