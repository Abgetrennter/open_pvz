# 第四阶段第一批可执行任务清单

> 这份清单把“第四阶段前置准备”进一步拆成可直接认领的 `Epic / Issue / checklist`。它的目标不是马上铺开完整 PvZ-like 战斗层，而是先让第四阶段首批系统在不打散第三阶段冻结边界的前提下，具备正式开工条件。

---

## 文档定位

这篇文档主要回答：

- 第四阶段第一批现在可以直接开干的任务有哪些
- 每项任务的产出物是什么
- 每项任务依赖什么前置条件
- 每项任务完成后如何判断“算做完”

这篇文档不负责：

- 代替第四阶段完整路线图
- 一次性展开全部玩法系统协议
- 直接开始完整 UI、拖拽交互和表现层制作
- 把状态系统、场上物件与机关一起塞进第一批主线
- 把参考项目的 `MainGameManager` 式大一统结构搬回当前仓库

## 当前判断

结合当前 wiki 和仓库主干状态，第四阶段第一批应按下面的现实前提来理解：

1. 第三阶段第一轮冻结已经完成，项目当前主线已经从“继续扩写规则协议”转成“为玩法系统接入做前置准备”。
2. 当前主干已经有稳定的最小运行时底座：
   - `BattleScenario -> BattleSpawnEntry -> EntityTemplate -> EntityFactory`
   - `TriggerBinding / EffectDef / ProjectileTemplate`
   - `ProtocolValidator`
   - 批量回归入口和验证矩阵
3. 当前仓库还没有第四阶段四个主域的正式最小概念草案：
   - 阳光与资源
   - 棋盘、占位与放置
   - 卡片、费用、冷却与手牌
   - 波次、战局阶段与胜负条件
4. 当前验证矩阵也还没有为第四阶段玩法层专项预留正式位置。

因此，第四阶段第一批最合理的定义不是“先写玩法代码”，而是：

> 先完成系统清单、四份最小概念草案和玩法层验证入口预留，再锁定真正进入建设的首批系统。

结合 [第四阶段前置清单](./第四阶段前置清单.md) 的推荐顺序，这一批结束后建议正式进入建设的系统是：

1. 阳光与资源
2. 棋盘、格子、占位与放置
3. 卡片、费用、冷却与手牌

`波次、刷怪与战局流程` 仍属于第一批必须写完草案的范围，但更适合作为完成前置准备后的第二顺位实现项；`状态系统` 和 `场上物件与机关` 继续留在后续批次。

---

## 使用方式

建议把下面的条目按两层理解：

- `Epic`：一组需要连续推进的主任务
- `Issue`：可直接认领、可单独验收的任务

如果当前人力有限，推荐按下面顺序推进：

1. `Epic A` 首批范围与依赖收口
2. `Epic B` 阳光与资源最小概念草案
3. `Epic C` 棋盘、占位与放置最小概念草案
4. `Epic D` 卡片、费用、冷却与手牌最小概念草案
5. `Epic E` 波次、战局阶段与胜负条件最小概念草案
6. `Epic F` 验证矩阵与回归入口预留

这个顺序背后的原则是：

- 先锁定首批范围，再写各系统草案
- 先明确概念和边界，再进入玩法实现
- 先预留验证口，再避免玩法层再次回到“先做再补守卫”

---

## 第一批边界提醒

第四阶段第一批最容易失控的地方，不是“任务不够多”，而是把本该后置的玩法宇宙和表现层一起卷进来。

当前应坚持下面 5 条边界：

1. 第一批只正式覆盖：
   - 阳光与资源
   - 棋盘、占位与放置
   - 卡片、费用、冷却与手牌
   - 波次、战局阶段与胜负条件
2. `状态系统` 与 `场上物件与机关` 只保留 backlog 占位，不进入第一批主线。
3. 不先做卡牌 UI、拖拽交互、阳光动画和完整关卡外循环。
4. 不为第一批重新引入植物大枚举、僵尸大枚举或 `MainGameManager` 式硬编码协调器。
5. 每个 Issue 都必须同时回答：
   - 概念模型是什么
   - 依赖谁
   - 最小验证入口放在哪里

    - 波次、战局阶段与胜负条件
  - 每个能力域都至少写清：
    - 建议验证项名
    - 预期场景命名
    - 关键观察点
    - 当前状态为“预留”或等价状态
  - 不要求现在就把所有场景接入 `tools/validation_scenarios.json`，但要先把命名和观察点定下来
- checklist：
  - [x] 在验证矩阵新增第四阶段玩法层专项区块
  - [x] 为 `sun_resource_validation / board_placement_validation / card_flow_validation / wave_flow_validation` 或等价命名预留行
  - [x] 写清楚每行的关键观察点和正负向关注点
  - [x] 标明这些验证位在场景资源和批量回归中的后续接入顺序

## 第一批结束后的进入条件

这一批不要求现在就写出完整玩法代码，但至少应满足：

- 已有一份正式的战斗玩法层系统清单
- 已有 4 份最小概念草案：
  - 阳光与资源
  - 棋盘、占位与放置
  - 卡片、费用、冷却与手牌
  - 波次、战局阶段与胜负条件
- 已锁定首批正式进入建设的 3 个系统：
  - 阳光与资源
  - 棋盘、占位与放置
  - 卡片、费用、冷却与手牌
- 验证矩阵已为第四阶段玩法层预留专项位置
- `状态系统` 和 `场上物件与机关` 仍明确留在后续批次，不回流到当前主线
- 不会一进入第四阶段就回退成 `MainGameManager` 式大一统实现
