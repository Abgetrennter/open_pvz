# 第四阶段阶段总结

> 这份文档用于正式沉淀第四阶段（战斗玩法系统接入期）的实现结果。它回答的不是“第四阶段原本想做什么”，而是“当前仓库里第四阶段已经真正做成了什么、验证到了什么、还留下了哪些明确后续项”。

这篇文档主要回答：

- 第四阶段当前已经完成了哪些玩法系统

- 当前为什么可以认为第四阶段已经完成

## 一句话结论

第四阶段已经从“玩法层接入前置准备”推进到“核心战斗玩法系统已正式落地主干，并有专项验证和 demo 场景证明闭环”。

当前已经完成的核心结果包括：

1. 阳光与资源最小经济闭环已落地。
2. 棋盘、占位与放置已形成正式棋盘层，而不是继续散落在卡片或管理器特判里。
3. 卡片、费用、冷却与手牌已形成正式上游控制链，并接入资源与棋盘。
4. 波次、战局阶段与胜负条件已形成最小战局流程闭环。
5. 状态系统已用代表状态正式进入主干。
6. 场上物件与机关已用代表性物件正式进入主干。

因此当前项目已经不再处于“第四阶段玩法系统接入前置期”，而是：

> 第四阶段已完成，项目当前应转入第五阶段（错误技、扩展与工具期）。

## 第四阶段完成项总览

- `BattleEconomyState`

- `SunDropEntry`
- `BattleResourceSpendRequest`

## Formal Slot Type 与角色分层收口

第四阶段完成时，棋盘层已经不再只是“一个 slot 上放一个实体”。

当前已形成的正式语义包括：

- formal slot type 白名单：
  - `ground`
  - `water`
  - `roof`
  - `air_only`
- 角色分层：
  - `surface`
  - `primary`
  - `cover`
  - `blocker`
- 模板静态放置约束：
  - `placement_role`
  - `allowed_slot_types`
  - `required_placement_tags`
  - `granted_placement_tags`
  - `required_present_roles`
  - `required_empty_roles`

这意味着第四阶段结束时，棋盘层已经具备：

- `surface + primary` 共存
- `primary + cover` 共存
- `blocker` 拦截其他角色
- `roof` 需先有 `surface` 容器
- `air_only` 只接受匹配模板

## Demo 场景结果

第四阶段结束时，仓库里已经有正式 demo 场景：

- [`scenes/demo/demo_level.tscn`](../scenes/demo/demo_level.tscn)
- [`scenes/demo/demo_level.tres`](../scenes/demo/demo_level.tres)

当前 demo 已覆盖的代表性能力包括：

## 第四阶段完成定义核对

回看第四阶段目标，当前已经满足：

- [x] 至少 2 到 3 个核心战斗玩法系统正式进入建设
- [x] 它们和引擎核之间的边界是清楚的
- [x] 没有直接退回“大管理器 + 裸脚本”实现方式
- [x] 验证矩阵开始并持续覆盖战斗玩法层

并且已经超出原最小目标，额外完成了：

- 状态系统的代表性接入
- 场上物件与机关的代表性接入
- demo 场景沉淀

因此当前可正式判定：

> 第四阶段已完成。

## 当前未完成但明确后续的内容

第四阶段完成不等于整个项目完成。当前仍明确留待后续阶段的主线包括：

- 更丰富的错误技样例与组合验证
- 扩展包 / 数据包入口与兼容策略
- 更强的调试与可视化工具
- 选关、存档、图鉴、商店、花园等外循环系统
- 更多状态、物件和玩法模式扩展

这些内容应进入第五阶段任务单，而不是继续混在第四阶段里扩写。
