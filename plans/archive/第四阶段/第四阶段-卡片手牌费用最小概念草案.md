# 第四阶段-卡片手牌费用最小概念草案

> 这份草案用于回答第四阶段第一批里“卡片如何成为正式的玩家上游入口”。目标不是先做界面，而是先把 `CardDef`、手牌状态、扣费和冷却的最小闭环定下来。

---

## 文档定位

这篇文档主要回答：

- `CardDef` 最小字段应包含什么
- 卡片如何连接模板、资源消耗和放置请求
- 手牌状态由谁持有
- 冷却由谁计时
- 出牌失败时的最小语义是什么

这篇文档不负责：

- 完整选卡界面
- 全套卡槽表现层
- 收藏、解锁和卡池系统

配套阅读建议：

- [第四阶段-战斗玩法层系统清单](./第四阶段-战斗玩法层系统清单.md)
- [第四阶段-阳光与资源最小概念草案](./第四阶段-阳光与资源最小概念草案.md)
- [第四阶段-棋盘占位放置最小概念草案](./第四阶段-棋盘占位放置最小概念草案.md)

## 一句话结论

当前建议把卡片层正式定义为 `CardDef + HandState + BattleCardState` 三段结构：`CardDef` 通过 `entity_template_id` 指向内容模板，`HandState` 持有选择状态，`BattleCardState` 持有冷却状态；出牌时先过冷却和棋盘合法性，再在放置确认阶段发起资源扣费，成功后才启动冷却。

---

## 最小概念模型

### 1. `CardDef`

当前建议 `CardDef` 或等价资源至少包含下面字段：

| 字段 | 作用 |
|------|------|
| `card_id` | 稳定卡片标识 |
| `display_name` | 展示名，允许后续 UI 读取 |
| `entity_template_id` | 卡片对应的实体模板入口 |
| `sun_cost` | 当前第一批固定使用的阳光费用 |
| `cooldown_seconds` | 冷却时长 |
| `placement_tags` | 对放置层提供最小放置元数据 |

当前建议：

- 卡片默认通过 `entity_template_id` 连接 `EntityTemplate`
- 不让 UI 直接拿模板资源去实例化实体

### 2. `HandState`

当前建议 `HandState` 或等价运行时状态负责：

- 当前战斗中可用的卡片顺序
- 当前是否存在 `selected_card_id`

第一批建议：

- 先用轻量有序列表表达手牌，不急着做固定视觉槽位系统

### 3. `BattleCardState`

当前建议为每张卡维护最小运行时状态：

- `cooldown_remaining`
- 最近一次出牌请求结果

当前建议：

- 冷却计时归属卡片运行时状态，不归属 UI

---

## 出牌最小流程

当前建议的最小控制链为：

`card selected -> placement intent -> board legality pass -> economy spend -> placement commit -> entity.spawned -> cooldown started`

当前建议的关键判断顺序：

1. 选中卡片不扣费，也不启动冷却。
2. 如果卡片处于冷却中，直接拒绝出牌。
3. 如果目标格非法，直接拒绝出牌，不进入扣费。
4. 只有在放置合法且准备提交时，才发起资源消耗。
5. 成功生成后启动冷却。

这样做的原因是：

- 资源失败和放置失败都不该靠回滚 UI 状态修复
- 选卡状态与真实出牌提交应分层

---

## 最小事件语义

当前建议最少使用下面 4 个事件：

| 事件名 | 何时触发 | 关键字段建议 |
|------|------|------|
| `card.selected` | 玩家进入持卡待放置状态时 | `card_id` |
| `card.play_requested` | 发起正式出牌请求时 | `card_id`、`lane_id`、`slot_index` |
| `card.play_rejected` | 因冷却、资源不足或非法格而失败时 | `card_id`、`reason` |
| `card.cooldown_started` | 出牌正式提交成功后 | `card_id`、`cooldown_seconds` |

当前建议：

- 资源不足时，同时会伴随 `resource.spend_failed`
- 非法格时，同时会伴随 `placement.rejected`
- 第一批不急着补 `card.cooldown_finished` 等完整事件宇宙

---

## 状态分层建议

当前建议把“选中卡片”和“等待放置结果”视为同一条上游控制链中的两个阶段，但不要压成一个模糊布尔值。

推荐最小状态：

- `idle`
- `selected`
- `committing`

失败后的处理建议：

| 失败原因 | 当前建议 |
|------|------|
| `on_cooldown` | 直接 `card.play_rejected`，不触发棋盘或资源动作 |
| `insufficient_resource` | 由资源层拒绝，返回 `card.play_rejected`，不触发实体生成 |
| `invalid_placement` | 由棋盘层拒绝，返回 `card.play_rejected`，不触发扣费 |

当前第一批不建议：

- 出牌失败后仍然启动冷却
- 非法格失败后先扣费再退款

---

## 最小验证口径

### 最小正向验证

建议预留：

- `card_flow_validation.tres`

至少覆盖下面 4 个观察点：

1. 卡片被选中后出现 `card.selected`
2. 合法出牌请求出现 `card.play_requested`
3. 资源成功扣除并出现 `resource.changed(reason=spend)`
4. 最终出现 `placement.accepted -> entity.spawned -> card.cooldown_started`

### 最小负向验证

至少覆盖下面 2 个观察点之一：

1. 冷却中再次出牌时出现 `card.play_rejected(reason=on_cooldown)`
2. 资源不足时出现 `resource.spend_failed` 与 `card.play_rejected(reason=insufficient_resource)`

两种失败都不应继续出现：

- `placement.accepted`
- `entity.spawned`

---

## 当前明确不做的事

1. 不先做完整选卡 UI。
2. 不让 UI 节点直接持有卡片真状态。
3. 不让卡片层绕过棋盘和资源层直接实例化实体。
4. 不先做收藏、解锁和卡池系统。

---

## 进入实现前必须具备

- 已定义 `CardDef` 最小字段集
- 已明确卡片通过 `entity_template_id` 连接模板
- 已明确手牌状态与冷却状态的归属对象
- 已明确出牌的最小判断顺序
- 已明确 `card.play_rejected` 的失败原因口径
- 已在验证矩阵中预留 `card_flow_validation` 位置
