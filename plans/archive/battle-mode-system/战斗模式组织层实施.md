# 战斗模式组织层实施路线图 v1

## Summary

目标是在不继续大改 `core runtime / archetype 编译链` 的前提下，为当前 `battle` 底盘补一层正式的“模式组织层”，让项目能够稳定承载更多 **保留 PVZ 棋盘/格子/车道语义** 的风味小游戏与模式变体，并且不把差异继续塞进 `BattleManager`、`BattleFlowState`、`InputRouter` 或各个 battle state。

本路线图按 **7 个阶段** 推进。实现完成后，项目会得到一套可扩展的 mode 框架：`BattleScenario` 继续定义对局资源，新增 `BattleModeDef / BattleRuleModule / BattleInputProfile / BattleObjectiveDef / BattleModeHost` 负责 mode 身份、规则、输入和目标。第一版优先支持“规则偏转型小游戏”，不追求自由移动或脱离棋盘的 action substrate。

## Important Interfaces

### 新增资源与运行时对象

- `scripts/battle/mode/battle_mode_def.gd`
  - 定义一个模式的身份、分类、默认输入、默认目标、默认规则模块。
- `scripts/battle/mode/battle_rule_module.gd`
  - mode 的最小规则单元，仅描述规则和参数，不直接承担整局主控。
- `scripts/battle/mode/battle_input_profile.gd`
  - 定义一个 mode 中允许哪些交互动作。
- `scripts/battle/mode/battle_objective_def.gd`
  - 定义 mode 专属胜负条件。
- `scripts/battle/mode/battle_mode_host.gd`
  - battle 运行时唯一新增 host，负责解析 mode_def、组织 rule module、驱动 mode-level runtime state 和 objective 评估。

### 现有对象新增字段

- `scripts/battle/battle_scenario.gd`
  - 新增：
    - `mode_def: Resource`
    - `mode_rule_modules: Array`
    - `objective_override: Resource`
    - `input_profile_override: Resource`
- `scripts/input/input_router.gd`
  - `setup(...)` 新增 `input_profile` 参数。
- `scripts/battle/battle_subsystem_host.gd`
  - 在标准子系统后创建并 setup `BattleModeHost`。
- `scripts/battle/battle_manager.gd`
  - 仅新增 mode host 获取和生命周期转发，不引入 mode 分支逻辑。
- `scripts/battle/battle_flow_state.gd`
  - 第一版保留现有 `battle_goal / defeat_conditions`，与 `BattleObjectiveDef` 并存，不立即替换。

## Phase Plan

### Phase 1：模式层骨架与目录落位

目标：把模式层正式落到仓库结构中，但不改变现有 battle 行为。

实施内容：

- 新建目录：
  - `scripts/battle/mode/`
  - `data/combat/modes/`
- 新建最小资源类：
  - `battle_mode_def.gd`
  - `battle_rule_module.gd`
  - `battle_input_profile.gd`
  - `battle_objective_def.gd`
- 新建最小运行时 host：
  - `battle_mode_host.gd`
- `battle_mode_host.gd` 第一版只做：
  - 读取 scenario 上的 `mode_def`
  - 合并 scenario 级 override
  - 暴露调试快照
  - 不执行任何规则
- `BattleSubsystemHost` 增加：
  - `_mode_host` 字段
  - `get_mode_host()`
  - `reset_runtime_services()` 中创建/销毁/初始化 mode host
- `BattleManager` 增加只读访问器：
  - `get_mode_host()`

完成信号：

- 所有现有场景在没有 mode_def 时行为完全不变。
- `BattleModeHost` 能在 debug snapshot 中出现。
- 全量运行时不新增 mode 分支副作用。

### Phase 2：Scenario 接入 mode 定义

目标：让 `BattleScenario` 成为 battle 资源入口，`BattleModeDef` 成为 mode 规则入口，二者形成稳定关系。

实施内容：

- 在 `battle_scenario.gd` 增加字段：
  - `mode_def`
  - `mode_rule_modules`
  - `objective_override`
  - `input_profile_override`
- 在 `BattleModeHost.setup()` 中建立合并规则：
  - 先取 `scenario.mode_def`
  - 再叠加 `scenario.mode_rule_modules`
  - `objective_override` 覆盖 `mode_def.objective_def`
  - `input_profile_override` 覆盖 `mode_def.input_profile`
- 建立第一版模式合并结果结构：
  - `resolved_mode_id`
  - `resolved_input_profile`
  - `resolved_objective_def`
  - `resolved_rule_modules`
- 在 `BattleModeHost` 中暴露：
  - `get_resolved_input_profile()`
  - `get_resolved_objective_def()`
  - `get_resolved_rule_modules()`
  - `get_mode_runtime_snapshot()`

完成信号：

- `BattleScenario` 可以在不影响现有内容资源的前提下声明 mode。
- mode 定义、scenario 覆盖、最终生效结果三者可被调试观察。
- 不需要 mode 的 battle scenario 继续零改动工作。

### Phase 3：InputRouter 模式化

目标：把当前 `InputRouter` 从“默认只服务标准选卡/放卡”提升为“由 mode 配置解释输入”，但不重写整个输入系统。

实施内容：

- `BattleInputProfile` 第一版字段固定为：
  - `enable_card_select`
  - `enable_card_place`
  - `enable_slot_click`
  - `enable_entity_click`
  - `enable_slot_drag`
  - `enable_swap`
  - `enable_manual_skill`
  - `enable_rhythm_hit`
  - `enable_cancel`
  - `input_tags`
- `InputRouter.setup(...)` 增加 `input_profile: Resource = null`
- 在 `InputRouter` 中新增 gating 逻辑：
  - 无 profile 时，完全保留现有行为
  - 有 profile 时，仅在对应开关启用时发射相关 `input.action.*`
- 第一版不新增复杂输入事件树，只补两个新入口：
  - `request_entity_click(entity_id, metadata = {})`
  - `request_slot_drag(from_lane, from_slot, to_lane, to_slot, metadata = {})`
- 第一版仅把这些新入口打通到 `EventBus`，不强制立刻被 mode 消费
- `BattleModeHost` 负责把 `resolved_input_profile` 注入给 `InputRouter`

完成信号：

- 标准模式行为不变。
- 非标准模式可以通过 input profile 关闭“选卡/放卡”，并保留其他交互。
- `InputRouter` 不新增针对具体小游戏的 if/switch。

### Phase 4：RuleModule 执行骨架

目标：建立“规则模块是 mode 变化的唯一主要承载点”的执行通路。

实施内容：

- `BattleRuleModule` 只作为 Resource 定义：
  - `module_id`
  - `display_name`
  - `enabled`
  - `priority`
  - `params`
  - `tags`
- `BattleModeHost` 中增加模块生命周期：
  - `on_mode_setup`
  - `on_battle_start`
  - `on_game_tick`
  - `on_event`
  - `on_mode_teardown`
- 运行时不要直接实例化每个 module 为 Node；第一版采用：
  - `BattleModeHost` 内部维护 module handler registry
  - `module_id -> callable/handler`
- 新建 `scripts/battle/mode/battle_mode_module_registry.gd`
  - 类似轻量 registry，不进 autoload
- 第一批只实现 2 类无侵入 module：
  - `conveyor_cards`
  - `manual_entity_skill`
- 同时为后续预留但不实现：
  - `slot_swap`
  - `combo_score`
  - `vase_populator`
  - `izombie_role_flip`
  - `bowling_lane_limit`

完成信号：

- mode rules 有统一执行入口。
- 新规则不需要继续写进 `BattleManager` 或各 state。
- 第一批 module 能够被 host 调度但不破坏现有 battle。

### Phase 5：Objective 层正式化

目标：把小游戏的胜负条件从当前固定 `battle_goal / defeat_conditions` 中分离出来，但保持兼容。

实施内容：

- `BattleObjectiveDef` 第一版字段固定为：
  - `objective_id`
  - `objective_type`
  - `params`
  - `failure_conditions`
  - `summary_tags`
- 第一版支持的 objective type：
  - `all_waves_cleared`
  - `survive_duration`
  - `protect_template`
  - `score_threshold`
  - `combo_threshold`
  - `clear_special_targets`
  - `collect_resource`
  - `defeat_named_spawn`
- 在 `BattleModeHost` 中新增 objective evaluator：
  - 若无 objective_def，继续完全走现有 `BattleFlowState + WaveRunner`
  - 若有 objective_def，则由 `BattleModeHost` 追加 mode-level 完成/失败判定
- 保持原则：
  - `BattleFlowState` 仍是唯一 battle phase 状态机
  - `BattleModeHost` 只调用 `mark_victory()` / `mark_defeat()`
  - 不在第一版重写 `WaveRunner` 的旧判定逻辑
- 新增调试字段：
  - `objective_progress`
  - `objective_completed`
  - `objective_failed`

完成信号：

- 标准战斗仍可只靠旧 `battle_goal`。
- mode 战斗可额外挂 objective_def。
- 新小游戏不用把 score/combo 目标硬塞回 `BattleFlowState`。

### Phase 6：第一个试点 mode

目标：不是追求完整小游戏，而是用一个 mode 验证这套架构足够承载未来风味小游戏。

推荐试点：`manual_aim_lane_battle`

原因：

- 保留棋盘 / 车道 / archetype / projectile / effect
- 只偏转交互和局内规则
- 能一次验证 4 个新增对象协同是否成立

实施内容：

- 新增一个 mode 资源：
  - `data/combat/modes/manual_aim_lane_battle.tres`
- 它包含：
  - `BattleInputProfile`
    - 允许实体点击 / manual skill
    - 可关闭标准 card select，或仅允许少量卡
  - `BattleObjectiveDef`
    - 例如限定击杀数 / 限时 survive / 击败指定敌人
  - `BattleRuleModule[]`
    - `manual_entity_skill`
- 新增一个试点 scenario：
  - `scenes/validation/mode_manual_aim_lane_validation.tres`
- 第一版试点目标不要做“真正手动弹道瞄准”，而做：
  - 选中植物
  - 点目标 lane/slot
  - 触发 mode 规则发射一次专属攻击
- 这能避免第一版就引入复杂拖拽/连击/UI 大改

完成信号：

- mode_def 能真实驱动一个非标准 battle。
- 不需要改 `BattleManager` 主循环。
- mode 特性只通过 input profile / rule module / objective 生效。

### Phase 7：验证、治理与文档收口

目标：让模式层不是“只在代码里存在”，而是成为受治理、可验证、可继续扩展的正式结构。

实施内容：

- 新增验证：
  - `mode_manual_aim_lane_validation`
  - `mode_input_profile_guard_validation`
  - `mode_objective_score_validation`
  - 至少 1 条 module-level guardrail
- 更新：
  - `tools/validation_scenarios.json`
  - `wiki/03-content-validation/32-验证矩阵.md`
  - `wiki/01-overview/23-当前阶段与实现路线.md`
  - `wiki/04-roadmap-reference/26-开发路线图.md`
  - `agent.md`
- 文档规范：
  - mode 相关草案留在 `plans/` 根目录，不进入 archive 索引
  - mode 正式化后再提升到 `wiki/` 正文
- 为未来扩展预留：
  - `BattleModeRuntimeState`
  - `BattleHUDProfile`
  - `BoardRuleLayer`
  - 但不进入 v1 实施范围

完成信号：

- 新增模式层有正式验证入口。
- 文档与实现口径一致。
- 后续新增小游戏不会再次回到“先往 BattleManager 塞特判”的路径。

## Test Plan

实施完成后至少覆盖以下测试与场景：

- 兼容性回归
  - 现有标准 battle scenario 不挂 `mode_def` 时，行为必须与当前一致。
  - `card_flow_validation`
  - `board_slot_tag_validation`
  - `battlefield_phase6_validation`
  - `wave_flow_validation`
- 输入层验证
  - 关闭 `enable_card_place` 后，标准点格不再触发放卡。
  - 开启 `enable_entity_click` 后，能发出 mode 语义事件。
  - `enable_cancel` 在 mode 间切换行为正确。
- 规则模块验证
  - mode host 能按 priority 装配规则模块。
  - 同一 scenario 的追加 module 能覆盖/扩展 mode_def 默认 module 集。
- 目标层验证
  - 无 `objective_def` 时旧 `battle_goal` 逻辑不变。
  - 有 `objective_def` 时 mode 可附加 score/combo/collect 目标。
- 试点 mode 验证
  - `manual_aim_lane_battle` 能完成最小闭环。
  - 相关输入、规则、目标都通过 scenario 验证。
- guardrail
  - 缺失 `mode_id`
  - 非法 `objective_type`
  - 非法输入 profile 组合
  - 未注册 `module_id`
  - 都应在协议层被拦下，而不是静默回退

## Assumptions

- 范围限定为“PVZ 棋盘/格子/车道底盘上的风味小游戏”，不处理自由移动 action substrate。
- 第一版 mode 层是 battle 上层组合器，不替换现有 `BattleScenario / BattleFlowState / InputRouter`，只在其上叠加。
- `BattleRuleModule` 第一版采用 host + registry 分发，不做复杂脚本化插件系统。
- 第一个试点 mode 目标是验证架构，而不是直接做最复杂的迷你游戏。
- mode 草案文档应保留在 `plans/` 下而不是 archive；正式稳定后再提升到 `wiki/`。
