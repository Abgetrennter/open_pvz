# 战斗模式层架构草案 v1

> 文档性质：架构草案
> 所属圈层：战斗玩法层之上的模式组织层
> 状态：待评审
> 适用范围：以 PVZ 棋盘/格子/车道/实体语义为底盘的风味小游戏与模式变体

---

## 1. 背景

当前仓库已经具备稳定的 battle substrate：

- `BattleScenario` 提供对局资源入口
- `BattleBoardState / BattleCardState / BattleEconomyState / BattleFlowState / WaveRunner` 提供标准战斗子系统
- `BattlefieldPreset` 提供棋盘尺寸与格子语义
- `InputRouter` 提供标准“选卡 -> 点格子 -> 放置”输入链
- `Archetype + Mechanic[]` 提供实体能力表达
- 验证体系已经能长期守住这些路径

这意味着项目已经很擅长描述：

- 植物、僵尸、投射体、场上物件
- 放置、资源、状态、生命周期、投射物、波次
- 规则执行与验证

但当前仍缺一层正式组织结构去承载“更多带其他游戏风味的 PVZ 小游戏”，例如：

- 传送带变体
- 手动技能释放变体
- 锤僵尸/节奏点击类模式
- 格子交换/连锁触发类模式
- 限时挑战、Boss Rush、限定 roster 模式

当前如果直接把这些需求继续塞进：

- `BattleManager`
- `BattleFlowState`
- `InputRouter`
- 各个 battle state

很容易重演原版 `Challenge` God Class，或走向 vendor 参考实现那种“巨型关卡 Resource + 布尔组合爆炸”的问题。

因此需要补一层正式的 **battle 模式层**，但这层必须满足两条硬约束：

1. 不推翻现有棋盘/格子/PVZ 战斗底盘。
2. 不继续对 core runtime / archetype 编译链大动刀。

---

## 2. 设计目标

### 2.1 目标

本草案目标是让项目可以稳定承载：

- 多个 PVZ 风味小游戏 / mode
- mode 间规则差异的资源化表达
- mode 规则与 battle 子系统的清晰边界
- mode 专属目标、输入、附加规则的可验证接入

### 2.2 非目标

本草案不试图支持：

- 完全自由移动的动作游戏 substrate
- 脱离棋盘和 slot 语义的纯 2D action / STG 引擎
- 统一所有未来小游戏到“零成本配置”
- 立即替换当前 `BattleScenario` / `BattleFlowState` / `InputRouter`

---

## 3. 参考项目给出的启示

### 3.1 原版 PVZ 的正面启示

来自 [23-原版PVZ模式系统逆向分析](E:/Code/open_pvz/wiki/04-roadmap-reference/23-原版PVZ模式系统逆向分析.md) 的可借鉴点：

- 模式与场景应分层理解：`GameMode` 不等于 `GameScene`
- 模式差异本质上是对同一 battle 底盘做规则注入
- 模式规则至少会影响：
  - 视觉/棋盘
  - 战术/波次
  - 经济/种子供给
  - 模式专属物件与交互
  - 运行时 Update/输入解释

### 3.2 原版 PVZ 的反面启示

不要复刻：

- 单一 `Challenge` 巨类
- 散落各处的 `if (IsXMode())`
- 用模式枚举贯穿全系统分支

### 3.3 vendor 参考实现的正面启示

来自 [39-vendor参考实现-游戏模式分析](E:/Code/open_pvz/wiki/04-roadmap-reference/39-vendor参考实现-游戏模式分析.md) 的可借鉴点：

- 模式差异尽量用数据声明
- 同一 battle 框架可以承接多种 mode
- 关卡资源和模式参数应尽量编辑器可见

### 3.4 vendor 参考实现的反面启示

不要复刻：

- 单个 `ResourceLevelData` 承载所有模式参数
- `is_zombie_mode / is_pot_mode / is_bowling_stripe` 这类布尔组合爆炸
- 模式逻辑分散进 20+ 个文件靠布尔开关互相缠绕

## 4. 总体方案

### 4.1 新增层级

在当前 battle 子系统之上新增一层：

```text
BattleScenario
  ├── BattlefieldPreset          # 现有：棋盘/格子语义
  ├── WaveDef / CardDef / Spawn  # 现有：对局资源
  └── BattleModeDef              # 新增：这一局是哪种 mode
        ├── BattleInputProfile   # 新增：这一局允许哪些交互
        ├── BattleObjectiveDef   # 新增：这一局如何赢/输
        └── BattleRuleModule[]   # 新增：这一局叠加哪些特殊规则
```

这层不替代 battle substrate，而是作为其上层组合器。

### 4.2 模式层的职责边界

模式层负责：

- 声明“这是哪种小游戏/变体”
- 声明“开启哪些附加规则”
- 声明“玩家能做哪些交互”
- 声明“这一局如何结算”

模式层不负责：

- 替代 archetype / mechanic
- 重写投射体、状态、生命周期、命中模型
- 重写棋盘基础语义
- 让 `BattleManager` 变成模式 God Object

## 5. 新增对象

## 5.1 `BattleModeDef`

建议路径：

- `scripts/battle/mode/battle_mode_def.gd`

建议职责：

- 唯一标识一个 mode
- 提供 mode 的高层说明
- 挂接输入、目标、规则模块

建议字段：

```gdscript
class_name BattleModeDef
extends Resource

@export var mode_id: StringName = StringName()
@export var display_name := ""
@export_multiline var description := ""
@export var category: StringName = &"challenge"

@export var input_profile: Resource = null          # BattleInputProfile
@export var objective_def: Resource = null          # BattleObjectiveDef
@export var rule_modules: Array[Resource] = []      # BattleRuleModule[]

@export var ui_profile_id: StringName = StringName()
@export var tags: PackedStringArray = PackedStringArray()
```

说明：

- `category` 不是 gameplay 逻辑关键字，只用于组织与 UI。
- `ui_profile_id` 当前只是预留，不要求先实现完整 HUD profile。
- `rule_modules` 是 mode 的核心扩展点。

---

## 5.2 `BattleRuleModule`

建议路径：

- `scripts/battle/mode/battle_rule_module.gd`

建议职责：

- mode 规则的最小可组合单元
- 向 battle 注入特殊逻辑，但不篡改底层架构

建议字段：

```gdscript
class_name BattleRuleModule
extends Resource

@export var module_id: StringName = StringName()
@export var display_name := ""
@export var enabled := true
@export var priority := 100
@export var params: Dictionary = {}
@export var tags: PackedStringArray = PackedStringArray()
```

建议配套运行时接口：

```gdscript
class_name BattleModeHost
extends Node

func setup(battle: Node, scenario: Resource, mode_def: BattleModeDef) -> void
func teardown() -> void
```

RuleModule 不直接继承 Node。  
真正的运行时分发由 `BattleModeHost` 完成。

`BattleModeHost` 负责：

- 合并 mode_def 与 scenario 的规则模块
- 按 priority 初始化模块
- 给模块提供统一 battle 上下文
- 转发生命周期钩子与 event bus 事件

建议模块可挂接的生命周期：

- `on_setup`
- `on_start`
- `on_tick`
- `on_event`
- `on_teardown`

但第一版不要做成复杂脚本接口树，建议先以：

- 资源声明
- host 读取参数
- 少量内置 module 运行时实现

这种方式起步。

### 推荐的第一批模块类型

- `conveyor_cards`
- `manual_entity_skill`
- `slot_swap`
- `combo_score`
- `vase_populator`
- `izombie_role_flip`
- `bowling_lane_limit`
- `hazard_slot`
- `spawn_on_objective`

---

## 5.3 `BattleInputProfile`

建议路径：

- `scripts/battle/mode/battle_input_profile.gd`

建议职责：

- 定义当前 mode 中哪些输入动作有效
- 约束 `InputRouter` 的解释方式

建议字段：

```gdscript
class_name BattleInputProfile
extends Resource

@export var profile_id: StringName = StringName()

@export var enable_card_select := true
@export var enable_card_place := true
@export var enable_slot_click := true
@export var enable_entity_click := false
@export var enable_slot_drag := false
@export var enable_swap := false
@export var enable_manual_skill := false
@export var enable_rhythm_hit := false
@export var enable_cancel := true

@export var input_tags: PackedStringArray = PackedStringArray()
```

这层不要一开始就做“动作游戏控制器”。

它解决的是：

- 这个 mode 里玩家允许什么交互
- 哪些输入需要被屏蔽
- `InputRouter` 应把鼠标/键盘解释成什么语义动作

也就是说，当前更合适的定位是：

> 交互权限模型，而不是角色操控模型。

---

## 5.4 `BattleObjectiveDef`

建议路径：

- `scripts/battle/mode/battle_objective_def.gd`

建议职责：

- 把 mode 专属胜负条件从 `BattleFlowState` 的固定字段里提升出来

建议字段：

```gdscript
class_name BattleObjectiveDef
extends Resource

@export var objective_id: StringName = StringName()
@export var objective_type: StringName = &"all_waves_cleared"
@export var params: Dictionary = {}
@export var failure_conditions: PackedStringArray = PackedStringArray()
@export var summary_tags: PackedStringArray = PackedStringArray()
```

建议支持的第一批 objective type：

- `all_waves_cleared`
- `survive_duration`
- `protect_template`
- `score_threshold`
- `combo_threshold`
- `clear_special_targets`
- `collect_resource`
- `defeat_named_spawn`

注意：

- 第一版不要替换掉现有 `battle_goal / defeat_conditions`
- 先做“兼容叠加”

即：

- 若 scenario 未挂 objective_def，继续走旧路径
- 若挂了 objective_def，则由 mode host / objective evaluator 解释

---

## 5.5 `BattleModeHost`

这是本草案里唯一建议新增的 battle 运行时 node。

建议路径：

- `scripts/battle/mode/battle_mode_host.gd`

建议职责：

- 解析 `BattleScenario.mode_def`
- 管理 mode runtime 生命周期
- 持有 mode-level runtime state
- 让 mode 规则不要直接污染 `BattleManager`

推荐接入点：

- 在 [battle_subsystem_host.gd](E:/Code/open_pvz/scripts/battle/battle_subsystem_host.gd) 中，在标准子系统创建后追加创建 `BattleModeHost`
- 顺序建议放在 `BattleFlowState` 与 `WaveRunner` 之后

原因：

- 它依赖 board/card/flow 等子系统已存在
- 但它不应该替代这些子系统

## 6. 与现有架构的接法

## 6.1 `BattleScenario`

在 [battle_scenario.gd](E:/Code/open_pvz/scripts/battle/battle_scenario.gd) 上追加：

```gdscript
@export var mode_def: Resource = null
@export var mode_rule_modules: Array = []
@export var objective_override: Resource = null
@export var input_profile_override: Resource = null
```

含义：

- `mode_def` 是该对局默认 mode
- `mode_rule_modules` 允许 scenario 临时追加规则模块
- `objective_override` / `input_profile_override` 允许 scenario 局部覆盖 mode 默认配置

这样不会推翻 `BattleScenario`，而是给它补一个 mode 入口。

---

## 6.2 `BattleFlowState`

当前 [battle_flow_state.gd](E:/Code/open_pvz/scripts/battle/battle_flow_state.gd) 已经承载：

- phase
- battle_goal
- defeat_conditions
- survival_duration
- protected_template_id

建议：

- 第一版不重写它
- 让 `BattleModeHost` 额外驱动 mode objective 的判断
- `BattleFlowState` 继续作为 battle 会话主状态机

也就是说：

- `BattleFlowState` 保持 battle-level state
- `BattleObjectiveDef` 负责 mode-level win/lose interpretation

---

## 6.3 `InputRouter`

当前 [input_router.gd](E:/Code/open_pvz/scripts/input/input_router.gd) 已经很适合作为 mode 输入解释层的起点。

建议第一版改法：

- `InputRouter.setup(...)` 新增 `input_profile: BattleInputProfile = null`
- 通过 input profile 控制：
  - 是否允许选卡
  - 是否允许点格子
  - 是否允许 cancel
  - 是否解释 entity click / swap / rhythm hit

第一版不要重做整个输入系统。  
只要让 `InputRouter` 从“默认只服务标准 card/board 输入”升级为“能被 BattleInputProfile 配置”即可。

---

## 6.4 `BattleManager`

当前 [battle_manager.gd](E:/Code/open_pvz/scripts/battle/battle_manager.gd) 已经很大。

因此原则上：

- 不把小游戏逻辑继续写进它
- 让它仍保持 orchestrator 身份

建议它只新增两件事：

1. 暴露 `get_mode_host()`
2. 在 reset / ready 生命周期里调用 `BattleModeHost.setup() / teardown()`

除此之外不继续加小游戏 if/switch。

---

## 7. 推荐的最小增量实现顺序

### Phase 1

只加：

- `BattleModeDef`
- `BattleInputProfile`
- `BattleModeHost`

目标：

- mode 可以声明
- input 可以按 mode 配置
- 不改变现有 battle 目标系统

### Phase 2

新增：

- `BattleRuleModule`
- 第一批内置 rule module

目标：

- 做出第一个真实的小游戏风味 mode

### Phase 3

新增：

- `BattleObjectiveDef`

目标：

- 把小游戏的胜负规则从固定 `battle_goal` 中解耦出来

## 8. 第一个试点 mode 建议

推荐不要直接做最复杂的模式。

建议第一个试点是：

### `manual_aim_lane_battle`

特征：

- 仍然保留棋盘/格子/车道
- 仍然使用 archetype / projectile / effect
- 玩家可以选中植物并触发手动技能或指定目标 lane/slot
- 有 mode 专属输入 profile
- 有 mode 专属 objective

它可以一次性验证：

- `BattleModeDef`
- `BattleInputProfile`
- `BattleRuleModule`
- `BattleObjectiveDef`

是否真的能协同工作。

---

## 9. 为什么这版草案足够但不至于过度设计

这版只新增 4 个资源对象 + 1 个 host：

- `BattleModeDef`
- `BattleRuleModule`
- `BattleInputProfile`
- `BattleObjectiveDef`
- `BattleModeHost`

它的好处是：

- 能支撑很多“PVZ 风味小游戏”
- 不要求底层继续重构
- 不强迫未来所有模式零成本配置
- 不制造新的 God Class
- 不把 mode 差异继续塞回 `BattleManager`

这符合当前项目的真实目标：

> 能支持更多风味模式，而且不至于混乱。

---

## 10. 后续扩展边界

如果以后小游戏越来越多，再考虑是否继续拆：

- `BattleModeRuntimeState`
- `BattleHUDProfile`
- `BoardRuleLayer`

但这些都不应该在 v1 先上。

v1 的关键是先把 mode 的身份、规则、目标、输入这 4 件事正式化。

---

## 11. 结论

结合当前架构与两份参考文档，本草案的核心判断是：

1. 不再对 core runtime 大动刀。
2. 在 battle 上层新增正式模式层。
3. 用 `mode + rule modules + objective + input profile` 承接未来更多风味小游戏。
4. 让 battle manager 继续做 orchestrator，而不是 mode God Object。

如果后续实现这版方案，项目将得到一条很清晰的增长路径：

- 主线 PVZ battle 继续稳定
- 风味小游戏通过 mode 层接入
- 两者共享实体能力、状态、投射体、验证体系
- 但不会互相污染实现结构
