# 帧率与仿真时间重构路线图草案

- 状态：草案
- 目标：在不破坏现有验证基线的前提下，将 Open PVZ 的规则时间从可变 delta / 低频语义 tick 逐步迁移到 100Hz 固定仿真帧。
- 关联草案：
  - [帧率与仿真时间设计草案](wiki-futures/帧率与仿真时间设计草案.md)
  - [战斗距离语义化重构路线图草案](战斗距离语义化重构路线图草案.md)
  - [帧率设计对比分析-de-pvz-vs-PVZ-Godot-Dream](../帧率设计对比分析-de-pvz-vs-PVZ-Godot-Dream.md)

---

## 路线判断

当前不需要一次性重写所有战斗系统。现有 `BattleManager`、`GameState`、`EventBus`、`TriggerRegistry`、`ControllerComponent` 和 projectile movement 已经形成可运行主链。

真正需要解决的是：

1. `GameState.current_time` 当前由渲染 `_process(delta)` 推进。
2. `game.tick` 当前由 `tick_interval = 0.25` 驱动，频率约 4Hz，不适合作为统一仿真 tick。
3. 连续行为、周期触发器、状态持续、波次和验证 deadline 缺少同一套权威时间。
4. 少量运行时逻辑仍使用墙钟时间，影响暂停、倍速和确定性。
5. 原版 100Hz 数值迁移缺少正式承载层。

因此路线应是渐进式：

> 先确立时间口径，再引入固定仿真时钟，随后迁移 gameplay 时间消费者，最后收口倍速、验证和 guardrail。

---

## 阶段 0：冻结当前口径

目标：先把现有时间字段和回调语义讲清楚，不急着改行为。

任务：

- 梳理现有时间入口：
  - `BattleManager._process(delta)`
  - `BattleManager._physics_process(delta)`
  - `GameState.advance_time(delta)`
  - `BattleManager.tick_interval`
  - `TriggerRegistry` 的 interval / start_delay
  - `ControllerRegistry` 中的 `Time.get_ticks_msec()`
  - validation auto quit / deadline
  - UI cooldown / visual feedback / Tween
- 明确当前 `tick_interval = 0.25` 是 legacy 语义 tick，不是最终 simulation tick。
- 文档声明：规则时间不应长期依赖渲染 delta。
- 选定 6 到 8 个代表性验证样本：
  - 基础射击链
  - proximity 触发
  - 波次刷新
  - 卡片冷却
  - 状态持续
  - 投射物命中
  - 割草机 sweep
  - validation deadline

退出信号：

- 文档口径完成。
- 不改变任何验证结果。
- 开发者能区分 legacy 语义 tick、Godot physics tick、渲染帧和未来 simulation tick。

## 阶段 1：引入仿真时钟字段

目标：新增固定仿真时间状态，但保持现有行为尽量不变。

建议文件与对象：

- 优先扩展 `autoload/GameState.gd`
- 如 `GameState` 开始膨胀，再拆出 `scripts/core/runtime/simulation_clock.gd`

新增字段候选：

- `simulation_tick_hz := 100`
- `fixed_dt := 0.01`
- `current_tick := 0`
- `simulation_speed := 1.0`
- `is_simulation_paused := false`

新增方法候选：

- `reset_simulation_time()`
- `advance_simulation_tick()`
- `set_simulation_speed(speed)`
- `set_simulation_paused(paused)`
- `get_simulation_snapshot()`

实现口径：

- `current_time` 由 `current_tick * fixed_dt` 派生。
- 迁移期可以保留 `advance_time(delta)`，但标记为 legacy 或仅用于兼容。
- `begin_battle()` 重置 `current_tick` 和 `current_time`。
- Debug snapshot 增加 tick 信息。

不做：

- 不改变 `game.tick` 发射频率。
- 不改变 physics fps。
- 不迁移所有系统。
- 不引入完整 replay。

验证：

- 运行 smoke/core validation。
- 增加一个最小 clock smoke：
  - reset 后 `current_tick = 0`
  - advance 1 tick 后 `current_time = 0.01`
  - advance 100 tick 后 `current_time = 1.0`

退出信号：

- 新字段稳定存在。
- 现有验证结果不变。
- 时间 debug 可观察。

## 阶段 2：建立固定仿真驱动

目标：让 `game.tick` 由固定 100Hz simulation tick 驱动。

建议改动：

- 将 `game.tick` 从 `_process(delta)` 的 `tick_interval` 累积器迁出。
- 在 `BattleManager._physics_process` 或专门 runtime driver 中推进仿真 tick。
- 设置项目 physics tick 为 100Hz。
- 每个仿真 tick 发出：

```gdscript
tick_event.core["tick_index"] = GameState.current_tick
tick_event.core["fixed_dt"] = GameState.fixed_dt
tick_event.core["game_time"] = GameState.current_time
```

兼容策略：

- 保留 `tick_interval` 字段一段时间，但不再作为 `game.tick` 间隔。
- 如果需要旧 4Hz 行为，新增明确事件或 subsystem interval，而不是复用 `game.tick`。
- 对 validation 预期事件数量做最小必要调整。

实施约束：

- 如果阶段 2 选择 Godot `_physics_process` 作为临时 tick source，必须明确它只是 1x 交互运行的过渡形态。
- 在阶段 4 支持 2x、pause、单步或单渲染帧消费多个 tick 前，需要引入中央 gameplay step 驱动，或提供等价机制让实体、投射物、controller、field object 等连续行为按每个仿真 tick 执行。
- 不允许只让 `BattleManager` 多发 `game.tick`，而实体 `_physics_process` 仍只按 Godot 回调次数运行；这会导致离散事件和连续运动脱节。

关键风险：

- 当前 `periodically` 和 `proximity` 会从 4Hz 变为 100Hz 评估。
- interval 判断能阻止过度触发，但 detection 评估成本会上升。
- 部分 validation 如果只期望“至少一次 tick”应不受影响；如果断言次数范围，需要更新。

验证：

- `game.tick` 事件携带 tick 信息。
- `GameState.current_time` 与 `current_tick` 对齐。
- 运行基础射击链、proximity、波次、卡片、状态验证。
- 检查 validation 输出中 tick 事件数量变化是否符合预期。

退出信号：

- `game.tick` 成为 100Hz 权威仿真事件。
- 现有核心行为可通过验证。
- 未发现因 tick 频率提高导致的事件风暴或性能问题。

## 阶段 3：迁移规则时间消费者

目标：把玩法逻辑从渲染 delta、墙钟时间和隐式物理 delta 迁移到仿真时间。

优先迁移点：

1. `GameState.current_time`
   - 移除 `_process(delta)` 中的权威推进。
   - `current_time` 只由 fixed tick 推导。
2. `TriggerRegistry`
   - interval / start_delay 继续读 `game_time`，但来源变为 simulation time。
   - 可选：内部用 tick 比较提升稳定性。
3. `ControllerRegistry`
   - 替换 `Time.get_ticks_msec()` 为 `GameState.current_time` 或 `current_tick`。
   - controller strategy 接收的 `delta` 语义明确为 fixed dt。
4. Projectile movement
   - movement component 使用 fixed dt。
   - lifetime 使用 simulation time 或 fixed dt 累积。
5. Entity movement / attack cooldown
   - cooldown 按 fixed dt 或 tick 减少。
   - death visual fade 仍可保留 `_process(delta)`，因为它是表现层。
6. Runtime delayed callbacks
   - `EffectDelayRunner` 这类延迟效果不能继续用 `_process(delta)` 决定玩法事件时机。
   - 延迟投射物、延迟爆炸、burst interval 等应由 simulation time 或 tick queue 驱动。
7. Collectible gameplay delay
   - `SunCollectible` 的 auto collect delay 会影响资源事件，应迁移到 simulation time。
   - 纯视觉漂浮、淡出、吸附动画仍可留在显示层。
8. Validation tracker
   - deadline、auto quit 的规则部分读 simulation time。
   - 进程退出延迟可继续用真实时间或单独 UI/runtime delay。

不迁移：

- debug overlay 刷新频率。
- UI hover / drag / card bar 视觉动画。
- visual feedback / particles / Tween。
- 纯显示层死亡淡出，除非它影响验证或实体活动状态。

验证：

- 暂停或低渲染帧率下，规则结果稳定。
- 改变渲染 FPS 不改变事件顺序和最终状态。
- 投射物 lifetime、攻击 cooldown、状态 duration 与 simulation time 一致。

退出信号：

- 玩法结果不依赖 `_process(delta)`。
- 墙钟时间不再决定战斗规则。
- 连续行为 `delta` 语义统一为 fixed dt。

## 阶段 4：倍速、暂停与单步

目标：建立正式 simulation speed / pause 协议，替代规则层对 `Engine.time_scale` 的依赖。

新增能力：

- `simulation_speed`
- `simulation_paused`
- `step_simulation_ticks(count)`
- 可选 debug 快捷键或验证参数。

运行模式：

| 模式 | 行为 |
|------|------|
| 交互 1x | 每真实秒消费 100 tick |
| 交互 2x | 每真实秒消费 200 tick |
| 暂停 | 不消费 simulation tick，UI 可继续 |
| 单步 | 手动推进指定 tick |
| headless validation | 固定步进或高上限补帧 |

累积器 guardrail：

- `max_accumulated_time`，建议从 0.2s 起步。
- `max_steps_per_frame`，交互运行建议较低，headless 可较高。
- 超出上限记录 DebugService warning。

不做：

- 不把 `Engine.time_scale` 作为正式规则倍速。
- 不要求视觉动画和规则倍速完全同步。
- 不引入复杂 replay UI。

验证：

- pause 后规则事件停止增长。
- UI/debug 可继续响应。
- 2x 下 1 秒真实时间约消费 200 tick。
- 单步推进 1 tick 只发出 1 个 `game.tick`。

退出信号：

- 倍速和暂停协议可被 validation 观察。
- `Engine.time_scale` 从规则层剥离。
- 单步调试可用。

## 阶段 5：原版数值迁移支持

目标：让原版时间数值可以直接或简单映射到 Open PVZ 内容层。

建议新增迁移口径文档或 helper：

- `classic_tick_hz = 100`
- `classic_tick_to_seconds(ticks) = ticks / 100.0`
- `seconds_to_simulation_ticks(seconds) = round(seconds * 100.0)`

与距离语义化联动：

- 原版 `px_per_tick` 不直接变成 world speed。
- 先换算为 `slots_per_sec`。
- 再由 `BattlefieldMetrics` 转换为 world unit。

示例：

```text
original_attack_interval_ticks = 150
attack_interval_seconds = 1.5
attack_interval_ticks = 150

original_move_px_per_tick = 0.4
speed_slots_per_sec = 0.4 / 80.0 * 100.0
```

任务：

- 在原版植物迁移文档中加入时间字段映射表。
- 为 archetype / mechanic 编写约定补充 tick 口径。
- 对至少一个普通射手、一个普通僵尸、一个投射物 profile 做样本迁移记录。

验证：

- 样本实体在 100Hz 下触发频率符合预期。
- 原版 tick 映射不随渲染 FPS 变化。

退出信号：

- 原版时间数值迁移有正式公式。
- 内容作者知道何时使用 tick、seconds 或 slots_per_sec。

## 阶段 6：收口与守卫

目标：避免时间语义重新分散。

可选 guardrail：

- 扫描 gameplay 目录中新增的 `Time.get_ticks_msec()` / `Time.get_ticks_usec()`。
- 扫描 battle/runtime 中直接使用 Timer 节点驱动规则的新增代码。
- ProtocolValidator 对不合理 interval 做提示。
- validation report 输出 simulation clock 信息。
- DebugService 记录 tick overrun、max step clamp、pause/speed change。

文档更新：

- 更新 `AGENTS.md` 中运行与验证部分。
- 更新 runtime protocol wiki。
- 更新原版实体迁移路线。
- 更新调试与验证指南。

不建议：

- 立即禁止所有 `_process(delta)`。
- 把视觉表现层强行改成 fixed tick。
- 自动批量重写全部 delta 逻辑。
- 开放扩展包替换 simulation clock。

退出信号：

- 新 gameplay 代码默认使用 simulation time。
- 视觉代码和规则代码的时间边界清楚。
- 验证能发现典型误用。

## 风险与控制

| 风险 | 控制方式 |
|------|----------|
| `game.tick` 从 4Hz 变 100Hz 导致事件量上升 | interval 判断保留；先跑代表性验证；必要时优化 detection |
| validation 断言次数变化 | 区分 tick 事件数量变化和玩法事件变化，只更新必要规则 |
| `GameState` 膨胀 | 时钟职责保持薄；必要时拆出 `SimulationClock` |
| 视觉动画受影响 | 表现层继续使用 `_process(delta)`，只消费规则状态 |
| 卡顿后补帧过多 | 设置 accumulator 和 steps 上限，DebugService 记录 |
| 倍速改变玩法顺序 | 倍速只改变消费 tick 数，不改变单 tick 内事件顺序 |
| 原版迁移过度承诺 | 时间 tick 可直通，空间和动画仍通过独立语义层校准 |

## 新问题处理建议

| 问题 | 建议处理 |
|------|----------|
| Godot physics 配置字段名 | 实施前确认 Godot 4.x 项目配置实际键名，避免写错配置 |
| 100Hz 性能压力 | 先用核心验证和 profiler 观察；detection 可按 interval 或 dirty flag 优化 |
| 同 tick 多事件顺序 | 保持同步发射顺序；未来需要时引入显式 deferred queue |
| Timer 节点边界 | 规则层不新增 Timer；已有视觉 Timer 不迁移 |
| 输入采样 | UI 输入排队，simulation tick 统一消费 |
| 快进验证 | headless 模式可绕过真实时间，以固定步进推进到目标 tick |
| Debug snapshot 频率 | 从 frame interval 改为 tick interval 或支持两种模式 |

## 完成定义

这条路线完成时，应满足：

1. Open PVZ 的战斗规则由 100Hz 固定仿真 tick 推进。
2. 渲染 FPS、VSync、窗口大小和 UI 缩放不影响玩法结果。
3. `GameState.current_time` 与 `current_tick` 一致，可复现、可验证。
4. gameplay 连续行为使用统一 fixed dt。
5. 倍速、暂停和单步由 simulation clock 控制。
6. 原版 tick 数值可以直接或简单映射。
7. 视觉层仍可现代化、高刷新、异步表现，但不反向决定规则。
