# 战斗距离语义化重构路线图草案

- 状态：草案
- 目标：在不破坏现有验证基线的前提下，将战斗距离从裸 world unit 逐步迁移到棋盘语义单位。
- 关联草案：
  - [战斗距离与棋盘度量设计草案](wiki-futures/战斗距离与棋盘度量设计草案.md)
  - [帧率与仿真时间设计草案](wiki-futures/帧率与仿真时间设计草案.md)

---

## 路线判断

当前不需要重写坐标系统。现有 `BattleBoardState`、`BoardSlot`、`DetectionRegistry`、Projectile movement 已经能支撑稳定运行。

真正需要解决的是：

1. 距离数值缺少统一解释口径。
2. 内容资源里的 `scan_range`、`speed`、`impact_radius` 与棋盘尺度隐式耦合。
3. 未来 UI 现代化、棋盘 preset 差异和内容迁移会放大裸数值调参成本。

因此路线应是渐进式：

> 先集中度量，再新增语义字段，最后分批迁移内容。

---

## 阶段 0：冻结当前口径

目标：先把“当前裸数字是什么”讲清楚，不急着改运行时。

任务：

- 梳理现有距离字段：
  - `scan_range`
  - `impact_radius`
  - `collision_padding`
  - `speed`
  - `move_speed`
  - `detection_range`
  - `board_slot_spacing`
- 明确当前字段均解释为 legacy world unit。
- 在 wiki 或开发文档中写明：legacy world unit 不是 UI 像素。
- 选择 5 到 8 个代表性实体作为迁移样本：
  - 豌豆射手
  - 小喷菇 / 海蘑菇
  - 大喷菇
  - 土豆雷
  - 投手
  - 普通僵尸
  - 割草机

退出信号：

- 文档口径完成。
- 不改变任何验证结果。
- 新内容编写者知道裸数值属于 legacy 字段。

---

## 阶段 1：引入 BattlefieldMetrics 最小对象

目标：集中棋盘几何换算，但不改变现有行为。

建议文件：

- `scripts/battle/battlefield_metrics.gd`

建议形态：

- 独立运行时只读值对象。
- 优先实现为 `RefCounted`，暂不做 `.tres` Resource。
- 从 `BattlefieldPreset` / scenario 派生，不把运行时查询方法塞进 `BattlefieldPreset`。

最小职责：

- `slot_origin_x`
- `slot_spacing`
- `lane_y_positions`
- `playable_min_x`
- `playable_max_x`
- `slot_position(lane_id, slot_index)`
- `slots_to_world(slot_count)`
- `world_to_slot_index(x)`

接入点：

- `BattleBoardState.setup()` 从 scenario / battlefield preset 创建 metrics。
- `BattleBoardState.get_slot_world_position()` 内部可委托 metrics。
- 保持旧字段 `board_slot_origin_x`、`board_slot_spacing` 可用。
- lane Y 坐标迁移期优先读 preset / scenario，缺失时 fallback 到 `BattleManager.get_lane_y()`。
- 后续让 `BattleManager.get_lane_y()` 委托 metrics，而不是继续持有最终解释权。

不做：

- 不接管放置校验。
- 不接管实体扫描。
- 不发事件。
- 不读写 `GameState`。
- 不修改 Mechanic family。

验证：

- 运行 board placement 相关验证。
- 增加一个最小 metrics smoke 场景或现有 validation probe：
  - 给定 origin 和 spacing，slot 位置稳定。
  - 改变 spacing 后 slot 位置按预期变化。

退出信号：

- 现有验证全通过。
- 运行时行为与接入前一致。
- metrics 保持纯几何查询对象。

---

## 阶段 2：编译链支持语义距离字段

目标：让新内容可以写“几格远”，旧内容继续不变。

前置约束：

- 射程、半径和静态距离字段只依赖 `BattlefieldMetrics`，可以先接入。
- `speed_slots_per_sec`、`move_speed_slots_per_sec` 必须按固定仿真秒解释，应与 `SimulationClock.fixed_dt` 或等价权威时间同步接入。
- 在权威仿真时钟落地前，不建议批量迁移速度字段，避免“每秒”语义被可变 delta 或墙钟时间污染。

新增字段候选：

- `scan_range_slots`
- `impact_radius_slots`
- `collision_padding_slots`
- `distance_slots`
- `speed_slots_per_sec`
- `move_speed_slots_per_sec`
- `detection_range_slots`
- `range_mode = full_lane`

字段命名采用 `_slots`。不引入 `_board_units`，避免过度泛化。

编译策略：

- 如果存在语义字段，编译为 legacy runtime params：
  - `scan_range = metrics.slots_to_world(scan_range_slots)`
  - `impact_radius = metrics.slots_to_world(impact_radius_slots)`
  - `speed = metrics.slots_to_world(speed_slots_per_sec)`
- 如果不存在语义字段，继续使用现有 legacy 字段。
- `range_mode = full_lane` 保留为 targeting / detection 语义参数，由 detection strategy 结合 metrics 计算到 playable edge 的范围。
- 不允许用 `99999.0`、`4000.0` 这类超大数值模拟整行射程。
- `per_sec` 一律指 simulation second，不指 render delta、墙钟时间或 UI 动画时间。

关键边界：

- `.tres` 可以同时保留 legacy 字段和语义字段，但语义字段优先。
- ProtocolValidator 应阻止负数语义距离。
- 不在此阶段批量迁移所有资源。

验证：

- 新增一个语义距离 smoke archetype。
- 同一 battlefield 下语义字段编译结果应等价于预期 world unit。
- 改变 `board_slot_spacing` 后，语义字段对应的实际 world range 随 spacing 变化。
- 速度字段验证应固定 tick 数：给定 `speed_slots_per_sec` 和 `N` 个 fixed tick，最终位移稳定等于预期值。
- legacy 资源行为保持不变。

退出信号：

- 新旧字段并存稳定。
- 新内容可完全不写裸 `scan_range` / `speed`。
- 验证能证明语义字段与棋盘 spacing 绑定。

## 阶段 3：迁移代表性内容

目标：用少量高价值实体验证语义单位是否适合实际玩法。

优先迁移顺序：

1. 整行射手：
   - 豌豆射手
   - 寒冰射手
   - 双发射手
2. 短射程植物：
   - 小喷菇
   - 海蘑菇
   - 大喷菇
3. 近身触发植物：
   - 土豆雷
   - 魅惑菇
   - 缠绕海草
4. 投射物 profile：
   - `linear_ground`
   - `linear_fast_swept`
   - `parabola_arc`
   - `parabola_terminal_blast`
5. 移动实体：
   - 普通僵尸
   - 快速僵尸
   - 割草机

每次迁移规则：

- 一次只迁移一组语义相近的内容。
- 保留必要注释或迁移记录。
- 同步跑对应 validation。
- 不在迁移 commit 中做额外平衡调整。

验证：

- 基础射击命中。
- 短射程触发边界。
- 近身触发边界。
- 抛物线终点半径。
- 僵尸到达、啃咬和割草机扫线。

退出信号：

- 样本内容验证通过。
- 新字段表达没有明显歧义。
- 没有出现“为了迁移而改玩法”的情况。

---

## 阶段 4：扩大迁移并清理裸数值

目标：把语义字段变成正式内容默认写法。

任务：

- 批量扫描剩余裸距离字段。
- 只迁移运行时语义明确的字段。
- 对确实属于特效视觉偏移的字段保留 world unit。
- 更新 archetype 编写约定。
- 更新原版实体复刻工作流：
  - 原版 px 先换算为 slot 语义
  - 再由 metrics 转为 world unit
- 在验证矩阵中新增距离语义覆盖说明。

退出信号：

- 新增正式内容默认使用语义字段。
- legacy 字段只用于历史资源、视觉偏移或暂未迁移内容。
- 文档不再把战斗 world unit 误称为 UI 像素。

---

## 阶段 5：收口与守卫

目标：避免裸距离重新扩散。

可选 guardrail：

- ProtocolValidator 对新增正式资源提示优先使用语义字段。
- 工具脚本扫描新增 `.tres` 中的裸 `scan_range` / `speed`。
- 文档明确 legacy 字段允许范围：
  - 兼容旧资源
  - 特殊视觉偏移
  - 明确需要 world unit 的连续行为

不建议：

- 立即禁止所有 legacy 字段。
- 自动批量改写所有资源。
- 把 metrics 暴露为扩展包可覆盖的全局服务。

退出信号：

- 内容层新写法稳定。
- 验证层能发现典型误用。
- `BattlefieldMetrics` 仍保持纯几何对象，没有膨胀成 God Class。

## 风险与控制

| 风险 | 控制方式 |
|------|----------|
| `BattlefieldMetrics` 膨胀成 God Class | 限制为只读几何查询，不接管实体、规则、事件 |
| 迁移改变玩法手感 | 样本先行，每组迁移独立验证，不混入平衡调整 |
| 字段双轨长期存在 | 设置阶段 4 / 5 的清理目标，但不急于硬禁 |
| full lane 语义不清 | 明确它是 detection / targeting 语义，不是巨大 magic number |
| UI 现代化影响判定 | Camera/UI 适配只能消费 world position，不能反向改变 runtime distance |

## 新问题处理建议

这些问题不阻塞阶段 0 / 1，但应在阶段 2 前形成实现约束。

| 问题 | 建议处理 |
|------|----------|
| lane Y 配置 | `lane_origin_y + lane_spacing` 为默认；`lane_y_positions` 为显式覆盖，存在时优先 |
| playable 边界 | 字段拆开：`playable_min_x/max_x`、`plant_zone_min_x/max_x`、`spawn_x`、`projectile_despawn_x`；阶段 1 只强制 playable 与 despawn |
| X/Y 度量 | `slot_spacing` 只表示 X；Y 使用 `lane_spacing` 或 `lane_y_positions` |
| hitbox 语义化 | 玩法半径支持 `_slots`；`hitbox_size` 保持 world unit，未来再评估 `hitbox_size_slots` |
| 速度迁移 | 内容层移动和投射物使用 `*_slots_per_sec`；视觉 Tween、粒子、UI 动画不迁移 |
| `range_mode` 范围 | 第一阶段支持 `lane_forward` / `lane_backward` 的 `full_lane`；multi-lane 按每条 lane 复用；global track 不套用 full lane |
| 迁移样本 | 六组样本：整行射手、短射程、近身触发、投手、移动实体、割草机 |
| debug 输出 | metrics 几何参数进入 debug snapshot 或 validation report，不输出实体关系 |

## 额外遗漏与解决方案

1. 朝向与阵营语义：
   - 风险：`forward` 长期绑定 `+X` 会限制反向植物、僵尸侧玩法和镜像模式。
   - 方案：保留 `lane_forward` 语义，但由 direction policy 决定实际 X 方向；第一阶段默认植物向 `+X`。

2. 非均匀棋盘与屋顶坡度：
   - 风险：只用 `lane_y_positions` 无法表达随 X 变化的地面高度。
   - 方案：第一阶段只做 lane Y；预留 `ground_position_for(lane_id, x)` 查询名，未来屋顶或坡面再实现。

3. 地面位置、视觉位置与高度：
   - 风险：视觉偏移混入距离判定会导致 UI/动画影响规则。
   - 方案：索敌和距离默认使用 `get_ground_position()`；高度命中继续走 `hit_height_band` / projectile height；metrics 不处理 sprite 偏移。

4. 数据版本与迁移记录：
   - 风险：legacy 字段和 `_slots` 字段并存后，内容来源难以追踪。
   - 方案：建立迁移记录；Validator 先提示，不硬禁；必要时在验证报告记录距离语义版本。

5. 扩展包边界：
   - 风险：扩展包替换 metrics 算法会破坏核心验证稳定性。
   - 方案：扩展包可贡献 battlefield preset 数据和 `_slots` 内容字段，但不得替换核心 metrics 算法，除非未来正式开放 geometry slot。

6. 输入映射一致性：
   - 风险：输入层和规则层各自维护棋盘常量，点击格子与规则格子不一致。
   - 方案：BoardCoordinateMapper / BoardOverlay 消费 `BattleBoardState` 或 metrics 暴露的位置，不维护独立几何常量。

7. 浮点边界：
   - 风险：`slots_to_world()` 后的边界判定出现 off-by-one 风格问题。
   - 方案：检测和命中统一使用明确的 `<=` 口径；验证覆盖边界内、边界上、边界外。

8. 仿真时间依赖：
   - 风险：`*_slots_per_sec` 字段先于固定仿真时钟落地，导致同一内容在不同帧率下位移不同。
   - 方案：速度字段迁移依赖 `SimulationClock.fixed_dt` 或等价权威时间；射程/半径语义化可以先行，速度迁移后置到时间口径稳定后。

## 建议优先级

P1：

- 阶段 0 文档口径
- 阶段 1 metrics 最小对象

P2：

- 阶段 2 编译链语义字段
- 代表性内容 smoke validation
- 固定仿真时间接入后再迁移 `*_slots_per_sec`

P3：

- 阶段 3 样本内容迁移
- 阶段 4 文档和工作流更新

P4：

- 阶段 5 guardrail
- 工具脚本扫描和提示

## 完成定义

这条路线完成时，应满足：

1. 原版 `px` 被正式解释为设计单位参考，而非现代 UI 像素。
2. Open PVZ 运行时距离以稳定 world unit 执行。
3. 内容作者优先使用 slot 语义描述射程、半径和速度。
4. UI 缩放、窗口尺寸和素材分辨率不影响战斗判定。
5. `BattlefieldMetrics` 是纯几何度量对象，没有管理实体、事件和规则。
