# 第四阶段前置清单

> 这份清单不是第四阶段的完整实现计划，也不是把所有玩法系统一次性细化成协议字段。它的目标是回答：在第三阶段已经完成第一轮协议冻结之后，`Open PVZ` 进入第四阶段（战斗玩法系统接入期）之前，到底还要先想清哪些边界、依赖和最小实现顺序。

---

## 文档定位

这篇文档主要回答：

- 第四阶段真正要接入哪些系统域
- 这些系统域各自依赖当前哪些前置条件
- 哪些内容可以借鉴参考项目，哪些绝不能照搬
- 第四阶段开始前，最小需要准备哪些概念、协议、验证和资源入口

这篇文档不负责：

- 代替第四阶段完整路线图
- 直接展开所有玩法字段
- 提前实现 UI 包装、外循环和大型工具链
- 把参考项目的业务层管理器直接搬到当前仓库

## 一句话结论

第四阶段现在不缺“要做的系统名字”，真正缺的是一份不会把玩法层重新写回大管理器特判的进入清单。

因此第四阶段开始前最重要的不是：

- 赶紧做阳光 UI
- 赶紧做完整格子系统
- 赶紧做完整关卡流程

而是：

> 先把玩法层的系统清单、边界、依赖顺序和最小验证口径定下来，再让系统按顺序进入建设。

## 1. 参考项目真正提供了什么提醒

结合 [PVZ-Godot-Dream 架构总览](../wiki/04-roadmap-reference/pvz-godot-dream/01-架构总览.md)、[PVZ-Godot-Dream 管理器与全局服务](../wiki/04-roadmap-reference/pvz-godot-dream/04-管理器与全局服务.md) 和 [参考实现迁移策略](../wiki/04-roadmap-reference/25-参考实现迁移策略.md)，参考项目对第四阶段最有价值的，不是某个系统怎么照抄，而是它提醒了我们完整 PvZ-like 战斗层天然会同时长出下面几组系统：

- 资源系统：
  - `DaySunsManager`
  - `CreateSunComponent`
- 卡牌和手持系统：
  - `AllCards`
  - `CardManager`
  - `HandManager`
- 棋盘和格子系统：
  - `PlantCellManager`
- 敌方节奏系统：
  - `ZombieManager`
  - 波次管理子模块
- 主战斗流程系统：
  - `MainGameManager`
  - 胜负、回合、下一波次

这对我们当前的意义是：

1. 这些系统域都必须正式进入第四阶段视野。
2. 但它们不能以“把参考项目的大管理器拆名搬过来”的方式进入。
3. 我们应借系统地图、管理器拆分意识和资源分层，不借原版业务层耦合。

---

## 2. 第四阶段当前的进入前提

按当前仓库状态，第四阶段前置条件里已经成立的部分包括：

- 第一轮规则协议冻结已完成
- `TriggerBinding` / `EntityTemplate` / `ProjectileTemplate` 主链已成立
- 残余兼容入口已退出主干
- 批量回归和负向守卫矩阵已存在
- 高度、轨迹、命中和模板工厂主干已稳定

所以第四阶段当前不再需要回头补的，不应再成为进入前提：

- 再补一轮 `runtime_behavior`
- 再补一轮场景兼容参数
- 再继续扩写触发器/效果宇宙

第四阶段当前真正需要补的是：

- 玩法层系统清单
- 系统依赖关系
- 每个系统的最小概念模型
- 每个系统的最小验证入口

---

## 3. 第四阶段总原则

### 1. 玩法层消费主干，不反向塑造主干

阳光、卡片、放置、波次这些系统要建立在：

- 事件链
- 模板链
- 已冻结规则协议

之上。

不要让它们重新定义：

- 实体生成方式
- 效果执行方式
- 命中或空间模型

### 2. 先系统清单，再协议草案，再最小实现

第四阶段每个玩法系统都应先回答：

- 它在世界里到底是什么
- 它依赖谁
- 它最小先实现哪一段

不要一上来就做：

- 大量字段
- 大量 UI
- 大量 manager 间互相回调

### 3. 先做最小闭环，不做完整复刻

第四阶段不是“现在就把原版 PvZ 战斗层全部做完”。

每个系统都应先做一个最小闭环，例如：

- 阳光：先跑通生产、收集、消耗
- 放置：先跑通选卡、校验、实例化
- 波次：先跑通一组波次定义到胜负判定

### 4. 每个玩法系统都必须有验证位置

第四阶段开始后不能回到“玩法层先凭手感开发，回头再补验证”。

每个要进入建设的系统都应至少对应：

- 一个最小验证场景
- 一个批量回归入口位置
- 一条矩阵记录

---

## 4. 第四阶段系统域前置清单

下面这 6 组系统是第四阶段真正的主线候选，但不建议同时开工。

### 4.1 阳光与资源

参考项目提醒来源：

- `DaySunsManager`
- `CreateSunComponent`
- `MainGameManager.test_change_sun_value`

当前要先回答的问题：

- 战斗资源由谁持有：
  - `BattleManager`
  - 独立 `BattleEconomyState`
  - 其他轻量服务
- 资源变化走什么语义事件
- 掉落阳光和植物产阳光是否走同一条资源收集路径
- 消耗发生在选卡时、放置确认时，还是实例化完成时

最小前置清单：

- [ ] 明确资源状态归属对象
- [ ] 明确 `sun.spawned / sun.collected / resource.changed / resource.spend_failed` 或等价事件语义
- [ ] 明确资源数值是否进入统一快照和验证日志
- [ ] 明确至少 1 个被动产阳光入口和 1 个掉落阳光入口
- [ ] 明确一个最小正向验证：
  - 产出资源
  - 收集资源
  - 资源数值变化正确
- [ ] 明确一个最小负向验证：
  - 资源不足时阻止消耗

当前非目标：

- 阳光动画和完整拾取表现
- 多资源体系
- 商店/外循环经济

### 4.2 棋盘、格子、占位与放置

参考项目提醒来源：

- `PlantCellManager`
- `HandManager`
- `plant_cell.gd`

当前要先回答的问题：

- 当前项目需要的是完整 `PlantCell`，还是更薄的 `BoardSlot / LaneSlot`
- 占位语义按：
  - lane + slot_index
  - world cell
  - placement tag
  - occupant list
  哪种组合表达
- 放置合法性校验属于：
  - 棋盘层
  - 卡牌层
  - 实体模板层
  哪一层

最小前置清单：

- [ ] 明确棋盘抽象最小模型
- [ ] 明确“占位成功”与“占位失败”的语义事件
- [ ] 明确是否允许同格多实体、覆盖、附着等后续扩展位
- [ ] 明确放置合法性校验入口，不回退到 `BattleManager` 特判
- [ ] 明确一个最小正向验证：
  - 合法格放置成功
- [ ] 明确一个最小负向验证：
  - 非法格放置失败且不生成实体

当前非目标：

- 原版全部地形规则
- 花盆、荷叶、墓碑、梯子等完整交互宇宙
- 完整拖拽 UI

### 4.3 卡片、费用、冷却与手牌

参考项目提醒来源：

- `AllCards`
- `CardManager`
- `HandManager`

当前要先回答的问题：

- 卡片是否直接引用 `EntityTemplate`
- 费用和冷却是写进：
  - `CardDef`
  - 模板扩展字段
  - 关卡局部配置
- 手牌是固定槽位，还是轻量列表
- 选卡状态与放置状态如何分层

最小前置清单：

- [ ] 明确 `CardDef` 或等价资源最小字段集
- [ ] 明确卡片与模板、资源消耗、放置请求之间的关系
- [ ] 明确冷却计时归属对象
- [ ] 明确手牌选择状态，不回退到 UI 直接驱动实例化
- [ ] 明确一个最小正向验证：
  - 有资源时出牌成功
- [ ] 明确一个最小负向验证：
  - 冷却中或资源不足时出牌失败

当前非目标：

- 完整选卡界面
- 全套卡槽表现层
- 收藏/解锁卡池

### 4.4 波次、刷怪与战局流程

参考项目提醒来源：

- `ZombieManager`
- 波次子管理器
- `MainGameManager.start_next_round_game()`

当前要先回答的问题：

- 敌方生成是按时间轴、波次块，还是事件驱动
- 胜负条件放在：
  - 关卡定义
  - 战局阶段机
  - 特定 manager
  哪一层
- 波次流程是否需要单独 `WaveDef / SpawnWaveResource`
- 关卡流程和战斗中的局部刷新如何分层

最小前置清单：

- [ ] 明确 `WaveDef` 或等价资源最小结构
- [ ] 明确战局阶段最小集合：
  - 准备
  - 进行中
  - 胜利
  - 失败
- [ ] 明确敌方生成入口，不回退到硬编码定时器散落在管理器里
- [ ] 明确一个最小正向验证：
  - 至少一波敌人按定义进入
  - 可触发胜利或失败条件
- [ ] 明确一个最小负向验证：
  - 无效波次定义被拦截

当前非目标：

- 完整原版 Adventure 流程
- 多模式小游戏流程
- 菜单和选关 UI

### 4.5 状态系统

参考项目提醒来源：

- `SleepComponent`
- `ScaredyComponent`
- `MagnetComponent`
- 各类特化组件说明了状态系统迟早会来

当前要先回答的问题：

- 状态是临时 modifier、标签，还是独立状态资源
- 状态如何影响：
  - 移动
  - 攻击
  - 目标选择
  - 事件链
- 第四阶段最小先做哪 1 到 2 个状态

最小前置清单：

- [ ] 明确状态最小抽象
- [ ] 只选 1 到 2 个最小状态进入第四阶段首批
- [ ] 明确状态叠加、刷新、移除的最小语义
- [ ] 明确一个最小正向验证：
  - 状态施加后行为发生可观测变化

当前非目标：

- 一次性做全套状态宇宙
- 为每个原版植物/僵尸特例补状态脚本

### 4.6 场上物件与机关

参考项目提醒来源：

- `GameItemManager`
- `GIM_LawnMover`
- `GIM_Brain`

当前要先回答的问题：

- 场上物件与普通实体是否共用模板链
- 割草机、脑子、障碍物等是否需要单独的 `BoardObjectTemplate`
- 战局胜负条件和场上物件如何关联

最小前置清单：

- [ ] 明确场上物件是否共用实体运行时底座
- [ ] 只选 1 个代表性物件进入第四阶段首批
- [ ] 明确与战局阶段、胜负判定的连接点
- [ ] 明确一个最小正向验证：
  - 物件成功进入战局并触发一次代表性行为

当前非目标：

- 一次性把脑子、割草机、墓碑、罐子、障碍物全做完

## 5. 推荐进入顺序

基于当前主干状态，第四阶段最合理的进入顺序不是“哪个最显眼先做哪个”，而是：

1. 阳光与资源
2. 棋盘、格子、占位与放置
3. 卡片、费用、冷却与手牌
4. 波次、刷怪与战局流程
5. 状态系统
6. 场上物件与机关

原因是：

- 没有资源，就没有最小经济闭环
- 没有棋盘和放置，就没有玩家指令入口
- 没有卡片和手牌，放置行为就缺正式上游
- 没有波次和战局流程，就只有静态演示，不算真正玩法层
- 状态和场上物件更适合作为后续扩展，而不是第四阶段第一刀

## 6. 当前明确不要照搬的参考项目做法

结合 [参考实现迁移策略](../wiki/04-roadmap-reference/25-参考实现迁移策略.md)，第四阶段尤其要克制下面几件事。

### 1. 不照搬 `MainGameManager`

参考项目的大管理器说明了系统地图，但不适合作为当前架构模板。

当前应坚持：

- 小 manager / 小 service
- 明确边界
- 明确验证

而不是：

- 把资源、卡片、手持、波次、胜负全部塞回单个 Godot 节点

### 2. 不照搬完整 `PlantCell`

参考项目的 `PlantCell` 很有价值，但它承载的是原版 PvZ 业务层全套规则。

当前更适合先做：

- 薄 `BoardSlot`
- 明确占位
- 明确合法性

而不是完整原版格子宇宙。

### 3. 不照搬原版业务枚举

第四阶段当前应继续使用：

- 模板 ID
- 标签
- 资源引用

而不是为了玩法层再引入：

- 植物大枚举
- 僵尸大枚举
- 子弹大枚举

### 4. 不把 UI 或表现层当主线

第四阶段当然会碰到卡牌 UI、阳光表现、拖拽交互。

但当前优先级仍应是：

- 概念模型
- 协议边界
- 最小验证闭环

不是：

- 先把界面做得像成品

## 7. 第四阶段开始前的最小交付批次

如果只做一轮最小前置准备，建议先完成下面 6 项。

- [x] 补一份战斗玩法层系统清单
- [x] 为阳光与资源写一份最小概念与事件草案
- [x] 为棋盘/占位/放置写一份最小概念与边界草案
- [x] 为卡片/手牌/费用写一份最小概念与边界草案
- [x] 为波次/战局阶段/胜负条件写一份最小概念与边界草案
- [x] 在验证矩阵中为第四阶段预留玩法层专项位置

完成这 6 项后，第四阶段就会从“知道以后要做什么”推进到“知道现在先让哪几个系统正式进入建设”。

---

## 8. 完成定义

第四阶段前置准备不要求现在就写出玩法系统代码，但至少应满足：

- 已明确第四阶段首批要进入建设的 2 到 3 个玩法系统
- 每个系统的边界、依赖和非目标已经写清楚
- 参考项目的可借鉴点和不可照搬点已经明确
- 不会一进入第四阶段就回退成 `MainGameManager` 式大一统实现
- 已为玩法层预留验证矩阵和回归入口位置
