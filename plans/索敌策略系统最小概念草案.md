# 索敌策略系统最小概念草案

> 这份草案用于回答"植物如何判断前方是否有敌人、仅在检测到敌人时才发射"这一问题。它的目标不是现在就做完整的目标选择 AI，而是先把检测策略的注册、调用和最小验证口径定下来，为三线攻击、五向攻击等后续扩展打好基础。

---

## 文档定位

这篇文档主要回答：

- 索敌（目标检测）在四层模型中应该落在哪一层
- 检测策略的注册表、函数签名和返回值结构如何设计
- 检测策略如何与现有 `periodically` 触发器结合
- 第一批内置检测策略有哪些、各自扫描什么范围
- 最小正向和负向验证口径怎么定义
- 冻结协议需要做什么程度的变更

这篇文档不负责：

- 完整目标优先级排序系统
- 基于 Area2D 的实时进出范围检测
- 多段索敌和链式索敌
- 索敌结果对效果层（如自动瞄准）的影响

## 一句话结论

建议引入独立的 `DetectionRegistry`（检测策略注册表），将索敌逻辑抽象为可注册、可组合的 Callable 策略函数。检测策略由 `periodically` 触发器在时间条件满足后调用，通过返回值中的 `has_target: bool` 决定是否真正触发效果。`detection_id` 和 `detection_params` 作为触发器条件参数写入 `TriggerBinding` 的 `condition_values`，保持数据驱动。

---

## 当前问题

当前 `periodically` 触发器只检查时间间隔，不检查前方是否有敌人：

```
game.tick → TriggerComponent → TriggerInstance.should_trigger()
  → 时间间隔满足？ → 是 → 立即触发 → spawn_projectile → 抛射体飞出
```

这导致：

1. 植物在没有敌人的情况下也会持续发射，抛射体飞入空空间
2. 无法表达"三线射手只在有敌人的车道发射"
3. 无法表达"星星果只在检测到敌人时才攻击"
4. 每个抛射体的生成和运动都消耗运行时资源

---

## 设计定位

### 在四层模型中的位置

索敌本质上是"是否触发"的判断条件，属于**触发器层**的职责扩展：

| 层 | 当前职责 | 索敌的影响 |
|---|---|---|
| 语义事件层 | `game.tick` | 不变 |
| 行为效果层 | `spawn_projectile` | 不变 |
| 组合装配层 | `TriggerBinding` | 新增 `detection_id` / `detection_params` 条件参数 |
| 连续行为层 | 抛射体运动 | 不变 |

### 核心原则

1. **检测策略是纯函数** -- 给定源实体和参数，返回检测结果。不持有状态，不发出事件。
2. **注册表驱动** -- 遵循 `EffectRegistry` / `TriggerRegistry` 已有的 `register_strategy(id, callable)` 模式。
3. **数据驱动** -- 检测模式通过 `.tres` 的 `condition_values` 配置，不需要修改 GDScript。
4. **向后兼容** -- 不指定 `detection_id` 时等同于不检查（`always` 策略），所有现有模板行为不变。

## 检测策略注册表

### 新增：`autoload/detection_registry.gd`

遵循项目已有的 Registry 模式：

| 结构 | 说明 |
|---|---|
| 存储 | `var _strategies: Dictionary = {}` -- `StringName -> Callable` |
| 注册 | `register_strategy(detection_id, callable)` |
| 查找 | `get_strategy(detection_id) -> Callable` |
| 调用 | `evaluate(detection_id, owner, params) -> Dictionary` |

### 策略函数签名

```gdscript
# 每个策略函数的签名：
func(owner: Node, params: Dictionary) -> Dictionary

# 返回值结构：
{
    "has_target": bool,          # 是否有目标
    "targets": Array[Node2D],   # 所有发现的目标
    "primary_target": Node2D,   # 主目标（最近/最高优先级）
}
```

### 调用链

```
periodically 策略（时间条件通过后）
  → 读取 condition_values["detection_id"]
  → 如果非空，调用 DetectionRegistry.evaluate(detection_id, owner_entity, detection_params)
  → has_target == false → 返回 false，不触发
  → has_target == true → 继续触发，执行效果
```

## 第一批内置检测策略

### `lane_forward` -- 单线前向扫描

扫描当前车道前方（X 轴正方向）的敌方实体。

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `scan_range` | `float` | `900.0` | 最大检测距离（ground_position 距离） |

扫描规则：

- 只扫描当前车道的敌方实体
- 方向：源实体 X 坐标右侧（前方）
- 距离：`ground_position` 欧氏距离 <= `scan_range`
- 筛选：`team != owner.team`、`lane_id == owner.lane_id`、`has_method("take_damage")`、`is Node2D`
- 返回：`primary_target` 为距离最近的敌人

适用场景：豌豆射手、双发射手、机枪射手等单线攻击植物。

### `lane_bidirectional` -- 单线双向扫描

扫描当前车道双向的敌方实体。

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `scan_range` | `float` | `900.0` | 最大检测距离 |

扫描规则：

- 同 `lane_forward`，但不限制方向（前后均检测）

适用场景：分裂豆等前后双向攻击植物。

### `three_lane_forward` -- 三线前向扫描

扫描当前车道及上下相邻车道的前方敌方实体。

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `scan_range` | `float` | `900.0` | 最大检测距离 |

扫描规则：

- 扫描 `lane_id - 1`、`lane_id`、`lane_id + 1` 三个车道
- 方向：仅前方
- 边界处理：超出车道范围的 lane_id 自动跳过（不报错）

适用场景：三线射手。

注意：此策略只回答"三个车道中是否有敌人"。各车道是否分别发射、如何分配抛射体，属于效果层或触发器绑定层的数据配置。

### `always` -- 始终通过

不做任何扫描，直接返回 `has_target = true`。

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| 无 | -- | -- | -- |

此策略用于：

- 不需要索敌的攻击类型（如向日葵产阳光的 `periodically` 触发）
- 显式标注"此攻击不需要检测目标"

## 共享扫描工具

当前引擎中有三处独立的实体扫描逻辑：

| 位置 | 方法 | 扫描范围 | 用途 |
|---|---|---|---|
| `BattleManager` | `_find_nearest_enemy()` | 全实体、同车道、欧氏距离 | 抛射体目标解析 |
| `ZombieRoot` | `_find_attack_target()` | 兄弟节点、同车道、X 距离、方向性 | 僵尸近战目标 |
| `EffectRegistry` | `_resolve_targets()` | 全实体、可选车道、半径 | 爆炸范围目标 |

建议在 `DetectionRegistry` 中提取一个内部共享扫描函数：

```gdscript
func _scan_enemies(source: Node, params: Dictionary) -> Array[Node2D]:
    # 参数：
    #   lanes: int / Array[int]  -- 车道过滤器（默认 source.lane_id）
    #   range: float             -- 最大距离（默认 INF）
    #   direction: String        -- "forward" / "backward" / "both"（默认 "both"）
    #   combat_active_only: bool -- 是否只返回战斗活跃的实体（默认 false）
    # 返回：所有符合条件的 Node2D 敌方实体
```

此函数可被所有检测策略和未来的其他扫描需求复用。`BattleManager._find_nearest_enemy()` 和 `EffectRegistry._resolve_targets()` 中的扫描逻辑可在后续版本中迁移到此处，但迁移不属于第一批范围。

## 协议层落点选项

当前 `periodically` 触发器的 `condition_params` 已冻结，`allow_extra_conditions = false`。引入检测策略需要选择如何与冻结协议交互。以下列出三种选项的利弊，不在本草案中做最终决定。

### 选项 A：扩展 `periodically` 的 `condition_params`

在 `TriggerRegistry` 中给 `periodically` 的 `condition_params` 新增 `detection_id`（`string_name`）和 `detection_params`（`dictionary`）两个参数。

**优点**：

- 改动量最小，只改 `TriggerRegistry.gd` 和 `ProtocolValidator.gd`
- 完全数据驱动，通过 `.tres` 配置
- 向后兼容 -- 不指定时默认为空，等同不检查
- 语义清晰 -- "是否触发"就在触发器层判断

**缺点**：

- 需要扩展冻结协议，需要设计审批
- `periodically` 的参数职责从纯时间扩展到包含空间查询

**对冻结协议的影响**：

- `FROZEN_TRIGGER_BEHAVIOR_SPECS` 不变（`attack -> periodically` 映射不变）
- `periodically` 的 `condition_params` 从 `[{interval}]` 扩展为 `[{interval}, {detection_id}, {detection_params}]`
- `ProtocolValidator` 的 `_normalize_trigger_event_and_conditions()` 需要识别新参数

### 选项 B：新增 `targeted_attack` 行为键

在 `FROZEN_TRIGGER_BEHAVIOR_SPECS` 中新增 `"targeted_attack"` 映射到 `periodically` 触发器，但使用不同的参数模板（包含检测参数）。

**优点**：

- 不修改现有 `attack` 行为的参数定义
- 概念上区分"盲目攻击"和"有目标攻击"

**缺点**：

- 引入新的冻结协议实体，仍需设计审批
- 与 `periodically` 核心逻辑重复
- 内容作者需要知道用 `attack` 还是 `targeted_attack`

### 选项 C：效果层静默检查

不改触发器协议，在 `spawn_projectile` 效果策略中检测是否有敌人。

**优点**：

- 冻结协议零修改
- 改动范围极小

**缺点**：

- 语义不精确 -- 触发器日志显示"已触发"但效果被跳过
- 违反四层模型分层原则（"是否做"属于触发器层）
- 只对 `spawn_projectile` 有效，不可复用

### 当前建议

本草案推荐**选项 A**，但最终决定需要在实现前经设计审批确认。理由是索敌本质是触发条件的一部分，放在触发器层语义最正确，且向后兼容性最好。如果后续检测需求变得非常复杂（Area2D 碰撞、实时信号），可以在 A 的基础上演进为组件模式。

## 数据层配置示例

以豌豆射手为例（选项 A）：

```
# plant_attack_basic_linear.tres -- 修改前
condition_values = { "interval": 0.6 }

# plant_attack_basic_linear.tres -- 修改后
condition_values = {
    "interval": 0.6,
    "detection_id": &"lane_forward",
    "detection_params": { "scan_range": 900.0 }
}
```

三线射手：

```
# plant_attack_three_lane.tres（新）
condition_values = {
    "interval": 0.6,
    "detection_id": &"three_lane_forward",
    "detection_params": { "scan_range": 900.0 }
}
```

不检测的攻击（如产阳光）：

```
# 不变，不指定 detection_id 即可
condition_values = { "interval": 0.6 }
```

## 事件语义

检测策略本身不发出事件。但如果需要调试或验证，建议在 `DebugService` 中记录：

| 日志级别 | 内容 |
|---|---|
| `trigger.blocked_by_detection` | 触发器因检测无目标而被阻止，包含 `detection_id`、`owner_entity` |

不建议新增专用的检测事件（如 `detection.scan`、`detection.target_found`），因为检测是触发器的内部条件判断，不属于语义事件层。

## 最小验证口径

### 最小正向验证

建议预留：`target_acquisition_validation.tres`

至少覆盖以下观察点：

1. 植物配置 `detection_id: lane_forward`、`scan_range: 400`
2. 场景中无敌人时，不出现 `projectile.spawned` 事件
3. 在范围内放置敌人后，出现 `projectile.spawned` 事件
4. 敌人移出范围后，停止发射

### 最小负向验证

建议在同一场景中覆盖：

1. 敌人在范围外时不发射（距离 > scan_range）
2. 敌人在相邻车道时不发射（车道隔离）
3. 同车道的友方实体不被视为目标（队伍隔离）
4. `detection_id` 为空或 `always` 时始终发射（向后兼容）

### 三线扫描验证

建议预留：`target_acquisition_three_lane_validation.tres`

至少覆盖：

1. 当前车道无敌人但相邻车道有敌人时，触发（`has_target == true`）
2. 三个车道都无敌人时，不触发
3. 边界车道（最上/最下）不会越界扫描

## 当前明确不做的事

1. 不做基于 Area2D 的实时进出范围检测（当前只需每 tick 一次性扫描）。
2. 不做目标优先级排序（如"优先攻击飞行单位"、"优先攻击血量最低"）。
3. 不做检测结果的缓存（每次触发独立扫描）。
4. 不把检测策略的结果直接传递给效果层（如自动瞄准）。
5. 不修改 `BattleManager._find_nearest_enemy()` 和 `EffectRegistry._resolve_targets()`（迁移属于后续批次）。
6. 不为检测策略引入新的 `Resource` 类型（当前只需 Callable 函数 + Dictionary 参数）。

- 已确定协议层落点（选项 A / B / C 的最终决定）

- 已在验证矩阵中预留 `target_acquisition_validation` 位置
