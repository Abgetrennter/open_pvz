## 核心原则

原版图像移植不是“把一整个 `.reanim` 直接转成一个 Godot 动画然后播放”。原版的 reanimation 系统有自己的分层、挂接和动画段语义，尤其是植物常见的 body/head/blink 分离结构。

Open PVZ 的目标是复用原版视觉比例和动作语义，但不能让运行时依赖私有素材，也不能把原版像素当成现代屏幕像素。

必须遵守：

- 原版素材只放在 `vendor/out_files/` 等本地 ignored 目录，不进入发布物。
- `vendor/de-pvz/` 只作参考，不直接修改。
- 原版 `px` 按 800x600 固定画布设计单位理解。
- OpenPVZ 战斗坐标仍然是 combat/world unit，不随 UI 缩放变化。
- 视觉反馈不得影响战斗判定。

## 高层反思与方法论

这轮 Peashooter、Sunflower、ThreePeater 的移植验证暴露出的核心问题，不是单纯的坐标偏差或导入器缺字段，而是最初容易把 `.reanim` 错看成“一个可直接播放的动画文件”。更准确的理解是：`.reanim` 是原版 reanimation definition，里面混合了 marker、视觉层、attachment 挂点、overlay、body 时基和 head 时基。

因此原版图像移植不能按“文件转文件”推进，必须先还原原版运行时组合语义，再选择 Godot 表达方式。

### 已踩过的坑

1. **把 `anim_*` 名字当语义**
   - Peashooter 里有些 `anim_*` 是 marker。
   - Sunflower 里 `anim_idle` 是头部视觉层，`anim_blink` 是 overlay。
   - ThreePeater 里 `anim_head1/2/3` 是 attachment 挂点。
   - 结论：不能靠名字分类，必须看 track 内容，再结合 `de-pvz` 的调用方式。

2. **把局部动画当整株动画**
   - Peashooter 的 `idle` 不是整株 idle，而是 body 段。
   - `full_idle` 可用于调试整株预览，但不适合和 `shooting` 叠加作为运行时结构。
   - ThreePeater 的 `head_idle2` 不是 body 时基，拿它驱动 body 会把茎叶姿态拉歪。

3. **忽视 body / head / overlay 的分层**
   - Peashooter 是 body + head。
   - Sunflower 是底层整株 pose + blink overlay。
   - ThreePeater 是 body + 三个 head actor + 三个动态挂点。
   - 结论：这些结构不能统一压成“一个 actor 播一个 AnimationPlayer 动画”。

4. **把 attachment 当固定坐标**
   - Peashooter 头部要跟随 `anim_stem`。
   - ThreePeater 三个头要跟随 `anim_head1/2/3`。
   - Sunflower blink 要跟随头部视觉层。
   - 固定位置只会在某一帧看起来正确，动起来必然漂移。

5. **忽视多段串联 `.reanim`**
   - ThreePeater 的 head1 / head3 / head2 / body 分布在不同帧段。
   - body 有效段在文件尾部，而不是开头。
   - 导入器生成的 animation name 只说明 marker 段，不自动说明 body 段。

6. **把“能显示”误认为“语义正确”**
   - 静态结构正常，不代表发射动作正常。
   - 头部能 attach，不代表 body 时基正确。
   - body 正常，不代表 attachment track 已参与动态更新。
   - 必须观察循环、切动作、动作结束回 idle、发射/产出瞬间和实际尺寸。

### 需要坚持的原则

- **先查原版行为，再写导入逻辑。**  
  `Plant.cpp` 决定 body/head 如何组合，`Reanimator.cpp` 决定 attach 和 `SetFramesForLayer()` 语义。素材文件只提供数据，不能单独决定运行时结构。

- **先做语义分类，再做 Godot 表达。**  
  每个 track 至少要分类为 marker、body visual、head visual、overlay、attachment anchor、未处理层。缺少这层分类，后续只能靠截图猜。

- **组合优先于单动画。**  
  原版植物常由多个 reanimation 叠加和挂接而成。OpenPVZ 应优先用 wrapper scene / visual composition 表达组合关系，而不是强行把所有层压进一个 actor。

- **body 时基和 head 时基要分离。**  
  body 负责站场摇摆和 attachment 挂点，head 负责 idle / shooting。head action 结束后要回 idle，并尽量重新对齐 body `animTime`。

- **overlay 永远叠加，不替换主体。**  
  blink、产阳光、发光、短特效都应该叠加到底层 pose 上，不能驱动整株，也不能让底层消失。

- **验证必须覆盖动态过程。**  
  单帧预览不够。每个迁移对象至少要看 idle 循环、动作切换、动作结束回 idle、attachment 跟随、发射/产出时机、实际棋盘尺寸。

### 工具层目标

后续导入器不应只输出 `actor.tscn + AnimationPlayer`，还应输出语义报告：

- marker tracks
- visual tracks
- suspected attachment tracks
- suspected overlay tracks
- inherited transform summary
- clip table with source track
- attachment base pose frames
- effective body ranges
- effective head/action ranges
- unresolved / unused tracks
- recommended composition skeleton

目标不是第一步就自动完美还原所有植物，而是让人工确认时有足够事实，减少靠截图反复猜。Peashooter、Sunflower、ThreePeater 已覆盖三类核心模式：单头挂接、overlay 叠加、多头挂点。后续植物迁移应先套这个分析框架，再进入具体调参。

## 原版 Reanimator 运行模型

这次重新查 `de-pvz` 后，需要把 `.reanim` 的地位说得更准确：它不是一个 actor 文件，而是一个 `ReanimationType` 对应的 reanimation definition。运行时可以创建多个 `Reanimation` 实例引用同一份 definition，每个实例各自维护 `mFrameStart/mFrameCount/mAnimTime/mAnimRate/mLoopType/mOverlayMatrix`，再通过 attachment 组合成最终 actor。

### 定义加载

原版先通过 `gLawnReanimationArray` 建立 `ReanimationType -> reanim 文件路径` 的映射，例如：

- `REANIM_PEASHOOTER -> reanim\PeaShooterSingle.reanim`
- `REANIM_REPEATER -> reanim\PeaShooter.reanim`
- `REANIM_SUNFLOWER -> reanim\SunFlower.reanim`
- `REANIM_THREEPEATER -> reanim\ThreePeater.reanim`

正式运行时优先读 `.reanim.compiled`，`de-pvz` 的 debug/反编译环境可从 XML `.reanim` 编译加载。图片字段 `i` 先按资源名直接查 ResourceManager；查不到时再按前缀路径尝试，例如 `IMAGE_REANIM_*` 会尝试 `reanim\` 和 `images\`。

### 数据补齐

`.reanim` 中每个 track 的 `<t>` 是逐帧 transform。原版加载后会对缺省字段做前值继承：

- `x/y/kx/ky/sx/sy/f/a` 缺省时继承上一帧。
- `i/font/text` 缺省时继承上一帧。
- 初始默认值大致是 `x/y/kx/ky = 0`、`sx/sy = 1`、`f = 0`、`a = 1`，但素材通常会在前导帧显式写 `<f>-1</f>` 让该层隐藏。

这解释了为什么很多 track 只有第一帧写完整字段，后面只写变化字段；导入器必须先完成同样的继承，再做可见性和矩阵计算。

### 采样与绘制

原版不是给每个 track 单独播放自己的时间轴，而是当前 `Reanimation` 实例先确定一个全局帧区间，然后所有 track 都在这个区间内按同一个 `mAnimTime` 采样。

采样规则：

- `x/y/kx/ky/sx/sy/a` 在前后帧之间线性插值。
- `i/font/text` 使用前一整数帧，不做插值。
- `f` 使用前一整数帧，`f < 0` 表示该 track 当前不绘制。
- `mAnimRate` 以原版 100Hz update 推进：每次 update 按 `0.01 * mAnimRate / mFrameCount` 增加 `mAnimTime`。
- `mLoopType` 决定循环、播放一次、播放后死亡、播放后停在末帧，以及是否把最后一帧纳入采样区间。

绘制规则：

- 每个可见 track 按 definition 中的 track 顺序绘制。
- render group 可以隐藏某些 track，例如 blink 时临时隐藏 `anim_eye` 前缀。
- attachment 挂在某个 track 上时，会在父 track 绘制后绘制其子 reanimation。
- `mIsAttachment = true` 的 reanimation 不作为 Board 顶层 reanimation 单独绘制，只通过父 attachment 出现在画面里。

### 运行时组合

原版植物常见结构是多个 `Reanimation` 实例叠加，而不是一个实例播放全部内容：

- body 实例负责底层、茎、叶、attachment anchor。
- head 实例负责头部 idle / shooting，并挂到 body 的 anchor track。
- blink/light/glow 实例是短 overlay，挂到 body 或 head 的语义父层。

因此 OpenPVZ 的导入器可以先输出单文件 actor 作为调试视图，但真正的运行时移植要走 wrapper scene / visual composition。

## `.reanim` 结构理解

`.reanim` 中不能只分成“动画名”和“图片层”。更贴近原版运行时的分类是：

- clip selector / marker track：通常只靠 `f` 标出一个可播放片段，供 `SetFramesForLayer(track_name)` 选帧段。
- visual track：携带 `i/x/y/sx/sy/kx/ky/a` 等字段，当前帧 `f >= 0` 时绘制。
- attachment anchor track：不一定绘制图片，但必须提供每帧 transform，供子 reanimation 挂接。
- overlay track：blink、glow、light 等短层，通常作为单独 reanimation 叠加。
- attacher track：名称前缀为 `attacher__`，通过 `text` 字段描述要动态创建并挂接的子 reanimation，主要见 credits 等复杂场景。

不能只靠名字判断 `anim_*` 都是 marker。Peashooter 中：

- `anim_full_idle`、`anim_shooting`、`anim_head_idle`、`anim_idle` 是 marker track。
- `anim_stem`、`anim_face`、`anim_blink` 是视觉 track。

Sunflower 中则更容易误判：

- `SunFlower.reanim` 当前解包版本没有 marker track，导入报告中 `marker_count = 0`。
- `anim_idle` 是头部视觉层，不是 idle marker。
- `anim_blink` 是眨眼视觉 overlay，不是 blink marker。
- 身体、茎、叶、花瓣等主要层在前 0-3 帧仍处于隐藏或局部状态，第 4 帧后才形成整株稳定姿态。

导入器应采用内容判断：

- 如果 track 名以 `anim_` 开头，且帧内只出现 `f/frame` 这类 marker 信息，才视为 marker。
- 如果出现 `i/x/y/sx/sy/kx/ky/a/bm` 等视觉字段，应视为视觉 track。
- 如果出现 `x/y/sx/sy/kx/ky` 但没有 `i`，且原版代码把它作为 `AttachToAnotherReanimation()` 的目标，应视为 attachment anchor。
- 如果没有 marker track，不能默认认为 `all: 0..end` 可以直接作为运行时 idle 循环；需要额外识别整株有效帧段。

ThreePeater 进一步说明：`anim_*` 还可能是 attachment 挂点 track。当前解包版本中：

- `anim_head_idle1/2/3`、`anim_shooting1/2/3`、`anim_blink1/2/3` 是 marker track。
- `anim_head1/2/3` 是 body reanim 上的 attachment 挂点，不是要显示的头部贴图。
- `anim_face1/2/3` 是各头的视觉层。
- body/stem/leaf 与 `anim_head1/2/3` 的有效动态段在文件尾部 `124..148`，不能用任一 `head_idle*` marker 段代替 body 时基。

判断顺序建议：

1. 先按内容区分 marker track 和视觉/变换 track。
2. 再按原版 `Plant.cpp` 判定 track 语义：body 层、head 层、blink overlay、attachment 挂点。
3. attachment 挂点即使名字像 `anim_head1`，也必须参与 body 动画更新；只是不要在 body actor 上绘制出来。

## 字段映射

当前 v0 字段映射：

| `.reanim` 字段 | Godot 字段 | 说明 |
|---|---|---|
| `f` | `visible` | `-1` 表示隐藏，其他表示显示 |
| `i` | `Sprite2D.texture` | 通过 `resources.xml` 映射到 png |
| `x/y` | `position` | 原版设计单位 |
| `sx/sy` | `scale` | 局部缩放 |
| `kx` | `rotation` | 当前按角度转弧度 |
| `ky` | `skew` 派生 | 当前用 `ky - kx` 近似 |
| `a` | `self_modulate.a` | 透明度 |
| `font/text` | report / 后续文本层 | v0 暂不绘制文本 |
| `bm` | report | `de-pvz` 当前 reanim schema 未直接体现；若解包素材出现，v0 只记录，不还原 blend material |

`Sprite2D.centered` 应为 `false`，否则所有 reanim 局部坐标会整体偏移。

注意：`kx/ky` 在原版矩阵中是 skew angle，不是 Godot 普通单轴 rotation。当前 v0 的 `rotation + skew` 是近似表达，够做素材识别和 demo，但不是完整矩阵还原。后续如果要高保真，应直接使用 `Transform2D` 还原：

```text
m00 = cos(-kx) * sx
m10 = -sin(-kx) * sx
m01 = sin(-ky) * sy
m11 = cos(-ky) * sy
m02 = x
m12 = y
```

## 原版播放语义

### 动画段切分

`vendor/de-pvz/Sexy.TodLib/Reanimator.cpp` 中的 `SetFramesForLayer()` 会调用 `GetFramesForLayer(track_name)`。

核心语义：

- 通过指定 track 找到第一个 `mFrame >= 0` 的帧作为起点；这个 track 可以是 marker，也可以是视觉层或挂点层。
- 从起点继续向后，最后一个 `mFrame >= 0` 的帧决定动画段长度。
- 播放时不是复制一份新动画资源，而是在同一 reanimation definition 上设置 `mFrameStart/mFrameCount`。
- 设置完帧段后，所有 track 都用这个帧段采样，不是只有指定 track 在播放。

Godot 导入可以把每个 marker 段导成一个 `AnimationPlayer` 动画，但要保留原版的段语义：

- idle 类动画循环。
- shooting / attack / special 类动画通常是一次性播放。
- 切段时需要 blend，而不是硬切。
- `StartBlend(blend_time)` 会记录所有当前可见 track 的 transform，并在后续若干 update 内混合回新片段；只硬切帧段会丢掉这层柔和过渡。
- 动作结束回 idle 时，head 类实例通常会把 `mAnimRate` 和 `mAnimTime` 重新对齐 body，避免回到 idle 后节奏跳动。

### Peashooter 示例

`vendor/de-pvz/Lawn/Plant.cpp` 中 Peashooter/Snow Pea/Repeater/Gatling Pea 的结构是：

- body reanim：
  - `SetFramesForLayer("anim_idle")`
  - `mLoopType = REANIM_LOOP`
  - `mAnimRate = RandRangeFloat(15.0f, 20.0f)`
- head reanim：
  - 独立 AddReanimation
  - `SetFramesForLayer("anim_head_idle")`
  - `mLoopType = REANIM_LOOP`
  - attach 到 body 的 `anim_stem`，如果没有 `anim_stem` 才 fallback 到 `anim_idle`

发射时：

- 对 head reanim 执行 `StartBlend(20)`
- `mLoopType = REANIM_PLAY_ONCE_AND_HOLD`
- `mAnimRate = 35.0f`
- `SetFramesForLayer("anim_shooting")`
- `mShootingCounter = 33`

射击动画完成后：

- 再对 head reanim `StartBlend(20)`
- 切回 `anim_head_idle`
- `mAnimRate` 和 `mAnimTime` 重新对齐 body

结论：

- `full_idle` 适合调试整株预览，不适合作为 Peashooter 的运行时主结构。
- `idle` 只对应 body/stem 站场，不包含完整头部语义。
- `head_idle` 与 `shooting` 应在 head reanim 上切换。
- 不应使用 `full_idle + shooting` 叠加，否则会出现双头或双嘴。

### Sunflower 示例

`SunFlower.reanim` 与 Peashooter 不同，当前解包素材中没有 marker track。第一版导入器会生成一个 `all` 动画，但直接从第 0 帧循环会暴露原版 reanim 的前导局部帧：

- 产阳或循环边界处只剩眼睛/局部 blink。
- 头部、花瓣、茎叶没有同时出现。
- blink overlay 固定在第 0 帧坐标，不跟随当前头部摆动。

当前实际演示场景采用的 v0 处理：

- 底层整株 idle 只循环有效区间：`IDLE_START_FRAME = 4`、`IDLE_END_FRAME = 28`，源 fps 为 `12`。
- `anim_idle` 作为头部视觉层保留在底层整株 pose 中。
- `anim_blink` 作为单独 overlay，在底层 seek 完成后重新叠加。
- blink 位置基于当前 `anim_idle` 头部 Sprite 的 transform，而不是写死 `anim_blink` 第 0 帧绝对坐标。
- 当前局部微调偏移为 `BLINK_HEAD_OFFSET = Vector2(12.8, 6.2)`，这是导入器尚未完整还原 attachment/overlay 语义前的演示层校准值。

结论：

- 无 marker 的 reanim 需要先做可见性区间分析，找到整株稳定帧段。
- 眨眼、发光、受击闪烁等短 overlay 不能替换底层整株动画。
- overlay 必须跟随其语义父层，例如眼睛跟随 head，头部跟随 stem。
- 如果 overlay 使用固定绝对坐标，会在底层 idle 摆动时漂移。

### ThreePeater 示例

`vendor/de-pvz/Lawn/Plant.cpp` 中 ThreePeater 的结构是：

- body reanim：
  - 不切到 `anim_head_idle*`。
  - `mAnimRate = RandRangeFloat(15.0f, 20.0f)`。
  - 负责茎、叶、body 挂点的连续摆动。
- head reanim 1：
  - 独立 AddReanimation。
  - `SetFramesForLayer("anim_head_idle1")`。
  - attach 到 body 的 `anim_head1`。
- head reanim 2：
  - `SetFramesForLayer("anim_head_idle2")`。
  - attach 到 body 的 `anim_head2`。
- head reanim 3：
  - `SetFramesForLayer("anim_head_idle3")`。
  - attach 到 body 的 `anim_head3`。

发射时：

- 根据上/中/下 lane 是否可攻击，分别把对应 head 切到 `anim_shooting1/2/3`。
- `StartBlend(10)`。
- `mAnimRate = 20.0f`。
- `mShootingCounter = 35`。
- `mShootingCounter == 1` 时发射对应 lane 的 projectile。

当前解包素材的帧段布局需要特别注意：

| 段 | 帧范围 | 语义 |
|---|---:|---|
| `anim_head_idle1` | `4..28` | 下方头 idle |
| `anim_shooting1` | `29..44` | 下方头 shooting |
| `anim_head_idle3` | `45..69` | 上方头 idle |
| `anim_shooting3` | `70..85` | 上方头 shooting |
| `anim_head_idle2` | `86..110` | 中间头 idle |
| `anim_shooting2` | `111..123` | 中间头 shooting |
| body / stem / `anim_head1/2/3` | `124..148` | 整株 body 摆动和三个 attachment 挂点 |

结论：

- 不能用 `head_idle2` 之类的 marker 动画驱动 body；这会把局部头部段的累计状态错误套到茎叶层上，导致身体扭曲。
- body 需要单独播放或解析自己的有效动态段。
- `anim_head1/2/3` 必须纳入 body 动画更新，因为它们是 head attachment 的动态父层。
- `anim_head1/2/3` 在 body actor 上应隐藏或不绘制，但 transform 必须保留。
- 三个 head actor 自己播放 `head_idle* / shooting*`，并通过 `body_actor.transform * current_anchor_transform * base_anchor_transform.affine_inverse()` 挂接。

## Attachment 语义

原版 head 不是固定在 body 同一原点，也不是简单复制 body transform。它通过 `AttachToAnotherReanimation(body, "...")` 挂接。

`Reanimator.cpp` 的关键逻辑：

```text
parent_current = current_parent_track_matrix * parent_overlay_matrix
parent_base = parent_track_matrix_at_base_pose
attachment_overlay = parent_current * inverse(parent_base) * attach_offset
```

迁移到 Godot 时，v0 可以采用最小近似：

```gdscript
head_actor.transform = body_actor.transform * current_stem_transform * base_stem_transform.affine_inverse()
```

其中：

- `base_stem_transform`：body attachment track 在 idle 起始姿态的 transform。
- `current_stem_transform`：body attachment track 当前帧 transform。
- head actor 自己仍播放 `head_idle` 或 `shooting`。

`AttachToAnotherReanimation(child, parent, track)` 第一次挂接时，如果 parent 还没有显式 base pose，原版会把 parent 当前 `mFrameStart` 作为 `mFrameBasePose`。因此 wrapper scene 要保存“挂接发生时的父片段基准帧”，不能每次动作切换都重新拿当前帧当基准，否则子层会被重新吸到错误位置。

如果不做这一步，会出现头部钉在原地、身体和茎在下面摆动的问题。

注意：attachment track 与 head 视觉层不是一回事。ThreePeater 的 `anim_head1/2/3` 是 body 上的挂点，应参与 body 动画更新；`anim_face1/2/3`、`ThreePeater_mouth*`、`ThreePeater_head*_blink` 等才是 head 视觉层。

## 推荐移植流程

### 1. 准备素材

确认本地素材池存在：

```powershell
vendor/out_files/reanim/<Name>.reanim
vendor/out_files/properties/resources.xml
```

确认 `vendor/out_files/` 被 `.gitignore` 覆盖。

### 2. 运行导入器

示例：

```powershell
.\Godot_v4.6.1-stable_win64_console.exe --headless --script res://tools/reanim_importer/reanim_import_one.gd -- --source res://vendor/out_files/reanim/Peashooter.reanim --image-root res://vendor/out_files/reanim --resources res://vendor/out_files/properties/resources.xml --out-dir res://vendor/out_files/_openpvz_import/peashooter --profile-id local.original.peashooter
```

### 3. 查看报告

检查 `import_report.json`：

- `actor_loadable == true`
- `unresolved_texture_count == 0`
- `loaded_texture_count > 0`
- marker 数量符合预期
- 动画名符合预期

Peashooter 期望至少：

- `idle`
- `head_idle`
- `shooting`
- `full_idle`

### 4. 分类原版结构

对每个植物先查 `vendor/de-pvz/Lawn/Plant.cpp`：

- 是单 body 还是 body + head？
- 是否有多头，例如 Threepeater / Split Pea？
- blink 是否独立 attachment？
- shooting 是 body 播放还是 head 播放？
- 是否需要 attach 到 `anim_stem`、`anim_head1`、`anim_idle` 等 track？
- 发射时 `mAnimRate`、`mShootingCounter` 是多少？

不要凭动画名猜运行时结构。

### 5. 生成 OpenPVZ 组合方式

v0 可采用 wrapper scene 组合导入 actor：

- 单 body 类：一个 actor，播放 body 动画。
- Peashooter 类：两个 actor，body 播 `idle`，head 播 `head_idle/shooting`，head 挂到 body 的 `anim_stem`。
- 多头类：body + 多个 head actor，各自挂到对应 track。
- blink 类：单独 blink actor 或后续导入为同 actor 局部层。
- Sunflower 类无 marker actor：一个底层 actor 循环整株有效帧段，再把 blink 等局部 overlay 按父层 transform 叠加。
- ThreePeater 类：body 单独播放 body 有效段，三个 head actor 分别播放 `head_idle1/2/3`，并 attach 到 body 的 `anim_head1/2/3`。

中期建议让导入器输出更明确的组合描述，而不是让验证脚本手写：

```text
visual_parts:
  body: idle
  head: head_idle, attach_to=anim_stem, action.shooting=shooting
  blink: blink, attach_to=anim_face
  effective_idle_range: 4..28
  body_range: 124..148
  heads:
    - idle=head_idle1, shooting=shooting1, attach_to=anim_head1
    - idle=head_idle2, shooting=shooting2, attach_to=anim_head2
    - idle=head_idle3, shooting=shooting3, attach_to=anim_head3
```

### 6. 设置播放参数

idle：

- loop
- speed scale 约 `15~20 / source_fps`
- 对 head idle，最好与 body `animTime` 对齐

shooting：

- one-shot / hold last frame
- blend in
- Peashooter 类 speed scale 约 `35 / source_fps`
- 完成后 blend 回 `head_idle`
- projectile spawn 应按原版 counter 延迟，不应在切动画第一帧立刻生成

### 7. 实际尺寸验证

至少准备两个本地验证入口：

- 放大调试预览：轮播所有导入动画，用于看贴图是否缺失、层级是否错。
- 实际尺寸演示：使用 `slot_spacing = 80`、actor scale `1.0`，放在 5 lane / 9 slot 棋盘上观察战场占比。

实际尺寸场景应优先回答：

- 是否和棋盘格宽关系合理？
- 是否锚到地面/格子中心合理？
- 发射点和 projectile 轨迹是否自然？
- head/body attachment 是否连续？

## 常见问题

### 只有头或只有身子

原因通常是把 `idle` 当成整株动画。Peashooter 的 `idle` 是 body 站场段，头部需要单独 `head_idle`。

### 产阳或循环时只剩眼睛

原因通常是无 marker 的 `all` 动画从第 0 帧直接循环。Sunflower 的前导帧中 blink/head 局部层会先出现，整株稳定姿态在后续帧才完整。

正确做法是：

- 不把产阳动作当成替换整株的动画。
- 底层整株 idle 持续播放或手动循环有效帧段。
- 阳光、眨眼等短效果作为 overlay 叠加。
- 如果没有 marker，先通过 track 可见性和截图验证确定有效 idle 区间。

### 眨眼层漂移或盖不准

原因通常是 blink overlay 使用了 `.reanim` 中的固定绝对坐标。这个坐标只对应某个基准帧；当头部 idle 左右/上下摆动时，眼睛必须跟随头部当前 transform。

正确做法是：

- 找到 blink 的语义父层，Sunflower 当前为 `anim_idle` 头部层。
- 每次底层 pose 更新后，再更新 blink overlay。
- blink position/scale/rotation/skew 基于父层当前 transform。
- v0 可使用局部偏移微调；后续工具化应使用矩阵或子节点 attachment 表达。

### 发射时双头/双嘴

原因通常是 `full_idle + shooting` 叠加。`full_idle` 已经包含整株，`shooting` 又包含头部动作，叠加必然重复。

正确做法是：

- body：`idle`
- head：`head_idle -> shooting -> head_idle`

### 头部不跟身体动

原因是没有复刻原版 attachment。head 必须跟随 body 的 `anim_stem` 相对基准姿态变化。

ThreePeater 还需要确认 `anim_head1/2/3` 这类挂点 track 参与了 body 动画更新。如果只是隐藏了这些 track，或把它们误判为 head 视觉层排除，三个头会固定在初始位置。

### 多头植物身体扭曲

原因通常是拿某个 head marker 动画驱动 body。ThreePeater 的 `head_idle1/2/3` 只描述各自头部段，不能代表整株 body 时基。

正确做法是：

- 先找 body/stem/attachment track 的有效帧段。
- body actor 播放或手动应用该有效段。
- head actor 只播放对应 head marker。
- attachment track 在 body 上隐藏但保持 transform 更新。

### 发射不连贯

常见原因：

- shooting 被错误设置为 loop。
- 切动画时硬切，没有 blend。
- shooting 播放速率仍是 source fps，没有使用原版 `mAnimRate`。
- projectile 在切动画第一帧立即生成，没有按 `mShootingCounter` 延迟。

### 尺寸看起来不对

先确认是否在实际尺寸场景中看，而不是放大调试预览中看。

OpenPVZ 当前第一版口径：

```text
原版 80 设计单位 == OpenPVZ 80 world unit
actor scale = 1.0
```

不要用屏幕窗口缩放判断战斗尺寸。

## 后续工具化方向

v0 先允许手写 wrapper 场景验证视觉语义。确认多个植物后，再推进工具化：

1. 导入器输出 marker/visual track 分类详情。
2. 导入器先完成原版式字段继承，再输出每个 track 的可见帧区间、图片引用、是否仅 transform、是否疑似 anchor。
3. 导入器在无 marker 时输出可见性区间分析，至少报告首个整株稳定帧和尾部可循环帧。
4. 导入器输出 clip table：`clip_name -> frame_start/frame_count/source_track`，并保留 source track 是否真 marker。
5. 导入器输出 attachment table：`anchor_track -> base_pose_frame/current_transform_provider`。
6. 导入器输出 overlay table：blink/glow/light 等短层及其推荐父层。
7. 导入器支持生成 body/head/blink part scene。
8. 增加 per-plant visual composition 配置。
9. `VisualProfile` 扩展 attachment/action/overlay 描述。
10. `VisualActorComponent` 支持 attachment part 播放，而不是只代理一个 `AnimationPlayer`。
11. 增加本地视觉验证 manifest，但不依赖 ignored 资源进入正式发布验证。

工具化前不要过早抽象。先用 Peashooter、Repeater、Threepeater、Sunflower、Wallnut 这几类覆盖典型结构，再收敛通用模型。
