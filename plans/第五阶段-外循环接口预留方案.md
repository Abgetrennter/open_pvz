# 第五阶段-外循环接口预留方案

> 这份文档用于完成 `第五阶段-v1规范化-4到8周执行路线图` 中的 `Epic F：外循环接口预留`。它的目标不是现在就做完整选关、存档、图鉴、商店和花园，而是先把这些系统需要消费的接口、数据边界和后续阶段输入整理出来。

---

## 文档定位

这篇文档主要回答：

- 当前外循环系统应如何占位
- 每个系统当前为什么后置
- 后续一定需要哪些接口位
- 战斗结果应该向外输出什么
- 第六阶段开工前需要哪些输入

这篇文档不负责：

- 直接实现完整外循环 UI
- 定义最终版存档格式
- 定义最终版经济系统
- 提前展开图鉴/商店/花园全部细节

---

## 当前判断

外循环系统当前之所以仍然明显后置，不是因为它们不重要，而是因为：

1. 当前主线仍然是第五阶段 `v1` 规范化
2. 协议、扩展和回归边界还在收口
3. 如果现在直接做外循环，很容易反向把战斗内状态边界打散

因此当前最合理的动作不是“开始做系统”，而是：

> 先把外循环系统会消费哪些接口和数据整理出来，保证第六阶段开始时不需要回头拆主干。

---

## 外循环系统地图

### 1. 选关与模式入口

当前为什么后置：

- 当前还没有真正稳定的关卡目录和模式目录
- Showcase 仍主要承担演示职责，不等同于正式选关

未来一定需要的接口位：

- `scenario_id`
- `display_name`
- `description`
- `mode_tag`
- `is_unlocked`
- `entry_scene_or_route`

### 2. 存档与进度

当前为什么后置：

- 当前还没有稳定的外循环状态模型
- 现在保存太多内容会放大协议漂移成本

未来一定需要的接口位：

- profile 身份
- 已完成关卡
- 已解锁内容
- 历史最佳成绩
- 最近运行版本或数据版本

### 3. 解锁

当前为什么后置：

- 当前还没有正式 campaign 结构
- 现在就写解锁规则会过早绑定内容顺序

未来一定需要的接口位：

- 解锁条件输入
- 关卡完成结果输入
- 内容元数据输入
- 解锁状态存储入口

### 4. 图鉴

当前为什么后置：

- 当前内容仍以验证和样例为主
- 模板元数据还没完全收敛成面向玩家展示的数据

未来一定需要的接口位：

- `template_id`
- `display_name`
- `entity_kind`
- `tags`
- 可展示的 summary 数据
- 是否已发现/已解锁

### 5. 商店

当前为什么后置：

- 当前经济系统只覆盖战斗内阳光，不覆盖战斗外经济
- 如果现在做商店，很容易把战斗内资源语义污染成长期货币

未来一定需要的接口位：

- 可购买项 ID
- 价格定义
- 解锁前置
- 购买状态
- 长期资源入口

### 6. 花园

当前为什么后置：

- 它本质上是战斗外持久状态和内容表现系统
- 当前没有稳定的持久化边界和互动模型

未来一定需要的接口位：

- 可持有单位或内容实例
- 培育状态
- 互动状态
- 长期收益入口

---

## 战斗内状态与战斗外状态边界

### 战斗内状态

下面内容应继续视为战斗内状态：

- 当前资源（sun）
- 当前战局 phase
- 当前活跃波次
- 当前实体状态
- 当前 slot 占位
- 当前 battle validation 状态

这些状态默认不应直接写入长期存档。

### 战斗外状态

下面内容应优先视为战斗外状态：

- 关卡完成记录
- 已解锁内容
- 图鉴发现状态
- 外循环货币
- 花园持久状态
- 玩家设置或 profile

这些状态应通过明确的战斗结果输出接口更新，而不是直接读取 battle runtime 内部对象。

---

## 第一版外围数据边界

### 1. Battle Result

未来外循环至少需要一份最小战斗结果：

```text
BattleResult
- scenario_id
- completed
- result_type (victory / defeat / aborted)
- victory_reason
- defeat_reason
- elapsed_time
- completed_wave_ids
- remaining_plants
- defeated_enemy_count
```

当前设计要求：

- 它是战斗对外输出的数据
- 它不是 battle 内部所有状态的完整快照

### 2. Unlock Input

解锁系统未来至少需要消费：

```text
UnlockInput
- scenario_id
- result_type
- completion_stars_or_grade (可选，后置)
- first_clear
- best_clear
```

### 3. Almanac Metadata

图鉴系统未来至少需要消费：

```text
AlmanacEntrySource
- template_id
- display_name
- entity_kind
- tags
- short_summary
- source_pack_id (可选)
```

### 4. Save Snapshot

长期存档未来至少需要承载：

```text
SaveSnapshot
- profile_id
- unlocked_scenarios
- unlocked_templates
- discovered_entries
- persistent_currency
- external_system_state
```

## 当前需要的接口与当前不需要的 UI

### 当前需要先占位的接口

- 战斗结果输出结构
- 关卡目录与模式目录的最小元数据
- 模板元数据的图鉴消费边界
- 解锁系统消费哪些 battle result 字段
- 长期存档至少会保存哪些摘要

### 当前不需要实现的 UI

- 完整关卡地图界面
- 完整存档槽 UI
- 完整图鉴翻页界面
- 完整商店购买界面
- 完整花园互动界面

---

## 第六阶段输入清单

第六阶段真正开始前，至少应具备：

### 已稳定项

- `v1` 协议边界
- 扩展包 manifest 边界
- 分层回归体系
- 内容工作流
- `BattleManager` 入场守则

### 已预留项

- 外循环系统地图
- Battle Result 最小输出结构
- Unlock Input 最小输入结构
- Almanac Metadata 最小消费结构
- Save Snapshot 最小存储结构

### 仍允许试验项

- 更细的评分系统
- 更复杂的存档格式
- 图鉴展示字段
- 商店经济规则
- 花园交互细节

### 当前明确不提前实现

- 完整关卡 UI
- 完整 campaign 流程
- 完整商店闭环
- 完整花园玩法
- 完整社区/分享系统

---

## 建议的第六阶段优先顺序

如果进入第六阶段，建议顺序如下：

1. Battle Result 正式化
2. 关卡目录与模式目录元数据
3. 解锁系统最小闭环
4. 存档最小闭环
5. 图鉴最小入口
6. 商店/花园继续后置

## Epic F 完成口径

这份文档完成后，`Epic F` 的三个 Issue 可按下面口径验收：

- `Issue F1`：已梳理选关、存档、解锁、图鉴、商店、花园六类外循环系统地图。
- `Issue F2`：已明确 Battle Result、Unlock Input、Almanac Metadata、Save Snapshot 四类外围数据边界。
- `Issue F3`：已给出第六阶段输入清单，并区分已稳定项、已预留项、可试验项和明确不提前实现的内容。

---

## 后续衔接

完成 `Epic F` 后，这一轮 `Epic CDEF` 的核心收口就基本完成。

后续应优先回到：

- 回归分层是否顺手
- 内容工作流是否足够低摩擦
- `BattleManager` 的第一批代码级收口是否值得启动

而不是直接跳去做完整外循环 UI。
