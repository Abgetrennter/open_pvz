# 第五阶段可执行任务清单

> 这份清单把“第五阶段（错误技、扩展与工具期）”拆成可直接认领的 `Epic / Issue / checklist`。它的目标不是立刻把所有外循环和工具都做完，而是在第四阶段已完成的基础上，优先推进最能验证项目北极星的那一批工作。

---

## 文档定位

这篇文档主要回答：

- 第五阶段现在可以直接开干的任务有哪些
- 每项任务的产出物是什么
- 每项任务依赖什么前置条件
- 每项任务完成后如何判断“算做完”

这篇文档不负责：

- 代替完整路线图
- 一次性规划最终版扩展生态
- 提前展开全部外循环 UI
- 把编辑器、可视化和扩展平台一起抬成同等优先级

## 当前判断

结合当前仓库状态，第五阶段的起点应按下面前提理解：

1. 第四阶段核心玩法系统已经完成最小正式接入。
2. 当前主干已经同时具备：
   - 资源系统
   - 棋盘与角色分层
   - 卡片系统
   - 波次与战局阶段
   - 状态系统
   - 场上物件代表性接入
3. 当前项目已经不缺“玩法闭环”，下一阶段最值得验证的是：
   - 规则表达力
   - 错误技样例
   - 扩展入口
   - 调试与内容工具

因此第五阶段当前最合理的定义是：

> 先用一批手工错误技样例和扩展入口验证主干表达力，再逐步补工具与外围系统，而不是立刻冲完整外循环。

---

## 使用方式

建议把下面的条目按两层理解：

- `Epic`：一组需要连续推进的主任务
- `Issue`：可直接认领、可单独验收的任务

如果当前人力有限，推荐按下面顺序推进：

1. `Epic A` 错误技样例第一批
2. `Epic B` 错误技验证矩阵与样例分类
3. `Epic C` 扩展入口与数据包最小主链
4. `Epic D` 调试与可视化增强
5. `Epic E` 外循环与工具 backlog 收口

---

## 第五阶段边界提醒

当前应坚持下面 5 条边界：

1. 先做手工错误技样例，不先做完整随机生成器。
2. 先做可验证的扩展入口，不先做完整社区工坊体系。
3. 先增强现有调试和验证面，不先做大而全编辑器。
4. 先把错误技和扩展建立在当前主干上，不反向打散第四阶段边界。
5. 外循环系统继续作为规划位和后续任务，而不是抢占错误技主线。

## Epic A：错误技样例第一批

当前这一组任务的收口文档建议统一沉淀到：

- `plans/第五阶段-错误技样例第一批清单.md`

### Issue A1：确定第一批错误技样例范围

- 目标：
  - 把错误技从抽象方向收敛成一批可直接制作和验证的代表性样例
- 产出物：
  - `plans/第五阶段-错误技样例第一批清单.md`
- 依赖：
  - 无
- 完成判定：
  - 至少明确 5 类代表样例：
    - 受伤反击型
    - 死亡爆炸型
    - 命中分裂型
    - 追踪抛射型
    - 周期召唤型
- checklist：
  - [ ] 列出第一批错误技样例及其目标能力域
  - [ ] 说明每个样例主要验证哪条主干
  - [ ] 标出哪些样例只需现有系统组合即可完成
  - [ ] 标出哪些样例需要扩新增效果或扩展入口

### Issue A2：把第一批错误技样例接成正式场景

- 目标：
  - 让错误技样例成为正式可运行、可观察、可回归的内容，而不是散落构想
- 产出物：
  - 一组 `scenes/validation/*_chaos_validation.tres` 或等价命名场景
- 依赖：
  - `Issue A1`
- 完成判定：
  - 至少 3 个样例进入正式验证场景
  - 样例差异主要来自模板与规则组合，而不是单独业务脚本
- checklist：
  - [ ] 至少落地 3 个错误技样例场景
  - [ ] 每个场景有明确 goals 和 validation_rules
  - [ ] 每个场景能通过现有运行时入口启动
  - [ ] 避免为了样例临时往 `BattleManager` 塞新特判

## Epic B：错误技验证矩阵与样例分类

### Issue B1：在验证矩阵中补错误技专项区块

- 目标：
  - 让错误技不再只是“以后做”的方向，而是有正式矩阵位置
- 产出物：
  - 更新后的 [验证矩阵](../wiki/03-content-validation/32-验证矩阵.md)
- 依赖：
  - `Issue A2`
- 完成判定：
  - 已新增错误技专项区块
  - 每个样例场景有明确观察点和状态
- checklist：
  - [ ] 新增错误技专项区块
  - [ ] 为每个错误技样例补矩阵行
  - [ ] 记录关键观察点和风险说明
  - [ ] 接入 `tools/validation_scenarios.json`

### Issue B2：建立样例分类与覆盖表

- 目标：
  - 避免后续错误技样例越加越乱
- 产出物：
  - `plans/第五阶段-错误技样例覆盖表.md`
- 依赖：
  - `Issue A1`
- 完成判定：
  - 能按能力域和样例类型快速知道“现有样例已经覆盖了什么”
- checklist：
  - [ ] 用表格整理错误技样例和对应能力域
  - [ ] 标出哪些能力尚无样例覆盖
  - [ ] 标出哪些样例适合作为 smoke test
  - [ ] 标出哪些样例适合作为回归基线

## Epic C：扩展入口与数据包最小主链

### Issue C1：确定最小扩展包入口边界

- 目标：
  - 让“扩展内容从哪里进来”成为正式边界，而不是继续散落在未来规划里
- 产出物：
  - `plans/第五阶段-扩展入口最小主链.md`
- 依赖：
  - 无
- 完成判定：
  - 已明确第一版扩展入口只覆盖什么：
    - 模板
    - 场景
    - 样例
    - 可能的效果扩展位
- checklist：
  - [ ] 明确第一版扩展内容范围
  - [ ] 明确哪些内容仍不进入扩展包主线
  - [ ] 明确加载、注册和验证入口
  - [ ] 明确与主仓模板树的边界

### Issue C2：做一个最小扩展包 smoke test

- 目标：
  - 用一个最小外部内容包证明扩展主链可行
- 产出物：
  - 一个最小扩展包目录和 smoke test 场景
- 依赖：
  - `Issue C1`
- 完成判定：
  - 外部内容可以在不修改主干代码的前提下被加载并验证
- checklist：
  - [ ] 建立一个最小扩展包目录
  - [ ] 至少接入一个模板或验证场景
  - [ ] 跑通一个扩展包 smoke test
  - [ ] 补最小失败日志和守卫口径

## Epic D：调试与可视化增强

### Issue D1：补战斗玩法层可视化观察点

- 目标：
  - 让错误技和扩展阶段的排错成本下降
- 产出物：
  - 调试覆盖增强，可能包括 overlay、event log 或 runtime snapshot 扩展
- 依赖：
  - 无
- 完成判定：
  - 至少能更直观看到：
    - 当前资源
    - 当前战局阶段
    - 活跃波次
    - slot 角色占位
    - 活跃状态
- checklist：
  - [ ] 补玩法层核心状态的调试可见性
  - [ ] 避免让调试逻辑侵入业务主链
  - [ ] 补一份使用说明
  - [ ] 保证 headless 跑法仍可工作

### Issue D2：整理专项调试命令和工作流

- 目标：
  - 把“怎么调某类问题”从口头经验变成可复用流程
- 产出物：
  - 一份新的调试工作流文档或现有文档更新
- 依赖：
  - `Issue D1`
- 完成判定：
  - 能快速知道如何调：
    - 错误技组合问题
    - 扩展包加载问题
    - 战局流程问题
- checklist：
  - [ ] 整理常用命令
  - [ ] 补日志位置和快照位置
  - [ ] 补常见故障定位顺序
  - [ ] 让文档可直接被新协作者使用

---

## 第五阶段完成定义

这一阶段不要求把所有扩展和外围系统都做完，但至少应满足：

- 已有一批正式错误技样例进入验证体系
- 错误技差异主要来自规则组合而不是专用业务脚本
- 已有一个最小扩展入口 smoke test
- 调试与验证面已能支撑错误技和扩展问题定位
- 外循环与大型工具链仍被清楚留在后续 backlog，而不是重新抢主线
