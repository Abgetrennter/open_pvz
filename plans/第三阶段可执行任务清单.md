# 第三阶段可执行任务清单

> 这份清单把“第三阶段（规则协议期）”进一步拆成可直接开工的 issue/checklist。它的目标不是把所有未来系统都提前协议化，而是在当前真实样例、验证矩阵和协议守卫基础上，冻结第一版可校验、可复用、可回归的规则协议。

---

## 文档定位

这篇文档主要回答：

- 第三阶段现在可以直接开干的任务有哪些
- 每项任务的产出物是什么
- 每项任务依赖什么前置条件
- 每项任务完成后如何判定“算做完”

这篇文档不负责：

- 代替整体路线图
- 提前展开第四阶段玩法系统
- 一次性冻结所有未来触发器、效果和玩法协议
- 把编辑器、DSL、扩展包格式拉成当前主线

配套阅读建议：

- [当前阶段与实现路线](../wiki/01-overview/23-当前阶段与实现路线.md)
- [开发路线图](../wiki/04-roadmap-reference/26-开发路线图.md)
- [第三阶段协议冻结输入清单](../wiki/05-governance/36-第三阶段协议冻结输入清单.md)
- [验证矩阵](../wiki/03-content-validation/32-验证矩阵.md)

## 使用方式

建议把下面的条目按两层理解：

- `Epic`：一组需要连续推进的主任务
- `Issue`：可直接开工、可单独验收的任务

如果当前人力有限，推荐先按下面顺序推进：

1. `Epic A` 冻结范围收口
2. `Epic B` 第一版触发器协议冻结
3. `Epic C` 第一版效果协议冻结
4. `Epic D` 兼容入口收缩与迁移
5. `Epic E` 协议验证矩阵与冻结结论

这个顺序背后的原则是：

- 先缩范围，再冻字段
- 先稳高频主链，再处理兼容入口
- 先补验证和迁移，再谈下一轮扩展

---

## 第三阶段的边界提醒

第三阶段最容易失控的地方，不是“字段不够多”，而是把本该属于后续阶段的问题一起卷进来。

当前应坚持下面 4 条边界：

1. 只冻结当前高频、已被样例和验证共同证明需要稳定的那一小批触发器、效果和目标语义。
2. 不把阳光、卡片、放置、波次、状态系统等第四阶段玩法协议提前展开。
3. 不为了“看起来完整”而引入新的协议层语言、独立配置编译器或大型编辑器工具链。
4. 不把错误技生成器当成第三阶段主线，最多只保留少量表达力 smoke test。

## Epic A：冻结范围收口

当前这一组任务的收口文档建议统一沉淀到：

- [第三阶段第一轮冻结范围与兼容计划](../wiki/05-governance/37-第三阶段第一轮冻结范围与兼容计划.md)

### Issue A1：确认第一轮冻结范围

- 目标：
  - 把第三阶段从“未来可能很大”收敛成“这一轮实际冻结什么”
- 产出物：
  - 一份冻结范围清单
  - 对应路线图和当前阶段文档更新
- 依赖：
  - 无
- 完成判定：
  - 第一轮明确只覆盖：
    - 触发器：`periodically / when_damaged / on_death`
    - 效果：`damage / spawn_projectile / explode`
    - 高频目标语义和资源类型约束
  - `heal / apply_modifier / spawn_entity` 被明确标成后续候选，而不是当前硬冻
- checklist：
  - [ ] 对照当前样例库确认高频触发器和效果
  - [ ] 对照负向专项确认必须先拦截的非法组合
  - [ ] 在路线图中写清楚第一轮冻结范围和非目标
  - [ ] 把后续候选项整理进 backlog，而不是继续混在当前阶段

### Issue A2：明确默认值来源和覆盖优先级

- 目标：
  - 避免同一个字段同时在模板、绑定、场景覆盖和运行时里各自偷带默认值
- 产出物：
  - 一份字段来源表
  - 对应的校验与文档更新
- 依赖：
  - `Issue A1`
- 完成判定：
  - 每个高频字段都能回答：
    - 默认值从哪里来
    - 哪一层可以覆盖它
    - 覆盖顺序是什么
  - 不再存在“实现碰到空值就顺手补一个”的隐式默认值
- checklist：
  - [ ] 梳理 `TriggerBinding`、`EntityTemplate`、`ProjectileTemplate`、`BattleSpawnEntry` 的高频字段来源
  - [ ] 标出只允许模板定义、只允许场景覆盖和只允许运行时临时写入的字段
  - [ ] 更新协议输入文档和模板编写约定
  - [ ] 为关键覆盖链补至少 1 条回归验证

### Issue A3：明确兼容入口和废弃计划

- 目标：
  - 让第三阶段不是继续兼容面越滚越大，而是开始收口
- 产出物：
  - 一份兼容入口清单
  - 一份废弃节奏说明
- 依赖：
  - `Issue A1`
  - `Issue A2`
- 完成判定：
  - 明确哪些入口仍保留兼容：
    - `runtime_behavior`
    - 旧 `params` 兼容写法
    - 老式覆盖字段
  - 明确哪些入口第三阶段后半段应转成 warning 或 hard error
- checklist：
  - [ ] 列出当前仍在被核心样例使用的兼容入口
  - [ ] 标出“允许继续读但不再建议新增”的入口
  - [ ] 标出“本阶段收尾前必须迁完”的入口
  - [ ] 在 wiki 中写清楚新内容应该走哪条主入口

## Epic B：第一版触发器协议冻结

### Issue B1：冻结第一版触发器白名单

- 目标：
  - 把当前真正跑通的触发器整理成明确协议，而不是继续默认“都能扩”
- 产出物：
  - 更新后的触发器协议文档
  - 对应 `TriggerDef` 白名单与校验同步
- 依赖：
  - `Issue A1`
- 完成判定：
  - `periodically / when_damaged / on_death` 的字段子集、类型和边界都被写清楚
  - 新内容作者可以明确知道什么能写、什么不能写
- checklist：
  - [x] 固定 `periodically.interval` 的类型和下界
  - [x] 固定 `when_damaged.min_damage` 的类型和默认语义
  - [x] 固定 `on_death` 的无条件语义
  - [x] 明确 `event_name` 是否允许内容层改写

### Issue B2：收紧 `TriggerBinding` 的作者入口

- 目标：
  - 让“模板如何声明行为”成为稳定入口，而不是在 `runtime_behavior` 里继续长新键
- 产出物：
  - 更新后的 `TriggerBinding` 写法约定
  - 对应模板样例迁移
- 依赖：
  - `Issue B1`
  - `Issue A3`
- 完成判定：
  - 第一轮冻结范围内的新样例默认走 `TriggerBinding`
  - `runtime_behavior` 不再承担新增正式协议字段
- checklist：
  - [x] 列出 `TriggerBinding` 当前正式支持的字段
  - [x] 标出与 `runtime_behavior` 的一一对应关系
  - [x] 迁移当前高频样例到统一入口
  - [x] 更新模板编写约定

### Issue B3：统一触发器错误语义

- 目标：
  - 让触发器非法输入不是“碰到哪里炸哪里”，而是在入口处可解释地失败
- 产出物：
  - 触发器侧错误拦截规则
  - 对应负向验证
- 依赖：
  - `Issue B1`
- 完成判定：
  - 非法条件类型、缺失必需字段、超出白名单的额外字段都能稳定进入守卫路径
  - 失败结果能在日志和验证里解释清楚
- checklist：
  - [x] 覆盖错误类型输入
  - [x] 覆盖缺失条件字段
  - [x] 覆盖多余条件字段
  - [x] 补触发器侧负向专项

## Epic C：第一版效果协议冻结

### Issue C1：冻结第一版效果参数子集

- 目标：
  - 先把当前最高频、最成熟的效果参数写稳，而不是继续堆新效果
- 产出物：
  - 更新后的效果协议文档
  - 对应 `EffectDef` 白名单和默认值定义
- 依赖：
  - `Issue A1`
  - `Issue A2`
- 完成判定：
  - 当前第一轮只正式冻结：
    - `damage.amount / target_mode`
    - `explode.amount / target_mode / radius`
    - `spawn_projectile.speed / direction / damage / movement_mode / flight_profile / projectile_template / turn_rate / travel_duration / impact_radius / collision_padding`
  - 其他效果继续保留规划位，但不挤进当前主线
- checklist：
  - [x] 固定每个参数的类型、默认值和是否必填
  - [x] 明确哪些参数必须来自资源，哪些允许场景覆盖
  - [x] 更新效果系统文档
  - [x] 更新样例和验证引用

### Issue C2：冻结目标选择、子槽位和链路边界

- 目标：
  - 不只冻结“字段名字”，还冻结它们如何组合
- 产出物：
  - 一份第一版目标语义和槽位语义说明
  - 对应运行时与校验同步
- 依赖：
  - `Issue C1`
- 完成判定：
  - 当前正式支持的高频 `target_mode` 子集被写清楚
  - `spawn_projectile.on_hit` 的可用子效果白名单固定
  - 链深、嵌套和终止保护仍然保留现有防护，不额外发明新层
- checklist：
  - [x] 固定高频 `target_mode` 子集
  - [x] 固定 `spawn_projectile.on_hit` 只允许 `damage / explode`
  - [x] 复核深度上限和嵌套保护
  - [x] 补组合约束文档

### Issue C3：冻结资源类型和组合约束

- 目标：
  - 把当前负向专项里已经出现的真实边界，上升成正式协议
- 产出物：
  - 一份组合约束清单
  - 对应 validator 和负向验证更新
- 依赖：
  - `Issue C1`
  - `Issue C2`
- 完成判定：
  - 至少下面几类组合被稳定拦截：
    - 错误资源脚本类型
    - `movement_mode` 与 `flight_profile` 不一致
    - 非法子效果
    - 白名单外参数
    - 不支持的目标模式
- checklist：
  - [x] 固定资源脚本类型约束
  - [x] 固定飞行配置一致性约束
  - [x] 固定白名单外参数默认报错
  - [x] 补协议守卫验证

## Epic D：兼容入口收缩与迁移

### Issue D1：让协议定义和校验保持同一套事实来源

- 目标：
  - 避免文档、`Def` 资源、validator、样例各自说一套
- 产出物：
  - 一次协议对齐清理
  - 对应实现和文档同步记录
- 依赖：
  - `Issue B1`
  - `Issue C1`
- 完成判定：
  - 新增或修改一个冻结字段时，能明确知道需要同步哪些位置
  - 不再长期存在“文档写允许，但 validator 已禁用”或相反情况
- checklist：
  - [x] 对齐 `TriggerDef`、`EffectDef`、validator 和 wiki
  - [x] 对齐正向样例和负向样例的字段集合
  - [x] 清掉明显过期的兼容描述
  - [x] 记录同步规则

### Issue D2：迁移现有高频样例到冻结子集

- 目标：
  - 证明冻结后的协议不是纸面清单，而是真能承载当前主干样例
- 产出物：
  - 更新后的模板、绑定和验证样例
- 依赖：
  - `Issue B2`
  - `Issue C3`
- 完成判定：
  - 当前核心植物、僵尸、投射物和验证样例都落在冻结子集内
  - 行为差异主要来自协议组合，而不是临时特判
- checklist：
  - [x] 复核第一批植物样例
  - [x] 复核第一批僵尸样例
  - [x] 复核投射体模板和飞行配置样例
  - [x] 清理仅因历史兼容保留的写法

### Issue D3：把兼容收缩结果写成作者可执行约定

- 目标：
  - 让后续新增内容的人知道现在该写什么，不该写什么
- 产出物：
  - 更新后的模板编写约定
  - 一份第三阶段后的作者说明
- 依赖：
  - `Issue D1`
  - `Issue D2`
- 完成判定：
  - 新内容作者能根据文档直接判断：
    - 行为写 `TriggerBinding` 还是兼容 `runtime_behavior`
    - 效果参数有哪些正式入口
    - 哪些老字段已不建议继续使用
- checklist：
  - [x] 更新模板编写约定
  - [x] 更新协议冻结输入清单
  - [x] 更新路线图引用
  - [x] 明确 warning/hard error 的切换时机

## 最小可交付批次建议

如果只先做一轮“最小可交付”，建议以这 8 个 issue 为第一批：

- [ ] `Issue A1` 确认第一轮冻结范围
- [ ] `Issue A2` 明确默认值来源和覆盖优先级
- [ ] `Issue B1` 冻结第一版触发器白名单
- [ ] `Issue C1` 冻结第一版效果参数子集
- [ ] `Issue C2` 冻结目标选择、子槽位和链路边界
- [ ] `Issue C3` 冻结资源类型和组合约束
- [ ] `Issue E1` 补齐协议正向样例覆盖
- [ ] `Issue E2` 扩协议守卫负向矩阵

完成这 8 项后，第三阶段就会从“知道要冻协议”推进到“已经有一版真正可复用、可回归的协议边界”。

## 完成定义

第三阶段不要求把规则系统一次性写成终局版本，但至少应满足：

- 当前高频触发器和效果已有第一版正式白名单
- 默认值来源、覆盖优先级和资源类型约束已写清楚
- 非法组合能在入口处被稳定拦截
- 当前核心样例已迁到冻结子集内
- 验证矩阵能同时证明“哪些组合被允许”和“哪些组合被禁止”

### 当前建议

第三阶段最重要的不是“再发明更多触发器和效果”，而是：

- 收口当前主干真实在用的那一批协议
- 让样例、校验、文档和回归指向同一套事实
- 把玩法系统和大型工具链继续留在后续阶段

只有这样，第四阶段玩法系统接入时，才不会重新踩回“大管理器 + 临时特判”的老路。
