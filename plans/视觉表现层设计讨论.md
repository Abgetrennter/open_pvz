## 1. 设计前提

从项目 wiki 和代码中提取的关键约束：

1. **逻辑与呈现分离** — 实体是纯逻辑节点，视觉层只"观察"不应"驱动"规则
2. **Resource 驱动** — 所有配置走 `.tres`，而非硬编码
3. **事件驱动** — 视觉变化应由 `EventBus` 事件触发（`entity.damaged` → 受击闪烁）
4. **扩展包友好** — 逻辑 ID → 实际资源的间接映射，支持换皮/主题
5. **当前实体用 `_draw()` 占位** — 视觉层需要平滑替换这些占位代码，向后兼容
6. **阶段定位是支撑层规范化** — 当前重点是让正式战斗内容更可见、更可维护，而不是把视觉层当成决定主线能否推进的前置门槛
7. **素材与内容解耦** — 内容包只持有逻辑素材 ID，由 `AssetRegistry` 解析实际资源；同一内容包可搭配不同素材包实现换皮

### 参考项目（PVZ-Godot-Dream）的视觉架构

参考项目使用继承层级 + 组件混合的模式：

```
Character000Base (Node2D)
├── Body (BodyCharacter) — 中央视觉状态管理，颜色调制/效果叠加
├── Shadow (Sprite2D) — 阴影渲染
├── AnimComponent (AnimComponentBase) — 动画控制器（策略模式）
│   ├── AnimComponentNorm — AnimationTree 驱动
│   ├── AnimComponentPlayer — AnimationPlayer 驱动
│   └── AnimComponentEmpty — 无动画
├── HpComponent — 血条 UI
└── HurtBoxComponent — 碰撞检测
```

**BodyCharacter 核心能力：**
- 基础颜色 + 多层效果叠加（受击白色、冰冻蓝色、催眠紫色、焦黑）
- Shader 遮罩（地下/水面效果）
- Tween 驱动的闪烁/脉冲动画

**参考项目的局限：** 继承层级深（Character000Base → Plant000Base/Zombie000Base → 具体），视觉效果种类硬编码为枚举 `E_ChangeColors`，不易扩展。Open PVZ 的组合优先设计可以做得更好。

### 原版 de-pvz 与动画迁移工作的补充启发

结合 `de-pvz` 原版和 `PVZ-Godot-Dream` 的动画组件 / 迁移工具链，还可以再提炼出 3 点：

1. **原版真正稳定的是"轨道 / 挂点 / 覆盖层"思想，而不是具体引擎实现。**
   - `de-pvz` 的 `Reanimation` 运行时支持轨道实例、颜色覆盖、附加色、挂点附着和按轨道替换图片。
   - 这说明后续如果 Open PVZ 真的要进入正式动画管线，最值得保留的是"部件级状态和挂点抽象"，不是照搬原版数据结构。

2. **`PVZ-Godot-Dream` 值得借的是动画驱动分层。**
   - `AnimComponentBase -> AnimComponentNorm / AnimComponentPlayer / AnimComponentEmpty`
   - 它把"速度倍率修正、动画结束信号、空实现兜底"从角色逻辑中抽出来了。

3. **原版 reanim 资源迁移应视为离线工作流，不应成为当前运行时前置依赖。**
   - `PVZ-Godot-Dream` 自带 `R2Ga_PVZ` 转换工具，说明它也是把"原版资源导入"与"Godot 播放"分开处理。
   - 因此 Open PVZ 当前更合理的策略是：运行时只消费 Godot 原生 actor scene；原版 `.reanim` 只作为本地 ignored 的离线输入，由导入工具转换为 Godot 可播放资源、组合场景和语义报告。
   - 简单角色可以由 `AnimationPlayer` 直接承载；Peashooter、Sunflower、ThreePeater 这类有 body/head/overlay/attachment 语义的角色，应由 wrapper actor scene 自己封装组合关系。

## 2. 架构方案：组件桥接模式

### 2.1 核心思路

视觉表现作为**可插拔组件**接入实体，不改变实体的逻辑职责。实体不知道自己"长什么样"，只暴露状态查询接口。

```
BaseEntity (Node2D) — 纯逻辑
  ├── HealthComponent       ← 已有
  ├── TriggerComponent      ← 已有
  ├── HitboxComponent       ← 已有
  ├── MovementComponent     ← 已有
  └── VisualActorComponent  ← 当前落地点
        ├── ActorRoot (Node2D) — VisualProfileDef.actor_scene 实例
        │   ├── AnimationPlayer / AnimatedSprite2D / AnimationTree
        │   └── 可选 wrapper 脚本：body/head/overlay/attachment 组合
        ├── ShadowAnchor — 抛射体投影阴影锚点
        └── 后续 FeedbackStack — 反馈效果栈
```

当前代码已经采用 `VisualActorComponent + VisualProfileDef.actor_scene` 的轻量桥接形态。旧草案里的 `SpriteSet` / `AnimDriver` 仍可作为简单素材和未来扩展策略使用，但不再要求所有视觉资源都先拆成 `SpriteSet`；对于原版 `.reanim` 移植，`actor_scene` 是更合适的运行时边界。

### 2.2 关键决策：VisualActorComponent 的位置

| 方案 | 描述 | 优势 | 劣势 |
|------|------|------|------|
| **A. 实体子组件** | VisualActorComponent 作为 BaseEntity 子节点 | 符合已有组件模式，EntityFactory 统一装配 | 视觉生命周期绑死实体 |
| **B. 独立渲染层** | 场景树中单独一层 VisualLayer，通过 entity_id 关联 | 最干净的逻辑/视觉分离 | 需同步位置/状态，复杂度高 |
| **C. 混合桥接** | 实体上挂轻量桥接组件，实际渲染节点独立管理 | 平衡分离与简洁 | 接口设计需谨慎 |

**倾向：方案 A 作为起始。** 理由：

- BaseEntity 是 `Node2D`，子节点变换自动跟随，无需手动同步位置
- EntityFactory 不需要大改，在现有装配流程中加一步即可
- 如果未来需要独立渲染层，VisualActorComponent 接口可保持不变，只改内部实现
- YAGNI — 在真正遇到性能瓶颈前不引入独立渲染层的复杂度

### 2.3 职责边界

**VisualActorComponent 负责：**
- 监听 EventBus 事件 → 触发视觉反馈
- 查询 EntityState → 同步动画状态
- 实例化并管理 `VisualProfileDef.actor_scene`
- 对外暴露稳定的 actor scene 播放协议
- 管理 ShadowAnchor / 后续 FeedbackStack 的生命周期

**VisualActorComponent 不做：**
- 不影响游戏逻辑（不修改 health、不改变位置）
- 不发射影响规则的事件（纯消费端）
- 不直接解析原版 `.reanim`
- 不内置具体植物的 body/head/overlay 特判

## 3. 资源定义设计

### 3.1 VisualProfile（视觉档案）

```gdscript
# scripts/core/defs/visual_profile.gd
class_name VisualProfile
extends Resource

@export var logical_id: StringName           # 逻辑资源 ID，AssetRegistry 用作解析键
@export var sprites: SpriteSet               # 精灵资源集
@export var animations: AnimationMapping     # 动画状态映射
@export var feedback_map: FeedbackMap        # 状态→效果映射表
@export var shadow_config: ShadowConfig      # 阴影配置
@export var default_scale: Vector2 = Vector2.ONE
@export var default_offset: Vector2 = Vector2.ZERO
@export var facing_direction: int = 1        # 1=右, -1=左（僵尸翻转）
```

### 3.1.1 当前落地：VisualProfileDef

当前仓库已经落地的是更薄的 `VisualProfileDef`，它继承统一 contributor 定义，通过 `VisualProfileRegistry` 注册，供 `CombatArchetype.visual_profile_id` 间接引用：

```gdscript
class_name VisualProfileDef
extends RegistryContributorDef

@export var actor_scene: PackedScene = null
@export var animation_map: Dictionary = {}
@export var state_animation_map: Dictionary = {}
@export var action_animation_map: Dictionary = {}
@export var status_visual_map: Dictionary = {}
@export var damage_stage_defs: Array[Dictionary] = []
@export var shadow_policy: Dictionary = {}
@export var z_policy: Dictionary = {}
@export var default_scale: Vector2 = Vector2.ONE
@export var ground_offset: Vector2 = Vector2.ZERO
```

其中 `actor_scene` 是当前最重要的边界：运行时只实例化 Godot scene，不关心它来自手工制作、SpriteFrames、AnimationPlayer，还是本地 `.reanim` 导入器。`state_animation_map` 负责状态到动画的常规映射，`action_animation_map` 负责把视觉动作 ID 映射到 actor 可理解的动作或动画；`animation_map` 保留为动画别名层，方便导入产物把 `shoot` 映射到 `shooting` 之类的实际动画名。

`default_scale` 与 `ground_offset` 只做最小装配修正，不承担原版 attachment 或 overlay 校准。复杂组合应放在 actor scene 自己的 wrapper 脚本中。

### 3.1.2 动画语义化原则

后续正式视觉配置不应把“动画文件名”当成稳定契约，而应把 `idle`、`shoot`、`explode`、`squash`、`produce_sun` 这类**语义动作**作为契约。clip 名、部件组合、特效时机都属于某个素材实现的细节，可以被替换。

例如战斗或视觉 cue 不应直接要求“播放 `ThreePeater_head_idle2`”，而应要求“执行 `shoot`”。具体表现可以由视觉配置决定：

- 普通 Peashooter：head 播 `shooting`，muzzle 生成普通豌豆视觉。
- Repeater：head 播 `shooting`，muzzle 分两次生成普通豌豆视觉。
- 各异三线射手：上头双发普通豌豆，中头发寒冰豌豆，下头发火焰豌豆；三个 head 都挂在同一个 body 的不同 part slot 上。

这让视觉层和 Mechanic-first 主链保持同一种设计风格：逻辑动作稳定，视觉实现可替换、可组合。

### 3.2 SpriteSet（精灵资源集）

SpriteSet 同时支持两种引用模式，与扩展系统的 AssetRegistry 对齐：

```gdscript
# scripts/core/defs/sprite_set.gd
class_name SpriteSet
extends Resource

@export var use_logical_ids: bool = false    # false=直接引用, true=通过 AssetRegistry 解析

# 直接引用模式（主仓占位、快速原型阶段使用）
@export var idle_frames: SpriteFrames
@export var attack_frames: SpriteFrames
@export var damage_frames: SpriteFrames
@export var death_frames: SpriteFrames
@export var move_frames: SpriteFrames
@export var special_frames: Dictionary       # StringName -> SpriteFrames

# 逻辑 ID 模式（素材包系统就绪后使用）
@export var idle_frames_id: StringName
@export var attack_frames_id: StringName
@export var damage_frames_id: StringName
@export var death_frames_id: StringName
@export var move_frames_id: StringName
@export var special_frames_ids: Dictionary   # StringName -> StringName
```

**解析优先级：** 当 `use_logical_ids == true` 时，运行时优先走 `AssetRegistry.resolve(resource_id)` 回退链（当前激活素材包 → 依赖素材包 → 默认占位素材包），如果解析失败则 fallback 到直接引用字段。这样保证了过渡期兼容性。

### 3.3 AnimationMapping（动画状态映射）

```gdscript
# scripts/core/defs/animation_mapping.gd
class_name AnimationMapping
extends Resource

@export var state_map: Dictionary = {
    &"idle": &"idle_frames",
    &"attack": &"attack_frames",
    &"damaged": &"damage_frames",
    &"dying": &"death_frames",
    &"moving": &"move_frames",
}
@export var driver_type: StringName = &"frames"   # frames / anim_player / tree / null
```

`driver_type` 决定视觉层内部选用哪个 AnimDriver 策略。内置策略（`frames` / `null`）属于级别 A 数据扩展；外部策略（如 `tree`）属于级别 C 策略扩展，需要 `trusted_script` 信任级别。

### 3.4 FeedbackMap（反馈映射表）

```gdscript
# scripts/core/defs/feedback_map.gd
class_name FeedbackMap
extends Resource

@export var on_damaged: FeedbackEntry
@export var on_death: FeedbackEntry
@export var on_healed: FeedbackEntry
@export var on_spawn: FeedbackEntry
@export var status_effects: Dictionary       # StringName -> FeedbackEntry
```

```gdscript
# scripts/core/defs/feedback_entry.gd
class_name FeedbackEntry
extends Resource

@export var effect_type: StringName = &""    # flash / tint / scale / shader / fade
@export var color: Color = Color.WHITE
@export var duration: float = 0.15
@export var intensity: float = 1.0
@export var easing: int = Tween.EASE_OUT
```

FeedbackMap 是扩展系统**级别 A（数据扩展）**的典型场景：新增状态反馈（如 `burning`）只需在 `.tres` 的 `status_effects` 字典里加一条 `FeedbackEntry`，无需改代码，无需策略扩展。

### 3.5 ShadowConfig（阴影配置）

```gdscript
# scripts/core/defs/shadow_config.gd
class_name ShadowConfig
extends Resource

@export var enabled: bool = true
@export var texture: Texture2D = null
@export var offset: Vector2 = Vector2(0, 8)
@export var scale: Vector2 = Vector2(1.0, 0.4)
@export var modulate: Color = Color(0, 0, 0, 0.3)
```

### 3.6 资源引用策略

资源引用分三个阶段演进：

| 阶段 | 策略 | 说明 |
|------|------|------|
| **当前（V0–V2）** | 直接引用 | `SpriteSet.use_logical_ids = false`，VisualProfile 直接引用 SpriteFrames 等资源 |
| **过渡（E1 完成后）** | 双模式 | `SpriteSet.use_logical_ids = true`，优先走 AssetRegistry，解析失败 fallback 到直接引用 |
| **正式（素材包系统就绪后）** | 逻辑 ID 为主 | 内容包只写逻辑 ID，由 `AssetRegistry` 按回退链解析实际资源 |

当前仓库在 `CombatArchetype` 中使用视觉档案 ID 间接引用：

```gdscript
@export var visual_profile_id: StringName = StringName()
```

向后兼容：`visual_profile_id` 为空时保留现有 `_draw()` 占位逻辑，已有验证场景不受影响。

## 4. 运行时组件设计

### 4.1 Actor Scene Contract（当前优先）

`VisualActorComponent` 对外保持轻量 API，并优先把请求委托给 `actor_scene` 根节点：

```gdscript
func play_animation(animation_name: StringName) -> bool
func play_state(state_id: StringName) -> bool
func play_action(action_id: StringName) -> bool
func set_visual_speed(speed_scale: float) -> bool
func get_anchor(anchor_name: StringName) -> Node2D
```

委托规则：

- 如果 `ActorRoot` 实现了同名方法，优先调用 `ActorRoot`，由场景脚本自己决定如何播放、混合和回到 idle。
- 如果 `ActorRoot` 没有实现 `play_animation`，则回退到查找子级 `AnimationPlayer` 并播放同名动画。
- 如果 `ActorRoot` 没有实现 `play_state`，则通过 `VisualProfileDef.state_animation_map` 找动画名，再调用 `play_animation`。
- 如果 `ActorRoot` 没有实现 `play_action`，则通过 `VisualProfileDef.action_animation_map` / `animation_map` 找动画名，再调用 `play_animation`。
- `get_anchor` 用于后续 FX 或组合 actor 查询锚点；无自定义实现时按节点名递归查找 `Node2D`。

这个协议让简单 actor 保持零脚本：只要有 `AnimationPlayer` 即可播放。复杂 actor 则可以在 wrapper scene 内部处理 body/head/overlay/attachment，例如 ThreePeater 的三个 head 跟随 `anim_head1/2/3`，而不把这些植物细节泄漏进 `VisualActorComponent`。

### 4.2 Semantic Action Recipe（动作语义编排）

Actor Scene Contract 解决的是“怎么调用 actor”；Action Recipe 解决的是“一个语义动作怎么演”。它不是战斗逻辑，不决定伤害、命中、冷却或目标死亡，只描述视觉编排。

核心边界：

- `.reanim` 或其他动画资源只提供 clip、track、part、anchor 等素材线索。
- `VisualProfileDef` 声明实体支持哪些语义 state/action。
- Actor scene 或其 wrapper 根据 Action Recipe 把语义 action 拆成 clip 播放、部件动作、root motion、局部 cue、世界 FX cue。
- 战斗逻辑仍由 `CombatArchetype + CombatMechanic[]` 决定，视觉动作只能观察或响应事件。

第一版 Action Recipe 可以先作为本地 `.reanim` manifest 字段存在，字段稳定后再考虑抽成 `VisualActionRecipeDef` Resource。示意：

```json
{
  "actions": {
    "squash": {
      "steps": [
        {"op": "play_action", "action": "look_right", "duration": 0.55},
        {"op": "play_action", "action": "jump_up", "duration": 0.55},
        {"op": "root_motion", "to": "target", "offset": [78, 0], "duration": 0.35},
        {"op": "play_action", "action": "jump_down", "duration": 0.50},
        {"op": "emit_local_cue", "cue": "impact", "at": 0.05},
        {"op": "return_home", "duration": 0.25}
      ]
    }
  }
}
```

这里的 `root_motion` 只移动 actor 的视觉根或本地 offset，不移动实体逻辑节点。Squash 可以看起来扑到目标身上，但实体真实坐标、目标查询、碰撞、伤害结算都不因此改变。

适合首批验证的 op：

- `play_state` / `play_action`：调用 Actor Scene Contract。
- `root_motion`：移动 actor 视觉根，支持 to self / target / anchor / offset。
- `part_action`：让某个 part slot 执行动作。
- `emit_local_cue`：给演示或 actor 内部使用的非战斗 cue，例如目标占位消失、impact 圈。
- `spawn_world_fx`：转交 `VisualActionRunner` / `VisualFxRegistry`，生成火焰、爆炸、喷雾等世界层效果。
- `return_home` / `hide_actor`：一次性植物的视觉收尾。

Squash、Jalapeno、Doom-shroom 是最合适的首批样本：

- Squash：本体 clip 负责看、跳、砸；recipe 负责视觉 root motion 和 impact cue。
- Jalapeno：本体 clip 负责 explode；recipe 负责整行火焰 cue。
- Doom-shroom：本体 clip 负责 explode/sleep；recipe 负责大范围爆炸 cue 和睡眠视觉切换。

### 4.3 Part Slot Composite Actor（部件槽组合）

Part Slot 是对 body/head/overlay/attachment 的语义化命名。它不关心某个原版 track 具体叫什么，而是声明“这里有一个可替换部件槽”，并把它绑定到 actor 内的 anchor 或 attachment track。

典型结构：

```text
CompositeActor
├── body              # 主体 actor，负责整体 idle / sway
├── upper_head        # part slot，挂到 anim_head1
├── middle_head       # part slot，挂到 anim_head2
├── lower_head        # part slot，挂到 anim_head3
└── overlay/blink     # overlay part，跟随语义父层
```

这样可以支持“各异三线射手”这类组合视觉：

```json
{
  "part_slots": {
    "upper_head": {
      "anchor": "anim_head1",
      "profile": "local.original.repeater_head",
      "lane_offset": -1
    },
    "middle_head": {
      "anchor": "anim_head2",
      "profile": "local.original.icepea_head",
      "lane_offset": 0
    },
    "lower_head": {
      "anchor": "anim_head3",
      "profile": "local.custom.firepea_head",
      "lane_offset": 1
    }
  },
  "actions": {
    "shoot": [
      {"part": "upper_head", "action": "shoot_twice", "projectile_visual": "pea", "times": [0.0, 0.18]},
      {"part": "middle_head", "action": "shoot", "projectile_visual": "ice_pea", "times": [0.05]},
      {"part": "lower_head", "action": "shoot", "projectile_visual": "fire_pea", "times": [0.10]}
    ]
  }
}
```

战斗逻辑也可以有对应的 emission 配置，但两者不应互相替代。推荐做法是：战斗 emission 声明真实发射次数、弹种、lane offset；视觉 recipe 通过语义 ID 绑定到对应 part slot 和 projectile visual cue。这样同一个逻辑可以换皮，同一个视觉也能用于不同规则组合。

### 4.4 AnimDriver（动画驱动策略）

```gdscript
# scripts/components/visual/anim_driver.gd
class_name AnimDriver
extends RefCounted

var _anim_node: Node2D

func setup(node: Node2D) -> void:
    _anim_node = node

func play(state: StringName, blend: float = 0.1) -> void:
    push_error("AnimDriver.play() not implemented")

func stop() -> void:
    push_error("AnimDriver.stop() not implemented")

func set_speed_scale(scale: float) -> void:
    pass

func get_current_state() -> StringName:
    return &""

func is_playing() -> bool:
    return false
```

具体实现及与扩展系统的对应关系：

| 驱动 | 适用场景 | 扩展级别 | 说明 |
|------|---------|---------|------|
| `FramesAnimDriver` | 大部分植物、简单实体 | 级别 A（数据扩展） | 只需提供 SpriteFrames 资源，引擎内置 |
| `AnimPlayerDriver` | 需 AnimationPlayer 精细控制的实体 | 级别 A / B | 第一轮正式实现候选 |
| `NullAnimDriver` | 无需动画的实体/占位 | 无需扩展 | 空实现，引擎内置 |
| `TreeAnimDriver` | 僵尸（复杂状态机） | 级别 C（策略扩展） | 需注册新策略 + `trusted_script`，第二批实现 |

AnimDriver 的注册机制对齐扩展系统的 Registry 模式：

- 引擎启动时注册内置策略：`frames` → `FramesAnimDriver`、`null` → `NullAnimDriver`
- 外部规则包可通过 `RuleRegistry` 注册新的 `driver_type`（如 `tree`），由视觉层工厂方法按 `driver_type` 查找并实例化
- 未识别的 `driver_type` 自动降级为 `NullAnimDriver`，保证运行时健壮性

补充建议：

- `AnimPlayerDriver` 应作为第一轮正式实现候选，而不是自动后置到复杂阶段。
- `TreeAnimDriver` 保留为复杂角色和正式动画资源增多后的第二批实现。
- 后续如要接原版 reanim，应新增"离线转换产物 -> AnimationPlayer / SpriteFrames 资源"的工作流，而不是让运行时直接解析原版格式。

### 4.5 FeedbackStack（反馈效果栈）

核心设计原则：**反馈效果是可叠加的，后入的覆盖先入的同类效果。**

```gdscript
# scripts/components/visual/feedback_stack.gd
class_name FeedbackStack
extends RefCounted

var _layers: Dictionary = {}  # StringName(layer_id) -> FeedbackLayer

func push(layer_id: StringName, entry: FeedbackEntry) -> void:
    _layers[layer_id] = FeedbackLayer.new(layer_id, entry)

func pop(layer_id: StringName) -> void:
    _layers.erase(layer_id)

func resolve() -> FeedbackResult:
    var result := FeedbackResult.new()
    for layer in _layers.values():
        result.merge(layer)
    return result
```

```gdscript
class_name FeedbackResult
extends RefCounted

var modulate: Color = Color.WHITE
var scale: Vector2 = Vector2.ONE
var speed_scale: float = 1.0
```

**相比参考项目的改进：** 参考项目用硬编码枚举 `E_ChangeColors` 管理效果种类。Open PVZ 用通用的栈结构 + FeedbackEntry Resource，新增效果类型只需定义新的 `.tres`，无需改代码。这符合项目"组合规则"的核心哲学。

### 4.6 VisualActorComponent 主流程

```gdscript
# scripts/components/visual_actor_component.gd (当前落地骨架)
class_name VisualActorComponent
extends Node

var _owner: Node
var _profile_def: Resource
var _actor_root: Node2D

func bind_profile(profile_def: Resource, owner: Node) -> void:
    _owner = owner
    _profile_def = profile_def
    if profile_def.actor_scene:
        _actor_root = profile_def.actor_scene.instantiate()
        add_child(_actor_root)
        _apply_profile_transform()
    _subscribe_events()
    play_state(&"idle")

func play_state(state_id: StringName) -> bool:
    if _actor_root and _actor_root.has_method(&"play_state"):
        return bool(_actor_root.call(&"play_state", state_id))
    var animation_name := _profile_def.state_animation_map.get(state_id, StringName())
    return play_animation(animation_name)

func play_animation(animation_name: StringName) -> bool:
    if _actor_root and _actor_root.has_method(&"play_animation"):
        return bool(_actor_root.call(&"play_animation", animation_name))
    var player := _find_animation_player()
    if player and player.has_animation(animation_name):
        player.play(animation_name)
        return true
    return false
```

素材包系统就绪前，`VisualActorComponent` 只消费直接引用的 `VisualProfileDef.actor_scene`。未来接入 `AssetRegistry` 时，解析点应放在 `VisualProfileRegistry` 或 `EntityFactory` 装配之前，而不是让 actor scene 自己去查素材包。

## 5. 数据流设计

### 5.1 事件 → 视觉反馈

```
EventBus: entity.damaged
  ↓ (VisualActorComponent 订阅)
VisualActorComponent._on_entity_damaged(event_data)
  ↓
FeedbackMap.lookup("on_damaged") → FeedbackEntry(flash, white, 0.15s)
  ↓
FeedbackStack.push("damaged_flash", entry)
  ↓
Tween: SpriteRoot.modulate → 白色 → 恢复（0.15s）
  ↓
FeedbackStack.pop("damaged_flash")
```

### 5.2 状态 → 动画切换

```
EntityState.health <= 0
  ↓ (VisualActorComponent 在 _process 中检测，或由 entity.died 事件触发)
VisualActorComponent._on_entity_died()
  ↓
AnimationMapping.state_map["dying"] → "death_frames"
  ↓
VisualActorComponent.play_state(&"dying")
  ↓
ActorRoot.play_state(...) 或 AnimationPlayer.play(...)
  ↓
同时 FeedbackStack.push("death_fade", fadeEntry)
```

### 5.3 Status 效果 → 视觉叠加

```
EventBus: entity.status_applied { status_id: "frozen", duration: 3.0 }
  ↓
VisualActorComponent._on_status_applied()
  ↓
FeedbackMap.status_effects["frozen"] → FeedbackEntry(tint, cyan, 3.0)
  ↓
FeedbackStack.push("status:frozen", entry)
  ↓ (持续 3 秒)
FeedbackStack.pop("status:frozen") → 恢复原色
```

### 5.4 语义动作 → Actor / Cue 编排

```
Combat / VisualCue: request action "shoot" or "explode"
  ↓
VisualActorComponent.play_action("shoot")
  ↓
ActorRoot.play_action("shoot")
  ↓
ActionRecipe.resolve("shoot")
  ↓
步骤编排：
  1. body/head/part 播放对应 clip
  2. 必要时对 actor visual root 做 root_motion
  3. 到达 cue moment 时 spawn_world_fx / emit_local_cue
  4. one-shot 结束后回到 next_state
```

这个链路中的 cue moment 只能触发视觉表现，例如火焰线、爆炸圈、喷雾、muzzle flash、目标占位隐藏。真实 projectile 生成、伤害结算、死亡移除仍由战斗事件链负责。若视觉和战斗需要对齐，应通过共享的语义 ID 或事件时间点连接，而不是让视觉层直接修改战斗状态。

## 6. 与扩展系统的集成

本节描述视觉表现层如何嵌入扩展系统的包类型体系、覆盖策略和信任分级。详见 [扩展系统总体规划](../wiki/04-roadmap-reference/38-扩展系统总体规划.md)。

### 6.1 视觉资源的包类型归属

| Resource | 归属包类型 | 扩展级别 | 说明 |
|----------|----------|---------|------|
| `VisualProfile` | `content_pack` | 级别 A | 内容包携带逻辑 ID，不直接内嵌素材 |
| `SpriteSet` | `content_pack`（引用）+ `asset_pack`（实际资源） | 级别 A | 直接引用模式用于原型，逻辑 ID 模式用于正式发布 |
| `AnimationMapping` | `content_pack` | 级别 A | 纯声明式配置 |
| `FeedbackMap` / `FeedbackEntry` | `content_pack` | 级别 A | 纯数据，新增反馈类型无需改代码 |
| `ShadowConfig` | `content_pack` | 级别 A | 纯配置 |
| AnimDriver 策略实现 | `rule_pack` | 级别 C | 新驱动类型需注册策略脚本，要求 `trusted_script` |
| Shader / 材质 | `asset_pack` | 级别 A | 素材包范围内 |

### 6.2 覆盖策略与视觉层

扩展系统总规提出 4 种覆盖模式（`add` / `override` / `patch` / `theme_override`），视觉层的具体落地场景：

| 覆盖场景 | 覆盖模式 | 例子 |
|----------|---------|------|
| 添加新实体的视觉 | `add` | 扩展包新单位带自己的 VisualProfile |
| 换皮/主题切换 | `theme_override` | 同一逻辑实体换不同画风的 SpriteSet，通过 `AssetRegistry` 切换 |
| 修改已有反馈效果 | `patch` | 把受击闪烁持续时间从 0.15s 改成 0.3s |
| 完全替换视觉表现 | `override` | 用 AnimationTree 替换 SpriteFrames（需 `trusted_script`） |

`theme_override` 是视觉层最核心的覆盖模式：集合包或素材包可以通过 `AssetRegistry` 将同一 `logical_id` 映射到不同画风资源，实现换皮而不改任何逻辑内容。

### 6.3 AssetRegistry 集成

视觉层与 `AssetRegistry` 的交互点：

```
CombatArchetype.visual_profile_id
  ↓
AssetRegistry.resolve_visual_profile(profile_id)
  ↓ 当前启用 asset_pack 的 asset_index.json
  ↓ 返回解析后的 VisualProfileDef
  ↓ 失败时回退 VisualProfileRegistry.get_def(profile_id)
VisualActorComponent.bind_profile(resolved_profile, entity)
```

当前 `AssetRegistry` 最小闭环只支持 `visual_profile`，不做主题覆盖、音频、FX、依赖排序或完整回退链。`AssetIndexCatalog` 负责读取已启用 asset_pack 的 `asset_index.json`，`AssetRegistry` 负责把逻辑表现 ID 解析到 `VisualProfileDef`；`VisualProfileRegistry` 继续作为 `.tres` 扫描注册表和回退路径。

### 6.4 默认占位素材包

主仓必须内置一套默认占位素材，作为 `AssetRegistry` 回退链的终点：

```
data/combat/built_in_placeholder/
├── placeholder_visual_profile.tres      # 占位 VisualProfile
├── placeholder_sprites.tres             # 占位 SpriteSet（色块精灵）
├── placeholder_feedback_map.tres        # 最小反馈映射（受击白色闪烁、死亡淡出）
└── placeholder_shadow_config.tres       # 默认阴影配置
```

- 所有无 `visual_profile` 的实体自动使用占位配置
- NullAnimDriver 在此模式下渲染占位色块，替代现有 `_draw()` 逻辑
- 未来正式素材包只需在 `AssetRegistry` 中覆盖 `logical_id` → 真实资源的映射

## 7. 与现有系统的集成点

### 7.1 EntityFactory 装配流程

```
现有：
  CombatArchetype → EntityFactory.instantiate_entity()
    → 挂 HealthComponent / TriggerComponent / HitboxComponent / MovementComponent

扩展：
  CombatArchetype.visual_profile_id → VisualProfileRegistry.get_def()
    → 挂 HealthComponent / TriggerComponent / HitboxComponent / MovementComponent
    → if profile_def != null:
        → 创建 VisualActorComponent
        → VisualActorComponent.bind_profile(profile_def, entity)
    → else:
        → 保留 BaseEntity._draw() 占位（不破坏已有验证场景）
```

### 7.2 3D-2D 投影集成

抛射体已有高度系统。VisualActorComponent 在 `_process` 中读取高度，偏移 actor scene 或阴影锚点：

```
ProjectileRoot: ground_position + height
  → BaseEntity.position = ground_position
  → VisualActorComponent: ActorRoot.position.y = -height * projection_scale
                         ShadowAnchor.global_position = ground_position
```

### 7.3 验证场景兼容性

- 所有已有验证场景不设置 `visual_profile`，走 `_draw()` 路径
- 新增专门的视觉验证场景（如 `visual_component_validation`）测试视觉功能
- 视觉组件不应影响逻辑验证的 pass/fail 判定

### 7.4 VisualCue 与 Action Recipe 的关系

`VisualCueDef` 适合描述“某类运行时事件发生时，要触发哪些视觉动作”，例如 `entity.damaged` 触发 flash、`projectile.hit` 触发 hit splat。Action Recipe 适合描述“某个 actor 收到语义动作后，这个动作内部如何编排”，例如 Squash 的 `squash`、Jalapeno 的 `explode`。

二者可以串联，但职责不同：

```
EventBus event
  → VisualCueDef actions
  → VisualActionRunner
  → VisualActorComponent.play_action(action_id)
  → ActorRoot ActionRecipe
```

短期落地建议：

- 本地 `.reanim` manifest 先承载 Action Recipe，用于验证 root motion、part action、cue moment。
- `VisualActionRunner` 保持运行时入口，负责从事件上下文解析 target/source/position/layer。
- `spawn_world_fx` 这类需要场景层 host 的步骤，最终应转交 `VisualActionRunner` 或复用其解析逻辑。
- Actor 内部只处理本体和部件；不要让 actor 自己查战斗目标、造成伤害或删除实体。

## 8. 实施路线

实施路线与扩展系统总规的分阶段落地顺序对齐，明确两者之间的依赖关系。

| 阶段 | 内容 | 验证标准 | 扩展系统依赖 |
|------|------|---------|-------------|
| **V0** | 定义 `VisualProfileDef` 并接入 `VisualProfileRegistry` | 可注册 `core.placeholder_*` profile | 无 |
| **V1** | `VisualActorComponent` 骨架 + EntityFactory 装配 | `CombatArchetype.visual_profile_id` 可挂载 actor | 无 |
| **V2** | Actor Scene Contract + AnimationPlayer fallback | 简单 actor 无脚本可播放，复杂 actor 可自定义 `play_*` | 无 |
| **V3** | 原版 `.reanim` 本地离线导入 smoke | ignored 私有素材可生成 `actor.tscn / visual_profile.tres / import_report.json` | 无 |
| **V4** | FeedbackStack + Tween 反馈（受击闪烁、死亡淡出、生成弹入） | 伤害/死亡事件有可观察的视觉反馈 | 无 |
| **V5** | Status 视觉映射（冰冻染色、减速动画放慢） | status_system_validation 场景有视觉反馈 | 无 |
| **V6** | 投影集成（高度偏移、阴影） | 抛射体验证场景视觉正确 | 无 |
| **V7** | Composite actor recipe（body/head/overlay/attachment） | Peashooter / Sunflower / ThreePeater wrapper 可复用协议 | 无 |
| **V7.1** | Semantic Action Recipe（root motion / action cue / next_state） | Squash / Jalapeno / Doom-shroom 不再依赖 per-demo 手写流程即可完成动作编排 | 无 |
| **V7.2** | Part Slot Composite Actor（可替换 body/head/overlay 部件槽） | 各异三线射手样本可用同一 body 挂载不同 head profile，并分别播放动作 | 无 |
| **V7.3** | Emission-to-Visual Binding（战斗 emission 与视觉 part/cue 对齐） | 多 lane / 多弹种发射的视觉 cue 与真实 emission 通过语义 ID 对齐 | 无 |
| **E1** | 素材包系统基础 + AssetRegistry 骨架 | 可注册/解析逻辑 ID，回退链工作 | 扩展总规第二步 |
| **V8** | `SpriteSet.use_logical_ids` / actor scene 逻辑 ID 接入 AssetRegistry | 内容包只写逻辑 ID，素材包可切换主题 | 依赖 E1 |
| **V9** | 默认占位素材包正式化 | 回退链终点可验证，无 visual_profile 的实体有基本呈现 | 依赖 E1 |

**关键依赖：** V0–V7.3 可独立推进，不依赖扩展系统。V8–V9 需要素材包系统（E1）完成后才能进行。在此设计下，视觉层的运行时桥接、actor scene 协议、本地 `.reanim` 导入验证和语义动作编排都不阻塞扩展系统落地。

每阶段独立可验证，不阻塞其他子系统开发。

## 9. 开放问题

1. ~~**SpriteFrames 来源** — 手动在 Godot 编辑器中制作，还是需要从参考项目提取精灵图的工具链？~~ → 当前决策：简单素材可继续用 SpriteFrames；原版 `.reanim` 走本地离线导入到 actor scene。
2. **动画状态粒度** — `idle/attack/damaged/dying` 四态够用，还是需要更细（`walking`、`eating`、`special`）？当前运行时用 `play_state` / `play_action` 分层，避免把所有动作都塞进 state。
3. ~~**AssetRegistry 时机** — 先做直接引用，还是现在就设计逻辑 ID 间接映射？~~ → 已决策：SpriteSet 同时支持两种模式（§3.2），过渡期用直接引用，素材包就绪后切逻辑 ID。
4. ~~**Fallback 策略** — 是否需要一套默认色块图形，让无视觉配置的实体也有基本可辨识的呈现？~~ → 已决策：主仓内置默认占位素材包（§6.4），作为 AssetRegistry 回退链终点。
5. **Shader 效果范围** — 初始阶段是否引入 shader（地下遮罩、水面效果），还是只用 modulate/scale？
6. **死亡序列** — 参考项目有复杂的"死亡语言"（断头、焦黑、灰烬）。Open PVZ 是否需要通用的死亡序列系统，还是由 on_death 触发器 + 视觉反馈组合表达？
7. **AnimDriver 策略注册入口** — 外部规则包注册新 AnimDriver 策略时，是否走现有 `ExtensionPackCatalog` 的 `register` 流程，还是需要独立的注册机制？当前不急于实现，优先使用 Actor Scene Contract。
8. **素材包与视觉验证** — 素材包切换后，视觉验证场景如何保证不退化？是否需要素材包级的 smoke test？
9. ~~**Composite actor recipe 格式** — Peashooter / Sunflower / ThreePeater 的 body/head/overlay/attachment 组合是否先用本地 JSON recipe，还是先保持 wrapper scene 手写，等 5 类植物稳定后再抽象？~~ → 当前决策：先在本地 `.reanim` manifest 中验证 Action Recipe / Part Slot，字段稳定后再抽成正式 Resource。
10. **Action Recipe 字段稳定性** — 第一版只收敛 `play_action`、`root_motion`、`part_action`、`emit_local_cue`、`spawn_world_fx`、`return_home` 等少量 op；不要提前设计完整时间轴语言。
11. **Emission-to-Visual Binding 的 ID 口径** — 战斗 emission 与视觉 part/cue 使用同一个语义 ID，还是在 VisualProfile 中维护映射表？建议先在样本中验证，再决定是否进入 `VisualProfileDef`。
