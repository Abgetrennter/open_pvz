# 视觉表现层设计讨论

> 日期：2026-04-19
> 状态：设计讨论阶段
> 背景：基于 [比较目前项目的开发情况](比较目前项目的开发情况.md) 中识别的 P0 缺口，深入讨论视觉表现层的架构方案。

## 1. 设计前提

从项目 wiki 和代码中提取的关键约束：

1. **逻辑与呈现分离** — 实体是纯逻辑节点，视觉层只"观察"不应"驱动"规则
2. **Resource 驱动** — 所有配置走 `.tres`，而非硬编码
3. **事件驱动** — 视觉变化应由 `EventBus` 事件触发（`entity.damaged` → 受击闪烁）
4. **扩展包友好** — 逻辑 ID → 实际资源的间接映射，支持换皮/主题
5. **当前实体用 `_draw()` 占位** — 视觉层需要平滑替换这些占位代码，向后兼容

### 参考项目（PVZ-Godot-Dream）的视觉架构

参考项目使用继承层级 + 组件混合的模式：

```
Character000Base (Node2D)
├── Body (BodyCharacter) — 中央视觉状态管理，颜色调制/效果叠加
├── Shadow (Sprite2D) — 阴影渲染
├── AnimComponent (AnimComponentBase) — 动画控制器（策略模式）
│   ├── AnimComponentNorm — AnimationTree 驱动
│   ├── AnimComponentPlayer — AnimationPlayer 驱动
│   └── AnimComponentEmpty — 无动画
├── HpComponent — 血条 UI
└── HurtBoxComponent — 碰撞检测
```

**BodyCharacter 核心能力：**
- 基础颜色 + 多层效果叠加（受击白色、冰冻蓝色、催眠紫色、焦黑）
- Shader 遮罩（地下/水面效果）
- Tween 驱动的闪烁/脉冲动画

**参考项目的局限：** 继承层级深（Character000Base → Plant000Base/Zombie000Base → 具体），视觉效果种类硬编码为枚举 `E_ChangeColors`，不易扩展。Open PVZ 的组合优先设计可以做得更好。

## 2. 架构方案：组件桥接模式

### 2.1 核心思路

视觉表现作为**可插拔组件**接入实体，不改变实体的逻辑职责。实体不知道自己"长什么样"，只暴露状态查询接口。

```
BaseEntity (Node2D) — 纯逻辑
  ├── HealthComponent       ← 已有
  ├── TriggerComponent      ← 已有
  ├── HitboxComponent       ← 已有
  ├── MovementComponent     ← 已有
  └── VisualComponent       ← 新增
        ├── SpriteRoot (AnimatedSprite2D / Node2D) — 精灵/动画容器
        ├── ShadowNode (Sprite2D) — 阴影
        ├── AnimDriver (RefCounted) — 动画驱动策略
        └── FeedbackStack (RefCounted) — 反馈效果栈
```

### 2.2 关键决策：VisualComponent 的位置

| 方案 | 描述 | 优势 | 劣势 |
|------|------|------|------|
| **A. 实体子组件** | VisualComponent 作为 BaseEntity 子节点 | 符合已有组件模式，EntityFactory 统一装配 | 视觉生命周期绑死实体 |
| **B. 独立渲染层** | 场景树中单独一层 VisualLayer，通过 entity_id 关联 | 最干净的逻辑/视觉分离 | 需同步位置/状态，复杂度高 |
| **C. 混合桥接** | 实体上挂轻量桥接组件，实际渲染节点独立管理 | 平衡分离与简洁 | 接口设计需谨慎 |

**倾向：方案 A 作为起始。** 理由：

- BaseEntity 是 `Node2D`，子节点变换自动跟随，无需手动同步位置
- EntityFactory 不需要大改，在现有装配流程中加一步即可
- 如果未来需要独立渲染层，VisualComponent 接口可保持不变，只改内部实现
- YAGNI — 在真正遇到性能瓶颈前不引入独立渲染层的复杂度

### 2.3 职责边界

**VisualComponent 负责：**
- 监听 EventBus 事件 → 触发视觉反馈
- 查询 EntityState → 同步动画状态
- 管理 SpriteRoot / ShadowNode 的生命周期
- 维护 FeedbackStack 效果栈

**VisualComponent 不做：**
- 不影响游戏逻辑（不修改 health、不改变位置）
- 不发射影响规则的事件（纯消费端）
- 不直接持有游戏状态引用

## 3. 资源定义设计

### 3.1 VisualProfile（视觉档案）

```gdscript
# scripts/core/defs/visual_profile.gd
class_name VisualProfile
extends Resource

@export var logical_id: StringName           # 逻辑资源 ID（为 AssetRegistry 预留）
@export var sprites: SpriteSet               # 精灵资源集
@export var animations: AnimationMapping     # 动画状态映射
@export var feedback_map: FeedbackMap        # 状态→效果映射表
@export var shadow_config: ShadowConfig      # 阴影配置
@export var default_scale: Vector2 = Vector2.ONE
@export var default_offset: Vector2 = Vector2.ZERO
@export var facing_direction: int = 1        # 1=右, -1=左（僵尸翻转）
```

### 3.2 SpriteSet（精灵资源集）

```gdscript
# scripts/core/defs/sprite_set.gd
class_name SpriteSet
extends Resource

@export var idle_frames: SpriteFrames        # Godot 原生 SpriteFrames
@export var attack_frames: SpriteFrames
@export var damage_frames: SpriteFrames
@export var death_frames: SpriteFrames
@export var move_frames: SpriteFrames        # 僵尸行走
@export var special_frames: Dictionary       # StringName -> SpriteFrames（扩展用）
```

### 3.3 AnimationMapping（动画状态映射）

```gdscript
# scripts/core/defs/animation_mapping.gd
class_name AnimationMapping
extends Resource

# 逻辑状态 → 精灵帧组名称的映射
@export var state_map: Dictionary = {
    &"idle": &"idle_frames",
    &"attack": &"attack_frames",
    &"damaged": &"damage_frames",
    &"dying": &"death_frames",
    &"moving": &"move_frames",
}
# 动画驱动类型：frames / tree / null
@export var driver_type: StringName = &"frames"
```

### 3.4 FeedbackMap（反馈映射表）

```gdscript
# scripts/core/defs/feedback_map.gd
class_name FeedbackMap
extends Resource

@export var on_damaged: FeedbackEntry        # 受击：白色闪烁
@export var on_death: FeedbackEntry          # 死亡：淡出/缩小
@export var on_healed: FeedbackEntry         # 治愈：绿色脉冲
@export var on_spawn: FeedbackEntry          # 生成：弹入动画
@export var status_effects: Dictionary       # StringName -> FeedbackEntry
```

```gdscript
# scripts/core/defs/feedback_entry.gd
class_name FeedbackEntry
extends Resource

@export var effect_type: StringName = &""    # flash / tint / scale / shader / fade
@export var color: Color = Color.WHITE
@export var duration: float = 0.15
@export var intensity: float = 1.0
@export var easing: int = Tween.EASE_OUT     # Tween 缓动类型
```

### 3.5 ShadowConfig（阴影配置）

```gdscript
# scripts/core/defs/shadow_config.gd
class_name ShadowConfig
extends Resource

@export var enabled: bool = true
@export var texture: Texture2D = null
@export var offset: Vector2 = Vector2(0, 8)  # 相对实体位置的偏移
@export var scale: Vector2 = Vector2(1.0, 0.4)
@export var modulate: Color = Color(0, 0, 0, 0.3)
```

### 3.6 资源引用策略

| 阶段 | 策略 | 说明 |
|------|------|------|
| **当前** | 直接引用 | VisualProfile 直接引用 SpriteFrames 等资源 |
| **未来** | 分层引用 | AssetRegistry 作为可选覆盖层，通过 `logical_id` 解析实际资源 |

EntityTemplate 中新增：

```gdscript
@export var visual_profile: VisualProfile = null  # 可选，null 则用占位
```

向后兼容：`visual_profile` 为 null 时保留现有 `_draw()` 占位逻辑，已有验证场景不受影响。

## 4. 运行时组件设计

### 4.1 AnimDriver（动画驱动策略）

```gdscript
# scripts/components/visual/anim_driver.gd
class_name AnimDriver
extends RefCounted

var _anim_node: Node2D  # AnimatedSprite2D 或 AnimationPlayer 所在节点

func setup(node: Node2D) -> void:
    _anim_node = node

func play(state: StringName, blend: float = 0.1) -> void:
    push_error("AnimDriver.play() not implemented")

func stop() -> void:
    push_error("AnimDriver.stop() not implemented")

func set_speed_scale(scale: float) -> void:
    pass

func get_current_state() -> StringName:
    return &""

func is_playing() -> bool:
    return false
```

具体实现：

| 驱动 | 适用场景 | 说明 |
|------|---------|------|
| `FramesAnimDriver` | 大部分植物、简单实体 | AnimatedSprite2D + SpriteFrames，按状态切换帧组 |
| `TreeAnimDriver` | 僵尸（复杂状态机） | AnimationTree + AnimationPlayer，状态过渡 |
| `NullAnimDriver` | 无需动画的实体/占位 | 空实现 |

**初始阶段只实现 FramesAnimDriver 和 NullAnimDriver**（YAGNI）。

### 4.2 FeedbackStack（反馈效果栈）

核心设计原则：**反馈效果是可叠加的，后入的覆盖先入的同类效果。**

```gdscript
# scripts/components/visual/feedback_stack.gd
class_name FeedbackStack
extends RefCounted

# 反馈层
var _layers: Dictionary = {}  # StringName(layer_id) -> FeedbackLayer

func push(layer_id: StringName, entry: FeedbackEntry) -> void:
    _layers[layer_id] = FeedbackLayer.new(layer_id, entry)

func pop(layer_id: StringName) -> void:
    _layers.erase(layer_id)

func resolve() -> FeedbackResult:
    # 计算所有活跃层的合成结果
    var result := FeedbackResult.new()
    for layer in _layers.values():
        result.merge(layer)
    return result
```

```gdscript
class_name FeedbackResult
extends RefCounted

var modulate: Color = Color.WHITE
var scale: Vector2 = Vector2.ONE
var speed_scale: float = 1.0
```

**相比参考项目的改进：** 参考项目用硬编码枚举 `E_ChangeColors` 管理效果种类。Open PVZ 用通用的栈结构 + FeedbackEntry Resource，新增效果类型只需定义新的 `.tres`，无需改代码。这符合项目"组合规则"的核心哲学。

### 4.3 VisualComponent 主流程

```gdscript
# scripts/components/visual_component.gd (伪代码骨架)
class_name VisualComponent
extends Node2D

var _entity: BaseEntity
var _profile: VisualProfile
var _anim: AnimDriver
var _feedback: FeedbackStack
var _sprite_root: Node2D
var _shadow: Sprite2D

func setup(entity: BaseEntity, profile: VisualProfile) -> void:
    _entity = entity
    _profile = profile
    _create_visual_nodes()
    _create_anim_driver()
    _subscribe_events()

func _create_visual_nodes() -> void:
    # 创建 SpriteRoot + ShadowNode
    pass

func _subscribe_events() -> void:
    EventBus.subscribe(&"entity.damaged", _on_entity_damaged)
    EventBus.subscribe(&"entity.died", _on_entity_died)
    EventBus.subscribe(&"entity.status_applied", _on_status_applied)
    EventBus.subscribe(&"entity.status_removed", _on_status_removed)

func _on_entity_damaged(data: EventData) -> void:
    if data.core.get("entity_id") != _entity.entity_id:
        return
    var entry := _profile.feedback_map.on_damaged
    if entry:
        _feedback.push(&"damaged_flash", entry)
        _apply_feedback_tween(entry)

func _process(delta: float) -> void:
    # 同步高度偏移（3D-2D 投影）
    var h := _entity.get_height()
    _sprite_root.position.y = -h * _projection_scale
    _shadow.position.y = h * _shadow_config.scale.y
    _shadow.visible = h > 0.0

    # 同步动画状态（从 EntityState 推断）
    _sync_anim_state()

    # 应用反馈合成
    var result := _feedback.resolve()
    _sprite_root.modulate = result.modulate
    _sprite_root.scale = result.scale
    _anim.set_speed_scale(result.speed_scale)
```

## 5. 数据流设计

### 5.1 事件 → 视觉反馈

```
EventBus: entity.damaged
  ↓ (VisualComponent 订阅)
VisualComponent._on_entity_damaged(event_data)
  ↓
FeedbackMap.lookup("on_damaged") → FeedbackEntry(flash, white, 0.15s)
  ↓
FeedbackStack.push("damaged_flash", entry)
  ↓
Tween: SpriteRoot.modulate → 白色 → 恢复（0.15s）
  ↓
FeedbackStack.pop("damaged_flash")
```

### 5.2 状态 → 动画切换

```
EntityState.health <= 0
  ↓ (VisualComponent 在 _process 中检测，或由 entity.died 事件触发)
VisualComponent._on_entity_died()
  ↓
AnimationMapping.state_map["dying"] → "death_frames"
  ↓
AnimDriver.play(&"dying")
  ↓
同时 FeedbackStack.push("death_fade", fadeEntry)
```

### 5.3 Status 效果 → 视觉叠加

```
EventBus: entity.status_applied { status_id: "frozen", duration: 3.0 }
  ↓
VisualComponent._on_status_applied()
  ↓
FeedbackMap.status_effects["frozen"] → FeedbackEntry(tint, cyan, 3.0)
  ↓
FeedbackStack.push("status:frozen", entry)
  ↓ (持续 3 秒)
FeedbackStack.pop("status:frozen") → 恢复原色
```

## 6. 与现有系统的集成点

### 6.1 EntityFactory 装配流程

```
现有：
  EntityTemplate → EntityFactory._assemble_entity()
    → 挂 HealthComponent / TriggerComponent / HitboxComponent / MovementComponent

扩展：
  EntityTemplate → EntityFactory._assemble_entity()
    → 挂 HealthComponent / TriggerComponent / HitboxComponent / MovementComponent
    → if template.visual_profile != null:
        → 创建 VisualComponent
        → VisualComponent.setup(entity, template.visual_profile)
    → else:
        → 保留 BaseEntity._draw() 占位（不破坏已有验证场景）
```

### 6.2 3D-2D 投影集成

抛射体已有高度系统。VisualComponent 在 `_process` 中读取高度，偏移精灵位置：

```
ProjectileRoot: ground_position + height
  → BaseEntity.position = ground_position
  → VisualComponent: sprite_root.position.y = -height * projection_scale
                      shadow_node.position.y = height * shadow_scale
```

### 6.3 验证场景兼容性

- 所有已有验证场景不设置 `visual_profile`，走 `_draw()` 路径
- 新增专门的视觉验证场景（如 `visual_component_validation`）测试视觉功能
- 视觉组件不应影响逻辑验证的 pass/fail 判定

## 7. 实施路线

| 阶段 | 内容 | 验证标准 |
|------|------|---------|
| **0** | 定义 Resource 类（VisualProfile、SpriteSet、AnimationMapping、FeedbackMap、FeedbackEntry、ShadowConfig） | 可在编辑器中创建 .tres，ProtocolValidator 校验通过 |
| **1** | VisualComponent 骨架 + NullAnimDriver + 占位色块精灵 | 替换现有 `_draw()`，验证场景视觉不退化 |
| **2** | FramesAnimDriver + AnimatedSprite2D 集成 | 给一个植物（如豌豆射手）添加真实帧动画 |
| **3** | FeedbackStack + Tween 反馈（受击闪烁、死亡淡出、生成弹入） | 伤害/死亡事件有可观察的视觉反馈 |
| **4** | Status 视觉映射（冰冻染色、减速动画放慢） | status_system_validation 场景有视觉反馈 |
| **5** | 投影集成（高度偏移、阴影） | 抛射体验证场景视觉正确 |
| **6** | EntityTemplate.visual_profile + EntityFactory 集成 | 完全模板化驱动，添加新植物无需改代码 |

每阶段独立可验证，不阻塞其他子系统开发。

## 8. 开放问题

1. **SpriteFrames 来源** — 手动在 Godot 编辑器中制作，还是需要从参考项目提取精灵图的工具链？
2. **动画状态粒度** — `idle/attack/damaged/dying` 四态够用，还是需要更细（`walking`、`eating`、`special`）？
3. **AssetRegistry 时机** — 先做直接引用，还是现在就设计逻辑 ID 间接映射？
4. **Fallback 策略** — 是否需要一套默认色块图形，让无视觉配置的实体也有基本可辨识的呈现？
5. **Shader 效果范围** — 初始阶段是否引入 shader（地下遮罩、水面效果），还是只用 modulate/scale？
6. **死亡序列** — 参考项目有复杂的"死亡语言"（断头、焦黑、灰烬）。Open PVZ 是否需要通用的死亡序列系统，还是由 on_death 触发器 + 视觉反馈组合表达？
