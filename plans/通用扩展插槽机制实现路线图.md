# 通用扩展插槽机制实现路线图

- 状态：已按 v1 路线落地；正式机制文档见 [通用扩展插槽机制](../wiki/04-roadmap-reference/42-通用扩展插槽机制.md)

## Summary

目标是把当前“局部存在”的扩展插槽机制规范化为可落地的工程体系：主仓提供统一 `Slot / Contributor / Registry / Runtime 分发 / Guardrail` 模型，扩展包通过资源化贡献项注册新能力，运行时只依赖 registry，不为具体扩展能力写硬编码分支。

默认实施顺序：

```text
阶段 0：基线收口
阶段 1：ProjectileMovement 插槽落地
阶段 2：MechanicCompiler 扩展接入
阶段 3：Registry 行为统一
阶段 4：manifest 能力与信任声明
阶段 5：文档提升与治理收口
```

### 阶段 1：ProjectileMovement 通用插槽落地

新增核心接口：

```text
ProjectileMovementDef
ProjectileMovementRegistry
```

建议行为：

- `ProjectileMovementDef` 作为 `Resource`，字段包含 `move_mode`、`movement_script`、`param_defs`、`default_hit_strategy`、`default_terminal_hit_strategy`、`tags`。
- `ProjectileMovementRegistry` 作为 autoload，提供 `register_def()`、`has()`、`get_def()`、`create_component()`、`list_ids()`、`rebuild_registry()`。
- 内置 `linear / parabola / track` 改为通过 registry 注册，同时保留兼容别名：
  - `linear -> core.linear`
  - `parabola -> core.parabola`
  - `track -> core.track`
- `ProjectileRoot` 不再用 `match move_mode` 选择脚本，改为通过 `ProjectileMovementRegistry.create_component(move_mode)` 创建 movement component。
- 找不到 `move_mode` 时记录 `protocol.issue`，安全回退到 `core.linear`。
- `ProtocolValidator` 短期移除 `movement_mode` 固定 options 限制，改为接受 `StringName`；中期再接入 registry 存在性校验。
- 新增扩展包示例 `my_pack.zigzag`，只作为验证插槽能力，不把 `zigzag` 写进主仓硬编码分支。

### 阶段 2：MechanicCompiler 扩展接入

新增或规范化：

```text
MechanicCompilerDef
```

建议行为：

- 扩展包可以注册现有 family 下的新 `type_id`，例如 `Trajectory.my_pack.zigzag` 实际表达为：
  - `family = Trajectory`
  - `type_id = my_pack.zigzag`
- `MechanicCompilerDef` 字段包含 `type_id`、`family`、`compiler_script`、`param_defs`。
- compiler script 必须暴露固定接口，例如 `compile(mechanic, archetype, merged_params) -> Dictionary`。
- `MechanicCompilerRegistry` 支持从扩展包扫描 compiler def，并拒绝：
  - 空 `type_id`
  - 非冻结 family
  - 扩展包注册 `core.*`
  - 重复 `type_id`
  - compiler 返回非 `Dictionary`
- `my_pack.zigzag` compiler 输出 `movement_mode = my_pack.zigzag` 及其轨迹参数，交给阶段 1 的 movement registry 消费。

### 阶段 3：Registry 行为统一

统一现有 registry 的基础行为，不急于重写所有运行时：

```gdscript
register_def(def, source) -> bool
has(id) -> bool
list_ids() -> PackedStringArray
rebuild_registry() -> void
```

重点收口：

- `EffectRegistry` 保持现有成熟路径，但补齐来源信息、命名空间检查和重复 id 策略。
- `ControllerRegistry / DetectionRegistry / TriggerRegistry` 保留内置策略注册方式，先统一错误报告和 `list_ids()`。
- 所有扩展注册失败必须进入 `DebugService.record_protocol_issue()`。
- registry rebuild 必须可重复执行，不能因为重复扫描导致重复注册或状态污染。
- v1 默认不做跨包 override；冲突即拒绝。

### 阶段 4：manifest 能力与信任声明

扩展 `extension.json` 过渡格式：

```json
{
  "pack_id": "my_pack",
  "enabled_by_default": false,
  "register": ["resources", "effects", "projectile_movement", "mechanic_compilers"],
  "trust_level": "trusted_runtime",
  "capabilities": ["projectile_movement", "mechanic_compiler"]
}
```

实现策略：

- `ExtensionPackCatalog.ALLOWED_REGISTER_KINDS` 增加新 register kind。
- 低信任包不能注册运行时代码：
  - `data_only`：只能资源内容
  - `rule_extended`：可注册受控 effect strategy
  - `trusted_runtime`：可注册 projectile movement / compiler / controller 级运行时代码
- manifest 校验失败仍走现有 `extension_manifest` protocol issue。
- 兼容现有扩展包：没有 `trust_level` 的旧包按当前行为保守映射，现有 `minimal_chaos_pack / phase5_chaos_pack / phase5_guardrail_pack` 不应被破坏。

## Test Plan

- 阶段 1 必测：
  - 内置 `linear / parabola / track` 全部原验证通过。
  - 新增 `projectile_movement_zigzag_smoke_validation`：扩展 movement 能注册、生成 projectile、发出 `projectile.spawned` 和 `projectile.hit`。
  - 新增 guardrail：未知 `move_mode` 记录 protocol issue 并回退。
  - 新增 guardrail：movement script 未继承 `ProjectileMovementBase` 被拒绝。

- 阶段 2 必测：
  - 扩展 compiler 注册 `my_pack.zigzag` 成功。
  - `Trajectory` mechanic 编译后能注入 `movement_mode = my_pack.zigzag`。
  - 非冻结 family、`core.*` 覆盖、重复 `type_id`、非法 compiler 返回值均被拒绝。

- 阶段 3 必测：
  - registry rebuild 幂等。
  - 重复扫描扩展包不会重复注册。
  - registry `list_ids()` 输出稳定。
  - 现有 effect 扩展验证不回退。

- 阶段 4 必测：
  - manifest 新 register kind 合法。
  - 非法 register kind 仍被 guardrail 捕获。
  - `data_only` 包注册 runtime script 被拒绝。
  - 旧扩展包在未声明新字段时仍可运行现有验证。

- 回归命令：
  - `pwsh tools/run_all_validations.ps1`
  - 阶段内可先跑 `extension` 与 `guardrail` 分层，再跑全量。

## Assumptions

- v1 优先落地 `ProjectileMovement`，因为它是当前最明显缺口。
- `zigzag` 只作为扩展包示例，命名使用 `my_pack.zigzag`，不进入 `core.*`。
- 不允许扩展包覆盖主仓 `core.*`。
- 不新增 Mechanic family，只允许扩展现有 family 下的 type。
- 先采用“资源化贡献项 + registry 扫描”的方式，不引入任意启动脚本。
- 所有代码注释继续跟随现有 GDScript 英文注释风格；文档保持中文。
