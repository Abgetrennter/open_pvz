## TL;DR

上方植物卡牌所在位置应拆成两类概念：

1. **行动栏容器**：例如 `BattleActionBar` / `SeedBankBar`，负责把阳光、卡牌槽、铲子、未来工具按钮组合在同一条 HUD 区域。
2. **卡牌呈现面板**：例如 `CardBar`，只负责渲染固定手牌和传送带卡牌，不承载所有临时卡玩法。

卡槽长度可以随卡牌数量变化，但必须由外层行动栏提供最大宽度和 overflow 策略。传送带可以作为上方 Bar 的一种供给模式；种子雨、战场掉落临时卡不属于上方卡槽，应进入独立的 `CardDropLayer` / `TemporaryCardLayer`。

底层建议逐步从 `card_id` 列表升级到 **CardPacketState（卡牌实例）**，用于支持重复卡、传送带、临时掉落和一次性卡。`BattleCardState` 仍然是规则真相；UI 只做状态投影和输入出口。

## 背景与目标

当前 `CardBar` 已经可以从场景读取 `card_defs`，订阅 `resource.changed`、`card.cooldown_started`、`card.hand_updated` 等事件，并提供点击选卡信号。它适合作为卡牌列表中介者，但不适合继续扩张成“整个上方 UI Bar”。

本草案目标是回答三个问题：

1. 上方卡牌位置的 Bar 代码边界如何拆分。
2. 自定义卡牌数量时，Bar 是否以及如何随数量变长。
3. 传送带、种子雨、临时卡这类非普通手牌是否应共用同一套底层协议。

## 非目标

- 不讨论视觉表现、贴图、动画风格或最终布局美术。
- 不把种子雨和战场掉落临时卡塞进上方固定卡槽。
- 不修改 Mechanic family，不引入新的玩法 family。
- 不在 UI 层实现卡牌生成规则、费用扣除、冷却结算或放置校验。
- 不直接复制 `vendor/` 参考项目代码。

## 当前 OpenPVZ 状态

### 已具备

- `BattleCardState` 是卡牌规则真相，负责手牌、费用、冷却、放置请求和程序化 `card_play_requests`。
- `CardDef` 已提供 `card_id`、`display_name`、`archetype_id`、`sun_cost`、`cooldown_seconds`、`placement_tags`。
- `CardBar` 已经从 `BattleCardState` 或 scenario 读取手牌，订阅资源和冷却事件，能动态重建 UI。
- `BattleHUD` 已经作为 HUD 容器存在，负责给子面板调用 `panel_setup()` / `panel_teardown()`。
- `InputRouter` 已经接入 `CardBar.card_selected` 和 board click，能完成卡牌选择到放置的链路。

### 主要缺口

- `CardBar` 既处理内部 UI 状态，又承担点击入口；后续如果继续塞阳光、铲子、传送带、临时卡，会违反单一职责。
- `InputRouter` 直接引用 `_card_bar`，并反向调用 `select_card()` / `deselect_card()`，不利于替换不同卡牌 presenter。
- 当前 `BattleCardState` 用 `card_id` 做字典、冷却和选择状态。若同一张卡在传送带里出现多次，会出现实例身份不足的问题。
- 目前没有行动栏级别的布局容器，demo 中仍然通过 offset 手工摆放 `SunCounter` 和 `CardBar`。

## 推荐模型

### 1. BattleActionBar：上方行动栏组合容器

新增概念：`BattleActionBar` 或 `SeedBankBar`。

职责：

- 组合 `SunCounter`、`CardBar`、未来 `ShovelButton`、模式按钮。
- 管理上方 Bar 的布局约束、最大宽度、overflow 策略。
- 接收 `BattleHUD.setup()` 传入的 battle/scenario/theme，再转交给内部面板。
- 不订阅玩法事件，不调用 `BattleCardState.play_card()`。

不负责：

- 不判断费用、冷却、放置是否合法。
- 不生成传送带卡。
- 不渲染种子雨或战场掉落临时卡。

### 2. CardBar：固定手牌与传送带 presenter

`CardBar` 保持为卡牌列表面板，但只渲染特定 presentation：

- `bank_slot`：普通固定手牌。
- `conveyor_slot`：传送带卡牌。

不渲染：

- `world_pickup`：种子雨、战场掉落卡。
- `floating_overlay`：特殊小游戏浮动卡。
- `shop_offer`：商店或构筑候选卡。

`CardBar` 的长度由可见 packet 数决定，外层行动栏决定是否允许继续增长。

### 3. CardSlotView：单卡槽叶子控件

建议后续把单张卡槽拆成独立叶子控件：

```gdscript
class_name CardSlotView
extends Control

func apply_state(state: Dictionary) -> void:
	# 只更新显示，不订阅 EventBus，不调用玩法接口。
	pass
```

`CardBar` 作为中介者订阅事件并批量更新 `CardSlotView`。叶子控件不接触 `EventBus`，也不接触 `BattleCardState`。

### 4. CardPacketState：卡牌实例协议

为传送带和临时卡预留实例身份：

```text
packet_id: StringName
card_id: StringName
source_kind: normal_hand / conveyor / seed_rain / temp_drop
presentation_kind: bank_slot / conveyor_slot / world_pickup / floating_overlay
state: ready / selected / consumed / cooling / expired
order_index: int
spawned_at: float
expires_at: float
```

说明：

- `card_id` 指向 `CardDef`。
- `packet_id` 标识“一张实际可用的卡牌实例”。
- 普通手牌可以先保持 `card_id` 兼容；传送带和临时卡必须使用 `packet_id`。
- `presentation_kind` 只决定显示位置，不决定玩法规则。
- `source_kind` 表示来源规则，供 validation、debug 和模式逻辑观测。

### 5. CardDropLayer / TemporaryCardLayer：非上方卡槽 presenter

种子雨和大多数临时卡不属于上方 Bar，应由独立层处理：

```text
CardDropLayer
  -> 渲染 presentation_kind == world_pickup 的 CardPacketState
  -> 点击后通知 InputRouter
  -> 选择/放置仍走 BattleCardState
```

这能避免 `CardBar` 同时处理固定卡槽、传送带、战场掉落、浮动限时卡，保持边界清晰。

## 动态长度策略

`CardBar` 应支持内容驱动宽度：

```text
content_width = packet_count * slot_width + max(packet_count - 1, 0) * slot_spacing
```

但不能无限挤占 HUD。外层 `BattleActionBar` 应提供布局配置：

```gdscript
class_name CardBarLayoutProfile
extends Resource

@export var slot_size := Vector2(72.0, 90.0)
@export var slot_spacing := 6.0
@export var max_inline_slots := 10
@export_enum("expand", "compact", "scroll", "page") var overflow_mode := "compact"
```

建议阶段策略：

| 数量 | 行为 |
|------|------|
| 1 到 `max_inline_slots` | 直接随数量增长 |
| 超出 `max_inline_slots` | 第一阶段可以拒绝或压缩；后续再实现滚动或分页 |
| 传送带 | 由 `max_packets` 控制可见上限，满了后规则层暂停生成或排队 |
| 种子雨 | 不进入 `CardBar`，不影响上方 Bar 长度 |

## 传送带设计边界

传送带能实现，但不应作为 `CardBar` 内部自发动画。

推荐链路：

```text
BattleModeHost / ConveyorRuleModule
  -> 按 GameState.current_time 和确定性随机生成 packet
  -> BattleCardState.enqueue_packet(packet)
  -> EventBus: card.packet_added / card.hand_updated
  -> CardBar 渲染 conveyor_slot
  -> 玩家选择 packet
  -> BattleCardState.play_packet(packet_id, lane_id, slot_index)
  -> 成功后 consume/remove packet
```

注意：

- 玩法生成时间走固定 tick / `GameState.current_time`，不使用 Godot `Timer`。
- UI tween 可以做传送带移动表现，但不能决定 packet 何时可用。
- 同一 `card_id` 可重复出现，因此必须有 `packet_id`。

## 种子雨与临时卡边界

种子雨不是上方卡槽变体。

推荐链路：

```text
SeedRainRuleModule
  -> 生成 presentation_kind == world_pickup 的 packet
  -> CardDropLayer 创建屏幕/战场空间节点
  -> 玩家点击掉落卡
  -> InputRouter.request_card_packet_select(packet_id)
  -> 玩家点击格子
  -> BattleCardState.play_packet(packet_id, lane_id, slot_index)
  -> packet consumed 或 expired
```

临时卡按出现位置区分：

| 类型 | presentation_kind | Presenter |
|------|-------------------|-----------|
| 传送带临时卡 | `conveyor_slot` | `CardBar` |
| 战场掉落卡 | `world_pickup` | `CardDropLayer` |
| 屏幕浮动奖励卡 | `floating_overlay` | `TemporaryCardLayer` |
| 构筑候选卡 | `shop_offer` / `chooser_slot` | 未来选卡 UI，不属于战斗上方 Bar |

## 与 InputRouter 的关系

当前 `InputRouter.setup(card_bar, board_visual, card_state, flow_state, input_profile)` 直接依赖 `CardBar`。后续建议调整为语义接口：

```text
CardBar.card_packet_selected(packet_id)
  -> InputRouter.request_card_packet_select(packet_id)

InputRouter.card_selected(packet_id)
  -> CardBar / CardDropLayer 通过信号更新高亮
```

兼容阶段可以保留：

```text
request_card_select(card_id)
```

但新增能力应优先走：

```text
request_card_packet_select(packet_id)
```

这样 `InputRouter` 不需要知道选择来自固定卡槽、传送带还是种子雨。

## 方案对比

| 方案 | 优点 | 问题 | 结论 |
|------|------|------|------|
| 扩大 `CardBar`，把阳光、铲子、传送带、临时卡全塞进去 | 短期改动少 | 职责膨胀；种子雨位置错误；InputRouter 更耦合具体 UI | 不推荐 |
| 新增 `BattleActionBar`，`CardBar` 只管卡槽类 presenter，临时卡独立 presenter | 边界清晰；符合现有 `BattleHUD` 和 `UIPanelBase`；可分阶段 | 需要新增组合容器和接线 | 推荐 |
| 新建完整 UIModel 层同步所有卡牌状态 | 表达力强 | 与 `BattleCardState` 双写，当前阶段过度设计 | 暂缓 |

## 分阶段路线

### Phase 1：行动栏组合容器

- 新增 `BattleActionBar`。
- 把 demo 中手工摆放的 `SunCounter` + `CardBar` 收进 `BattleActionBar`。
- `CardBar` 支持根据卡牌数量计算自身最小宽度。
- 不改变 `BattleCardState` 和 `InputRouter` 规则语义。

验收：

- 1、5、10 张卡都能正确生成卡槽。
- Bar 长度随卡牌数量变化。
- 现有 `card_place_validation` 不受影响。

### Phase 2：CardSlotView 与 InputRouter 解耦

- 抽出 `CardSlotView`。
- `CardBar` 作为中介者批量更新叶子控件。
- `InputRouter` 从直接持有 `CardBar` 过渡为接收语义请求和发出语义反馈信号。

验收：

- 键盘选卡、鼠标选卡、取消选中都与现状等价。
- 替换 `CardBar` 具体实现不需要改 `InputRouter` 核心逻辑。

### Phase 3：CardPacketState 兼容层

- 在 `BattleCardState` 内引入 packet 概念，但保留 `play_card(card_id, lane_id, slot_index)` 兼容入口。
- 普通手牌可自动映射为稳定 packet。
- 新增 `play_packet(packet_id, lane_id, slot_index)`。

验收：

- 普通手牌路径行为不变。
- 两个 packet 可以指向同一 `card_id`，不会互相覆盖选择/消耗状态。

### Phase 4：传送带供给模式

- 在 mode/rule module 中生成 `source_kind == conveyor`、`presentation_kind == conveyor_slot` 的 packet。
- `CardBar` 渲染传送带 packet。
- 成功出牌后移除 packet。
- 生成规则走固定 tick 和确定性随机。

验收：

- 传送带满载时不会无限增长。
- 同名卡重复出现时可独立选择和消耗。
- 不使用 Godot `Timer` 决定玩法生成。

### Phase 5：种子雨与临时卡层

- 新增 `CardDropLayer` / `TemporaryCardLayer`。
- 种子雨 packet 使用 `presentation_kind == world_pickup`。
- 点击掉落卡后进入统一选择/放置链。
- `CardBar` 不显示这些 packet。

验收：

- 种子雨卡不会增加上方 Bar 长度。
- 掉落卡可被选中、放置、过期或消耗。
- InputRouter 不区分来源，只处理 packet 选择和格子点击。

## 验证建议

| 场景 | 覆盖点 |
|------|--------|
| `ui_card_bar_dynamic_length_validation` | 不同手牌数量下 slot 数和最小宽度正确 |
| `card_packet_duplicate_validation` | 两个 packet 指向同一 `card_id` 时可独立选择/消耗 |
| `mode_conveyor_packet_validation` | 传送带生成、满载、消耗、重复卡 |
| `temporary_card_layer_exclusion_validation` | `world_pickup` packet 不进入 `CardBar` |
| `input_router_packet_selection_validation` | 固定卡槽、传送带、掉落卡都能走同一语义选择链 |

## 开放问题

1. 普通固定手牌是否需要立刻暴露 `packet_id`，还是先由 `card_id` 兼容入口自动映射。
2. 普通手牌冷却应继续按 `card_id` 共享，还是未来允许同一 `card_id` 多 slot 独立冷却。
3. `BattleActionBar` 的默认 `max_inline_slots` 是否固定为 10，还是由 scenario/mode 配置。
4. 传送带满载后应暂停生成、丢弃新 packet，还是维护一个不可见队列。
5. 种子雨点击后是“立即选择等待放置”，还是“先收入临时队列再由玩家选择”。

## 当前推荐的最小闭环

第一步只做 `BattleActionBar + CardBar 动态长度`，不碰 packet 协议。这样能先解决上方 Bar 的代码边界和自定义卡牌数目问题。

第二步再解耦 `InputRouter` 与具体 `CardBar`。

第三步才引入 `CardPacketState`，为传送带和临时卡打开统一入口。
