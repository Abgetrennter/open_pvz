# 实体活跃性与状态控制系统设计草案

> 状态：已收敛为正式计划。后续以 `plans/实体活跃性与状态控制计划.md` 为准。

## 问题

一个实体从诞生到死亡之间，不是只有"工作"和"消亡"两种状态。蘑菇需要睡觉但不消失，Potato Mine 需要埋入地下不可被攻击，Chomper 消化期间不能攻击但可以被吃掉，Ice-shroom 需要冻结全场的移动但保留触发器响应。

当前系统用五个独立发明的机制分别处理这些需求：

| 机制 | 位置 | 控制什么 |
|------|------|----------|
| `is_combat_active()` | BaseEntity | 外部实体能否选它为目标 |
| `is_attack_blocked()` | BaseEntity + status | 攻击行为是否暂停 |
| `get_effective_movement_scale()` | BaseEntity + status | 移动速度倍率 |
| `required_state` | TriggerRegistry | 触发器只在某状态下执行 |
| `activate()` / `deactivate()` | FieldObjectRoot | 场上物件的开关 |

五个机制互相不知道对方的存在，缺乏统一的语义模型。每个新行为都需要自己发明一套"暂停/恢复"方案。

同时，StateComponent 本身也有三个缺口：

| 缺口 | 说明 | 影响 |
|------|------|------|
| 状态事件链不完整 | 只有 `entity.state_entered`，没有 exit 和 transition-failed 事件 | 下游消费方（视觉、音频）无法完整响应状态生命周期 |
| 状态转换无约束 | 任何状态可以跳转到任何状态，没有声明式转换表 | 复杂实体的状态图无法保证正确性 |
| 扁平状态模型 | 无法表达 Chomper（idle→biting→chewing→digesting）等有优先级和互斥关系的多状态 | 复杂行为正确性 |

这三个缺口和活跃性问题高度关联：**活跃性的"状态→liveness profile"映射 + 声明式转换约束，足以替代分层状态机（HSM）的需求**。本文将活跃性系统和状态机增强统一设计。

## 核心概念

### 活跃性维度

一个实体的"活跃程度"不是一个标量，而是多个独立维度的组合。这些维度分为两组。

**内省维度**——控制实体自身子系统是否工作：

| 维度 | 控制的子系统 | 典型用途 |
|------|-------------|----------|
| triggers | TriggerComponent 是否响应 EventBus 事件 | 蘑菇睡觉时关闭 |
| state | StateComponent 是否推进时间驱动的状态转换 | 蘑菇睡觉时保持开启（唤醒是事件驱动） |
| movement | 实体是否执行位移 | Ice-shroom 冻结时关闭 |
| controllers | ControllerComponent 是否执行连续行为策略 | Chomper 消化时关闭 bite |

**外视维度**——控制外部世界如何对待这个实体：

| 维度 | 含义 | 典型用途 |
|------|------|----------|
| targetable | DetectionRegistry 的索敌逻辑能否发现它 | Potato Mine 地下阶段不可选中 |
| damageable | take_damage 是否产生实际效果 | Potato Mine 地下阶段不可受伤 |
| collidable | 抛射体碰撞检测是否命中它 | 与 targetable 的区别：targetable 是主动索敌过滤，collidable 是被动碰到过滤 |

两组维度正交。一个实体可以"自己的触发器全关"同时"外部仍然可以攻击它"——这正是 Chomper 消化中需要的语义。

### 维度值

每个维度只有两个值：**ENABLED** 和 **DISABLED**。

冲突解决不靠第三个枚举值，而是靠 **数字优先级**。具体规则：对每个维度，取所有 override 中优先级最高的那个值。相同优先级时 DISABLED 赢（保守策略）。

为什么不需要第三个值（如 FORCED_ON）：如果两个效果在"强制开关"上打架，说明它们不应该同时作用于同一个实体，应该在更高层的游戏规则上避免这种冲突。用数字优先级表达"谁更强"，比用类型枚举更简单、更统一，也不会陷入"需要一个比强制更强制的东西"的无限升级。

### Override 栈

一个维度在任何时刻的实际值，不是某个单一来源决定的，而是多层 override 的合成结果。

**来源分层**：

| 层 | 来源 | 优先级 | 说明 |
|----|------|--------|------|
| 0 | 基础 profile | 0 | 实体类型默认值。Plant/Zombie 全部 ENABLED，FieldObject 外视维度 DISABLED |
| 1 | 状态驱动 | 10 | StateComponent 当前状态对应的 profile |
| 2 | Status 效果驱动 | 20 | 活跃 status effect 产生的 profile |
| 3 | 外部直接驱动 | 30-50 | 效果链或能力直接 push 的 profile，优先级由调用者指定 |

**合成规则**：逐层合并。对每个维度取所有 override 中优先级最高的那个值，同优先级 DISABLED 赢。

**关键特性**：来源之间不需要知道彼此的存在。蘑菇的 sleeping 状态关掉了 triggers，Coffee Bean 不需要知道是谁关的，它只需要 push 一个更高优先级的 override 就能覆盖。当 sleeping 状态结束时 pop 掉自己的 override，Coffee Bean 的 override 仍在栈上，最终效果仍然是"开启"。Coffee Bean 效果也结束后 pop 掉，一切回到默认值。

### 查询 API

各子系统不直接读取 override 栈，而是通过 BaseEntity 上的统一查询方法：

| 查询方法 | 消费者 |
|----------|--------|
| `is_liveness_enabled(axis)` | 各子系统在执行前查询自己的维度 |
| `is_targetable()` | DetectionRegistry 索敌过滤、ProjectileRoot 目标搜索 |
| `is_damageable()` | HealthComponent 的 take_damage 入口 |
| `is_collidable()` | ProjectileRoot 碰撞检测 |

### 状态转换约束

在 StateSpec 层面新增 `allowed_transitions` 字段，声明式定义哪些状态可以转换到哪些状态：

```gdscript
# StateSpec 扩展
var allowed_transitions: Dictionary = {}
# key: target_state_id, value: 转换条件描述
# 不在白名单中的转换会被拒绝
```

转换约束和活跃性正交：
- 活跃性回答"转换完成后各子系统是否工作"
- 转换约束回答"这个转换能不能发生"

两者组合后，Chomper 的完整行为可以表达为：

```
States: idle → biting → chewing → digesting → idle

Transition constraints:
  idle → {biting: on_enemy_in_range}
  biting → {chewing: after_bite_timer}
  chewing → {digesting: after_chew_timer}
  digesting → {idle: after_digest_timer}
  （任何不在白名单里的转换，如 digesting→biting，被自动拒绝）

State → Liveness mapping:
  idle:      all ENABLED
  biting:    triggers: DISABLED
  chewing:   triggers: DISABLED, controllers: DISABLED
  digesting: triggers: DISABLED, controllers: DISABLED, targetable: ENABLED
```

### 为什么不需要分层状态机（HSM）

HSM 解决的核心问题是"子状态继承父状态行为"和"状态优先级互斥"。活跃性 + 转换约束可以用更低的复杂度覆盖相同的场景：

| HSM 能力 | 活跃性 + 约束如何替代 |
|----------|----------------------|
| 子状态继承父状态的 liveness | 扁平状态各自声明 liveness profile，不需要继承链 |
| 优先级互斥（消化中不能攻击） | 转换约束白名单 + triggers/controllers DISABLED |
| 状态层级（idle 中有 sub-state） | 大多数实体不需要 sub-state；如果确实需要（如 Gatling Pea 连射弹药计数），后续可在 StateSpec 中增加 `parent_state` 字段做简单层级，不需要完整 HSM 框架 |

**不引入 HSM 的理由**：当前所有已知的复杂行为（Chomper、Potato Mine、蘑菇、Ice-shroom）都可以用"扁平状态 + 转换约束 + 活跃性 profile"完整表达。HSM 的复杂度成本（状态栈、enter/exit 传播、历史状态）在当前阶段没有回报。

### 状态事件链

扩展 StateComponent 的事件发射，覆盖状态生命周期的完整环节：

| 事件 | 发射时机 | 消费者 |
|------|----------|--------|
| `entity.state_entered` | 已有。进入新状态后 | VisualActorComponent、触发器、活跃性系统 |
| `entity.state_exited` | 新增。离开旧状态后 | VisualActorComponent、音频层、调试 |
| `entity.state_transition_rejected` | 新增。转换被约束拒绝时 | DebugService、开发者调试 |

**重要**：活跃性 override 的 pop/push 在 `_try_execute_transition` 内部同步完成，不依赖事件链的异步性。事件只做通知，不做驱动。这样避免 pop→push 之间出现"无 override"窗口。

## 与现有系统的集成

### 与 State 状态机的集成

当前状态转换声明（`_build_state_spec_inline` 输出的字典）包含 from_state、to_state、触发条件等字段。扩展方式：

1. 增加 `state_liveness` 字段，声明每个状态对应的活跃性 profile
2. 增加 `allowed_transitions` 字段，声明式转换白名单

StateComponent 的转换执行流程变为：

```
_try_execute_transition:
  1. 检查 allowed_transitions 白名单 → 不通过则发射 state_transition_rejected，返回
  2. 发射 state_exited（旧状态）
  3. 同步 pop 旧状态的 liveness override
  4. 更新 current_state
  5. 同步 push 新状态的 liveness override
  6. 发射 state_entered（新状态）
```

Archetype 作者不需要手动写 liveness 或转换约束配置。只要用了 `core.sleeping` type_id，sleeping 状态自动携带 dormant profile（triggers: DISABLED, controllers: DISABLED）和到 "awake" 的单一转换约束。只要用了 `core.arming`，arming 状态自动携带 underground profile（targetable/damageable/collidable: DISABLED）和到 "active" 的单一转换约束。这是编译器隐式提供的约定。

编译器硬编码 liveness 约定的理由：State family 的 type 是冻结的，新增 type 需要审批，一起定义 liveness 约定是合理的。避免增加 archetype 编写负担。

### 与 Status 系统的集成

`apply_status` 的 properties 字典中，`blocks_attack` 删除。取而代之的是一个新字段 `liveness_overrides`，显式声明这个 status 要改变哪些维度：

```
apply_status("frozen", 5.0, {
    movement_scale: 0.0,
    liveness_overrides: {
        movement: DISABLED,
        controllers: DISABLED,
    }
})
```

apply_status 内部从 `liveness_overrides` 构建 profile 并 push。过期时自动 pop。没有 `liveness_overrides` 的 status 和以前一样。

`movement_scale` 保留不变——活跃性控制"动不动"，movement_scale 控制"动多快"，两者共存。

### 与 Effect 效果链的集成

EffectRegistry 新增一个效果策略，允许效果链直接操作 override 栈，传入维度 + 值 + 优先级。这是 Coffee Bean 唤醒、临时 buff 等一次性能力的主要接口。

### required_state 的位置

`required_state` 保留在 TriggerRegistry 中不动。它和活跃性是两个不同层面的概念：

- 活跃性回答"触发器这个子系统现在能不能工作"
- required_state 回答"触发器在工作的情况下，当前状态是否满足执行条件"

两者正交共存。

## 要删除的旧机制

以下机制直接删除，不做兼容代理：

| 旧机制 | 删除后由什么替代 |
|--------|-----------------|
| `is_combat_active()` | `is_targetable() && is_damageable()` |
| `is_attack_blocked()` | `is_liveness_enabled(controllers)` |
| `blocks_attack` (status property) | `liveness_overrides` 字段 |
| `activate()` / `deactivate()` (FieldObjectRoot) | push/pop_liveness_override |
| FieldObjectRoot 硬编码 `is_combat_active() -> false` | 基础 profile 中外视维度默认 DISABLED |

## 各子系统查询点的改动

| 子系统 | 当前检查方式 | 改为 |
|--------|-------------|------|
| TriggerComponent._on_event | 无检查 | `is_liveness_enabled(triggers)` → 不通过则跳过 |
| BaseEntity.simulation_step | 无检查 | `is_liveness_enabled(state)` → 不通过则跳过 |
| ZombieRoot 移动逻辑 | `get_effective_movement_scale()` | `is_liveness_enabled(movement)` |
| ZombieRoot 攻击逻辑 | `is_attack_blocked()` | `is_liveness_enabled(controllers)` |
| ProjectileRoot 目标搜索 | `is_combat_active()` | `is_targetable()` |
| HealthComponent.take_damage | 无检查 | `is_damageable()` → 不通过则静默返回 |
| ProjectileRoot 碰撞检测 | `is_combat_active()` | `is_collidable()` |
| StateComponent._try_execute_transition | 无约束 | 检查 `allowed_transitions` 白名单 |

## 场景验证

### 蘑菇睡觉 → Coffee Bean 唤醒

1. Puffshroom 实例化，State family core.sleeping 编译
2. initial_state = "sleeping"，编译器附带声明：sleeping → dormant profile（triggers: DISABLED, controllers: DISABLED），allowed_transitions = {awake: {event: "entity.wake"}}
3. StateComponent 自动 push override（优先级 10）
4. TriggerComponent 每次收到 game.tick → is_liveness_enabled(triggers) → false → 跳过
5. Coffee Bean 效果执行 → push override（全部 ENABLED，优先级 30）
6. 优先级 30 > 10，triggers 恢复为 ENABLED
7. 同时触发 entity.wake 事件 → StateComponent 检查 allowed_transitions → awake 在白名单中 → 转换执行
8. pop 旧 sleeping override → push 新 awake override（无 liveness 声明 → 默认 all ENABLED）
9. 最终：所有维度 ENABLED

### Potato Mine 布防→激活→爆炸

1. 放置，initial_state = "arming"，编译器附带声明：arming → underground profile（targetable/damageable/collidable: DISABLED），allowed_transitions = {active: {after: 15.0}}
2. 抛射体飞过时 → is_targetable() → false → 不命中
3. arming 时间到 → 检查 allowed_transitions → active 在白名单中 → 转换执行
4. pop underground override → push active override（无 liveness 声明 → all ENABLED）
5. 所有维度恢复 ENABLED，proximity 触发器开始工作
6. 敌人靠近 → proximity 触发 → 效果链 → damage → entity.died

### Chomper 完整行为

1. 实例化，initial_state = "idle"，编译器附带声明完整状态图：
   ```
   allowed_transitions = {
     idle: {biting: {event: "enemy_in_range"}},
     biting: {chewing: {after: 0.5}},
     chewing: {digesting: {after: 1.0}},
     digesting: {idle: {after: 30.0}}
   }
   state_liveness = {
     idle: all ENABLED,
     biting: {triggers: DISABLED},
     chewing: {triggers: DISABLED, controllers: DISABLED},
     digesting: {triggers: DISABLED, controllers: DISABLED}
   }
   ```
2. 敌人进入范围 → proximity 触发器 → 发射 enemy_in_range 事件
3. StateComponent 检查 allowed_transitions[idle] → biting 在白名单中 → 转换
4. pop idle override → push biting override（triggers: DISABLED）→ 发射 state_exited/idle + state_entered/biting
5. biting 定时器到 → 检查 allowed_transitions[biting] → chewing 在白名单中 → 转换
6. chewing 定时器到 → 检查 allowed_transitions[chewing] → digesting 在白名单中 → 转换
7. 消化中：triggers DISABLED（不能再攻击），但 targetable 保持 ENABLED（可以被僵尸吃）
8. 如果此时另一个 Chomper 试图触发 enemy_in_range → triggers DISABLED → 无反应
9. 消化定时器到 → 检查 allowed_transitions[digesting] → idle 在白名单中 → 转换
10. 如果 digesting 期间收到 enemy_in_range 事件 → allowed_transitions[digesting] 中没有 biting → 发射 state_transition_rejected → 无效转换被正确拒绝

### Ice-shroom 冻结全场 + Magnet-shroom 同时吸装备

1. Ice-shroom 爆炸 → 对每个敌人 apply_status("frozen", 5.0, { liveness_overrides: { movement: DISABLED, controllers: DISABLED } })
2. 每个 zombie push override（优先级 20）
3. Magnet-shroom 的效果链 → 对目标 zombie push override（triggers: ENABLED, 优先级 50）
4. 合成结果：movement DISABLED（来自 frozen），controllers DISABLED（来自 frozen），triggers ENABLED（优先级 50 > 20）
5. frozen 过期 → pop status_frozen → movement 和 controllers 恢复
6. magnet 效果完成后 pop → triggers 恢复正常

### 割草机

1. FieldObjectRoot 的基础 profile：targetable/damageable/collidable DISABLED
2. 同车道僵尸触发割草机 → 效果链 push override（targetable/collidable ENABLED, 优先级 30）
3. 割草机开始扫掠，可以命中僵尸
4. 扫掠结束 → pop override → 回到默认的"不参与战斗"状态

## 不改变的东西

- **不新增 Mechanic family**。活跃性是 State family compiler 的内部扩展和 BaseEntity 的运行时能力
- **不改变冻结协议**。state_liveness 和 allowed_transitions 是编译器输出的一部分，不是协议语义
- **不改变 EventBus 的事件模型**。活跃性检查是子系统内部的前置守卫，不改变事件分发
- **不改变 movement_scale 倍率机制**。活跃性是开关，movement_scale 是倍率，两个独立轴
- **不改变 required_state**。它是触发器业务逻辑，和活跃性正交
- **不引入分层状态机（HSM）**。扁平状态 + 转换约束 + 活跃性 profile 覆盖所有已知需求

## 处理不了的边界

以下场景不属于活跃性范畴，需要其他机制：

| 场景 | 需要的机制 |
|------|-----------|
| Hypno-shroom 换队 | team 属性直接修改，不是子系统开关 |
