## 背景与目标

### 为什么需要棋盘多样性

当前 Open PVZ 的棋盘抽象层支持 ground / water / roof / air_only 四种槽位类型，以及覆盖 placed slot_config 的 per-slot 覆盖能力。但运行时棋盘构造仍然以"全 ground"为默认值：`_rebuild_slots()` 对所有 lane 的所有 slot 统一赋值 `slot_type = &"ground"`，然后通过 `_apply_slot_configs()` 覆盖个别格子。

这种设计对"全草地关卡"足够，但对以下场景缺乏支持：

- **泳池关卡**：第 1 行和第 6 行是草地，第 2-5 行是泳池。泳池行需要`water`槽位类型，但只有一小部分槽位允许放荷叶。
- **屋顶关卡**：所有行是屋顶地形，同行的 Y 坐标随 X 位置变化（坡度），投射物需要弹道修正。
- **夜间关卡**：没有天降阳光、蘑菇植物需要初始睡眠状态。
- **迷雾关卡**：右侧区域被迷雾覆盖，特定植物可驱散迷雾。
- **墓地地形**：部分格子被墓碑占据，不可种植，僵尸可从墓碑生成。

### 当前状态评估

现有系统已经具备的基础：

- 4 种 slot type (ground / water / roof / air_only) 和对应的默认标签
- 5 种放置角色 (primary / support / surface / cover / blocker / upgrade)
- Surface / substrate 模式 (water_pod, flower_pot_surface)
- 8 项放置验证链 (validate_request)
- 5 级 HeightBand
- BattlefieldPreset + BattlefieldMetrics 骨架
- BoardSlotConfig 支持 per-slot 覆盖 (slot_type, placement_tags)
- BattleRuleModule 支持运行时规则模块
- BattleModeHost 支持模式级模块加载

缺失的关键能力：

- **行级元数据**：没有 per-row 的地形类型声明，所有行从同一个 `_rebuild_slots()` 流程生成。
- **行级 Y 坐标配置**：lane Y 仍由 `BattleManager.get_lane_y()` 作为最终事实源，不是由预设数据驱动。
- **运行时环境效果**：没有日/夜切换、迷雾系统、蘑菇睡眠/唤醒等运行时机制。
- **僵尸生成过滤**：WaveRunner 不按行类型过滤僵尸生成（泳池行不应生成陆地僵尸）。
- **行级视觉主题**：泳池行和草地行应有不同的视觉表现。

## 当前抽象层

### 槽位类型与标签

当前系统通过 `BoardSlotCatalog` 定义 4 种槽位类型及其默认标签：

| 槽位类型 | 默认标签 | 用途 |
|----------|----------|------|
| `ground` | `["ground", "supports_primary"]` | 标准草地格子 |
| `water` | `["water"]` | 水面格子（需荷叶等 substrate） |
| `roof` | `["roof"]` | 屋顶格子 |
| `air_only` | `["air_only", "supports_air"]` | 仅空中单位可放置 |

### 放置角色系统

5 种放置角色用于 slot 占位管理：

| 角色 | 用途 |
|------|------|
| `primary` | 主力植物（常规格子） |
| `support` | 支援装饰（如南瓜壳内层） |
| `surface` | 水面/表面部署（荷叶、花盆） |
| `cover` | 保护覆盖（南瓜壳） |
| `blocker` | 阻挡角色 |
| `upgrade` | 升级替换（如路灯花→加强路灯花） |

### Surface/substrate 模式

现有系统支持 surface 角色作为"基底"：荷叶 (`water_pod`) 占用 surface 角色后将 `water` 标签改为 `ground` + `supports_primary`，使主力植物可在水面放置。

### 放置验证链

`validate_request()` 按顺序检查 8 项约束：

1. request 非空
2. lane 有效
3. slot_index 在范围内
4. slot 存在
5. placement_tags 非空
6. slot 标签匹配
7. 实体类型（植物/僵尸）对应的限制
8. 角色占位冲突

### HeightBand 系统

5 个高度带用于投射物命中判定：`low / mid_low / mid / mid_high / high`。实体声明 `hit_height_range`，投射物声明飞行高度，通过 overlap 检测。

### BattlefieldPreset + BattlefieldMetrics

`BattlefieldPreset` 是数据入口 Resource，声明棋盘基础参数：

```gdscript
@export var lane_count := 2
@export var board_slot_count := 5
@export var board_slot_origin_x := 160.0
@export var board_slot_spacing := 96.0
@export var board_slot_configs: Array = []
```

`BattlefieldMetrics` 是运行时只读值对象，提供 slot_position()、slots_to_world()、world_to_slot_index()、resolve_range() 等几何换算。

### 缺失项总结

- **行级类型声明**：没有 lane-level 的数据结构描述该行是草地、泳池、屋顶还是 dirt（不可种植行）。
- **行级高度配置**：lane Y 坐标未数据化，Roof 坡度无法表达。
- **环境元数据**：BattlefieldPreset 没有 is_night、has_fog、fog_start_column 等字段。
- **运行时环境模块**：BattleRuleModule 系统已存在，但没有针对夜间/迷雾的规则模块。
- **生成过滤**：WaveRunner 不根据行类型过滤可生成的僵尸类型。

## 原版 PVZ 参考

### 背景类型系统

原版 PVZ 使用 `BackgroundType` 枚举区分关卡大类型：

| 背景 | 属性 |
|------|------|
| DAY_GRASS | 白天草地，无泳池，无迷雾，无屋顶 |
| NIGHT_GRASS | 夜间草地，无泳池，有迷雾（但非全部场景） |
| DAY_POOL | 白天泳池，有泳池行，无迷雾 |
| NIGHT_POOL | 夜间泳池，有泳池行，有迷雾 |
| DAY_ROOF | 白天屋顶，无泳池，有屋顶坡度 |
| NIGHT_ROOF | 夜间屋顶，无泳池，有迷雾 |
| ZEN_GARDEN / TREE_OF_WISDOM / ... | 特殊背景 |

实际运行时通过布尔组合描述差异：

- `is_night`：影响是否天降阳光、蘑菇行为
- `has_pool`：影响是否出现泳池行
- `has_fog`：影响迷雾渲染与清除机制
- `has_roof`：影响坡度 Y 偏移与投射物弧线

### 行类型 (PlantRowType)

`PlantRowType` 枚举定义每行的地形类型，保存在 `PlantRowType[]` 数组中（每行一个）：

| 行类型 | 含义 |
|--------|------|
| NORMAL | 普通草地行 |
| POOL | 泳池行（有水，需荷叶） |
| DIRT | 不可种植行（原版仅用于泳池关最后一行） |
| HIGH_GROUND | 屋顶行（有坡度，需花盆） |

原版泳池关卡的行配置示例（6 行）：

```text
[0] NORMAL (草地，陆生僵尸行走)
[1] POOL (水面，水生僵尸行走)
[2] POOL (水面，水生僵尸行走)
[3] POOL (水面，水生僵尸行走)
[4] POOL (水面，水生僵尸行走)
[5] DIRT (不可种植)
```

屋顶关卡的行配置示例（5 行）：

```text
[0] HIGH_GROUND (屋顶，FlowerPot 需求，坡度 Y)
[1] HIGH_GROUND
[2] HIGH_GROUND
[3] HIGH_GROUND
[4] HIGH_GROUND
```

### 格子类型 (GridSquareType)

`GridSquareType` 二维数组由 `PlantRowType + X 位置` 派生。部分格子可能有特殊标记，如原版泳池关卡的泳池行最左侧 1-2 格不可通过荷叶种植（水太浅）。

### 泳池 (Pool)

泳池关卡的关键机制：

- 6 行布局：1 草地 + 4 泳池 + 1 dirt（不可种植）
- 泳池行需要荷叶 (LilyPad) 作为 surface 基底才能放植物
- 泳池行使用特定水生僵尸（Duckytube Zombie 等）
- 泳池行视觉上为蓝色水面，草地行为绿色草地
- dirt 行在 grass 风格的泳池关卡中为不可种植区域

### 屋顶 (Roof)

屋顶关卡的关键机制：

- 所有行为 HIGH_GROUND，需要花盆 (FlowerPot) 作为基底
- 每行的 Y 坐标随 X 增加而增大（坡度效果），坡度因子约为 `0.2 * X` 像素
- 投射物需要弧线修正：抛射体在屋顶地形上应使用抛物线轨迹补偿坡度
- 部分原版屋顶关卡有"天花板"限制抛射体高度

### 迷雾 (Fog)

迷雾机制：

- 迷雾覆盖右侧部分棋盘（通常从列 5-6 开始），向左渐淡
- 棋盘格有独立的 `fog_alpha` 属性，决定迷雾透明度
- 迷雾清除源：Plantern（半俓 4 格清除）、Torchwood（半径 1 格清除）、Blover（全屏短暂清除）
- 迷雾中僵尸不可见，但碰撞和伤害判定仍然存在
- 迷雾在夜间泳池/夜间屋顶关卡中出现

### 夜间 (Night)

夜间机制：

- 没有天降阳光（Sky Sun Drops 关闭）
- 蘑菇植物初始状态为 `sleeping`，需唤醒后才可攻击
- Coffee Bean 可唤醒蘑菇（一次性消耗品）
- 夜间关卡视觉为暗色调

### 墓碑 (Grave)

墓碑机制：

- 墓碑是占用格子的静态物体，不可放置植物
- 墓碑可以从格子中生成僵尸
- 墓碑可被特定植物破坏（如 Grave Buster）
- 墓碑在夜间草地关卡中出现

### 位掩码地形系统 (PVZ-Godot-Dream 参考)

PVZ-Godot-Dream 移植版使用 `GridBitmask` 实现更精细的格子颗粒度：

- 每个格子有独立的 bitmask（如可种植位、通行位、有墓碑位等）
- 地形变化通过修改格子的 bitmask 实现
- 放置验证直接查询目标格子的 bitmask

当前方案不采用 bitmask 路线，而是复用现有的 tag 系统。理由见后文"不做的事"。

## 设计方案

### 总体架构

棋盘多样性采用两层分离架构：

```
[地形布局层 (Terrain Layout)]
  固定、数据驱动、运行时不可变
  LaneConfig[] + BattlefieldPreset 增强
  -> 决定 slot 类型、坐标、视觉主题

[环境效果层 (Environment Effects)]
  可变、规则驱动、可被模式/模块替换
  DayNightRuleModule / FogRuleModule
  -> 决定夜间行为、迷雾状态、蘑菇睡眠
```

两层互相独立：地形布局不依赖环境模块存在与否，环境模块消费地形布局但不修改它。

### 方案 A: 行类型数据层 (LaneConfig)

#### LaneConfig Resource

新增 `LaneConfig` Resource，声明单行的地形配置：

```gdscript
# scripts/battle/lane_config.gd (草案)
extends Resource
class_name LaneConfig

@export var lane_index := 0
@export var lane_type: StringName = &"grass"
@export var slot_type_override: StringName = StringName()
@export var base_tags_override: PackedStringArray = PackedStringArray()
@export var height_offset := 0.0
@export var visual_theme: StringName = StringName()
@export var spawn_filter_tags: PackedStringArray = PackedStringArray()
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `lane_index` | int | 行号，从 0 开始 |
| `lane_type` | StringName | 行类型标识符，如 `&"grass"` / `&"pool"` / `&"roof"` / `&"dirt"` |
| `slot_type_override` | StringName | 非空时覆盖该行所有 slot 的 slot_type（如 `&"water"`） |
| `base_tags_override` | PackedStringArray | 非空时覆盖该行所有 slot 的默认标签（如 `["water"]`） |
| `height_offset` | float | 该行相对于基准 Y 的偏移量（用于屋顶坡度计算） |
| `visual_theme` | StringName | 视觉主题标识，如 `&"pool_water"` / `&"roof_tile"` |
| `spawn_filter_tags` | PackedStringArray | 该行允许生成的僵尸 tag 过滤（空表示不限制） |

关键设计决策：`lane_type` 使用 `StringName` 而非枚举。原因：

- 与现有 `slot_type` 命名风格一致（也是 StringName）
- 扩展性好（后续新增地下、空中等行类型不需要改枚举）
- 不需要注册表查重，StringName 本身就是驻留标识符

#### BattlefieldPreset 增强

在现有 `BattlefieldPreset` 基础上新增字段：

```gdscript
# 新增字段
@export var lane_configs: Array[Resource] = []  # LaneConfig[]
@export var is_night := false
@export var has_fog := false
@export var fog_start_column := 6
@export var lane_y_positions: Array[float] = []  # 显式 Y 坐标覆盖
@export var lane_origin_y := 0.0
@export var lane_spacing := 0.0
```

#### 消费端: BattleBoardState._build_slots()

`_rebuild_slots()` 改为读取 `lane_configs` 数据：

```gdscript
# 伪代码示意
func _rebuild_slots() -> void:
    var lane_ids = battle.get_lane_ids()
    for lane_id in lane_ids:
        var lane_config = _resolve_lane_config(lane_id)
        for slot_index in range(board_slot_count):
            var slot = BoardSlot.new()
            var slot_type = lane_config.slot_type_override if lane_config.slot_type_override != StringName() else &"ground"
            var base_tags = lane_config.base_tags_override if not lane_config.base_tags_override.is_empty() else BoardSlotCatalog.default_tags_for(slot_type)
            var world_pos = _compute_slot_position(lane_id, slot_index, lane_config)
            slot.configure(lane_id, slot_index, world_pos, slot_type, base_tags)
            _slots[_slot_key(lane_id, slot_index)] = slot
    _apply_slot_configs()
```

关键变更：

- 行级 `slot_type_override` 和 `base_tags_override` 在 slot 层次优先于 `BoardSlotConfig`。
- `BoardSlotConfig` 的 per-slot 覆盖仍然可以覆盖特定格子（如泳池行最左侧不可种植格）。
- `height_offset` 传给 `BattlefieldMetrics` 用于该行 slot_position 的 Y 坐标计算。

#### 消费端: WaveRunner 僵尸生成过滤

WaveRunner 在生成僵尸时检查目标行和僵尸的 tag 匹配：

```gdscript
# 伪代码示意
func _can_spawn_in_lane(lane_id: int, spawn_entry_tags: PackedStringArray) -> bool:
    var lane_config = _resolve_lane_config(lane_id)
    if lane_config.spawn_filter_tags.is_empty():
        return true  # 无限制
    for tag in spawn_entry_tags:
        if lane_config.spawn_filter_tags.has(tag):
            return true
    return false
```

泳池行的 `spawn_filter_tags` 可设为 `["aquatic"]`，只允许带 `aquatic` tag 的僵尸生成在该行。

#### LaneType 预设枚举

虽然 lane_type 使用 StringName，但建议核心定义一组标准类型常量，供内容资源和 `BoardSlotCatalog` 消费：

| lane_type | slot_type_override | base_tags_override | spawn_filter_tags | 说明 |
|-----------|--------------------|--------------------|--------------------|------|
| `grass` | ground | `["ground", "supports_primary"]` | 空 | 标准草地 |
| `pool` | water | `["water"]` | `["aquatic"]` | 泳池水面 |
| `roof` | roof | `["roof"]` | 空 | 屋顶 |
| `dirt` | dirt | `[]`（空，自然不可放置） | 空 | 不可种植行 |

dirt 行的处理方式：给 `slot_type = &"dirt"`、`base_tags = []`（默认无标签）。放置验证时因为无法满足 `placement_tags` 匹配，所有植物不可放置。不需要额外的 dirt 逻辑分支。

### 方案 C: 环境 Rule Module 层

环境效果通过 `BattleRuleModule` 框架实现，每个环境模块是独立的、可组合的 `BattleRuleModule` 实例。

#### DayNightRuleModule

```gdscript
# 草案参数
{
    "module_id": &"environment.day_night",
    "params": {
        "is_night": true,                    # 初始是否为夜间
        "sun_drop_enabled": false,           # 是否启用天降阳光
        "nocturnal_tag": "nocturnal",        # 夜间植物的 tag
        "sleep_state": "sleeping",           # 初始状态
        "wake_trigger_event": "entity.woke", # 唤醒后发射的事件
        "can_toggle": false,                 # 是否支持运行时昼夜切换
        "toggle_interval": 120,              # 切换间隔（仿真秒）
        "toggle_initial_phase": 0,           # 初始相位
    }
}
```

职责：

- 启动时将 `BattleEconomyState.sun_drop_enabled` 设为 `is_night` 的取反。
- 对 spawn 事件和 archetype 编译结果进行后处理：如果 `is_night == true` 且植物有 `nocturnal` tag，将 `StateComponent` 初始态设为 `sleeping`。
- 监听 `entity.woke` 事件（Coffee Bean 使用效果后发射，或通过效果 `apply_status` 清除 sleeping）。
- 如果 `can_toggle == true`，通过 `toggle_interval` 和 `toggle_initial_phase` 驱动日/夜交替。

#### FogRuleModule

```gdscript
# 草案参数
{
    "module_id": &"environment.fog",
    "params": {
        "fog_start_column": 6,               # 迷雾起始列
        "fog_max_alpha": 0.85,               # 迷雾最大不透明度
        "fog_clearer_tag": "fog_clearer",    # 可清除迷雾的实体 tag
        "fog_clear_radius_slots": 4,          # 清除半径（格）
        "fog_clearer_min_radius": 1,          # 最小清除半径（用于 Torchwood）
        "blover_clear_duration": 10,          # Blover 全屏清除时长（仿真秒）
        "fog_visual_layer": "fog",           # 视觉层 ID
    }
}
```

职责：

- 初始计算迷雾覆盖范围：从 `fog_start_column` 到棋盘最右侧，每格计算 `fog_alpha`（越靠右越浓）。
- 注册 `fog_clearer` tag 检测：对棋盘上的实体按 tag 查询，更新其周围半径内的迷雾 alpha（降低到 0）。
- 提供 `is_position_in_fog(world_position)` 查询，供视觉层和检测层使用。
- 支持 Blover 效果的临时全屏清除：清除后按 `blover_clear_duration` 内迷雾 alpha 为 0，到期后恢复。
- 视觉层通过 `VisualStageLayerService` 激活 `fog` 层，消费迷雾 alpha 数据做半透明覆盖渲染。

#### 模块加载方式

模式定义 (BattleModeDef) 中声明需要加载的环境模块：

```gdscript
# BattleModeDef 增强：增加 modules 列表
{
    "mode_id": &"adventure_pool_night",
    "modules": [
        {"module_id": &"environment.day_night", "params": {"is_night": true}},
        {"module_id": &"environment.fog", "params": {"fog_start_column": 5}},
    ]
}
```

也可以通过在 `BattleRuleModule` 中设置 `module_id` 直接引用。

BattleModeHost 在 `_setup_modules()` 时根据 module_id 从 `BattleModeModuleRegistry` 查找对应的 callable，创建并 attach 模块实例。模块通过 EventBus 订阅事件，通过 BattleManager 访问子系统状态。

### 两层分离的原则

```
地形布局层 (LaneConfig)
- 数据来源: BattlefieldPreset
- 加载时机: 战斗 setup 时
- 可变性: fixed (运行时不可变)
- 影响: slot 类型、slot 标签、槽位坐标、行 Y 偏移
- 不依赖: 环境模块、模式

环境效果层 (RuleModule)
- 数据来源: BattleModeDef.modules / BattleRuleModule
- 加载时机: 战斗 setup 后、模式初始化时
- 可变性: runtime (可被事件/定时器改变)
- 影响: 经济行为 (阳光)、实体状态 (睡眠)、视觉 (迷雾)
- 依赖: 地形布局 (查询 lane 类型、slot 位置)
```

## 数据流设计

### 完整数据流

```text
                          BattleScenario
                               |
                      BattlefieldPreset
                    /                    \
         lane_configs[]            is_night / has_fog
               |                         |
     BattleBoardState           BattleModeHost
     _rebuild_slots()               |
               |              DayNightRuleModule
     per-lane slot 类型          FogRuleModule
     坐标、Y 偏移                    |
               |                    |
     BattlefieldMetrics         EventBus 订阅
     slot_position()            modify game state
     get_lane_y()
               |
     SpatialIndex / Detection
     WaveRunner (生成过滤)
```

### 关键路径分解

**1. BattlefieldPreset -> LaneConfig -> BattleBoardState slot 类型推导**

```
BattlefieldPreset.lane_configs[]
  -> _rebuild_slots()
  -> per lane: lane_config.slot_type_override 非空则使用，否则取默认 ground
  -> per lane: lane_config.base_tags_override 非空则使用，否则取 BoardSlotCatalog.default_tags_for(slot_type)
  -> BoardSlotConfig 覆盖仍可 per-slot 微调
  -> slot 构造时传入 height_offset 到 world_position.y 计算
```

**2. BattlefieldPreset.is_night -> DayNightRuleModule 初始状态**

```
BattlefieldPreset.is_night (初始值)
  -> BattleModeHost 构造时读取
  -> 传给 DayNightRuleModule params.is_night
  -> DayNightRuleModule._setup():
     -> 设置 BattleEconomyState.sun_drop_enabled = !is_night
     -> 对所有已生成实体检查 nocturnal tag -> 设置 sleeping 状态
     -> 对新生成实体 spawn 事件 -> 同上检查
```

**3. BattlefieldPreset.has_fog -> FogRuleModule 激活**

```
BattlefieldPreset.has_fog
  -> BattleModeHost 构造时读取
  -> 传给 FogRuleModule params
  -> FogRuleModule._setup():
     -> 计算 fog alpha 网格
     -> 注册 fog_clearer 实体检测
     -> 激活视觉 fog layer
```

**4. LaneConfig.height_offset -> BattlefieldMetrics lane Y 计算**

```
BattlefieldMetrics.lane_y_positions 构建时:
  -> 如果 BattlefieldPreset.lane_y_positions 非空，直接使用
  -> 否则使用 lane_origin_y + lane_id * lane_spacing + lane_config.height_offset
  -> 如果仍未有值，fallback 到 BattleManager.get_lane_y()
```

**5. LaneConfig.lane_type -> WaveRunner 僵尸生成过滤**

```
WaveRunner 尝试 spawn:
  -> 读取 spawrn_entry 的目标 lane
  -> 查询该 lane 的 LaneConfig.spawn_filter_tags
  -> 非空时检查 spawrn_entry 的 placement_tags / archetype tags
  -> 不匹配则跳过该 lane 寻找下一个可用 lane
```

### Phase 1: LaneConfig Resource + BattleBoardState 消费 + 新 Preset

**范围**：核心数据结构改造和新预设配置

**具体任务**：

- 新增 `LaneConfig` Resource（`scripts/battle/lane_config.gd`）
- `BattlefieldPreset` 增加 `lane_configs` / `is_night` / `has_fog` / `fog_start_column` / `lane_y_positions` / `lane_origin_y` / `lane_spacing` 字段
- `BattleBoardState._rebuild_slots()` 改造：读取 lane_configs 生成 per-row slot 类型和标签
- `BattlefieldMetrics` 改造：支持从 lane_configs 读取 Y 坐标偏移和显式 lane_y_positions
- `BoardSlotCatalog` 增加 `dirt` 类型（空标签，不可放置）
- 新增 3 个预设 `.tres`：
  - `day_5lane`：标准 5 行草地
  - `pool_6lane`：泳池关卡（1 grass + 4 pool + 1 dirt）
  - `roof_5lane`：屋顶关卡（5 roof）
- 更新 `_apply_slot_configs()`：确保 per-slot config 在 lane_config 之后覆盖
- 验证：新预设下地面放置仍然正常工作

**验证场景**：

- 3 个新预设的基本 smoke 验证（实体生成、槽位创建、放置成功）
- 多行类型混合预设（泳池）的槽位类型 tag 正确性
- `dirt` 行拒绝放置验证

### Phase 2: Pool 行僵尸过滤 + NightRuleModule

**范围**：僵尸生成按行过滤 + 夜间环境模块

**具体任务**：

- `WaveRunner` 增加行级生成过滤：查询 `LaneConfig.spawn_filter_tags`
- 水生僵尸 archetype 增加 `aquatic` tag（如 Ducky Tube Zombie 等）
- 实现 `DayNightRuleModule`：
  - 控制 `BattleEconomyState.sun_drop_enabled`
  - 监听 `entity.spawned` 事件，对有 `nocturnal` tag 的植物设置 `sleeping` 初始状态
  - 提供 `toggle_day_night()` 接口支持运行时切换
- 现有蘑菇植物 archetype 增加 `nocturnal` tag
- 注册 `environment.day_night` 到 `BattleModeModuleRegistry`
- `BattleEconomyState` 暴露 `sun_drop_enabled` getter/setter（当前可能无此公开接口）

**验证场景**：

- 泳池关：水生僵尸只在 pool 行生成，陆地僵尸只在 grass 行生成
- 夜间关：蘑菇初始为 sleep 状态，不发射攻击
- 白天关：蘑菇初始为 active 状态
- 夜间关无天降阳光

### Phase 3: Roof 坡度 + 投射物弧线修正 + FogRuleModule

**范围**：屋顶的几何修正 + 投射物弹道 + 迷雾模块

**具体任务**：

- **Roof 坡度 Y 偏移**：
  - `BattlefieldMetrics.slot_position()` 读取 `LaneConfig.height_offset`，叠加基于 X 位置的坡度因子
  - 坡度算法草案：`lane_y = base_y + slot_index * slope_factor`，其中 slope_factor 由 `LaneConfig` 参数控制（屋顶约为 2-3 world unit per slot）
  - `BattleManager.get_lane_y()` fallback: 如果有 `BattlefieldPreset.lane_y_positions` 或 metrics 已计算 lane Y，委派给 metrics
- **投射物弧线修正**：
  - 屋顶关卡的植物发出的非抛物线投射物（如直线豌豆）应自动附加轻微弧线补偿，避免与坡度地面发生碰撞判定异常
  - 检查 `ProjectileMovementBase` 的 ground_position 计算是否依赖 Y 坐标
  - 不做硬编码屋顶逻辑；改为使 `ProjectileFlightProfile` 或 `ProjectileMovementDef` 支持 `roof_arc_compensation` 可选参数
- **FogRuleModule**：
  - 实现迷雾网格计算（每列 `fog_alpha`）
  - 注册 `fog_clearer` tag 检测
  - Blover `apply_status` effect 扩展：触发 FogRuleModule 的全屏清除
  - 视觉层：通过 `VisualStageLayerService` 激活 fog layer，消费 alpha 数据

**验证场景**：

- 屋顶关 slot Y 坐标正确偏移
- 屋顶关投射物击中地面实体（无异常浮空）
- 迷雾覆盖范围正确
- Plantern/Torchwood 清除迷雾半径正确
- Blover 临时清除迷雾并在到期后恢复

### Phase 4: Grave / GridItem 系统（延后）

**范围**：墓碑等格子级障碍物的完整系统（当前阶段不做，仅做设计预留）

**设计预留**：

- 如果未来引入墓碑/障碍物系统，应该作为 `GridItem` Resource 附着在 `BoardSlot` 上
- `GridItem` 应包含：`item_type`（如 "grave"）、`block_planting`（bool）、`block_movement`（bool）、`spawn_zombie_config`（可选）

- 此系统设计上独立于 LaneConfig 和环境模块，不阻塞 Phase 1-3 的落地

## 不做的事

以下内容明确排除在棋盘多样性系统的当前设计范围之外：

1. **不修改冻结 Mechanic family**：触发器、效果、detection、controller 等 10 个 family 保持不变。夜间/迷雾行为通过 RuleModule 实现，不通过 Mechanic 扩展。

2. **不实现位掩码 (bitmask) 地形系统**：当前 tag 系统足够表达 slot 的地形属性。位掩码的精度虽高，但引入复杂度不匹配当前需求。后续如果出现更精细的格子级修改需求，可重新评估。

3. **不实现完整的墓碑/GridItem 系统**：墓碑/障碍物是独立的功能模块，不阻塞 Phase 1-3 的落地。Phase 4 仅做设计预留。

4. **不做 per-cell 地形修改**：初始版本不支持运行时改变单个格子的地形类型。水面变草地（如原版池塘清理）等需求不在第一版覆盖。

5. **不把 lane_type 作为扩展点**：lankton_type 的集合由核心定义，扩展包不得注册新的 lane_type。如有合理的新类型需求，通过 ADR 加入核心集合。

6. **不实现禅境花园 / 智慧树背景**：特殊模式背景不在棋盘多样性的当前目标内。

7. **不修改 `BattleManager` 的直接特判**：行类型相关的逻辑通过 `LaneConfig` 查询，不在 `BattleManager` 中添加 if/switch 分支。

8. **不会一次迁移所有现有 `.tres`**：Phase 1 的新预设只增加不修改，现有验证场景不受影响。

## 决策记录

### D1: lane_type 使用 StringName 而非枚举

- **问题**：行类型应该用枚举还是 StringName？
- **决策**：StringName
- **理由**：
  - 与现有 `slot_type` 命名风格一致（`slot_type: StringName`）
  - 扩展性好，新增行类型不需要改枚举
  - 不需要注册表查重，StringName 本身就是驻留标识符
- **后果**：内容作者需注意拼写错误，建议核心提供一组常量（`Constants.LANE_TYPE_GRASS`）

### D2: dirt 行通过空标签实现"不可放置"

- **问题**：不可种植行如何表示？
- **决策**：`slot_type = &"dirt"`，`base_tags = []`（空标签数组）
- **理由**：放置验证检查 placement_tags 必须在 slot 的 effective_tags 中存在。空标签数组将自然拒绝所有放置请求。不需要额外的逻辑分支。
- **后果**：如果以后需要 dirt 行支持某些特殊放置（如 Grave Buster），只需要在 dirt 行添加对应标签，而不是修改验证逻辑。

### D3: 僵尸生成过滤复用 tag 系统

- **问题**：泳池行如何防止陆地僵尸生成？
- **决策**：通过 `LaneConfig.spawn_filter_tags` 限制允许的 tag；僵尸 archetype 声明 `aquatic` 等 tag
- **理由**：
  - 复用现有 tag 系统，不需要额外的僵尸类型枚举
  - 扩展灵活：新增"空中僵尸""地下僵尸"只需要加 tag
  - ztype 的 spawn_entry 已有 tags 字段
- **后果**：需要梳理僵尸 archetype 的 tags，确保每个僵尸都有正确的生成 tag

### D4: is_night / has_fog 放在 BattlefieldPreset，可由 RuleModule 覆盖

- **问题**：环境标志（is_night / has_fog）应该放在哪里？
- **决策**：在 `BattlefieldPreset` 中作为初始值声明，`BattleModeHost` 可选择通过 RuleModule 参数覆盖运行时值
- **理由**：
  - BattlefieldPreset 负责"这个预设是什么"（数据声明）
  - BattleModeHost / RuleModule 负责"运行时实际是什么"（行为覆盖）
  - 两者并行不悖：预设提供默认，模式可覆盖
  - 例如：同一个泳池预设可以用在白天模式或夜间模式，只需改变 `is_night` 参数
- **后果**：规则模块初始化时需要读取 preset 的初始值作为默认，然后检查模式参数是否覆盖

### D5: 日/夜交替可通过 DayNightRuleModule 参数驱动

- **问题**：是否支持运行时日/夜交替？
- **决策**：阶段性支持。Phase 2 只支持固定夜间或固定白天（`can_toggle = false`），后续通过 `can_toggle = true` + `toggle_interval` 开启交替。
- **理由**：
  - 第一优先级是基础夜间功能（蘑菇睡眠、无阳光）
  - 日/夜交替目前无紧迫需求，但设计上预留
  - toggle 机制通过 RuleModule 参数配置，不需要新增基础设施
- **后果**：DayNightRuleModule 的 toggle 逻辑在 Phase 2 不做完整验证，仅保证接口可调用

### D6: 两层分离 (地形 vs 环境)

- **问题**：地形布局和环境效果应该合在一个系统还是分离？
- **决策**：分离。地形布局固定、数据驱动；环境效果可变、规则驱动。
- **理由**：
  - 单一职责：每层只做自己该做的事
  - 可测试性：地形布局可以在没有环境模块时独立验证
  - 可组合性：同一个地形预设可以搭配不同环境模块
  - 与现有架构一致：BattlefieldPreset 做数据，BattleModeHost/BattleRuleModule 做行为
- **后果**：系统边界清晰，但需要额外的模块注册和参数传递

### D7: 屋顶坡度不硬编码到 BattleManager

- **问题**：屋顶的 Y 坐标偏移逻辑放在哪里？
- **决策**：放在 `BattlefieldMetrics` 中，通过 `lane_config.height_offset` 和可选的 `slope_factor` 参数计算
- **理由**：
  - 符合"metrics 是尺子"的设计原则
  - BattleManager.get_lane_y() 迁移期 fallback 后最终委派 metrics
  - 不把屋顶逻辑变成 BattleManager 的特判
- **后果**：metrics 需要支持 slope 模式，增加少量复杂度

### D8: 迷雾模块不依赖 Mechanic 扩展

- **问题**：迷雾清除效果（Plantern / Torchwood / Blover）如何实现？
- **决策**：通过在 `FogRuleModule` 内部监听 `entity.spawned` 和效果触发事件，查询有 `fog_clearer` tag 的实体来计算清除半径。不清除效果本身通过 Mechanic 系统注册新 effect。
- **理由**：
  - 不违反"效果通过 EffectRegistry 注册"的总原则
  - 但清除迷雾是环境行为，不是实体行为，更适合在模块内完成
  - 不需要为迷雾修改冻结效果集
- **后果**：模块需要主动查询和订阅，不能依赖编译器处理
