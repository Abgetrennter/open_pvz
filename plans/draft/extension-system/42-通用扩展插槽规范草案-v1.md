# 通用扩展插槽规范草案-v1

> 这份文档用于把扩展系统中的“通用插槽”概念固定下来，避免后续在 `EffectRegistry`、`ControllerRegistry`、`ProjectileMovement`、`MechanicCompilerRegistry` 等子系统里各自发展出互不兼容的扩展方式。它回答的不是“某个具体能力怎么实现”，而是“主仓如何提供可插拔入口，扩展包如何声明贡献项，运行时如何安全分发”。

---

## 文档定位

这篇文档主要回答：

- 什么是通用扩展插槽
- 插槽、贡献项、注册表、运行时分发之间是什么关系
- 扩展包注册新能力时必须满足哪些命名、信任和验证约束
- 已有系统哪些部分已经具备插槽雏形，哪些部分仍缺失
- `ProjectileMovement` 应如何从硬编码分发演进为通用插槽

这篇文档不负责：

- 直接实现新的 registry
- 直接改变现有扩展包格式
- 直接批准新增 Mechanic family
- 直接定义最终安装器、包管理器或 UI 交互

## 一句话结论

通用扩展插槽是：

> 主仓提供稳定、可校验、可注册、可分发的抽象入口；扩展包通过资源化贡献项挂入具体能力；运行时只依赖注册表和统一接口，不直接依赖具体扩展实现。

它不是：

- 为某个扩展能力在主仓写一个新 `match` 分支
- 让扩展包随意执行启动脚本并修改全局状态
- 绕过协议校验直接把任意参数塞进运行时
- 为了扩展包重新打开新增一级 Mechanic family 的口子

## 当前仓库状态判断

当前项目已经有“通用插槽”的概念和局部实现，但尚未形成统一规范。

### 已较成熟

`EffectRegistry` 已经具备较完整的扩展插槽雏形：

```text
EffectDef
-> strategy_script
-> EffectRegistry 扫描 extension effect 资源
-> register_strategy(effect_id, Callable)
-> EffectExecutor 按 effect_id 分发
```

这说明扩展包可以新增 effect 行为，而不需要改 `EffectExecutor` 主链。

### 已有注册表，但扩展包接入不完整

以下系统已经有 registry 或类似接口：

- `MechanicCompilerRegistry`
- `ControllerRegistry`
- `DetectionRegistry`
- `TriggerRegistry`

但它们当前主要服务主仓内置注册。扩展包自动发现、贡献项资源格式、命名空间守卫、重复注册守卫和信任等级检查尚未统一。

### 当前明显缺口

`ProjectileMovement` 当前仍以硬编码分发为主：

```text
move_mode = parabola -> ProjectileMovementParabola
move_mode = track    -> ProjectileMovementTrack
else                 -> ProjectileMovementLinear
```

因此扩展包若要新增 `my_pack.zigzag` 这类全新轨迹语义，当前不能只靠扩展包资源和脚本完成；主仓缺少 `ProjectileMovementRegistry` 或等价插槽。

## 核心术语

### Slot

插槽。主仓声明的可扩展入口。

示例：

```text
effect
projectile_movement
controller
detection
mechanic_compiler
mode_module
```

插槽回答：

- 这里允许扩展什么
- 扩展项的 key 是什么
- 扩展项需要实现什么接口
- 哪些包类型和信任等级允许注册
- 运行时如何调用它

### Contributor

贡献项。扩展包提供给插槽的具体能力声明。

示例：

```text
EffectDef(effect_id = "my_pack.chain_bounce")
ProjectileMovementDef(move_mode = "my_pack.zigzag")
ControllerDef(controller_id = "my_pack.orbit")
MechanicCompilerDef(type_id = "my_pack.volley")
```

贡献项应优先是 `Resource`，因为 Resource 可以被校验、索引、报告来源，并进入验证场景。

### Registry

注册表。主仓维护的 `id -> definition / strategy / factory` 映射。

注册表负责：

- 接收贡献项
- 校验 id、命名空间、接口、重复项和信任等级
- 提供查询和分发入口
- 记录协议问题

### Strategy

策略。贡献项背后的运行时代码，通常是 `Callable`、`Script` 或特定基类实例。

策略不能直接决定自己是否被信任。信任判断应由 manifest、slot 规则和 registry 共同完成。

---

## 统一模型

所有通用插槽都应尽量收敛成同一条链：

```text
扩展包 manifest
        |
        v
贡献项 Resource
        |
        v
Slot Registry 校验并注册
        |
        v
运行时通过 id 查询 registry
        |
        v
统一接口调用 Strategy
        |
        v
DebugService / validation 记录结果
```

主仓只负责“提供插槽和分发协议”，扩展包负责“提供具体贡献项”。

---

## Slot 定义规范

每个 slot 必须显式定义以下字段。

### 1. 基本身份

```text
slot_id
display_name
runtime_owner
registry_autoload
```

示例：

```text
slot_id = projectile_movement
runtime_owner = ProjectileRoot
registry_autoload = ProjectileMovementRegistry
```

### 2. key 规则

```text
key_property
core_namespace
extension_namespace_rule
allow_core_override
```

默认规则：

- 主仓内置能力使用 `core.*`
- 扩展包能力使用 `<pack_id>.*`
- 扩展包默认不得注册或覆盖 `core.*`
- 重复 id 默认拒绝，不做隐式覆盖

### 3. 信任等级

```text
required_trust_level
allowed_pack_types
```

建议分级：

```text
data_only       只能贡献资源和参数
rule_extended   可以贡献受控规则脚本
trusted_runtime 可以贡献运行时代码
```

示例：

- `EffectDef` 带 `strategy_script`：至少 `rule_extended`
- `ProjectileMovementDef` 带 movement script：至少 `trusted_runtime`
- 纯 archetype / mechanic 资源：可为 `data_only`

### 4. 接口签名

每个 slot 必须定义策略接口。

示例：

```text
effect strategy:
  execute(context, params, node) -> EffectResult

projectile movement:
  movement_script extends ProjectileMovementBase
  configure_movement(params: Dictionary) -> void
  physics_process_projectile_move(delta: float) -> ProjectileMoveResult

controller strategy:
  process(owner: Node, spec: Dictionary, delta: float, blackboard: Dictionary) -> void
```

### 5. 失败语义

每个 slot 必须定义：

- 找不到 id 时如何回退
- 接口不合法时是否拒绝注册
- 运行时失败时是否中断行为链
- 是否向 `DebugService.record_protocol_issue()` 报告

默认规则：

```text
注册失败：拒绝注册，记录 protocol issue
运行时查无 id：记录 protocol issue，使用安全默认值或取消本次行为
策略执行失败：不崩溃主循环，记录 failure note
```

## Contributor 资源规范

所有可资源化的贡献项都应遵循同一类结构。

通用字段：

```text
contributor_id
slot_id
namespace
display_name
tags
param_defs
strategy_script 或 strategy_resource
source_pack_id
minimum_engine_version
```

具体 slot 可以增加字段。

### ProjectileMovementDef 建议形态

```text
ProjectileMovementDef
  move_mode: StringName
  movement_script: Script
  param_defs: Array[Dictionary]
  default_hit_strategy: StringName
  default_terminal_hit_strategy: StringName
  tags: PackedStringArray
```

示例：

```text
move_mode = "my_pack.zigzag"
movement_script = res://extensions/my_pack/scripts/projectile/projectile_movement_zigzag.gd
param_defs = [
  { name = "amplitude", type = "float", min = 0.0, max = 200.0, default = 24.0 },
  { name = "frequency", type = "float", min = 0.1, max = 20.0, default = 4.0 }
]
default_hit_strategy = "swept_segment"
```

### MechanicCompilerDef 建议形态

```text
MechanicCompilerDef
  type_id: StringName
  family: StringName
  compiler_script: Script
  param_defs: Array[Dictionary]
```

约束：

- `family` 必须属于 10 个冻结 family
- 扩展包不得新增一级 family
- `type_id` 必须使用 `<pack_id>.*`
- compiler 输出必须是 `Dictionary`

## Registry 统一接口

所有 slot registry 应尽量具备同一组基础方法。

```gdscript
func register_def(def: Resource, source: Dictionary = {}) -> bool
func has(id: StringName) -> bool
func get_def(id: StringName) -> Resource
func get_strategy(id: StringName) -> Callable
func list_ids() -> PackedStringArray
func rebuild_registry() -> void
```

如果某个 slot 的策略不是 `Callable`，可以提供等价方法：

```gdscript
func create_component(id: StringName) -> Node
func compile_type(id: StringName, mechanic, archetype, merged_params: Dictionary) -> Dictionary
func process(id: StringName, args...) -> Variant
```

但注册、查询、重建、列举、校验这几个动作不应各自发明新名字。

## 命名空间与覆盖规则

默认规则：

```text
core.*       主仓保留
<pack_id>.*  扩展包私有
```

扩展包不得：

- 注册 `core.*`
- 覆盖已有 `core.*`
- 注册与其他已启用扩展包冲突的 id
- 使用空 namespace

后续如果确实需要覆盖，应另设显式 override 机制，并满足：

- manifest 明确声明 override
- 被覆盖对象允许 override
- 依赖图中显式依赖被覆盖包
- guardrail 验证覆盖顺序

在当前草案阶段，默认不开放覆盖。

## 与 Mechanic-first 的关系

通用插槽不改变 Mechanic-first 主链：

```text
CombatArchetype + CombatMechanic[]
-> NormalizedMechanicSet
-> RuntimeSpec
-> EntityFactory
```

它补的是“type 背后的运行时能力如何扩展”。

例如 `Trajectory`：

```text
CombatMechanic:
  family = Trajectory
  type_id = my_pack.zigzag

MechanicCompiler:
  my_pack.zigzag -> movement_mode = my_pack.zigzag

ProjectileMovementRegistry:
  my_pack.zigzag -> ProjectileMovementZigzag
```

这里 `Trajectory` family 没有新增，新增的是 family 下的 type 和对应运行时策略。

## ProjectileMovement 插槽建议

### 当前问题

当前 projectile movement 的两个关键点仍偏硬编码：

1. `ProjectileRoot` 根据 `move_mode` 选择具体 movement script。
2. `spawn_projectile` 的 `movement_mode` 协议选项只列出 `linear / track / parabola`。

这导致扩展包无法自然新增：

```text
my_pack.zigzag
my_pack.boomerang
my_pack.spiral
```

### 目标结构

建议新增：

```text
scripts/core/defs/projectile_movement_def.gd
autoload/ProjectileMovementRegistry.gd
```

内置能力也通过 registry 注册：

```text
core.linear   -> ProjectileMovementLinear
core.parabola -> ProjectileMovementParabola
core.track    -> ProjectileMovementTrack
```

为了兼容现有资源，短期可保留别名：

```text
linear   -> core.linear
parabola -> core.parabola
track    -> core.track
```

### 运行时分发

`ProjectileRoot` 不再直接 `match move_mode`，而是：

```text
move_mode
-> ProjectileMovementRegistry.get_def(move_mode)
-> create movement component
-> configure_movement(params)
```

如果找不到：

```text
记录 protocol issue
回退到 core.linear
```

### 协议校验

`spawn_projectile.movement_mode` 不应继续写死为固定 options。

可选演进：

1. 短期：允许任意 `StringName`，运行时 registry 检查。
2. 中期：ProtocolValidator 查询 `ProjectileMovementRegistry.has(move_mode)`。
3. 长期：编辑器工具从 registry 动态生成可选项。

## 扩展包加载顺序

通用插槽加载应遵循稳定顺序：

```text
1. 读取 extension manifest
2. 解析 pack_id / enabled / trust_level / register
3. 扫描资源内容
4. 注册 contributor defs
5. 注册 strategy scripts
6. 重建依赖它们的内容索引
7. 运行 guardrail 或 smoke validation
```

关键点：

- 不允许扩展包在任意时机自行修改 registry
- registry rebuild 必须可重复执行
- 禁用扩展包后 registry 应能清理对应贡献项
- DebugService 应能报告贡献项来源路径

---

## Guardrail 验证要求

每个新 slot 至少需要 4 类验证。

### 1. Smoke

合法扩展项可以注册并运行。

示例：

```text
my_pack.zigzag projectile spawned
projectile.spawned.core.move_mode == my_pack.zigzag
projectile.hit emitted
```

### 2. Duplicate

重复 id 被拒绝。

示例：

```text
两个扩展包都注册 my_pack.zigzag -> 后者失败
扩展包注册 core.linear -> 失败
```

### 3. Interface

策略接口不合法时拒绝注册。

示例：

```text
movement_script 未继承 ProjectileMovementBase -> 失败
effect strategy 缺少 execute(context, params, node) -> 失败
```

### 4. Trust

低信任包不能注册运行时代码。

示例：

```text
data_only pack 携带 movement_script -> 失败
rule_extended pack 注册 trusted_runtime slot -> 失败
```

## 分阶段实施建议

### 第一阶段：固定规范并补 ProjectileMovement

目标：

- 新增 `ProjectileMovementDef`
- 新增 `ProjectileMovementRegistry`
- 内置 `linear / parabola / track` 走 registry
- 扩展包示例注册 `my_pack.zigzag`
- 增加 smoke 和 guardrail 验证

不做：

- 不重构全部 projectile 命中系统
- 不一次性迁移所有 registry
- 不开放覆盖 `core.*`

### 第二阶段：MechanicCompiler 扩展接入

目标：

- 资源化 `MechanicCompilerDef`
- 扩展包可注册现有 family 下的新 `type_id`
- compiler 输出仍为 `Dictionary`
- 禁止扩展包新增 family

### 第三阶段：统一 Registry 行为

目标：

- 对齐 `EffectRegistry / ControllerRegistry / DetectionRegistry / TriggerRegistry`
- 统一 `register_def / rebuild_registry / list_ids`
- 统一 DebugService 协议问题格式

### 第四阶段：manifest 能力声明

目标：

- manifest 显式声明 `capabilities`
- manifest 显式声明 `trust_level`
- 加载器按 slot 权限决定是否扫描和注册

## 设计原则

### KISS

先补最小稳定插槽，不引入通用插件 VM、复杂热重载或安装器 UI。

### YAGNI

当前不开放覆盖 `core.*`，不设计复杂冲突仲裁，先以拒绝冲突为主。

### DRY

注册、查询、重建、重复 id 检查、命名空间检查应沉淀成统一模式，避免每个 registry 复制一套不同规则。

### SOLID

运行时依赖 slot 抽象和 registry，不依赖具体扩展实现。扩展能力通过新增贡献项开放，而不是修改主链分支。

---

## 决策边界

这份草案固定以下边界：

1. 通用插槽是后续扩展系统的统一抽象。
2. 扩展包默认只能新增现有 family 下的 type，不能新增 Mechanic family。
3. `core.*` 命名空间属于主仓，扩展包默认不得覆盖。
4. 新运行时语义必须通过明确 slot 注册，不应通过主链特判扩展包 id。
5. `ProjectileMovement` 是当前最需要补齐通用插槽的子系统。
6. 每个 slot 必须配套 smoke、duplicate、interface、trust guardrail。

未决问题：

1. contributor def 是否统一继承一个基类。
2. manifest 的 `trust_level` 具体字段名和默认值。
3. 禁用扩展包后的 registry 清理策略。
4. 编辑器内动态 options 如何从 registry 暴露。
5. 多扩展包依赖顺序和 override 是否在 v1 开放。

## 提升到正式 wiki 的条件

这份草案应在满足以下条件后再提升为正式 wiki 文档：

1. 至少一个新 slot 按本文模型落地。
2. 至少一个扩展包通过该 slot 注册运行时代码。
3. 对应 smoke 和 guardrail 验证进入批量回归。
4. 实际代码中的字段名、接口签名和本文一致。
5. ADR 或正式 wiki 已确认 trust level 与 namespace 规则。
