# 汉字图像默认视觉身份系统草案

> 日期：2026-05-04
> 状态：概念草案，供后续 spike 与设计评审使用
> 关联主题：Mechanic-first 编译链、视觉表现层、素材包系统、程序化默认美术

---

## 文档定位

本文记录一个早期方向：为每个最终编译后的植物模板生成一个默认名字与一张汉字风格图像。图像由植物的突出规则特征半自动组合而来，作为无正式美术时的默认视觉身份。

本文不作为当前实现依据，不修改冻结协议，不要求立刻进入主干。它的目标是为未来落地提供足够清晰的边界、输入输出、特征提取思路和风险清单。

本文讨论的是：

- 如何从 `RuntimeSpec` 提取足够突出的最终编译特征
- 如何把特征映射为汉字部件与结构
- 如何通过外部 Kage 工具链生成 SVG/PNG
- 如何接入 Godot 的默认 visual/icon 回退链
- 如何保留人工 override 与内容作者控制权

本文不讨论：

- 完整汉字造字语法
- Kage 引擎源码移植
- 正式素材包 manifest 最终规范
- 正式动画系统
- 与原版 PVZ 视觉资源的迁移

---

## 一句话结论

汉字图像系统可以作为 Open PVZ 的“程序化默认视觉身份层”：规则由 Mechanic-first 编译链生成行为，视觉身份由同一份最终运行时事实提取突出特征后生成图像。

它不应替代正式美术，也不应成为运行时规则的一部分。更合理的定位是：

```text
CombatArchetype + CombatMechanic[]
-> MechanicCompiler
-> RuntimeSpec
-> Visual Feature Extractor
-> Glyph Recipe
-> External Kage Tool
-> SVG/PNG Default Visual
```

## 设计动机

Open PVZ 的核心是组合规则与涌现式玩法。传统内容管线里，新增一个植物通常需要同步准备名字、卡片图标、战场形象、图鉴图等资产。对规则引擎项目来说，视觉资产会很容易成为内容扩展速度的瓶颈。

汉字图像系统尝试把这个瓶颈变成系统能力：

1. 组合规则生成行为。
2. 突出行为特征生成视觉语义。
3. 视觉语义组合成汉字风格图像。
4. 正式美术缺失时仍有统一、可读、可替换的默认表现。

这套系统的价值不在于精确表达每个机制，而在于用少量稳定视觉部件表达最重要的身份信息。例如：

- 射手 + 豌豆 + 冰 = 冰豌豆射手
- 爆破 + 火 + 整行 = 火焰行爆植物
- 生产 + 阳光 + 周期 = 阳光生产植物
- 防御 + 高生命 + 覆盖 = 坚固覆盖植物

## 核心原则

### 1. 展示层独立

汉字图像系统只消费编译结果，不反向影响规则。

不得因为字形系统新增 Mechanic family、修改效果语义、改变编译协议。

### 2. 以最终编译事实为准

特征提取应优先读取 `RuntimeSpec`，而不是直接根据原始 `.tres` 资源猜测。原因是同一个 mechanic 在不同组合上下文中可能产生不同运行时意义。

优先级：

```text
RuntimeSpec 字段
> 编译后的 EffectNode / controller_specs / state_specs / placement_spec
> tags / archetype_id / display_name
> 人工 override
```

### 3. 显著特征优先

系统不追求包含全部意义，只提取最容易被玩家识别的 2 到 4 个特征。

推荐上限：

- 1 个主职业特征
- 1 个核心输出物或核心元素
- 0 到 1 个范围/结构特征
- 0 到 1 个生境/触发/状态修饰

### 4. 离线生成优先

Kage 作为外部工具链调用，不内嵌 Godot 运行时。Godot 运行时只加载已经生成好的 SVG/PNG 或 Godot 导入纹理。

### 5. 人工 override 始终优先

自动结果必须可被内容作者覆盖。程序化图像是默认值，不是权威定义。

---

## 版权与许可证边界

当前可用的 Python Kage 实现可作为外部工具评估，但不建议基于其源码移植到 Godot。

原因：

- 基于 GPLv3 源码重写、翻译、移植通常仍可能被视为派生作品。
- 将 GPLv3 代码嵌入主仓或运行时会引入许可证兼容风险。
- 外部开发期工具的风险边界通常更清晰，但仍需要后续许可证评估。

建议策略：

1. 第一阶段仅把 Kage 作为开发期外部命令调用。
2. 主仓不复制 Kage 源码。
3. 生成资产、GlyphWiki 数据来源、工具分发方式分别评估授权。
4. 若未来必须内置造字能力，采用 clean-room 重实现，而不是源码移植。

## 特征提取模型

### FeatureSignal

特征提取器内部可以使用统一结构：

```text
FeatureSignal
- id: String
- group: String
- score: int
- source: String
- evidence: Dictionary
```

示例：

```json
{
  "id": "role.shooter",
  "group": "role",
  "score": 100,
  "source": "trigger_specs[].effect_root.effect_id",
  "evidence": {
    "effect_id": "spawn_projectile"
  }
}
```

### 特征分组

推荐分组：

| 分组 | 作用 | 示例 |
|------|------|------|
| `role` | 决定主体身份 | shooter, bomber, producer, wall, summoner, melee, controller |
| `payload` | 决定核心输出物 | pea, melon, cabbage, spore, bone, spike, sun |
| `element` | 决定元素修饰 | fire, frost, poison, light, magnet, sleep, hypnosis |
| `range` | 决定结构倾向 | lane, radius, pierce, multi_lane, spread, tracking |
| `trigger` | 决定触发修饰 | periodic, on_place, on_death, when_damaged, proximity |
| `placement` | 决定生境/底部 | ground, water, roof, air, cover |
| `state` | 决定状态标记 | arming, sleeping, growth, rage |
| `shape` | 决定体型/防御倾向 | tall, small, bulky, fragile |

### 主职业识别

主职业应优先由编译后的效果与控制器决定。

| 条件 | 特征 | 分数建议 |
|------|------|----------|
| `effect_root.effect_id == "spawn_projectile"` | `role.shooter` | 100 |
| `effect_root.effect_id == "explode"` | `role.bomber` | 100 |
| `effect_root.effect_id == "produce_sun"` | `role.producer` | 100 |
| `effect_root.effect_id == "spawn_entity"` | `role.summoner` | 95 |
| `controller_id == "core.bite"` | `role.melee` | 95 |
| `controller_id == "core.sweep"` | `role.sweeper` | 90 |
| `effect_root.effect_id == "apply_status"` | `role.controller` | 85 |
| 高生命且缺少攻击效果 | `role.wall` | 85 |

高生命防御的初始判断可以简单一些：

```text
max_health >= 600
and no strong attack effect
-> role.wall
```

后续可按实际植物数值校准。

### 输出物识别

输出物优先从 `projectile_template`、`effect_root.params`、`tags` 中识别。

| 证据 | 特征 |
|------|------|
| projectile path/name 包含 `pea` | `payload.pea` |
| 包含 `melon` | `payload.melon` |
| 包含 `cabbage` | `payload.cabbage` |
| 包含 `spore` | `payload.spore` |
| 包含 `bone` | `payload.bone` |
| 包含 `spike` | `payload.spike` |
| `effect_id == "produce_sun"` | `payload.sun` |
| `effect_id == "damage"` 且无 projectile | `payload.direct_damage` |

输出物通常比伤害数值更适合进入字形。

### 元素与状态识别

元素可从以下位置提取：

- `on_hit` 子效果
- `effect_root.effect_id`
- `effect_root.params.status_id`
- `tags`
- `projectile_template` 名称
- `state_specs`

示例规则：

| 证据 | 特征 |
|------|------|
| status/tag/projectile 包含 `frost` / `ice` | `element.frost` |
| tag/effect 包含 `fire` / `explode` / `jalapeno` | `element.fire` |
| effect/tag 包含 `hypno` / `team_switch` | `element.hypnosis` |
| tag/effect 包含 `magnet` | `element.magnet` |
| `state_specs.initial_state == "sleeping"` | `state.sleeping` |
| `state_specs.initial_state == "arming"` | `state.arming` |
| `state_specs.initial_state == "small"` 且存在 growth transition | `state.growth` |

元素只建议保留最高分的一个，避免字形拥挤。

### 范围与结构识别

范围特征更适合影响字形布局，而不是占用主体部件。

| 证据 | 特征 | 可能布局影响 |
|------|------|--------------|
| `detection_id == "lane_forward"` | `range.lane_forward` | 左右结构，朝向标记 |
| `target_mode == "enemies_in_lane"` | `range.lane` | 横向结构 |
| `target_mode == "enemies_in_radius"` 或 radius 显著 | `range.radius` | 包围/中心放射 |
| hit policy 为 pierce | `range.pierce` | 贯穿笔画 |
| emission 为 multi_lane | `range.multi_lane` | 三列/多横结构 |
| emission 为 spread | `range.spread` | 分叉结构 |
| trajectory 为 track/global_track | `range.tracking` | 回旋/钩形标记 |

### 触发方式识别

触发方式作为低优先级修饰。

| event/trigger | 特征 |
|---------------|------|
| `game.tick` + `periodically` | `trigger.periodic` |
| `placement.accepted` | `trigger.on_place` |
| `entity.died` | `trigger.on_death` |
| `entity.damaged` | `trigger.when_damaged` |
| `proximity` | `trigger.proximity` |

通常只有 `on_place`、`on_death`、`when_damaged` 这类强身份触发值得进入图像。

### 放置与生境识别

放置特征从 `placement_spec.slot_type_hint` 和 placement tags 提取。

| slot/tag | 特征 |
|----------|------|
| ground | `placement.ground` |
| water | `placement.water` |
| roof | `placement.roof` |
| air | `placement.air` |
| cover role / cover tags | `placement.cover` |

这类特征适合做底部、水旁、屋顶或外框，不宜抢主体。

## 特征选择策略

推荐采用简单稳定的选择规则。

```text
1. 收集所有 FeatureSignal。
2. 按 group 内分数降序去重。
3. 选择最高分 role，若无 role 则从 tags 兜底。
4. 选择最高分 payload 或 element。
5. 选择最高分 range。
6. 选择最高分 placement/state/trigger 中最有辨识度的一项。
7. 总数超过 4 时丢弃低优先级项。
```

建议优先级：

```text
role
> payload
> element
> range
> state
> placement
> trigger
> shape
```

例外：

- `on_place + explode` 的一次性植物，`trigger.on_place` 可以保留。
- `sleeping` 对蘑菇身份强，可保留。
- `water` 对水生植物身份强，可保留。
- `wall` 植物的 shape/health 可上升为 role。

## Glyph Recipe

### 目标

`glyph_recipe` 是 Kage 工具消费的中间描述。它不需要暴露给运行时，也不需要完整表达全部视觉细节。

### 推荐结构

```json
{
  "layout": "left_right",
  "primary": "bow",
  "secondary": "pea",
  "marks": ["frost"],
  "style": {
    "stroke_weight": "normal",
    "density": "medium"
  }
}
```

### 布局枚举

第一版只支持少量结构：

| layout | 用途 |
|--------|------|
| `left_right` | 射手 + 输出物，控制 + 状态 |
| `top_bottom` | 生产、生长、蓄力 |
| `surround` | 防御、范围爆炸、覆盖 |
| `center_mark` | 单一强身份 + 元素点缀 |

### 部件映射建议

以下只是初始方向，不是最终字源规范。

| 特征 | 部件意象 |
|------|----------|
| `role.shooter` | 弓、矢 |
| `role.bomber` | 火、爆、灬 |
| `role.producer` | 日、光、生 |
| `role.wall` | 囗、土、盾意象 |
| `role.summoner` | 子、生、召意象 |
| `role.melee` | 口、牙、刃意象 |
| `payload.pea` | 豆 |
| `payload.melon` | 瓜 |
| `payload.cabbage` | 艹、菜意象 |
| `payload.spore` | 菌、孢意象 |
| `payload.sun` | 日、光 |
| `element.frost` | 冫 |
| `element.fire` | 火、灬 |
| `placement.water` | 氵 |
| `placement.air` | 羽 |
| `range.lane` | 一、川、行意象 |
| `range.radius` | 囗、回、环意象 |

## 命名系统

命名可以和图像使用同一套特征，但不必完全一致。

### 默认命名思路

```text
[元素/范围修饰] + [输出物/身份] + [职业]
```

示例：

- 冰 + 豆 + 射手
- 火 + 行 + 爆
- 日 + 生
- 水 + 菇 + 射手

### 命名边界

第一版只建议生成内部默认中文名或 debug 名，不要替换已有 `display_name`。

更安全的落点：

- `generated_display_name`
- `visual_identity.display_name`
- 或 manifest 中的 `suggested_name`

已有人工 `display_name` 始终优先。

## 工具链方案

### 阶段 A：Godot 导出编译特征

新增 headless 工具脚本，负责：

1. 扫描植物 archetype。
2. 通过现有 `MechanicCompiler` 得到 `RuntimeSpec`。
3. 提取 `FeatureSignal`。
4. 写出 `runtime_visual_features.json`。

该阶段必须在 Godot 侧完成，避免 Python 重写 `.tres` 解析与编译逻辑。

### 阶段 B：Python 生成字形资产

Python 工具负责：

1. 读取 `runtime_visual_features.json`。
2. 应用特征选择规则。
3. 生成 `glyph_recipe`。
4. 调用外部 Kage 工具。
5. 输出 SVG/PNG。
6. 写出资产 manifest。

### 阶段 C：Godot 加载默认图像

后续视觉层或 AssetRegistry 可按以下回退链解析：

```text
正式素材包 override
-> archetype 指定 visual/icon
-> generated glyph asset
-> DebugViewComponent / 占位绘制
```

## 与现有系统的关系

### 与 Mechanic-first 的关系

只读 `RuntimeSpec`，不参与编译决策。

### 与素材包系统的关系

生成的汉字图像可以视为主仓默认 asset pack 的一部分，或作为生成资产缓存。正式素材包可以覆盖这些默认图像。

### 与视觉表现层的关系

汉字图像是视觉输入之一，可用于：

- 卡片 icon
- 图鉴占位图
- 战场默认 visual
- debug overlay
- 内容编辑器预览

它不是动画系统，也不要求实体视觉组件立即完整落地。

### 与扩展系统的关系

扩展包中的新植物也可以走同一生成链，只要它们能被现有编译器编译为 `RuntimeSpec`。

## 人工 override

自动生成必须允许局部覆盖。

### override 粒度

1. 覆盖名称
2. 覆盖特征选择
3. 覆盖 glyph recipe
4. 覆盖最终资产路径

### 示例

```json
{
  "archetype_original_snowpea": {
    "suggested_name": "冰豆射手",
    "features": ["role.shooter", "payload.pea", "element.frost"],
    "glyph_recipe": {
      "layout": "left_right",
      "primary": "bow",
      "secondary": "pea",
      "marks": ["frost"]
    }
  }
}
```

## 验证策略

第一阶段不需要复杂测试框架，建议用小规模黄金样本验证。

### 样本集

初始选 10 到 15 个植物：

- Peashooter
- Snow Pea
- Repeater
- Sunflower
- Wall-nut
- Cherry Bomb
- Jalapeno
- Cabbage-pult
- Melon-pult
- Puff-shroom
- Chomper
- Pumpkin
- Lily Pad
- Starfruit
- Spikeweed

### 验证目标

1. 每个样本都能导出 `FeatureSignal`。
2. 每个样本至少有 1 个主职业特征。
3. 每个样本生成的 `glyph_recipe` 不超过 4 个语义部件。
4. 生成资产路径稳定。
5. 人工 override 能覆盖自动结果。
6. 小尺寸预览仍可辨识。

### 视觉 QA

建议导出一张 overview sheet：

```text
archetype_id | display_name | selected features | generated glyph
```

人工检查优先于自动像素检查。

## 风险

### 1. 字形可读性不足

组合过多会导致小尺寸不可读。

缓解：

- 限制部件数量。
- 限制布局数量。
- 自动检测过高笔画密度。
- 提供人工 override。

### 2. 语义映射不稳定

同一 mechanic 在不同组合中可能意义不同。

缓解：

- 以 `RuntimeSpec` 为准。
- 特征规则只识别突出行为。
- 不承诺完整语义还原。

### 3. 审美风格不统一

不同部件组合可能视觉风格割裂。

缓解：

- 固定 Kage 字形风格参数。
- 建立部件库审查清单。
- 每轮只扩展少量新部件。

### 4. 许可证边界不清

外部 Kage 工具、GlyphWiki 数据、生成资产授权都需要确认。

缓解：

- 不复制 GPL 源码进主仓。
- 只做外部工具调用。
- 记录工具版本与输入来源。
- 进入发布阶段前做许可证审查。

### 5. 被误用为正式美术替代

程序化默认图像可能遮蔽正式视觉管线需求。

缓解：

- 文档明确其定位是默认视觉身份和占位资产。
- 正式素材包 override 始终优先。
- 不把生成图像写死到内容规则中。

## 分阶段落地建议

### P0：纸面与样本确认

- 确认 10 到 15 个样本植物。
- 手工写出期望特征与 glyph recipe。
- 验证部件映射是否有辨识度。

### P1：RuntimeSpec 特征导出

- 新增 Godot headless 脚本导出 runtime visual feature manifest。
- 只支持植物 archetype。
- 只读取现有编译结果，不改协议。

### P2：Python 生成链 spike

- Python 读取 manifest。
- 应用特征选择规则。
- 调外部 Kage 生成 SVG。
- 转 PNG。
- 生成 overview sheet。

### P3：Godot 预览接入

- 在卡片栏、debug visual 或图鉴原型中显示生成图像。
- 验证 fallback 链。
- 加入人工 override。

### P4：提升为正式设计

满足以下条件后，才考虑从 `plans/draft` 提升到 wiki：

1. 至少一个生成链脚本进入主仓。
2. 至少 10 个植物样本可生成稳定图像。
3. 视觉回退链有实际 Godot 入口。
4. override 流程可用。
5. 许可证边界完成初步确认。
