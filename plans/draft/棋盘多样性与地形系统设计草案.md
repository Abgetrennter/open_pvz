## 背景与目标

### 为什么需要棋盘多样性

当前 Open PVZ 的棋盘抽象层支持 `ground` / `water` / `roof` / `air_only` 四种槽位类型，以及覆盖 placed slot_config 的 per-slot 覆盖能力。但运行时棋盘构造仍然以“全 ground”为默认值：`_rebuild_slots()` 对所有 lane 的所有 slot 统一赋值 `slot_type = &"ground"`，然后通过 `_apply_slot_configs()` 覆盖个别格子。

这种设计对“全草地关卡”足够，但对以下场景缺乏支持：

- **泳池关卡**：第 1 行和第 6 行是草地，第 2-5 行是泳池。泳池行需要 `water` 槽位类型，但只有一小部分槽位允许放荷叶。
- **屋顶关卡**：所有行是屋顶地形，同行的地面高度随 X 位置变化，投射物需要按地形高度解释飞行高度。
- **阶梯/高台地形**：同一行不同格子可能有离散高度，投射物和视觉投影需要统一查询地面高度。
- **墓地地形**：部分格子被墓碑占据，不可种植，僵尸可从墓碑生成。完整 GridItem 系统另立设计，本文只保留接口边界。

夜晚、迷雾、晴天、阴天和烈日不属于棋盘结构。它们是环境状态，应与棋盘正交组合。

### 当前状态评估

现有系统已经具备的基础：

- 4 种 slot type (`ground` / `water` / `roof` / `air_only`) 和对应的默认标签
- 5 种放置角色 (`primary` / `support` / `surface` / `cover` / `blocker` / `upgrade`)
- Surface / substrate 模式 (`water_pod`, `flower_pot_surface`)
- 8 项放置验证链 (`validate_request`)
- 5 级 `HeightBand`
- `BattlefieldPreset + BattlefieldMetrics` 骨架
- `BoardSlotConfig` 支持 per-slot 覆盖 (`slot_type`, `placement_tags`)
- `BattleRuleModule` 和 `BattleModeHost` 已能承载正交的模式/环境行为

缺失的关键能力：

- **行级元数据**：没有 per-row 的地形类型声明，所有行从同一个 `_rebuild_slots()` 流程生成。
- **行级 Y 坐标配置**：lane Y 仍由 `BattleManager.get_lane_y()` 作为最终事实源，不是由预设数据驱动。
- **地形高度采样**：没有统一的 `lane + x -> terrain elevation` 查询，Roof、小山、阶梯或未来地形变化只能落成特例。
- **生成入口未解耦**：`WaveRunner` 直接按 wave entry 生成，当前缺少独立的 `SpawnZone` / `BattleSpawnResolver` 来匹配生成入口能力与僵尸生成需求。
- **行级视觉主题**：泳池行、草地行、屋顶行应有不同的视觉表现。

## 当前抽象层

### 槽位类型与标签

当前系统通过 `BoardSlotCatalog` 定义 4 种槽位类型及其默认标签：

| 槽位类型 | 默认标签 | 用途 |
|----------|----------|------|
| `ground` | `["ground", "supports_primary"]` | 标准草地格子 |
| `water` | `["water"]` | 水面格子，需荷叶等 substrate |
| `roof` | `["roof"]` | 屋顶格子 |
| `air_only` | `["air_only", "supports_air"]` | 仅空中单位可放置 |

### 放置角色系统

5 种放置角色用于 slot 占位管理：

| 角色 | 用途 |
|------|------|
| `primary` | 主力植物，常规格子 |
| `support` | 支援装饰，如南瓜壳内层 |
| `surface` | 水面/表面部署，如荷叶、花盆 |
| `cover` | 保护覆盖，如南瓜壳 |
| `blocker` | 阻挡角色 |
| `upgrade` | 升级替换 |

### Surface/substrate 模式

现有系统支持 surface 角色作为“基底”：荷叶 (`water_pod`) 占用 `surface` 角色后将 `water` 标签改为 `ground + supports_primary`，使主力植物可在水面放置。

### 放置验证链

`validate_request()` 按顺序检查 8 项约束：

1. request 非空
2. lane 有效
3. slot_index 在范围内
4. slot 存在
5. placement_tags 非空
6. slot 标签匹配
7. 实体类型对应的限制
8. 角色占位冲突

### HeightBand 系统

5 个高度带用于投射物命中判定：`low / mid_low / mid / mid_high / high`。实体声明 `hit_height_range`，投射物声明飞行高度，通过 overlap 检测。

### 投射物地面位置与高度

当前投射物系统已经具备 3D 逻辑雏形：`ProjectileRoot` 保存 `ground_position` 与 `height`，再通过 `global_position = ground_position + Vector2(0, -height * projection_scale)` 映射到 2D 渲染位置。

现有限制是：`height` 目前等价于“相对默认平地的飞行高度”，没有纳入地形表面的高度。若 Roof、小山或阶梯改变了某个 `lane + x` 的地面高度，投射物命中和视觉投影需要比较：

```text
absolute_z = terrain_elevation_at(lane, x) + height_above_ground
```

因此后续不应将屋顶修正写成 `roof_arc_compensation` 特例，而应将 `height` 明确定义为“离地高度”，由 `BattlefieldMetrics` 提供地形高度采样。

### BattlefieldPreset + BattlefieldMetrics

`BattlefieldPreset` 是棋盘数据入口 Resource，声明棋盘基础参数：

```gdscript
@export var lane_count := 2
@export var board_slot_count := 5
@export var board_slot_origin_x := 160.0
@export var board_slot_spacing := 96.0
@export var board_slot_configs: Array = []
```

`BattlefieldMetrics` 是运行时只读值对象，提供 `slot_position()`、`slots_to_world()`、`world_to_slot_index()`、`resolve_range()` 等几何换算。

### 缺失项总结

- **行级类型声明**：没有 lane-level 的数据结构描述该行是草地、泳池、屋顶还是 dirt。
- **行级高度配置**：lane Y 坐标未数据化，Roof 坡度、小山或阶梯式地形无法通过统一采样表达。
- **生成入口语义**：地形系统尚未暴露稳定的 terrain traits，生成系统也缺少独立的 spawn zone / spawn requirement 匹配层。
- **环境元数据外移**：`is_night`、`has_fog`、`weather` 等不应放进棋盘预设；应由独立环境草稿定义。

## 原版 PVZ 参考

### 背景类型系统的历史绑定

原版 PVZ 使用 `BackgroundType` 枚举区分关卡大类型：

| 背景 | 棋盘属性 | 环境属性 |
|------|----------|----------|
| `DAY_GRASS` | 草地 | 白天 |
| `NIGHT_GRASS` | 草地 | 夜晚，可有墓碑 |
| `DAY_POOL` | 草地 + 泳池 | 白天 |
| `NIGHT_POOL` | 草地 + 泳池 | 夜晚 + 迷雾 |
| `DAY_ROOF` | 屋顶 | 白天 |
| `NIGHT_ROOF` | 屋顶 | 夜晚 |

Open PVZ 不应继续把这些维度绑死。本文只抽取棋盘维度：`has_pool`、`has_roof`、行级空间描述、格子类型和地形高度。白天/夜晚/迷雾/天气迁移到动态环境系统。

### 行类型 (PlantRowType)

`PlantRowType` 枚举定义每行的地形类型，保存在 `PlantRowType[]` 数组中：

| 行类型 | 含义 |
|--------|------|
| `NORMAL` | 普通草地行 |
| `POOL` | 泳池行，有水，需荷叶 |
| `DIRT` | 不可种植行 |
| `HIGH_GROUND` | 屋顶行，有坡度，需花盆 |

原版泳池关卡的行配置示例：

```text
[0] NORMAL
[1] POOL
[2] POOL
[3] POOL
[4] POOL
[5] DIRT
```

屋顶关卡的行配置示例：

```text
[0] HIGH_GROUND
[1] HIGH_GROUND
[2] HIGH_GROUND
[3] HIGH_GROUND
[4] HIGH_GROUND
```

### 格子类型 (GridSquareType)

`GridSquareType` 二维数组由 `PlantRowType + X 位置` 派生。部分格子可能有特殊标记，如泳池行最左侧 1-2 格不可通过荷叶种植。

### 泳池 (Pool)

泳池关卡的关键机制：

- 6 行布局：1 草地 + 4 泳池 + 1 dirt
- 泳池行需要荷叶 (`LilyPad`) 作为 surface 基底才能放植物
- 泳池行使用特定水生僵尸
- 泳池行视觉上为水面，草地行为草地
- dirt 行为不可种植区域

### 屋顶 (Roof)

屋顶关卡的关键机制：

- 所有行为 high ground，需要花盆 (`FlowerPot`) 作为基底
- 每行的地面高度随 X 增加而变化
- 投射物需要按地形高度解释飞行高度
- 部分原版屋顶关卡有高度限制，可作为后续碰撞边界设计

### 墓碑 (Grave)

墓碑机制：

- 墓碑是占用格子的静态物体，不可放置普通植物
- 墓碑可以从格子中生成僵尸
- 墓碑可被特定植物破坏
- 本文不实现完整 GridItem，只保留与棋盘 slot 的边界

### 位掩码地形系统参考

部分参考实现使用 `GridBitmask` 实现更精细的格子颗粒度：

- 每个格子有独立 bitmask
- 地形变化通过修改格子的 bitmask 实现
- 放置验证直接查询目标格子的 bitmask

当前方案不采用 bitmask 路线，而是复用现有 tag 系统。理由见“不做的事”。

## 设计方案

### 总体架构

棋盘多样性采用两层地形架构：

```text
[行/格子布局层]
  固定、数据驱动、战斗 setup 后不可变
  LaneConfig[] + BattlefieldPreset
  -> 决定 slot 类型、slot 标签、地形 trait、行视觉主题

[地形高度采样层]
  固定采样函数，未来可叠加 runtime modifier
  LaneTerrainProfile + BattlefieldMetrics
  -> 决定 lane + x 的地面高度、slot_position、ground_position_for
```

动态环境系统是第三条正交轴，不属于本文实现：

```text
BattleScenario / BattleModeDef
  -> battlefield_preset
  -> environment_profile
  -> rule_modules
```

棋盘系统决定“空间是什么”；环境系统决定“当前世界状态如何影响规则”。环境不直接改 slot，棋盘也不内置 sunny / night / fog。

### 方案 A: 行级地形描述层 (LaneConfig)

#### LaneConfig Resource

新增 `LaneConfig` Resource，声明单行的空间和地形配置。它不是规则层，不直接决定生成、环境或模式行为。

```gdscript
# scripts/battle/lane_config.gd (草案)
extends Resource
class_name LaneConfig

@export var lane_index := 0
@export var lane_type: StringName = &"grass"
@export var slot_type_override: StringName = StringName()
@export var base_tags_override: PackedStringArray = PackedStringArray()
@export var lane_traits: PackedStringArray = PackedStringArray()
@export var height_offset := 0.0
@export var slope_y_per_slot := 0.0
@export var terrain_profile: Resource = null
@export var visual_theme: StringName = StringName()
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `lane_index` | int | 行号，从 0 开始 |
| `lane_type` | StringName | 作者友好的默认模板 ID，如 `&"grass"` / `&"pool"` / `&"roof"` / `&"dirt"`；只用于展开默认字段 |
| `slot_type_override` | StringName | 非空时覆盖该行所有 slot 的 slot_type |
| `base_tags_override` | PackedStringArray | 非空时覆盖该行所有 slot 的默认标签 |
| `lane_traits` | PackedStringArray | 行的地形特征，如 `terrain.grass` / `surface.water` / `terrain.high_ground`，供生成、视觉、环境查询消费 |
| `height_offset` | float | 该行相对于基准 Y 的整体视觉偏移量；第一阶段可作为简化字段 |
| `slope_y_per_slot` | float | 同一行随 slot_index 增加的视觉 Y 偏移量；后续优先由 `terrain_profile` 表达 |
| `terrain_profile` | Resource | 可选 `LaneTerrainProfile`，描述该行的地形高度采样规则 |
| `visual_theme` | StringName | 视觉主题标识，如 `&"pool_water"` / `&"roof_tile"` |

关键设计决策：`lane_type` 降级为 `LaneTypeDef` / 内置模板 shorthand，而不是运行时逻辑判断依据。内容加载或校验阶段可以用它填充默认 `slot_type_override`、`base_tags_override`、`lane_traits`、`visual_theme` 和默认 `terrain_profile`。运行时系统消费展开后的字段，不写 `if lane_type == &"pool"` 之类分支。

核心仍必须提供常量和 `ProtocolValidator` 校验，拼写错误不能静默退回 ground。

`lane_traits` 只描述地形特征，不表达“允许哪些僵尸生成”。生成系统可以消费这些 trait 派生默认生成入口，但匹配逻辑不放在 `LaneConfig` 中。

`height_offset` / `slope_y_per_slot` 可以保留为 Phase 1 的轻量兼容字段，但 Roof、小山、阶梯与未来地形变动不应继续扩展这一组字段。复杂高度语义统一进入 `terrain_profile`，由 `BattlefieldMetrics` 提供采样查询。

#### BattlefieldPreset 增强

在现有 `BattlefieldPreset` 基础上新增棋盘字段：

```gdscript
@export var lane_configs: Array[Resource] = [] # LaneConfig[]
@export var lane_y_positions: Array[float] = []
@export var lane_origin_y := 0.0
@export var lane_spacing := 0.0
```

不在 `BattlefieldPreset` 中新增 `is_night`、`has_fog`、`weather`、`sun_drop_scale` 等环境字段。这些由 `BattleEnvironmentProfile` 或对应 rule module 负责。

#### 消费端: BattleBoardState._rebuild_slots()

`_rebuild_slots()` 改为读取 `lane_configs` 数据：

```gdscript
func _rebuild_slots() -> void:
    var lane_ids = battle.get_lane_ids()
    for lane_id in lane_ids:
        var lane_config = _resolve_lane_config(lane_id)
        for slot_index in range(board_slot_count):
            var slot = BoardSlot.new()
            var slot_type = lane_config.slot_type_override if lane_config.slot_type_override != StringName() else &"ground"
            var base_tags = lane_config.base_tags_override if not lane_config.base_tags_override.is_empty() else BoardSlotCatalog.default_tags_for(slot_type)
            var world_pos = _compute_slot_position(lane_id, slot_index, lane_config)
            slot.configure(lane_id, slot_index, world_pos, slot_type, base_tags)
            _slots[_slot_key(lane_id, slot_index)] = slot
    _apply_slot_configs()
```

关键变更：

- 行级 `slot_type_override` 和 `base_tags_override` 在 slot 层次先于 `BoardSlotConfig`。
- `BoardSlotConfig` 的 per-slot 覆盖仍然可以覆盖特定格子。
- 行级高度配置进入 `BattlefieldMetrics`，不让 `BattleBoardState` 直接理解地形曲线。

#### LaneTypeDef 内置模板

虽然 `lane_type` 使用 `StringName`，但它只引用核心内置的 `LaneTypeDef` 模板。模板用于减少内容重复，不作为运行时分支条件：

| lane_type | slot_type_override | base_tags_override | lane_traits | 说明 |
|-----------|--------------------|--------------------|-------------|------|
| `grass` | `ground` | `["ground", "supports_primary"]` | `["terrain.grass", "surface.ground"]` | 标准草地 |
| `pool` | `water` | `["water"]` | `["terrain.pool", "surface.water"]` | 泳池水面 |
| `roof` | `roof` | `["roof"]` | `["terrain.roof", "terrain.high_ground", "surface.roof"]` | 屋顶 |
| `dirt` | `dirt` | `[]` | `["terrain.dirt", "placement.blocked"]` | 不可种植行 |

模板展开规则：

```text
LaneTypeDef 默认值
  -> LaneConfig 显式字段覆盖
  -> BoardSlotConfig per-slot 覆盖
```

运行时事实源：

```text
slot_type / base_tags / lane_traits / terrain_profile / visual_theme
```

`lane_type` 本身只保留在 debug snapshot、protocol issue 和编辑器提示中，方便内容作者定位来源。

dirt 行的处理方式：给 `slot_type = &"dirt"`、`base_tags = []`。放置验证时因为无法满足 `placement_tags` 匹配，所有普通植物不可放置。不需要额外的 dirt 逻辑分支。

### 方案 B: 生成入口层边界 (SpawnZoneConfig + BattleSpawnResolver)

地形系统不直接决定哪些僵尸可以生成。更干净的边界是：

```text
LaneConfig:
  描述地形类型、slot、trait、高度和视觉主题

SpawnZoneConfig:
  描述生成入口在哪里、支持什么生成介质

ZombieArchetype / BattleSpawnEntry:
  描述生成需求

BattleSpawnResolver:
  匹配入口能力和生成需求，返回生成位置或 protocol.issue
```

#### SpawnZoneConfig Resource

新增 `SpawnZoneConfig` Resource，属于生成系统，而不是 `LaneConfig` 的字段：

```gdscript
# scripts/battle/spawn_zone_config.gd (草案)
extends Resource
class_name SpawnZoneConfig

@export var zone_id: StringName = StringName()
@export var lane_id := 0
@export var x := 0.0
@export var zone_group: StringName = &"default"
@export var zone_tags: PackedStringArray = PackedStringArray()
@export var enabled := true
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `zone_id` | StringName | 生成入口 ID |
| `lane_id` | int | 所属行 |
| `x` | float | 生成入口 X 坐标 |
| `zone_group` | StringName | 入口组，如 `default` / `grave` / `ambush` |
| `zone_tags` | PackedStringArray | 入口能力，如 `spawn.medium.ground` / `spawn.medium.water` |
| `enabled` | bool | 入口是否启用 |

`BattlefieldPreset` 可声明静态 spawn zones：

```gdscript
@export var spawn_zones: Array[Resource] = [] # SpawnZoneConfig[]
```

也可以在预设为空时由生成系统根据 `lane_traits` 派生默认入口。例如：

| lane trait | 默认 zone_tags |
|------------|----------------|
| `surface.ground` | `["spawn.medium.ground"]` |
| `surface.water` | `["spawn.medium.water"]` |
| `surface.roof` | `["spawn.medium.roof", "spawn.medium.ground"]` |
| `placement.blocked` | 不派生默认入口 |

关键点：派生逻辑属于生成系统初始化，不属于 `LaneConfig` 的运行时判断。

#### 生成需求

生成需求可以先从 zombie archetype tags 读取，后续再收口为独立 spawn profile：

| 实体/条目 | 需求示例 |
|-----------|----------|
| 普通陆地僵尸 | `required_spawn_tags = ["spawn.medium.ground"]` |
| 水生僵尸 | `required_spawn_tags = ["spawn.medium.water"]` |
| 屋顶兼容僵尸 | `required_spawn_tags = ["spawn.medium.roof"]` 或兼容 ground |
| 墓碑生成 | `zone_group = "grave"`，另走 GridItem/墓碑入口 |

第一版可以使用 archetype tag 作为桥接，例如 `spawn.medium.ground` / `spawn.medium.water`。不要复用 `ground` / `water` 这类放置或视觉标签，避免语义混淆。

#### BattleSpawnResolver

`BattleSpawnResolver` 负责生成匹配：

```gdscript
func resolve_spawn(entry: Resource, archetype: Resource) -> Dictionary:
    # 返回 { "ok": bool, "lane_id": int, "x": float, "zone_id": StringName, "reason": StringName }
```

职责：

- 根据 wave entry 的 lane / zone_id / zone_group 找候选 `SpawnZoneConfig`
- 读取候选入口的 `zone_tags`
- 读取 archetype 或 spawn profile 的 `required_spawn_tags`
- 判断入口能力是否满足生成需求
- 返回稳定生成位置或记录 `protocol.issue`

`WaveRunner` 只负责时间和波次调度：

```text
WaveRunner
  -> BattleSpawnResolver.resolve_spawn(entry, archetype)
  -> BattleSpawner / BattleManager 实例化
```

若目标入口不匹配，第一版建议记录 `protocol.issue` 并跳过该 spawn entry。“寻找下一个可用入口”需要 wave entry 明确声明候选 zone group 或 fallback 策略，不应隐式随机改道。

### 方案 C: 地形高度采样层 (LaneTerrainProfile + BattlefieldMetrics)

Roof 坡度与投射物弧线修正不应作为 roof 专用逻辑处理。更通用的抽象是：棋盘地形提供“某行某个 X 位置的地面高度”，实体和投射物再声明“相对地面的高度”。

#### 核心高度语义

```text
terrain_z = BattlefieldMetrics.terrain_elevation_at(lane_id, x)
height_above_ground = entity/projectile local height
absolute_z = terrain_z + height_above_ground
ground_y = lane_base_y - terrain_z * projection_scale
projected_y = ground_y - height_above_ground * projection_scale
```

这样可以把多个地形统一为同一种采样问题：

| 地形 | 表达方式 | 说明 |
|------|----------|------|
| 平地 | `terrain_z = 0` | 默认草地、泳池 |
| 屋顶坡度 | `terrain_z = base + x * slope` | 线性上升或下降 |
| 阶梯 | `terrain_z = slot_elevations[slot_index]` | 分格离散高度 |
| 小山 / 反斜面 | 采样点线性插值 | 可表达上坡、下坡、局部高台 |
| 未来地形变动 | base profile + runtime modifiers | 由后续 `BattleTerrainState` 叠加，不修改静态 preset |

#### LaneTerrainProfile Resource

新增可选 `LaneTerrainProfile` Resource，专门描述一行的地形高度采样。它不负责放置合法性、生成过滤或环境效果。

```gdscript
# scripts/battle/lane_terrain_profile.gd (草案)
extends Resource
class_name LaneTerrainProfile

@export var profile_id: StringName = StringName()
@export var elevation_mode: StringName = &"flat" # flat / linear / stepped / sampled
@export var base_elevation := 0.0
@export var elevation_per_slot := 0.0
@export var slot_elevations: PackedFloat32Array = PackedFloat32Array()
@export var sample_points: PackedVector2Array = PackedVector2Array() # x -> elevation
@export var interpolation: StringName = &"step" # step / linear
@export var projection_y_scale := 1.0
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `profile_id` | StringName | 地形高度配置 ID |
| `elevation_mode` | StringName | 高度模式：`flat` / `linear` / `stepped` / `sampled` |
| `base_elevation` | float | 基础地面高度 |
| `elevation_per_slot` | float | 每格高度变化量，适用于屋顶等线性坡度 |
| `slot_elevations` | PackedFloat32Array | 每个 slot 的离散地面高度，适用于阶梯或高台 |
| `sample_points` | PackedVector2Array | X 位置到地面高度的采样点，适用于小山和反斜面 |
| `interpolation` | StringName | 采样点之间使用阶梯或线性插值 |
| `projection_y_scale` | float | 高度投影到 2D Y 轴的比例 |

第一版建议仅实现 `flat` / `linear` / `stepped`。`sampled` 可先作为设计预留，等出现真实小山或反斜面内容时再落地。

#### BattlefieldMetrics 查询接口

`BattlefieldMetrics` 继续保持“尺子”职责，只做纯查询：

```gdscript
func terrain_elevation_at(lane_id: int, x: float) -> float
func terrain_elevation_at_slot(lane_id: int, slot_index: int) -> float
func ground_position_for(lane_id: int, x: float) -> Vector2
func slot_position(lane_id: int, slot_index: int) -> Vector2
func terrain_sample_at(lane_id: int, x: float) -> Dictionary
```

职责边界：

- `terrain_elevation_at()` 返回地面高度，不判断地形能否种植。
- `ground_position_for()` 返回该 lane / x 的 2D 地面投影位置，可用于放置、投射物目标点和视觉定位。
- `slot_position()` 应使用 `terrain_elevation_at_slot()` 影响视觉 Y，但 slot 的标签和占位仍由 `BoardSlot` / `LaneConfig` 管理。
- `terrain_sample_at()` 可返回 `{ "elevation": float, "slope": float, "normal": Vector2 }`，第一版只要求 elevation。

#### 投射物高度参考

当前投射物的 `height` 应收口为“离当前地面高度”，命中判定时再加上地形高度得到绝对高度：

```text
projectile_abs_z = terrain_elevation_at(projectile_lane, projectile_x) + projectile_height
target_abs_min = terrain_elevation_at(target_lane, target_x) + target_hit_height_min
target_abs_max = terrain_elevation_at(target_lane, target_x) + target_hit_height_max
```

`ProjectileFlightProfile` / movement params 建议新增高度参考语义，而不是 `roof_arc_compensation`：

| 参数 | 含义 |
|------|------|
| `height_reference = &"terrain_follow"` | 始终保持相对当前地面固定高度，适用于直线豌豆 |
| `height_reference = &"launch_absolute"` | 保持发射时的绝对高度，适用于特殊直线弹或激光类效果 |
| `height_reference = &"ballistic_to_target"` | 起点地面高度与目标地面高度插值，再叠加弧高，适用于投手 |

Roof 直线投射物第一版应优先使用 `terrain_follow`，投手使用 `ballistic_to_target`。这样屋顶、小山和反斜面共享同一套逻辑。

#### 动态地形预留

未来如果需要地形变动，不应修改 `LaneConfig` 的静态数据，而应新增运行时状态层，例如 `BattleTerrainState`：

```text
terrain_elevation = lane_terrain_profile.sample(lane, x)
                  + BattleTerrainState.runtime_elevation_delta(lane, x)
```

动态地形系统应只提供高度 modifier 和必要的失效通知；放置合法性变化仍通过 BoardSlot modifier / GridItem / slot tag 体系另行设计，避免把地形高度系统变成新的万能规则容器。

### 与动态环境系统的组合边界

棋盘系统和环境系统正交组合：

```text
BattleScenario / BattleModeDef
  -> battlefield_preset: day_5lane / pool_6lane / roof_5lane / custom_terrain
  -> environment_profile: sunny / cloudy / night / fog / harsh_sun / timeline
```

边界规则：

- 棋盘系统不直接声明 `is_night`、`has_fog`、`weather` 或自然阳光参数。
- 环境系统不直接修改 `BoardSlot.slot_type`、`placement_tags`、占位角色或 `LaneTerrainProfile`。
- 若环境需要“局部雾”“烈日”“夜晚”等效果，应通过环境查询影响经济、可见度、视觉或实体状态。
- 若运行时需要改变格子可放置性，应通过 BoardSlot modifier / GridItem 设计，不借用天气系统。
- 若运行时需要改变地形高度，应通过未来 `BattleTerrainState` modifier，不借用环境 profile。

## 数据流设计

### 完整数据流

```text
BattleScenario
  |
  +-- BattlefieldPreset
        |
        +-- lane_configs[]
        |     |
        |     +-- BattleBoardState._rebuild_slots()
        |           -> per-lane slot 类型
        |           -> per-lane slot 标签
        |           -> per-lane terrain traits
        |
        +-- spawn_zones[]
        |     |
        |     +-- BattleSpawnResolver
        |           -> 生成入口能力匹配
        |
        +-- lane_y_positions / lane_origin_y / lane_spacing
        +-- LaneTerrainProfile
              |
              +-- BattlefieldMetrics
                    -> slot_position()
                    -> ground_position_for()
                    -> terrain_elevation_at()
                    |
                    +-- SpatialIndex / Detection
                    +-- Projectile / Controller
                    +-- BattleSpawnResolver

BattleScenario / BattleModeDef
  +-- EnvironmentProfile / RuleModule
        -> 见动态环境与天气系统草稿
```

### 关键路径分解

**1. BattlefieldPreset -> LaneConfig -> BattleBoardState slot 类型推导**

```text
BattlefieldPreset.lane_configs[]
  -> _rebuild_slots()
  -> 先用 lane_type 引用的 LaneTypeDef 展开默认字段
  -> per lane: lane_config.slot_type_override 非空则覆盖默认 slot_type，否则取 LaneTypeDef / 默认 ground
  -> per lane: lane_config.base_tags_override 非空则覆盖默认 base_tags，否则取 LaneTypeDef / BoardSlotCatalog.default_tags_for(slot_type)
  -> per lane: lane_traits 使用 LaneTypeDef 默认 + LaneConfig 显式覆盖或补充
  -> BoardSlotConfig 覆盖仍可 per-slot 微调
  -> slot 构造时通过 BattlefieldMetrics.slot_position() 计算 world_position
```

**2. LaneConfig.terrain_profile -> BattlefieldMetrics 地形高度采样**

```text
BattlefieldMetrics.lane_y_positions 构建时:
  -> 如果 BattlefieldPreset.lane_y_positions 非空，直接使用
  -> 否则使用 lane_origin_y + lane_id * lane_spacing + lane_config.height_offset
  -> terrain_elevation_at(lane, x) 从 lane_config.terrain_profile 采样地面高度
  -> slot_position(lane, slot) 使用 lane 基准 Y + terrain profile 的视觉投影偏移
  -> ground_position_for(lane, x) 作为投射物、目标点和视觉层的统一地面坐标入口
  -> 迁移期由 BattleManager._rebuild_lane_config() 先消费 preset 更新 lane_y_map
  -> 避免 BattleManager.get_lane_y() 与 BattlefieldMetrics 互相委派形成循环
```

**3. LaneConfig.lane_traits -> SpawnZoneConfig 默认入口派生**

```text
BattlefieldPreset.spawn_zones 为空或需要补默认入口:
  -> 生成系统读取 LaneConfig.lane_traits
  -> surface.ground 派生 spawn.medium.ground 入口
  -> surface.water 派生 spawn.medium.water 入口
  -> surface.roof 派生 spawn.medium.roof / spawn.medium.ground 入口
  -> placement.blocked 不派生默认入口
  -> 派生结果进入 BattleSpawnResolver 的候选入口集合
```

**4. BattleSpawnEntry -> BattleSpawnResolver -> 实例化**

```text
WaveRunner 到达 spawn 时间:
  -> resolved archetype
  -> BattleSpawnResolver.resolve_spawn(entry, archetype)
  -> 根据 lane / zone_id / zone_group 找候选 SpawnZoneConfig
  -> 用 zone_tags 匹配 required_spawn_tags
  -> 成功: 返回 lane_id / x / zone_id
  -> 失败: 记录 protocol.issue 并跳过该 spawn entry
```

**5. 地形高度 -> 投射物高度与命中判定**

```text
ProjectileMovement 输出:
  -> ground_position(lane, x)
  -> height_above_ground
ProjectileRoot / 命中路径:
  -> projectile_abs_z = terrain_elevation_at(lane, x) + height_above_ground
  -> target_abs_range = target_terrain_z + target.hit_height_range
  -> 使用 absolute_z overlap 判断高度命中
Visual projection:
  -> ground_y 已包含 terrain_elevation 的视觉投影
  -> projected_y = ground_y - height_above_ground * projection_scale
```

## 实现分批

### Phase 1: LaneConfig Resource + BattleBoardState 消费 + 新 Preset

**范围**：核心数据结构改造和新预设配置

**具体任务**：

- 新增 `LaneConfig` Resource（`scripts/battle/lane_config.gd`）
- 新增或固化 `LaneTypeDef` 内置模板：`grass` / `pool` / `roof` / `dirt`
- `BattlefieldPreset` 增加 `lane_configs` / `lane_y_positions` / `lane_origin_y` / `lane_spacing` / `spawn_zones` 字段
- `BattleBoardState._rebuild_slots()` 改造：读取 `lane_configs` 生成 per-row slot 类型和标签
- `LaneConfig` 增加 `lane_traits`，只描述地形特征，不描述僵尸生成过滤
- 内容加载/校验阶段根据 `lane_type` 展开默认字段；运行时不以 `lane_type` 做规则分支
- 新增 `SpawnZoneConfig` 草案脚本或先在文档中固定字段；Phase 1 可只做静态数据校验，不强制接入 `WaveRunner`
- `BattlefieldMetrics` 改造：支持从 `lane_configs` 读取 Y 坐标偏移、显式 `lane_y_positions`，并提供默认返回 0 的 `terrain_elevation_at()`
- 新增 `LaneTerrainProfile` Resource 草案脚本或先在文档中固定字段；Phase 1 只要求 `flat` 默认语义，不强制启用复杂 profile
- `BattlefieldMetrics` 增加 `ground_position_for(lane_id, x)`，让输入、投射物和视觉层拥有统一地面坐标入口
- `BattleManager._rebuild_lane_config()` 迁移期消费 preset 的 lane 几何，保证 `get_lane_y()` fallback 稳定
- `BoardSlotCatalog` 增加 `dirt` 类型（空标签，不可放置）
- `ProtocolValidator` 增加 `LaneTypeDef` / `LaneConfig` / `LaneTerrainProfile` / `SpawnZoneConfig` / `lane_configs` / `lane_y_positions` / `spawn_zones` 校验：类型、范围、重复 lane、lane_type 是否已知、slot_type 是否已知、terrain mode 是否受支持、spawn zone 是否指向有效 lane
- 新增 3 个预设 `.tres`：
  - `day_5lane`：标准 5 行草地
  - `pool_6lane`：泳池关卡（1 grass + 4 pool + 1 dirt）
  - `roof_5lane`：屋顶关卡（5 roof）
- 更新 `_apply_slot_configs()`：确保 per-slot config 在 lane_config 之后覆盖

**验证场景**：

- 3 个新预设的基本 smoke 验证（实体生成、槽位创建、放置成功）
- 多行类型混合预设（泳池）的槽位类型 tag 正确性
- `lane_type` 模板展开后的 `slot_type` / `base_tags` / `lane_traits` 与预期一致
- `spawn_zones` 指向无效 lane 时触发 protocol issue
- `dirt` 行拒绝放置验证
- 默认 `lane_configs` 为空时回退现有行为

### Phase 2: SpawnZone + BattleSpawnResolver

**范围**：生成入口与生成需求匹配

**具体任务**：

- 新增 `BattleSpawnResolver`
- `BattlefieldPreset.spawn_zones` 非空时直接作为候选生成入口
- `spawn_zones` 为空时，生成系统可按 `lane_traits` 派生默认入口
- 僵尸 archetype 或临时 spawn profile 增加生成需求 tag：`spawn.medium.ground` / `spawn.medium.water` / `spawn.medium.roof`
- `WaveRunner` 调用 `BattleSpawnResolver.resolve_spawn(entry, archetype)`，不直接读取 `LaneConfig`
- 第一版不隐式改道：目标入口不匹配时记录 `protocol.issue` 并跳过该 spawn entry
- 验证失败信息必须指出 spawn entry、目标 zone/lane、zone_tags 和缺失的 required_spawn_tags

**验证场景**：

- 泳池关：水生僵尸只匹配 `spawn.medium.water` 入口
- 草地行：陆地僵尸只匹配 `spawn.medium.ground` 入口
- 屋顶行：屋顶兼容僵尸匹配 `spawn.medium.roof` 或显式兼容入口
- dirt 行：默认不派生生成入口
- 缺失生成需求 tag 的 zombie archetype 触发 protocol issue
- `WaveRunner` 不直接依赖 `LaneConfig.spawn_*` 字段

### Phase 3: 地形高度采样 + 投射物高度参考

**范围**：屋顶/阶梯的地形高度采样 + 投射物相对地面高度语义

**具体任务**：

- `BattlefieldMetrics.terrain_elevation_at(lane_id, x)` 支持 `flat` / `linear` / `stepped` 三种模式
- Roof 预设改用 `linear` terrain profile 表达坡度，而不是硬编码 roof 分支
- 阶梯或高台样本使用 `stepped` terrain profile 表达每格高度
- `slot_position(lane, slot)` 和 `ground_position_for(lane, x)` 消费 terrain profile 的视觉投影偏移
- 明确 `ProjectileRoot.height` 为 `height_above_ground`
- `ProjectileFlightProfile` 或 movement params 增加 `height_reference`：`terrain_follow` / `launch_absolute` / `ballistic_to_target`
- 直线豌豆类使用 `terrain_follow`，保持相对当前地面固定高度
- 投手类使用 `ballistic_to_target`，在起点地面高度和目标地面高度之间插值并叠加弧高
- 命中高度比较从相对高度升级为 `terrain_z + hit_height_range` 的 absolute_z overlap
- 不新增 `roof_arc_compensation`，Roof 只是 terrain profile 的一个样本

**验证场景**：

- 平地 profile 与现有行为完全一致
- 屋顶 linear profile 下 slot Y 坐标正确偏移
- 阶梯 stepped profile 下不同 slot 的地面高度可被 metrics 查询
- `terrain_follow` 直线投射物在屋顶坡度上保持相对地面高度并能命中目标
- `ballistic_to_target` 投射物可从低地打到高地或从高地打到低地
- absolute_z 命中比较能区分高台目标与低地飞行物

### Phase 4: Grave / GridItem 系统（延后）

**范围**：墓碑等格子级障碍物的完整系统。当前阶段不做，仅做设计预留。

**设计预留**：

- GridItem 不应作为纯 Resource 附着物，而应作为 `entity_kind = &"field_object"` 的 Archetype 驱动实体，由 `BattleGridItemState` 绑定到 BoardSlot。
- 阻挡型 GridItem 第一版建议同时占用 `grid_item` role 与现有 `blocker` role，复用当前 `required_empty_roles = ["blocker"]` 放置约束并兼容 Grave Buster。
- `validate_request()` 不应只靠一个粗粒度 `blocks_planting` tag 拒绝所有植物，否则 Grave Buster 这类交互型植物也会被拒绝。若后续需要 tag 级阻挡，应配套 bypass / required_grid_item_tags 语义。
- 墓碑破坏通过 GridItem 专用 effect 或现有 damage 链实现；现有 `archetype_tombstone_blocker` 是 plant/blocker 过渡实现，迁移时需要单独验证兼容性。
- 此系统设计上独立于 `LaneConfig` 和动态环境模块，不阻塞 Phase 1-3 的落地。

## 需要补充的能力

现有系统需要以下小规模改造以支撑 Phase 1-3：

### SpatialIndex

- 未来可增加通过 lane 类型过滤（查询指定 lane_type 上的实体）
- 若投射物命中改为 absolute_z，`height_range` 查询需要明确传入绝对高度范围，或由调用方在 filter 中补 terrain_z 比较
- 此项非 Phase 1-2 所必需

### BattlefieldMetrics / LaneTerrainProfile

- `BattlefieldMetrics` 增加 `terrain_elevation_at(lane_id, x)`，默认返回 0，确保现有平地场景不变
- 增加 `ground_position_for(lane_id, x)`，避免投射物、输入和视觉层各自重算 lane Y / slope
- `slot_position(lane_id, slot_index)` 通过 `ground_position_for()` 派生，避免 slot 和连续对象使用两套地面坐标
- `LaneTerrainProfile` 第一版只实现 `flat` / `linear` / `stepped`，暂不实现任意曲线编辑
- `ProtocolValidator` 校验 terrain profile 的 mode、数组长度、采样点顺序和 projection scale

### ProjectileRoot / ProjectileMovement

- 明确 `ProjectileRoot.height` 为 `height_above_ground`，并在 state snapshot 中区分 `height_above_ground` 与 `absolute_z`
- movement params 增加 `height_reference`，至少支持 `terrain_follow` / `ballistic_to_target`
- `ProjectileMovementBase` 提供 `_terrain_elevation_at(position)` helper，从 `GameState.current_battle.get_battlefield_metrics()` 查询
- 命中路径比较 absolute_z，避免高低地目标用同一相对高度误判
- `ProjectileFlightProfile` 增加 `height_reference` 字段，并由 `ProtocolValidator` 限定可选值

### SpawnZoneConfig / BattleSpawnResolver

- 新增 `SpawnZoneConfig` Resource，描述生成入口位置、入口组和入口能力 tag
- 新增 `BattleSpawnResolver`，负责从 `BattleSpawnEntry`、archetype 和 `SpawnZoneConfig` 中解析合法生成位置
- 第一版可从 resolved zombie archetype.tags 读取 `required_spawn_tags`，后续再收口为独立 spawn profile
- Zombie archetype 需要增加 `spawn.medium.ground` / `spawn.medium.water` / `spawn.medium.roof` 等生成需求 tag
- 目标入口不匹配时记录 protocol issue 并跳过；候选入口重选需要 wave entry 明确声明策略

### WaveRunner

- 只负责波次时间和 spawn entry 调度
- 不直接读取 `LaneConfig` 或地形类型
- 通过 `BattleSpawnResolver.resolve_spawn()` 获取最终 lane/x/zone

### BattleModeHost / BattleScenario

- 允许同一个 `battlefield_preset` 与不同 `environment_profile` 组合
- 棋盘默认值只负责空间结构，不再承载环境初始值
- 环境 profile 的加载和生命周期由动态环境草稿定义

## 不做的事

以下内容明确排除在棋盘多样性系统的当前设计范围之外：

1. **不修改冻结 Mechanic family**：触发器、效果、detection、controller 等 10 个 family 保持不变。
2. **不实现动态环境与天气系统**：晴天、阴天、夜晚、浓雾、烈日、昼夜交替和自然阳光修正由独立草稿负责。
3. **不实现位掩码 (bitmask) 地形系统**：当前 tag 系统足够表达 slot 的地形属性。位掩码的精度虽高，但引入复杂度不匹配当前需求。
4. **不实现完整的墓碑/GridItem 系统**：墓碑/障碍物是独立功能模块，不阻塞 Phase 1-3 的落地。
5. **不做 per-cell 地形修改**：初始版本不支持运行时改变单个格子的地形类型。地形高度可以预留 runtime modifier 方向，但不与 slot tag / 放置合法性变化混在一起实现。
6. **不把 lane_type 作为运行时扩展点**：`lane_type` 只是核心内置模板 ID，不作为运行时规则分支。扩展包不得注册新的 lane_type。如有合理的新模板需求，通过 ADR 加入核心集合。
7. **不实现禅境花园 / 智慧树背景**：特殊模式背景不在棋盘多样性的当前目标内。
8. **不修改 BattleManager 的直接特判**：行级空间数据通过 `LaneConfig` / `BattlefieldMetrics` 查询，不在 `BattleManager` 中添加 `lane_type` if/switch 分支。
9. **不会一次迁移所有现有 .tres**：Phase 1 的新预设只增加不修改，现有验证场景不受影响。
10. **不实现完整 3D 地形系统**：本设计只提供 `lane + x -> terrain elevation` 的规则采样，不引入自由曲面、三维物理碰撞、导航网格或编辑器级地形刷。

## 决策记录

### D1: lane_type 降级为 LaneTypeDef 模板 ID

- **问题**：`lane_type` 是否应该作为运行时行类型系统存在？
- **决策**：不作为运行时类型系统。`lane_type` 仅作为 `LaneTypeDef` / 内置模板 ID，使用 `StringName` 表达。
- **理由**：
  - 真正的运行时事实源应是展开后的 `slot_type`、`base_tags`、`lane_traits`、`terrain_profile` 和 `visual_theme`
  - 避免各系统写 `if lane_type == &"pool"` 形成新一层规则分支
  - 保留作者友好的 shorthand，减少泳池/屋顶等整行配置重复
  - 与现有 `StringName` 标识风格一致
- **后果**：内容加载或校验阶段需要执行模板展开；`ProtocolValidator` 必须校验未知 lane_type，不允许静默退回 ground。

### D2: dirt 行通过空标签实现不可放置

- **问题**：不可种植行如何表示？
- **决策**：`slot_type = &"dirt"`，`base_tags = []`。
- **理由**：放置验证检查 placement_tags 必须在 slot 的 effective_tags 中存在。空标签数组将自然拒绝所有普通放置请求。
- **后果**：如果以后需要 dirt 行支持某些特殊放置，只需要在 dirt 行添加对应标签，而不是修改验证逻辑。

### D3: 僵尸生成通过 SpawnZone / SpawnResolver 解耦

- **问题**：泳池行如何防止陆地僵尸生成？
- **决策**：不让 `LaneConfig` 直接过滤僵尸。地形只暴露 `lane_traits`；生成系统通过 `SpawnZoneConfig.zone_tags` 和 zombie / spawn profile 的 `required_spawn_tags` 匹配。
- **理由**：
  - 单一职责：地形描述“我是什么”，生成系统决定“从哪里生成什么”
  - 避免棋盘行配置耦合 zombie 分类和 wave 编排
  - 复用 tag 匹配，不需要额外的僵尸类型枚举
  - 后续墓碑、伏击、空降等入口可以通过 `zone_group` / `zone_tags` 扩展
- **后果**：需要新增 `SpawnZoneConfig` 和 `BattleSpawnResolver`；WaveRunner 不再直接读取 `LaneConfig`。

### D4: 棋盘系统与环境系统正交拆分

- **问题**：棋盘地形和天气/昼夜/雾是否应该合在一个草稿和一个系统里？
- **决策**：拆分。棋盘系统负责空间结构，环境系统负责时间变化和全局/区域条件修正。
- **理由**：
  - 单一职责：棋盘决定“哪里是什么”，环境决定“当前世界状态如何影响规则”
  - 可组合性：同一个泳池 preset 可以搭配晴天、夜晚、浓雾或烈日
  - 可测试性：棋盘布局可在没有环境模块时独立验证
- **后果**：`BattleScenario / BattleModeDef` 需要能同时引用 `battlefield_preset` 和 `environment_profile`。

### D5: 屋顶坡度抽象为地形高度采样

- **问题**：屋顶的坡度和未来小山、阶梯、反斜面是否应该分别实现？
- **决策**：统一抽象为 `BattlefieldMetrics.terrain_elevation_at(lane_id, x)` 地形高度采样。Roof 只是 `linear` terrain profile 的一个内容样本。
- **理由**：
  - 符合“metrics 是尺子”的设计原则
  - 不把屋顶逻辑变成 `BattleManager` 的特判
  - 未来阶梯、小山、反斜面可复用同一接口
- **后果**：metrics 需要支持 `flat` / `linear` / `stepped` 等采样模式，投射物和视觉层需要从 metrics 查询地面高度。

### D6: 投射物高度采用离地高度 + 地形高度

- **问题**：Roof 上的投射物是否应该通过 `roof_arc_compensation` 之类专用参数修正？
- **决策**：不新增 roof 专用补偿。投射物的 `height` 收口为 `height_above_ground`，命中和投影时计算 `absolute_z = terrain_z + height_above_ground`。
- **理由**：
  - 当前投射物系统已有 `ground_position + height` 的基础，可直接推广为相对地面高度
  - 直线、追踪、抛物线只需要选择不同 `height_reference`，不需要知道地形类型
  - 高低地命中可通过 absolute_z overlap 统一判断
- **后果**：`ProjectileFlightProfile` / movement params 需要新增 `height_reference`，命中路径需要比较 absolute_z 而不是裸 `height`。
