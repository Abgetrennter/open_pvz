## 文档定位

本文记录动态环境与天气系统的早期设计方向。目标是把晴天、阴天、夜晚、浓雾、烈日、昼夜交替等“世界状态”从棋盘系统中拆出，形成可组合、可动态变化、可由模式控制的环境层。

核心判断：

```text
棋盘系统：决定空间是什么。
环境系统：决定当前世界状态如何影响规则。
```

因此同一个棋盘 preset 可以搭配不同环境 profile：

```text
草地 + 晴天
草地 + 夜晚
泳池 + 阴天
泳池 + 浓雾
屋顶 + 烈日
屋顶 + 昼夜交替
```

本文讨论的是：

- 环境系统与棋盘系统的正交边界
- 静态环境 profile 与动态 timeline
- 晴天、阴天、烈日对自然阳光经济的影响
- 昼夜状态对自然阳光、nocturnal 植物和视觉主题的影响
- 浓雾对可见度、视觉遮罩和清雾交互的影响
- `BattleRuleModule` / `BattleEnvironmentState` 的落地方式
- 分阶段实施方案和验证方向

本文不讨论：

- `LaneConfig`、`LaneTerrainProfile`、泳池/屋顶/阶梯等棋盘结构，详见 [棋盘多样性与地形系统设计草案.md](棋盘多样性与地形系统设计草案.md)
- 修改冻结 Mechanic family
- 完整气候模拟、粒子天气物理或真实时间天气
- 通过环境系统直接修改 slot 类型、slot tag 或棋盘占位
- 把天气做成实体专属硬编码逻辑

## 背景与目标

原版 PVZ 往往把背景、棋盘和环境绑定在同一个 `BackgroundType` 中，例如：

| 背景 | 棋盘维度 | 环境维度 |
|------|----------|----------|
| `DAY_GRASS` | 草地 | 白天 |
| `NIGHT_GRASS` | 草地 | 夜晚 |
| `DAY_POOL` | 泳池 | 白天 |
| `NIGHT_POOL` | 泳池 | 夜晚 + 迷雾 |
| `DAY_ROOF` | 屋顶 | 白天 |
| `NIGHT_ROOF` | 屋顶 | 夜晚 |

Open PVZ 的目标不是复刻固定背景枚举，而是让规则组合成为核心能力。棋盘和环境拆分后，内容作者可以复用同一套空间布局，组合不同环境条件。

当前项目已经具备：

- `BattleModeHost` 和 `BattleRuleModule`：可承载模式/环境规则模块
- `BattleEconomyState`：已管理阳光资源、收集、消费和 scenario 脚本化阳光掉落
- `StateComponent`：已有 `sleeping` 等状态基础
- `EventBus`：可发布 `environment.changed`、`environment.fog_clear_requested` 等语义事件
- `VisualStageLayerService`：已有视觉层骨架，可承载 night tint / fog overlay
- `DetectionRegistry` / `SpatialIndex`：可扩展目标查询和可见度过滤

缺失的关键能力：

- 没有统一的 `BattleEnvironmentProfile` 描述初始环境和变化时间线
- 没有 `BattleEnvironmentState` 暴露当前环境查询
- 当前没有真正的自然阳光循环；`sun_drop_entries` 是 scenario 脚本化掉落，不应被天气倍率隐式修改
- 自然阳光掉落没有独立调度器和环境修正入口
- 昼夜和天气切换没有固定 tick 驱动的确定性流程
- 迷雾缺少统一的可见度/视觉 alpha 数据源
- nocturnal 植物的白天睡眠、夜间唤醒语义尚未收口

## 核心边界

### 与棋盘系统的边界

```text
BattleScenario / BattleModeDef
  -> battlefield_preset     # 空间布局
  -> environment_profile    # 世界状态
  -> rule_modules           # 模式和特殊规则
```

职责划分：

| 系统 | 负责 | 不负责 |
|------|------|--------|
| 棋盘系统 | lane、slot、地形高度、生成过滤、放置合法性 | 天气、昼夜、雾、自然阳光倍率 |
| 环境系统 | 光照、天气、可见度、自然阳光修正、环境视觉/音频 | 修改 slot 类型、修改 lane_type、改变占位角色 |
| 模式系统 | 组合 preset/profile/module，覆盖参数 | 在 `BattleManager` 写特判 |

### 与规则模块的关系

第一版建议通过 `BattleRuleModule` 落地，而不是新增 autoload：

```text
BattleModeHost
  -> EnvironmentRuleModule
      -> BattleEnvironmentState
      -> on_mode_setup()
      -> on_battle_start()
      -> on_game_tick()
      -> on_event()
```

理由：

- 与现有模式层一致，不新增全局单例
- 环境本身是战斗实例内状态，不应跨战斗共享
- 可通过 mode/scenario 参数覆盖
- 支持后续多个环境模块或 profile 组合

如果后续多个系统都需要查询当前环境，可由 `BattleManager` 暴露：

```gdscript
func get_environment_state() -> BattleEnvironmentState
```

落地前收口：

- 第一版不在 `BattleScenario` 或 `BattleModeDef` 增加一等 `environment_profile` 字段。
- `BattleEnvironmentProfile` 通过 `BattleRuleModule.params.environment_profile` 传入。
- `EnvironmentRuleModule` 在 setup 时把 `BattleEnvironmentState` 注册到 `BattleManager`，其他系统通过 `battle.get_environment_state()` 查询。
- Fog 不进入 Phase 1，继续放在 Phase 3。

## 环境维度

### 光照 (Light)

光照描述当前世界的昼夜状态和可见光水平：

| 条件 | 规则影响 | 视觉影响 |
|------|----------|----------|
| `day` | 默认自然阳光；nocturnal 植物可进入 sleeping | 标准色调 |
| `dusk` | 可作为昼夜切换过渡；自然阳光倍率降低 | 暖色或低光 |
| `night` | 默认关闭自然阳光；nocturnal 植物不应被强制睡眠 | 暗色调 |

语义收口：

- `nocturnal` 不等于“夜晚睡眠”。建议语义为：白天可睡眠，夜晚应可活动。
- 当前蘑菇资源已经挂载 `core.sleeping`，落地时采用资源迁移口径：`core.sleeping` 不再作为 nocturnal 资源默认机制，而由环境模块在白天注入或请求进入 `sleeping`。
- 夜晚不强制睡眠；已经处于 sleeping 的 nocturnal 实体应通过 `entity.wake` 或公共状态切换接口恢复可行动状态。
- 唤醒事件沿用现有 `entity.wake`，不新增 `entity.woke`。

### 天气 (Weather)

天气描述当前自然阳光经济和环境表现：

| 条件 | 自然阳光影响 | 说明 |
|------|--------------|------|
| `sunny` | `interval_scale = 1.0`，`value_scale = 1.0` | 默认晴天 |
| `cloudy` | 掉落间隔变长或阳光值变小 | 阴天 |
| `harsh_sun` | 掉落更频繁或阳光值更高 | 烈日 |
| `rain` | 第一版不做规则影响 | 先作为视觉/音频预留 |

第一版只做自然阳光修正，不做温度、湿度、风力、移动速度变化等复杂模拟。

### 可见度 (Visibility)

浓雾属于环境可见度，不属于棋盘地形：

| 条件 | 规则影响 | 视觉影响 |
|------|----------|----------|
| `fog` | 限制目标发现/可见性查询，可选影响 UI 可见 | fog overlay |
| `clear` | 无可见度限制 | 无遮罩 |

雾的边界：

- 迷雾中实体仍然存在，碰撞和伤害判定不因视觉隐藏而消失。
- detection 是否受雾影响应显式查询环境状态，不能隐式改 `SpatialIndex` 的实体集合。
- Plantern / Torchwood / Blover 等清雾行为进入 `FogRuleModule` 或 `environment.fog_clear_requested` 事件。
- 不扩展 `apply_status` 来承载全局清雾；清雾是环境状态变化，不是实体状态。

## 数据模型

### BattleEnvironmentProfile Resource

新增 `BattleEnvironmentProfile` Resource，描述环境初始状态和动态 timeline：

```gdscript
# scripts/battle/environment/battle_environment_profile.gd (草案)
extends Resource
class_name BattleEnvironmentProfile

@export var profile_id: StringName = StringName()
@export var initial_conditions: PackedStringArray = [&"day", &"sunny"]
@export var natural_sun_enabled := true
@export var sun_interval_scale := 1.0
@export var sun_value_scale := 1.0
@export var light_level := 1.0
@export var visibility_range_slots := -1 # -1 表示不限制
@export var fog_start_column := -1
@export var fog_max_alpha := 0.85
@export var visual_environment_id: StringName = StringName()
@export var audio_environment_id: StringName = StringName()
@export var timeline: Array[Resource] = [] # EnvironmentTimelineEntry[]
```

字段说明：

| 字段 | 说明 |
|------|------|
| `profile_id` | 环境配置 ID |
| `initial_conditions` | 初始条件集合，如 `day`、`sunny`、`fog` |
| `natural_sun_enabled` | 是否启用自然阳光 |
| `sun_interval_scale` | 自然阳光掉落间隔倍率，越大越慢 |
| `sun_value_scale` | 每次自然阳光数值倍率 |
| `light_level` | 光照强度，供视觉和规则查询 |
| `visibility_range_slots` | 可见范围格数，`-1` 表示无限制 |
| `fog_start_column` | 迷雾起始列，`-1` 表示无迷雾 |
| `fog_max_alpha` | 最大迷雾不透明度 |
| `visual_environment_id` | 视觉环境主题 |
| `audio_environment_id` | 音频环境主题 |
| `timeline` | 动态变化时间线 |

### EnvironmentTimelineEntry Resource

```gdscript
# scripts/battle/environment/environment_timeline_entry.gd (草案)
extends Resource
class_name EnvironmentTimelineEntry

@export var at_seconds := 0.0
@export var add_conditions: PackedStringArray = PackedStringArray()
@export var remove_conditions: PackedStringArray = PackedStringArray()
@export var set_conditions: PackedStringArray = PackedStringArray()
@export var natural_sun_enabled_override := -1 # -1 keep, 0 false, 1 true
@export var sun_interval_scale := -1.0
@export var sun_value_scale := -1.0
@export var light_level := -1.0
@export var visibility_range_slots := -2 # -2 keep, -1 unlimited
@export var transition_seconds := 0.0
```

第一版可以只支持离散切换，`transition_seconds` 作为视觉平滑预留。规则层只在固定 tick 上改变状态，避免 Tween 或视觉过渡影响战斗结果。

### BattleEnvironmentState

`BattleEnvironmentState` 是运行时只读/受控写的状态对象：

```gdscript
# scripts/battle/environment/battle_environment_state.gd (草案)
extends RefCounted
class_name BattleEnvironmentState

func has_condition(condition_id: StringName) -> bool
func get_conditions() -> PackedStringArray
func is_natural_sun_enabled() -> bool
func get_sun_interval_scale() -> float
func get_sun_value_scale() -> float
func get_light_level() -> float
func get_visibility_range_slots() -> int
func fog_alpha_at_slot(lane_id: int, slot_index: int) -> float
func is_position_visible(lane_id: int, x: float) -> bool
func get_visual_environment_id() -> StringName
```

状态变化时发事件：

```text
environment.changed
environment.fog_changed
environment.fog_clear_requested
```

事件 payload 应包含变化前后条件、变化来源和当前仿真 tick，方便验证场景断言。

## 规则接入

### EnvironmentRuleModule

第一版可以使用一个统一模块：

```gdscript
# BattleRuleModule params 草案
{
    "module_id": &"environment.core",
    "params": {
        "environment_profile": preload("res://data/combat/environments/env_sunny_day.tres"),
        "allow_timeline": true
    }
}
```

职责：

- 在 `on_mode_setup()` 中创建 `BattleEnvironmentState`
- 在 `on_battle_start()` 中应用初始环境到经济、状态、视觉层
- 在 `on_game_tick()` 中按 100Hz 仿真时间推进 timeline
- 在 `on_event()` 中消费 `environment.fog_clear_requested`、`entity.spawned`、`entity.wake` 等事件
- 状态改变时发布 `environment.changed`

如果后续环境维度变复杂，可以拆成多个模块：

| 模块 | 职责 |
|------|------|
| `environment.core` | profile 加载、timeline、状态查询 |
| `environment.day_night` | 昼夜/nocturnal/sun enabled |
| `environment.weather` | 晴天/阴天/烈日阳光倍率 |
| `environment.fog` | 迷雾 alpha、清雾、可见度 |

第一版优先用一个核心模块，避免过早拆分。

### BattleEconomyState 接入

环境系统影响自然阳光，不应直接改阳光资源总量。

代码审查结论：当前 `BattleEconomyState` 没有自然天降阳光循环，只有 `BattleScenario.sun_drop_entries` 驱动的 scheduled drops。`sun_drop_entries` 常用于验证和脚本化场景，不能被晴天/阴天/烈日倍率隐式修改。

因此第一版必须区分两类阳光：

| 类型 | 来源 | 是否受环境影响 | 用途 |
|------|------|----------------|------|
| scheduled sun drops | `BattleScenario.sun_drop_entries` | 否 | 验证、脚本化关卡、固定事件 |
| natural sun drops | `BattleEconomyState` 新增自然调度器 | 是 | 晴天、阴天、烈日、夜晚等环境规则 |

推荐接口：

```gdscript
func set_natural_sun_enabled(enabled: bool) -> void
func set_natural_sun_interval_scale(scale: float) -> void
func set_natural_sun_value_scale(scale: float) -> void
```

`BattleEconomyState` 新增内部字段草案：

```gdscript
var natural_sun_enabled := false
var natural_sun_interval_seconds := 8.0
var natural_sun_value := 25
var natural_sun_interval_scale := 1.0
var natural_sun_value_scale := 1.0
var _next_natural_sun_drop_time := -1.0
```

环境变化路径：

```text
EnvironmentRuleModule 收到环境变化
  -> 计算 natural_sun_enabled / interval_scale / value_scale
  -> 调用 BattleEconomyState setter
  -> BattleEconomyState 后续 tick 按修正值生成自然阳光
```

`_on_game_tick()` 的职责拆分：

```text
_process_scheduled_drops(game_time)    # 现有逻辑，不受天气影响
_process_natural_sun_drops(game_time)  # 新增逻辑，受环境影响
_process_scheduled_spends(game_time)   # 现有逻辑
```

自然阳光的 spawn 仍复用 `spawn_sun()`，但 `source_type` 应明确为 `&"natural_sun"` 或继续使用受文档约束的 `&"sky_drop"`。第一版建议使用 `&"natural_sun"`，便于验证场景区分 scripted drops 与环境自然掉落。

这样 `BattleEconomyState` 不需要知道 sunny/cloudy/night 等条件名，只消费明确数值。

### StateComponent 接入

昼夜与 nocturnal 植物关系：

```text
day:
  nocturnal 植物可进入 sleeping
night:
  nocturnal 植物应进入 awake/active 或至少不被强制 sleeping
```

需要补充能力：

- 第一版采用资源迁移口径：`nocturnal` 资源默认不再自带 `core.sleeping`；白天由环境模块注入或请求进入 `sleeping`。
- `StateComponent` 提供公共状态切换接口，避免 RuleModule 直接改内部字段。
- 或提供事件驱动路径：环境模块发送 `entity.wake` / `entity.sleep_requested`，由组件消费。
- 当前已有 Coffee Bean `entity.wake` 路径，应优先复用。

### Detection / SpatialIndex 接入

迷雾影响目标发现，不应从 `SpatialIndex` 中移除实体。

推荐路径：

```text
Detection strategy:
  -> 正常从 SpatialIndex 查询候选
  -> 如果当前环境有 fog:
       调用 environment_state.is_position_visible(lane, x)
       过滤不可见目标
  -> 返回可见候选
```

第一版可以只让特定 detection 消费可见度，避免一次性改变全部目标查询语义。

### Visual / Audio 接入

视觉层消费环境状态，但不反向影响战斗结果：

| 环境状态 | 视觉/音频动作 |
|----------|---------------|
| `night` | 设置暗色调或低光 profile |
| `dusk` | 设置过渡色调 |
| `fog` | 激活 fog overlay，消费每格 alpha |
| `rain` | 播放雨声和视觉粒子，第一版可仅预留 |
| `harsh_sun` | 高亮/热浪视觉，第一版可仅预留 |

接口草案：

```gdscript
VisualStageLayerService.set_environment_theme(theme_id: StringName)
VisualStageLayerService.set_fog_alpha_grid(alphas: Dictionary)
AudioCueRegistry.play_environment_cue(cue_id: StringName)
```

### Fog 清除接入

主动清雾不应扩展 `apply_status`。清雾是环境状态变更，不是实体 status，也不是棋盘 slot 修改。

推荐分两层：

```text
clear_fog effect:
  内容作者可配置的效果入口
  只负责把 effect 参数翻译成事件

environment.fog_clear_requested:
  FogRuleModule 消费的运行时事件
  是唯一进入 fog 状态变更的公共请求入口
```

这保留了 EffectRegistry 的内容装配入口，同时避免普通 effect 直接持有或修改环境状态。

事件 payload 示例：

```gdscript
{
    "source_entity_id": source_id,
    "scope": &"radius", # radius / full_board / lane / column
    "lane_id": lane_id,
    "slot_index": slot_index,
    "radius_slots": 4,
    "duration_seconds": 0.0,
    "strength": 1.0
}
```

字段语义：

| 字段 | 说明 |
|------|------|
| `source_entity_id` | 清雾来源实体。持续型清雾源可随实体死亡/移除自动失效 |
| `scope` | 清雾范围：`radius` / `full_board` / `lane` / `column` |
| `lane_id` / `slot_index` | `radius`、`lane` 或 `column` 的定位信息 |
| `radius_slots` | 半径型清雾范围，单位为格 |
| `duration_seconds` | `0` 表示持续型或瞬时重算；大于 0 表示临时清除，到期恢复 |
| `strength` | 清除强度。第一版固定为 `1.0`，预留半清除 |

`FogRuleModule` 是唯一修改 fog alpha 状态的地方。Plantern、Torchwood、Blover 不直接改 fog grid，只注册/请求清雾源。

#### FogRuleModule 内部状态

Fog 状态不应只存一个最终 alpha grid。推荐拆成三层：

```text
base_fog_alpha_grid
clear_sources
final_fog_alpha_grid
```

计算口径：

```text
final_alpha = base_alpha * (1.0 - max_clear_strength_at_cell)
```

设计收益：

- 基础雾区可由 profile / timeline 决定。
- 清雾源消失后可以自然恢复。
- Blover 的临时全屏清雾只是一个带过期时间的 clear source。
- 多个清雾源叠加时第一版取最大值即可，避免复杂叠加规则。

清雾源内部结构草案：

```gdscript
{
    "source_id": StringName(),
    "source_entity_id": 0,
    "scope": &"radius",
    "lane_id": 2,
    "slot_index": 5,
    "radius_slots": 4,
    "expires_at_tick": -1, # -1 表示跟随实体存活或由模块显式移除
    "strength": 1.0
}
```

#### 典型清雾来源

Plantern：

```text
entity.spawned / entity.removed
  -> FogRuleModule 检测 fog_clearer tag
  -> 从实体参数或 tag 映射读取 radius_slots
  -> 注册/移除持续清雾源
```

Torchwood：

```text
entity.spawned / entity.removed
  -> FogRuleModule 检测 fog_clearer tag 或 fog_clear_radius 参数
  -> 注册较小半径的持续清雾源
```

Blover：

```text
clear_fog effect
  -> 发布 environment.fog_clear_requested(scope = full_board, duration_seconds = 10)
  -> FogRuleModule 添加临时 full_board clear source
  -> expires_at_tick 到期后自动移除并重算 final_fog_alpha_grid
```

第一版建议 Plantern/Torchwood 都走持续清雾源，Blover 走临时全屏清雾源。这样三者共享同一套状态模型。

#### Detection 可见度过滤

雾不应该改变碰撞或伤害事实，只影响“能不能发现/看见”。推荐路径：

```text
Detection strategy
  -> SpatialIndex 查候选
  -> 如果 detection params.respect_visibility == true:
       environment_state.is_position_visible(lane, x)
       过滤不可见目标
  -> 返回候选
```

第一版只让明确声明 `respect_visibility` 的 detection 消费雾可见度，避免全局改变所有目标发现语义。碰撞、伤害和已经锁定的特殊效果是否受雾影响，需要各自显式设计，不由 FogRuleModule 隐式改变。

## 数据流设计

### 静态环境启动

```text
BattleScenario / BattleModeDef
  -> environment_profile
  -> BattleModeHost._resolve_rule_modules()
  -> environment.core.on_mode_setup()
  -> BattleEnvironmentState 初始化
  -> apply_to_economy / apply_to_visual / apply_to_state
  -> environment.changed
```

### 动态 timeline

```text
game.tick
  -> EnvironmentRuleModule.on_game_tick()
  -> 使用 GameState 仿真 tick 计算 elapsed_seconds
  -> 检查 timeline entry
  -> 更新 BattleEnvironmentState
  -> 发布 environment.changed
  -> 同步 economy / detection / visual 所需缓存
```

禁止使用 `Timer` 或 `OS.get_ticks_*` 驱动游戏逻辑。所有切换都走 100Hz 固定 tick。

### 昼夜交替示例

```text
0s:   day + sunny
90s:  dusk
120s: night
210s: dawn
240s: day + sunny
```

对应行为：

- `day`：自然阳光启用，nocturnal 可睡眠
- `dusk`：自然阳光倍率降低，只影响经济和视觉
- `night`：自然阳光关闭，nocturnal 唤醒或保持可行动
- `dawn`：视觉过渡，准备回到 day

### 天气切换示例

```text
0s:   sunny
60s:  cloudy
120s: harsh_sun
180s: sunny
```

经济参数示例：

| 条件 | natural_sun_enabled | interval_scale | value_scale |
|------|---------------------|----------------|-------------|
| `sunny` | true | 1.0 | 1.0 |
| `cloudy` | true | 1.5 | 0.8 |
| `harsh_sun` | true | 0.75 | 1.25 |
| `night` | false | 1.0 | 1.0 |

当多个条件同时存在时，第一版建议使用 profile/timeline 显式写最终参数，避免叠乘规则过早复杂化。

## 实现分批

### Phase 1: 静态 EnvironmentProfile + 阳光经济修正

**范围**：静态环境 profile、自然阳光开关、晴天/阴天/烈日倍率

**具体任务**：

- 新增 `BattleEnvironmentProfile`
- 新增 `BattleEnvironmentState`
- 新增 `EnvironmentRuleModule` 或注册 `environment.core`
- 第一版通过 `BattleRuleModule.params.environment_profile` 传入 profile，不新增 `BattleScenario.environment_profile`
- `BattleManager` 增加 `set_environment_state()` / `get_environment_state()`，由 `EnvironmentRuleModule` 在 setup/teardown 时注册和清理
- `BattleEconomyState` 暴露自然阳光开关、间隔倍率和值倍率 setter
- `BattleEconomyState` 新增 natural sun 调度器；现有 `sun_drop_entries` scheduled drops 保持不受环境影响
- 增加环境事件 `environment.changed`
- 提供样本 profile：
  - `env_day_sunny`
  - `env_day_cloudy`
  - `env_day_harsh_sun`
  - `env_night_clear`

**验证场景**：

- 晴天自然阳光与现有行为一致
- 阴天自然阳光掉落变慢或数值变小
- 烈日自然阳光掉落变快或数值变大
- 夜晚自然阳光关闭
- scenario `sun_drop_entries` 在阴天/夜晚仍按原始时间和值触发，不受自然阳光倍率影响
- `battle.get_environment_state()` 可返回当前环境状态
- 环境状态初始化后能发出 `environment.changed`

### Phase 2: 昼夜状态 + nocturnal sleep/wake

**范围**：固定白天/夜晚与可切换昼夜

**具体任务**：

- 收口 `nocturnal` 语义：资源默认不自带 `core.sleeping`；白天环境注入 sleeping，夜晚醒或保持活动
- 迁移现有 nocturnal mushroom 资源，移除默认 `core.sleeping` 或提供等价迁移桥接验证
- 复用现有 `entity.wake` 路径
- `StateComponent` 增加安全公共状态切换接口，或补充事件消费路径
- 支持 timeline 离散切换 `day -> dusk -> night -> dawn -> day`
- 切换时同步新生成实体状态

**验证场景**：

- 白天关：`nocturnal` 蘑菇初始为 sleeping，不发射攻击
- 夜间关：`nocturnal` 蘑菇初始为 awake/active，能正常触发攻击或生产
- 昼夜切换时，已存在与新生成的 nocturnal 实体状态一致
- Coffee Bean 通过 `entity.wake` 仍能唤醒目标

### Phase 3: FogRuleModule + 可见度查询

**范围**：浓雾 alpha 数据、清雾交互、视觉层接入

**具体任务**：

- `BattleEnvironmentProfile` 支持 `fog_start_column` / `fog_max_alpha`
- `BattleEnvironmentState` 提供 `fog_alpha_at_slot()`、`is_position_visible()` 和必要的 fog grid snapshot 查询
- 实现 `FogRuleModule` 或在 `environment.core` 中实现 fog 子状态
- `FogRuleModule` 维护 `base_fog_alpha_grid` / `clear_sources` / `final_fog_alpha_grid`
- 注册 `fog_clearer` tag 检测，将 Plantern/Torchwood 等实体登记为持续清雾源
- 新增 `clear_fog` effect，执行时只发布 `environment.fog_clear_requested` 事件
- `environment.fog_clear_requested` 支持 `radius` / `full_board` / `lane` / `column` 范围和 `duration_seconds`
- Blover 使用临时 `full_board` clear source，到期后自动恢复基础雾
- 视觉层通过 `VisualStageLayerService` 激活 fog overlay
- 第一版只让声明 `respect_visibility` 的 detection 受雾影响，避免大范围行为回归

**验证场景**：

- 迷雾覆盖范围正确
- fog alpha 随列变化正确
- Plantern/Torchwood 注册持续清雾源，实体移除后雾能恢复
- Blover 临时全屏清除迷雾并在到期后恢复
- 多个清雾源重叠时 final alpha 取最大清除强度，结果稳定
- `clear_fog` effect 只发布 `environment.fog_clear_requested`，不直接修改 fog state
- 声明 `respect_visibility` 的 detection 会过滤雾中目标；未声明的 detection 保持现有行为
- 雾中实体仍能被碰撞/伤害，只是目标发现或视觉受限

### Phase 4: 局部天气与环境事件（延后）

**范围**：区域环境、短时天气、未来扩展

**设计预留**：

- 局部浓雾、短时烈日、局部阴影可作为 environment runtime modifier
- 随机天气必须走 battle_seed 派生的确定性随机协议
- 不实现真实气候模拟
- 不让天气直接改棋盘 slot；如需改变可放置性，走 BoardSlot modifier / GridItem 体系

## 需要补充的能力

### BattleEnvironmentProfile / BattleEnvironmentState

- 新增 Resource 和运行时状态对象
- Profile 做数据声明，State 做当前状态查询
- State 不直接持有视觉节点或实体引用，避免环境层变成万能宿主
- 第一版 profile 通过 `BattleRuleModule.params.environment_profile` 引用

### BattleModeHost / BattleModeModuleRegistry

- 注册 `environment.core`
- 支持通过 `BattleModeDef.rule_modules` 或 `BattleScenario.mode_rule_modules` 加载环境模块
- 不新增 `BattleScenario.environment_profile` 一等字段；通过覆盖同名 `environment.core` rule module 实现 scenario 级 profile 替换
- 明确同一个棋盘 preset 可组合不同 environment profile

### BattleManager

- 增加 `set_environment_state(state: BattleEnvironmentState)` 和 `get_environment_state()`
- `EnvironmentRuleModule.on_mode_setup()` 注册 state，`on_mode_teardown()` 清理 state
- 其他系统只通过 `battle.get_environment_state()` 查询，不直接依赖 `BattleModeHost` runtime 字典

### BattleEconomyState

- 暴露自然阳光开关、间隔倍率和值倍率 setter
- 新增 natural sun 调度器，与现有 `sun_drop_entries` scheduled drops 分离
- 当 `natural_sun_enabled == false` 时，`_on_game_tick()` 只跳过 natural sun，不跳过 scheduled drops
- `_process_scheduled_drops(game_time)` 保持现有语义，不读取环境倍率
- `_process_natural_sun_drops(game_time)` 使用 `natural_sun_interval_scale` / `natural_sun_value_scale`
- 自然掉落建议使用 `source_type = &"natural_sun"`，便于验证和统计
- 倍率必须有 validator 下限，避免 0 或负数导致异常

### StateComponent

- 提供安全的状态切换入口，供环境模块让 nocturnal 在白天进入 `sleeping`
- 复用 `entity.wake` 事件，避免新增重复唤醒语义
- nocturnal 资源默认不自带 `core.sleeping`；白天睡眠由环境模块负责
- 不允许环境模块直接改组件私有字段

### Detection / SpatialIndex

- detection 可以显式查询 `BattleEnvironmentState.is_position_visible()`
- `SpatialIndex` 仍负责实体空间索引，不负责雾过滤
- 第一版通过 detection params 的 `respect_visibility` 显式启用雾过滤，避免全局改写所有 detection 行为

### FogRuleModule

- 维护 `base_fog_alpha_grid`、`clear_sources`、`final_fog_alpha_grid`
- 监听 `entity.spawned` / `entity.removed`，将 `fog_clearer` 实体注册/移除为持续清雾源
- 监听 `environment.fog_clear_requested`，创建半径、整板、行或列范围的清雾源
- 使用 `expires_at_tick` 管理临时清雾源，所有过期判断走固定 tick
- final alpha 由 base alpha 与 clear sources 重算得到，不允许外部直接写 final grid
- 清雾源重叠时第一版取最大 `strength`，不做复杂叠乘或叠加

### VisualStageLayerService / AudioCueRegistry

- 视觉层支持 environment theme 和 fog alpha grid
- 音频层支持 environment cue
- 视觉和音频不能反向影响战斗逻辑

### EffectRegistry

- 若选择 effect 路径，新增 `clear_fog` effect
- `clear_fog` 只发布 `environment.fog_clear_requested`，不直接修改 fog state
- `clear_fog` 支持 `scope`、`radius_slots`、`duration_seconds`、`strength` 等参数，第一版 `strength` 固定或校验为 `1.0`
- Fog state 修改集中在环境模块

## 不做的事

1. **不修改棋盘 slot**：环境系统不直接修改 `slot_type`、`placement_tags` 或占位角色。
2. **不新增 Mechanic family**：环境行为通过 RuleModule、EffectRegistry 或已有事件系统接入。
3. **不使用 Timer 或真实时间驱动规则**：动态环境变化走 100Hz 固定 tick。
4. **不让视觉影响战斗结果**：Tween、遮罩、粒子和音效只消费环境状态，不产生规则效果。
5. **不做完整气候模拟**：第一版只做光照、天气条件、阳光经济和可见度。
6. **不硬编码具体实体逻辑**：不写 `MushroomSleepLogic`、`PlanternFogLogic` 这类实体专属模块；通过 tag、effect 和事件组合。
7. **不实现非确定性随机天气**：未来随机天气必须走 battle_seed 派生链。

## 决策记录

### D1: 环境系统与棋盘系统正交拆分

- **问题**：晴天、阴天、夜晚、浓雾等是否应该放在棋盘草稿中？
- **决策**：拆分为独立环境草稿和独立运行时环境层。
- **理由**：
  - 棋盘是空间结构，环境是时间变化和全局/区域条件
  - 同一棋盘可组合多种环境
  - 测试矩阵更清晰
- **后果**：`BattleScenario / BattleModeDef` 需要能同时引用 `battlefield_preset` 和 `environment_profile`。

### D2: 第一版使用 EnvironmentProfile + RuleModule，不新增 autoload

- **问题**：环境系统应该是全局单例还是战斗实例内模块？
- **决策**：使用 `BattleEnvironmentProfile + BattleEnvironmentState + EnvironmentRuleModule`。
- **理由**：
  - 环境状态属于单场战斗
  - 现有 `BattleRuleModule` 已能承载模式规则
  - 不增加全局生命周期复杂度
- **后果**：第一版通过 `BattleRuleModule.params.environment_profile` 传入 profile，不新增 `BattleScenario.environment_profile` 一等字段；需要通过 `BattleManager.get_environment_state()` 给其他系统提供查询入口。

### D3: 天气先表现为阳光经济修正

- **问题**：晴天、阴天、烈日第一版应该影响什么？
- **决策**：只影响自然阳光开关、掉落间隔和值倍率，视觉/音频作为可选表现。
- **理由**：
  - 玩法收益明确
  - 对现有系统侵入小
  - 避免过早引入复杂天气模拟
- **后果**：雨天、风、温度等复杂效果延后。

### D3.1: scheduled drops 与 natural drops 分离

- **问题**：天气倍率是否应该影响现有 `BattleScenario.sun_drop_entries`？
- **决策**：不影响。`sun_drop_entries` 是 scheduled/scripted drops，用于验证、剧情和固定事件；天气只影响新增的 natural sun 调度器。
- **理由**：
  - 当前代码没有自然天降阳光循环，`sun_drop_entries` 是精确时间表
  - 直接修改 scheduled drops 会破坏验证场景的确定断言
  - 夜晚关闭自然阳光时，仍应允许脚本化阳光事件存在
- **后果**：`BattleEconomyState` 需要新增 `_process_natural_sun_drops()`，并保持 `_process_scheduled_drops()` 现有语义不变。

### D4: 动态环境切换由固定 tick 驱动

- **问题**：昼夜交替和天气变化是否可以用 Godot Timer 或真实时间？
- **决策**：不可以。必须走 `GameState` 仿真 tick。
- **理由**：
  - 保持确定性
  - 符合项目时间协议
  - 验证场景可稳定断言
- **后果**：timeline entry 使用仿真秒或 tick，模块在 `on_game_tick()` 中推进。

### D5: 浓雾是可见度系统，不是棋盘地形

- **问题**：浓雾是否应该改棋盘格？
- **决策**：不改。浓雾提供 `fog_alpha` 和可见度查询。
- **理由**：
  - 雾中实体仍存在并参与碰撞/伤害
  - 视觉遮罩和目标发现是环境行为
  - 不污染 `BoardSlot` 的放置语义
- **后果**：detection 需要显式接入环境可见度查询。

### D6: nocturnal 语义收口为白天睡、夜晚醒

- **问题**：夜晚是否让蘑菇睡眠？
- **决策**：不。`nocturnal` 植物资源默认不再自带 `core.sleeping`；白天由环境模块注入 `sleeping`，夜晚应可活动。
- **理由**：
  - 符合原版直觉
  - 与当前“夜间不强制蘑菇睡眠”的修订一致
  - Coffee Bean 唤醒路径可复用 `entity.wake`
- **后果**：现有蘑菇资源的 `core.sleeping` 语义需要迁移；实现期必须补白天注入 sleeping 和夜间保持/恢复 awake 的验证。

### D7: 清雾通过 clear source 分层维护

- **问题**：Plantern、Torchwood、Blover 是否可以直接修改 fog alpha grid？
- **决策**：不可以。FogRuleModule 统一维护 `base_fog_alpha_grid`、`clear_sources` 和 `final_fog_alpha_grid`。实体和效果只能注册清雾源或发送 `environment.fog_clear_requested`。
- **理由**：
  - 清雾是环境状态变更，不是实体 status 或棋盘 slot 修改
  - 持续清雾源消失后可以自然恢复基础雾
  - 临时清雾源可用 `expires_at_tick` 确定性到期
  - 多个清雾源重叠时可用最大 strength 保持第一版规则简单稳定
- **后果**：需要为 `clear_sources` 定义稳定 ID、生命周期和重算时机；`clear_fog` effect 只发布请求事件，不直接写 fog state。

### D8: Fog 延后到 Phase 3

- **问题**：Fog 是否进入第一阶段实现？
- **决策**：不进入。Phase 1 聚焦 EnvironmentProfile、EnvironmentState 和自然阳光经济；FogRuleModule、清雾源和 detection 可见度过滤放到 Phase 3。
- **理由**：
  - Fog 涉及视觉层、detection 和清雾 effect，影响面明显大于阳光经济
  - 先完成环境状态与经济修正，可降低首轮实现风险
  - Fog 依赖 `BattleEnvironmentState` 查询入口，适合作为后续使用者验证
- **后果**：Phase 1 profile 可以声明 fog 字段但不启用 fog 规则；验证只覆盖阳光和基础环境状态。

## 当前建议

下一步最小行动不是直接实现完整天气系统，而是先确认 5 个 spike：

1. **环境 profile 样本**：写出 `env_day_sunny`、`env_day_cloudy`、`env_day_harsh_sun`、`env_night_clear` 四个样本，确认字段足够。
2. **经济修正 spike**：让 `BattleEconomyState` 支持 natural sun 调度器、自然阳光开关、间隔倍率和值倍率，并验证 scheduled drops 不受天气影响。
3. **昼夜/nocturnal spike**：迁移 nocturnal 资源默认 sleeping 语义，确认白天环境注入 sleeping、夜晚醒或保持活动所需的最小 StateComponent 接口。
4. **timeline spike**：用固定 tick 驱动一次 `day -> night -> day` 切换，并发出 `environment.changed`。
5. **fog clear source spike**：在视觉层做最小 fog alpha grid，验证 `base_fog_alpha_grid + clear_sources + final_fog_alpha_grid`、Plantern/Torchwood 持续清雾源和 Blover 临时全屏清雾源。

如果这 5 项确认完成，再进入 Phase 1 的工程实现。
