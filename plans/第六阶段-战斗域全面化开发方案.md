# 第六阶段-战斗域全面化开发方案

> 这份方案用于承接第五阶段核心收口之后的正式开发主线。它回答的不是“还能再验证什么”，而是“在当前引擎主干已经证明可行之后，接下来应如何把战斗域从一套可表达的规则主干，推进成一套真正可持续扩张、可沉淀正式内容、可支撑长期玩法设计的战斗系统”。

---

## 文档定位

这篇文档主要回答：

- 为什么第六阶段应该从“验证可行性”切换到“战斗域全面化开发”
- 第六阶段的正式开发目标到底是什么
- 第六阶段应该优先展开哪些战斗域内容
- 哪些内容属于本阶段主线，哪些仍然不应抢占优先级
- 第六阶段应产出哪些正式能力、正式内容和正式边界
- 这一阶段结束时，项目应达到什么状态

这篇文档不负责：

- 代替完整项目路线图
- 重新回到协议冻结主线
- 提前展开完整外循环系统
- 承诺最终版社区平台、工坊或编辑器产品
- 一次性定义所有未来玩法、数值和内容宇宙

## 一句话结论

第六阶段不是继续证明“当前主干还能不能再多表达几种效果”，而是：

> 以当前已经稳定的运行时、模板主链、扩展入口和验证体系为基础，正式进入战斗域全面化开发，开始系统补齐植物、僵尸、规则能力、战场语义、波次结构和正式关卡内容，让项目从“能表达”走向“能支撑完整战斗内容开发”。

---

## 当前阶段切换判断

第五阶段当前已经证明了下面几件事：

1. 错误技样例已经能够建立在统一主干之上。
2. 扩展包入口和扩展 effect 已经形成正式闭环。
3. 协议守卫、回归矩阵和调试工作流已经足够支撑持续开发。
4. `BattleManager` 的第一轮代码级收口已经完成，主干编排边界开始明确。

这意味着当前项目最紧缺的，已经不再是：

- 还能不能再做一个新 effect 原子
- 还能不能再做一个更复杂的 showcase
- 是否还要继续围绕基础协议做一轮抽象修饰

当前真正紧缺的是：

- 更完整的植物内容结构
- 更完整的僵尸内容结构
- 更完整的交互矩阵
- 更完整的战场语义与关卡结构
- 一套真正面向长期开发的战斗内容版图

因此，第六阶段的主线应明确切换为：

> 正式补齐战斗域，而不是继续以样例和协议为主线。

---

## 为什么第六阶段不应优先做外围

第六阶段当前不应优先切到：

- 完整选关 UI
- 完整存档
- 图鉴系统
- 商店
- 花园

原因不是这些系统不重要，而是：

1. 现在最缺的不是“战斗外怎么组织”，而是“战斗内到底有没有足够完整的正式内容版图”。
2. 如果战斗域内容还不够全面，外循环系统只会把当前内容稀薄的问题暴露得更明显。
3. 外循环应该建立在更稳定的关卡内容、单位 roster、战斗目标和地图语义之上。

所以第六阶段当前更合理的策略不是“跳过战斗域去做产品壳”，而是：

> 先把战斗域做成真正的正式开发对象，再让外围系统来组织它。

---

## 第六阶段的正式目标

第六阶段的目标不是“再做一批样例”，而是完成下面 5 个正式化结果。

### 1. 正式植物内容版图

项目应不再只有若干验证样例植物，而是开始形成正式的植物内容谱系，例如：

- 输出型
- 资源型
- 防御型
- 控制型
- 召唤型
- 支援 / 光环型

### 2. 正式僵尸内容版图

项目应不再只有若干验证样例僵尸，而是开始形成正式的敌方压力结构，例如：

- 基础推进型
- 快速压迫型
- 高血 / 护甲型
- 特殊攻击型
- 空中型
- 干扰 / 支援型

### 3. 正式战斗交互矩阵

项目应开始明确：

- 哪些植物克制哪些敌方能力
- 哪些敌方能力可以压制哪些布阵
- 状态、标记、击退、召唤、光环、爆炸、穿透、分裂等能力如何进入正式对抗关系

### 4. 正式战场语义和关卡语义

项目应不再只依赖“一个通用平地 demo 场景”，而是开始形成：

- 不同 slot type 的实际意义
- 不同 lane / board 结构对内容选择的影响
- 不同 battle goal / defeat condition 的差异

### 5. 正式战斗内容套件

项目应开始沉淀一组真正面向后续外循环的战斗内容，例如：

- 一批正式关卡模板
- 一批正式波次结构模板
- 一批正式内容组合基线

---

## 第六阶段不负责什么

为了防止阶段目标失焦，第六阶段明确不负责下面这些事：

1. 不继续把“新 effect 原子数量”当成阶段主 KPI。
2. 不把 showcase 数量当成内容深度替代品。
3. 不优先做完整外循环 UI。
4. 不把商店、花园、图鉴、社区平台拉进当前主线。
5. 不回到“先抽象、后内容”的模式。
6. 不为了战斗域全面化而重写当前主干。

---

## 第六阶段的开发原则

### 1. 内容优先于抽象

从第六阶段开始，应坚持：

> 先用当前主干认真做内容；只有在内容真的被主干卡住时，才回头补抽象。

### 2. 抽象服务正式内容，而不是服务 showcase

以后新增：

- 事件
- effect
- trigger
- 模板字段
- manifest 字段

都应首先回答：

- 它是否真的被正式战斗内容需要
- 它是否在多个内容点上重复出现
- 它是否值得成为长期能力位

### 3. 优先做战斗内完整性，而不是外围完整性

第六阶段优先补的是：

- 植物 / 僵尸谱系
- 互动矩阵
- 地图 / 波次 /目标

不是：

- 外围菜单
- 经济系统壳层
- 存档 UI

### 4. 新内容默认走模板和内容链，不走业务特判

第六阶段开始后，新增正式内容默认应通过：

- `EntityTemplate`
- `ProjectileTemplate`
- `TriggerBinding`
- `BattleScenario`
- `WaveDef`
- 扩展包（如适合）

而不是继续往 `BattleManager` 塞内容级分支。

### 5. 每个新内容族都必须可验证

第六阶段不是“先做内容，回头再补验证”。

每个正式内容族都应同时带来：

- 正向验证
- 关键交互验证
- 必要时的负向守卫

---

## 第六阶段的五条建设轴

### A. 植物内容轴

这条轴回答：

- 玩家有哪些正式手段
- 这些手段的定位是否清晰
- 它们之间是否形成选择空间

建议优先补齐的植物大类：

1. 基础输出型
2. 连续输出 / 高频输出型
3. 抛投 / 远距覆盖型
4. 资源生产型
5. 防御 / 拦截型
6. 控制型
7. 支援 / 光环型
8. 召唤 / 分体型

当前阶段要求：

- 不追求一开始就做很多数量
- 先让每个大类至少有正式代表
- 每个大类都应明确它的战术职责

### B. 僵尸内容轴

这条轴回答：

- 敌方压力从哪里来
- 玩家为什么需要不同类型的植物
- 战局节奏如何变得真实而不是单一

建议优先补齐的僵尸大类：

1. 基础推进型
2. 快速冲压型
3. 高血 / 护甲型
4. 远程或特殊攻击型
5. 空中型
6. 支援 / 召唤 / 干扰型

当前阶段要求：

- 每一类都应给战局引入新的压力维度
- 不只是“换一个数值模板”
- 要让玩家内容谱系真的有对抗意义

### C. 交互矩阵轴

这条轴回答：

- 内容之间如何发生正式对抗
- 哪些关系是克制，哪些关系是压制，哪些关系是交换

当前建议重点补齐：

- 状态与移动的对抗
- 高度与目标选择的对抗
- 护甲 / 高血与爆发 / 持续输出的对抗
- 召唤 / 分裂与范围 / 跳链的对抗
- 支援 / 光环与推进压力的对抗

第六阶段要求：

- 不只是做“新单位”
- 而是同步整理“这些单位为什么存在”

### D. 战场语义轴

这条轴回答：

- 地图和棋盘为什么会影响内容选择
- 不同 slot / lane / board 结构如何改变战斗

建议优先补齐：

- `ground / water / roof / air_only` 的正式战术意义
- `surface / primary / cover / blocker` 的实际布阵意义
- 场上物件、障碍、容器的进一步接入策略

第六阶段要求：

- 不再只把 board 视为一个放置数组
- 而是开始让地图语义参与内容设计

### E. 波次与关卡内容轴

这条轴回答：

- 正式战斗内容如何组织
- 玩家面对的不是单个样例，而是一整局设计

建议优先补齐：

- 正式 `WaveDef` 结构族
- 多种 battle goal
- 多种 defeat condition
- 一组正式关卡模板

第六阶段要求：

- 不再只验证单点能力
- 而是验证完整战斗体验的结构

---

## 第六阶段建议的建设顺序

### 第一步：补齐正式植物 / 僵尸基线 roster

优先做什么：

- 每个大类至少 1 个正式代表
- 不先追求数量，而先追求“谱系完整性”

目标：

- 让玩家与敌方都具备基本战术分层

### 第二步：补齐正式交互矩阵

优先做什么：

- 植物对不同僵尸压力的响应
- 状态、标记、召唤、范围伤害、击退等正式战斗关系

目标：

- 让单位不是平铺的，而是相互形成对抗结构

### 第三步：把地图 / 棋盘语义真正接入内容设计

优先做什么：

- 让 `water / roof / air_only / cover / blocker` 不只是验证项
- 而是成为正式内容设计的约束

目标：

- 让地图差异真正影响内容选择

### 第四步：补正式波次结构和关卡模板

优先做什么：

- 基础波次模板
- 压迫型波次模板
- 护甲 / 快速 / 空中 / 支援混合波次模板

目标：

- 让战斗从“样例拼盘”变成“正式关卡内容”

### 第五步：做第一批正式战斗内容套件

优先做什么：

- 一组正式战斗模板
- 一组正式对抗组合
- 一组面向后续外循环的可复用内容包

目标：

- 为第七阶段及外循环系统提供真实内容基础

---

## 第六阶段建议的正式输出

这一阶段结束时，建议至少形成下面 6 组输出。

### 1. 正式植物 roster v1

要求：

- 不是临时样例集合
- 而是按职责划分的正式单位集

### 2. 正式僵尸 roster v1

要求：

- 不是若干 dummy 和 tank 的扩写
- 而是带有明确压力结构的敌方谱系

### 3. 正式交互矩阵 v1

要求：

- 能回答主要单位族之间的对抗关系
- 能支持后续平衡和关卡设计

### 4. 正式战场语义说明 v1

要求：

- 地图 / slot / role 语义开始真正影响内容生产

### 5. 正式波次 / 关卡模板集 v1

要求：

- 不再只是一组验证场景
- 而是一组可以用于正式战斗内容建设的模板

### 6. 正式战斗内容回归区块 v1

要求：

- 新增内容不只是能跑
- 而是进入稳定回归

---

## 第六阶段的边界提醒

当前最容易失衡的地方有 4 个。

### 1. 继续回到 showcase 驱动

风险：

- 看起来内容很多
- 实际上仍然只是演示素材

### 2. 继续回到协议驱动

风险：

- 内容还没全面展开
- 又把主线拉回抽象打磨

### 3. 直接跳去外围产品层

风险：

- 战斗内容底盘还不够厚
- 外循环只能围绕稀薄内容打转

### 4. 用数值堆叠代替内容体系

风险：

- 单位数量增长了
- 但大类职责和交互关系依然不清楚

---

## 第六阶段不应优先做的事

当前不建议优先把精力放在：

1. 新一轮大规模协议冻结
2. 继续扩 showcase 分组
3. 完整选关与存档 UI
4. 完整商店和花园
5. 随机错误技生成器
6. 社区工坊和分享平台

---

## 阶段退出条件

当第六阶段结束时，建议至少满足下面条件：

### A. 内容结构条件

- 植物正式大类已经成形
- 僵尸正式大类已经成形
- 至少一轮交互矩阵整理已经成立

### B. 战斗结构条件

- 地图 / slot / board 语义已进入正式内容设计
- 波次和 battle goal 不再只是最小闭环
- 已有一批正式关卡模板

### C. 工程条件

- 新正式内容仍以模板和内容链接入
- 新正式内容都进入分层回归
- 没有明显回退到 `BattleManager` 内容特判

### D. 阶段判断条件

- 项目已经不再只是“规则引擎 + 样例”
- 而是已经开始具备“正式战斗内容开发平台”的形态

---

## 和第五阶段的关系

第五阶段解决的是：

- 能不能表达
- 能不能扩展
- 能不能验证

第六阶段解决的是：

- 能不能做出完整战斗内容版图
- 能不能形成正式战斗内容开发主线

所以这两个阶段不是重复关系，而是前后关系：

> 第五阶段证明这套主干可行，第六阶段开始认真用它做完整战斗内容。

第六阶段不是外围产品化阶段，也不是再做一轮协议打磨阶段，而是：

> 当前引擎主干已经被证明可行之后，正式进入战斗域全面化开发阶段，开始用这套主干认真构建完整战斗内容。
