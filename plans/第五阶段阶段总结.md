## 一句话结论

第五阶段已经从“先做几组手工错误技样例”推进到“错误技样例、受控扩展入口、扩展 effect 家族、守卫专项和可视化入口均已形成正式闭环”。

当前已经完成的核心结果包括：

1. 第一批正式错误技样例已经进入统一验证体系。
2. 最小扩展入口已经支持：
   - `EntityTemplate`
   - `ProjectileTemplate`
   - `TriggerBinding`
   - `EffectDef`
   - effect `strategy`
3. 扩展包已经不只是“加模板”，而是已经能新增一批真正的 effect 原子能力。
4. 扩展 effect 主链已经有注册期和运行期守卫专项。
5. 第五阶段样例已经有统一 showcase 分组，而不只是验证资源。

因此当前项目已经不再处于“第五阶段起步讨论期”，而是：

> 第五阶段核心起步收口已经完成，后续主线应转入“继续扩扩展 effect 家族与工具深度”，而不是再回退到主干补丁式开发。

## 第五阶段完成项总览

### 1. 正式错误技样例库

当前已落地 7 个正式错误技验证场景：

- `reactive_retaliation_chaos_validation`
- `death_blossom_chaos_validation`
- `tracking_barrage_chaos_validation`
- `chain_explosion_cascade_validation`
- `splash_zone_cascade_validation`
- `fast_pursuit_cascade_validation`
- `multi_lane_retaliation_cascade_validation`

已成立的能力：

- 受伤反击
- 死亡爆炸
- 空地追踪
- 多源级联
- 多目标溅射
- 高速追踪
- 多 lane 反击隔离

这些样例当前都证明了一件事：

> 错误技差异主要来自模板、触发器、效果和资源组合，而不是新增业务脚本。

### 2. 扩展入口最小主链

当前最小扩展入口已正式支持：

- 验证场景
- `EntityTemplate`
- `ProjectileTemplate`
- `TriggerBinding`
- `EffectDef`
- effect `strategy`

当前目录约定已经稳定为：

```text
extensions/<pack_id>/
  data/combat/
    effects/
    entity_templates/
    projectile_templates/
    trigger_bindings/
  scripts/effects/
  scenes/validation/
```

对应实现入口：

- [SceneRegistry.gd](../autoload/SceneRegistry.gd)
- [EffectRegistry.gd](../autoload/EffectRegistry.gd)

已成立的能力：

- 主仓会自动扫描扩展模板、绑定和验证场景
- `EffectRegistry` 会自动扫描扩展 `EffectDef`
- 带 `strategy_script` 的扩展效果可进入统一执行链
- 扩展内容复用主仓运行时，不需要改主干代码

### 3. 扩展 effect 家族

当前已经正式落地并进入回归的扩展 effect 家族包括：

#### A. 命中响应类

- `split_projectiles`
- `chain_bounce`

这类效果主要验证：

- `spawn_projectile.on_hit` 不再只局限于内置名字表
- hit-response 扩展位已经能承接新的 follow-up 逻辑

#### B. 生成类

- `spawn_entity`

这类效果主要验证：

- 扩展包已经能把新实体接回共享 spawn 主链
- 扩展入口不再只是静态内容挂载

#### C. 控制类

- `apply_status`
- `knockback`
- `aura`

这类效果主要验证：

- 扩展 effect 已经不只会造成伤害
- 扩展能力能直接复用现有状态系统与实体运行时

### 4. 守卫专项

第五阶段当前已形成两类正式守卫：

#### 1. 扩展包正常路径 smoke test

- `extension_pack_smoke_validation`

作用：

- 证明外部内容包可以加载、注册并进入统一验证主链

#### 2. 扩展 effect 守卫专项

- `extension_effect_guardrail_validation`

作用：

- 拦截重复 `effect_id`
- 拦截缺失 `effect_id`
- 拦截非法参数 schema
- 拦截缺失 `execute()`
- 拦截 slot tag 不匹配
- 拦截无 strategy 的 effect
- 拦截运行前的坏参数类型

这意味着第五阶段当前已经不只是“扩展能跑”，而是：

> 扩展 effect 主链已经具备正式守卫和回归位置。

### 5. Showcase 与体验层

当前主界面已新增“第五阶段错误技”分组，内含：

- 4 个正式级联样例 showcase

当前已落地 showcase 包括：

## 当前支持的扩展 effect 家族

按能力语义看，当前已经支持的扩展 effect 家族可整理为下面 6 组。

| 家族 | 当前 effect | 主要语义 |
|------|------|------|
| hit-response | `split_projectiles` | 命中后分裂 follow-up 投射物 |
| summon | `spawn_entity` | 生成新的运行时实体 |
| status-control | `apply_status` | 施加共享状态系统内的控制效果 |
| displacement | `knockback` | 对目标施加最小受控位移 |
| bounce-damage | `chain_bounce` | 在邻近目标间跳链伤害并衰减 |
| aura-control | `aura` | 周期扫描范围内目标并持续施加控制 |

这 5 组的意义在于：

- 已经跨出了“只有伤害和爆炸”的阶段
- 已经覆盖了：
  - 直接伤害后续
  - 生成
  - 控制
  - 位移
  - 多目标链路
  - 持续范围控制

也就是说，第五阶段当前已经初步证明：

> 扩展入口可以承接“新效果原子”，而不只是继续组合旧的内置效果。

## 当前仍未覆盖的边界

第五阶段核心收口已经完成，不代表扩展生态已经完整。当前仍明确未覆盖的边界主要包括：

### 1. 持续型与区域型扩展效果

当前还没有正式落地：

- `aura`
- `mark`
- 持续区块伤害 / 持续区块控制

### 2. 延迟型与时序型扩展效果

当前还没有正式落地：

- `delayed_trigger`
- `delayed_explode`
- 倒计时型效果链

### 3. 更复杂的位移与空间效果

当前 `knockback` 仍是最小水平位移版本，还没有覆盖：

- 多轴位移
- 推挤碰撞
- 吸附 / 拉拽
- 地块交互型位移

### 4. 扩展包管理体验

当前仍未进入主线：

- 扩展包 UI
- 扩展包启停管理
- 扩展包版本兼容 UI
- 最终版工坊 / 分享平台

### 5. 错误技随机生成

当前仍明确后置：

- 完整随机错误技生成器
- 大规模组合空间搜索
- 面向作者的生成调参工具

换句话说，第五阶段当前已经把“能不能扩”验证清楚了，但还没有把“扩展生态最终怎么管理”做完。

---

## 当前最合理的下一步

如果继续顺着第五阶段主线推进，当前最合理的后续顺序是：

1. 继续扩下一批 effect 家族：
   - `aura / mark / delayed_trigger`
2. 继续补扩展 effect 守卫矩阵：
   - 更多 slot capability
   - 更多参数非法组合
3. 再考虑更完整的扩展包管理体验

这里的顺序仍然不能颠倒：

- 先继续扩 effect 家族与守卫
- 再做扩展管理体验
- 最后才考虑更完整生态
