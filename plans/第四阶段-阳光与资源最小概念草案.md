# 第四阶段-阳光与资源最小概念草案

> 这份草案用于回答第四阶段第一批里“资源闭环到底怎么起步”。它的目标不是现在就做完整经济系统，而是先把阳光在战斗里的归属、事件、路径和最小验证口径定下来。

---

## 文档定位

这篇文档主要回答：

- 战斗资源状态由谁持有
- 阳光产出、收集、消耗分别走什么路径
- 资源变化进入什么语义事件
- 最小正向和负向验证怎么定义

这篇文档不负责：

- 阳光动画和拾取表现
- 多资源体系
- 商店、选关和外循环经济

配套阅读建议：

- [第四阶段-战斗玩法层系统清单](./第四阶段-战斗玩法层系统清单.md)
- [第四阶段第一批可执行任务清单](./第四阶段第一批可执行任务清单.md)
- [第四阶段前置清单](./第四阶段前置清单.md)

## 一句话结论

第四阶段第一批建议引入独立的 `BattleEconomyState` 或等价轻量服务，作为战斗资源的权威状态承载；植物产阳光和场景掉落阳光都统一走“生成可收集阳光 -> 收集 -> 资源变化”的同一条路径；资源消耗发生在放置确认阶段，而不是选卡阶段。

---

## 最小概念模型

| 对象 | 职责 | 当前建议 |
|------|------|------|
| `BattleEconomyState` | 持有当前战斗资源权威状态 | 由战斗层持有，不挂在 UI，不散落在 `BattleManager` 私有变量里 |
| `SunCollectible` | 场上可收集阳光对象 | 天降阳光和植物产阳光共用同一类可收集对象 |
| `ResourceSpendRequest` | 资源消耗请求 | 由卡片/放置层发起，资源层只负责判断能否扣除并返回结果 |

当前最小权威数据建议包括：

- `current_sun`
- 最近一次资源变化的 `reason`
- 当前活跃可收集阳光的最小标识集合

当前不建议一开始就引入：

- 多资源字典宇宙
- 脱离战斗上下文的全局经济状态
- 直接由 UI 控制资源数字

---

## 资源状态归属

当前建议：

1. 资源状态归属到独立 `BattleEconomyState`，由战斗层创建和消费。
2. `BattleManager` 或后续战斗协调节点只负责调用资源服务，不直接成为资源数字的权威持有者。
3. UI 只读资源状态和事件，不拥有资源真值。

这样做的原因是：

- 卡片系统和资源系统会形成稳定依赖关系，不应靠 UI 回调串起来
- 后续验证需要稳定读取资源快照，不能依赖临时节点变量
- 能避免第四阶段一开始就回退成“大 manager 里加一个 `sun_value`”

---

## 统一事件语义

当前建议使用下面 4 个最小事件：

| 事件名 | 何时触发 | 关键字段建议 |
|------|------|------|
| `sun.spawned` | 一个可收集阳光对象进入战局时 | `sun_id`、`source_type`、`source_id`、`lane_id`、`value` |
| `sun.collected` | 阳光被玩家或收集逻辑吸收时 | `sun_id`、`collector_id`、`value` |
| `resource.changed` | 资源权威值发生变化时 | `resource_id`、`before`、`delta`、`after`、`reason` |
| `resource.spend_failed` | 扣费请求因资源不足被拒绝时 | `resource_id`、`cost`、`current`、`reason` |

当前建议：

- `resource_id` 在第一批中固定为 `sun`
- 扣费成功不额外发明 `resource.spent`，直接使用 `resource.changed` 且 `reason = spend`
- 资源事件继续沿用当前主干的语义事件风格，而不是发明“植物专属资源事件”

资源变化进入快照和验证日志时，至少应能观察到：

- 变化前数值
- 变化后数值
- 变化原因
- 关联请求或来源对象

## 三条最小路径

### 1. 植物被动产阳光

建议路径：

`periodic trigger -> sun.spawned -> 玩家收集 -> sun.collected -> resource.changed`

当前建议：

- 植物侧先只要求“能定期产出一个阳光对象”
- 不要求现在就做完整动画、磁吸和拾取轨迹

### 2. 场景掉落阳光

建议路径：

`战局定时或阶段逻辑 -> sun.spawned -> 玩家收集 -> sun.collected -> resource.changed`

当前建议：

- 场景掉落与植物产出共用同一种 `SunCollectible`
- 二者只在 `source_type` 上区分，例如 `plant_generated` / `sky_drop`

### 3. 资源消耗

建议路径：

`card play request -> board legality pass -> economy spend check -> resource.changed(reason=spend) -> placement commit`

当前建议：

1. 资源消耗不发生在“选中卡片”时。
2. 资源消耗发生在“放置合法性已通过，准备正式提交放置”时。
3. 如果资源不足，直接发出 `resource.spend_failed`，并终止后续放置和实体生成。

这样做的原因是：

- 非法格尝试不应先扣资源再回滚
- 卡片选择状态和资源真值应解耦
- 第一批不需要额外引入复杂退款语义

---

## 最小验证口径

### 最小正向验证

建议预留：

- `sun_resource_validation.tres`

至少覆盖下面 3 个观察点：

1. 一次植物产阳光或一次场景掉落能触发 `sun.spawned`
2. 收集后触发 `sun.collected -> resource.changed`
3. `resource.changed` 的 `before / delta / after` 与最终资源数值一致

### 最小负向验证

建议预留：

- 同一专项场景中的负向子用例，或单独 `sun_resource_guardrail_validation.tres`

至少覆盖下面 2 个观察点：

1. 资源不足时出现 `resource.spend_failed`
2. 失败后不会继续出现 `placement.accepted` 或 `entity.spawned`

---

## 当前明确不做的事

1. 不把阳光系统做成完整动画和表现工程。
2. 不在第一批就扩成多资源系统。
3. 不把资源状态放到 UI 或全局外循环里。
4. 不把扣费时机放到选卡阶段。

---

## 进入实现前必须具备

- 已明确 `BattleEconomyState` 或等价轻量服务的权威归属
- 已固定 `sun.spawned / sun.collected / resource.changed / resource.spend_failed` 或等价事件
- 已明确植物产出和场景掉落共用可收集阳光路径
- 已明确扣费发生在放置确认阶段
- 已在验证矩阵中预留 `sun_resource_validation` 位置
